Chapter 06 · Phase 3: React Fundamentals

State Management and Hooks

Make your React components interactive with state. Master useState for local state, useEffect for side effects, useContext for shared state, and learn to build custom hooks that encapsulate reusable logic across your application.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Components without state are static — they render the same output every time. State is what makes React applications interactive. When a user types in a form, clicks a button, or data arrives from an API, state changes trigger re-renders that update the UI.

Hooks are the mechanism React provides for adding state and side effects to functional components. Before hooks (React 16.8, February 2019), you needed class components for state. Today, hooks are the standard. They are simpler, more composable, and easier to test. Understanding hooks is understanding modern React.

Key Insight

State is not the same as props. Props come from the parent (read-only). State is owned by the component (read-write). When state changes, React re-renders the component and its children. This is the core of React's reactivity model.

Key Concepts

1. useState — Local Component State

useState is the most fundamental hook. It declares a state variable and returns the current value plus a function to update it. Every call to the setter triggers a re-render.

jsx
import { useState } from "react";

function Counter() {
  // Declare state: [currentValue, setterFunction] = useState(initialValue)
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}

// State with objects — always spread the previous state
function UserForm() {
  const [form, setForm] = useState({ name: "", email: "" });

  const handleChange = (e) => {
    setForm(prev => ({
      ...prev,                        // keep existing fields
      [e.target.name]: e.target.value  // update changed field
    }));
  };

  return (
    <form>
      <input name="name" value={form.name} onChange={handleChange} />
      <input name="email" value={form.email} onChange={handleChange} />
    </form>
  );
}

// State with arrays
function TodoList() {
  const [todos, setTodos] = useState([]);

  const addTodo = (text) => {
    setTodos(prev => [...prev, { id: Date.now(), text, done: false }]);
  };

  const toggleTodo = (id) => {
    setTodos(prev =>
      prev.map(t => t.id === id ? { ...t, done: !t.done } : t)
    );
  };

  const deleteTodo = (id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  };
  // ... render JSX
}
Never Mutate State Directly

Never do state.push(item) or state.name = "new". Always create a new object or array. React compares references to detect changes. If you mutate the existing reference, React does not know anything changed and will not re-render.

2. useEffect — Side Effects

useEffect runs code after the component renders. Use it for side effects: fetching data, setting up subscriptions, updating the document title, or interacting with browser APIs. The dependency array controls when the effect re-runs.

jsx
import { useState, useEffect } from "react";

function ArticleList() {
  const [articles, setArticles] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  // Fetch data on mount
  useEffect(() => {
    const fetchArticles = async () => {
      try {
        const res = await fetch("/api/v1/articles");
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        const data = await res.json();
        setArticles(data);
      } catch (err) {
        setError(err.message);
      } finally {
        setLoading(false);
      }
    };

    fetchArticles();
  }, []); // Empty array = run once on mount

  if (loading) return <Spinner />;
  if (error) return <ErrorMessage message={error} />;
  return <ArticleGrid articles={articles} />;
}

// Effect with dependencies — re-runs when searchTerm changes
function SearchResults({ searchTerm }) {
  const [results, setResults] = useState([]);

  useEffect(() => {
    if (!searchTerm) return;

    const controller = new AbortController();

    fetch(`/api/v1/search?q=${searchTerm}`, { signal: controller.signal })
      .then(res => res.json())
      .then(setResults)
      .catch(() => {}); // ignore abort errors

    // Cleanup function — runs before next effect or unmount
    return () => controller.abort();
  }, [searchTerm]); // Re-run when searchTerm changes

  return <ResultList results={results} />;
}

3. useContext — Shared State Without Prop Drilling

When multiple components need the same state (like the current user or theme), passing it through props at every level becomes tedious ("prop drilling"). Context provides a way to share values across the component tree without explicit props.

jsx
import { createContext, useContext, useState } from "react";

// 1. Create the context
const AuthContext = createContext(null);

// 2. Create a provider component
function AuthProvider({ children }) {
  const [user, setUser] = useState(null);

  const login = async (email, password) => {
    const res = await fetch("/api/v1/session", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ email, password }),
    });
    const data = await res.json();
    setUser(data.user);
  };

  const logout = () => setUser(null);

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// 3. Use the context in any child component
function Navbar() {
  const { user, logout } = useContext(AuthContext);

  return (
    <nav>
      {user ? (
        <>
          <span>Welcome, {user.name}</span>
          <button onClick={logout}>Logout</button>
        </>
      ) : (
        <a href="/login">Login</a>
      )}
    </nav>
  );
}

4. Custom Hooks — Reusable Logic

Custom hooks extract stateful logic into reusable functions. Any function that starts with use and calls other hooks is a custom hook. They are the React equivalent of Ruby modules — shared behavior without inheritance.

jsx
// hooks/useApi.js — reusable data fetching hook
import { useState, useEffect } from "react";

export function useApi(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);

    fetch(url, { signal: controller.signal })
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then(setData)
      .catch(err => {
        if (err.name !== "AbortError") setError(err.message);
      })
      .finally(() => setLoading(false));

    return () => controller.abort();
  }, [url]);

  return { data, loading, error };
}

// Usage — clean, declarative components
function ArticlesPage() {
  const { data: articles, loading, error } = useApi("/api/v1/articles");

  if (loading) return <Spinner />;
  if (error) return <ErrorMessage message={error} />;
  return <ArticleList articles={articles} />;
}

// hooks/useLocalStorage.js — persist state to localStorage
export function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    const stored = localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initialValue;
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}
Pro Tip

If you find yourself copying the same useState + useEffect pattern across multiple components, extract it into a custom hook. Good hooks to create early: useApi (data fetching), useForm (form handling), useDebounce (delayed input), and useLocalStorage (persisted state).

5. Rules of Hooks

Hooks have two strict rules that must never be violated. Breaking these rules causes bugs that are extremely hard to debug because React relies on the call order of hooks being stable between renders.

  Rules of Hooks
  ══════════════

  1. Only call hooks at the TOP LEVEL
     ✓  const [count, setCount] = useState(0);
     ✗  if (condition) { const [x, setX] = useState(0); }
     ✗  for (let i = 0; i < n; i++) { useState(0); }

  2. Only call hooks from REACT FUNCTIONS
     ✓  Inside functional components
     ✓  Inside custom hooks (functions starting with "use")
     ✗  Inside regular JavaScript functions
     ✗  Inside class components
     ✗  Inside event handlers (call the setter, not the hook)
Figure 1: The two rules of hooks — violating them causes unpredictable behavior

Self-Check

Can you explain the useEffect dependency array?

Empty array [] = run once on mount. No array = run after every render (usually a bug). Array with values [a, b] = run when a or b changes. Getting this wrong causes infinite loops or stale data.

Do you know when to use state vs props vs context?

State: data that changes within a component (form inputs, toggles). Props: data passed from parent to child (configuration, callbacks). Context: data needed by many components at different nesting levels (current user, theme).

Can you write a custom hook from scratch?

Write a useWindowSize hook that returns the current window width and height, updating on resize. If you can do this in under 10 minutes including cleanup, you understand hooks well enough to move on.
