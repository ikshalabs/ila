Chapter 06 · Phase 2: React Front-End Development

State Management and Hooks

Master React's built-in hooks for managing component state, side effects, and shared context — useState, useEffect, useContext, useReducer, and the art of building custom hooks. This is how React components come alive.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

In Chapter 05, you built components that receive data through props and render static UI. But real applications are interactive: users click buttons, type in forms, toggle menus, and trigger data fetches. State is what makes your components respond to these interactions. When state changes, React automatically re-renders only the affected components, keeping the UI in sync with the data without any manual DOM manipulation.

React Hooks are the modern API for managing state and side effects in functional components. Before hooks (introduced in React 16.8), you needed class components for anything stateful. Hooks eliminated that requirement, making functional components the standard for all React development. Understanding hooks deeply — not just the syntax, but the mental model of how React tracks state across renders — is the key to writing bug-free React code.

The hooks covered in this chapter (useState, useEffect, useContext, useReducer) are the building blocks for everything you will do in React. API calls, form management, authentication flows, real-time updates — they all rely on these primitives. Once you master them, you can build custom hooks that encapsulate complex logic into reusable packages, which is how professional React codebases stay maintainable.

Key Insight

React re-renders a component whenever its state changes. Each render is a snapshot in time — the component function runs again, reads the current state values, and returns new JSX. The old JSX is compared with the new JSX (reconciliation), and only the differences are applied to the DOM. This is why you must never mutate state directly — React would not know anything changed.

Key Concepts

1. useState — Local Component State

useState is the most fundamental hook. It declares a state variable that persists across re-renders. It returns a pair: the current value and a setter function. When you call the setter, React schedules a re-render with the new value. State can hold any JavaScript value: strings, numbers, booleans, arrays, objects, or null.

jsx 
import { useState } from "react";

// Basic counter
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}

// Functional updates (when new state depends on previous state)
function SafeCounter() {
  const [count, setCount] = useState(0);

  const incrementThree = () => {
    // Without functional updates, these would all read the same count
    setCount((prev) => prev + 1);
    setCount((prev) => prev + 1);
    setCount((prev) => prev + 1);
  };

  return <button onClick={incrementThree}>Add 3 (count: {count})</button>;
}

// Object state (always spread to create new reference)
function UserForm() {
  const [user, setUser] = useState({
    name: "",
    email: "",
    role: "viewer",
  });

  const handleChange = (field, value) => {
    setUser((prev) => ({ ...prev, [field]: value }));
  };

  return (
    <form>
      <input
        value={user.name}
        onChange={(e) => handleChange("name", e.target.value)}
        placeholder="Name"
      />
      <input
        value={user.email}
        onChange={(e) => handleChange("email", e.target.value)}
        placeholder="Email"
      />
    </form>
  );
}

// Array state (always create new arrays, never mutate)
function TodoList() {
  const [todos, setTodos] = useState([]);
  const [input, setInput] = useState("");

  const addTodo = () => {
    setTodos((prev) => [...prev, { id: Date.now(), text: input, done: false }]);
    setInput("");
  };

  const removeTodo = (id) => {
    setTodos((prev) => prev.filter((t) => t.id !== id));
  };

  const toggleTodo = (id) => {
    setTodos((prev) =>
      prev.map((t) => (t.id === id ? { ...t, done: !t.done } : t))
    );
  };

  return (
    <div>
      <input value={input} onChange={(e) => setInput(e.target.value)} />
      <button onClick={addTodo}>Add</button>
      <ul>
        {todos.map((todo) => (
          <li key={todo.id}>
            <span onClick={() => toggleTodo(todo.id)}>
              {todo.done ? "[x]" : "[ ]"} {todo.text}
            </span>
            <button onClick={() => removeTodo(todo.id)}>Delete</button>
          </li>
        ))}
      </ul>
    </div>
  );
}
Never mutate state directly

Writing user.name = "Alice" or todos.push(newTodo) will not trigger a re-render. React only re-renders when you call the setter with a new reference. Always create new objects and arrays: use spread (...prev), .map(), .filter(), or .concat() instead of mutating methods like .push(), .splice(), or direct property assignment.

2. useEffect — Side Effects and Lifecycle

useEffect lets you run code after React has rendered the component to the DOM. It is the hook for side effects: data fetching, subscriptions, timers, manual DOM manipulation, and logging. The effect runs after every render by default, but you control when it re-runs using the dependency array.

jsx 
import { useState, useEffect } from "react";

// Run on every render (no dependency array)
useEffect(() => {
  console.log("Component rendered");
});

// Run once on mount (empty dependency array)
useEffect(() => {
  console.log("Component mounted");
}, []);

// Run when specific values change
useEffect(() => {
  console.log(`User ID changed to: ${userId}`);
}, [userId]);

// Practical example: fetching data on mount
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;  // prevent state updates after unmount

    async function fetchUser() {
      try {
        setLoading(true);
        const res = await fetch(`/api/users/${userId}`);
        const data = await res.json();
        if (!cancelled) setUser(data);
      } catch (err) {
        if (!cancelled) setError(err);
      } finally {
        if (!cancelled) setLoading(false);
      }
    }

    fetchUser();

    // Cleanup function: runs before re-running effect or on unmount
    return () => { cancelled = true; };
  }, [userId]);  // re-fetch when userId changes

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error.message}</p>;
  return <div><h2>{user.name}</h2><p>{user.email}</p></div>;
}

// Cleanup example: event listeners and timers
function WindowSize() {
  const [size, setSize] = useState({ width: window.innerWidth, height: window.innerHeight });

  useEffect(() => {
    const handleResize = () => {
      setSize({ width: window.innerWidth, height: window.innerHeight });
    };

    window.addEventListener("resize", handleResize);
    return () => window.removeEventListener("resize", handleResize);
  }, []);

  return <p>{size.width} x {size.height}</p>;
}

  useEffect Dependency Array Behavior
  ════════════════════════════════════

  useEffect(fn)           No array       Runs after EVERY render
  useEffect(fn, [])       Empty array    Runs ONCE after first render
  useEffect(fn, [a, b])   With deps      Runs when a or b changes

  Cleanup Function:
  ──────────────────
  useEffect(() => {
    // Setup: subscribe, add listener, start timer
    const id = setInterval(tick, 1000);

    return () => {
      // Cleanup: unsubscribe, remove listener, clear timer
      clearInterval(id);
    };
  }, [deps]);

3. useContext — Shared State Without Prop Drilling

When multiple components at different nesting levels need the same data, passing props through every intermediate component (prop drilling) becomes tedious and fragile. useContext solves this by providing a way to share values across the component tree without explicitly passing props at every level. Common use cases include themes, authentication state, and locale settings.

jsx 
import { createContext, useContext, useState } from "react";

// Step 1: Create the context
const AuthContext = createContext(null);

// Step 2: Create a provider component
function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [token, setToken] = useState(null);

  const login = async (email, password) => {
    const res = await fetch("/api/auth/login", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ email, password }),
    });
    const data = await res.json();
    setUser(data.user);
    setToken(data.token);
  };

  const logout = () => {
    setUser(null);
    setToken(null);
  };

  return (
    <AuthContext.Provider value={{ user, token, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// Step 3: Create a custom hook for convenience
function useAuth() {
  const context = useContext(AuthContext);
  if (!context) {
    throw new Error("useAuth must be used within an AuthProvider");
  }
  return context;
}

// Step 4: Use anywhere in the tree
function NavBar() {
  const { user, logout } = useAuth();

  return (
    <nav>
      {user ? (
        <>
          <span>Hello, {user.name}</span>
          <button onClick={logout}>Sign Out</button>
        </>
      ) : (
        <a href="/login">Sign In</a>
      )}
    </nav>
  );
}

// Step 5: Wrap your app
function App() {
  return (
    <AuthProvider>
      <NavBar />
      <MainContent />
    </AuthProvider>
  );
}

4. useReducer — Complex State Logic

When state logic becomes complex — multiple sub-values, state transitions that depend on the previous state, or actions that affect multiple state fields at once — useReducer provides a structured alternative to useState. It follows the same pattern as Redux: you define a reducer function that takes the current state and an action, and returns the new state. This makes state transitions explicit, testable, and traceable.

jsx 
import { useReducer } from "react";

// Define action types as constants
const ACTIONS = {
  ADD_TODO:    "ADD_TODO",
  TOGGLE_TODO: "TOGGLE_TODO",
  DELETE_TODO: "DELETE_TODO",
  SET_FILTER:  "SET_FILTER",
};

// Initial state
const initialState = {
  todos: [],
  filter: "all",   // "all" | "active" | "completed"
  nextId: 1,
};

// Reducer function (pure function — no side effects)
function todoReducer(state, action) {
  switch (action.type) {
    case ACTIONS.ADD_TODO:
      return {
        ...state,
        todos: [...state.todos, { id: state.nextId, text: action.payload, done: false }],
        nextId: state.nextId + 1,
      };
    case ACTIONS.TOGGLE_TODO:
      return {
        ...state,
        todos: state.todos.map((t) =>
          t.id === action.payload ? { ...t, done: !t.done } : t
        ),
      };
    case ACTIONS.DELETE_TODO:
      return {
        ...state,
        todos: state.todos.filter((t) => t.id !== action.payload),
      };
    case ACTIONS.SET_FILTER:
      return { ...state, filter: action.payload };
    default:
      return state;
  }
}

// Component using the reducer
function TodoApp() {
  const [state, dispatch] = useReducer(todoReducer, initialState);
  const [input, setInput] = useState("");

  const filteredTodos = state.todos.filter((t) => {
    if (state.filter === "active") return !t.done;
    if (state.filter === "completed") return t.done;
    return true;
  });

  const handleAdd = (e) => {
    e.preventDefault();
    if (!input.trim()) return;
    dispatch({ type: ACTIONS.ADD_TODO, payload: input });
    setInput("");
  };

  return (
    <div>
      <form onSubmit={handleAdd}>
        <input value={input} onChange={(e) => setInput(e.target.value)} />
        <button type="submit">Add</button>
      </form>
      <div>
        {["all", "active", "completed"].map((f) => (
          <button key={f} onClick={() => dispatch({ type: ACTIONS.SET_FILTER, payload: f })}>
            {f}
          </button>
        ))}
      </div>
      <ul>
        {filteredTodos.map((todo) => (
          <li key={todo.id}>
            <span onClick={() => dispatch({ type: ACTIONS.TOGGLE_TODO, payload: todo.id })}>
              {todo.done ? "[x]" : "[ ]"} {todo.text}
            </span>
            <button onClick={() => dispatch({ type: ACTIONS.DELETE_TODO, payload: todo.id })}>
              Delete
            </button>
          </li>
        ))}
      </ul>
    </div>
  );
}

5. Custom Hooks — Reusable Logic

Custom hooks are the most powerful pattern in React. A custom hook is simply a function whose name starts with use and that calls other hooks. It lets you extract component logic into reusable functions that can be shared across components. The calling component does not need to know how the logic works internally — it just uses the returned values. This is similar to modules in Ruby.

jsx 
// Custom hook: useLocalStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    const stored = localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initialValue;
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(value));
  }, [key, value]);

  return [value, setValue];
}

// Custom hook: useFetch
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    let cancelled = false;

    async function doFetch() {
      try {
        setLoading(true);
        const res = await fetch(url);
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        const json = await res.json();
        if (!cancelled) setData(json);
      } catch (err) {
        if (!cancelled) setError(err);
      } finally {
        if (!cancelled) setLoading(false);
      }
    }

    doFetch();
    return () => { cancelled = true; };
  }, [url]);

  return { data, loading, error };
}

// Custom hook: useDebounce
function useDebounce(value, delay = 300) {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);

  return debounced;
}

// Using the custom hooks
function SearchPage() {
  const [query, setQuery] = useLocalStorage("search-query", "");
  const debouncedQuery = useDebounce(query, 500);
  const { data, loading, error } = useFetch(
    debouncedQuery ? `/api/search?q=${debouncedQuery}` : null
  );

  return (
    <div>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      {loading && <p>Searching...</p>}
      {error && <p>Error: {error.message}</p>}
      {data && <ResultsList results={data} />}
    </div>
  );
}
Pro Tip

If you find yourself copying the same useState + useEffect pattern across multiple components, it is time to extract a custom hook. Think of custom hooks as the React equivalent of Ruby modules — they encapsulate reusable behavior that can be mixed into any component.

6. State Patterns and Rules of Hooks

React hooks have strict rules that must be followed. Understanding these rules and common state patterns will prevent subtle bugs and help you structure your components correctly. The two rules are: (1) only call hooks at the top level of your component or custom hook, and (2) only call hooks from React functions (components or custom hooks), never from regular JavaScript functions.

jsx 
// WRONG: conditional hooks (React tracks hooks by call order)
function Bad({ showExtra }) {
  const [name, setName] = useState("");
  if (showExtra) {
    const [extra, setExtra] = useState("");  // BUG: breaks hook ordering
  }
}

// RIGHT: always call all hooks, conditionally use the values
function Good({ showExtra }) {
  const [name, setName] = useState("");
  const [extra, setExtra] = useState("");  // always called

  return (
    <div>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      {showExtra && (
        <input value={extra} onChange={(e) => setExtra(e.target.value)} />
      )}
    </div>
  );
}

// Pattern: Lifting state up
function Parent() {
  const [selectedId, setSelectedId] = useState(null);

  return (
    <div>
      <ItemList onSelect={setSelectedId} />
      <ItemDetail itemId={selectedId} />
    </div>
  );
}

// Pattern: Derived state (compute, do not store)
function FilteredList({ items }) {
  const [query, setQuery] = useState("");

  // Derived — computed during render, not stored in state
  const filtered = items.filter((item) =>
    item.name.toLowerCase().includes(query.toLowerCase())
  );

  return (
    <div>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <p>{filtered.length} results</p>
      <ul>{filtered.map((i) => <li key={i.id}>{i.name}</li>)}</ul>
    </div>
  );
}
Do not store what you can compute

If a value can be calculated from existing state or props, do not put it in useState. Compute it during render instead. For example, filteredItems should not be state if it can be derived from items and query. Storing derived values leads to synchronization bugs where the derived state gets out of sync with its source.

Self-Check

Before moving to Chapter 07, honestly answer these questions. If any answer is "no", go back and practice more.

Can you explain what happens when you call a useState setter function?

When you call a setter (e.g., setCount(5)), React queues a re-render with the new value. The current function call continues with the old value. On the next render, the component function runs again and useState returns the new value. If you need the new value to depend on the previous value, use the functional form: setCount(prev => prev + 1).

Can you write a useEffect with proper cleanup for a WebSocket connection?

You should be able to open a WebSocket in a useEffect, set up message handlers, and close the connection in the cleanup function. The dependency array should include the URL or any values that, when changed, require a new connection. If this pattern is unclear, practice with simpler side effects like setInterval first.

Do you know when to use useState vs useReducer?

Use useState for simple, independent state values (a string, a number, a boolean). Use useReducer when state has multiple sub-values that change together, when the next state depends on the previous state in complex ways, or when you want to make state transitions explicit and testable. A shopping cart or form with validation are good candidates for useReducer.

Can you create a context provider with a custom hook from memory?

You should be able to write createContext, a Provider component with state, and a useXxx hook that reads the context and throws if used outside the provider. This three-step pattern (create, provide, consume) is the foundation for all shared state in React applications.

Can you extract a custom hook from a component that mixes state and effects?

Given a component with useState, useEffect, and event handlers all mixed together, you should be able to identify the reusable logic, pull it into a useXxx function, and return the values the component needs. The component should become a thin rendering layer that delegates all logic to the hook.
