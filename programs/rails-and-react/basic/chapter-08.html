Chapter 08 · Phase 4: Production Readiness

Testing Full-Stack Applications

Build confidence in your Rails + React applications through comprehensive testing — RSpec for backend validation, React Testing Library for component behavior, Jest for JavaScript logic, and integration tests that verify the full stack works together.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Shipping code without tests is shipping hope. In a full-stack Rails + React application, every layer of the stack can break independently — a model validation changes, an API endpoint shifts its response format, a React component renders stale data. Without automated tests, you discover these failures in production, when real users are affected and your credibility as an engineer takes the hit.

Testing is not about achieving 100% coverage for a badge on your README. It is about building a safety net that lets you change code confidently. When you refactor a service object and 47 RSpec examples still pass green, you know the refactor worked. When you update a React component's state management and your Testing Library specs still verify the correct user-facing behavior, you ship with certainty instead of anxiety.

The testing landscape for a Rails + React application spans two ecosystems. On the Ruby side you have RSpec, factory_bot, and Capybara. On the JavaScript side you have Jest and React Testing Library. This chapter teaches you both sides and, critically, how to write integration tests that verify the contract between your Rails API and your React frontend. By the end, you will have a test suite that runs in CI and catches regressions before they reach production.

Key Insight

The testing pyramid remains the best mental model: many fast unit tests at the base, fewer integration tests in the middle, and a small number of slow end-to-end tests at the top. Each layer catches different categories of bugs. Skipping any layer leaves a blind spot in your safety net.

Key Concepts

1. RSpec Fundamentals and Project Setup

RSpec is the de facto testing framework for Ruby and Rails applications. Its syntax is designed to read like English specifications, making your test suite serve double duty as living documentation. Before writing any tests, you need to configure RSpec correctly for a Rails API application, including support files, database cleaner strategy, and factory integration.

ruby 
# spec/rails_helper.rb
require 'spec_helper'
ENV['RAILS_ENV'] ||= 'test'
require_relative '../config/environment'
require 'rspec/rails'

# Load support files
Dir[Rails.root.join('spec/support/**/*.rb')].sort.each { |f| require f }

RSpec.configure do |config|
  # Use transactional fixtures for speed
  config.use_transactional_fixtures = true

  # Include FactoryBot methods (create, build, etc.)
  config.include FactoryBot::Syntax::Methods

  # Infer spec type from file location
  config.infer_spec_type_from_file_location!

  # Filter Rails backtrace for cleaner output
  config.filter_rails_from_backtrace!
end
Transactional vs Truncation

Transactional fixtures wrap each test in a database transaction that rolls back at the end. This is fast but does not work with tests that spawn separate threads or processes (like system tests with a real browser). For those cases, use database_cleaner with a truncation strategy. Never mix the two without understanding the tradeoffs.

2. Model Specs and Validations

Model specs are the fastest tests in your Rails suite because they do not involve HTTP requests or view rendering. They test your business logic, validations, associations, scopes, and callbacks. Every model in your application should have a corresponding spec file that verifies its behavior under both valid and invalid conditions.

ruby 
# spec/models/article_spec.rb
require 'rails_helper'

RSpec.describe Article, type: :model do
  # Associations
  describe "associations" do
    it { is_expected.to belong_to(:author).class_name('User') }
    it { is_expected.to have_many(:comments).dependent(:destroy) }
    it { is_expected.to have_many(:tags).through(:article_tags) }
  end

  # Validations
  describe "validations" do
    it { is_expected.to validate_presence_of(:title) }
    it { is_expected.to validate_length_of(:title).is_at_most(255) }
    it { is_expected.to validate_presence_of(:body) }
    it { is_expected.to validate_uniqueness_of(:slug) }
  end

  # Scopes
  describe ".published" do
    let!(:published)   { create(:article, status: "published") }
    let!(:draft)       { create(:article, status: "draft") }

    it "returns only published articles" do
      expect(Article.published).to contain_exactly(published)
    end
  end

  # Instance methods
  describe "#reading_time" do
    it "calculates minutes based on word count" do
      article = build(:article, body: "word " * 500)
      expect(article.reading_time).to eq(2)  # 500 / 250 wpm
    end
  end
end

3. factory_bot: Test Data That Makes Sense

Hard-coding test data in every spec file creates a maintenance nightmare. factory_bot centralizes your test data definitions into factories — blueprints for creating model instances with sensible defaults. Factories support traits for common variations and sequences for unique attributes, making your test setup concise and readable.

ruby 
# spec/factories/articles.rb
FactoryBot.define do
  factory :article do
    # Sequences generate unique values
    sequence(:title) { |n| "Article Title #{n}" }
    body { Faker::Lorem.paragraphs(number: 3).join("\n\n") }
    status { "draft" }
    association :author, factory: :user

    # Traits for common variations
    trait :published do
      status { "published" }
      published_at { Time.current }
    end

    trait :with_comments do
      after(:create) do |article|
        create_list(:comment, 3, article: article)
      end
    end

    trait :long_form do
      body { Faker::Lorem.paragraphs(number: 20).join("\n\n") }
    end
  end
end

# Usage in specs:
article = create(:article)                      # basic article (saved to DB)
article = build(:article)                       # in-memory only (faster)
article = create(:article, :published)            # with published trait
article = create(:article, :published, :with_comments) # multiple traits
articles = create_list(:article, 5)              # create 5 articles

4. Request Specs for API Endpoints

Request specs test your Rails API endpoints end-to-end: the HTTP request hits routing, passes through middleware, invokes the controller, runs business logic, touches the database, and returns a JSON response. These are the most important tests for a Rails API because they verify the contract your React frontend depends on. Always test status codes, response body structure, and error handling.

ruby 
# spec/requests/api/v1/articles_spec.rb
require 'rails_helper'

RSpec.describe "Api::V1::Articles", type: :request do
  let(:user) { create(:user) }
  let(:headers) { auth_headers_for(user) }

  describe "GET /api/v1/articles" do
    let!(:articles) { create_list(:article, 3, :published) }

    it "returns a list of published articles" do
      get "/api/v1/articles", headers: headers
      expect(response).to have_http_status(200)

      json = JSON.parse(response.body)
      expect(json["data"].length).to eq(3)
      expect(json["data"].first).to have_key("title")
      expect(json["data"].first).to have_key("slug")
    end

    it "supports pagination" do
      get "/api/v1/articles", params: { page: 1, per: 2 }, headers: headers
      json = JSON.parse(response.body)
      expect(json["data"].length).to eq(2)
      expect(json["meta"]["total_pages"]).to eq(2)
    end
  end

  describe "POST /api/v1/articles" do
    let(:valid_params) do
      { article: { title: "New Post", body: "Content here" } }
    end

    context "with valid parameters" do
      it "creates an article and returns 201" do
        expect {
          post "/api/v1/articles", params: valid_params, headers: headers
        }.to change(Article, :count).by(1)

        expect(response).to have_http_status(201)
      end
    end

    context "with invalid parameters" do
      it "returns 422 with error messages" do
        post "/api/v1/articles",
             params: { article: { title: "" } },
             headers: headers

        expect(response).to have_http_status(422)
        json = JSON.parse(response.body)
        expect(json["errors"]).to include("Title can't be blank")
      end
    end
  end
end
Pro Tip

Create a shared helper for authentication headers. Define auth_headers_for(user) in spec/support/auth_helpers.rb so every request spec can authenticate without duplicating JWT token generation logic. Include it via config.include AuthHelpers, type: :request.

5. React Testing Library and Jest

React Testing Library is built on a fundamental philosophy: test what the user sees, not implementation details. Instead of checking component state or prop values, you query the DOM the way a user would — by visible text, labels, roles, and placeholders. This makes your tests resilient to refactors. If you rename an internal state variable but the user experience stays the same, your tests should still pass.

jsx 
// src/components/__tests__/ArticleCard.test.jsx
import { render, screen } from '@testing-library/react';
import { ArticleCard } from '../ArticleCard';

describe('ArticleCard', () => {
  const article = {
    id: 1,
    title: 'Testing React Components',
    excerpt: 'Learn how to test your React components effectively.',
    author: 'Jane Doe',
    publishedAt: '2025-01-15',
  };

  it('renders the article title as a heading', () => {
    render(<ArticleCard article={article} />);
    expect(screen.getByRole('heading', {
      name: 'Testing React Components'
    })).toBeInTheDocument();
  });

  it('displays the author name', () => {
    render(<ArticleCard article={article} />);
    expect(screen.getByText('Jane Doe')).toBeInTheDocument();
  });

  it('shows the excerpt text', () => {
    render(<ArticleCard article={article} />);
    expect(screen.getByText(/test your React components/i))
      .toBeInTheDocument();
  });

  it('links to the full article', () => {
    render(<ArticleCard article={article} />);
    const link = screen.getByRole('link', { name: /read more/i });
    expect(link).toHaveAttribute('href', '/articles/1');
  });
});
jsx 
// Testing user interactions and async behavior
import { render, screen, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { CreateArticleForm } from '../CreateArticleForm';

describe('CreateArticleForm', () => {
  const mockOnSubmit = jest.fn();

  beforeEach(() => {
    mockOnSubmit.mockClear();
  });

  it('submits the form with entered values', async () => {
    const user = userEvent.setup();
    render(<CreateArticleForm onSubmit={mockOnSubmit} />);

    await user.type(
      screen.getByLabelText('Title'), 'My New Article'
    );
    await user.type(
      screen.getByLabelText('Body'), 'Article content here'
    );
    await user.click(
      screen.getByRole('button', { name: /publish/i })
    );

    await waitFor(() => {
      expect(mockOnSubmit).toHaveBeenCalledWith({
        title: 'My New Article',
        body: 'Article content here',
      });
    });
  });

  it('shows validation error for empty title', async () => {
    const user = userEvent.setup();
    render(<CreateArticleForm onSubmit={mockOnSubmit} />);

    await user.click(
      screen.getByRole('button', { name: /publish/i })
    );

    expect(screen.getByText('Title is required'))
      .toBeInTheDocument();
    expect(mockOnSubmit).not.toHaveBeenCalled();
  });
});

6. Integration Tests and Test Coverage

Unit tests verify individual pieces work correctly. Integration tests verify those pieces work together. In a full-stack application, the most critical integration point is the API contract between Rails and React. Use Mock Service Worker (MSW) to intercept HTTP requests in your React tests and return realistic API responses. On the Rails side, configure SimpleCov to track coverage and set minimum thresholds that fail CI if not met.

jsx 
// Mocking API calls with MSW (Mock Service Worker)
import { rest } from 'msw';
import { setupServer } from 'msw/node';
import { render, screen, waitFor } from '@testing-library/react';
import { ArticleList } from '../ArticleList';

const server = setupServer(
  rest.get('/api/v1/articles', (req, res, ctx) => {
    return res(ctx.json({
      data: [
        { id: 1, title: 'First Post', slug: 'first-post' },
        { id: 2, title: 'Second Post', slug: 'second-post' },
      ],
      meta: { total_pages: 1 }
    }));
  })
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

describe('ArticleList', () => {
  it('fetches and displays articles', async () => {
    render(<ArticleList />);

    // Loading state
    expect(screen.getByText(/loading/i)).toBeInTheDocument();

    // Articles appear after fetch
    await waitFor(() => {
      expect(screen.getByText('First Post')).toBeInTheDocument();
      expect(screen.getByText('Second Post')).toBeInTheDocument();
    });
  });

  it('handles API errors gracefully', async () => {
    server.use(
      rest.get('/api/v1/articles', (req, res, ctx) => {
        return res(ctx.status(500));
      })
    );
    render(<ArticleList />);

    await waitFor(() => {
      expect(screen.getByText(/something went wrong/i))
        .toBeInTheDocument();
    });
  });
});

  Full-Stack Testing Pyramid
  ═══════════════════════════════════════════════════

         /\          E2E Tests (Cypress / Playwright)
        /  \         - Full browser, real API
       /    \        - Slow but high confidence
      /──────\
     /        \      Integration Tests (MSW + RTL / Request Specs)
    /          \     - Mock API boundaries
   /            \    - Medium speed, medium confidence
  /──────────────\
 /                \   Unit Tests (Jest / RSpec Model Specs)
/                  \  - Single functions, components, models
/────────────────────\ - Fast, high volume, low scope

  Speed:       FAST ──────────────────── SLOW
  Scope:       NARROW ────────────────── BROAD
  Count:       MANY ──────────────────── FEW
Figure 1: The testing pyramid for full-stack Rails + React applications
Avoid Snapshot Testing as a Crutch

Jest snapshot tests are easy to write but create false confidence. When a snapshot fails, developers often blindly update it with -u instead of verifying the change is correct. Use snapshots sparingly for complex serialized outputs. Prefer explicit assertions with React Testing Library queries that describe what the user sees.

Assignments

Complete these in order. Each level builds on the previous one. All code should be tested and pass CI before moving to the next level.

Assignment 1
Level 1 — Foundation

Model Specs with factory_bot

Write comprehensive model specs for all existing models using factory_bot for test data.

  • Create factories for all models with sensible defaults and at least 2 traits per factory
  • Write validation specs using shoulda-matchers for presence, uniqueness, and length
  • Test all model associations (belongs_to, has_many, has_many through)
  • Test all custom scopes with both positive and negative cases
  • Achieve 100% model coverage as reported by SimpleCov
Done when

Running bundle exec rspec spec/models passes all examples green. SimpleCov reports 100% line coverage for the app/models directory.

Assignment 2
Level 2 — Professional

Request Specs and React Component Tests

Write request specs for every API endpoint and RTL tests for every user-facing component.

  • Write request specs for all CRUD endpoints testing success, validation errors, and authentication
  • Test pagination, filtering, and sorting in your index endpoints
  • Write RTL tests for all form components testing submission, validation, and error display
  • Test loading states, empty states, and error states in list components
  • Use MSW to mock API responses in React component tests
Done when

Both bundle exec rspec spec/requests and npx jest pass all tests green. Every API endpoint and every React component has at least one test file.

Assignment 3
Level 3 — Production-Ready

CI Pipeline with Full Coverage Enforcement

Set up a GitHub Actions CI pipeline that runs the entire test suite and enforces coverage thresholds.

  • Configure GitHub Actions to run RSpec, Jest, and ESLint on every push and pull request
  • Set up SimpleCov with a minimum 90% coverage threshold that fails the build if not met
  • Configure Jest coverage with --coverageThreshold enforcing 85% across the board
  • Add parallel test execution using parallel_tests gem to keep CI under 5 minutes
  • Write at least 2 end-to-end tests with Cypress or Playwright covering critical user flows
Done when

Every push to GitHub triggers a green CI build. Coverage reports are generated and accessible. E2E tests verify that a user can sign up, create an article, and view it on the frontend.

Self-Check

This is the final chapter. Before completing the course, honestly answer these questions. If any answer is "no", go back and practice more.

Can you write an RSpec model spec with factory_bot from memory?

Open a new file and write a spec for a Product model without looking at notes. Include validation tests, association tests, a scope test, and use create, build, and traits from a factory. If you cannot do this fluently, review the model specs and factory_bot sections until the patterns are automatic.

Can you write a request spec that tests a JSON API endpoint including error handling?

Write a request spec for a POST /api/v1/products endpoint. Test the happy path (valid params, returns 201), the validation error path (invalid params, returns 422 with error messages), and the authentication failure path (no token, returns 401). If you struggle with the structure, review the request specs section.

Can you test a React component using React Testing Library queries and user events?

Write a test for a login form component. Verify that the form displays email and password fields, that clicking submit with valid data calls the onSubmit handler, and that submitting with an empty email shows a validation error. Use getByLabelText, getByRole, and userEvent. If you reach for getByTestId first, revisit the RTL query priority documentation.

Can you explain the difference between unit, integration, and end-to-end tests?

Unit tests verify a single function, method, or component in isolation. They are fast and numerous. Integration tests verify that multiple units work together correctly, such as a React component fetching data from an API endpoint. End-to-end tests drive a real browser against a real server, simulating actual user behavior. Each layer catches different categories of bugs. A healthy test suite has all three.

Do you have a CI pipeline that runs your entire test suite automatically?

A test suite that only runs on your laptop is a test suite that will be ignored. Your CI pipeline should run RSpec, Jest, linters, and coverage checks on every push. If a test fails, the build fails, and the pull request cannot be merged. This is the foundation of professional software development. If your CI is not set up, that is your top priority before moving on.
