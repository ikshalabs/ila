Chapter 08 · Phase 4: Full-Stack Integration

Testing Full-Stack Applications

Write tests that give you confidence to ship. Master RSpec for testing Rails models, controllers, and API endpoints. Learn React Testing Library for testing components the way users interact with them. Build integration tests that verify your full stack works end to end.

Estimated: 5–6 days
3 Advancement Levels
3 Assignments

Why This Matters

Code without tests is a liability. Every change becomes a gamble — you never know if you broke something until a user reports it. Tests give you the confidence to refactor, add features, and deploy without fear. In a professional environment, untested code does not get merged.

Full-stack testing is more complex than testing a single layer because you need different tools for different parts of the stack. Rails has RSpec for models, controllers, and request specs. React has React Testing Library for component tests. And you need integration tests that verify the entire flow from UI click to database write and back. This chapter teaches all three.

Key Insight

The testing pyramid is your guide: many fast unit tests at the base (model specs, component tests), fewer integration tests in the middle (request specs, component integration), and a small number of end-to-end tests at the top. This gives maximum confidence with minimum test runtime.

Key Concepts

1. RSpec Setup and Model Specs

RSpec is the de facto testing framework for Rails. It provides a readable, expressive DSL for writing tests. Model specs test your business logic — validations, associations, scopes, and custom methods — without touching the network or browser.

bash
# Add RSpec and testing gems to Gemfile
group :development, :test do
  gem "rspec-rails", "~> 6.0"
  gem "factory_bot_rails"
  gem "faker"
end

# Install RSpec
bundle install
rails generate rspec:install
ruby
# spec/models/article_spec.rb
require "rails_helper"

RSpec.describe Article, type: :model do
  # Test validations
  describe "validations" do
    it "is valid with a title and body" do
      article = build(:article)
      expect(article).to be_valid
    end

    it "is invalid without a title" do
      article = build(:article, title: nil)
      expect(article).not_to be_valid
      expect(article.errors[:title]).to include("can't be blank")
    end

    it "is invalid with a title shorter than 5 characters" do
      article = build(:article, title: "Hi")
      expect(article).not_to be_valid
    end
  end

  # Test associations
  describe "associations" do
    it "belongs to a user" do
      association = described_class.reflect_on_association(:user)
      expect(association.macro).to eq(:belongs_to)
    end

    it "has many comments" do
      association = described_class.reflect_on_association(:comments)
      expect(association.macro).to eq(:has_many)
    end
  end

  # Test scopes
  describe ".published" do
    it "returns only published articles" do
      published = create(:article, status: "published")
      draft = create(:article, status: "draft")

      expect(Article.published).to include(published)
      expect(Article.published).not_to include(draft)
    end
  end

  # Test custom methods
  describe "#reading_time" do
    it "calculates reading time based on word count" do
      article = build(:article, body: "word " * 500)
      expect(article.reading_time).to eq(3) # 500 words / 200 wpm = 2.5, rounded up
    end
  end
end

# spec/factories/articles.rb
FactoryBot.define do
  factory :article do
    title { Faker::Lorem.sentence(word_count: 5) }
    body { Faker::Lorem.paragraphs(number: 3).join("\n\n") }
    status { "published" }
    association :user

    trait :draft do
      status { "draft" }
    end
  end
end

2. Request Specs — Testing API Endpoints

Request specs test your API endpoints from the outside — sending HTTP requests and asserting on the response status, headers, and JSON body. They are the most important specs for an API application because they test the full request/response cycle through the router, controller, serializer, and back.

ruby
# spec/requests/api/v1/articles_spec.rb
require "rails_helper"

RSpec.describe "Api::V1::Articles", type: :request do
  let(:user) { create(:user) }
  let(:token) { JsonWebToken.encode(user_id: user.id) }
  let(:auth_headers) { { "Authorization" => "Bearer #{token}" } }

  describe "GET /api/v1/articles" do
    it "returns a list of published articles" do
      create_list(:article, 3, status: "published")
      create(:article, status: "draft")

      get "/api/v1/articles"

      expect(response).to have_http_status(:ok)
      expect(JSON.parse(response.body).length).to eq(3)
    end

    it "returns articles with expected fields" do
      create(:article, title: "Test Article")

      get "/api/v1/articles"

      json = JSON.parse(response.body)
      expect(json.first).to include(
        "id",
        "title" => "Test Article",
        "body",
        "created_at"
      )
    end
  end

  describe "POST /api/v1/articles" do
    context "with valid params" do
      it "creates a new article" do
        article_params = { article: { title: "New Post", body: "Content here" } }

        expect {
          post "/api/v1/articles", params: article_params,
               headers: auth_headers, as: :json
        }.to change(Article, :count).by(1)

        expect(response).to have_http_status(:created)
        expect(JSON.parse(response.body)["title"]).to eq("New Post")
      end
    end

    context "with invalid params" do
      it "returns 422 with validation errors" do
        post "/api/v1/articles",
             params: { article: { title: "" } },
             headers: auth_headers, as: :json

        expect(response).to have_http_status(:unprocessable_entity)
        json = JSON.parse(response.body)
        expect(json["errors"]["title"]).to be_present
      end
    end

    context "without authentication" do
      it "returns 401 unauthorized" do
        post "/api/v1/articles",
             params: { article: { title: "Test" } }, as: :json

        expect(response).to have_http_status(:unauthorized)
      end
    end
  end

  describe "DELETE /api/v1/articles/:id" do
    it "deletes the article if owned by user" do
      article = create(:article, user: user)

      expect {
        delete "/api/v1/articles/#{article.id}", headers: auth_headers
      }.to change(Article, :count).by(-1)

      expect(response).to have_http_status(:no_content)
    end

    it "returns 403 if not the owner" do
      other_article = create(:article)

      delete "/api/v1/articles/#{other_article.id}", headers: auth_headers

      expect(response).to have_http_status(:forbidden)
    end
  end
end
Pro Tip: Use let and let! wisely

let is lazy — it only creates the object when first referenced. let! creates it before each test runs. Use let by default and let! when you need the record to exist in the database before the test body executes (e.g., for listing endpoints).

3. React Testing Library — Testing Components

React Testing Library (RTL) tests components the way users interact with them — by finding elements by their visible text, labels, or roles, not by CSS classes or component internals. This makes tests resilient to refactoring. If the user experience has not changed, the tests should not break.

bash
# React Testing Library is included with Create React App
# For Vite projects, install it:
npm install --save-dev @testing-library/react @testing-library/jest-dom
npm install --save-dev @testing-library/user-event vitest jsdom
jsx
// ArticleCard.test.jsx
import { render, screen } from "@testing-library/react";
import { ArticleCard } from "./ArticleCard";

describe("ArticleCard", () => {
  const article = {
    id: 1,
    title: "Getting Started with Rails",
    body: "Rails is a web framework...",
    author: { name: "Jane Doe" },
    created_at: "2025-01-15",
  };

  it("renders the article title", () => {
    render(<ArticleCard article={article} />);

    // Query by visible text — the way users find elements
    expect(screen.getByText("Getting Started with Rails")).toBeInTheDocument();
  });

  it("renders the author name", () => {
    render(<ArticleCard article={article} />);

    expect(screen.getByText("Jane Doe")).toBeInTheDocument();
  });

  it("links to the article detail page", () => {
    render(<ArticleCard article={article} />);

    const link = screen.getByRole("link", { name: /getting started/i });
    expect(link).toHaveAttribute("href", "/articles/1");
  });
});

// LoginForm.test.jsx — Testing user interactions
import { render, screen, waitFor } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { LoginForm } from "./LoginForm";

describe("LoginForm", () => {
  it("submits the form with email and password", async () => {
    const mockLogin = vi.fn().mockResolvedValue({ name: "Jane" });
    const user = userEvent.setup();

    render(<LoginForm onLogin={mockLogin} />);

    // Find by label — accessible and resilient
    await user.type(screen.getByLabelText(/email/i), "jane@example.com");
    await user.type(screen.getByLabelText(/password/i), "secret123");
    await user.click(screen.getByRole("button", { name: /log in/i }));

    expect(mockLogin).toHaveBeenCalledWith("jane@example.com", "secret123");
  });

  it("shows validation error for empty email", async () => {
    const user = userEvent.setup();

    render(<LoginForm onLogin={vi.fn()} />);

    await user.click(screen.getByRole("button", { name: /log in/i }));

    expect(screen.getByText(/email is required/i)).toBeInTheDocument();
  });

  it("disables the button while submitting", async () => {
    const mockLogin = vi.fn(() => new Promise(() => {})); // never resolves
    const user = userEvent.setup();

    render(<LoginForm onLogin={mockLogin} />);

    await user.type(screen.getByLabelText(/email/i), "jane@example.com");
    await user.type(screen.getByLabelText(/password/i), "secret123");
    await user.click(screen.getByRole("button", { name: /log in/i }));

    expect(screen.getByRole("button")).toBeDisabled();
    expect(screen.getByText(/logging in/i)).toBeInTheDocument();
  });
});
Query Priority

React Testing Library recommends this query priority: (1) getByRole — accessible queries work for all users. (2) getByLabelText — great for form fields. (3) getByText — for non-interactive elements. (4) getByTestId — last resort when nothing else works. Avoid querying by class name or component hierarchy.

4. Mocking API Calls in React Tests

When testing React components that fetch data, you do not want real network requests. Use msw (Mock Service Worker) to intercept requests at the network level. This gives you realistic tests without needing a running backend.

jsx
// Install Mock Service Worker
// npm install --save-dev msw

// test/mocks/handlers.js
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("http://localhost:3000/api/v1/articles", () => {
    return HttpResponse.json([
      { id: 1, title: "First Article", body: "Content..." },
      { id: 2, title: "Second Article", body: "More content..." },
    ]);
  }),

  http.post("http://localhost:3000/api/v1/articles", async ({ request }) => {
    const body = await request.json();
    return HttpResponse.json(
      { id: 3, ...body.article },
      { status: 201 }
    );
  }),

  http.post("http://localhost:3000/api/v1/auth/login", async ({ request }) => {
    const { email, password } = await request.json();
    if (email === "jane@example.com" && password === "secret123") {
      return HttpResponse.json({
        token: "fake-jwt-token",
        user: { id: 1, email, name: "Jane" },
      });
    }
    return HttpResponse.json(
      { error: "Invalid credentials" },
      { status: 401 }
    );
  }),
];

// test/mocks/server.js
import { setupServer } from "msw/node";
import { handlers } from "./handlers";

export const server = setupServer(...handlers);

// test/setup.js — start/stop MSW around tests
import { server } from "./mocks/server";

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

// ArticlesPage.test.jsx — now uses MSW mock
import { render, screen, waitFor } from "@testing-library/react";
import { ArticlesPage } from "./ArticlesPage";

it("loads and displays articles", async () => {
  render(<ArticlesPage />);

  // Initially shows loading state
  expect(screen.getByText(/loading/i)).toBeInTheDocument();

  // Wait for data to load
  await waitFor(() => {
    expect(screen.getByText("First Article")).toBeInTheDocument();
    expect(screen.getByText("Second Article")).toBeInTheDocument();
  });
});

5. The Testing Pyramid — What to Test Where

Not all tests are equal. The testing pyramid helps you balance speed, coverage, and confidence. Knowing what to test at each level prevents both under-testing (bugs in production) and over-testing (slow test suites that nobody wants to run).

  Testing Pyramid for React + Rails
  ═══════════════════════════════════

           /\
          /  \         E2E Tests (Cypress / Playwright)
         /    \        - Full user flows: login, create, edit
        / Few  \       - Slow but high confidence
       /--------\      - Test critical paths only
      /          \
     /            \    Integration Tests
    / Request Specs \  - Rails: API endpoints with auth
   / Component Tests \ - React: components with mocked APIs
  /  Medium amount    \
 /--------------------\
/                      \  Unit Tests
/ Model Specs           \ - Rails: validations, scopes, methods
/ Component Render Tests \ - React: renders correct output
/ Many, fast, isolated    \
/══════════════════════════\

  Speed:       Fast ──────────────────────► Slow
  Count:       Many ──────────────────────► Few
  Confidence:  Low  ──────────────────────► High
  Isolation:   High ──────────────────────► Low
Figure 1: The testing pyramid balances speed and confidence. Most tests should be fast unit tests. Few should be slow E2E tests.
What to Test

Rails: Always test validations, custom methods, scopes, and API endpoints. Skip testing basic CRUD that Rails provides — test your logic, not the framework. React: Test user interactions (forms, clicks), conditional rendering, and data display. Skip testing implementation details (state values, lifecycle timing).

6. Running Tests in CI

Tests are only useful if they run automatically. A CI (Continuous Integration) pipeline runs your entire test suite on every push and pull request. If tests fail, the code cannot be merged. This is the safety net that keeps your main branch deployable.

yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  rails-tests:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: password
        ports: ["5432:5432"]
        options: --health-cmd pg_isready --health-interval 10s

    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          ruby-version: "3.3"
          bundler-cache: true

      - name: Setup database
        env:
          DATABASE_URL: postgres://postgres:password@localhost:5432/test
          RAILS_ENV: test
        run: |
          bin/rails db:create db:schema:load

      - name: Run RSpec
        env:
          DATABASE_URL: postgres://postgres:password@localhost:5432/test
          RAILS_ENV: test
        run: bundle exec rspec --format documentation

  react-tests:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./frontend

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
          cache-dependency-path: frontend/package-lock.json

      - run: npm ci
      - run: npm test -- --run
      - run: npm run lint
CI Best Practice

Run Rails and React tests in parallel jobs (as shown above). This cuts your CI time in half. Also add branch protection rules that require CI to pass before merging pull requests. This makes testing non-negotiable for the entire team.

Self-Check

Can you explain the difference between model specs, request specs, and component tests?

Model specs test Ruby business logic in isolation: validations, scopes, custom methods. No HTTP, no browser. Request specs send real HTTP requests to your Rails app and assert on status codes and JSON responses. They test the full request cycle (routing, controller, serializer). Component tests render React components in a simulated DOM and assert on what users see and can interact with. They test rendering, user events, and async behavior.

Do you know when to use getByRole vs getByText vs getByTestId?

getByRole is the preferred query because it tests accessibility. Use it for buttons, links, headings, and form elements. getByLabelText is best for form inputs. getByText works for any visible text content. getByTestId is the last resort when the element has no accessible role, label, or unique text. Adding data-testid attributes couples tests to implementation, so avoid it when possible.

Can you set up a CI pipeline that runs both Rails and React tests?

Use GitHub Actions with two parallel jobs. The Rails job needs a PostgreSQL service container, Ruby setup, database creation, and bundle exec rspec. The React job needs Node.js setup and npm test -- --run. Running them in parallel (separate jobs, not separate steps) halves your CI time. Add branch protection rules to require both jobs to pass before merging.
