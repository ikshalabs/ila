Chapter 05 · Phase 2: React Front-End Development

React Components and JSX

Learn to build user interfaces with React's component model and JSX syntax — functional components, props, composition patterns, conditional rendering, lists with keys, and event handling. This is where your Rails API meets the browser.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

React is the most widely adopted front-end library in professional web development, and for good reason. It introduced a component-based architecture that fundamentally changed how developers think about building user interfaces. Instead of manipulating the DOM directly with imperative code, you describe what the UI should look like for any given state, and React figures out how to update the browser efficiently. This declarative approach eliminates entire categories of bugs related to DOM synchronization.

In a full-stack Rails + React application, your Rails API handles data persistence, business logic, authentication, and authorization. React handles everything the user sees and interacts with. The boundary between these two layers is clean JSON over HTTP. But building a React front-end that is maintainable, testable, and performant requires understanding components deeply — not just how to render HTML, but how to decompose a UI into reusable pieces, pass data between them, and handle user interactions elegantly.

JSX is the syntax that makes React practical. It looks like HTML embedded in JavaScript, but it is actually syntactic sugar for function calls. Understanding that JSX compiles to React.createElement calls helps you debug rendering issues, understand error messages, and write more intentional component code. This chapter builds the foundation for everything React that follows — state management, API integration, routing, and testing all depend on solid component fundamentals.

Key Insight

React components are just functions that return JSX. A component receives data through props (like function arguments) and returns a description of what the UI should look like. There is no magic — if you can write a JavaScript function, you can write a React component. The power comes from composing many small components together.

Key Concepts

1. JSX Syntax and Expressions

JSX stands for JavaScript XML. It is a syntax extension that allows you to write HTML-like markup directly inside JavaScript files. Under the hood, a build tool like Babel or SWC transforms JSX into React.createElement() calls. This means JSX is not a template language — it has the full power of JavaScript. You can embed any JavaScript expression inside JSX by wrapping it in curly braces {}.

jsx 
// JSX looks like HTML but lives inside JavaScript
const element = <h1>Hello, World!</h1>;

// Embedding JavaScript expressions with curly braces
const name = "Alice";
const greeting = <h1>Hello, {name}!</h1>;

// Any valid JavaScript expression works inside {}
const math = <p>Two plus two is {2 + 2}</p>;
const upper = <p>{name.toUpperCase()}</p>;
const today = <p>Date: {new Date().toLocaleDateString()}</p>;

// JSX attributes use camelCase (className, not class)
const styledElement = (
  <div className="container">
    <label htmlFor="email">Email</label>
    <input
      id="email"
      type="email"
      placeholder="you@example.com"
      autoComplete="email"
    />
  </div>
);

// Inline styles use a JavaScript object with camelCase properties
const inlineStyled = (
  <div style={{ backgroundColor: "#f0fdfa", padding: "1rem", borderRadius: "8px" }}>
    <p style={{ color: "#0f766e", fontWeight: 600 }}>Styled with objects</p>
  </div>
);

// JSX must have a single root element (use fragments to avoid extra divs)
const fragment = (
  <>
    <h1>Title</h1>
    <p>Paragraph</p>
  </>
);
JSX is not HTML

JSX uses className instead of class, htmlFor instead of for, and all attributes are camelCase (onClick, tabIndex, autoFocus). This is because JSX compiles to JavaScript, and class and for are reserved keywords in JavaScript. Self-closing tags like <img> and <input> must include the closing slash: <img />.

2. Functional Components

In modern React, all components are functions. A functional component is a JavaScript function that accepts a single argument (an object called props) and returns JSX. The function name must start with an uppercase letter so React can distinguish your components from regular HTML elements. Components let you split the UI into independent, reusable pieces that you can think about in isolation.

jsx 
// The simplest possible component
function Welcome() {
  return <h1>Welcome to ILA!</h1>;
}

// Arrow function syntax (equally valid, common in codebases)
const Greeting = () => {
  return <h1>Welcome to ILA!</h1>;
};

// Arrow function with implicit return (single expression)
const ShortGreeting = () => <h1>Welcome to ILA!</h1>;

// Component that accepts props
function UserGreeting({ name, role }) {
  return (
    <div className="user-greeting">
      <h2>Hello, {name}!</h2>
      <p>Your role: {role}</p>
    </div>
  );
}

// Using the component (props passed as JSX attributes)
function App() {
  return (
    <div>
      <UserGreeting name="Alice" role="admin" />
      <UserGreeting name="Bob" role="editor" />
      <UserGreeting name="Charlie" role="viewer" />
    </div>
  );
}

// A realistic component with multiple elements
function ProductCard({ title, price, imageUrl, inStock }) {
  return (
    <article className="product-card">
      <img src={imageUrl} alt={title} className="product-image" />
      <div className="product-details">
        <h3 className="product-title">{title}</h3>
        <span className="product-price">${price.toFixed(2)}</span>
        <span className={inStock ? "in-stock" : "out-of-stock"}>
          {inStock ? "In Stock" : "Out of Stock"}
        </span>
      </div>
    </article>
  );
}

3. Props, Children, and Composition

Props are how data flows in React — always downward from parent to child. A parent component passes data to a child via props, and the child reads them as a plain JavaScript object. Props are read-only: a child component must never modify its own props. This one-way data flow makes React applications predictable and easy to debug, because you can always trace where data comes from by following the component tree upward.

jsx 
// Props with default values
function Button({ label, variant = "primary", size = "md", disabled = false }) {
  return (
    <button
      className={`btn btn-${variant} btn-${size}`}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

// Using children prop for composition
function Card({ title, children }) {
  return (
    <div className="card">
      <div className="card-header">
        <h3>{title}</h3>
      </div>
      <div className="card-body">
        {children}
      </div>
    </div>
  );
}

// Composition: nesting components inside other components
function Dashboard() {
  return (
    <div className="dashboard">
      <Card title="User Statistics">
        <p>Total users: 1,234</p>
        <p>Active today: 567</p>
        <Button label="View All" variant="secondary" />
      </Card>

      <Card title="Recent Orders">
        <p>Processing: 12</p>
        <p>Shipped: 45</p>
        <Button label="Manage Orders" />
      </Card>
    </div>
  );
}

// Spreading props to pass them through
function TextInput({ label, error, ...inputProps }) {
  return (
    <div className="form-field">
      <label>{label}</label>
      <input {...inputProps} className={error ? "input-error" : "input"} />
      {error && <span className="error-message">{error}</span>}
    </div>
  );
}

// Usage: all standard input attributes pass through
<TextInput
  label="Email"
  type="email"
  placeholder="you@example.com"
  required
  error="Email is required"
/>
Pro Tip

Favor composition over configuration. Instead of building a single component with 20 props to control every variation, build smaller components and compose them together. A <Card> that accepts children is more flexible than a <Card content={...} footer={...} sidebar={...} /> that tries to anticipate every use case.

4. Conditional Rendering

React does not have a special template syntax for conditionals like v-if or *ngIf. Instead, you use plain JavaScript. This is one of React's greatest strengths: there is no new syntax to learn for control flow. The three most common patterns are the logical AND operator, the ternary operator, and early returns.

jsx 
// Pattern 1: Logical AND (&&) — show or hide
function Notification({ message, isVisible }) {
  return (
    <div>
      {isVisible && (
        <div className="notification">
          <p>{message}</p>
        </div>
      )}
    </div>
  );
}

// Pattern 2: Ternary operator — either/or
function LoginButton({ isLoggedIn, userName }) {
  return (
    <header>
      {isLoggedIn ? (
        <span>Welcome back, {userName}!</span>
      ) : (
        <button>Sign In</button>
      )}
    </header>
  );
}

// Pattern 3: Early return — guard clauses (like Ruby!)
function UserProfile({ user, isLoading, error }) {
  if (isLoading) return <div className="spinner">Loading...</div>;
  if (error) return <div className="error">Error: {error.message}</div>;
  if (!user) return <div>No user found</div>;

  return (
    <div className="profile">
      <h2>{user.name}</h2>
      <p>{user.email}</p>
      {user.isAdmin && <span className="badge">Admin</span>}
    </div>
  );
}

// Pattern 4: Rendering different components based on a value
function StatusIcon({ status }) {
  const icons = {
    success: <span className="icon-success">OK</span>,
    warning: <span className="icon-warning">!!</span>,
    error:   <span className="icon-error">X</span>,
  };

  return icons[status] || <span className="icon-unknown">?</span>;
}
Watch out for falsy value rendering

The expression {count && <span>{count}</span>} will render 0 on screen when count is zero, because 0 is falsy but still a valid React child. Use explicit checks instead: {count > 0 && <span>{count}</span>} or a ternary operator.

5. Lists, Keys, and Iteration

Rendering lists is one of the most common operations in React. You use JavaScript's .map() method to transform an array of data into an array of JSX elements. React requires a key prop on each element in a list so it can efficiently update the DOM when items are added, removed, or reordered. Keys must be unique among siblings and stable across re-renders.

jsx 
// Basic list rendering with .map()
function UserList({ users }) {
  return (
    <ul className="user-list">
      {users.map((user) => (
        <li key={user.id}>
          {user.name} — {user.email}
        </li>
      ))}
    </ul>
  );
}

// Rendering a list of components
function ProductGrid({ products }) {
  return (
    <div className="product-grid">
      {products.map((product) => (
        <ProductCard
          key={product.id}
          title={product.name}
          price={product.price}
          imageUrl={product.imageUrl}
          inStock={product.inStock}
        />
      ))}
    </div>
  );
}

// Filtering and mapping together
function ActiveUsers({ users }) {
  const activeUsers = users.filter((u) => u.isActive);

  if (activeUsers.length === 0) {
    return <p className="empty-state">No active users found.</p>;
  }

  return (
    <div>
      <h3>Active Users ({activeUsers.length})</h3>
      <ul>
        {activeUsers.map((user) => (
          <li key={user.id}>
            <strong>{user.name}</strong>
            <span>Last seen: {user.lastActive}</span>
          </li>
        ))}
      </ul>
    </div>
  );
}

// Nested lists with separate keys per level
function CategoryMenu({ categories }) {
  return (
    <nav>
      {categories.map((category) => (
        <div key={category.id} className="category-group">
          <h4>{category.name}</h4>
          <ul>
            {category.items.map((item) => (
              <li key={item.id}>{item.label}</li>
            ))}
          </ul>
        </div>
      ))}
    </nav>
  );
}

6. Event Handling

React handles events using synthetic events — a cross-browser wrapper around the browser's native event system. Event handlers in JSX are passed as camelCase props (onClick, onChange, onSubmit) and receive a synthetic event object. You can prevent default behavior, stop propagation, and access the target element just like with native DOM events.

jsx 
// Basic click handler
function Counter() {
  const [count, setCount] = useState(0);

  const handleIncrement = () => {
    setCount(count + 1);
  };

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={handleIncrement}>Increment</button>
    </div>
  );
}

// Form handling with controlled inputs
function LoginForm() {
  const [email, setEmail] = useState("");
  const [password, setPassword] = useState("");

  const handleSubmit = (e) => {
    e.preventDefault();  // prevent page reload
    console.log("Logging in:", { email, password });
  };

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        placeholder="Email"
      />
      <input
        type="password"
        value={password}
        onChange={(e) => setPassword(e.target.value)}
        placeholder="Password"
      />
      <button type="submit">Sign In</button>
    </form>
  );
}

// Passing data with event handlers (common pattern)
function TaskList({ tasks, onDelete }) {
  return (
    <ul>
      {tasks.map((task) => (
        <li key={task.id}>
          <span>{task.title}</span>
          <button onClick={() => onDelete(task.id)}>
            Delete
          </button>
        </li>
      ))}
    </ul>
  );
}

// Keyboard and focus events
function SearchBox({ onSearch }) {
  const [query, setQuery] = useState("");

  const handleKeyDown = (e) => {
    if (e.key === "Enter") {
      onSearch(query);
    }
    if (e.key === "Escape") {
      setQuery("");
    }
  };

  return (
    <input
      type="search"
      value={query}
      onChange={(e) => setQuery(e.target.value)}
      onKeyDown={handleKeyDown}
      onFocus={() => console.log("Search focused")}
      placeholder="Search..."
    />
  );
}
What is the difference between onClick={handler} and onClick={handler()}?

onClick={handler} passes a reference to the function. React will call it when the user clicks. This is what you want.

onClick={handler()} calls the function immediately during rendering and passes the return value as the handler. This is almost always a bug — the function runs on every render, not on click.

If you need to pass arguments, use an arrow function: onClick={() => handler(id)}.

When should I extract a component versus keeping things inline?

Extract a component when any of these apply: (1) the JSX is reused in multiple places, (2) the logic is complex enough that it deserves its own name, (3) the file is getting long and hard to scan, or (4) you want to test the piece independently. A good rule of thumb: if you need to scroll past a chunk of JSX to understand the parent component, that chunk should probably be its own component.

Self-Check

Before moving to Chapter 06, honestly answer these questions. If any answer is "no", go back and practice more.

Can you create a functional component that accepts props and returns JSX from memory?

Open a new file and write a UserCard component that takes name, email, and avatarUrl props without consulting documentation. It should use destructuring, default values, and conditional rendering for the avatar. If you need to look up the syntax, practice writing 10 small components until it becomes automatic.

Can you explain the difference between props and children?

Props are named values passed as attributes: <Button label="Click" />. Children is the content between opening and closing tags: <Card>content here</Card>. Children is actually just a special prop called children. Use named props for structured data and children for flexible content composition.

Can you render a list from an array with proper keys and handle the empty state?

Given an array of objects from an API, you should be able to use .map() to render a list, choose appropriate keys (always use unique IDs, never array indices for dynamic lists), and show a fallback message when the array is empty. If this pattern is not second nature, build three different list components with different data shapes.

Can you handle form submission with controlled inputs?

You should be able to create a form where each input's value is controlled by React state, updated via onChange, and submitted via onSubmit with e.preventDefault(). The form should be clearable after submission. This is the pattern you will use for every form in your Rails + React application.

Do you understand why React needs keys on list items?

Keys help React's reconciliation algorithm identify which items have changed, been added, or been removed. Without stable keys, React may re-render the entire list instead of only the changed items. Using array indices as keys causes bugs when items are reordered or deleted. Always use unique, stable identifiers from your data (like database IDs).
