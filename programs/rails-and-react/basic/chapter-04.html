Chapter 04 · Phase 2: API Development & React Integration

RESTful API Design

Design and build production-grade APIs with Rails — REST principles, API-only mode, JSON serialization, HTTP status codes, API versioning, consistent error handling, and the patterns that make your back end a reliable contract for your React front end.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

In this course, you are building a decoupled architecture: Rails serves as the API back end, and React serves as the front end. The API is the contract between these two systems. If the API is well-designed, your React developers (and any future mobile developers) can work independently and confidently. If the API is inconsistent, every front-end feature becomes a negotiation about data formats, error structures, and URL patterns. Good API design is what makes the entire stack productive.

REST (Representational State Transfer) is not a technology — it is a set of architectural constraints that guide how you design your API. Resources are identified by URLs. Actions on resources are expressed through HTTP methods (GET, POST, PATCH, DELETE). Status codes communicate outcomes. When you follow REST consistently, your API becomes predictable: any developer who knows REST conventions can use your API without reading documentation for every endpoint.

Rails was built for REST. The resources routing DSL, controller conventions, and response helpers all align with RESTful design. Rails API mode strips out browser-specific middleware (cookies, sessions, CSRF protection) to give you a lean, fast API server. This chapter teaches you to leverage these Rails strengths to build APIs that are clean, consistent, and ready for production.

Key Insight

A great API is one that your React developer can integrate with by reading the route names and HTTP status codes alone. If every endpoint follows the same conventions for success responses, error responses, and pagination, the front-end team never has to ask "what format does this endpoint return?" — they already know.

Key Concepts

1. REST Principles

REST defines six architectural constraints. In practice, the ones that matter most for Rails API development are: resources identified by URIs, manipulation through HTTP methods, self-descriptive messages (proper status codes and content types), and statelessness (each request carries all information needed to process it). 

  RESTful Resource Mapping
  ═══════════════════════════════════════════════════════

  HTTP Method   URL                  Action     Purpose
  ───────────   ──────────────────   ────────   ──────────────────────
  GET           /api/v1/articles     index      List all articles
  POST          /api/v1/articles     create     Create a new article
  GET           /api/v1/articles/42  show       Get one article
  PATCH/PUT     /api/v1/articles/42  update     Update an article
  DELETE        /api/v1/articles/42  destroy    Delete an article

  Key Rules:
  - URLs identify RESOURCES (nouns), not actions (verbs)
  - HTTP methods express the ACTION on the resource
  - /articles      = collection (plural noun)
  - /articles/42   = individual resource (identified by ID)

  BAD:  GET  /getArticle/42     (verb in URL)
  BAD:  POST /deleteArticle/42  (wrong method)
  GOOD: GET  /articles/42       (noun + method = intent)
  GOOD: DELETE /articles/42
Figure 1: Mapping HTTP methods to CRUD operations on resources

2. Rails API Mode

When you create a Rails application with the --api flag, Rails configures itself for API-only operation. It removes middleware for cookies, sessions, flash messages, and CSRF protection — things only needed for browser-based applications. Controllers inherit from ActionController::API instead of ActionController::Base, which is lighter and faster.

bash 
# Create a new Rails API application
rails new my_api --api --database=postgresql -T

# What --api does:
# 1. Sets config.api_only = true in application.rb
# 2. ApplicationController inherits from ActionController::API
# 3. Removes browser middleware (cookies, sessions, CSRF)
# 4. Generators skip view templates and helpers
# 5. Removes asset pipeline configuration

# What -T does:
# Skips default test framework (we use RSpec instead)

# After creation, add RSpec:
# Add to Gemfile: gem 'rspec-rails', group: [:development, :test]
bundle install
rails generate rspec:install
ruby 
# config/application.rb (generated with --api flag)
module MyApi
  class Application < Rails::Application
    config.load_defaults 7.1
    config.api_only = true

    # Add back middleware we need for API development
    config.middleware.insert_before 0, Rack::Cors do
      allow do
        origins "localhost:3001", "localhost:5173"
        resource "*",
          headers: :any,
          methods: [:get, :post, :put, :patch, :delete, :options],
          expose:  ["Authorization"]
      end
    end
  end
end

# app/controllers/application_controller.rb (API mode)
class ApplicationController < ActionController::API
  # Note: inherits from API, not Base
  # This means no cookie/session/CSRF support
end

3. JSON Serialization

When your API returns data, you need to control exactly which attributes are included in the JSON response. Rendering @article.to_json exposes every column, including internal fields like updated_at or password_digest that clients do not need. Serializers give you explicit control over the response shape.

ruby 
# ═══════════════════════════════════════════
# APPROACH 1: render json with options
# ═══════════════════════════════════════════

# Simple — works for quick prototyping
render json: @article, only: [:id, :title, :body, :status]

# With associations
render json: @article, include: {
  comments: { only: [:id, :body], include: {
    user: { only: [:id, :name] }
  }}
}

# ═══════════════════════════════════════════
# APPROACH 2: Serializer classes (recommended)
# ═══════════════════════════════════════════

# app/serializers/article_serializer.rb
class ArticleSerializer
  def initialize(article)
    @article = article
  end

  def as_json(*)
    {
      id:         @article.id,
      title:      @article.title,
      body:       @article.body,
      slug:       @article.slug,
      status:     @article.status,
      word_count: @article.word_count,
      author:     UserSerializer.new(@article.user).as_json,
      tags:       @article.tags.pluck(:name),
      created_at: @article.created_at.iso8601,
      comment_count: @article.comments.count
    }
  end
end

# app/serializers/user_serializer.rb
class UserSerializer
  def initialize(user)
    @user = user
  end

  def as_json(*)
    {
      id:   @user.id,
      name: @user.name,
      avatar_url: @user.avatar_url
    }
  end
end

# In the controller:
def show
  render json: ArticleSerializer.new(@article)
end

def index
  articles = Article.published.includes(:user, :tags).recent
  render json: articles.map { |a| ArticleSerializer.new(a) }
end
Pro Tip

Always use ISO 8601 format for dates (created_at.iso8601) in your API responses. This is the universal standard that JavaScript's new Date() constructor understands natively. Never return Ruby-formatted dates like "2024-01-15 14:30:00 +0000" — they are ambiguous across time zones and harder to parse on the front end.

4. HTTP Status Codes

Status codes are how your API communicates the outcome of a request. Using the correct status code lets your React front end handle responses consistently — all 2xx responses are success, 4xx responses are client errors, and 5xx responses are server errors. Rails provides symbolic names for every status code so you never need to memorize numbers.

ruby 
# ═══════════════════════════════════════════
# SUCCESS CODES (2xx)
# ═══════════════════════════════════════════

# 200 OK — standard success response
render json: @articles, status: :ok

# 201 Created — new resource was created
render json: @article, status: :created

# 204 No Content — success, no body (used for DELETE)
head :no_content

# ═══════════════════════════════════════════
# CLIENT ERROR CODES (4xx)
# ═══════════════════════════════════════════

# 400 Bad Request — malformed request syntax
render json: { error: "Missing required parameter" },
       status: :bad_request

# 401 Unauthorized — authentication required
render json: { error: "Authentication required" },
       status: :unauthorized

# 403 Forbidden — authenticated but not authorized
render json: { error: "You do not have permission" },
       status: :forbidden

# 404 Not Found — resource does not exist
render json: { error: "Article not found" },
       status: :not_found

# 422 Unprocessable Entity — validation errors
render json: { errors: @article.errors.full_messages },
       status: :unprocessable_entity

# ═══════════════════════════════════════════
# RAILS SYMBOLIC STATUS CODES REFERENCE
# ═══════════════════════════════════════════

# :ok                    => 200
# :created               => 201
# :no_content            => 204
# :bad_request           => 400
# :unauthorized          => 401
# :forbidden             => 403
# :not_found             => 404
# :unprocessable_entity  => 422
# :internal_server_error => 500

5. API Versioning

APIs evolve over time. New features require new fields, and breaking changes require new endpoints. Versioning lets you make changes without breaking existing clients. The most common approach is URL-based versioning with a namespace like /api/v1/. When you need to make breaking changes, you create /api/v2/ while keeping v1 running.

ruby 
# config/routes.rb
Rails.application.routes.draw do
  namespace :api do
    namespace :v1 do
      resources :articles do
        resources :comments, only: [:index, :create, :destroy]
        member do
          patch :publish
          patch :archive
        end
      end
      resources :users, only: [:index, :show, :create]
      resources :tags,  only: [:index]
    end
  end
end

# app/controllers/api/v1/base_controller.rb
module Api
  module V1
    class BaseController < ApplicationController
      # Shared error handling for all v1 controllers
      rescue_from ActiveRecord::RecordNotFound, with: :not_found
      rescue_from ActiveRecord::RecordInvalid,  with: :unprocessable
      rescue_from ActionController::ParameterMissing, with: :bad_request

      private

      def not_found(exception)
        render json: {
          error: "not_found",
          message: exception.message
        }, status: :not_found
      end

      def unprocessable(exception)
        render json: {
          error: "unprocessable_entity",
          message: "Validation failed",
          details: exception.record.errors.full_messages
        }, status: :unprocessable_entity
      end

      def bad_request(exception)
        render json: {
          error: "bad_request",
          message: exception.message
        }, status: :bad_request
      end
    end
  end
end

# app/controllers/api/v1/articles_controller.rb
module Api
  module V1
    class ArticlesController < BaseController
      before_action :set_article, only: [:show, :update, :destroy, :publish]

      def index
        articles = Article.published
          .includes(:user, :tags)
          .recent
          .limit(params[:limit] || 20)
          .offset(params[:offset] || 0)

        render json: {
          data: articles.map { |a| ArticleSerializer.new(a) },
          meta: { total: Article.published.count }
        }
      end

      # ... show, create, update, destroy ...

      private

      def set_article
        @article = Article.find(params[:id])
      end
    end
  end
end

6. Consistent Error Handling

Every API response should follow a consistent structure. Your React front end should never have to guess whether errors are in response.error, response.errors, or response.message. Define a standard error format and use it everywhere. The rescue_from pattern in a base controller ensures consistency across all endpoints.

ruby 
# Consistent response structure for your API

# SUCCESS: Single resource
# {
#   "data": { "id": 1, "title": "Hello", ... }
# }

# SUCCESS: Collection with metadata
# {
#   "data": [ { "id": 1, ... }, { "id": 2, ... } ],
#   "meta": { "total": 142, "limit": 20, "offset": 0 }
# }

# ERROR: Consistent error format
# {
#   "error": "not_found",
#   "message": "Couldn't find Article with 'id'=999"
# }

# ERROR: Validation errors with details
# {
#   "error": "unprocessable_entity",
#   "message": "Validation failed",
#   "details": ["Title can't be blank", "Body is too short"]
# }

# Helper module for consistent responses
module Api
  module V1
    module Respondable
      extend ActiveSupport::Concern

      private

      def render_success(data, status: :ok, meta: nil)
        response = { data: data }
        response[:meta] = meta if meta
        render json: response, status: status
      end

      def render_error(error, message, status:, details: nil)
        response = { error: error, message: message }
        response[:details] = details if details
        render json: response, status: status
      end

      def render_created(data)
        render_success(data, status: :created)
      end
    end
  end
end
bash 
# Testing your API with curl

# GET — list all articles
curl http://localhost:3000/api/v1/articles | jq

# GET — single article
curl http://localhost:3000/api/v1/articles/1 | jq

# POST — create an article
curl -X POST http://localhost:3000/api/v1/articles \
  -H "Content-Type: application/json" \
  -d '{"article": {"title": "New Post", "body": "Content here..."}}' | jq

# PATCH — update an article
curl -X PATCH http://localhost:3000/api/v1/articles/1 \
  -H "Content-Type: application/json" \
  -d '{"article": {"title": "Updated Title"}}' | jq

# DELETE — delete an article
curl -X DELETE http://localhost:3000/api/v1/articles/1 -v

# Test error handling — missing resource
curl http://localhost:3000/api/v1/articles/99999 | jq
# => {"error":"not_found","message":"Couldn't find Article..."}
Never expose stack traces in production

In development, Rails shows detailed error pages with stack traces. In production, you must ensure errors return clean JSON without internal implementation details. The rescue_from pattern ensures this. Also check that config.consider_all_requests_local = false is set in config/environments/production.rb (it is by default).

Self-Check

Before moving to Chapter 05, honestly answer these questions. If any answer is "no", go back and practice more.

Can you list the correct HTTP status code for every CRUD operation?

Index: 200. Show: 200 (or 404). Create: 201 (or 422). Update: 200 (or 422/404). Destroy: 204 (or 404). Authentication failure: 401. Authorization failure: 403. Malformed request: 400. If you had to look any of these up, review the status codes section. Your React front end depends on these being consistent.

Can you set up a namespaced /api/v1/ route with a base controller?

You need: namespace :api do namespace :v1 do resources :articles end end in routes, app/controllers/api/v1/base_controller.rb with rescue_from handlers, and app/controllers/api/v1/articles_controller.rb inheriting from BaseController. The file paths must match the module nesting exactly. If this structure is not automatic for you, practice building it from scratch.

Can you explain why you need CORS and how to configure it?

CORS (Cross-Origin Resource Sharing) is a browser security mechanism. When your React app at localhost:3001 makes a request to your Rails API at localhost:3000, the browser blocks it by default because they are different origins (different ports). The rack-cors gem adds headers that tell the browser "this origin is allowed to call my API." Without it, every fetch call from React will fail with a CORS error.

Do you understand the difference between render json: and a serializer?

render json: @article calls to_json on the model, which serializes every column. render json: @article, only: [:id, :title] limits fields but gets messy with associations. A serializer class gives you full control: you define exactly which fields to include, how to format dates, which associations to nest, and what computed fields to add. Serializers also make your response format testable and reusable.

Can you write a rescue_from block that handles all common API errors?

At minimum, your base controller should handle: ActiveRecord::RecordNotFound (return 404), ActiveRecord::RecordInvalid (return 422 with error details), and ActionController::ParameterMissing (return 400). Each handler should return a consistent JSON structure with error and message keys. If you need to look up the syntax, practice writing it from memory.
