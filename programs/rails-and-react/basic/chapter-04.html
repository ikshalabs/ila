Chapter 04 · Phase 2: Building APIs

RESTful API Design

Build production-quality JSON APIs with Rails. Learn RESTful conventions, API-mode controllers, serializers for shaping JSON responses, proper HTTP status codes, error handling, and versioning strategies that keep your API stable as it evolves.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

In a Rails + React architecture, the Rails backend serves as a pure API. It receives HTTP requests, processes business logic, and returns JSON responses. The React frontend consumes this API to render the user interface. This clean separation means your API must be well-designed, consistent, and predictable — because the frontend team (or your future self) depends on it.

REST (Representational State Transfer) is not a protocol — it is a set of architectural constraints for designing networked applications. When applied correctly, RESTful APIs are intuitive, cacheable, and stateless. Rails was built for REST, and its routing system maps directly to RESTful conventions.

Key Insight

A good API is like a contract. The frontend says "give me all published articles sorted by date" and the API always returns the same shape of data. When you change the API, you version it. When something goes wrong, you return meaningful error messages with appropriate HTTP status codes. Predictability is the goal.

Key Concepts

1. API Routes and Namespacing

API routes live under a namespace to separate them from any web routes. Versioning is built into the URL structure so you can evolve the API without breaking existing clients.

ruby
# config/routes.rb
Rails.application.routes.draw do
  namespace :api do
    namespace :v1 do
      resources :articles, only: [:index, :show, :create, :update, :destroy] do
        resources :comments, only: [:index, :create]
      end
      resources :users, only: [:index, :show, :create]
      resource  :session, only: [:create, :destroy]
    end
  end

  # Generates routes like:
  # GET    /api/v1/articles
  # POST   /api/v1/articles
  # GET    /api/v1/articles/:id
  # PATCH  /api/v1/articles/:id
  # DELETE /api/v1/articles/:id
  # GET    /api/v1/articles/:article_id/comments
  # POST   /api/v1/articles/:article_id/comments
end

2. API Controllers

API controllers inherit from ActionController::API instead of ActionController::Base. They skip middleware for sessions, cookies, and CSRF protection (which are not needed for stateless API calls). Every response is JSON.

ruby
# app/controllers/api/v1/articles_controller.rb
module Api
  module V1
    class ArticlesController < ApplicationController
      before_action :set_article, only: [:show, :update, :destroy]

      # GET /api/v1/articles
      def index
        articles = Article.published
                         .includes(:user)
                         .order(created_at: :desc)
                         .page(params[:page])

        render json: articles, each_serializer: ArticleSerializer,
               meta: { total: articles.total_count, page: articles.current_page }
      end

      # GET /api/v1/articles/:id
      def show
        render json: @article, serializer: ArticleDetailSerializer
      end

      # POST /api/v1/articles
      def create
        article = current_user.articles.build(article_params)

        if article.save
          render json: article, serializer: ArticleSerializer,
                 status: :created,
                 location: api_v1_article_url(article)
        else
          render json: { errors: article.errors.full_messages },
                 status: :unprocessable_entity
        end
      end

      # PATCH /api/v1/articles/:id
      def update
        if @article.update(article_params)
          render json: @article, serializer: ArticleSerializer
        else
          render json: { errors: @article.errors.full_messages },
                 status: :unprocessable_entity
        end
      end

      # DELETE /api/v1/articles/:id
      def destroy
        @article.destroy
        head :no_content
      end

      private

      def set_article
        @article = Article.find(params[:id])
      end

      def article_params
        params.require(:article).permit(:title, :body, :published)
      end
    end
  end
end

3. Serializers — Shaping JSON

Serializers control exactly what data appears in your JSON response. Never expose raw model attributes — always go through a serializer. This gives you control over field names, nested data, computed fields, and what stays private.

ruby
# app/serializers/article_serializer.rb
class ArticleSerializer < ActiveModel::Serializer
  attributes :id, :title, :summary, :published, :created_at

  belongs_to :user, serializer: UserCompactSerializer

  def summary
    object.body.truncate(200)
  end

  def created_at
    object.created_at.iso8601
  end
end

# app/serializers/user_compact_serializer.rb
class UserCompactSerializer < ActiveModel::Serializer
  attributes :id, :name, :avatar_url
end

4. HTTP Status Codes

Status codes tell the client what happened without parsing the response body. Using them correctly is not optional — it is what makes your API machine-readable.

  Essential HTTP Status Codes for APIs
  ════════════════════════════════════

  2xx Success
  ──────────────────────────────────
  200 OK           — Standard success (GET, PATCH)
  201 Created      — Resource created (POST)
  204 No Content   — Success with no body (DELETE)

  4xx Client Errors
  ──────────────────────────────────
  400 Bad Request          — Malformed request body
  401 Unauthorized         — Not authenticated
  403 Forbidden            — Authenticated but not allowed
  404 Not Found            — Resource does not exist
  422 Unprocessable Entity — Validation errors

  5xx Server Errors
  ──────────────────────────────────
  500 Internal Server Error — Bug in your code
  503 Service Unavailable   — Server overloaded / maintenance
Figure 1: Status codes your API should use — clients depend on these for error handling

5. Consistent Error Handling

Every error response from your API should follow the same structure. This lets the frontend display errors without guessing the format. A base controller with rescue handlers ensures consistency across all endpoints.

ruby
# app/controllers/api/v1/application_controller.rb
module Api
  module V1
    class ApplicationController < ActionController::API
      rescue_from ActiveRecord::RecordNotFound, with: :not_found
      rescue_from ActiveRecord::RecordInvalid,  with: :unprocessable
      rescue_from ActionController::ParameterMissing, with: :bad_request

      private

      def not_found(exception)
        render json: {
          error: "not_found",
          message: exception.message
        }, status: :not_found
      end

      def unprocessable(exception)
        render json: {
          error: "validation_failed",
          message: exception.record.errors.full_messages
        }, status: :unprocessable_entity
      end

      def bad_request(exception)
        render json: {
          error: "bad_request",
          message: exception.message
        }, status: :bad_request
      end
    end
  end
end
Pro Tip

Always include a machine-readable error key (like "not_found" or "validation_failed") alongside human-readable message text. The frontend can switch on the error key for logic, and display the message to users.

6. Testing Your API

API tests (request specs in RSpec) send HTTP requests and assert on the response status and body. They are the most important tests for an API application because they test the full request cycle.

ruby
# spec/requests/api/v1/articles_spec.rb
require "rails_helper"

RSpec.describe "Api::V1::Articles", type: :request do
  describe "GET /api/v1/articles" do
    it "returns published articles" do
      create_list(:article, 3, published: true)
      create(:article, published: false)

      get "/api/v1/articles"

      expect(response).to have_http_status(:ok)
      expect(json_body.length).to eq(3)
    end
  end

  describe "POST /api/v1/articles" do
    context "with valid parameters" do
      it "creates an article and returns 201" do
        post "/api/v1/articles", params: {
          article: { title: "New Post", body: "Content here" }
        }, headers: auth_headers

        expect(response).to have_http_status(:created)
        expect(json_body["title"]).to eq("New Post")
      end
    end

    context "with invalid parameters" do
      it "returns 422 with error messages" do
        post "/api/v1/articles", params: {
          article: { title: "" }
        }, headers: auth_headers

        expect(response).to have_http_status(:unprocessable_entity)
        expect(json_body["errors"]).to include("Title can't be blank")
      end
    end
  end
end

Basic CRUD API with JSON Responses

Build a Rails API-mode application with full CRUD for a single resource.

  • Create a Rails app with rails new api_app --api --database=postgresql
  • Create namespaced routes under /api/v1/ for a Product resource
  • Implement all 5 actions (index, show, create, update, destroy) returning JSON

Assignment 2
Level 2 — Professional

Multi-Resource API with Serializers and Error Handling

Add serializers, consistent error handling, and nested resources to the API.

  • Add active_model_serializers gem and create serializers for all resources
  • Add a base API controller with rescue_from for consistent error responses
  • Add nested routes for comments on products
  • Add query parameter support: filtering, sorting, and pagination
  • Write RSpec request specs for all endpoints including error cases

All error responses follow the same JSON structure. Serializers control exactly what data is exposed. All request specs pass.

Assignment 3
Level 3 — Production-Ready

Production API with Versioning and Documentation

Add API versioning, rate limiting, and self-documenting endpoint descriptions.

  • Implement content-type based versioning alongside URL versioning
  • Add rate limiting with rack-attack gem (100 requests per minute)
  • Add API documentation using rswag or a Postman collection
  • Add CORS configuration with rack-cors for the React frontend
  • Add a /api/v1/health endpoint that checks database connectivity
Done when

The API has documentation, rate limiting, CORS headers, and a health endpoint. A React app running on a different port can successfully call the API.

Self-Check

Can you explain the difference between 401, 403, and 404?

401 Unauthorized: The client has not authenticated (no token or invalid token). 403 Forbidden: The client is authenticated but does not have permission for this resource. 404 Not Found: The resource does not exist. Getting these right matters because frontend code handles each differently.

Do your serializers prevent leaking sensitive data?

Check your serializers right now. Are you exposing password_digest, email, or internal IDs you should not? Serializers are security boundaries — they define what the outside world can see.

Can you add a new resource endpoint in under 15 minutes?

Time yourself adding a /api/v1/categories endpoint with index, show, and create. If it takes longer than 15 minutes, you need more practice with the Rails API workflow.
