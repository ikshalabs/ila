Chapter 02 · Phase 1: Ruby & Rails Fundamentals

Rails MVC Architecture

Understand the Model-View-Controller pattern that organizes every Rails application. Learn how models represent data, controllers handle requests, views render responses, and routing ties everything together with elegant conventions.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Rails does not just use MVC — it is MVC. Every file in a Rails project lives in a specific directory determined by its role: models go in app/models, controllers in app/controllers, views in app/views. This is not optional. Understanding MVC is understanding how Rails thinks.

The MVC pattern separates concerns: models handle business logic and data, controllers coordinate between the user request and the model, and views format the response. This separation means you can change how data is stored without touching the UI, or redesign the UI without touching business logic.

Rails takes MVC further with convention over configuration. If you create a Post model, Rails expects a posts database table, a PostsController, and views in app/views/posts/. Follow the conventions and Rails writes boilerplate for you. Fight them and you write it yourself.

Key Insight

Convention over configuration means Rails makes decisions for you. A User model maps to a users table. A UsersController#show action renders app/views/users/show.html.erb. You can override all of this, but you almost never need to. The conventions are the framework.

Key Concepts

1. The Rails Request Lifecycle

Every HTTP request to a Rails app follows the same path: the router matches the URL to a controller action, the controller queries models and prepares data, and the view renders the response. Understanding this flow is essential for debugging.

2. Routing — The Front Door

The router is defined in config/routes.rb. It maps HTTP verbs and URL patterns to controller actions. Rails provides resources as a shorthand that generates all seven RESTful routes at once.

ruby
# config/routes.rb
Rails.application.routes.draw do
  # Root route — what loads at "/"
  root "pages#home"

  # RESTful resource — generates 7 routes automatically
  resources :articles
  #   GET    /articles          => articles#index
  #   GET    /articles/new       => articles#new
  #   POST   /articles           => articles#create
  #   GET    /articles/:id       => articles#show
  #   GET    /articles/:id/edit  => articles#edit
  #   PATCH  /articles/:id       => articles#update
  #   DELETE /articles/:id       => articles#destroy

  # Nested resources — articles have many comments
  resources :articles do
    resources :comments, only: [:create, :destroy]
  end

  # API namespace — prefix all routes with /api/v1
  namespace :api do
    namespace :v1 do
      resources :users, only: [:index, :show, :create]
    end
  end

  # Custom route
  get "about", to: "pages#about"
end
bash
# View all routes in your application
bin/rails routes

# Filter routes by controller
bin/rails routes -c articles

# Filter routes by HTTP verb
bin/rails routes -g POST

3. Controllers — Traffic Directors

Controllers receive requests from the router, interact with models to fetch or save data, and determine what response to send back. Each public method in a controller is an action. Rails conventions keep controllers thin — they coordinate but should not contain business logic.

ruby
# app/controllers/articles_controller.rb
class ArticlesController < ApplicationController
  before_action :set_article, only: [:show, :edit, :update, :destroy]

  # GET /articles
  def index
    @articles = Article.all.order(created_at: :desc)
  end

  # GET /articles/:id
  def show
  end

  # GET /articles/new
  def new
    @article = Article.new
  end

  # POST /articles
  def create
    @article = Article.new(article_params)

    if @article.save
      redirect_to @article, notice: "Article was successfully created."
    else
      render :new, status: :unprocessable_entity
    end
  end

  # PATCH /articles/:id
  def update
    if @article.update(article_params)
      redirect_to @article, notice: "Article was successfully updated."
    else
      render :edit, status: :unprocessable_entity
    end
  end

  # DELETE /articles/:id
  def destroy
    @article.destroy
    redirect_to articles_path, notice: "Article was successfully deleted."
  end

  private

  def set_article
    @article = Article.find(params[:id])
  end

  def article_params
    params.require(:article).permit(:title, :body, :published)
  end
end
Strong Parameters are Non-Negotiable

Never trust user input directly. params.require(:article).permit(:title, :body) ensures only whitelisted attributes can be mass-assigned. Without this, attackers could inject admin privileges or modify attributes you never intended to expose.

4. Models — Business Logic Home

Models inherit from ApplicationRecord (which inherits from ActiveRecord::Base). They represent database tables and contain your business logic: validations, associations, scopes, and custom methods.

ruby
# app/models/article.rb
class Article < ApplicationRecord
  belongs_to :author, class_name: "User"
  has_many :comments, dependent: :destroy

  validates :title, presence: true, length: { maximum: 200 }
  validates :body,  presence: true

  scope :published, -> { where(published: true) }
  scope :recent,    -> { order(created_at: :desc) }
  scope :by_author, ->(user) { where(author: user) }

  def summary
    body.truncate(150)
  end
end

5. Views and Templates

Views are ERB templates (Embedded Ruby) that combine HTML with Ruby code. Instance variables set in the controller (@articles) are automatically available in the corresponding view. For API applications, you will use JSON serializers instead of ERB, but understanding the view layer is still important.

erb
<%# app/views/articles/index.html.erb %>
<h1>Articles</h1>

<% @articles.each do |article| %>
  <article>
    <h2><%= link_to article.title, article %></h2>
    <p><%= article.summary %></p>
    <span>By <%= article.author.name %></span>
  </article>
<% end %>
ERB Tags

<%= %> outputs the result to HTML (use for displaying data). <% %> executes Ruby but does not output (use for loops and conditionals). Forgetting the = is the most common ERB mistake.

6. Rails Generators

Generators scaffold files following Rails conventions. They save time and ensure consistency. The most common generators create models, controllers, and migrations.

bash
# Generate a model with attributes and migration
bin/rails generate model Article title:string body:text published:boolean author:references

# Generate a controller with specific actions
bin/rails generate controller Articles index show new create edit update destroy

# Generate a full scaffold (model + controller + views + tests + migration)
bin/rails generate scaffold Product name:string price:decimal description:text

# Undo a generator
bin/rails destroy model Article

# Dry run — see what files would be created
bin/rails generate model Article --pretend
Scaffold Warning

rails generate scaffold is useful for learning, but professional Rails developers rarely use it in production. It generates too many files you will need to customize or delete. Prefer generating models and controllers separately.

Rails naming conventions cheat sheet
text
Model:        Article              (singular, CamelCase)
Table:        articles             (plural, snake_case)
Controller:   ArticlesController   (plural + "Controller")
File:         articles_controller.rb
View dir:     app/views/articles/  (plural)
Route:        resources :articles  (plural symbol)
URL helper:   articles_path        (plural)
              article_path(@a)     (singular with instance)
Migration:    CreateArticles       (CamelCase, descriptive)

Self-Check

Before moving to Chapter 03, honestly answer these questions.

Can you trace an HTTP request from URL to rendered response?

Given GET /articles/5, can you explain: which route matches, which controller action runs, what model query executes, and which view renders? If you cannot trace this flow, review the request lifecycle diagram.

Do you understand why controllers should be thin?

A controller action should be 5-10 lines. If your create action has 30 lines of logic, that logic belongs in a model method or service object. Controllers coordinate — they do not compute.

Can you create a new resource from scratch without generators?

Try creating a Category resource by hand: write the migration, model, controller, routes, and views without any generator. This proves you understand what generators do under the hood.
