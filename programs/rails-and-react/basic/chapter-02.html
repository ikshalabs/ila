Chapter 02 · Phase 1: Ruby & Rails Fundamentals

Rails MVC Architecture

Understand the Model-View-Controller pattern that organizes every Rails application — routing, generators, the request/response cycle, folder structure, and why Rails favors convention over configuration to keep your code predictable and fast to write.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Ruby on Rails is not just a collection of libraries glued together — it is an opinionated framework built around the Model-View-Controller architectural pattern. MVC provides a clear separation of concerns: data logic lives in Models, presentation logic in Views, and coordination logic in Controllers. Every file you create in Rails, every generator you run, and every URL your application responds to is shaped by this pattern. If you do not understand MVC, you will constantly fight the framework instead of letting it accelerate your development.

Rails takes MVC further than most frameworks by adding an aggressive set of conventions. Where other frameworks ask you to configure database table names, file paths, URL structures, and class relationships, Rails derives all of this from naming conventions. A model named Article automatically maps to an articles database table, lives in app/models/article.rb, and has a controller at app/controllers/articles_controller.rb. This is the principle of Convention over Configuration, and it is the single biggest reason Rails developers can build production applications faster than developers using less opinionated frameworks.

Understanding the full request/response cycle — how a browser request enters your application through the router, gets dispatched to a controller action, interacts with models, and returns a response — gives you a mental map for debugging any issue. When something goes wrong in a Rails app, knowing which layer is responsible lets you locate the problem in seconds instead of minutes. This chapter builds that mental map.

Key Insight

Rails generators are not magic — they create the same files you could write by hand. But they enforce naming conventions automatically. Once you understand what each generated file does and where it lives, generators become a productivity tool rather than a black box. Always inspect what a generator creates before moving on.

Key Concepts

1. The MVC Pattern Explained

MVC divides your application into three interconnected layers. The Model handles data, business logic, and database interactions. The View handles presentation and what the user sees. The Controller sits between them, receiving user requests, asking Models for data, and deciding which View to render. In a Rails API application (which is what we build in this course), the View layer is replaced by JSON serialization since React handles the actual UI rendering on the front end.

Thin Controllers, Fat Models

A common Rails antipattern is putting business logic in controllers. Controllers should only coordinate — receive the request, call the model, return the response. Business rules, validations, and data transformations belong in Models or service objects. If your controller action is longer than 10 lines, something probably belongs in the model layer.

2. Routing in Rails

The router is the entry point of every Rails application. It maps incoming HTTP requests (a combination of URL path and HTTP method) to a specific controller action. Rails routing is defined in config/routes.rb and provides a powerful DSL for declaring RESTful routes, nested resources, and custom endpoints. Understanding routing is essential because every feature you build starts with a route definition.

ruby 
# config/routes.rb
Rails.application.routes.draw do
  # Root route — the homepage
  root "pages#home"

  # RESTful resource — generates 7 standard routes
  resources :articles
  # GET    /articles          => articles#index
  # GET    /articles/new       => articles#new
  # POST   /articles           => articles#create
  # GET    /articles/:id       => articles#show
  # GET    /articles/:id/edit  => articles#edit
  # PATCH  /articles/:id       => articles#update
  # DELETE /articles/:id       => articles#destroy

  # Limit routes to only what you need (API-style)
  resources :users, only: [:index, :show, :create, :update]

  # Nested resources — articles have many comments
  resources :articles do
    resources :comments, only: [:index, :create, :destroy]
  end
  # POST /articles/:article_id/comments => comments#create

  # Namespace — groups controllers under a module
  namespace :api do
    namespace :v1 do
      resources :articles
      resources :users
    end
  end
  # GET /api/v1/articles => api/v1/articles#index

  # Custom member and collection routes
  resources :articles do
    member do
      patch :publish    # PATCH /articles/:id/publish
      patch :archive    # PATCH /articles/:id/archive
    end
    collection do
      get :search      # GET /articles/search
      get :drafts      # GET /articles/drafts
    end
  end
end
bash 
# View all routes in your application
rails routes

# Filter routes by controller
rails routes -c articles

# Filter routes matching a pattern
rails routes -g POST

# Show routes in expanded format
rails routes --expanded

3. Controllers and Actions

Controllers are Ruby classes that inherit from ApplicationController. Each public method in a controller is an action that can respond to an HTTP request. The controller receives request parameters through params, interacts with models to fetch or persist data, and renders a response. In API mode, that response is typically JSON with an appropriate HTTP status code.

ruby 
# app/controllers/articles_controller.rb
class ArticlesController < ApplicationController
  # Callback — runs before specific actions
  before_action :set_article, only: [:show, :update, :destroy]

  # GET /articles
  def index
    @articles = Article.all.order(created_at: :desc)
    render json: @articles
  end

  # GET /articles/:id
  def show
    render json: @article
  end

  # POST /articles
  def create
    @article = Article.new(article_params)

    if @article.save
      render json: @article, status: :created
    else
      render json: { errors: @article.errors.full_messages },
             status: :unprocessable_entity
    end
  end

  # PATCH /articles/:id
  def update
    if @article.update(article_params)
      render json: @article
    else
      render json: { errors: @article.errors.full_messages },
             status: :unprocessable_entity
    end
  end

  # DELETE /articles/:id
  def destroy
    @article.destroy
    head :no_content
  end

  private

  # Strong parameters — whitelist allowed attributes
  def article_params
    params.require(:article).permit(:title, :body, :status, :category_id)
  end

  def set_article
    @article = Article.find(params[:id])
  end
end
Pro Tip

Strong parameters (params.require().permit()) are a security feature that prevents mass-assignment attacks. Without them, a malicious user could send extra fields like admin: true in a request body. Always whitelist exactly the attributes you expect — never use params.permit! which allows everything.

4. Rails Generators

Generators are command-line tools that scaffold boilerplate code following Rails conventions. They create models, controllers, migrations, tests, and more. While you should understand what every generated file does, generators save significant time by wiring up naming conventions automatically. Think of them as templates that Rails fills in for you.

bash 
# Create a new Rails API application
rails new blog_api --api --database=postgresql -T

# Generate a model (creates model file + migration)
rails generate model Article title:string body:text status:integer
# Creates:
#   app/models/article.rb
#   db/migrate/20240101000000_create_articles.rb
#   spec/models/article_spec.rb (if RSpec installed)

# Generate a controller
rails generate controller Articles index show create update destroy
# Creates:
#   app/controllers/articles_controller.rb
#   spec/controllers/articles_controller_spec.rb

# Generate a migration only
rails generate migration AddCategoryToArticles category:string
# Creates: db/migrate/20240102000000_add_category_to_articles.rb

# Scaffold — generates model, controller, routes, and views
rails generate scaffold Comment body:text article:references

# Undo any generator
rails destroy model Article
rails destroy controller Articles

# Preview what a generator will create without writing files
rails generate model Article --pretend
Do not blindly run scaffold

The scaffold generator creates a lot of code you may not need. In real projects, most teams generate models and controllers separately because scaffold adds views, helpers, and route entries that clutter API-only applications. Prefer individual generators and always inspect what each one creates before moving on.

5. Rails Folder Structure

Rails enforces a standardized directory layout. Every Rails application has the same structure, which means you can jump into any Rails project and immediately know where things live. This is convention over configuration in action — you never have to configure paths or autoloading because Rails knows exactly where to look for each type of file.

text

|   +-- database.yml            # Database connection settings
|   +-- application.rb          # Main app configuration
|   +-- environments/           # Per-environment settings

6. The Request/Response Cycle in Detail

Understanding exactly what happens when a request hits your Rails application is critical for debugging. Here is the complete lifecycle, step by step, from the moment a user clicks a link or React makes a fetch call to the moment a response arrives back in the browser.

ruby 
# Step 1: Request arrives — GET /articles/42
# The web server (Puma) receives the HTTP request
# and passes it to the Rack middleware stack.

# Step 2: Middleware runs in order
# Each middleware processes the request:
#   - ActionDispatch::SSL (force HTTPS in production)
#   - Rack::Cors (handle CORS headers for React)
#   - ActionDispatch::Cookies
#   - ActionDispatch::Session
#   - ActionDispatch::RequestId (adds unique ID)

# Step 3: Router matches the URL to a controller action
# config/routes.rb: resources :articles
# Match: GET /articles/:id => ArticlesController#show
# params[:id] = "42"

# Step 4: Controller filters and action run
class ArticlesController < ApplicationController
  before_action :set_article, only: [:show]

  def show
    # 4a: before_action fires first — calls set_article
    # 4b: The show action body runs
    render json: @article, status: :ok
  end

  private

  def set_article
    # Step 5: Model queries the database
    # SQL: SELECT * FROM articles WHERE id = 42 LIMIT 1
    @article = Article.find(params[:id])
  rescue ActiveRecord::RecordNotFound
    render json: { error: "Article not found" }, status: :not_found
  end
end

# Step 6: Response is serialized and sent
# HTTP/1.1 200 OK
# Content-Type: application/json; charset=utf-8
# X-Request-Id: abc-123-def-456
# {"id":42,"title":"Hello World","body":"..."}

7. Convention Over Configuration

Rails is built on a philosophy where sensible defaults eliminate the need for configuration files. If you follow the naming conventions, Rails wires everything together automatically. Here is a comprehensive mapping of how Rails derives names across the entire stack:

ruby 
# ═══════════════════════════════════════════
# RAILS NAMING CONVENTIONS
# ═══════════════════════════════════════════

# Model: singular, CamelCase
class Article < ApplicationRecord
end
# File:  app/models/article.rb
# Table: articles (plural, snake_case)

# Controller: plural, CamelCase, suffixed
class ArticlesController < ApplicationController
end
# File: app/controllers/articles_controller.rb

# Multi-word model
class BlogPost < ApplicationRecord
end
# File:       app/models/blog_post.rb
# Table:      blog_posts
# Controller: BlogPostsController
# Route:      /blog_posts

# Foreign key convention
class Comment < ApplicationRecord
  belongs_to :article
  # Expects: article_id column in comments table
end

# Namespaced model
class Admin::User < ApplicationRecord
end
# File:  app/models/admin/user.rb
# Table: users (default) or admin_users (configurable)

# Test the inflector in rails console
"BlogPost".underscore      # => "blog_post"
"blog_post".classify       # => "BlogPost"
"article".pluralize        # => "articles"
"people".singularize       # => "person"
"blog_post".camelize       # => "BlogPost"
"Article".tableize         # => "articles"
The Inflector

Rails uses an inflector to convert between singular/plural and CamelCase/snake_case. It even handles irregular plurals: "person".pluralize returns "people", and "octopus".pluralize returns "octopi". When in doubt, open rails console and test the inflections directly.

Self-Check

Before moving to Chapter 03, honestly answer these questions. If any answer is "no", go back and practice more.

Can you trace a request from URL to JSON response through all MVC layers?

Given GET /articles/42, you should be able to explain: the router matches resources :articles and dispatches to ArticlesController#show with params[:id] = "42". The before_action calls Article.find(42), which generates a SQL SELECT query. The model instance is serialized to JSON and returned with a 200 status code. If you cannot walk through this from memory, review the request cycle section.

Can you list the 7 RESTful routes that resources :articles generates?

It generates: GET /articles (index), GET /articles/new (new), POST /articles (create), GET /articles/:id (show), GET /articles/:id/edit (edit), PATCH /articles/:id (update), and DELETE /articles/:id (destroy). In API mode, you typically use only five of these (skipping new and edit since those are form pages). You should know the HTTP method and URL for each action without checking documentation.

Can you name the file path for a model, controller, migration, and route file?

Model: app/models/article.rb. Controller: app/controllers/articles_controller.rb. Migration: db/migrate/TIMESTAMP_create_articles.rb. Routes: config/routes.rb. If the controller is namespaced under Api::V1, the path becomes app/controllers/api/v1/articles_controller.rb. These paths are conventions that Rails uses for autoloading — they are not configurable.

Do you understand why strong parameters exist and how to use them?

Strong parameters prevent mass-assignment vulnerabilities. Without them, a user could submit { article: { title: "Hack", admin: true } } and potentially set fields you did not intend. The pattern params.require(:article).permit(:title, :body) ensures only title and body can be set from user input. Every create and update action must use strong parameters.

Can you explain Convention over Configuration with three concrete examples?

1. A model named Article automatically maps to a table named articles without any configuration. 2. A file at app/models/article.rb is automatically loaded by Rails without require statements. 3. A belongs_to :author association expects an author_id foreign key column without being told. In each case, Rails uses naming conventions to derive the connection, eliminating boilerplate configuration.
