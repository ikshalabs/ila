Chapter 03 · Phase 1: Ruby & Rails Fundamentals

ActiveRecord Fundamentals

Master Rails’ object-relational mapping layer — migrations for evolving your schema, associations for connecting models, validations for data integrity, callbacks for lifecycle hooks, and the query interface that lets you talk to your database without writing raw SQL.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

ActiveRecord is the Model layer in Rails MVC. It is an Object-Relational Mapping (ORM) that connects your Ruby classes to database tables, rows to objects, and columns to attributes — all without writing a single line of SQL. In a typical Rails application, you spend more time working with ActiveRecord than any other part of the framework. Your models define business rules, data relationships, and validation logic that ensure your application's data stays consistent and correct.

Migrations are how you evolve your database schema over time. Instead of manually running SQL ALTER TABLE statements, you write Ruby migration files that Rails can apply forward or roll back. Every team member runs the same migrations, and your CI/CD pipeline applies them automatically during deployment. This gives you a version-controlled, reproducible database schema that matches your code at every point in history.

Associations are what make ActiveRecord powerful for real applications. A blog post has_many :comments. A comment belongs_to :article. A student has_many :courses, through: :enrollments. These declarations give you intuitive Ruby methods for traversing relationships — no JOIN queries required. Understanding associations deeply is the difference between writing clean, efficient code and accidentally triggering N+1 query performance disasters.

Key Insight

ActiveRecord follows the Active Record pattern described by Martin Fowler: an object wraps a row in a database table, encapsulates database access, and adds domain logic on top. This means your Article model is simultaneously a Ruby class with behavior and a gateway to the articles table. This dual nature is what makes Rails models so productive — and why keeping them well-organized matters.

Key Concepts

1. Migrations

Migrations are Ruby classes that define changes to your database schema. Each migration file has a timestamp in its name, and Rails runs them in chronological order. You can add columns, remove columns, create tables, add indexes, and modify column types — all in Ruby, all version-controlled, and all reversible.

ruby 
# db/migrate/20240101000000_create_articles.rb
class CreateArticles < ActiveRecord::Migration[7.1]
  def change
    create_table :articles do |t|
      t.string   :title,   null: false
      t.text     :body
      t.integer  :status,  default: 0
      t.string   :slug,    null: false

      t.timestamps # creates created_at and updated_at
    end

    add_index :articles, :slug, unique: true
    add_index :articles, :status
  end
end

# db/migrate/20240102000000_create_comments.rb
class CreateComments < ActiveRecord::Migration[7.1]
  def change
    create_table :comments do |t|
      t.text       :body,    null: false
      t.references :article, null: false, foreign_key: true
      t.references :user,    null: false, foreign_key: true

      t.timestamps
    end
  end
end

# db/migrate/20240103000000_add_category_to_articles.rb
class AddCategoryToArticles < ActiveRecord::Migration[7.1]
  def change
    add_column    :articles, :category, :string
    add_index     :articles, :category
    change_column_default :articles, :status, from: nil, to: 0
  end
end
bash 
# Run pending migrations
rails db:migrate

# Roll back the last migration
rails db:rollback

# Roll back 3 migrations
rails db:rollback STEP=3

# Check migration status
rails db:migrate:status

# Generate a migration
rails generate migration AddPublishedAtToArticles published_at:datetime
Never edit a migration after it has been run in production

Once a migration is shared with other developers or deployed, treat it as immutable. If you need to change something, write a new migration. Editing old migrations causes schema mismatches between environments and breaks db:migrate for the entire team.

2. Associations

Associations declare relationships between models. They add methods to your models for querying related records, and they tell ActiveRecord how to construct JOIN queries. The three most common associations are belongs_to, has_many, and has_many :through. Getting associations right is critical for both correctness and performance.

ruby 
# ═══════════════════════════════════════════
# ONE-TO-MANY: Article has many Comments
# ═══════════════════════════════════════════

# app/models/article.rb
class Article < ApplicationRecord
  has_many :comments, dependent: :destroy
  belongs_to :user    # author of the article
  has_many :taggings
  has_many :tags, through: :taggings
end

# app/models/comment.rb
class Comment < ApplicationRecord
  belongs_to :article
  belongs_to :user
end

# ═══════════════════════════════════════════
# MANY-TO-MANY: Articles have many Tags through Taggings
# ═══════════════════════════════════════════

# app/models/tag.rb
class Tag < ApplicationRecord
  has_many :taggings
  has_many :articles, through: :taggings
end

# app/models/tagging.rb (join model)
class Tagging < ApplicationRecord
  belongs_to :article
  belongs_to :tag
end

# Usage in rails console:
article = Article.first
article.comments             # => [#<Comment...>, ...]
article.comments.count       # => 5 (SQL COUNT query)
article.comments.create!(    # Create a comment for this article
  body: "Great post!",
  user: current_user
)

article.tags                 # => [#<Tag name: "ruby">, ...]
article.tags << Tag.find_by(name: "rails")

comment = Comment.first
comment.article              # => #<Article...>
comment.article.title        # => "My First Post"

3. Validations

Validations ensure that only valid data gets saved to your database. They run automatically before save, create, and update. If validations fail, the record is not persisted and error messages are populated on the model. Validations are your first line of defense against bad data — always validate at the model level, even if you also validate in the front end.

ruby 
# app/models/article.rb
class Article < ApplicationRecord
  # Presence — field must not be blank
  validates :title, presence: true
  validates :body,  presence: true

  # Length constraints
  validates :title, length: { minimum: 5, maximum: 200 }
  validates :body,  length: { minimum: 20 }

  # Uniqueness — no duplicate slugs
  validates :slug, uniqueness: true, presence: true

  # Numericality
  validates :word_count, numericality: {
    only_integer: true,
    greater_than: 0
  }, allow_nil: true

  # Inclusion — must be one of these values
  validates :status, inclusion: {
    in: %w[draft published archived],
    message: "%{value} is not a valid status"
  }

  # Format — validates against a regex pattern
  validates :slug, format: {
    with: /\A[a-z0-9\-]+\z/,
    message: "must be lowercase letters, numbers, and hyphens only"
  }

  # Custom validation method
  validate :title_is_not_body

  private

  def title_is_not_body
    if title.present? && body.present? && title == body
      errors.add(:title, "cannot be the same as the body")
    end
  end
end

# Using validations
article = Article.new(title: "")
article.valid?               # => false
article.errors.full_messages # => ["Title can't be blank", ...]
article.errors[:title]       # => ["can't be blank", "is too short..."]

article.save                 # => false (does not raise)
article.save!                # => raises ActiveRecord::RecordInvalid

4. Callbacks

Callbacks are hooks into the ActiveRecord lifecycle. They let you run code automatically before or after key events like validation, saving, creating, updating, and destroying. Common use cases include generating slugs before saving, sending notifications after creation, and cleaning up related data before destruction.

ruby 
# app/models/article.rb
class Article < ApplicationRecord
  # Before validation — runs before validations
  before_validation :generate_slug, on: :create

  # Before save — runs before every save (create + update)
  before_save :calculate_word_count

  # After create — runs only after a new record is created
  after_create :notify_subscribers

  # After update — runs only after an existing record is updated
  after_update :clear_cache, if: :saved_change_to_body?

  # Before destroy — runs before a record is deleted
  before_destroy :check_if_destroyable

  private

  def generate_slug
    self.slug = title
      .downcase
      .gsub(/[^a-z0-9\s-]/, "")
      .gsub(/[\s-]+/, "-")
      .gsub(/^-|-$/, "")
  end

  def calculate_word_count
    self.word_count = body.to_s.split.size
  end

  def notify_subscribers
    # Queue a background job to send emails
    ArticleNotificationJob.perform_later(self.id)
  end

  def clear_cache
    Rails.cache.delete("article_#{id}")
  end

  def check_if_destroyable
    if comments.exists?
      errors.add(:base, "Cannot delete article with comments")
      throw :abort
    end
  end
end
Use callbacks sparingly

Callbacks are powerful but can make your code harder to reason about. When an after_create callback sends an email, every test that creates that model also sends an email (unless mocked). For complex side effects, prefer service objects that you call explicitly. Reserve callbacks for simple data transformations like slug generation and word count calculation.

5. Query Interface

ActiveRecord provides a rich query interface that translates Ruby method calls into SQL. Methods like where, order, limit, and joins are chainable and lazy — they do not execute SQL until you actually need the data. This lets you build complex queries incrementally.

ruby 
# ═══════════════════════════════════════════
# FINDING RECORDS
# ═══════════════════════════════════════════

# Find by primary key (raises RecordNotFound if missing)
Article.find(42)

# Find by attribute (returns nil if missing)
Article.find_by(slug: "hello-world")

# Find or raise
Article.find_by!(slug: "hello-world")

# ═══════════════════════════════════════════
# FILTERING WITH WHERE
# ═══════════════════════════════════════════

# Exact match
Article.where(status: "published")

# Multiple conditions
Article.where(status: "published", category: "ruby")

# Range
Article.where(created_at: 1.week.ago..)

# Array of values (IN clause)
Article.where(status: ["published", "archived"])

# SQL fragment (use ? for parameterized queries)
Article.where("word_count > ?", 500)

# NOT
Article.where.not(status: "draft")

# ═══════════════════════════════════════════
# ORDERING, LIMITING, AND CHAINING
# ═══════════════════════════════════════════

Article
  .where(status: "published")
  .order(created_at: :desc)
  .limit(10)
  .offset(20)

# Aggregations
Article.count                          # => 142
Article.where(status: "published").count # => 98
Article.average(:word_count)            # => 850.5
Article.maximum(:word_count)            # => 3200
Article.sum(:word_count)                # => 120870

# Pluck — returns array of values (skips model instantiation)
Article.where(status: "published").pluck(:title)
# => ["Hello World", "Ruby Tips", ...]

# Eager loading — prevent N+1 queries
Article.includes(:comments, :tags).where(status: "published")
Pro Tip

Use .to_sql on any ActiveRecord relation to see the exact SQL it will generate: Article.where(status: "published").order(:title).to_sql. This is invaluable for debugging queries and understanding what ActiveRecord is doing under the hood.

6. Scopes

Scopes are named queries defined on the model. They encapsulate common filtering logic and can be chained with other scopes or query methods. Scopes keep your controllers clean by moving query logic into the model where it belongs.

ruby 
# app/models/article.rb
class Article < ApplicationRecord
  # Simple scopes
  scope :published,  -> { where(status: "published") }
  scope :draft,      -> { where(status: "draft") }
  scope :archived,   -> { where(status: "archived") }
  scope :recent,     -> { order(created_at: :desc) }

  # Scope with an argument
  scope :by_category, ->(cat) { where(category: cat) }

  # Scope with a time range
  scope :created_after, ->(date) { where("created_at > ?", date) }
  scope :this_week, -> { where(created_at: 1.week.ago..) }
  scope :this_month, -> { where(created_at: 1.month.ago..) }

  # Scope with eager loading
  scope :with_comments, -> { includes(:comments) }
  scope :popular, -> {
    left_joins(:comments)
      .group(:id)
      .order("COUNT(comments.id) DESC")
  }
end

# Chaining scopes — reads like English
Article.published.recent.limit(10)
Article.published.by_category("ruby").this_week
Article.draft.created_after(3.days.ago)
Article.published.popular.limit(5)

# In a controller — clean and readable
class ArticlesController < ApplicationController
  def index
    @articles = Article
      .published
      .by_category(params[:category])
      .recent
      .limit(params[:limit] || 20)
      .offset(params[:offset] || 0)

    render json: @articles
  end
end
What is the difference between a scope and a class method?

Functionally, scopes and class methods are very similar. The key difference is that scopes always return an ActiveRecord relation (even if the lambda returns nil, it returns all), making them safely chainable. Class methods can return anything, including nil, which would break a chain. Use scopes for simple, reusable query fragments. Use class methods when you need more complex logic or conditional branching.

Self-Check

Before moving to Chapter 04, honestly answer these questions. If any answer is "no", go back and practice more.

Can you write a migration to create a table with columns, indexes, and foreign keys?

You should be able to write create_table :orders do |t| with various column types (string, text, integer, decimal, boolean, datetime, references), add indexes with add_index, and set up foreign key constraints with foreign_key: true. You should also know how to add and remove columns in subsequent migrations.

Can you set up has_many :through without referring to documentation?

A many-to-many relationship requires three models: the two main models and a join model. For example: Article has_many :taggings, Article has_many :tags, through: :taggings. Tag has_many :taggings, Tag has_many :articles, through: :taggings. Tagging belongs_to :article, Tagging belongs_to :tag. The join table needs both foreign key columns.

Can you chain scopes to build a complex query without writing raw SQL?

Given a request like "find published articles in the ruby category from this week, ordered by most comments, limited to 10", you should be able to write: Article.published.by_category("ruby").this_week.popular.limit(10). Each scope is a simple, reusable building block. If you cannot chain scopes fluently, practice defining and composing them in rails console.

Do you understand what an N+1 query is and how to prevent it?

An N+1 query happens when you load a list of records (1 query) and then access an association on each record individually (N queries). For example: Article.all.each { |a| a.comments.count } fires one query for articles plus one query per article for comments. Fix it with Article.includes(:comments).all, which loads all comments in a single additional query. Use .includes for associations you will access, and install the bullet gem to detect N+1 issues automatically.

Can you explain the difference between save and save!?

save returns true on success and false on failure (validations failed). save! returns true on success and raises ActiveRecord::RecordInvalid on failure. Use save in controllers where you want to handle the failure case (show errors to the user). Use save! in seeds, background jobs, and situations where failure indicates a bug that should not be silently ignored. The same bang/no-bang pattern applies to create/create! and update/update!.
