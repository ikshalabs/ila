Chapter 03 · Phase 2: Building APIs

ActiveRecord Fundamentals

Master the ORM that powers every Rails application. Learn migrations for schema evolution, associations for modeling relationships, validations for data integrity, callbacks for lifecycle hooks, and the query interface for efficient database access.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

ActiveRecord is the bridge between your Ruby code and the database. It implements the Active Record pattern — each model class maps to a database table, and each instance maps to a row. You never write raw SQL for standard operations. Instead, you call methods like User.find(1), article.save, or Post.where(published: true), and ActiveRecord translates them into optimized SQL.

In every Rails application, the database schema is the source of truth. Migrations let you evolve that schema incrementally and reversibly, associations let you express relationships between tables as Ruby method calls, and validations ensure bad data never reaches the database. Mastering ActiveRecord is not optional — it is the difference between an application that works and one that works correctly.

Key Insight

ActiveRecord is not just a query builder — it is a domain modeling tool. When you write has_many :comments, you are not just configuring a JOIN. You are declaring that an article can have comments, and Rails will give you methods to create, find, count, and delete those comments through the association.

Key Concepts

1. Migrations — Schema as Code

Migrations are Ruby classes that describe database schema changes. Each migration has a timestamp, runs in order, and can be rolled back. They are version control for your database — every developer and every server applies the same changes in the same order.

ruby
# db/migrate/20240115120000_create_products.rb
class CreateProducts < ActiveRecord::Migration[7.1]
  def change
    create_table :products do |t|
      t.string   :name,        null: false
      t.text     :description
      t.decimal  :price,       precision: 10, scale: 2, null: false
      t.integer  :stock_count, default: 0
      t.boolean  :active,      default: true
      t.references :category,  foreign_key: true

      t.timestamps # adds created_at and updated_at
    end

    add_index :products, :name
    add_index :products, [:category_id, :active]
  end
end
bash
# Generate a migration
bin/rails generate migration CreateProducts name:string price:decimal

# Run pending migrations
bin/rails db:migrate

# Rollback the last migration
bin/rails db:rollback

# Check migration status
bin/rails db:migrate:status

# Reset database (drop, create, migrate, seed)
bin/rails db:reset
Never Edit Old Migrations

Once a migration has been committed and run by other developers, never modify it. Create a new migration instead. Editing old migrations causes schema drift — your database will not match your teammates' databases. Always move forward.

2. Associations — Modeling Relationships

Associations declare relationships between models. Rails supports six types, but you will use three constantly: belongs_to, has_many, and has_many :through. These generate methods that let you navigate between related records without writing SQL JOINs.

ruby
# app/models/user.rb
class User < ApplicationRecord
  has_many :articles, dependent: :destroy
  has_many :comments
  has_many :commented_articles, through: :comments, source: :article
end

# app/models/article.rb
class Article < ApplicationRecord
  belongs_to :user
  has_many   :comments, dependent: :destroy
  has_many   :tags,     through: :article_tags
  has_many   :article_tags
end

# app/models/comment.rb
class Comment < ApplicationRecord
  belongs_to :user
  belongs_to :article
end

# Usage — these methods are auto-generated by associations
user = User.first
user.articles                  # all articles by this user
user.articles.create(title: "New Post")  # create with user_id set
user.articles.count            # COUNT query, not loading all records

article = Article.first
article.user                   # the author
article.comments.recent        # chain scopes on associations
article.tags                   # through join table

3. Validations — Data Integrity

Validations run before saving a record to the database. If any validation fails, the save is aborted and error messages are attached to the record. This is your first line of defense against bad data.

ruby
class Product < ApplicationRecord
  # Presence — field cannot be nil or empty
  validates :name,  presence: true
  validates :price, presence: true

  # Numericality — must be a number with constraints
  validates :price, numericality: { greater_than: 0 }
  validates :stock_count, numericality: { only_integer: true, greater_than_or_equal_to: 0 }

  # Uniqueness — no duplicates
  validates :sku, uniqueness: true, allow_nil: true

  # Length constraints
  validates :name, length: { minimum: 2, maximum: 200 }
  validates :description, length: { maximum: 5000 }

  # Format — regex pattern matching
  validates :sku, format: { with: /\A[A-Z]{2}-\d{6}\z/,
    message: "must be like AB-123456" }, allow_nil: true

  # Inclusion — must be one of these values
  validates :status, inclusion: { in: %w[draft active archived] }

  # Custom validation method
  validate :price_must_be_reasonable

  private

  def price_must_be_reasonable
    if price.present? && price > 1_000_000
      errors.add(:price, "seems unreasonably high — please verify")
    end
  end
end

# Working with validation errors
product = Product.new(name: "", price: -5)
product.valid?                # => false
product.errors.full_messages  # => ["Name can't be blank", "Price must be greater than 0"]
product.errors[:name]         # => ["can't be blank"]

4. Callbacks — Lifecycle Hooks

Callbacks let you hook into the lifecycle of an ActiveRecord object — before validation, after save, before destroy, and more. Use them sparingly for cross-cutting concerns like setting defaults or cleaning data. Avoid putting business logic in callbacks.

ruby
class Article < ApplicationRecord
  # Before validation — normalize data
  before_validation :normalize_title

  # Before save — set computed fields
  before_save :generate_slug

  # After create — trigger side effects
  after_create :notify_subscribers

  # Before destroy — clean up dependencies
  before_destroy :check_if_destroyable

  private

  def normalize_title
    self.title = title&.strip&.titleize
  end

  def generate_slug
    self.slug = title.parameterize if title_changed?
  end

  def notify_subscribers
    NotificationJob.perform_later(self.id)
  end

  def check_if_destroyable
    if comments.any?
      errors.add(:base, "Cannot delete article with comments")
      throw(:abort)
    end
  end
end
Callback Discipline

Callbacks are the most abused feature in Rails. Use them for data normalization (stripping whitespace, generating slugs) and simple side effects (enqueuing a background job). Never put complex business logic in callbacks — it makes your code hard to test, hard to debug, and creates invisible coupling between operations.

5. The Query Interface

ActiveRecord provides a chainable query interface that builds SQL incrementally. Queries are lazy — they do not execute until you need the results. This lets you build complex queries from simple, reusable pieces.

ruby
# Finding records
Product.find(1)                     # raises RecordNotFound if missing
Product.find_by(sku: "AB-123456")   # returns nil if missing
Product.find_by!(sku: "AB-123456")  # raises if missing

# Where clauses
Product.where(active: true)
Product.where("price > ?", 50)     # parameterized to prevent SQL injection
Product.where(price: 10..100)       # BETWEEN 10 AND 100
Product.where.not(status: "archived")

# Ordering and limiting
Product.order(price: :desc)
Product.order(:name)
Product.limit(10).offset(20)        # pagination building block

# Selecting specific columns
Product.select(:id, :name, :price)

# Aggregations
Product.count                        # SELECT COUNT(*)
Product.average(:price)              # SELECT AVG(price)
Product.sum(:stock_count)            # SELECT SUM(stock_count)
Product.maximum(:price)              # SELECT MAX(price)

# Chaining — build complex queries from simple parts
Product
  .where(active: true)
  .where("price > ?", 25)
  .order(created_at: :desc)
  .limit(10)
  .includes(:category)              # eager load to avoid N+1

# Scopes — named, reusable query fragments
class Product < ApplicationRecord
  scope :active,     -> { where(active: true) }
  scope :expensive,  -> { where("price > ?", 100) }
  scope :in_stock,   -> { where("stock_count > 0") }
  scope :search,     ->(q) { where("name ILIKE ?", "%#{q}%") }
end

# Scopes chain beautifully
Product.active.in_stock.expensive.search("widget")
N+1 Queries

The N+1 problem is the most common performance issue in Rails. If you load 100 articles and then call article.user.name for each one, Rails executes 101 queries (1 for articles + 100 for users). Use .includes(:user) to eager-load the association in a single query. The bullet gem can detect N+1 queries automatically.

6. Seeding and the Rails Console

The seed file (db/seeds.rb) populates your database with initial data. The Rails console (bin/rails console) is your REPL for interacting with models directly. Both are essential for development and debugging.

ruby
# db/seeds.rb
puts "Seeding categories..."
electronics = Category.find_or_create_by!(name: "Electronics")
clothing    = Category.find_or_create_by!(name: "Clothing")

puts "Seeding products..."
50.times do |i|
  Product.create!(
    name: "Product #{i + 1}",
    price: rand(10.0..500.0).round(2),
    stock_count: rand(0..100),
    category: [electronics, clothing].sample,
    status: %w[draft active archived].sample
  )
end

puts "Seeded #{Product.count} products."
Essential Rails console commands
ruby
# Start the console
# bin/rails console

# Create records
user = User.create!(name: "Alice", email: "alice@example.com")

# Query records
User.all
User.where(active: true).count
User.last

# Inspect SQL being generated
User.where(active: true).to_sql

# Reload a modified record from database
user.reload

# Sandbox mode — all changes rolled back on exit
# bin/rails console --sandbox

Self-Check

Before moving to Chapter 04, honestly answer these questions.

Can you write a migration, model, and validations from memory?

Without checking notes, create a Payment model with amount (decimal), status (string with inclusion), and a belongs_to user. Add validations and a scope. If you need references, practice until it is automatic.

Do you understand the N+1 problem and how to fix it?

Load 10 articles and display each author's name. Check your Rails log — do you see 11 queries or 2? If you see 11, you have an N+1 problem. Add .includes(:author) and verify it drops to 2 queries.

Can you explain when to use a callback vs a service object?

Callbacks for data normalization (generating slugs, stripping whitespace). Service objects for business operations (processing an order, sending notifications). If your callback calls an external service or has conditional logic, extract it to a service object.
