Chapter 07 · Phase 4: Full-Stack Integration

Connecting React to Rails

Bridge your frontend and backend. Learn to make HTTP requests from React to Rails APIs using fetch and Axios, configure CORS for cross-origin requests, implement JWT authentication flows, and build robust error handling with loading states.

Estimated: 5–6 days
3 Advancement Levels
3 Assignments

Why This Matters

You have built a Rails API (Chapters 3–4) and a React frontend (Chapters 5–6) as separate applications. Now they need to talk to each other. The connection between frontend and backend is where most full-stack bugs live: CORS failures that produce cryptic browser errors, authentication tokens that expire silently, network requests that leave users staring at infinite spinners.

This chapter teaches you to build the bridge correctly. You will learn two approaches for HTTP requests (the built-in fetch API and the popular Axios library), how to configure Rails for cross-origin requests, how to implement token-based authentication (JWT), and how to handle every edge case — loading states, error messages, request cancellation, and retry logic.

Key Insight

In a decoupled architecture, the React app and Rails API are separate deployments. React runs in the browser and makes HTTP requests to Rails running on a server. They communicate exclusively through JSON over HTTP. Understanding this boundary is essential to debugging integration issues.

Key Concepts

1. The Fetch API — Built-in HTTP Requests

Every modern browser includes the fetch function. It returns a Promise that resolves to a Response object. Unlike Axios, fetch does not throw on HTTP error status codes (404, 500) — you must check response.ok yourself.

jsx
// GET request with fetch
async function fetchArticles() {
  const response = await fetch("http://localhost:3000/api/v1/articles");

  // fetch does NOT throw on 404/500 — you must check manually
  if (!response.ok) {
    throw new Error(`HTTP error! Status: ${response.status}`);
  }

  const data = await response.json();
  return data;
}

// POST request with fetch
async function createArticle(article) {
  const response = await fetch("http://localhost:3000/api/v1/articles", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${localStorage.getItem("token")}`,
    },
    body: JSON.stringify({ article }),
  });

  if (!response.ok) {
    const errors = await response.json();
    throw new Error(errors.message || "Failed to create article");
  }

  return response.json();
}

// DELETE request
async function deleteArticle(id) {
  const response = await fetch(`http://localhost:3000/api/v1/articles/${id}`, {
    method: "DELETE",
    headers: {
      "Authorization": `Bearer ${localStorage.getItem("token")}`,
    },
  });

  if (!response.ok) throw new Error("Failed to delete");
  return true;
}
Fetch Gotcha

fetch only rejects on network errors (server unreachable, DNS failure). A 404 or 500 response is still a "successful" HTTP response from fetch's perspective. Always check response.ok or response.status before parsing the body.

2. Axios — A Better HTTP Client

Axios is a popular third-party HTTP client that improves on fetch in several ways: it automatically throws on error status codes, transforms JSON automatically, supports request/response interceptors, and makes cancellation easier. Most production React+Rails apps use Axios.

bash
# Install Axios in your React project
npm install axios
jsx
// api/client.js — Create a configured Axios instance
import axios from "axios";

const api = axios.create({
  baseURL: "http://localhost:3000/api/v1",
  timeout: 10000,  // 10 second timeout
  headers: {
    "Content-Type": "application/json",
    "Accept": "application/json",
  },
});

// Request interceptor — attach auth token to every request
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("token");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// Response interceptor — handle 401 globally
api.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem("token");
      window.location.href = "/login";
    }
    return Promise.reject(error);
  }
);

export default api;

// Using the configured client
import api from "./api/client";

// GET — Axios auto-parses JSON
const { data } = await api.get("/articles");

// POST — Axios auto-stringifies the body
const { data: newArticle } = await api.post("/articles", {
  article: { title: "Hello", body: "World" }
});

// PATCH
await api.patch(`/articles/${id}`, { article: { title: "Updated" } });

// DELETE
await api.delete(`/articles/${id}`);

3. CORS — Cross-Origin Resource Sharing

When your React app (running on localhost:5173) makes a request to your Rails API (running on localhost:3000), the browser blocks it by default. This is the Same-Origin Policy. CORS is the mechanism that allows your Rails server to tell the browser "yes, this origin is allowed to make requests to me."

ruby
# Gemfile
gem "rack-cors"

# config/initializers/cors.rb
Rails.application.config.middleware.insert_before 0, Rack::Cors do
  allow do
    # In development, allow your React dev server
    origins "http://localhost:5173",
            "http://localhost:3001"

    resource "/api/*",
      headers: :any,
      methods: [:get, :post, :put, :patch, :delete, :options],
      expose: ["Authorization"],
      max_age: 600
  end
end

# In production, use environment variables for origins
Rails.application.config.middleware.insert_before 0, Rack::Cors do
  allow do
    origins ENV.fetch("ALLOWED_ORIGINS", "").split(",")

    resource "/api/*",
      headers: :any,
      methods: [:get, :post, :put, :patch, :delete, :options],
      credentials: true
  end
end

Never Use origins "*" in Production

Setting origins "*" allows any website to make authenticated requests to your API. This is a serious security vulnerability. Always whitelist specific origins in production. In development, specify your exact dev server URLs.

4. JWT Authentication Flow

JSON Web Tokens (JWT) are the standard way to authenticate API requests in a decoupled architecture. The flow: user logs in, Rails returns a JWT, React stores it, and sends it with every subsequent request in the Authorization header.

ruby
# Gemfile
gem "jwt"
gem "bcrypt"

# app/models/user.rb
class User < ApplicationRecord
  has_secure_password
  validates :email, presence: true, uniqueness: true
end

# app/lib/json_web_token.rb
class JsonWebToken
  SECRET_KEY = Rails.application.credentials.secret_key_base

  def self.encode(payload, exp = 24.hours.from_now)
    payload[:exp] = exp.to_i
    JWT.encode(payload, SECRET_KEY)
  end

  def self.decode(token)
    body = JWT.decode(token, SECRET_KEY)[0]
    HashWithIndifferentAccess.new(body)
  rescue JWT::DecodeError, JWT::ExpiredSignature
    nil
  end
end

# app/controllers/api/v1/auth_controller.rb
class Api::V1::AuthController < ApplicationController
  def login
    user = User.find_by(email: params[:email])

    if user&.authenticate(params[:password])
      token = JsonWebToken.encode(user_id: user.id)
      render json: {
        token: token,
        user: { id: user.id, email: user.email, name: user.name }
      }
    else
      render json: { error: "Invalid email or password" },
             status: :unauthorized
    end
  end
end

# app/controllers/concerns/authenticatable.rb
module Authenticatable
  extend ActiveSupport::Concern

  private

  def current_user
    return @current_user if defined?(@current_user)
    header = request.headers["Authorization"]
    token = header&.split(" ")&.last
    decoded = JsonWebToken.decode(token)
    @current_user = User.find_by(id: decoded&.dig(:user_id))
  end

  def authenticate_user!
    render json: { error: "Not authorized" }, status: :unauthorized unless current_user
  end
end
jsx
// React: AuthContext with JWT
import { createContext, useContext, useState, useEffect } from "react";
import api from "../api/client";

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  // On mount, check if token exists and validate it
  useEffect(() => {
    const token = localStorage.getItem("token");
    if (token) {
      api.get("/auth/me")
        .then(({ data }) => setUser(data.user))
        .catch(() => localStorage.removeItem("token"))
        .finally(() => setLoading(false));
    } else {
      setLoading(false);
    }
  }, []);

  const login = async (email, password) => {
    const { data } = await api.post("/auth/login", { email, password });
    localStorage.setItem("token", data.token);
    setUser(data.user);
    return data.user;
  };

  const logout = () => {
    localStorage.removeItem("token");
    setUser(null);
  };

  return (
    <AuthContext.Provider value={{ user, login, logout, loading }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);

5. Loading States and Error Handling

Every network request has three states: loading, success, and error. Handling all three is what separates a professional application from a prototype. Users should always know what is happening — never leave them staring at a blank screen.

jsx
// hooks/useApi.js — Complete data fetching hook
import { useState, useEffect, useCallback } from "react";
import api from "../api/client";

export function useApi(url, options = {}) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  const fetchData = useCallback(async () => {
    const controller = new AbortController();
    setLoading(true);
    setError(null);

    try {
      const { data } = await api.get(url, {
        signal: controller.signal,
        ...options,
      });
      setData(data);
    } catch (err) {
      if (!axios.isCancel(err)) {
        setError(err.response?.data?.error || err.message);
      }
    } finally {
      setLoading(false);
    }

    return () => controller.abort();
  }, [url]);

  useEffect(() => {
    const cleanup = fetchData();
    return () => cleanup.then(abort => abort());
  }, [fetchData]);

  return { data, loading, error, refetch: fetchData };
}

// Using the hook in a component
function ArticlesPage() {
  const { data: articles, loading, error, refetch } = useApi("/articles");

  if (loading) {
    return (
      <div className="loading-skeleton">
        {[1, 2, 3].map(i => (
          <div key={i} className="skeleton-card" />
        ))}
      </div>
    );
  }

  if (error) {
    return (
      <div className="error-banner">
        <p>Something went wrong: {error}</p>
        <button onClick={refetch}>Try Again</button>
      </div>
    );
  }

  if (!articles?.length) {
    return <p>No articles yet. Create your first one!</p>;
  }

  return (
    <ul>
      {articles.map(article => (
        <li key={article.id}>{article.title}</li>
      ))}
    </ul>
  );
}
Pro Tip: Skeleton Loading

Instead of a spinner, use skeleton screens that match the shape of the content being loaded. They reduce perceived load time because users see the layout taking shape. Libraries like react-loading-skeleton make this easy.

6. Organizing an API Service Layer

As your application grows, raw API calls scattered across components become unmaintainable. Create a service layer that centralizes all API interactions. Each resource gets its own service file with methods for CRUD operations.

jsx
// services/articleService.js
import api from "../api/client";

const ArticleService = {
  getAll(params = {}) {
    return api.get("/articles", { params });
    // GET /api/v1/articles?page=1&per_page=20
  },

  getById(id) {
    return api.get(`/articles/${id}`);
  },

  create(articleData) {
    return api.post("/articles", { article: articleData });
  },

  update(id, articleData) {
    return api.patch(`/articles/${id}`, { article: articleData });
  },

  delete(id) {
    return api.delete(`/articles/${id}`);
  },

  search(query) {
    return api.get("/articles/search", { params: { q: query } });
  },
};

export default ArticleService;

// Usage in a component — clean and readable
import ArticleService from "../services/articleService";

function CreateArticleForm() {
  const [title, setTitle] = useState("");
  const [body, setBody] = useState("");
  const [submitting, setSubmitting] = useState(false);
  const [errors, setErrors] = useState({});

  const handleSubmit = async (e) => {
    e.preventDefault();
    setSubmitting(true);
    setErrors({});

    try {
      const { data } = await ArticleService.create({ title, body });
      navigate(`/articles/${data.id}`);
    } catch (err) {
      if (err.response?.status === 422) {
        setErrors(err.response.data.errors);
      } else {
        setErrors({ base: "Something went wrong. Please try again." });
      }
    } finally {
      setSubmitting(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      {errors.base && <div className="error">{errors.base}</div>}
      <input
        value={title}
        onChange={(e) => setTitle(e.target.value)}
        placeholder="Title"
      />
      {errors.title && <span className="field-error">{errors.title}</span>}
      <textarea
        value={body}
        onChange={(e) => setBody(e.target.value)}
        placeholder="Body"
      />
      <button type="submit" disabled={submitting}>
        {submitting ? "Creating..." : "Create Article"}
      </button>
    </form>
  );
}

Self-Check

Can you explain the difference between fetch and Axios?

fetch is built into the browser and does not throw on HTTP errors (you must check response.ok). It returns a Response object that requires .json() to parse. Axios throws on 4xx/5xx status codes, auto-parses JSON, supports interceptors and request cancellation out of the box, and has a simpler API. Use fetch for simple projects, Axios for anything with authentication or complex error handling.

Do you know what CORS is and why the browser blocks cross-origin requests?

CORS (Cross-Origin Resource Sharing) is a security mechanism. The Same-Origin Policy prevents a website on one origin (protocol + domain + port) from making requests to a different origin. Without this, any malicious website could make authenticated requests to your bank's API using your cookies. CORS allows the server to explicitly whitelist which origins are allowed by sending Access-Control-Allow-Origin headers. For non-simple requests (POST with JSON), the browser sends a preflight OPTIONS request first.

Can you implement a JWT authentication flow from scratch?

The flow: (1) User submits credentials to POST /api/v1/auth/login. (2) Rails verifies credentials with bcrypt, generates a JWT with JWT.encode containing the user_id and expiration. (3) React stores the token in localStorage. (4) Every subsequent request includes Authorization: Bearer <token> via an Axios interceptor. (5) Rails decodes the token, finds the user, and authorizes the request. (6) On 401, the interceptor clears the token and redirects to login.
