Chapter 07 · Phase 3: Full-Stack Integration

Connecting React to Rails

Bridge the gap between your React front-end and Rails API — HTTP requests with fetch and axios, CORS configuration, authentication tokens, loading and error states, and production-ready API integration patterns.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Until now, your Rails API and React front-end have lived in separate worlds. Rails serves JSON endpoints, and React renders components with hardcoded data. In production, these two halves must communicate seamlessly over HTTP. This chapter is where your full-stack application comes together. Every feature your users interact with — viewing a list of products, submitting a form, logging in — requires a React component to make an HTTP request to a Rails endpoint and handle the response.

The integration layer between React and Rails introduces real-world challenges that do not exist when working with either technology alone. CORS (Cross-Origin Resource Sharing) will block your requests if not configured correctly. Network requests are asynchronous, so your components need to handle loading states, error states, and the possibility that data arrives after the component unmounts. Authentication requires sending tokens with every request and handling token expiration gracefully.

Getting this integration right is critical. A poorly designed API layer leads to duplicated fetch logic scattered across components, inconsistent error handling, race conditions, and security vulnerabilities. A well-designed API layer centralizes HTTP logic, provides consistent loading and error patterns, and makes your application resilient to network failures. The patterns in this chapter are what separate a demo project from a production application.

Key Insight

The React front-end and Rails API are separate applications that happen to work together. React runs in the browser, Rails runs on a server. They communicate via HTTP, exchanging JSON. This separation means you can deploy, scale, and test them independently — but you must explicitly handle the boundary between them.

Key Concepts

1. The Fetch API — Making HTTP Requests

The Fetch API is built into every modern browser. It provides a clean, promise-based interface for making HTTP requests. Unlike older approaches like XMLHttpRequest, fetch uses Promises and integrates naturally with async/await. One important quirk: fetch does not reject the promise on HTTP error status codes (404, 500). You must check response.ok yourself.

jsx 
// GET request — fetching data
async function fetchUsers() {
  const response = await fetch("http://localhost:3000/api/v1/users");

  if (!response.ok) {
    throw new Error(`HTTP error: ${response.status}`);
  }

  const users = await response.json();
  return users;
}

// POST request — creating a resource
async function createUser(userData) {
  const response = await fetch("http://localhost:3000/api/v1/users", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Accept": "application/json",
    },
    body: JSON.stringify({ user: userData }),
  });

  if (!response.ok) {
    const errors = await response.json();
    throw new Error(errors.message || "Failed to create user");
  }

  return await response.json();
}

// PUT request — updating a resource
async function updateUser(id, updates) {
  const response = await fetch(`http://localhost:3000/api/v1/users/${id}`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ user: updates }),
  });

  if (!response.ok) throw new Error(`Update failed: ${response.status}`);
  return await response.json();
}

// DELETE request — removing a resource
async function deleteUser(id) {
  const response = await fetch(`http://localhost:3000/api/v1/users/${id}`, {
    method: "DELETE",
  });

  if (!response.ok) throw new Error(`Delete failed: ${response.status}`);
}

2. Axios — A Better HTTP Client

While fetch works well, many production applications use axios for HTTP requests. Axios provides automatic JSON parsing, request/response interceptors (perfect for adding auth tokens), timeout support, and better error handling. It also rejects the promise on HTTP error codes, which eliminates the response.ok boilerplate.

jsx 
import axios from "axios";

// Create a configured instance for your Rails API
const api = axios.create({
  baseURL: "http://localhost:3000/api/v1",
  timeout: 10000,  // 10 seconds
  headers: {
    "Content-Type": "application/json",
    "Accept": "application/json",
  },
});

// Request interceptor — add auth token to every request
api.interceptors.request.use((config) => {
  const token = localStorage.getItem("authToken");
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// Response interceptor — handle 401 globally
api.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem("authToken");
      window.location.href = "/login";
    }
    return Promise.reject(error);
  }
);

// Clean API functions
export const usersApi = {
  getAll: ()         => api.get("/users").then((r) => r.data),
  getOne: (id)       => api.get(`/users/${id}`).then((r) => r.data),
  create: (data)     => api.post("/users", { user: data }).then((r) => r.data),
  update: (id, data) => api.patch(`/users/${id}`, { user: data }).then((r) => r.data),
  remove: (id)       => api.delete(`/users/${id}`),
};

export default api;

3. CORS — Cross-Origin Resource Sharing

When your React app runs on localhost:5173 (Vite) and your Rails API runs on localhost:3000, the browser considers these different origins. By default, browsers block cross-origin requests for security. CORS is the mechanism that allows your Rails server to tell the browser which origins are permitted to make requests.

ruby 
# Gemfile
gem "rack-cors"

# config/initializers/cors.rb
Rails.application.config.middleware.insert_before 0, Rack::Cors do
  allow do
    # Development: allow your React dev server
    origins "http://localhost:5173",
            "http://localhost:3001"

    resource "/api/*",
      headers: :any,
      methods: [:get, :post, :put, :patch, :delete, :options],
      expose:  ["Authorization"],
      max_age: 600  # cache preflight for 10 minutes
  end
end
Never use origins "*" in production

Setting origins "*" allows any website to make requests to your API. This is a security risk. In production, always whitelist specific domains. Use environment variables to configure allowed origins differently for development and production.

4. API Integration Patterns in Components

The key to clean API integration is separating the HTTP logic from the component rendering logic. Use custom hooks to encapsulate data fetching, and let your components focus on displaying data and handling user interactions. This pattern makes your components testable (you can mock the hooks) and your API logic reusable.

jsx 
// hooks/useUsers.js — custom hook encapsulating all user API logic
import { useState, useEffect, useCallback } from "react";
import { usersApi } from "../api/users";

export function useUsers() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  const fetchUsers = useCallback(async () => {
    try {
      setLoading(true);
      setError(null);
      const data = await usersApi.getAll();
      setUsers(data);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  }, []);

  useEffect(() => { fetchUsers(); }, [fetchUsers]);

  const createUser = async (userData) => {
    const newUser = await usersApi.create(userData);
    setUsers((prev) => [...prev, newUser]);
    return newUser;
  };

  const deleteUser = async (id) => {
    await usersApi.remove(id);
    setUsers((prev) => prev.filter((u) => u.id !== id));
  };

  return { users, loading, error, createUser, deleteUser, refetch: fetchUsers };
}

// components/UsersPage.jsx — clean component using the hook
function UsersPage() {
  const { users, loading, error, createUser, deleteUser } = useUsers();

  if (loading) return <LoadingSpinner />;
  if (error) return <ErrorMessage message={error} />;

  return (
    <div>
      <h1>Users ({users.length})</h1>
      <CreateUserForm onSubmit={createUser} />
      <UserList users={users} onDelete={deleteUser} />
    </div>
  );
}

5. Loading and Error State Patterns

Every network request can be in one of three states: loading, success, or error. Your UI must handle all three. Users should see a loading indicator while data is being fetched, the actual content when it succeeds, and a helpful error message when it fails. Ignoring any of these states leads to a poor user experience.

jsx 
// Reusable loading/error wrapper component
function AsyncContent({ loading, error, onRetry, children }) {
  if (loading) {
    return (
      <div className="loading-container">
        <div className="spinner" />
        <p>Loading...</p>
      </div>
    );
  }

  if (error) {
    return (
      <div className="error-container">
        <p className="error-message">{error}</p>
        {onRetry && (
          <button onClick={onRetry} className="retry-button">
            Try Again
          </button>
        )}
      </div>
    );
  }

  return children;
}

// Usage
function ProductsPage() {
  const { products, loading, error, refetch } = useProducts();

  return (
    <AsyncContent loading={loading} error={error} onRetry={refetch}>
      <h1>Products</h1>
      <ProductGrid products={products} />
    </AsyncContent>
  );
}

// Optimistic updates pattern
const handleToggleFavorite = async (productId) => {
  // Immediately update UI (optimistic)
  setProducts((prev) =>
    prev.map((p) => p.id === productId ? { ...p, favorited: !p.favorited } : p)
  );

  try {
    await productsApi.toggleFavorite(productId);
  } catch (err) {
    // Revert on failure
    setProducts((prev) =>
      prev.map((p) => p.id === productId ? { ...p, favorited: !p.favorited } : p)
    );
    showToast("Failed to update favorite");
  }
};

6. Authentication Tokens and Protected Requests

Most Rails APIs use JWT (JSON Web Tokens) or session-based tokens for authentication. The React client receives a token during login, stores it (typically in localStorage or an HTTP-only cookie), and includes it in the Authorization header of every subsequent request. Your API layer should handle token management transparently so individual components never need to think about authentication.

jsx 
// api/auth.js — authentication API functions
import api from "./client";

export const authApi = {
  login: async (email, password) => {
    const { data } = await api.post("/auth/login", { email, password });
    localStorage.setItem("authToken", data.token);
    return data.user;
  },

  logout: () => {
    localStorage.removeItem("authToken");
  },

  getCurrentUser: async () => {
    const { data } = await api.get("/auth/me");
    return data;
  },

  signup: async (userData) => {
    const { data } = await api.post("/auth/signup", { user: userData });
    localStorage.setItem("authToken", data.token);
    return data.user;
  },
};

localStorage vs HTTP-only cookies

Storing tokens in localStorage is simple but vulnerable to XSS attacks. For higher security, use HTTP-only cookies set by Rails (they cannot be read by JavaScript). The trade-off is more complexity in CORS and CSRF configuration. For this course, localStorage is acceptable, but production applications should evaluate their security requirements carefully.

How do I handle token refresh without logging the user out?

Implement a refresh token pattern: when the access token expires, use a longer-lived refresh token to obtain a new access token. In your axios interceptor, catch 401 responses, call the refresh endpoint, update the stored token, and retry the original request. This provides a seamless experience where the user never sees an unexpected logout.

Should I use fetch or axios in production?

Both work. fetch has no dependencies and is built into browsers. axios adds ~13KB but provides interceptors, automatic transforms, timeout support, and request cancellation. For applications with authentication, pagination, and error handling, axios interceptors save significant boilerplate. Choose based on your project's complexity.

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is "no", go back and practice more.

Can you make a GET and POST request with fetch and handle the response?

You should be able to write a fetch call with the correct method, headers, and body, check response.ok, parse the JSON, and handle errors in a try/catch block. If you forget the Content-Type header on POST requests or forget to call response.json(), practice more.

Can you configure CORS in Rails to allow your React front-end?

You should know which gem to use (rack-cors), where the initializer lives, and how to set allowed origins, methods, and headers. You should also understand why origins "*" is dangerous in production and how preflight OPTIONS requests work.

Can you set up an axios instance with interceptors for auth tokens?

You should be able to create an axios instance with axios.create, add a request interceptor that reads the token from storage and attaches it to headers, and add a response interceptor that catches 401 errors and redirects to login. This pattern is the foundation of every authenticated React application.

Do you handle loading, success, and error states for every API call?

Every component that fetches data must show a loading indicator during the request, the actual content on success, and a meaningful error message on failure (ideally with a retry button). If you have components that show a blank screen during loading or crash on errors, fix them before proceeding.

Can you extract API logic into a custom hook that multiple components can reuse?

Given a component with inline fetch calls, useState, and useEffect, you should be able to extract all the data-fetching logic into a useXxx hook that returns { data, loading, error }. The component should become a pure rendering layer that calls the hook and displays the results.
