Chapter 01 · Phase 1: Python Foundations

Python Core Fundamentals

Master Python's core building blocks — variables, data types, control flow, functions, and modules. These fundamentals are the bedrock of every FastAPI application you will build throughout this program.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Python is the language behind FastAPI, machine learning pipelines, DevOps automation, and millions of production web services. Before you can build robust APIs, you need to think in Python — understanding how the language handles data, controls execution flow, and organizes code into reusable units.

Every bug you encounter in a FastAPI application traces back to a Python fundamental: a mutable default argument that silently corrupts state, a variable scope issue that returns None instead of data, or a missing __init__.py that breaks your import chain. Mastering these fundamentals now prevents weeks of debugging later.

Key Insight

Python's philosophy is captured in the Zen of Python: "Explicit is better than implicit" and "There should be one obvious way to do it." Understanding these principles is not academic — they directly inform how FastAPI was designed and how you should structure your code.

Key Concepts

1. Variables and Data Types

Python is dynamically typed — you do not declare types when creating variables. However, Python is also strongly typed, meaning it will not implicitly convert between incompatible types. Understanding Python's type system is critical because FastAPI leverages type hints extensively for request validation and documentation generation.

python 
# ── Fundamental Data Types ──

# Integers and floats
age = 28
price = 19.99
big_number = 1_000_000          # underscores for readability

# Strings — immutable sequences of Unicode
name = "Alice"
greeting = f"Hello, {name}!"    # f-strings (Python 3.6+)
multiline = """
This is a
multiline string.
"""

# Booleans
is_active = True
is_deleted = False

# None — Python's null equivalent
result = None

# Type checking at runtime
print(type(age))        # <class 'int'>
print(isinstance(name, str))  # True

Python's collection types are the workhorses of data manipulation. You will use them constantly when processing API request bodies, database query results, and configuration data:

python 
# ── Collection Types ──

# Lists — ordered, mutable, allows duplicates
users = ["alice", "bob", "charlie"]
users.append("diana")
first_user = users[0]           # "alice"
last_two = users[-2:]            # ["charlie", "diana"]

# Dictionaries — key-value pairs, ordered (Python 3.7+)
user = {
    "name": "Alice",
    "email": "alice@example.com",
    "age": 28,
    "roles": ["admin", "editor"],
}

# Safe access with .get() — returns None instead of KeyError
phone = user.get("phone")         # None
phone = user.get("phone", "N/A") # "N/A"

# Tuples — ordered, immutable
coordinates = (40.7128, -74.0060)
lat, lng = coordinates           # tuple unpacking

# Sets — unordered, unique elements
tags = {"python", "fastapi", "async"}
tags.add("python")               # no duplicate — still 3 items
Mutable Default Arguments Trap

Never use a mutable object (list, dict, set) as a default function argument. Python evaluates defaults once at function definition, not at each call. This causes shared state between calls — a notorious source of bugs. Use None as the default and create the mutable object inside the function body.

2. Control Flow

Python uses indentation (not braces) to define code blocks. This enforced readability is a deliberate design choice. Control flow in Python is straightforward but has nuances that affect how you write API endpoint handlers, validation logic, and error handling.

python 
# ── Conditionals ──
def classify_status_code(code: int) -> str:
    """Classify an HTTP status code into its category."""
    if 200 <= code < 300:
        return "success"
    elif 300 <= code < 400:
        return "redirect"
    elif 400 <= code < 500:
        return "client_error"
    elif 500 <= code < 600:
        return "server_error"
    else:
        return "unknown"

# ── Match Statement (Python 3.10+) ──
def handle_command(command: str):
    match command.split():
        case ["get", resource]:
            return f"Fetching {resource}"
        case ["delete", resource, id]:
            return f"Deleting {resource} #{id}"
        case _:
            return "Unknown command"

# ── Loops with useful patterns ──
items = ["apple", "banana", "cherry"]

# enumerate gives you index + value
for i, item in enumerate(items):
    print(f"{i}: {item}")

# zip pairs up multiple iterables
names = ["Alice", "Bob"]
scores = [95, 87]
for name, score in zip(names, scores):
    print(f"{name}: {score}")

# List comprehensions — the Pythonic way
squares = [x ** 2 for x in range(10)]
even_squares = [x ** 2 for x in range(10) if x % 2 == 0]

# Dict comprehension
user_map = {user["id"]: user for user in users_list}

3. Functions and Closures

Functions in Python are first-class objects — they can be assigned to variables, passed as arguments, and returned from other functions. This is foundational to FastAPI's dependency injection system and middleware patterns.

python 
# ── Function signatures with type hints ──
def create_user(
    name: str,
    email: str,
    age: int = 0,
    *,                          # everything after * is keyword-only
    is_admin: bool = False,
) -> dict:
    """Create a new user dictionary.

    Args:
        name: The user's display name.
        email: The user's email address.
        age: The user's age (default 0).
        is_admin: Whether the user has admin privileges.

    Returns:
        A dictionary representing the new user.
    """
    return {
        "name": name,
        "email": email,
        "age": age,
        "is_admin": is_admin,
    }

# ── *args and **kwargs ──
def log_event(event: str, *tags: str, **metadata: str):
    """Log an event with optional tags and metadata."""
    print(f"Event: {event}")
    print(f"Tags: {tags}")
    print(f"Metadata: {metadata}")

log_event("user_signup", "auth", "growth", source="web", plan="free")

# ── Lambda functions ──
sort_by_age = lambda user: user["age"]
sorted_users = sorted(users_list, key=sort_by_age)

# ── Closures — functions that capture outer state ──
def make_multiplier(factor: int):
    def multiply(n: int) -> int:
        return n * factor
    return multiply

double = make_multiplier(2)
print(double(5))   # 10
Pro Tip

Use keyword-only arguments (after the * separator) for boolean flags. Calling create_user("Alice", "a@b.com", 28, True) is unclear — what is True? Forcing is_admin=True makes the call self-documenting. FastAPI uses this pattern extensively.

4. Error Handling

Robust error handling is the difference between an API that returns helpful error messages and one that crashes with a 500 Internal Server Error. Python's exception hierarchy is rich and extensible.

python 
# ── Try/Except/Else/Finally ──
def safe_divide(a: float, b: float) -> float | None:
    try:
        result = a / b
    except ZeroDivisionError:
        print("Cannot divide by zero")
        return None
    except TypeError as e:
        print(f"Type error: {e}")
        return None
    else:
        # Runs only if no exception occurred
        print(f"Result: {result}")
        return result
    finally:
        # Always runs — cleanup code goes here
        print("Division attempted")

# ── Custom Exceptions ──
class UserNotFoundError(Exception):
    """Raised when a user lookup fails."""
    def __init__(self, user_id: int):
        self.user_id = user_id
        super().__init__(f"User {user_id} not found")

class ValidationError(Exception):
    """Raised when input validation fails."""
    def __init__(self, field: str, message: str):
        self.field = field
        self.message = message
        super().__init__(f"{field}: {message}")

# Using custom exceptions
def get_user(user_id: int) -> dict:
    user = database.find(user_id)
    if user is None:
        raise UserNotFoundError(user_id)
    return user

5. Modules, Packages, and pip

Python organizes code into modules (single files) and packages (directories with an __init__.py). Understanding this structure is essential because every FastAPI project is organized as a Python package with routers, models, and services split across modules.

python 
# ── Project Structure ──
# my_project/
#   __init__.py
#   main.py
#   models/
#     __init__.py
#     user.py
#     product.py
#   services/
#     __init__.py
#     auth.py

# ── Import Patterns ──
import json                           # import entire module
from pathlib import Path             # import specific item
from datetime import datetime, timedelta
from typing import Optional, List

# Relative imports (within a package)
from .models.user import User
from ..config import settings

# ── The __name__ guard ──
if __name__ == "__main__":
    # Only runs when this file is executed directly
    # Not when imported as a module
    main()
bash 
# ── pip and Virtual Environments ──

# Create a virtual environment
python -m venv .venv
source .venv/bin/activate      # Linux/macOS

# Install packages
pip install fastapi uvicorn sqlalchemy

# Freeze dependencies to a requirements file
pip freeze > requirements.txt

# Install from requirements file
pip install -r requirements.txt

# Modern alternative: use pyproject.toml with pip
pip install -e ".[dev]"
Never install globally

Always use a virtual environment. Installing packages globally leads to version conflicts between projects. If project A needs SQLAlchemy 1.4 and project B needs SQLAlchemy 2.0, global installs will break one of them. Virtual environments isolate each project's dependencies completely.

6. Decorators and Generators

Decorators modify or extend functions without changing their source code. FastAPI routes are defined using decorators (@app.get, @app.post). Generators produce values lazily, which is essential for memory-efficient data processing.

python 
# ── Decorators ──
import time
from functools import wraps

def timer(func):
    """Measure execution time of a function."""
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper

@timer
def slow_operation():
    time.sleep(1)
    return "done"

# ── Generators ──
def read_large_file(path: str):
    """Read a file line-by-line without loading it all into memory."""
    with open(path) as f:
        for line in f:
            yield line.strip()

# Generator expressions — like list comprehensions but lazy
total = sum(x ** 2 for x in range(1_000_000))
# Uses constant memory, not a 1M-element list
How decorators work under the hood

A decorator is syntactic sugar for wrapping a function. When you write @timer above a function definition, Python executes slow_operation = timer(slow_operation). The @wraps(func) decorator from functools preserves the original function's name and docstring, which matters for debugging and for FastAPI's automatic documentation generation.

When to use generators vs. lists

Use generators when you are processing data sequentially and do not need random access. If you are iterating over database rows, reading log files, or streaming API responses, generators keep memory usage constant regardless of data size. Use lists when you need to access elements by index, sort them, or iterate multiple times.

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer is "no", go back and review.

Can you explain the difference between mutable and immutable types?

Lists, dictionaries, and sets are mutable — they can be changed in place. Strings, tuples, integers, and frozensets are immutable — any "modification" creates a new object. This matters for function arguments (mutable default arguments are shared), dictionary keys (must be immutable), and understanding when Python copies data vs. creates a reference.

Can you write a decorator that accepts arguments?

A parameterized decorator is a function that returns a decorator. The pattern is three nested functions: the outer function takes the decorator's arguments, the middle function takes the wrapped function, and the inner function replaces the wrapped function. If you can write @retry(max_attempts=3), you understand closures, higher-order functions, and decorators.

Can you explain when and why to use virtual environments?

Virtual environments isolate project dependencies. Without them, installing a package for one project could break another. You should create a virtual environment for every project, activate it before working, and never install project dependencies globally. The venv module is built into Python 3.3+ and is the recommended approach.
