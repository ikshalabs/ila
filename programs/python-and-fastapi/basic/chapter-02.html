Chapter 02 · Phase 1: Python Foundations

Object-Oriented Python

Classes, inheritance, magic methods, dataclasses, and protocols. OOP in Python is the backbone of every Pydantic model, every SQLAlchemy entity, and every FastAPI dependency you will write.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Object-oriented programming is not just a paradigm — it is the structural foundation of the entire Python web ecosystem. Pydantic models are classes. SQLAlchemy entities are classes. FastAPI's dependency injection system revolves around callable objects and class instances. If you cannot fluently define classes, compose them through inheritance, and customize their behavior with special methods, you will hit a wall the moment you try to build a real API.

Modern Python OOP has evolved far beyond the traditional "class with __init__" pattern. Dataclasses eliminate boilerplate. Protocols enable structural subtyping (duck typing with type safety). Abstract base classes define contracts. Understanding when to use each approach is what separates maintainable codebases from fragile ones.

Key Insight

Python follows the principle of "we're all consenting adults". There are no true private members. A leading underscore is a convention, not enforcement. This philosophy prioritizes flexibility and trust over rigid access control, which makes Python OOP more pragmatic than Java or C++.

Key Concepts

1. Classes and Instances

A class is a blueprint for creating objects. Each instance has its own state. Understanding the relationship between class attributes and instance attributes is critical for avoiding shared-state bugs in web applications.

python 
class User:
    """Represents a user in the system."""

    # Class attribute — shared by all instances
    default_role = "viewer"

    def __init__(self, name: str, email: str, role: str | None = None):
        # Instance attributes — unique to each object
        self.name = name
        self.email = email
        self.role = role or self.default_role
        self._login_count = 0  # convention: "private"

    def login(self) -> None:
        """Record a login event."""
        self._login_count += 1

    @property
    def login_count(self) -> int:
        """Read-only access to login count."""
        return self._login_count

    @classmethod
    def from_dict(cls, data: dict) -> "User":
        """Alternate constructor from a dictionary."""
        return cls(name=data["name"], email=data["email"], role=data.get("role"))

    @staticmethod
    def validate_email(email: str) -> bool:
        """Check if an email address looks valid."""
        return "@" in email and "." in email

# Usage
user = User("Alice", "alice@example.com", "admin")
user.login()
print(user.login_count)  # 1 (via property)

user2 = User.from_dict({"name": "Bob", "email": "bob@example.com"})

2. Inheritance and Composition

Inheritance creates "is-a" relationships. Composition creates "has-a" relationships. Python supports multiple inheritance through the Method Resolution Order (MRO), but composition is usually preferred for flexibility and testability.

python 
from abc import ABC, abstractmethod

# ── Abstract Base Class ──
class Repository(ABC):
    """Interface for data storage."""

    @abstractmethod
    def save(self, entity: dict) -> str: ...

    @abstractmethod
    def find_by_id(self, entity_id: str) -> dict | None: ...

# ── Concrete Implementation ──
class InMemoryRepository(Repository):
    def __init__(self):
        self._store: dict[str, dict] = {}

    def save(self, entity: dict) -> str:
        import uuid
        entity_id = str(uuid.uuid4())
        self._store[entity_id] = {**entity, "id": entity_id}
        return entity_id

    def find_by_id(self, entity_id: str) -> dict | None:
        return self._store.get(entity_id)

# ── Composition over Inheritance ──
class UserService:
    """Business logic for users. Uses Repository via composition."""

    def __init__(self, repo: Repository):
        self._repo = repo  # dependency injection

    def create_user(self, name: str, email: str) -> str:
        if not User.validate_email(email):
            raise ValueError(f"Invalid email: {email}")
        return self._repo.save({"name": name, "email": email})
Favor composition over deep inheritance

Deep inheritance hierarchies (more than 2-3 levels) create fragile code where changes to a base class break distant subclasses. Composition through dependency injection is more flexible and testable — you can swap implementations without modifying consuming code.

3. Magic Methods (Dunder Methods)

Magic methods let you define how your objects behave with built-in Python operations. They are the mechanism behind len(), str(), comparison operators, and iteration.

python 
class Money:
    """Value object representing a monetary amount."""

    def __init__(self, amount: float, currency: str = "USD"):
        self.amount = round(amount, 2)
        self.currency = currency

    def __repr__(self) -> str:
        return f"Money({self.amount}, '{self.currency}')"

    def __str__(self) -> str:
        return f"${self.amount:.2f} {self.currency}"

    def __eq__(self, other: object) -> bool:
        if not isinstance(other, Money):
            return NotImplemented
        return self.amount == other.amount and self.currency == other.currency

    def __add__(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("Cannot add different currencies")
        return Money(self.amount + other.amount, self.currency)

    def __hash__(self) -> int:
        return hash((self.amount, self.currency))

# Now Money works naturally with Python
total = Money(19.99) + Money(1.60)  # Money(21.59, 'USD')
print(total)                            # $21.59 USD

4. Dataclasses

Dataclasses (Python 3.7+) automatically generate __init__, __repr__, __eq__, and optionally __hash__ and ordering methods. They are the stepping stone to Pydantic models.

python 
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class Task:
    title: str
    description: str
    priority: int = 1
    completed: bool = False
    tags: list[str] = field(default_factory=list)
    created_at: datetime = field(default_factory=datetime.now)

# Auto-generated __init__, __repr__, __eq__
task = Task("Write tests", "Add unit tests for auth module")

# ── Frozen dataclasses — immutable value objects ──
@dataclass(frozen=True)
class Coordinate:
    lat: float
    lng: float

# ── Post-init processing ──
@dataclass
class Product:
    name: str
    price: float
    quantity: int
    total: float = field(init=False)

    def __post_init__(self):
        self.total = self.price * self.quantity
        if self.price < 0:
            raise ValueError("Price cannot be negative")

5. Protocols and Structural Subtyping

Protocols (Python 3.8+) define structural subtyping — "static duck typing." Instead of requiring explicit inheritance, any class that implements the required methods satisfies the protocol.

python 
from typing import Protocol, runtime_checkable

@runtime_checkable
class Serializable(Protocol):
    def to_dict(self) -> dict: ...

class UserModel:
    def __init__(self, name: str):
        self.name = name

    def to_dict(self) -> dict:
        return {"name": self.name}

# No explicit inheritance needed!
def serialize(obj: Serializable) -> dict:
    return obj.to_dict()

serialize(UserModel("Alice"))  # Works — UserModel has to_dict()
print(isinstance(UserModel("Alice"), Serializable))  # True
Pro Tip

Prefer Protocols over ABC when you want type-checking without forced inheritance. This aligns with Python's duck typing philosophy. Use ABC when you need to enforce that subclasses implement specific methods at class creation time (not at usage time).

6. Slots and Descriptors

__slots__ restricts which attributes an instance can have, reducing memory usage significantly for classes with many instances. Descriptors control attribute access and are the mechanism behind property, classmethod, and staticmethod.

python 
# ── __slots__ for memory-efficient classes ──
class Point:
    __slots__ = ("x", "y")

    def __init__(self, x: float, y: float):
        self.x = x
        self.y = y
# Uses ~40% less memory per instance

# ── Custom Descriptor ──
class Validated:
    def __init__(self, min_val: float, max_val: float):
        self.min_val = min_val
        self.max_val = max_val

    def __set_name__(self, owner, name):
        self.name = name
        self.private = f"_{name}"

    def __get__(self, obj, objtype=None):
        return getattr(obj, self.private, None)

    def __set__(self, obj, value):
        if not self.min_val <= value <= self.max_val:
            raise ValueError(f"{self.name} must be {self.min_val}-{self.max_val}")
        setattr(obj, self.private, value)

class Temperature:
    celsius = Validated(-273.15, 1000)

    def __init__(self, celsius: float):
        self.celsius = celsius  # triggers descriptor __set__
When to use __slots__ vs. regular classes

Use __slots__ when you have many instances (thousands+) with a fixed set of attributes. Common examples: ORM row objects, data points, coordinates. Avoid when you need dynamic attributes or complex inheritance hierarchies.

How Pydantic uses descriptors internally

Pydantic v2 uses descriptors and __slots__ internally for model fields. When you define a Pydantic field with a validator, it creates a descriptor that intercepts attribute assignment and runs validation. Understanding this helps you debug validation errors and write custom field types.

Self-Check

Before moving to Chapter 03, answer these honestly.

Can you explain the difference between a Protocol and an ABC?

ABCs require explicit inheritance and check at class creation. Protocols use structural subtyping — any class with the right methods qualifies, no inheritance needed. Use ABCs for explicit opt-in; Protocols for flexibility and duck typing with type safety.

Can you implement __eq__ and __hash__ correctly?

If you override __eq__, Python sets __hash__ to None. You must define both for dict keys or set members. Hash must use the same attributes as equality, and those attributes should be immutable.

Do you know when to use dataclass vs. regular class vs. Pydantic model?

Use dataclasses for internal data structures without validation. Use regular classes for complex behavior or full control. Use Pydantic models (Chapter 06) for input validation, serialization, and settings management.
