Chapter 07 · Phase 4: Production Backend

Database Integration

SQLAlchemy, Alembic migrations, async database access, and the repository pattern. Connect your FastAPI application to a real database with proper schema management.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

A web API without a database is a toy. Real applications need persistent storage, schema migrations, transaction management, and efficient query patterns. SQLAlchemy is the most mature and powerful ORM in the Python ecosystem — it powers Instagram, Uber, Dropbox, and thousands of production applications.

In this chapter, you will connect FastAPI to PostgreSQL using SQLAlchemy 2.0 (the modern, type-safe API), manage schema changes with Alembic migrations, and implement the repository pattern for clean separation between your API layer and your data layer.

Key Insight

The repository pattern is not academic overengineering. It is the practical boundary between your business logic and your database. When you need to switch from PostgreSQL to MongoDB, add caching, or mock the database in tests, the repository pattern makes it a one-file change instead of a rewrite.

Key Concepts

1. SQLAlchemy 2.0 Models

SQLAlchemy 2.0 uses a declarative approach with type annotations that integrates naturally with Python's type system and Pydantic models.

python 
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, relationship
from sqlalchemy import String, ForeignKey
from datetime import datetime

class Base(DeclarativeBase):
    pass

class User(Base):
    __tablename__ = "users"

    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(String(100))
    email: Mapped[str] = mapped_column(String(255), unique=True)
    created_at: Mapped[datetime] = mapped_column(default=datetime.utcnow)
    posts: Mapped[list["Post"]] = relationship(back_populates="author")

2. Database Migrations with Alembic

Alembic tracks schema changes and applies them incrementally. Never manually modify database schemas in production — always use migrations.

bash 
# Initialize Alembic
alembic init alembic

# Generate a migration from model changes
alembic revision --autogenerate -m "add users table"

# Apply migrations
alembic upgrade head

# Rollback one step
alembic downgrade -1

3. Async Database Sessions

FastAPI is async-first. Using async database sessions prevents blocking the event loop during database queries, keeping your API responsive under load.

python 
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker

engine = create_async_engine(
    "postgresql+asyncpg://user:pass@localhost/db",
    echo=True,
)

async_session = sessionmaker(
    engine, class_=AsyncSession, expire_on_commit=False
)

async def get_db():
    async with async_session() as session:
        yield session

4. The Repository Pattern

The repository pattern abstracts database operations behind a clean interface, making your business logic independent of the storage technology.

python 
from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

class UserRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def create(self, name: str, email: str) -> User:
        user = User(name=name, email=email)
        self.session.add(user)
        await self.session.commit()
        await self.session.refresh(user)
        return user

    async def get_by_id(self, user_id: int) -> User | None:
        result = await self.session.execute(
            select(User).where(User.id == user_id)
        )
        return result.scalar_one_or_none()

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you explain the difference between sync and async SQLAlchemy sessions?

Sync sessions block the event loop during database queries. Async sessions (via asyncpg or aiosqlite) yield control back to the event loop while waiting for the database, allowing other requests to be processed concurrently. Always use async sessions in FastAPI.

Do you understand how Alembic autogenerate works?

Alembic compares your SQLAlchemy models to the current database schema and generates migration code for the differences. It detects new tables, new columns, type changes, and index changes. Always review autogenerated migrations before applying them — they can miss some changes.

Can you implement the repository pattern from memory?

A repository wraps database operations behind a clean interface: create, get_by_id, list, update, delete. It takes a database session in its constructor and uses it for all queries. This pattern lets you swap databases or add caching without changing business logic.
