Production Deployment

CI/CD pipelines, blue-green deployments, feature flags, infrastructure as code, and production runbooks. Ship with confidence.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

The final chapter brings everything together: CI/CD pipelines that automate testing and deployment, blue-green deploys for zero-downtime releases, feature flags for safe rollouts, and infrastructure as code for reproducible environments. This is the operational foundation that turns your application from a project into a product.

Production readiness is not about writing more code — it is about building the systems and processes that let you ship changes safely and respond to incidents quickly. A well-operated simple system beats a poorly-operated complex one every time.

Key Insight

Production readiness is a spectrum, not a binary. Start with CI/CD (automated testing and deployment). Add monitoring (know when things break). Add runbooks (know how to fix them). Add chaos engineering (know what breaks before it breaks). Each level increases your confidence and reduces your incident response time.

Key Concepts

1. GitHub Actions CI/CD Pipeline

A CI/CD pipeline automates testing, building, and deploying your application. Every commit triggers the pipeline, catching bugs before they reach production.

yaml 
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt
      - run: pytest --cov=app --cov-fail-under=80

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: docker build -t myapp .
      - run: docker push myregistry/myapp:$GITHUB_SHA

2. Blue-Green Deployment Strategy

Blue-green deployments run two identical environments. Traffic switches from blue (current) to green (new) instantly, with instant rollback capability.

bash 
# Blue-Green with Kubernetes
# 1. Deploy new version as "green"
kubectl apply -f deployment-green.yaml

# 2. Verify green is healthy
kubectl rollout status deployment/fastapi-green

# 3. Switch service to green
kubectl patch service fastapi-svc -p \
  '{"spec":{"selector":{"version":"green"}}}'

# 4. If problems, switch back to blue
kubectl patch service fastapi-svc -p \
  '{"spec":{"selector":{"version":"blue"}}}'

3. Feature Flags for Safe Rollouts

Feature flags decouple deployment from release. Deploy code to production behind a flag, then gradually enable it for users.

python 
from pydantic import BaseModel

class FeatureFlags(BaseModel):
    new_search: bool = False
    v2_api: bool = False
    dark_mode: bool = True

flags = FeatureFlags()

@app.get("/search")
async def search(q: str):
    if flags.new_search:
        return await new_search_engine(q)
    return await legacy_search(q)

4. Infrastructure as Code with Terraform

Terraform defines cloud infrastructure (servers, databases, networking) as code. Changes are reviewed, versioned, and applied predictably.

hcl 
# main.tf
resource "aws_ecs_service" "api" {
  name            = "fastapi-service"
  cluster         = aws_ecs_cluster.main.id
  task_definition = aws_ecs_task_definition.api.arn
  desired_count   = 3

  load_balancer {
    target_group_arn = aws_lb_target_group.api.arn
    container_name   = "api"
    container_port   = 8000
  }
}

CI/CD Pipeline

Create a complete CI/CD pipeline with GitHub Actions.

  • Deploy to staging automatically
  • Add manual approval gate for production deployment

Blue-Green Deployment System

Implement blue-green deployments with instant rollback capability.

  • Create blue and green deployment configurations
  • Automate traffic switching with a single command

Complete Production Platform

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you implement zero-downtime deployments?

Blue-green: run old and new side by side, switch traffic atomically. Rolling update: gradually replace old pods with new ones. Canary: route a small percentage of traffic to the new version first. All require health checks to verify the new version works before committing.

Do you understand feature flags?

Feature flags decouple deployment from release. Code is deployed to production but disabled via a flag. The flag can be toggled per-user, per-percentage, or per-environment. This enables gradual rollouts, A/B testing, and instant rollback without redeployment.

Can you define infrastructure as code?

IaC means all infrastructure (servers, databases, networking, DNS) is defined in version-controlled configuration files (Terraform, CloudFormation). Changes go through code review. Environments are reproducible. Disasters can be recovered by re-running the IaC. No manual clicking in cloud consoles.
