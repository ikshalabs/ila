Chapter 03 · Phase 2: Real-Time Systems

WebSockets and Real-Time

WebSocket endpoints, connection management, broadcasting, rooms, and Server-Sent Events. Build real-time features that scale.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Modern applications need real-time features: live notifications, chat, collaborative editing, dashboards that update without refresh. WebSockets and Server-Sent Events (SSE) provide the primitives for building these features, and FastAPI supports both natively through Starlette.

The challenge is not sending a single message — it is managing connections at scale: handling disconnects, broadcasting to rooms, surviving server restarts, and scaling across multiple instances. This chapter covers the patterns that make real-time features production-ready.

Key Insight

Choose the right tool: WebSockets for bidirectional communication (chat, gaming). SSE for server-to-client streaming (notifications, live feeds). HTTP polling for infrequent updates where simplicity matters more than latency.

Key Concepts

1. WebSocket Endpoints in FastAPI

WebSockets provide full-duplex communication between client and server. Unlike HTTP, the connection stays open for bidirectional message passing.

python 
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

@app.websocket("/ws")
async def websocket_endpoint(websocket: WebSocket):
    await websocket.accept()
    try:
        while True:
            data = await websocket.receive_text()
            await websocket.send_text(f"Echo: {data}")
    except WebSocketDisconnect:
        print("Client disconnected")

2. Connection Manager Pattern

A connection manager tracks active WebSocket connections and provides broadcasting, room management, and graceful disconnect handling.

python 
class ConnectionManager:
    def __init__(self):
        self.active: dict[str, list[WebSocket]] = defaultdict(list)

    async def connect(self, websocket: WebSocket, room: str):
        await websocket.accept()
        self.active[room].append(websocket)

    async def broadcast(self, room: str, message: str):
        for ws in self.active[room]:
            await ws.send_text(message)

3. Server-Sent Events (SSE)

SSE provides one-way streaming from server to client over HTTP. Simpler than WebSockets for notifications, live feeds, and progress updates.

python 
from fastapi.responses import StreamingResponse
import asyncio

async def event_stream():
    while True:
        data = await get_latest_event()
        yield f"data: {json.dumps(data)}\n\n"
        await asyncio.sleep(1)

@app.get("/events")
async def get_events():
    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream",
    )

4. Scaling WebSockets with Redis Pub/Sub

In-memory connection managers only work on a single server. Redis Pub/Sub enables broadcasting across multiple server instances.

python 
import redis.asyncio as redis

class RedisConnectionManager:
    def __init__(self):
        self.redis = redis.from_url("redis://localhost")
        self.local_connections: dict[str, list[WebSocket]] = {}

    async def publish(self, room: str, message: str):
        await self.redis.publish(room, message)

    async def listen(self, room: str):
        pubsub = self.redis.pubsub()
        await pubsub.subscribe(room)
        async for message in pubsub.listen():
            if message["type"] == "message":
                yield message["data"]

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you explain the difference between WebSockets and SSE?

WebSockets provide full-duplex (bidirectional) communication. SSE is server-to-client only over HTTP. Use WebSockets for chat, gaming, collaborative editing. Use SSE for notifications, live feeds, and dashboards where the client only receives data.

Do you understand connection lifecycle management?

WebSocket connections must be accepted, monitored for disconnects, and cleaned up. Failing to remove disconnected clients from your manager causes memory leaks and broadcast failures. Always use try/except/finally for cleanup.

Can you scale WebSockets across multiple servers?

In-memory connection managers only work on a single server. For multiple servers, use Redis Pub/Sub or a message queue. Each server subscribes to relevant channels and forwards messages to its local connections.
