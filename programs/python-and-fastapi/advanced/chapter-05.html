Microservice Patterns

Service discovery, API gateways, circuit breakers, saga pattern, and inter-service communication. Design systems that scale independently.

Why This Matters

Microservices are not a silver bullet — they are a trade-off. You exchange the simplicity of a monolith for the ability to scale, deploy, and evolve services independently. This chapter covers the patterns that make microservices work in practice: service communication, failure handling, and distributed transactions.

The patterns in this chapter — circuit breakers, sagas, API gateways — are not theoretical. They are battle-tested solutions to the real problems that emerge when you split a system into independent services that communicate over a network.

Key Insight

The hardest part of microservices is not building them — it is deciding where to split. Each service should own a single business domain with a clear API boundary. If two services need to update each other constantly, they should probably be one service.

Key Concepts

1. Designing Service Boundaries

The hardest part of microservices is deciding where to draw the boundaries. Each service should own a single business domain and communicate through well-defined APIs.

python 
# Service structure example
# user-service/   — owns user accounts, auth
# order-service/  — owns orders, payments
# product-service/ — owns catalog, inventory

# Each service has its own database
# Communication via HTTP or message queues

import httpx

class UserServiceClient:
    def __init__(self, base_url: str):
        self.client = httpx.AsyncClient(base_url=base_url)

    async def get_user(self, user_id: int):
        response = await self.client.get(f"/users/{user_id}")
        response.raise_for_status()
        return response.json()

2. Circuit Breaker Pattern

When a downstream service is failing, a circuit breaker prevents your service from wasting resources on requests that will fail. It 'opens' after N failures and 'closes' after a recovery period.

python 
import time

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30):
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.last_failure_time = 0
        self.state = "closed"  # closed, open, half-open

    async def call(self, func, *args):
        if self.state == "open":
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = "half-open"
            else:
                raise CircuitOpenError("Circuit is open")
        try:
            result = await func(*args)
            self._on_success()
            return result
        except Exception:
            self._on_failure()
            raise

3. Distributed Transactions with Sagas

In microservices, you cannot use database transactions across services. The Saga pattern coordinates multi-service operations through a sequence of local transactions with compensating actions.

python 
class OrderSaga:
    """Orchestrates order creation across services."""

    async def execute(self, order_data: dict):
        try:
            # Step 1: Reserve inventory
            reservation = await inventory_service.reserve(order_data)
            # Step 2: Process payment
            payment = await payment_service.charge(order_data)
            # Step 3: Create order
            order = await order_service.create(order_data)
            return order
        except PaymentError:
            # Compensate: release inventory
            await inventory_service.release(reservation)
            raise

4. API Gateway Pattern

An API gateway provides a single entry point for clients, handling routing, authentication, rate limiting, and request aggregation across multiple backend services.

python 
# Simple API gateway with FastAPI
from fastapi import FastAPI
import httpx

gateway = FastAPI()
SERVICES = {
    "users": "http://user-service:8001",
    "orders": "http://order-service:8002",
    "products": "http://product-service:8003",
}

@gateway.api_route("/{service}/{path:path}", methods=["GET","POST","PUT","DELETE"])
async def proxy(service: str, path: str, request: Request):
    if service not in SERVICES:
        raise HTTPException(404)
    url = f"{SERVICES[service]}/{path}"
    async with httpx.AsyncClient() as client:
        response = await client.request(
            request.method, url, content=await request.body()
        )
    return Response(content=response.content, status_code=response.status_code)

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you explain when to use microservices vs monolith?

Start with a monolith unless you have a proven need for independent scaling or deployment of specific components. Microservices add complexity (networking, distributed transactions, monitoring). They pay off when teams need to deploy independently or services have very different scaling requirements.

Do you understand the circuit breaker pattern?

A circuit breaker tracks failures to a downstream service. After N failures (threshold), it "opens" and immediately rejects calls for a recovery period. After the timeout, it lets one request through ("half-open"). If it succeeds, the circuit closes. This prevents cascade failures.

Can you implement a saga with compensation?

A saga is a sequence of local transactions with compensating actions. If step 3 fails, you run compensating actions for steps 2 and 1 in reverse order. The orchestrator manages state and ensures either all steps succeed or all are compensated.
