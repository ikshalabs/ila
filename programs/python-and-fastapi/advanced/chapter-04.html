Background Tasks and Queues

Celery, Redis queues, task scheduling, retry strategies, and monitoring. Offload heavy work from your API to background workers.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Not everything can happen inside a request/response cycle. Sending emails, processing images, generating reports, and syncing with external services should happen in the background so your API responds quickly. FastAPI provides built-in background tasks for simple cases, but production systems need a proper task queue.

Celery with Redis is the industry standard for Python task queues. It handles distributed processing, retry logic, scheduling, monitoring, and dead letter queues — all the things you need when background work is business-critical.

Key Insight

The key insight is separation of concerns. Your API should accept the request, validate it, enqueue the work, and respond immediately. The background worker processes the work asynchronously. This keeps your API responsive and your background processing reliable.

Key Concepts

1. Built-in Background Tasks

FastAPI's BackgroundTasks runs functions after the response is sent. Good for simple tasks like sending emails or logging.

python 
from fastapi import BackgroundTasks

async def send_email(to: str, subject: str):
    # This runs after the response is sent
    await email_client.send(to=to, subject=subject)

@app.post("/users")
async def create_user(user: UserCreate, bg: BackgroundTasks):
    new_user = await repo.create(user)
    bg.add_task(send_email, user.email, "Welcome!")
    return new_user

2. Celery for Distributed Task Processing

Celery is the industry standard for distributed task queues. It runs tasks on separate worker processes, with Redis or RabbitMQ as the message broker.

python 
from celery import Celery

celery_app = Celery(
    "tasks",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1",
)

@celery_app.task(bind=True, max_retries=3)
def process_image(self, image_id: int):
    try:
        image = download_image(image_id)
        resize_image(image)
        upload_result(image)
    except Exception as exc:
        self.retry(exc=exc, countdown=60)

3. Retry Strategies and Dead Letter Queues

Not all failures are permanent. Retry strategies with exponential backoff handle transient errors gracefully while dead letter queues capture permanently failed tasks.

python 
@celery_app.task(
    bind=True,
    max_retries=5,
    autoretry_for=(ConnectionError, TimeoutError),
    retry_backoff=True,       # exponential backoff
    retry_backoff_max=600,    # max 10 minutes
    retry_jitter=True,       # add randomness
)
def sync_external_api(self, endpoint: str):
    response = requests.get(endpoint, timeout=30)
    response.raise_for_status()
    return response.json()

4. Monitoring with Flower and Prometheus

Production task queues need monitoring. Flower provides a real-time web UI for Celery, while Prometheus metrics enable alerting.

bash 
# Start Flower monitoring UI
celery -A tasks flower --port=5555

# Prometheus metrics exporter
pip install celery-prometheus-exporter
celery-prometheus-exporter --broker-url redis://localhost:6379/0

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

When should you use BackgroundTasks vs Celery?

Use BackgroundTasks for lightweight, fire-and-forget tasks that run in the same process (sending an email, logging). Use Celery for heavy tasks that need retries, scheduling, monitoring, or should survive server restarts.

Can you implement retry with exponential backoff?

Set retry_backoff=True in Celery task decorator. The delay doubles each retry (1s, 2s, 4s, 8s...) with retry_backoff_max as ceiling and retry_jitter=True to prevent thundering herd.

Do you understand dead letter queues?

When a task exhausts all retries, it goes to the dead letter queue instead of being lost. This lets you inspect, debug, and replay permanently failed tasks. Essential for any production task queue.
