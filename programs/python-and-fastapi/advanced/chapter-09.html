Chapter 09 · Phase 5: Cloud Deployment

Container Orchestration

Docker multi-stage builds, Kubernetes manifests, health probes, horizontal scaling, and Helm charts. Package and orchestrate your application for the cloud.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Containers are the standard unit of deployment for modern applications. Docker packages your application with all its dependencies into an immutable image. Kubernetes orchestrates those containers at scale: deploying, scaling, healing, and updating them automatically.

This chapter covers the practical skills you need to containerize a FastAPI application and deploy it to Kubernetes: multi-stage builds for small images, health probes for reliability, HPA for auto-scaling, and Helm for packaging.

Key Insight

Containers solve "works on my machine." The same image runs identically on your laptop, in CI, and in production. Kubernetes solves "what happens when the machine dies." It automatically reschedules containers to healthy nodes.

Key Concepts

1. Multi-Stage Docker Builds

Multi-stage builds produce minimal production images by separating build dependencies from runtime dependencies.

dockerfile 
# Dockerfile
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /install /usr/local
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

2. Kubernetes Deployment and Service

Kubernetes manages your containers at scale. Deployments define how many replicas to run, Services expose them to the network.

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: fastapi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: fastapi-app
  template:
    spec:
      containers:
      - name: api
        image: myapp:latest
        ports:
        - containerPort: 8000
        livenessProbe:
          httpGet:
            path: /health/live
            port: 8000
        readinessProbe:
          httpGet:
            path: /health/ready
            port: 8000

3. Horizontal Pod Autoscaler

HPA automatically scales your deployment based on CPU, memory, or custom metrics like request rate.

yaml 
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: fastapi-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: fastapi-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

4. Packaging with Helm

Helm packages Kubernetes manifests into reusable, versioned charts with configurable values. It is the package manager for Kubernetes.

yaml 
# Chart.yaml
apiVersion: v2
name: fastapi-app
version: 1.0.0
appVersion: "1.0"

# values.yaml
replicaCount: 3
image:
  repository: myregistry/fastapi-app
  tag: latest
resources:
  requests:
    cpu: 100m
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 512Mi

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you explain multi-stage Docker builds?

Stage 1 (builder) installs all build dependencies and compiles code. Stage 2 (runtime) starts from a clean base image and copies only the compiled artifacts. This produces images 2-10x smaller because build tools (gcc, pip cache) are not included in the final image.

Do you understand Kubernetes health probes?

Liveness: "Is the process alive?" Fails = container restart. Readiness: "Can it serve traffic?" Fails = removed from load balancer. Startup: "Has it finished initializing?" Fails during startup = wait longer. Getting these wrong causes unnecessary restarts or routing to unready pods.

Can you explain Horizontal Pod Autoscaler?

HPA monitors resource metrics (CPU, memory, custom) and adjusts replica count to maintain target utilization. It scales up when load increases and scales down when load decreases, within configured min/max bounds. Scale-down is intentionally slower than scale-up to prevent flapping.
