Observability and Monitoring

Structured logging, OpenTelemetry, Prometheus metrics, distributed tracing, and alerting. See inside your running application.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

You cannot fix what you cannot see. In production, your application is a black box unless you have observability: structured logs tell you what happened, metrics tell you how the system is performing, and traces tell you where time is spent across services.

Observability is not logging more — it is logging smarter. Structured JSON logs with request IDs, OpenTelemetry traces that cross service boundaries, and Prometheus metrics that power dashboards and alerts. Together, they give you the ability to diagnose any production issue in minutes, not hours.

Key Insight

The three pillars of observability are logs (events), metrics (aggregates), and traces (request flows). You need all three. Logs alone do not show trends. Metrics alone do not show root causes. Traces alone do not show the big picture.

Key Concepts

1. Structured Logging with structlog

Structured logs output JSON instead of free-text, making them searchable, filterable, and parseable by log aggregation tools.

python 
import structlog

structlog.configure(
    processors=[
        structlog.stdlib.filter_by_level,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

log = structlog.get_logger()
log.info("user.created", user_id=42, email="alice@example.com")
# {"event":"user.created","user_id":42,"email":"alice@example.com","level":"info","timestamp":"2024-01-15T..."}

2. Distributed Tracing with OpenTelemetry

OpenTelemetry provides vendor-neutral instrumentation for tracing requests across microservices.

python 
from opentelemetry import trace
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor

FastAPIInstrumentor.instrument_app(app)
tracer = trace.get_tracer(__name__)

@app.get("/orders/{order_id}")
async def get_order(order_id: int):
    with tracer.start_as_current_span("fetch_order") as span:
        span.set_attribute("order.id", order_id)
        order = await order_repo.get(order_id)
    return order

Prometheus scrapes metrics from your application at regular intervals. Grafana visualizes them in dashboards. Together they provide real-time monitoring and alerting.

4. Liveness, Readiness, and Startup Probes

Kubernetes uses health probes to manage container lifecycle. Liveness checks if the process is alive, readiness checks if it can serve traffic, and startup checks if it has finished initializing.

@app.get("/health/ready")
async def readiness():
    db_ok = await check_database()
    redis_ok = await check_redis()
    if not (db_ok and redis_ok):
        raise HTTPException(503)
    return {"status": "ready", "db": db_ok, "redis": redis_ok}

Self-Check

Before moving on, honestly answer these questions. If any answer is "no", go back and review.

Can you explain why structured logging is better than print statements?

Structured logs are JSON objects that can be filtered, searched, and aggregated by any field. print produces free-text that requires regex to parse. With structured logs, finding all errors for user_id=42 in the last hour is a simple query, not a grep nightmare.

Do you understand distributed tracing?

A trace follows a single request across all services it touches. Each service adds a span with timing data. Trace context (trace_id, span_id) propagates via HTTP headers. Tools like Jaeger visualize the complete request flow and identify bottlenecks.

Can you define meaningful SLOs for your API?

SLOs (Service Level Objectives) define reliability targets: 99.9% of requests complete in under 200ms, error rate below 0.1%. They are based on what users need, not what the system can achieve. Alerts fire when SLOs are at risk, not when arbitrary thresholds are crossed.
