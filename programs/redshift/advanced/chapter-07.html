Chapter 07 · Redshift Advanced

Advanced Performance Tuning

Master the art of diagnosing slow queries and systematically improving performance in Amazon Redshift — from reading EXPLAIN plans, analyzing SVL system views, optimizing distribution and sort keys, leveraging compilation and result caching, to eliminating data skew that silently cripples throughput.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

Performance tuning in Redshift is not guesswork. Every query that runs on your cluster generates detailed execution metadata in system tables and views. The difference between a 30-second query and a 3-second query often comes down to a single poorly chosen distribution key, a missing sort key, or a join pattern that forces a massive data redistribution across the network. In production environments with hundreds of concurrent users, these inefficiencies compound into cluster-wide bottlenecks.

Unlike traditional row-based databases where you add indexes to speed up queries, Redshift is a columnar MPP (Massively Parallel Processing) system. Performance tuning here means understanding how data is physically distributed across slices, how the query compiler generates execution steps, and how to align your table design with your most critical query patterns. The tools are different, but the methodology is systematic: measure, diagnose, change, re-measure.

Key Insight

Redshift has no traditional indexes (B-tree, hash, etc.). Performance is determined entirely by three factors: distribution style (how data is spread across slices), sort keys (how data is ordered within each slice for zone map elimination), and compression encoding (how much data can fit into memory per I/O operation). Master these three, and you master Redshift performance.

Key Concepts

1. Reading EXPLAIN Plans

The EXPLAIN statement is your first diagnostic tool. It shows the query execution plan that Redshift's optimizer generates before running the query. The plan reveals the order of operations (scans, joins, aggregations), the join algorithms chosen (hash join, merge join, nested loop), and critically, whether data redistribution is required between nodes.

Each step in the plan has an estimated cost (in arbitrary units), rows (estimated cardinality), and width (bytes per row). The most important thing to look for is DS_BCAST_INNER (broadcasting the inner table to all nodes) and DS_DIST_BOTH (redistributing both tables) — these are the most expensive redistribution strategies and signal that your distribution keys are misaligned with the join columns.

sql 
-- Run EXPLAIN to view the query plan
EXPLAIN
SELECT
    d.campaign_name,
    f.event_date,
    COUNT(*) AS impressions,
    SUM(f.revenue) AS total_revenue
FROM analytics.fact_impressions f
JOIN analytics.dim_campaign d
    ON f.campaign_id = d.campaign_id
WHERE f.event_date BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY d.campaign_name, f.event_date
ORDER BY total_revenue DESC;

-- Typical EXPLAIN output interpretation:
--   XN Merge  (cost=... rows=... width=...)
--     -> XN Network  (cost=...)
--       -> XN Sort  (cost=...)
--         -> XN HashAggregate  (cost=...)
--           -> XN Hash Join DS_DIST_NONE  (cost=...)
--                              ^^^^^^^^^^^^
--              DS_DIST_NONE = no redistribution needed (ideal!)
--              DS_DIST_INNER = inner table redistributed
--              DS_BCAST_INNER = inner table broadcasted to all
--              DS_DIST_BOTH = BOTH tables redistributed (worst)
--
--             -> XN Seq Scan on fact_impressions f
--                  Filter: (event_date >= '2025-01-01')
--             -> XN Hash  (cost=...)
--               -> XN Seq Scan on dim_campaign d
Important

EXPLAIN only shows the estimated plan. The actual execution may differ if table statistics are stale. Always run ANALYZE on your tables before relying on EXPLAIN output. Stale statistics can cause the optimizer to choose catastrophically bad join orders and redistribution strategies.

2. SVL and STL System View Diagnostics

While EXPLAIN shows the planned execution, the SVL_ and STL_ system views show what actually happened. After a query runs, Redshift logs detailed per-step execution data: actual row counts, execution time per segment, bytes processed, disk spills, and network transfer volumes. This is where you find the real bottlenecks.

sql 
-- Find the slowest queries in the last 24 hours
SELECT
    query,
    TRIM(querytxt) AS query_text,
    starttime,
    endtime,
    DATEDIFF(ms, starttime, endtime) AS duration_ms,
    aborted
FROM stl_query
WHERE starttime > GETDATE() - INTERVAL '24 hours'
  AND userid > 1    -- exclude system queries
ORDER BY duration_ms DESC
LIMIT 20;

-- Detailed per-step breakdown for a specific query
SELECT
    query,
    segment,
    step,
    label,
    rows,
    bytes,
    DATEDIFF(us, starttime, endtime) AS duration_us,
    is_diskbased   -- TRUE = spilled to disk (bad!)
FROM svl_query_summary
WHERE query = 12345678
ORDER BY segment, step;

-- Check for queries that spilled to disk (hash joins too large for memory)
SELECT
    query,
    segment,
    step,
    label,
    workmem,
    rows,
    bytes,
    is_diskbased
FROM svl_query_summary
WHERE is_diskbased = 't'
  AND query > (SELECT MAX(query) - 5000 FROM stl_query)
ORDER BY bytes DESC
LIMIT 20;
sql 
-- Identify queries with high network transfer (redistribution)
SELECT
    q.query,
    TRIM(q.querytxt) AS query_text,
    n.rows AS redistributed_rows,
    n.bytes AS redistributed_bytes,
    DATEDIFF(ms, q.starttime, q.endtime) AS duration_ms
FROM stl_query q
JOIN stl_dist n ON q.query = n.query
WHERE q.starttime > GETDATE() - INTERVAL '24 hours'
  AND n.bytes > 1073741824   -- more than 1 GB redistributed
ORDER BY n.bytes DESC;

-- Check queue wait times (query was queued before execution)
SELECT
    query,
    TRIM(querytxt) AS query_text,
    queue_start_time,
    exec_start_time,
    DATEDIFF(ms, queue_start_time, exec_start_time) AS queue_wait_ms,
    DATEDIFF(ms, exec_start_time, exec_end_time) AS exec_duration_ms,
    service_class
FROM stl_wlm_query
WHERE queue_start_time > GETDATE() - INTERVAL '24 hours'
  AND DATEDIFF(ms, queue_start_time, exec_start_time) > 5000
ORDER BY queue_wait_ms DESC;

3. Distribution Key Optimization

The single most impactful performance decision in Redshift is choosing the right distribution key for each table. The distribution key determines which slice stores each row. When two tables are joined on their distribution keys, the join can happen locally on each slice without any network transfer. When they are joined on non-distribution columns, Redshift must redistribute one or both tables across the network — an operation that can be orders of magnitude slower for large tables.

There are four distribution styles in Redshift: KEY (distribute by a specific column), ALL (replicate the entire table to every slice), EVEN (round-robin across slices), and AUTO (let Redshift choose). For production systems, you should almost always choose KEY or ALL explicitly based on your join patterns.

sql 
-- Check current distribution style and skew for all tables
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    info.diststyle,
    info.sortkey1,
    info.tbl_rows,
    info.size AS size_mb,
    info.skew_rows,
    info.skew_sortkey1
FROM svv_table_info info
JOIN pg_class pgc ON info.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE pgn.nspname = 'analytics'
ORDER BY info.tbl_rows DESC;

-- Analyze data skew: rows per slice for a specific table
SELECT
    TRIM(name) AS table_name,
    slice,
    num_values AS rows_in_slice,
    minvalue,
    maxvalue
FROM svv_diskusage
WHERE name = 'fact_impressions'
  AND col = 0
ORDER BY slice;

-- Recommended: fact table distributed on most common join key
CREATE TABLE analytics.fact_impressions (
    impression_id    BIGINT,
    event_date       DATE,
    event_timestamp  TIMESTAMP,
    campaign_id      INTEGER,
    creative_id      INTEGER,
    device_type      VARCHAR(20),
    country_code     CHAR(2),
    bid_price_cpm    DECIMAL(10,4),
    win_price_cpm    DECIMAL(10,4),
    revenue          DECIMAL(12,4)
)
DISTSTYLE KEY
DISTKEY (campaign_id)    -- matches dim_campaign join column
COMPOUND SORTKEY (event_date, campaign_id);

-- Small dimension table: replicate to all slices
CREATE TABLE analytics.dim_campaign (
    campaign_id      INTEGER,
    campaign_name    VARCHAR(200),
    advertiser_id    INTEGER,
    status           VARCHAR(20),
    budget_amount    DECIMAL(12,2)
)
DISTSTYLE ALL         -- every slice has full copy, join is always local
SORTKEY (campaign_id);
Pro Tip

Use DISTSTYLE ALL for dimension tables under ~5 million rows. The storage cost of replicating across slices is negligible, and every join with fact tables becomes co-located. For very large dimension tables (hundreds of millions of rows), use DISTSTYLE KEY on the primary key column instead.

4. Sort Key Effectiveness and Zone Maps

Sort keys determine the physical order of rows within each 1 MB data block on disk. Redshift maintains metadata called zone maps for each block: the minimum and maximum values stored in that block. When a query includes a WHERE clause on a sort key column, Redshift can skip entire blocks that cannot contain matching rows. This is called zone map elimination and can reduce I/O by 90% or more on time-series data filtered by date.

There are two types of sort keys: COMPOUND (optimized for queries that filter or order by the leading columns in sequence) and INTERLEAVED (gives equal weight to all sort key columns, useful when queries filter by different columns independently). Compound sort keys are almost always the right choice unless you have a very specific workload that filters by different columns unpredictably.

sql 
-- Check how effectively sort keys are being used
SELECT
    TRIM(s.perm_table_name) AS table_name,
    SUM(s.rows_pre_filter) AS rows_scanned_before_filter,
    SUM(s.rows) AS rows_after_filter,
    CASE WHEN SUM(s.rows_pre_filter) > 0
         THEN ROUND(100.0 * (1 - SUM(s.rows)::FLOAT / SUM(s.rows_pre_filter)), 1)
         ELSE 0
    END AS filter_pct,
    SUM(s.rows_pre_user_filter) AS rows_after_zone_map,
    CASE WHEN SUM(s.rows_pre_filter) > 0
         THEN ROUND(100.0 * (1 - SUM(s.rows_pre_user_filter)::FLOAT / SUM(s.rows_pre_filter)), 1)
         ELSE 0
    END AS zone_map_elimination_pct
FROM stl_scan s
WHERE s.starttime > GETDATE() - INTERVAL '24 hours'
  AND s.perm_table_name NOT LIKE 'Internal%'
GROUP BY TRIM(s.perm_table_name)
HAVING SUM(s.rows_pre_filter) > 10000
ORDER BY zone_map_elimination_pct ASC;

-- Check VACUUM status: unsorted regions hurt zone map effectiveness
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    info.unsorted,
    info.stats_off,
    info.tbl_rows,
    info.size AS size_mb
FROM svv_table_info info
JOIN pg_class pgc ON info.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE info.unsorted > 20   -- more than 20% unsorted
ORDER BY info.unsorted DESC;

  ┌────────────────────────────────────────────────────────────────────┐
  │  COMPOUND SORTKEY (event_date, campaign_id)                        │
  │                                                                    │
  │  Block 1: event_date [2025-01-01 .. 2025-01-03]  campaign_id [1..50]  │
  │  Block 2: event_date [2025-01-03 .. 2025-01-05]  campaign_id [1..48]  │
  │  Block 3: event_date [2025-01-06 .. 2025-01-09]  campaign_id [2..45]  │
  │  Block 4: event_date [2025-01-09 .. 2025-01-12]  campaign_id [1..52]  │
  │  Block 5: event_date [2025-01-12 .. 2025-01-15]  campaign_id [3..50]  │
  │                                                                    │
  │  Query: WHERE event_date = '2025-01-07'                            │

5. Compilation Caching and Result Caching

Redshift compiles every query into C++ code before execution. This compilation step can take 2–10 seconds for complex queries, which adds significant latency to the first execution. However, Redshift caches compiled query plans in a compilation cache. Subsequent executions of the same query pattern (with different literal values) reuse the compiled plan, reducing cold-start latency dramatically.

In addition to compilation caching, Redshift has a result cache that stores the full result set of queries. If the same query is executed again and the underlying data has not changed, Redshift returns the cached result in milliseconds without re-executing the query at all. The result cache is enabled by default and is invalidated automatically when underlying tables are modified.

sql 
-- Check if a query was served from the result cache
SELECT
    query,
    TRIM(querytxt) AS query_text,
    starttime,
    endtime,
    DATEDIFF(us, starttime, endtime) AS duration_us,
    source_query   -- non-null = served from result cache
FROM svl_qlog
WHERE starttime > GETDATE() - INTERVAL '1 hour'
  AND source_query IS NOT NULL
ORDER BY starttime DESC;

-- Check compilation cache hit rate
SELECT
    DATEPART(hour, starttime) AS hour_of_day,
    COUNT(*) AS total_queries,
    SUM(CASE WHEN compile_time = 0 THEN 1 ELSE 0 END) AS cache_hits,
    ROUND(100.0 * SUM(CASE WHEN compile_time = 0 THEN 1 ELSE 0 END) / COUNT(*), 1) AS hit_rate_pct
FROM svl_compile
WHERE starttime > GETDATE() - INTERVAL '24 hours'
GROUP BY DATEPART(hour, starttime)
ORDER BY hour_of_day;

-- Disable result cache for benchmarking (session-level only)
SET enable_result_cache_for_session = off;

-- Re-enable result cache
SET enable_result_cache_for_session = on;

-- Use prepared statements to maximize compilation cache reuse
PREPARE daily_campaign_report (DATE, DATE) AS
SELECT
    d.campaign_name,
    f.event_date,
    COUNT(*) AS impressions,
    SUM(f.revenue) AS total_revenue
FROM analytics.fact_impressions f
JOIN analytics.dim_campaign d
    ON f.campaign_id = d.campaign_id
WHERE f.event_date BETWEEN $1 AND $2
GROUP BY d.campaign_name, f.event_date;

EXECUTE daily_campaign_report('2025-01-01', '2025-01-31');
Common Pitfall

Do not rely on the result cache for performance benchmarking. When testing query optimizations, always disable the result cache with SET enable_result_cache_for_session = off; so you measure actual query execution time. The result cache can make a 30-second query appear to take 5 milliseconds, hiding real performance problems.

6. Detecting and Eliminating Data Skew

Data skew occurs when one or more slices hold significantly more data than others. In an MPP system like Redshift, a query's speed is limited by the slowest slice. If one slice has 10x more data than the others, that single slice becomes a bottleneck for every query that touches the skewed table. Skew is typically caused by a poor distribution key choice — distributing on a column with low cardinality or highly uneven value distribution.

sql 
-- Identify tables with significant data skew
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    info.diststyle,
    info.tbl_rows,
    info.skew_rows,
    info.skew_sortkey1,
    CASE
        WHEN info.skew_rows > 4.0 THEN 'CRITICAL'
        WHEN info.skew_rows > 2.0 THEN 'WARNING'
        WHEN info.skew_rows > 1.5 THEN 'MODERATE'
        ELSE 'OK'
    END AS skew_status
FROM svv_table_info info
JOIN pg_class pgc ON info.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE pgn.nspname NOT IN ('pg_catalog', 'information_schema')
  AND info.tbl_rows > 100000
ORDER BY info.skew_rows DESC;

-- Deep-dive: check distribution key value distribution
SELECT
    campaign_id,
    COUNT(*) AS row_count,
    ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 2) AS pct_of_total
FROM analytics.fact_impressions
GROUP BY campaign_id
ORDER BY row_count DESC
LIMIT 20;

-- If one campaign_id has 40% of rows, it causes skew
-- Solution: choose a higher-cardinality distribution key
ALTER TABLE analytics.fact_impressions
ALTER DISTSTYLE KEY DISTKEY impression_id;

-- Or use EVEN distribution if no good key exists
ALTER TABLE analytics.fact_impressions
ALTER DISTSTYLE EVEN;

-- Detect processing skew at query execution time
SELECT
    query,
    segment,
    step,
    slice,
    label,
    rows,
    bytes,
    DATEDIFF(us, starttime, endtime) AS duration_us
FROM svl_query_report
WHERE query = 12345678
ORDER BY segment, step, slice;
-- Look for slices with much higher rows/bytes/duration than others
Important

The skew_rows column in svv_table_info is the ratio of the maximum number of rows on any single slice to the average. A value of 1.0 means perfect distribution. Values above 2.0 indicate significant skew and should be investigated. Values above 4.0 are critical — one slice has 4x the average workload, meaning that slice is the bottleneck for every query on that table.

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What do DS_DIST_NONE and DS_BCAST_INNER mean in an EXPLAIN plan, and which is preferable?

DS_DIST_NONE means no data redistribution is needed — the join can happen locally on each slice because both tables are already co-located (or one is DISTSTYLE ALL). This is the ideal scenario. DS_BCAST_INNER means the entire inner (smaller) table must be broadcast to every slice, which involves significant network transfer. Always aim for DS_DIST_NONE by aligning distribution keys with join columns or using DISTSTYLE ALL for small dimension tables.

How do zone maps reduce I/O, and what causes them to become ineffective?

Zone maps record the min/max values for each 1 MB data block. When a query filters on a sort key column, Redshift skips blocks whose min/max range does not overlap the filter predicate. Zone maps become ineffective when: (1) the table has a large unsorted region from recent INSERTs that bypassed the sort order; (2) the sort key has low cardinality causing wide min/max ranges per block; or (3) the wrong column was chosen as the sort key for the dominant query patterns. Running VACUUM SORT re-sorts unsorted data to restore zone map effectiveness.

When should you use INTERLEAVED SORTKEY instead of COMPOUND SORTKEY?

Use INTERLEAVED SORTKEY only when your workload has queries that frequently filter by different columns independently (e.g., sometimes by event_date, sometimes by campaign_id, sometimes by country_code, without a predictable leading-column pattern). COMPOUND is superior when queries consistently filter by the leading column(s) in order. Interleaved sort keys are much more expensive to VACUUM (requires full table rewrite) and should be avoided unless the query pattern clearly demands it.

Why is data skew especially damaging in an MPP system like Redshift?

In an MPP system, every query step waits for the slowest slice to finish before proceeding to the next step. If one slice has 10x the data of the average, that slice takes 10x longer to scan, filter, and aggregate its portion. All other slices finish quickly and then wait idle. The query duration is determined by the single most overloaded slice, not the average. This is why even moderate skew (2x) can effectively halve your cluster's effective throughput.

What is the difference between compilation caching and result caching in Redshift?

Compilation caching stores the compiled machine code (C++) for a query's execution plan. When the same query pattern runs again (even with different literal values), Redshift reuses the compiled code, saving 2–10 seconds of compilation overhead. Result caching stores the actual query output. If the exact same query runs again and no underlying data has changed, the cached result is returned in milliseconds without any execution. Compilation cache helps with repeated query patterns; result cache helps with repeated identical queries.
