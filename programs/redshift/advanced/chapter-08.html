Chapter 08 · Redshift Advanced

Concurrency and Resource Management

Learn to manage competing workloads on a shared Redshift cluster — configuring Workload Management (WLM) queues, using concurrency scaling for burst capacity, setting Query Monitoring Rules (QMR) to enforce guardrails, and preventing runaway queries from starving critical dashboards and ETL pipelines.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

A Redshift cluster is a shared resource. Your ETL pipelines, BI dashboards, ad-hoc analyst queries, and machine learning feature extraction jobs all compete for the same CPU, memory, and I/O bandwidth. Without explicit resource management, a single unoptimized analyst query can consume all available memory on every slice, causing critical dashboard queries to queue for minutes and ETL pipelines to miss their SLA windows.

Workload Management (WLM) is Redshift's mechanism for controlling this contention. It lets you define separate queues with guaranteed memory allocations, concurrency limits, and timeout thresholds. Combined with Query Monitoring Rules (QMR) and Concurrency Scaling, you can build a robust resource governance framework that ensures critical workloads always get the resources they need while still allowing exploratory queries to run without special intervention.

Key Insight

Redshift allocates memory per-query, not per-user. If you allow 15 concurrent queries and each gets an equal share of memory, a single complex hash join might not have enough memory and will spill to disk, running 10–100x slower. Conversely, if you allow only 5 concurrent queries, each gets 3x more memory but other users wait in the queue. WLM configuration is fundamentally about managing this tradeoff between concurrency and per-query memory.

Key Concepts

1. WLM Queue Configuration

Workload Management (WLM) lets you define multiple query queues, each with its own concurrency limit and memory allocation. Queries are routed to queues based on the user group or query group of the session. The default Redshift configuration has a single queue with a concurrency of 5, which is rarely appropriate for production workloads.

Redshift offers two WLM modes: Manual WLM (you explicitly set concurrency and memory percentages per queue) and Auto WLM (Redshift dynamically adjusts memory and concurrency based on workload). Auto WLM is the recommended default for most workloads, but understanding manual WLM is essential for troubleshooting and for workloads that need guaranteed resource isolation.

sql 
-- View current WLM configuration
SELECT
    service_class,
    name,
    num_query_tasks AS concurrency_slots,
    query_working_mem AS memory_per_slot_mb,
    max_execution_time AS timeout_ms,
    user_group_wild_card,
    query_group_wild_card
FROM stv_wlm_service_class_config
WHERE service_class > 5  -- user-defined queues start at 6
ORDER BY service_class;

-- Check current queue state: running and queued queries
SELECT
    service_class,
    num_queued_queries,
    num_executing_queries,
    num_executed_queries
FROM stv_wlm_service_class_state
WHERE service_class > 5;

-- Route a session to a specific query group
SET query_group TO 'etl_pipeline';

-- Run ETL queries here: they will be routed to the ETL queue
COPY staging.raw_impressions
FROM 's3://my-bucket/impressions/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftRole'
FORMAT AS PARQUET;

-- Reset to default queue
RESET query_group;

-- Alternative: route by user group (assigned at user creation)
CREATE USER etl_service_account PASSWORD 'SecurePass123!'
IN GROUP etl_users;

Important

WLM configuration changes are applied via the AWS Console or API through Parameter Groups — they cannot be set via SQL. After modifying WLM parameters, you must reboot the cluster for Manual WLM changes. However, Auto WLM and query priority changes take effect dynamically without a reboot. Always use Auto WLM unless you have a specific reason to use Manual WLM.

2. Auto WLM and Query Priorities

Auto WLM is Amazon's recommended approach for most workloads. Instead of manually specifying concurrency and memory allocations, Auto WLM dynamically determines the optimal number of concurrent queries and memory allocation based on the current workload. You can still define multiple queues, but instead of setting explicit concurrency numbers, you assign priority levels (Highest, High, Normal, Low, Lowest) to each queue.

sql 
-- With Auto WLM, you configure priorities in the Parameter Group.
-- The JSON configuration looks like this:
--
-- [
--   {
--     "name": "ETL Pipeline",
--     "query_group": ["etl_pipeline"],
--     "priority": "highest",
--     "auto_wlm": true
--   },
--   {
--     "name": "Dashboard",
--     "query_group": ["dashboard"],
--     "priority": "high",
--     "auto_wlm": true
--   },
--   {
--     "name": "Ad-Hoc Analysts",
--     "user_group": ["analysts"],
--     "priority": "normal",
--     "auto_wlm": true
--   },
--   {
--     "name": "Default",
--     "priority": "low",
--     "auto_wlm": true
--   }
-- ]

-- Monitor which priority each query is running at
SELECT
    w.query,
    TRIM(q.querytxt) AS query_text,
    w.service_class,
    w.query_priority,
    DATEDIFF(ms, w.queue_start_time, w.exec_start_time) AS queue_wait_ms,
    DATEDIFF(ms, w.exec_start_time, w.exec_end_time) AS exec_ms
FROM stl_wlm_query w
JOIN stl_query q ON w.query = q.query
WHERE w.queue_start_time > GETDATE() - INTERVAL '1 hour'
ORDER BY w.queue_start_time DESC
LIMIT 30;

-- Override priority for a specific session
SET query_group TO 'superuser';
CHANGE query_priority 12345678 TO 'HIGHEST';

3. Concurrency Scaling

Even with well-configured WLM queues, there are times when the cluster simply does not have enough capacity to serve all incoming queries without queuing. Concurrency Scaling addresses this by automatically adding temporary cluster capacity when needed. When queries start queuing, Redshift spins up additional clusters (concurrency scaling clusters) in the background and routes eligible read queries to them.

Concurrency scaling clusters are billed per-second of active use. Each cluster accumulates one hour of free concurrency scaling credits per day. For most workloads, the free credits cover burst periods adequately. You enable concurrency scaling per-queue in the WLM configuration.

sql 
-- Monitor concurrency scaling activity
SELECT
    service_class,
    query,
    TRIM(querytxt) AS query_text,
    starttime,
    endtime,
    DATEDIFF(ms, starttime, endtime) AS duration_ms,
    concurrency_scaling_status
FROM stl_query
WHERE concurrency_scaling_status > 0
  AND starttime > GETDATE() - INTERVAL '24 hours'
ORDER BY starttime DESC;
-- concurrency_scaling_status:
--   0 = ran on main cluster
--   1 = ran on concurrency scaling cluster
--   2 = eligible but ran on main (scaling cluster not yet available)

-- Check concurrency scaling usage and free credit balance
SELECT
    start_time,
    end_time,
    DATEDIFF(second, start_time, end_time) AS active_seconds,
    queries
FROM stl_concurrency_scaling_usage
WHERE start_time > GETDATE() - INTERVAL '7 days'
ORDER BY start_time DESC;

-- Disable concurrency scaling for a specific session
SET max_concurrency_scaling_clusters TO 0;

-- Limit to at most 3 additional clusters
SET max_concurrency_scaling_clusters TO 3;
Pro Tip

Enable concurrency scaling only on the queues that need it (typically dashboard and ad-hoc queues). ETL queues should not use concurrency scaling because write operations (COPY, INSERT, DELETE) are not eligible for scaling clusters. Only read queries (SELECT) can be offloaded. Enabling it on ETL queues wastes credits with no benefit.

4. Query Monitoring Rules (QMR)

Query Monitoring Rules (QMR) act as guardrails that detect and handle problematic queries in real time. You define rules based on metrics like execution time, CPU usage, rows returned, disk spill, and join type. When a running query triggers a rule, Redshift can automatically LOG the event, HOP the query to a different queue (with lower priority), or ABORT the query entirely.

sql 
-- QMR rules are defined in the WLM JSON configuration.
-- Example rules (applied via Parameter Group):
--
-- Rule 1: Abort queries that run longer than 10 minutes in the ad-hoc queue
-- {
--   "rule_name": "abort_long_adhoc",
--   "predicate": [
--     {"metric_name": "query_execution_time", "operator": ">", "value": 600}
--   ],
--   "action": "abort"
-- }
--
-- Rule 2: Log queries returning more than 1 billion rows
-- {
--   "rule_name": "log_massive_result",
--   "predicate": [
--     {"metric_name": "return_row_count", "operator": ">", "value": 1000000000}
--   ],
--   "action": "log"
-- }
--
-- Rule 3: Hop queries that spill to disk to a lower-priority queue
-- {
--   "rule_name": "hop_disk_spill",
--   "predicate": [
--     {"metric_name": "query_temp_blocks_to_disk", "operator": ">", "value": 100000}
--   ],
--   "action": "hop"
-- }

-- Check which queries were affected by QMR rules
SELECT
    query,
    TRIM(querytxt) AS query_text,
    rule_name,
    action,
    starttime,
    metric_name,
    metric_value
FROM stl_wlm_rule_action
WHERE starttime > GETDATE() - INTERVAL '24 hours'
ORDER BY starttime DESC;

-- Available QMR metrics:
--   query_execution_time      (seconds)
--   query_cpu_time             (seconds)
--   query_blocks_read          (1 MB blocks)
--   return_row_count           (rows returned to client)
--   scan_row_count             (rows scanned)
--   join_row_count             (rows produced by joins)
--   nested_loop_join_row_count (rows from nested loops, very bad!)
--   query_temp_blocks_to_disk  (1 MB blocks spilled)
--   spectrum_scan_row_count    (rows from Spectrum)
--   spectrum_scan_size_mb      (MB scanned by Spectrum)
--   cpu_usage_percent          (aggregate CPU percentage)
Common Pitfall

Never set the QMR ABORT action on the ETL queue unless you have robust retry logic in your orchestrator. Aborting a COPY command mid-load can leave staging tables in an inconsistent state. Use LOG or HOP actions for ETL queues, and reserve ABORT for ad-hoc analyst queues where query termination is acceptable.

5. Lock Management and Deadlock Prevention

Redshift uses table-level locks for DDL operations and write operations. When an ETL pipeline runs ALTER TABLE, DROP TABLE, or VACUUM, it acquires an exclusive lock on the table. Any concurrent queries on that table will wait until the lock is released. In production systems with overlapping ETL windows and dashboard queries, lock contention is a major source of latency spikes.

sql 
-- View currently held locks
SELECT
    l.table_id,
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    l.lock_owner,
    l.lock_owner_pid,
    l.lock_mode,
    l.lock_granted
FROM svv_transactions l
JOIN pg_class pgc ON l.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE l.lock_mode IN ('AccessExclusiveLock', 'ExclusiveLock')
ORDER BY l.txn_start;

-- View blocking chains: who is waiting on whom
SELECT
    blocked.pid AS blocked_pid,
    TRIM(blocked_q.querytxt) AS blocked_query,
    blocker.pid AS blocking_pid,
    TRIM(blocker_q.querytxt) AS blocking_query,
    blocked.txn_start AS blocked_since
FROM stv_locks blocked
JOIN stv_locks blocker
    ON blocked.table_id = blocker.table_id
    AND blocked.lock_granted = 'f'
    AND blocker.lock_granted = 't'
JOIN stv_recents blocked_q ON blocked.pid = blocked_q.pid
JOIN stv_recents blocker_q ON blocker.pid = blocker_q.pid;

-- Cancel a specific blocking query
SELECT pg_cancel_backend(12345);

-- Force-terminate a session (more aggressive than cancel)
SELECT pg_terminate_backend(12345);

-- Best practice: set statement_timeout for all sessions
ALTER USER analyst_user SET statement_timeout TO 600000;  -- 10 minutes
ALTER USER etl_service SET statement_timeout TO 3600000;  -- 1 hour
Important

Schedule VACUUM and ANALYZE operations during low-usage windows (late night or early morning). VACUUM acquires an exclusive lock on the table, blocking all concurrent queries. If VACUUM runs during business hours on a table used by dashboards, every dashboard query will queue until VACUUM completes. Consider using VACUUM DELETE ONLY for faster runs when full re-sorting is not needed.

6. Short Query Acceleration (SQA)

Short Query Acceleration (SQA) is an Auto WLM feature that identifies short-running queries and routes them to a dedicated fast lane, bypassing the normal queue. Redshift uses machine learning to predict a query's execution time. If the predicted time is below the SQA threshold (default: 5 seconds), the query runs immediately without waiting in any queue, even if all queue slots are occupied.

SQA is particularly valuable for interactive dashboards where users expect sub-second to few-second responses. Without SQA, a simple SELECT COUNT(*) FROM dim_campaign might wait behind a 5-minute analytical query in the same queue. With SQA enabled, the count query bypasses the queue entirely and returns instantly.

sql 
-- Check SQA effectiveness: queries that were accelerated
SELECT
    query,
    TRIM(querytxt) AS query_text,
    service_class,
    final_state,
    DATEDIFF(us, starttime, endtime) AS duration_us,
    CASE WHEN service_class = 14 THEN 'SQA' ELSE 'Regular' END AS execution_path
FROM stl_wlm_query
WHERE queue_start_time > GETDATE() - INTERVAL '1 hour'
  AND service_class = 14  -- SQA service class
ORDER BY queue_start_time DESC;

-- Measure SQA hit rate
SELECT
    COUNT(*) AS total_queries,
    SUM(CASE WHEN service_class = 14 THEN 1 ELSE 0 END) AS sqa_queries,
    ROUND(100.0 * SUM(CASE WHEN service_class = 14 THEN 1 ELSE 0 END) / COUNT(*), 1) AS sqa_pct
FROM stl_wlm_query
WHERE queue_start_time > GETDATE() - INTERVAL '24 hours';

-- SQA configuration is set in the Parameter Group:
-- "short_query_queue": true
-- "max_execution_time": 5  (seconds, default threshold)
--
-- You can adjust the threshold via the Parameter Group.
-- Raising it to 10-15 seconds captures more medium queries.

-- Comprehensive WLM monitoring view
SELECT
    DATE_TRUNC('hour', queue_start_time) AS hour_bucket,
    service_class,
    COUNT(*) AS queries_executed,
    AVG(DATEDIFF(ms, queue_start_time, exec_start_time)) AS avg_queue_wait_ms,
    MAX(DATEDIFF(ms, queue_start_time, exec_start_time)) AS max_queue_wait_ms,
    AVG(DATEDIFF(ms, exec_start_time, exec_end_time)) AS avg_exec_ms,
    SUM(CASE WHEN final_state = 'Aborted' THEN 1 ELSE 0 END) AS aborted_count
FROM stl_wlm_query
WHERE queue_start_time > GETDATE() - INTERVAL '24 hours'
GROUP BY DATE_TRUNC('hour', queue_start_time), service_class
ORDER BY hour_bucket, service_class;

Self-Check

Before moving to Chapter 09, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What is the fundamental tradeoff when increasing WLM queue concurrency?

Increasing concurrency means more queries can run simultaneously, but each query gets a smaller share of memory. If a complex hash join no longer fits in its memory allocation, it spills to disk and runs 10–100x slower. Decreasing concurrency gives each query more memory (fewer disk spills, faster joins) but forces other queries to wait in the queue. The optimal setting depends on your mix of simple and complex queries.

What types of queries are eligible for Concurrency Scaling, and what types are not?

Only read queries (SELECT) are eligible for concurrency scaling clusters. Write operations (INSERT, UPDATE, DELETE, COPY, UNLOAD), DDL statements (CREATE, ALTER, DROP), and VACUUM/ANALYZE are not eligible and always run on the main cluster. This means enabling concurrency scaling on ETL queues provides no benefit since ETL is primarily write operations.

What are the three QMR actions, and when should you use each?

LOG: Records the rule violation but does not interrupt the query. Use for monitoring and establishing baselines. HOP: Moves the query to a lower-priority queue where it gets fewer resources but continues running. Use for workloads that should eventually complete but should not hog resources. ABORT: Terminates the query immediately. Use for ad-hoc queues where runaway queries must be stopped, but never for ETL queues unless you have retry logic in your orchestrator.

How does Short Query Acceleration (SQA) determine which queries to fast-track?

SQA uses a machine learning model trained on the cluster's historical query execution data. Before a query enters a WLM queue, Redshift estimates its execution time based on the query plan, table statistics, and historical patterns for similar queries. If the estimated time is below the SQA threshold (configurable, default 5 seconds), the query is routed directly to the SQA lane, bypassing all WLM queues. If the prediction is wrong and the query runs longer than expected, it is moved back to a regular WLM queue.

Why should VACUUM and ANALYZE run during low-usage windows, not during business hours?

VACUUM acquires an exclusive lock on the table, blocking all other queries on that table until it completes. If VACUUM runs on fact_impressions during business hours, every dashboard query that touches that table will queue until VACUUM finishes, which can take minutes to hours depending on table size. ANALYZE reads table data to update statistics, which while not as blocking as VACUUM, still consumes I/O bandwidth and memory. Both should run during maintenance windows when user query volume is lowest.
