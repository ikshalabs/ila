Chapter 04 · Redshift Advanced

Aggregation and Refresh Strategies

Master materialized views, incremental aggregation pipelines, and refresh orchestration patterns that keep summary tables accurate without re-scanning terabytes of raw event data on every dashboard query.

Estimated: 55–70 min
6 Key Concepts
3 Assignments

Why This Matters

In a production Redshift warehouse, raw fact tables — impressions, clicks, bid logs — routinely contain billions of rows. Dashboard analysts and BI tools rarely need row-level detail. They need daily campaign summaries, hourly spend aggregates, or weekly performance roll-ups. Scanning the entire fact table for every dashboard refresh is not only slow; it consumes compute resources that other concurrent queries desperately need.

Materialized views solve this by precomputing aggregation results and storing them physically on disk. When an analyst runs a query that matches the materialized view definition, Redshift can serve the result from the precomputed data instead of re-scanning the base table. The challenge lies in keeping these views fresh — new data lands every hour, and stale aggregates lead to incorrect business decisions about budget pacing, bid adjustments, and campaign optimization.

This chapter teaches the full lifecycle: creating materialized views, choosing between full and incremental refresh, scheduling refresh pipelines, monitoring staleness, and handling edge cases like late-arriving data and schema evolution. By the end, you will design an aggregation layer that serves sub-second dashboard queries while staying within minutes of the latest raw data.

Key Insight

A well-designed aggregation layer typically reduces dashboard query scan volumes by 95-99%. A query that scans 500 million rows from fact_impressions might scan only 50,000 rows from a daily summary materialized view — finishing in milliseconds instead of seconds.

Key Concepts

1. Materialized View Fundamentals

A materialized view in Redshift is a database object that stores the result set of a query physically. Unlike a regular view (which is just a stored SQL definition executed on every access), a materialized view pre-executes the query and persists the output. Subsequent reads hit the stored result, not the underlying base tables. This is the fundamental mechanism behind dashboard acceleration.

Redshift materialized views support a substantial subset of SQL including GROUP BY, JOIN, aggregate functions (SUM, COUNT, AVG, MIN, MAX), CASE expressions, and window functions in certain configurations. They can reference base tables, other materialized views, and even external Spectrum tables — enabling you to build aggregation layers that span your entire data estate.

sql 
-- Create a daily campaign performance summary
CREATE MATERIALIZED VIEW analytics.mv_daily_campaign_summary
AUTO REFRESH YES
AS
SELECT
    fi.event_date,
    fi.campaign_id,
    dc.campaign_name,
    dc.advertiser_name,
    COUNT(*) AS impressions,
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN fi.is_viewable THEN 1 ELSE 0 END) AS viewable_impressions,
    SUM(fi.win_price_cpm) AS total_spend_cpm,
    AVG(fi.win_price_cpm) AS avg_cpm,
    MIN(fi.event_timestamp) AS first_event,
    MAX(fi.event_timestamp) AS last_event
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc
    ON fi.campaign_id = dc.campaign_id
GROUP BY
    fi.event_date,
    fi.campaign_id,
    dc.campaign_name,
    dc.advertiser_name;

The AUTO REFRESH YES clause instructs Redshift to automatically refresh this materialized view when the base tables change. Redshift determines the optimal time based on cluster workload. You can also trigger manual refreshes for immediate consistency when your ETL pipeline completes.

Important

Materialized views in Redshift are eventually consistent by default. Between the moment new data is loaded and the next refresh, queries against the MV return stale results. Design your pipelines with this latency budget in mind. For dashboards refreshing every 15 minutes, a 5-minute staleness window is usually acceptable.

2. Incremental vs. Full Refresh

Redshift supports two refresh modes for materialized views. Understanding when each is used is critical for production performance and cost control.

Incremental refresh processes only rows changed since the last refresh. Redshift tracks which base table blocks were modified and applies only those deltas. This is dramatically faster than a full rebuild. However, incremental refresh is only available when the MV uses INNER JOIN, supported aggregate functions, no subqueries in the SELECT list, and no DDL has been run on the base tables since the last refresh.

Full refresh (recompute) re-executes the entire defining query from scratch. This is required for MVs using OUTER JOIN, subqueries, certain window functions, or when the base table schema has changed. Full refresh is essentially dropping and recreating the MV data.

sql 
-- Manual refresh (Redshift picks incremental if eligible)
REFRESH MATERIALIZED VIEW analytics.mv_daily_campaign_summary;

-- Check the current refresh state of all MVs
SELECT
    schema_name,
    mv_name       AS materialized_view,
    state         AS refresh_state,
    autorefresh   AS auto_refresh_enabled,
    autorewrite   AS auto_rewrite_enabled
FROM svv_mv_info
WHERE schema_name = 'analytics';

-- Check refresh history and duration
SELECT
    mv_name,
    refresh_type,     -- 'Incremental' or 'Recompute'
    status,
    start_time,
    end_time,
    DATEDIFF('second', start_time, end_time) AS duration_sec
FROM svl_mv_refresh_status
WHERE mv_name = 'mv_daily_campaign_summary'
ORDER BY start_time DESC
LIMIT 10;
Pro Tip

If svl_mv_refresh_status shows refresh_type = 'Recompute' when you expected incremental, check whether any DDL was run on the base tables since the last refresh. An ALTER TABLE or TRUNCATE forces a full recompute. Design your ETL to avoid DDL on MV-backing tables during business hours.

3. Automatic Query Rewrite

One of Redshift's most powerful features is automatic query rewrite. When enabled, the query optimizer automatically recognizes that a user query against the base table can be satisfied by an existing materialized view, and transparently rewrites the execution to read from the MV instead.

This means analysts can continue writing queries against the raw fact table without knowing materialized views exist. The optimizer handles the routing transparently. For this to work, the MV must be recently refreshed, and the query must be a logical subset of the MV definition — same or fewer grouping columns, compatible WHERE clause, and compatible aggregate functions.

sql 
-- The analyst writes this against the raw fact table:
SELECT
    event_date,
    campaign_id,
    COUNT(*) AS impressions
FROM analytics.fact_impressions
WHERE event_date = '2025-06-15'
GROUP BY event_date, campaign_id;

-- Redshift auto-rewrites to read from the MV:
-- SELECT event_date, campaign_id, impressions
-- FROM analytics.mv_daily_campaign_summary
-- WHERE event_date = '2025-06-15';

-- Verify rewrite happened in the query plan:
EXPLAIN
SELECT event_date, campaign_id, COUNT(*) AS impressions
FROM analytics.fact_impressions
WHERE event_date = '2025-06-15'
GROUP BY event_date, campaign_id;
-- Look for "Scan mv_daily_campaign_summary" in the EXPLAIN output

4. Layered Aggregation Patterns

Production warehouses rarely use a single aggregation layer. Instead, they build a hierarchy of materialized views where each layer summarizes the one below. This is the aggregation pyramid pattern, and it is the standard architecture for high-performance analytics: 

  Layer 0 (Raw):     fact_impressions          ~500M rows/day
                            |
  Layer 1 (Hourly):  mv_hourly_campaign        ~24 x 5000 = 120K rows/day
                            |
  Layer 2 (Daily):   mv_daily_campaign         ~5000 rows/day
                            |
  Layer 3 (Weekly):  mv_weekly_campaign         ~700 rows/week
                            |
  Layer 4 (Monthly): mv_monthly_campaign        ~150 rows/month
Figure 1: Aggregation pyramid — each layer compresses row volume by orders of magnitude
sql 
-- Layer 1: Hourly aggregation from raw events
CREATE MATERIALIZED VIEW analytics.mv_hourly_campaign
AUTO REFRESH YES
AS
SELECT
    DATE_TRUNC('hour', event_timestamp) AS event_hour,
    campaign_id,
    device_type,
    country_code,
    COUNT(*) AS impressions,
    SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(win_price_cpm) AS total_spend,
    COUNT(win_price_cpm) AS spend_count  -- for correct AVG later
FROM analytics.fact_impressions
GROUP BY 1, 2, 3, 4;

-- Layer 2: Daily rollup built ON TOP of hourly MV
CREATE MATERIALIZED VIEW analytics.mv_daily_campaign_rollup
AUTO REFRESH YES
AS
SELECT
    DATE_TRUNC('day', event_hour) AS event_date,
    campaign_id,
    SUM(impressions) AS impressions,
    SUM(clicks) AS clicks,
    SUM(total_spend) AS total_spend,
    SUM(spend_count) AS spend_count,
    ROUND(
      SUM(clicks)::DECIMAL / NULLIF(SUM(impressions), 0) * 100, 4
    ) AS ctr_pct,
    ROUND(
      SUM(total_spend) / NULLIF(SUM(spend_count), 0), 4
    ) AS avg_cpm
FROM analytics.mv_hourly_campaign
GROUP BY 1, 2;
Warning

When chaining MVs, AVG does not compose correctly across layers. You cannot average averages. Store SUM and COUNT separately in lower layers and compute the average at query time: SUM(total_spend) / NULLIF(SUM(spend_count), 0). Failing to do this is one of the most common aggregation bugs in production.

5. Handling Late-Arriving Data

In real-time ad-tech pipelines, events arrive out of order. An impression from 14:00 might not arrive until 14:30 due to network delays, batch processing windows, or retry logic. If your hourly MV was already refreshed at 14:15, the late data will be missing until the next refresh cycle.

The standard pattern is the idempotent upsert with partition rebuild. Instead of simple appends, your ETL targets a specific time partition and replaces all data within that window:

sql 
-- Idempotent load for late-arriving impressions
-- Step 1: Delete the target partition (safe to re-run)
BEGIN;

DELETE FROM analytics.fact_impressions
WHERE event_date = '2025-06-15'
  AND event_timestamp BETWEEN '2025-06-15 14:00:00'
                        AND '2025-06-15 14:59:59';

-- Step 2: Insert the complete hour from staging
INSERT INTO analytics.fact_impressions
SELECT *
FROM staging.impressions_landing
WHERE event_date = '2025-06-15'
  AND event_timestamp BETWEEN '2025-06-15 14:00:00'
                        AND '2025-06-15 14:59:59';

COMMIT;

-- Step 3: Refresh the MV to pick up changes
REFRESH MATERIALIZED VIEW analytics.mv_hourly_campaign;

-- Step 4: Verify data integrity
SELECT
    COUNT(*) AS row_count,
    MIN(event_timestamp) AS earliest,
    MAX(event_timestamp) AS latest
FROM analytics.fact_impressions
WHERE event_date = '2025-06-15';
Pro Tip

Define a late-arrival window for your pipeline — typically 2 hours for ad-tech. Your ETL always reprocesses the current hour plus the previous 2 hours, guaranteeing that 99.9% of late events are captured without rebuilding the entire day.

6. Scheduling and Orchestrating Refreshes

While AUTO REFRESH is convenient for development, production pipelines need deterministic scheduling. You want refreshes after ETL loads complete, not at random intervals. The standard approach combines Redshift stored procedures with external orchestration:

sql 
-- Stored procedure to refresh all MVs in dependency order
CREATE OR REPLACE PROCEDURE analytics.sp_refresh_aggregations()
AS $$
BEGIN
    -- Layer 1: Hourly MVs (depend on base tables)
    REFRESH MATERIALIZED VIEW analytics.mv_hourly_campaign;
    REFRESH MATERIALIZED VIEW analytics.mv_hourly_geo;
    REFRESH MATERIALIZED VIEW analytics.mv_hourly_device;

    -- Layer 2: Daily roll-ups (depend on hourly MVs)
    REFRESH MATERIALIZED VIEW analytics.mv_daily_campaign_rollup;
    REFRESH MATERIALIZED VIEW analytics.mv_daily_geo_rollup;

    -- Log completion
    INSERT INTO analytics.refresh_log (procedure_name, completed_at)
    VALUES ('sp_refresh_aggregations', GETDATE());
END;
$$ LANGUAGE plpgsql;

-- Monitor staleness of all MVs
SELECT
    i.mv_name,
    i.state,
    DATEDIFF('minute',
      (SELECT MAX(end_time)
       FROM svl_mv_refresh_status s
       WHERE s.mv_name = i.mv_name
         AND s.status = 'Complete'),
      GETDATE()
    ) AS minutes_since_refresh
FROM svv_mv_info i
WHERE schema_name = 'analytics'
ORDER BY minutes_since_refresh DESC;
Important

Never run REFRESH MATERIALIZED VIEW inside a BEGIN...COMMIT transaction block. MV refresh manages its own transactions internally. Wrapping it causes an error. Each refresh statement must be its own autonomous operation within the stored procedure.

Self-Check

Before moving to Chapter 05, honestly answer these questions. Revisit any section where the answer is unclear.

What is the difference between a regular view and a materialized view in Redshift?

A regular view is a stored SQL definition re-executed on every access. A materialized view precomputes and stores the result physically on disk. Reads hit the stored data, not the base tables. The trade-off is freshness: MVs can become stale between refreshes.

When does Redshift use incremental refresh vs. full recompute?

Incremental refresh is used when the MV uses only INNER JOINs, supported aggregates, no subqueries, and no DDL has been run on base tables since last refresh. Otherwise Redshift falls back to full recompute. Check svl_mv_refresh_status.refresh_type to see which was used.

Why can you not average the AVG column from a lower-layer MV?

Averaging averages is mathematically incorrect because each average represents a different number of observations. If hour 1 has 1000 impressions at avg $2 CPM and hour 2 has 10 impressions at avg $10 CPM, the correct daily average is (2000+100)/1010 = $2.08, not ($2+$10)/2 = $6. Always store SUM and COUNT separately.

How does automatic query rewrite work?

The optimizer transparently redirects a query against a base table to read from a matching MV. The MV must be recently refreshed and the query must be a logical subset of the MV definition. Analysts never need to know MVs exist. Verify with EXPLAIN output.

What is the idempotent upsert pattern and why is it important?

It deletes all data in a target time partition, then re-inserts the complete dataset. Running it multiple times yields the same result. Critical for late-arriving data because the pipeline simply reruns the affected partition, incorporating both original and late events without duplicates.
