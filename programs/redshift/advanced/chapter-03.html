Chapter 03 · Redshift Advanced

Multi-Dimensional Fact Tables

Master dimensional modeling in Redshift — star schemas, fact table types (transaction, periodic snapshot, accumulating snapshot), dimension tables, slowly changing dimensions (SCD), and constellation schemas for complex analytics workloads.

Estimated: 60–80 min
5 Key Concepts
3 Assignments

Why This Matters

A data warehouse is only as useful as its data model. Without a coherent dimensional model, analysts spend more time figuring out how to join tables than answering business questions. Dimensional modeling — pioneered by Ralph Kimball — organizes data into fact tables (measurements and metrics) surrounded by dimension tables (descriptive context). This approach makes queries intuitive, enables fast aggregations, and maps directly to how business users think about their data.

In Redshift, dimensional modeling is not just a design philosophy — it directly impacts query performance. The choice of distribution keys, sort keys, and table structure must align with your dimensional model. A well-designed star schema in Redshift can deliver sub-second query times on billions of rows. A poorly designed one forces expensive redistribution and cross-node shuffles on every query.

Key Insight

Dimensional modeling is not about normalization — it is about query usability. In a star schema, dimension tables are intentionally denormalized so that every query against the fact table needs at most one join per dimension. This trades storage efficiency for query simplicity and performance, which is exactly the right trade-off for an analytical warehouse.

Key Concepts

1. Star Schema Fundamentals

A star schema places a central fact table at its core, surrounded by dimension tables that radiate outward like points of a star. The fact table contains the measurable, numeric data (impressions, clicks, revenue, bid prices) along with foreign keys to each dimension. The dimension tables provide the descriptive attributes (campaign name, advertiser, device type, geography) that give context to those measurements.

sql 
-- Central fact table: one row per ad impression event
CREATE TABLE analytics.fact_impressions (
    impression_id       BIGINT        IDENTITY(1,1),
    date_key            INTEGER       NOT NULL,
    campaign_key        INTEGER       NOT NULL,
    creative_key        INTEGER       NOT NULL,
    publisher_key       INTEGER       NOT NULL,
    geo_key             INTEGER       NOT NULL,
    device_key          INTEGER       NOT NULL,
    -- Measures
    impressions         INTEGER       NOT NULL DEFAULT 1,
    clicks              INTEGER       NOT NULL DEFAULT 0,
    conversions         INTEGER       NOT NULL DEFAULT 0,
    bid_price_cpm       DECIMAL(10,4),
    win_price_cpm       DECIMAL(10,4),
    revenue             DECIMAL(12,4)
)
DISTSTYLE KEY
DISTKEY (campaign_key)
SORTKEY (date_key, campaign_key);

-- Dimension: Date (distributed to ALL nodes for local joins)
CREATE TABLE analytics.dim_date (
    date_key            INTEGER       PRIMARY KEY,
    full_date           DATE          NOT NULL,
    day_of_week         VARCHAR(10),
    day_of_month        INTEGER,
    week_of_year        INTEGER,
    month_number        INTEGER,
    month_name          VARCHAR(15),
    quarter             INTEGER,
    year                INTEGER,
    is_weekend          BOOLEAN,
    is_holiday          BOOLEAN,
    fiscal_quarter      VARCHAR(6)
)
DISTSTYLE ALL;

2. Fact Table Types

Not all fact tables record the same kind of information. Kimball identifies three fundamental types, each suited to different analytical questions. Understanding which type to use for each business process is one of the most important modeling decisions you will make.

Transaction fact tables record one row per discrete event. An ad impression, a click, a bid request — each event generates exactly one row. These tables grow the fastest but support the most granular analysis. They are the default starting point for most data warehouses.

Periodic snapshot fact tables capture the state of a metric at regular intervals. Instead of recording every individual impression, you record the total impressions per campaign per day. These are smaller than transaction tables and ideal for trend analysis and dashboards that show daily, weekly, or monthly aggregates.

Accumulating snapshot fact tables track the lifecycle of a process that has a defined beginning and end. In ad-tech, this could track a conversion funnel: impression date, click date, landing page visit date, and conversion date. Each row is updated as the entity progresses through pipeline stages.

sql 
-- Periodic snapshot: daily campaign performance
CREATE TABLE analytics.fact_daily_campaign_snapshot (
    snapshot_date_key   INTEGER       NOT NULL,
    campaign_key        INTEGER       NOT NULL,
    total_impressions   BIGINT,
    total_clicks        BIGINT,
    total_conversions   INTEGER,
    total_spend         DECIMAL(14,4),
    avg_cpm             DECIMAL(10,4),
    avg_ctr             DECIMAL(8,6),
    remaining_budget    DECIMAL(14,4),
    PRIMARY KEY (snapshot_date_key, campaign_key)
)
DISTSTYLE KEY
DISTKEY (campaign_key)
SORTKEY (snapshot_date_key);

-- Accumulating snapshot: conversion funnel tracking
CREATE TABLE analytics.fact_conversion_funnel (
    funnel_id           BIGINT        IDENTITY(1,1),
    user_key            INTEGER       NOT NULL,
    campaign_key        INTEGER       NOT NULL,
    impression_date_key INTEGER,
    click_date_key      INTEGER,
    visit_date_key      INTEGER,
    conversion_date_key INTEGER,
    days_to_click       INTEGER,
    days_to_convert     INTEGER,
    conversion_revenue  DECIMAL(12,4),
    funnel_status       VARCHAR(20)
)
DISTSTYLE KEY
DISTKEY (campaign_key)
SORTKEY (impression_date_key);

3. Dimension Tables and DISTSTYLE ALL

Dimension tables are typically small relative to fact tables — a campaign dimension might have 50,000 rows while the impression fact has 50 billion. In Redshift, you should distribute small dimension tables using DISTSTYLE ALL, which places a complete copy of the table on every compute node. This eliminates network redistribution when joining the fact table to the dimension — every slice can perform the join locally.

sql 
-- Campaign dimension with denormalized attributes
CREATE TABLE analytics.dim_campaign (
    campaign_key        INTEGER       PRIMARY KEY,
    campaign_id         INTEGER       NOT NULL,
    campaign_name       VARCHAR(256)  NOT NULL,
    advertiser_name     VARCHAR(256),
    advertiser_industry VARCHAR(100),
    campaign_type       VARCHAR(50),
    campaign_objective  VARCHAR(50),
    start_date          DATE,
    end_date            DATE,
    total_budget        DECIMAL(14,4),
    daily_budget_cap    DECIMAL(10,4),
    target_audience     VARCHAR(500),
    is_active           BOOLEAN,
    -- SCD Type 2 metadata
    effective_date      DATE          NOT NULL,
    expiration_date     DATE          NOT NULL DEFAULT '9999-12-31',
    is_current          BOOLEAN       NOT NULL DEFAULT TRUE
)
DISTSTYLE ALL
SORTKEY (campaign_key);

-- Geography dimension
CREATE TABLE analytics.dim_geo (
    geo_key             INTEGER       PRIMARY KEY,
    country_code        CHAR(2),
    country_name        VARCHAR(100),
    region              VARCHAR(100),
    city                VARCHAR(100),
    dma_code            VARCHAR(10),
    dma_name            VARCHAR(100),
    timezone            VARCHAR(50)
)
DISTSTYLE ALL;
Important

Only use DISTSTYLE ALL for tables under approximately 3 million rows. Larger tables consume significant memory on every node and slow down writes because every insert, update, or delete must be replicated to all nodes. If a dimension exceeds this threshold, use DISTSTYLE KEY matching the fact table's distribution key for co-located joins.

4. Slowly Changing Dimensions (SCD)

Dimension attributes change over time. A campaign's budget gets adjusted mid-flight. An advertiser rebrands. A publisher changes its content category. Slowly Changing Dimensions (SCD) define how your warehouse handles these changes. The three most common types are:

  • SCD Type 1 (Overwrite) — Simply update the dimension record in place. You lose the history but always have the current value. Use this for corrections (fixing a typo in a campaign name).
  • SCD Type 2 (Add New Row) — Insert a new row with the updated values and mark the old row as expired. Preserves full history. Use this for attributes where historical accuracy matters (budget changes that affect past reporting).
  • SCD Type 3 (Add Column) — Add a "previous value" column alongside the current value. Preserves exactly one level of history. Rarely used in practice because it only tracks one change.
sql 
-- SCD Type 2: expire old record and insert new one
CREATE OR REPLACE PROCEDURE etl.apply_scd2_campaign(
    p_campaign_id    INTEGER,
    p_campaign_name  VARCHAR(256),
    p_budget         DECIMAL(14,4),
    p_effective_date DATE
)
AS $$
DECLARE
    v_max_key INTEGER;
BEGIN
    -- Step 1: Expire the current record
    UPDATE analytics.dim_campaign
    SET
        expiration_date = p_effective_date - 1,
        is_current = FALSE
    WHERE campaign_id = p_campaign_id
      AND is_current = TRUE;

    -- Step 2: Generate next surrogate key
    SELECT COALESCE(MAX(campaign_key), 0) + 1
    INTO v_max_key
    FROM analytics.dim_campaign;

    -- Step 3: Insert new version
    INSERT INTO analytics.dim_campaign
        (campaign_key, campaign_id, campaign_name, total_budget,
         effective_date, expiration_date, is_current)
    VALUES
        (v_max_key, p_campaign_id, p_campaign_name, p_budget,
         p_effective_date, '9999-12-31', TRUE);
END;
$$ LANGUAGE plpgsql;
Pro Tip

When querying SCD Type 2 dimensions, always filter by is_current = TRUE for current-state queries, or join on the date range (effective_date <= event_date AND expiration_date >= event_date) for historical accuracy. Create a view that defaults to the current version so analysts do not accidentally get duplicated results from historical rows.

5. Constellation Schema

A constellation schema (also called a galaxy schema) extends the star schema by sharing dimension tables across multiple fact tables. In ad-tech, you might have separate fact tables for impressions, clicks, and conversions, but they all share the same campaign, publisher, and date dimensions. This avoids dimension duplication and ensures consistent definitions across all metrics.

sql 
-- Constellation schema: multiple fact tables sharing dimensions

-- Fact: Clicks (shares dim_campaign, dim_date, dim_publisher)
CREATE TABLE analytics.fact_clicks (
    click_id            BIGINT        IDENTITY(1,1),
    date_key            INTEGER       NOT NULL,
    campaign_key        INTEGER       NOT NULL,
    creative_key        INTEGER       NOT NULL,
    publisher_key       INTEGER       NOT NULL,
    impression_id       BIGINT,
    click_timestamp     TIMESTAMP,
    landing_page_url    VARCHAR(2000),
    cost_per_click      DECIMAL(10,4)
)
DISTSTYLE KEY
DISTKEY (campaign_key)
SORTKEY (date_key, campaign_key);

-- Fact: Conversions (shares dim_campaign, dim_date)
CREATE TABLE analytics.fact_conversions (
    conversion_id       BIGINT        IDENTITY(1,1),
    date_key            INTEGER       NOT NULL,
    campaign_key        INTEGER       NOT NULL,
    click_id            BIGINT,
    conversion_type     VARCHAR(50),
    conversion_value    DECIMAL(12,4),
    attribution_model   VARCHAR(30)
)
DISTSTYLE KEY
DISTKEY (campaign_key)
SORTKEY (date_key);

-- Query across constellation: full funnel analysis
SELECT
    dd.full_date,
    dc.campaign_name,
    dc.advertiser_name,
    SUM(fi.impressions) AS impressions,
    COUNT(DISTINCT fcl.click_id) AS clicks,
    COUNT(DISTINCT fcv.conversion_id) AS conversions,
    SUM(fcv.conversion_value) AS total_revenue
FROM analytics.fact_impressions fi
JOIN analytics.dim_date dd ON fi.date_key = dd.date_key
JOIN analytics.dim_campaign dc ON fi.campaign_key = dc.campaign_key
LEFT JOIN analytics.fact_clicks fcl
    ON fi.impression_id = fcl.impression_id
LEFT JOIN analytics.fact_conversions fcv
    ON fcl.click_id = fcv.click_id
WHERE dd.full_date BETWEEN '2025-06-01' AND '2025-06-30'
  AND dc.is_current = TRUE
GROUP BY dd.full_date, dc.campaign_name, dc.advertiser_name
ORDER BY dd.full_date, impressions DESC;

Self-Check

Before moving to Chapter 04, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What is the difference between a transaction fact table and a periodic snapshot fact table?

A transaction fact table records one row per event (impression, click, purchase). It captures the finest grain of detail and grows rapidly. A periodic snapshot records the state of a metric at regular intervals (daily totals, weekly averages). Snapshots are smaller and faster for trend analysis but lose event-level granularity. Use transaction facts for drill-down analysis and periodic snapshots for dashboards.

Why should dimension tables use DISTSTYLE ALL in Redshift?

DISTSTYLE ALL places a complete copy of the dimension table on every compute node. This means every slice can join locally to the fact table without network redistribution. Since dimension tables are typically small (thousands to low millions of rows), the storage overhead is minimal but the query performance improvement is significant — especially for star schema queries that join one fact to multiple dimensions.

When would you use SCD Type 2 versus SCD Type 1?

Use SCD Type 2 when you need to preserve the historical value at the time an event occurred. For example, if a campaign's budget changed mid-month, you want impressions before the change to be associated with the old budget and impressions after with the new budget. Use SCD Type 1 when the change is a correction (fixing a typo) and you want all historical facts to reflect the corrected value.

What is a constellation schema and when is it preferable to a single star schema?

A constellation schema uses multiple fact tables that share common dimension tables. It is preferable when your warehouse models multiple business processes (impressions, clicks, conversions) that share the same dimensions (campaigns, dates, publishers) but measure different things. Sharing dimensions ensures consistent definitions and enables cross-process analytics like full-funnel analysis.
