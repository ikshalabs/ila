Chapter 09 · Redshift Advanced

Monitoring, Alerts, and Cost Control

Build a comprehensive observability layer for your Redshift cluster — from CloudWatch metrics and SNS alerts, through system table audit queries, Redshift Advisor recommendations, to cost optimization strategies including Reserved Instance planning, pause/resume scheduling, and Spectrum cost governance.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

Running a Redshift cluster without monitoring is like driving at night without headlights. You might make it to your destination, but you will not see problems until they hit you. In production environments, Redshift clusters can cost $10,000–$100,000+ per month. Without active cost governance, idle clusters burn budget 24/7, Spectrum scans over unpartitioned data lakes cost 10x what they should, and concurrency scaling charges silently accumulate. Meanwhile, without alerting, ETL failures go unnoticed for hours, disk space fills up silently, and query queues back up during peak hours without anyone knowing.

This chapter covers the complete observability and cost governance stack for Redshift: CloudWatch metrics for infrastructure-level monitoring, system table queries for query-level diagnostics, SNS-based alerting for automated incident detection, Redshift Advisor for optimization recommendations, and cost control strategies including Reserved Instances, pause/resume scheduling, and Spectrum cost management.

Key Insight

The most expensive Redshift problem is not a failed query — it is an idle cluster. A dc2.8xlarge 4-node cluster costs approximately $19.20 per hour ($14,000 per month) whether it is processing queries or sitting idle. If your development or staging cluster runs 24/7 but is only used during business hours (10 hours/day, 5 days/week), you are wasting 70% of your budget. Pause/resume scheduling alone can cut costs by $10,000/month in this scenario.

Key Concepts

1. CloudWatch Metrics and Dashboards

Amazon Redshift automatically publishes metrics to CloudWatch every minute. These infrastructure-level metrics cover CPU utilization, disk space usage, network throughput, read/write IOPS, database connections, query throughput, and WLM queue metrics. CloudWatch is your first line of defense for detecting cluster-level problems: disk running full, CPU pegged at 100%, or connection limits exhausted.

sql 
-- Key CloudWatch metrics to monitor for Redshift:
--
-- CPUUtilization           Average CPU across all nodes (alert > 80%)
-- PercentageDiskSpaceUsed  Disk usage across cluster (alert > 75%)
-- DatabaseConnections       Active connections (alert > 80% of limit)
-- ReadIOPS / WriteIOPS      I/O operations per second
-- ReadThroughput            MB/s read from disk
-- WriteThroughput           MB/s written to disk
-- QueriesCompletedPerSecond Throughput metric
-- QueryDuration             Average query execution time
-- WLMQueueLength            Queries waiting in WLM queues
-- WLMRunningQueries         Currently executing queries
-- MaintenanceMode           1 = cluster is in maintenance
-- HealthStatus              1 = healthy, 0 = unhealthy

-- Check current disk usage from within the cluster
SELECT
    owner AS node,
    diskno,
    ROUND(used::FLOAT / capacity * 100, 1) AS pct_used,
    used AS used_mb,
    capacity AS total_mb
FROM stv_partitions
WHERE type = 0  -- data partitions only
ORDER BY pct_used DESC;

-- Check connection count
SELECT
    COUNT(*) AS active_connections,
    MAX(TRIM(recordtime)) AS latest_connection
FROM stl_connection_log
WHERE event = 'initiating session'
  AND recordtime > GETDATE() - INTERVAL '5 minutes';

-- Table-level storage audit: which tables consume the most space?
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    info.tbl_rows,
    info.size AS size_mb,
    info.pct_used,
    info.unsorted,
    info.stats_off
FROM svv_table_info info
JOIN pg_class pgc ON info.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE pgn.nspname NOT IN ('pg_catalog', 'information_schema')
ORDER BY info.size DESC
LIMIT 30;

2. SNS Alerting and Event Notifications

CloudWatch Alarms combined with Amazon SNS provide automated alerting when metrics cross defined thresholds. You should configure alerts for disk space exhaustion (critical — cluster becomes read-only at 100%), high CPU utilization (sustained above 80%), WLM queue backups (queries waiting more than 60 seconds), and cluster health status changes.

In addition to CloudWatch Alarms, Redshift generates event notifications for operational events: cluster maintenance, snapshot creation, parameter group changes, and security group modifications. Subscribe to these events via SNS to stay informed about infrastructure changes that could affect your workloads.

bash 
# Create SNS topic for Redshift alerts
aws sns create-topic --name redshift-production-alerts

# Subscribe email to the topic
aws sns subscribe \
  --topic-arn arn:aws:sns:us-east-1:123456789012:redshift-production-alerts \
  --protocol email \
  --notification-endpoint ops-team@company.com

# Create CloudWatch alarm for high disk usage (>75%)
aws cloudwatch put-metric-alarm \
  --alarm-name redshift-disk-usage-high \
  --alarm-description "Redshift disk usage above 75%" \
  --metric-name PercentageDiskSpaceUsed \
  --namespace AWS/Redshift \
  --dimensions Name=ClusterIdentifier,Value=my-prod-cluster \
  --statistic Average \
  --period 300 \
  --threshold 75 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 2 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:redshift-production-alerts

# Create alarm for sustained high CPU (>80% for 15 minutes)
aws cloudwatch put-metric-alarm \
  --alarm-name redshift-cpu-high \
  --alarm-description "Redshift CPU above 80% for 15 minutes" \
  --metric-name CPUUtilization \
  --namespace AWS/Redshift \
  --dimensions Name=ClusterIdentifier,Value=my-prod-cluster \
  --statistic Average \
  --period 300 \
  --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 3 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:redshift-production-alerts

# Create alarm for WLM queue backup
aws cloudwatch put-metric-alarm \
  --alarm-name redshift-wlm-queue-backup \
  --alarm-description "WLM queue length exceeds 10" \
  --metric-name WLMQueueLength \
  --namespace AWS/Redshift \
  --dimensions Name=ClusterIdentifier,Value=my-prod-cluster \
  --statistic Maximum \
  --period 60 \
  --threshold 10 \
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods 5 \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:redshift-production-alerts

# Subscribe to Redshift event notifications
aws redshift create-event-subscription \
  --subscription-name prod-cluster-events \
  --sns-topic-arn arn:aws:sns:us-east-1:123456789012:redshift-production-alerts \
  --source-type cluster \
  --source-ids my-prod-cluster \
  --event-categories "configuration" "management" "security"
Common Pitfall

The PercentageDiskSpaceUsed alarm is critical. When a Redshift cluster reaches 100% disk utilization, it becomes read-only. All write operations (INSERT, COPY, CREATE TABLE) will fail, and your ETL pipelines will break. Set your alert threshold at 75% to give yourself time to respond. At 90%, treat it as a critical incident and immediately drop temp tables, run VACUUM DELETE ONLY, or resize the cluster.

3. System Table Audit Queries

Beyond CloudWatch metrics, Redshift's system tables (STL_, STV_, SVL_, SVV_) provide granular query-level telemetry. Building a regular audit cadence with system table queries is essential for proactive performance management. Key audit queries should cover: long-running query identification, aborted query tracking, storage growth trends, and user activity patterns.

sql 
-- Daily Audit Query 1: Top resource-consuming queries
SELECT
    q.query,
    TRIM(u.usename) AS username,
    SUBSTRING(TRIM(q.querytxt), 1, 100) AS query_text,
    q.starttime,
    DATEDIFF(ms, q.starttime, q.endtime) AS duration_ms,
    qs.scan_row_count,
    qs.join_row_count,
    qs.return_row_count,
    qs.query_temp_blocks_to_disk
FROM stl_query q
JOIN pg_user u ON q.userid = u.usesysid
LEFT JOIN svl_query_metrics_summary qs ON q.query = qs.query
WHERE q.starttime > GETDATE() - INTERVAL '24 hours'
  AND q.userid > 1
ORDER BY duration_ms DESC
LIMIT 25;

-- Daily Audit Query 2: Aborted and failed queries
SELECT
    q.query,
    TRIM(u.usename) AS username,
    SUBSTRING(TRIM(q.querytxt), 1, 200) AS query_text,
    q.starttime,
    q.aborted,
    TRIM(e.err_reason) AS error_reason
FROM stl_query q
JOIN pg_user u ON q.userid = u.usesysid
LEFT JOIN stl_error e ON q.query = e.query
WHERE q.starttime > GETDATE() - INTERVAL '24 hours'
  AND (q.aborted = 1 OR e.query IS NOT NULL)
ORDER BY q.starttime DESC;

-- Daily Audit Query 3: Storage growth trend
SELECT
    DATE_TRUNC('day', starttime) AS day,
    SUM(CASE WHEN TRIM(querytxt) LIKE '%COPY%' THEN 1 ELSE 0 END) AS copy_operations,
    SUM(CASE WHEN TRIM(querytxt) LIKE '%INSERT%' THEN 1 ELSE 0 END) AS insert_operations,
    SUM(CASE WHEN TRIM(querytxt) LIKE '%DELETE%' THEN 1 ELSE 0 END) AS delete_operations
FROM stl_query
WHERE starttime > GETDATE() - INTERVAL '30 days'
  AND userid > 1
GROUP BY DATE_TRUNC('day', starttime)
ORDER BY day DESC;

-- Daily Audit Query 4: User activity summary
SELECT
    TRIM(u.usename) AS username,
    COUNT(*) AS total_queries,
    ROUND(AVG(DATEDIFF(ms, q.starttime, q.endtime)), 0) AS avg_duration_ms,
    MAX(DATEDIFF(ms, q.starttime, q.endtime)) AS max_duration_ms,
    SUM(CASE WHEN q.aborted = 1 THEN 1 ELSE 0 END) AS aborted_queries
FROM stl_query q
JOIN pg_user u ON q.userid = u.usesysid
WHERE q.starttime > GETDATE() - INTERVAL '24 hours'
  AND q.userid > 1
GROUP BY TRIM(u.usename)
ORDER BY total_queries DESC;

4. Redshift Advisor Recommendations

Redshift Advisor is an automated optimization engine that analyzes your cluster's workload and provides specific, actionable recommendations. It identifies tables that would benefit from distribution key changes, sort key modifications, compression encoding updates, and VACUUM operations. Advisor recommendations are based on actual query patterns, not theoretical best practices, making them highly reliable.

sql 
-- View all current Advisor recommendations
SELECT
    TRIM(feature_name) AS recommendation_type,
    TRIM(condition) AS condition,
    TRIM(action) AS recommended_action,
    ranking
FROM svv_advisor_recommendations
ORDER BY ranking;

-- Common Advisor recommendation types:
--   "DISTRIBUTION KEY"    Table would benefit from a different dist key
--   "SORT KEY"            Table needs sort key for better zone map usage
--   "COMPRESSION"         Column encoding is suboptimal
--   "VACUUM"              Table has large unsorted or deleted regions
--   "ANALYZE"             Table statistics are stale
--   "TABLE DESIGN"        Schema design improvements

-- Check table-specific recommendations
SELECT
    TRIM(feature_name) AS feature,
    TRIM(condition) AS issue,
    TRIM(action) AS fix
FROM svv_advisor_recommendations
WHERE condition LIKE '%fact_impressions%';

-- Automate Advisor checks: create a stored procedure
CREATE OR REPLACE PROCEDURE monitoring.check_advisor()
AS $$
DECLARE
    rec_count INTEGER;
BEGIN
    SELECT COUNT(*) INTO rec_count
    FROM svv_advisor_recommendations;

    IF rec_count > 0 THEN
        INSERT INTO monitoring.advisor_log (check_time, recommendation_count)
        VALUES (GETDATE(), rec_count);

        INSERT INTO monitoring.advisor_details (check_time, feature, condition, action)
        SELECT GETDATE(), TRIM(feature_name), TRIM(condition), TRIM(action)
        FROM svv_advisor_recommendations;

        RAISE INFO 'Advisor found % recommendations', rec_count;
    END IF;
END;
$$ LANGUAGE plpgsql;

5. Cost Optimization Strategies

Redshift cost optimization operates at three levels: compute (cluster node type and count, Reserved Instances, pause/resume), storage (compression, data retention, archival to S3), and Spectrum (data scanned per query, partitioning, file format). A disciplined cost optimization program can reduce your Redshift bill by 40–70% without sacrificing performance.

bash 
# --- COMPUTE COST OPTIMIZATION ---

# Schedule cluster pause for non-production (saves 100% during pause)
# Use EventBridge (CloudWatch Events) + Lambda for scheduling

# Pause the dev cluster at 8 PM
aws redshift pause-cluster --cluster-identifier my-dev-cluster

# Resume the dev cluster at 7 AM
aws redshift resume-cluster --cluster-identifier my-dev-cluster

# Check Reserved Instance pricing (1-year, no upfront)
aws redshift describe-reserved-node-offerings \
  --node-type dc2.8xlarge \
  --duration 31536000

# Purchase Reserved Instances (saves ~30-40% vs on-demand)
aws redshift purchase-reserved-node-offering \
  --reserved-node-offering-id offering-id-here \
  --node-count 4

# --- STORAGE COST OPTIMIZATION ---

# Check which tables have the most dead rows (wasting space)
# Run this query inside Redshift:
sql 
-- Identify tables with high dead-row ratios (need VACUUM)
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    info.tbl_rows AS live_rows,
    info.size AS size_mb,
    info.empty AS empty_blocks_pct,
    info.unsorted AS unsorted_pct
FROM svv_table_info info
JOIN pg_class pgc ON info.table_id = pgc.oid
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
WHERE pgn.nspname NOT IN ('pg_catalog', 'information_schema')
  AND info.empty > 20   -- more than 20% empty blocks
ORDER BY info.size DESC;

-- Data retention: archive old partitions to S3 before deleting
UNLOAD ('
    SELECT * FROM analytics.fact_impressions
    WHERE event_date < ''''2024-01-01''''
')
TO 's3://my-archive-bucket/impressions/archive/pre-2024/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
FORMAT AS PARQUET
PARTITION BY (event_date)
ALLOWOVERWRITE;

-- Then delete archived data from Redshift
DELETE FROM analytics.fact_impressions
WHERE event_date < '2024-01-01';

VACUUM DELETE ONLY analytics.fact_impressions;

-- Query archived data via Spectrum without loading it back
SELECT event_date, COUNT(*) AS impressions
FROM spectrum_schema.fact_impressions_archive
WHERE event_date BETWEEN '2023-06-01' AND '2023-06-30'
GROUP BY event_date;
Pro Tip

The archive-then-delete pattern is one of the most powerful cost reduction strategies. Older data that is rarely queried but must be retained for compliance can live in S3 at approximately $0.023/GB/month (versus Redshift's effective storage cost of $0.25/GB/month with RA3 nodes). Use Redshift Spectrum to query archived data directly from S3 when needed, without re-loading it into the cluster.

6. Spectrum Cost Governance

Redshift Spectrum charges $5 per terabyte of data scanned from S3. Without proper governance, Spectrum queries over unpartitioned or uncompressed data lakes can generate surprisingly high bills. A single SELECT * over a 10 TB unpartitioned dataset costs $50 per execution. Running that query 20 times a day costs $1,000/day or $30,000/month.

The key cost control levers for Spectrum are: partitioning (so queries scan only relevant date/region partitions), columnar file formats (Parquet or ORC, so only needed columns are read), and file size optimization (128 MB–512 MB files for optimal scan parallelism).

sql 
-- Monitor Spectrum scan costs over the last 7 days
SELECT
    DATE_TRUNC('day', starttime) AS day,
    COUNT(*) AS spectrum_queries,
    SUM(s3_scanned_bytes) / 1099511627776.0 AS tb_scanned,
    ROUND(SUM(s3_scanned_bytes) / 1099511627776.0 * 5.0, 2) AS estimated_cost_usd
FROM svl_s3query_summary
WHERE starttime > GETDATE() - INTERVAL '7 days'
GROUP BY DATE_TRUNC('day', starttime)
ORDER BY day DESC;

-- Identify the most expensive Spectrum queries
SELECT
    query,
    TRIM(querytxt) AS query_text,
    s3_scanned_rows,
    s3_scanned_bytes / 1073741824.0 AS gb_scanned,
    ROUND(s3_scanned_bytes / 1099511627776.0 * 5.0, 4) AS estimated_cost_usd,
    s3query_returned_rows,
    CASE WHEN s3_scanned_rows > 0
         THEN ROUND(100.0 * s3query_returned_rows / s3_scanned_rows, 1)
         ELSE 0
    END AS selectivity_pct  -- low % = scanning too much data
FROM svl_s3query_summary
WHERE starttime > GETDATE() - INTERVAL '24 hours'
ORDER BY s3_scanned_bytes DESC
LIMIT 20;

-- Check partition pruning effectiveness
SELECT
    query,
    segment,
    assignment,
    TRIM(external_table_name) AS table_name,
    s3_scanned_rows,
    s3query_returned_rows,
    files,
    files_qualified     -- files after partition pruning
FROM svl_s3query
WHERE starttime > GETDATE() - INTERVAL '1 hour'
  AND files > files_qualified  -- partitions were pruned
ORDER BY starttime DESC;

-- Cost reduction: ensure data lake tables are partitioned and columnar
-- BAD: 10 TB of CSV in one S3 prefix -> $50 per full scan
-- GOOD: 10 TB of Parquet, partitioned by event_date
--   Query on 1 day = ~30 GB scanned = $0.15 per query
--   That is a 333x cost reduction!

Self-Check

Before moving to Chapter 10, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What happens when a Redshift cluster reaches 100% disk utilization, and how do you prevent it?

At 100% disk utilization, the cluster becomes read-only. All write operations (INSERT, COPY, CREATE TABLE) fail. The cluster may also become unresponsive for reads. To prevent this: (1) Set a CloudWatch alarm at 75% disk usage to give early warning. (2) Implement data retention policies that archive old data to S3 and delete it from Redshift. (3) Run VACUUM DELETE ONLY regularly to reclaim space from deleted rows. (4) Monitor table growth trends weekly and resize the cluster proactively before hitting capacity.

How do you calculate the cost of a Redshift Spectrum query before running it?

Spectrum charges $5 per TB of data scanned from S3. To estimate: (1) Determine the total data size in S3 for the external table. (2) If the table is partitioned and your query includes a partition filter (e.g., WHERE event_date = '2025-01-15'), only the matching partition is scanned. (3) If using a columnar format (Parquet/ORC) and your query selects specific columns, only those columns are read (column pruning). Multiply the estimated scan size by $5/TB. You can verify the actual cost after execution by querying svl_s3query_summary for the s3_scanned_bytes column.

What is the archive-then-delete pattern, and why is it cost-effective?

The pattern involves: (1) UNLOAD old data to S3 as Parquet with partitioning. (2) DELETE the same data from Redshift. (3) VACUUM DELETE ONLY to reclaim disk space. (4) Create an external table via Spectrum to query the archived data. This is cost-effective because S3 storage costs approximately $0.023/GB/month versus Redshift's effective storage cost of $0.10–$0.25/GB/month depending on node type. Archived data remains queryable via Spectrum at $5/TB scanned, which is economical for infrequent queries on historical data.

What are the key CloudWatch metrics to monitor, and what alarm thresholds should you set?

The essential metrics and recommended thresholds are: PercentageDiskSpaceUsed (alert at 75%, critical at 90%); CPUUtilization (alert at 80% sustained for 15 minutes); DatabaseConnections (alert at 80% of the cluster's connection limit); WLMQueueLength (alert when queue length exceeds 10 for 5+ minutes); HealthStatus (alert on any value other than 1/healthy). Additionally, monitor MaintenanceMode to know when the cluster is in maintenance and temporarily unavailable for writes.

How much can pause/resume scheduling save, and which clusters should use it?

A cluster paused during non-business hours (14 hours/day weekdays + all weekend = ~118 hours/week out of 168) saves approximately 70% of compute costs. For a 4-node dc2.8xlarge cluster at ~$19.20/hour, that saves ~$9,800/month. Pause/resume is appropriate for development, staging, QA, and test clusters. Production clusters typically should not be paused unless the business operates on a strict schedule. Note: you still pay for storage when paused (snapshot storage), but compute charges stop completely.
