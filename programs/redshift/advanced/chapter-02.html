Chapter 02 · Redshift Advanced

Stored Procedures for ETL

Learn to encapsulate complex ETL logic inside Redshift stored procedures using PL/pgSQL — including loop constructs, error handling with EXCEPTION blocks, transaction control, and production-grade orchestration patterns for reliable data pipelines.

Estimated: 55–75 min
6 Key Concepts
3 Assignments

Why This Matters

ETL pipelines in a modern data warehouse are rarely a single SQL statement. They involve staging raw data, validating it, transforming it across multiple tables, handling late-arriving facts, maintaining audit logs, and cleaning up temporary artifacts. Without stored procedures, teams scatter this logic across external orchestration tools, Python scripts, and ad-hoc SQL files — creating maintenance nightmares where business logic lives in three different places.

Amazon Redshift supports PL/pgSQL stored procedures that let you encapsulate multi-step ETL workflows directly inside the database engine. This means your transformation logic runs where the data lives, avoiding costly data movement. Stored procedures support variables, conditionals, loops, cursor-based iteration, explicit transaction control, and structured error handling — everything you need for production-grade pipelines.

Key Insight

Stored procedures in Redshift run as atomic units of work by default. If any statement fails mid-procedure, the entire transaction is rolled back automatically unless you have explicitly committed intermediate work. This behavior is fundamental to building ETL pipelines that never leave data in a half-transformed state.

Key Concepts

1. CREATE PROCEDURE Syntax

The CREATE OR REPLACE PROCEDURE statement defines a reusable block of PL/pgSQL code. Unlike functions, procedures do not return a value through a RETURN statement — they perform actions. You invoke them with the CALL statement. Procedures can accept input parameters, output parameters, or both via INOUT parameters.

sql 
-- Basic stored procedure skeleton
CREATE OR REPLACE PROCEDURE etl.load_daily_impressions(
    p_load_date DATE,
    p_source_schema VARCHAR(128) DEFAULT 'staging'
)
AS $$
DECLARE
    v_row_count   INTEGER;
    v_start_time  TIMESTAMP;
    v_procedure   VARCHAR(256) := 'etl.load_daily_impressions';
BEGIN
    v_start_time := GETDATE();

    -- Step 1: Delete existing data for idempotency
    DELETE FROM analytics.fact_impressions
    WHERE event_date = p_load_date;

    GET DIAGNOSTICS v_row_count = ROW_COUNT;
    RAISE INFO 'Deleted % existing rows for %', v_row_count, p_load_date;

    -- Step 2: Insert transformed data from staging
    INSERT INTO analytics.fact_impressions
    SELECT
        impression_id,
        event_timestamp,
        event_date,
        campaign_id,
        creative_id,
        bid_price_cpm,
        win_price_cpm
    FROM staging.raw_impressions
    WHERE event_date = p_load_date
      AND campaign_id IS NOT NULL;

    GET DIAGNOSTICS v_row_count = ROW_COUNT;
    RAISE INFO 'Inserted % rows for %', v_row_count, p_load_date;

END;
$$ LANGUAGE plpgsql;

-- Execute the procedure
CALL etl.load_daily_impressions('2025-06-15');

Notice the GET DIAGNOSTICS statement — this is how you capture the number of rows affected by the previous DML statement. The RAISE INFO statement sends informational messages to the client, which appear in your SQL client's messages pane and in the Redshift system log (STL_UTILITYTEXT).

2. PL/pgSQL in Redshift

Redshift's PL/pgSQL dialect is based on PostgreSQL's procedural language but has important differences. Understanding these differences prevents frustrating debugging sessions where code that works on PostgreSQL fails silently or behaves differently on Redshift.

  • No RETURN for procedures — Use INOUT parameters to pass values back to the caller. Functions use RETURN, but procedures use CALL and cannot appear in SELECT statements.
  • RECORD type is limited — You can declare RECORD variables, but they must be assigned from a SELECT INTO statement, not from a cursor FETCH into individual fields.
  • No cursors with SCROLL — Redshift cursors are forward-only. You cannot move backwards through a result set.
  • Dynamic SQL via EXECUTE — Use EXECUTE with string concatenation or format() to build and run SQL dynamically inside procedures.
  • Transaction boundaries — By default, the entire procedure runs inside a single transaction. Use COMMIT or ROLLBACK to create explicit transaction boundaries within the procedure.
sql 
-- Using INOUT parameter to return a result
CREATE OR REPLACE PROCEDURE etl.get_row_count(
    p_table_name VARCHAR(256),
    INOUT p_count BIGINT
)
AS $$
BEGIN
    EXECUTE 'SELECT COUNT(*) FROM ' || p_table_name
    INTO p_count;
END;
$$ LANGUAGE plpgsql;

-- Call it and capture the result
CALL etl.get_row_count('analytics.fact_impressions', NULL);

3. Loop Constructs

Loops are essential for iterating over date ranges, processing partitions, or batching large operations. Redshift PL/pgSQL supports FOR loops (numeric ranges), FOR...IN loops (query result iteration), WHILE loops, and simple LOOP with EXIT WHEN.

sql 
-- Backfill procedure: loop over a date range
CREATE OR REPLACE PROCEDURE etl.backfill_impressions(
    p_start_date DATE,
    p_end_date   DATE
)
AS $$
DECLARE
    v_current_date  DATE;
    v_row_count     INTEGER;
    v_total_rows    INTEGER := 0;
BEGIN
    v_current_date := p_start_date;

    WHILE v_current_date <= p_end_date LOOP
        -- Delete-then-insert for idempotent reload
        DELETE FROM analytics.fact_impressions
        WHERE event_date = v_current_date;

        INSERT INTO analytics.fact_impressions
        SELECT *
        FROM staging.raw_impressions
        WHERE event_date = v_current_date
          AND campaign_id IS NOT NULL;

        GET DIAGNOSTICS v_row_count = ROW_COUNT;
        v_total_rows := v_total_rows + v_row_count;

        RAISE INFO 'Loaded % rows for %', v_row_count, v_current_date;

        -- Commit each day separately so failures
        -- do not roll back the entire backfill
        COMMIT;

        v_current_date := v_current_date + 1;
    END LOOP;

    RAISE INFO 'Backfill complete: % total rows loaded', v_total_rows;
END;
$$ LANGUAGE plpgsql;

-- Backfill the first two weeks of June
CALL etl.backfill_impressions('2025-06-01', '2025-06-14');
Important

The COMMIT inside the loop creates a new transaction boundary for each iteration. If the procedure fails on day 8 of a 14-day backfill, the first 7 days remain committed. Without the intermediate COMMIT, a failure on day 8 would roll back all 7 previous days too.

4. Error Handling with EXCEPTION Blocks

Production ETL procedures must handle errors gracefully. Redshift PL/pgSQL supports EXCEPTION blocks within BEGIN...END sub-blocks. When an error occurs inside the sub-block, execution jumps to the EXCEPTION handler, where you can log the error, perform cleanup, or re-raise the exception.

sql 
CREATE OR REPLACE PROCEDURE etl.safe_load_impressions(
    p_load_date DATE
)
AS $$
DECLARE
    v_row_count     INTEGER;
    v_error_msg     VARCHAR(4000);
    v_sql_state     VARCHAR(5);
BEGIN
    -- Wrap the critical section in a sub-block
    BEGIN
        DELETE FROM analytics.fact_impressions
        WHERE event_date = p_load_date;

        INSERT INTO analytics.fact_impressions
        SELECT *
        FROM staging.raw_impressions
        WHERE event_date = p_load_date;

        GET DIAGNOSTICS v_row_count = ROW_COUNT;

        -- Log success to audit table
        INSERT INTO etl.audit_log
            (procedure_name, load_date, rows_affected, status, completed_at)
        VALUES
            ('safe_load_impressions', p_load_date, v_row_count,
             'SUCCESS', GETDATE());

    EXCEPTION WHEN OTHERS THEN
        -- Capture the error details
        v_error_msg := SQLERRM;
        v_sql_state := SQLSTATE;

        -- Log failure to audit table
        INSERT INTO etl.audit_log
            (procedure_name, load_date, rows_affected, status,
             error_message, completed_at)
        VALUES
            ('safe_load_impressions', p_load_date, 0,
             'FAILED', v_sql_state || ': ' || v_error_msg,
             GETDATE());

        -- Re-raise so the caller knows it failed
        RAISE;
    END;
END;
$$ LANGUAGE plpgsql;
Pro Tip

Always create an etl.audit_log table before building procedures. Include columns for procedure_name, load_date, rows_affected, status, error_message, and completed_at. This gives your operations team visibility into every ETL run without digging through CloudWatch logs.

5. Transaction Control Inside Procedures

Transaction management inside Redshift stored procedures differs significantly from traditional databases. By default, the entire procedure runs in the caller's transaction. However, you can use explicit COMMIT and ROLLBACK statements to create intermediate transaction boundaries. This is called nonatomic procedure behavior, and it is automatically enabled when you include COMMIT or ROLLBACK in your procedure body.

sql 
-- Multi-stage ETL with explicit transaction boundaries
CREATE OR REPLACE PROCEDURE etl.full_pipeline(
    p_load_date DATE
)
AS $$
BEGIN
    -- Stage 1: Load raw data into staging
    CALL etl.load_staging(p_load_date);
    COMMIT;
    RAISE INFO 'Stage 1 complete: staging loaded and committed';

    -- Stage 2: Transform and load fact table
    CALL etl.transform_facts(p_load_date);
    COMMIT;
    RAISE INFO 'Stage 2 complete: facts transformed and committed';

    -- Stage 3: Update aggregate tables
    CALL etl.refresh_aggregates(p_load_date);
    COMMIT;
    RAISE INFO 'Stage 3 complete: aggregates refreshed and committed';

    -- Stage 4: Cleanup staging data
    DELETE FROM staging.raw_impressions
    WHERE event_date = p_load_date;
    COMMIT;
    RAISE INFO 'Stage 4 complete: staging cleaned up';
END;
$$ LANGUAGE plpgsql;

Each COMMIT finalizes the work done since the last commit point. If Stage 3 fails, Stages 1 and 2 remain committed. The pipeline can be restarted from Stage 3 without re-processing Stages 1 and 2. This pattern is critical for large-scale ETL where re-running the entire pipeline from scratch is expensive.

Warning

You cannot use COMMIT or ROLLBACK inside an EXCEPTION handler in Redshift. If you need both error handling and transaction control, structure your procedure so that the EXCEPTION block is inside a sub-block that does not contain COMMIT, and place the COMMIT outside the sub-block after verifying success.

6. ETL Orchestration Patterns

Real-world ETL pipelines coordinate multiple procedures, validate data quality between stages, and implement retry logic. Here is a production-grade orchestration pattern that combines everything covered in this chapter.

sql 
-- Master orchestration procedure
CREATE OR REPLACE PROCEDURE etl.orchestrate_daily_load(
    p_load_date DATE,
    INOUT p_status VARCHAR(20)
)
AS $$
DECLARE
    v_staging_count   BIGINT;
    v_fact_count      BIGINT;
    v_quality_pass    BOOLEAN;
BEGIN
    p_status := 'RUNNING';

    -- Step 1: Validate source data exists
    SELECT COUNT(*) INTO v_staging_count
    FROM staging.raw_impressions
    WHERE event_date = p_load_date;

    IF v_staging_count = 0 THEN
        RAISE INFO 'No staging data for %. Skipping.', p_load_date;
        p_status := 'SKIPPED';
        RETURN;
    END IF;

    RAISE INFO 'Found % staging rows for %', v_staging_count, p_load_date;

    -- Step 2: Run the core ETL
    CALL etl.safe_load_impressions(p_load_date);
    COMMIT;

    -- Step 3: Data quality check
    SELECT COUNT(*) INTO v_fact_count
    FROM analytics.fact_impressions
    WHERE event_date = p_load_date;

    v_quality_pass := (v_fact_count > 0)
        AND (v_fact_count >= v_staging_count * 0.95);

    IF NOT v_quality_pass THEN
        RAISE WARNING 'Quality check failed: staging=%, fact=%',
            v_staging_count, v_fact_count;
        p_status := 'QUALITY_FAIL';
        RETURN;
    END IF;

    -- Step 4: Refresh downstream aggregates
    CALL etl.refresh_aggregates(p_load_date);
    COMMIT;

    p_status := 'SUCCESS';
    RAISE INFO 'Daily load complete for %: %', p_load_date, p_status;
END;
$$ LANGUAGE plpgsql;

-- Invoke the orchestrator
CALL etl.orchestrate_daily_load('2025-06-15', NULL);

Self-Check

Before moving to Chapter 03, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What is the difference between a stored procedure and a function in Redshift?

A stored procedure is invoked with CALL and can contain transaction control statements (COMMIT, ROLLBACK). It cannot be used in a SELECT statement. A function is invoked within a SQL expression, returns a value, and cannot contain transaction control. For ETL workflows, procedures are the right choice because they can manage multi-step operations with explicit commit points.

Why is delete-then-insert preferred over UPSERT for fact table loading in Redshift?

Redshift does not support native MERGE or UPSERT with the same efficiency as OLTP databases. The delete-then-insert pattern is idempotent — running it twice with the same input produces the same output. It also avoids the complexity of matching on surrogate keys. For fact tables partitioned by date, deleting and re-inserting a single day's data is fast because the sort key enables efficient block pruning.

When would you use COMMIT inside a stored procedure, and what are the risks?

Use COMMIT to create intermediate save points in long-running pipelines so that completed stages persist even if later stages fail. The risk is that a failure leaves the data in a partially-transformed state. You must design your procedures to be restartable — each stage should be idempotent so re-running from the failed stage does not corrupt previously committed work.

How do you capture error details inside an EXCEPTION handler?

Inside an EXCEPTION WHEN OTHERS THEN block, use the special variables SQLERRM (the error message text) and SQLSTATE (the 5-character SQL state code) to capture error details. Store these in your audit log table for post-mortem analysis. Use RAISE without arguments to re-raise the original exception after logging it.
