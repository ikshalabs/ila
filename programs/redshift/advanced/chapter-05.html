Redshift Spectrum and Data Lake

Extend your Redshift warehouse to query data directly in Amazon S3 using Spectrum — external schemas, external tables, Parquet and ORC formats, S3 partitioning strategies, Spectrum query optimization, and hybrid query patterns that combine local and external data.

Estimated: 60–80 min
6 Key Concepts
3 Assignments

Why This Matters

Not all data belongs inside Redshift. Historical data older than 90 days, raw event logs before transformation, machine learning training datasets, and third-party data feeds are often best stored in Amazon S3 as part of a data lake. But analysts still need to query this data alongside their warehouse tables. Without Spectrum, you would need to load all of it into Redshift — consuming expensive cluster storage and slowing down ETL pipelines.

Redshift Spectrum lets you run SQL queries directly against data stored in S3 without loading it into Redshift tables. Spectrum uses its own fleet of compute resources (separate from your cluster's compute nodes) to scan, filter, and aggregate S3 data. The results are then sent back to your Redshift cluster for final processing and joining with local tables. This creates a hybrid architecture where hot data lives in Redshift for fast queries and cold data stays in S3 for cost-effective storage.

Key Insight

Spectrum queries scale independently of your Redshift cluster. Whether you have 2 nodes or 20, Spectrum can process the same amount of S3 data at the same speed because it uses a shared pool of AWS-managed compute resources. You pay per terabyte of data scanned — making file format and partitioning choices critical for cost control.

Key Concepts

1. External Schemas and the AWS Glue Catalog

Before querying S3 data, you need to define an external schema that maps to a database in the AWS Glue Data Catalog (or an Apache Hive metastore). The Glue Catalog stores the metadata about your external tables — column names, data types, file formats, and S3 locations — without storing any of the actual data.

sql 
-- Create an external schema pointing to the Glue Data Catalog
CREATE EXTERNAL SCHEMA datalake
FROM DATA CATALOG
DATABASE 'adtech_datalake'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftSpectrumRole'
CREATE EXTERNAL DATABASE IF NOT EXISTS;

-- Verify the schema was created
SELECT *
FROM svv_external_schemas
WHERE schemaname = 'datalake';

The IAM role must have permissions to access both the Glue Data Catalog (glue:GetTable, glue:GetPartitions) and the S3 data (s3:GetObject, s3:ListBucket). This role is assumed by Spectrum when executing queries, so its permissions define what data your Redshift cluster can access in the data lake.

2. External Tables

External tables define the structure of data files in S3. They specify column names and types, the file format (Parquet, ORC, CSV, JSON), the S3 location, and optionally the partition scheme. External tables are read-only — you cannot INSERT, UPDATE, or DELETE rows in them through Redshift.

sql 
-- External table for historical impressions in Parquet format
CREATE EXTERNAL TABLE datalake.historical_impressions (
    impression_id       BIGINT,
    event_timestamp     TIMESTAMP,
    campaign_id         INTEGER,
    creative_id         INTEGER,
    publisher_id        INTEGER,
    device_type         VARCHAR(20),
    country_code        CHAR(2),
    bid_price_cpm       DECIMAL(10,4),
    win_price_cpm       DECIMAL(10,4),
    is_viewable         BOOLEAN,
    is_clicked          BOOLEAN
)
PARTITIONED BY (event_date DATE)
STORED AS PARQUET
LOCATION 's3://adtech-datalake-prod/impressions/';

-- Register partitions from the S3 folder structure
ALTER TABLE datalake.historical_impressions
ADD PARTITION (event_date='2024-01-01')
LOCATION 's3://adtech-datalake-prod/impressions/event_date=2024-01-01/';

-- Or use MSCK REPAIR TABLE to auto-discover Hive-style partitions
-- (requires Glue Catalog and Hive-style partition directories)

3. Parquet and ORC File Formats

File format is the single most impactful choice for Spectrum query performance and cost. Columnar formats like Parquet and ORC store data column-by-column, enabling Spectrum to read only the columns referenced in your query. Row-based formats like CSV and JSON require Spectrum to read entire rows even if you only need two columns.

  • Apache Parquet — The most widely used columnar format. Supports predicate pushdown, column pruning, and built-in compression (Snappy, GZIP, ZSTD). Choose Parquet as your default format for Spectrum.
  • Apache ORC — Similar benefits to Parquet with slightly better compression for certain data types. More common in Hive/Hadoop ecosystems. Use ORC if your data pipeline already produces it.
  • CSV/TSV — Row-based, no column pruning, no predicate pushdown. Use only for raw ingestion staging, never for production Spectrum queries.
  • JSON — Row-based with nested structures. Useful for semi-structured data but significantly slower than columnar formats for analytical queries.
Important

Spectrum charges per terabyte of data scanned from S3. Switching from CSV to Parquet can reduce the data scanned by 90% or more because Parquet enables column pruning (reading only needed columns) and better compression. A query that scans 1 TB of CSV data might scan only 50-100 GB of the same data in Parquet format — saving both money and query time.

4. S3 Partitioning Strategies

Partitioning your S3 data by frequently-filtered columns (typically date) allows Spectrum to skip entire directories of files that do not match your query's WHERE clause. This is called partition pruning and is the most effective optimization for Spectrum queries.

sql 
-- S3 directory structure with Hive-style partitioning:
-- s3://adtech-datalake-prod/impressions/
--   event_date=2024-01-01/
--     part-00000.parquet
--     part-00001.parquet
--   event_date=2024-01-02/
--     part-00000.parquet
--     part-00001.parquet

-- Multi-level partitioning for finer granularity
CREATE EXTERNAL TABLE datalake.raw_bid_logs (
    bid_request_id      VARCHAR(64),
    auction_id          VARCHAR(64),
    bidder_id           INTEGER,
    bid_amount_cpm      DECIMAL(10,4),
    bid_response_ms     INTEGER,
    user_agent          VARCHAR(500),
    ip_address          VARCHAR(45)
)
PARTITIONED BY (
    event_year   INTEGER,
    event_month  INTEGER,
    event_day    INTEGER
)
STORED AS PARQUET
LOCATION 's3://adtech-datalake-prod/bid-logs/';

-- This query prunes to a single day's partition
SELECT
    bidder_id,
    COUNT(*) AS total_bids,
    AVG(bid_amount_cpm) AS avg_bid
FROM datalake.raw_bid_logs
WHERE event_year = 2024
  AND event_month = 6
  AND event_day = 15
GROUP BY bidder_id
ORDER BY total_bids DESC;
Pro Tip

Target Parquet file sizes of 128 MB to 512 MB each. Files that are too small (under 32 MB) create excessive S3 LIST operations and metadata overhead. Files that are too large (over 1 GB) reduce parallelism because Spectrum cannot split a single file across multiple readers. Use tools like AWS Glue ETL or Apache Spark to repartition and compact small files.

5. Spectrum Query Optimization

Spectrum pushes as much processing as possible down to the S3 scan layer. Understanding what gets pushed down and what gets pulled back to Redshift helps you write efficient queries.

  • Column pruning — Only the columns in your SELECT, WHERE, GROUP BY, and JOIN are read from Parquet/ORC files.
  • Predicate pushdown — Simple filter predicates (=, <, >, BETWEEN, IN) on partition columns and data columns are applied during the S3 scan, before data is sent to Redshift.
  • Aggregation pushdown — Simple aggregations (COUNT, SUM, MIN, MAX) can be partially computed by Spectrum before sending results to Redshift.
  • File format matters — Parquet and ORC enable all three optimizations above. CSV and JSON only benefit from partition pruning.
sql 
-- Check how much data Spectrum scanned
SELECT
    query,
    segment,
    elapsed,
    s3_scanned_rows,
    s3_scanned_bytes,
    s3query_returned_rows,
    s3query_returned_bytes,
    files
FROM svl_s3query_summary
WHERE query = pg_last_query_id()
ORDER BY segment;

-- Verify partition pruning is working
EXPLAIN
SELECT COUNT(*)
FROM datalake.historical_impressions
WHERE event_date = '2024-06-15';
-- Look for "S3 Seq Scan" with partition filter in the plan

6. Hybrid Queries: Combining Local and External Data

The real power of Spectrum is joining external S3 data with local Redshift tables in a single query. This enables a tiered storage architecture where recent data (last 90 days) lives in Redshift for fast queries, and historical data stays in S3 for cost-effective long-term storage.

sql 
-- Hybrid query: join Spectrum external data with local dimensions
SELECT
    dc.campaign_name,
    dc.advertiser_name,
    hi.event_date,
    COUNT(*) AS impressions,
    SUM(CASE WHEN hi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    ROUND(AVG(hi.win_price_cpm), 2) AS avg_cpm
FROM datalake.historical_impressions hi
JOIN analytics.dim_campaign dc
    ON hi.campaign_id = dc.campaign_id
    AND dc.is_current = TRUE
WHERE hi.event_date BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY dc.campaign_name, dc.advertiser_name, hi.event_date
ORDER BY impressions DESC;

-- Union local and external data for a full timeline view
SELECT event_date, campaign_id, SUM(impressions) AS impressions
FROM (
    -- Recent data from local Redshift table
    SELECT event_date, campaign_id, 1 AS impressions
    FROM analytics.fact_impressions
    WHERE event_date >= CURRENT_DATE - 90

    UNION ALL

    -- Historical data from S3 via Spectrum
    SELECT event_date, campaign_id, 1 AS impressions
    FROM datalake.historical_impressions
    WHERE event_date < CURRENT_DATE - 90
      AND event_date >= '2024-01-01'
)
GROUP BY event_date, campaign_id
ORDER BY event_date;

Warning

Never use Spectrum for queries that scan the entire data lake without partition filters. A query against 10 TB of unpartitioned CSV data will scan all 10 TB, cost $50, and take minutes to complete. Always include partition predicates in your WHERE clause and use columnar file formats.

Self-Check

Before moving to Chapter 06, honestly answer these questions.

What is the difference between a Redshift local table and a Spectrum external table?

A local table stores data on the Redshift cluster's managed storage (SSD or RA3 managed storage). It supports full DML (INSERT, UPDATE, DELETE) and uses the cluster's compute nodes for scanning. An external table is a metadata definition that points to files in S3. It is read-only from Redshift and uses the Spectrum compute layer (separate from your cluster) for scanning. External tables do not consume cluster storage.

Why does Parquet dramatically reduce Spectrum scan costs compared to CSV?

Parquet stores data column-by-column with built-in compression. When your query needs only 3 of 20 columns, Spectrum reads only those 3 columns from the Parquet file, skipping the other 17 entirely. CSV stores data row-by-row, so Spectrum must read every column in every row even if the query only references a few. This column pruning typically reduces scanned data by 70-95%, directly reducing the per-TB scan cost.

How does partition pruning work and why is it critical for Spectrum cost control?

Partition pruning uses the WHERE clause predicates on partition columns to determine which S3 directories to scan. If data is partitioned by event_date and your query filters WHERE event_date = '2024-06-15', Spectrum only reads files in the event_date=2024-06-15/ directory, skipping all other dates. Without partition filters, Spectrum scans every file in the table's S3 location. For a table with a year of daily data, a single-day query with pruning scans 1/365th of the data.

When should you keep data in Redshift local tables versus querying it via Spectrum?

Keep data in local Redshift tables when it is queried frequently, needs sub-second response times, or is involved in complex multi-table joins. Use Spectrum for data that is accessed infrequently (historical data older than 90 days), is very large (would be too expensive to store on the cluster), or is already in S3 from other pipelines. The hybrid approach gives you the best of both worlds: fast queries on hot data and cost-effective access to the full historical dataset.
