Chapter 01 · Redshift Advanced

ETL Pipeline Architecture

Design and implement production-grade Extract-Transform-Load pipelines in Amazon Redshift — from S3 ingestion with COPY commands, staging table patterns, incremental loads, SCD handling, UNLOAD for data export, and orchestration strategies that keep your warehouse fresh and reliable.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

A data warehouse is only as valuable as the data that flows into it. You can have the most perfectly designed star schema in Redshift, with optimal distribution keys and sort keys, but if your ETL pipelines are brittle, slow, or produce duplicates, every downstream dashboard and report becomes unreliable. In production environments, ETL pipeline failures are the single most common source of data quality incidents.

Amazon Redshift provides a unique set of tools for ETL that differ significantly from traditional databases. The COPY command can ingest terabytes from S3 in parallel across all slices simultaneously. The UNLOAD command can export query results back to S3 in partitioned Parquet format. Staging tables with CREATE TABLE ... AS (CTAS) and ALTER TABLE APPEND enable atomic swap patterns that prevent users from ever seeing partial loads. Understanding these Redshift-specific patterns is what separates a reliable data pipeline from one that breaks every Monday morning.

Key Insight

Redshift is not Postgres. While it supports most PostgreSQL syntax, the performance characteristics of data loading are fundamentally different. A single INSERT INTO ... VALUES statement writes to one slice. A COPY command from S3 writes to all slices in parallel, achieving 10–100x higher throughput. Every ETL pipeline in Redshift should be built around COPY, not INSERT.

Key Concepts

1. The COPY Command — Parallel Ingestion from S3

The COPY command is Redshift's primary data ingestion mechanism. It reads files from Amazon S3, Amazon DynamoDB, or remote hosts via SSH, and loads them into a target table. The critical advantage of COPY is parallelism: Redshift distributes the load across all compute node slices simultaneously. If your cluster has 16 slices and you provide 16 or more input files, each slice reads one file independently, maximizing throughput.

For optimal performance, split your source data into multiple files. The ideal number of files is a multiple of the total number of slices in your cluster. For example, with an 8-slice cluster, use 8, 16, 32, or 64 input files. Each file should be between 1 MB and 1 GB after compression for best results.

sql 
-- Basic COPY from S3 with compressed CSV files
COPY staging.raw_impressions
FROM 's3://my-adtech-bucket/impressions/2025/01/15/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS CSV
IGNOREHEADER 1
GZIP
DATEFORMAT 'auto'
TIMEFORMAT 'auto'
MAXERROR 100
COMPUPDATE OFF
STATUPDATE OFF;

-- COPY from S3 with Parquet (schema auto-detected from file)
COPY staging.raw_impressions
FROM 's3://my-adtech-bucket/impressions/parquet/dt=2025-01-15/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS PARQUET;

-- COPY with manifest file for explicit file listing
COPY staging.raw_impressions
FROM 's3://my-adtech-bucket/manifests/impressions_20250115.manifest'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS CSV
GZIP
MANIFEST;
Important

Always use COMPUPDATE OFF and STATUPDATE OFF in production COPY commands. Automatic compression analysis and statistics updates add significant overhead to each load. Instead, set compression encodings when you create the table, and run ANALYZE as a separate step after loading completes. This alone can cut your COPY time by 30–50%.

2. Staging Table Patterns

Production ETL pipelines never load data directly into fact or dimension tables. Instead, you load raw data into a staging table, validate and transform it there, and then merge the results into production tables. This pattern provides several critical benefits: you can validate data quality before it reaches production, you can handle duplicates via deduplication logic, and if anything goes wrong, production tables remain untouched.

sql 
-- Step 1: Create staging table (temporary, drops on session end)
CREATE TEMP TABLE stg_impressions (
    event_timestamp   TIMESTAMP,
    campaign_id       INTEGER,
    creative_id       INTEGER,
    device_type       VARCHAR(20),
    country_code      CHAR(2),
    bid_price_cpm     DECIMAL(10,4),
    win_price_cpm     DECIMAL(10,4),
    impression_id_raw VARCHAR(64)
);

-- Step 2: COPY raw data into staging
COPY stg_impressions
FROM 's3://my-adtech-bucket/impressions/2025/01/15/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS CSV
IGNOREHEADER 1
GZIP
COMPUPDATE OFF
STATUPDATE OFF;

-- Step 3: Validate - check for nulls in required columns
SELECT COUNT(*) AS null_campaigns
FROM stg_impressions
WHERE campaign_id IS NULL;

-- Step 4: Deduplicate and insert into production
INSERT INTO analytics.fact_impressions (
    event_timestamp, event_date, campaign_id, creative_id,
    device_type, country_code, bid_price_cpm, win_price_cpm
)
SELECT
    s.event_timestamp,
    s.event_timestamp::DATE AS event_date,
    s.campaign_id,
    s.creative_id,
    s.device_type,
    s.country_code,
    s.bid_price_cpm,
    s.win_price_cpm
FROM stg_impressions s
WHERE NOT EXISTS (
    SELECT 1
    FROM analytics.fact_impressions f
    WHERE f.event_timestamp = s.event_timestamp
      AND f.campaign_id = s.campaign_id
      AND f.creative_id = s.creative_id
);

For very large loads where the NOT EXISTS deduplication is slow, consider the delete-and-reinsert pattern: delete all rows from the production table that match the staging table's date range, then insert all staging rows. This is often faster because Redshift optimizes bulk deletes much better than row-by-row existence checks.

3. Incremental and Full-Refresh Loading Strategies

There are two fundamental approaches to keeping your warehouse current: full-refresh (reload the entire table each time) and incremental load (load only new or changed records since the last load). The right choice depends on table size, change frequency, and how much latency your business can tolerate.

sql 
-- INCREMENTAL: Delete-and-reinsert for daily partitions
BEGIN TRANSACTION;

-- Remove existing data for the target date
DELETE FROM analytics.fact_impressions
WHERE event_date = '2025-01-15';

-- Insert fresh data from staging
INSERT INTO analytics.fact_impressions
SELECT
    event_timestamp,
    event_timestamp::DATE,
    campaign_id, creative_id, device_type,
    country_code, bid_price_cpm, win_price_cpm,
    is_viewable, is_clicked
FROM stg_impressions;

COMMIT;

-- Post-load maintenance
ANALYZE analytics.fact_impressions;
sql 
-- FULL REFRESH: Atomic table swap using CTAS + rename
CREATE TABLE analytics.dim_campaign_new
DISTSTYLE ALL
SORTKEY (campaign_id)
AS
SELECT
    c.campaign_id,
    c.campaign_name,
    c.advertiser_id,
    a.advertiser_name,
    c.budget_amount,
    c.start_date,
    c.end_date,
    c.status,
    c.updated_at
FROM staging.raw_campaigns c
JOIN staging.raw_advertisers a
    ON c.advertiser_id = a.advertiser_id;

-- Atomic swap: users never see an empty table
BEGIN TRANSACTION;
ALTER TABLE analytics.dim_campaign RENAME TO dim_campaign_old;
ALTER TABLE analytics.dim_campaign_new RENAME TO dim_campaign;
DROP TABLE analytics.dim_campaign_old;
COMMIT;
Pro Tip

The CTAS + rename pattern is the safest way to rebuild dimension tables. The CREATE TABLE ... AS statement creates the new table with data in a single atomic operation. The rename swap inside a transaction ensures that at no point does a query hit an empty or partially loaded table.

4. The UNLOAD Command — Exporting Data to S3

Just as COPY is the primary ingestion tool, UNLOAD is the primary export tool. It writes query results to S3 in parallel across all slices. UNLOAD supports CSV, JSON, and Parquet output formats. Parquet is strongly recommended for downstream consumption by Spark, Athena, or Redshift Spectrum because it preserves column types and enables predicate pushdown.

sql 
-- UNLOAD query results to S3 as Parquet with partitioning
UNLOAD ('
    SELECT
        event_date,
        campaign_id,
        device_type,
        country_code,
        COUNT(*) AS impressions,
        SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks,
        ROUND(AVG(win_price_cpm), 4) AS avg_cpm
    FROM analytics.fact_impressions
    WHERE event_date BETWEEN ''''2025-01-01'''' AND ''''2025-01-31''''
    GROUP BY event_date, campaign_id, device_type, country_code
')
TO 's3://my-adtech-bucket/exports/campaign_daily/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
FORMAT AS PARQUET
PARTITION BY (event_date)
ALLOWOVERWRITE;

-- UNLOAD as gzipped CSV for legacy consumers
UNLOAD ('SELECT * FROM analytics.dim_campaign')
TO 's3://my-adtech-bucket/exports/dim_campaign/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
CSV
HEADER
GZIP
PARALLEL ON
ALLOWOVERWRITE;

5. Error Handling and Load Validation

Production ETL pipelines must handle errors gracefully. The COPY command logs rejected rows to the STL_LOAD_ERRORS system table. You should always check this table after every load and alert on unexpected error counts. The MAXERROR parameter controls how many individual row errors are tolerated before the entire COPY fails.

sql 
-- Check for load errors after a COPY command
SELECT
    starttime,
    filename,
    line_number,
    colname,
    type,
    raw_field_value,
    err_reason
FROM stl_load_errors
WHERE starttime > GETDATE() - INTERVAL '1 hour'
ORDER BY starttime DESC
LIMIT 50;

-- Validation: row count reconciliation
SELECT
    'staging' AS source, COUNT(*) AS row_count
FROM stg_impressions
UNION ALL
SELECT
    'production', COUNT(*)
FROM analytics.fact_impressions
WHERE event_date = '2025-01-15';

-- Create an audit log table for ETL tracking
CREATE TABLE etl.load_audit (
    load_id          BIGINT IDENTITY(1,1),
    table_name       VARCHAR(128),
    load_timestamp   TIMESTAMP DEFAULT GETDATE(),
    rows_loaded      BIGINT,
    rows_rejected    BIGINT,
    source_file_path VARCHAR(512),
    load_duration_s  INTEGER,
    status           VARCHAR(20) DEFAULT 'SUCCESS'
)
DISTSTYLE AUTO
SORTKEY (load_timestamp);
Common Pitfall

Never set MAXERROR to a very high number in production. If your source data format changes unexpectedly, you could silently load millions of rows with NULL values in critical columns. Keep MAXERROR small (100–500), and investigate any errors that occur.

6. Orchestration and Scheduling

Individual COPY and INSERT statements are building blocks. A production ETL pipeline chains these together with dependency management, retries, and alerting. Common orchestration tools for Redshift ETL include AWS Step Functions, Apache Airflow, and dbt.

sql 
-- Complete ETL workflow in a single transaction
BEGIN TRANSACTION;

-- 1. Load raw data into staging
COPY staging.raw_clicks
FROM 's3://my-adtech-bucket/clicks/2025/01/15/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftCopyRole'
FORMAT AS PARQUET;

-- 2. Delete existing data for the partition
DELETE FROM analytics.fact_clicks
WHERE event_date = '2025-01-15';

-- 3. Transform and insert into production
INSERT INTO analytics.fact_clicks
SELECT
    click_timestamp,
    click_timestamp::DATE AS event_date,
    campaign_id,
    creative_id,
    impression_id,
    landing_page_url,
    COALESCE(revenue, 0.0) AS revenue
FROM staging.raw_clicks
WHERE campaign_id IS NOT NULL;

-- 4. Log the load
INSERT INTO etl.load_audit (table_name, rows_loaded, rows_rejected)
SELECT
    'analytics.fact_clicks',
    (SELECT COUNT(*) FROM staging.raw_clicks WHERE campaign_id IS NOT NULL),
    (SELECT COUNT(*) FROM staging.raw_clicks WHERE campaign_id IS NULL);

COMMIT;

-- 5. Post-load maintenance (outside transaction)
ANALYZE analytics.fact_clicks;
VACUUM DELETE ONLY analytics.fact_clicks;

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer is unclear, revisit the relevant section.

Why should you split source files into multiples of the slice count before running COPY?

Redshift assigns one file per slice during a COPY operation. If you have 16 slices but only 1 file, only one slice does all the work while the other 15 sit idle. Splitting into 16+ files means every slice reads data in parallel, maximizing ingestion throughput. The ideal file size is 1 MB to 1 GB after compression.

What is the difference between a full-refresh and an incremental load, and when would you choose each?

A full-refresh drops and recreates the entire table (or uses CTAS + rename). It is appropriate for small dimension tables or when the source lacks reliable change tracking. An incremental load processes only new or changed records. It is essential for large fact tables where reloading billions of rows daily is prohibitively slow. Use the delete-and-reinsert pattern partitioned by date.

Why should VACUUM and ANALYZE run outside the main ETL transaction?

VACUUM cannot run inside a transaction block — Redshift will reject it. Running ANALYZE separately lets the main transaction commit quickly, reducing lock contention. Schedule both during low-usage windows as separate post-load steps.

How does the CTAS atomic swap pattern prevent users from seeing empty or partially loaded tables?

The CREATE TABLE ... AS statement builds the new table completely before any rename occurs. The rename inside a transaction is a metadata-only change that is instantaneous. At no point does the production table name point to an empty table. Users will see either old data or new data, never nothing.
