Chapter 06 · Redshift Advanced

Cross-Database Queries and Data Sharing

Learn to query across multiple Redshift databases, share live data between clusters using datashare objects, implement producer/consumer architectures, run federated queries against Aurora and RDS, and build multi-team data mesh patterns without data duplication.

Estimated: 55–70 min
6 Key Concepts
3 Assignments

Why This Matters

Modern organizations rarely have a single database. The ad-tech team runs campaign analytics in one Redshift database, the data science team has ML feature tables in another, the finance team tracks billing in Aurora PostgreSQL, and the product team stores user metadata in RDS MySQL. Without cross-database capabilities, teams resort to ETL pipelines that copy data between systems — creating stale copies, inconsistent definitions, and escalating storage costs.

Redshift provides three mechanisms to break down these silos: cross-database queries (querying multiple databases on the same cluster), datashares (sharing live data between separate Redshift clusters without copying), and federated queries (querying external Aurora and RDS databases directly from Redshift SQL). Together, these features enable a data mesh architecture where each team owns their data but makes it accessible to others through governed, read-only interfaces.

Key Insight

Data sharing in Redshift is live, not copied. When a consumer cluster queries shared data, it reads directly from the producer's storage. There is no data duplication, no synchronization lag, and no additional storage cost on the consumer side. The producer maintains full control over what is shared and can revoke access at any time.

Key Concepts

1. Cross-Database Queries

A single Redshift cluster can host multiple databases. Cross-database queries allow you to reference tables in any database on the cluster using three-part naming: database.schema.table. You do not need to switch database context — a single query can join tables from different databases.

sql 
-- Query from 'analytics' database joining with 'billing' database
-- Connected to the 'analytics' database

SELECT
    a.campaign_id,
    a.campaign_name,
    SUM(a.impressions) AS total_impressions,
    b.monthly_invoice_amount,
    b.payment_status
FROM analytics.public.fact_campaign_daily a
JOIN billing.public.campaign_invoices b
    ON a.campaign_id = b.campaign_id
    AND b.invoice_month = '2025-06'
WHERE a.event_date BETWEEN '2025-06-01' AND '2025-06-30'
GROUP BY a.campaign_id, a.campaign_name,
         b.monthly_invoice_amount, b.payment_status
ORDER BY total_impressions DESC;

Cross-database queries work for both local tables and external schemas (Spectrum). You can also create cross-database views that provide a simplified interface for analysts who should not need to know which database owns which table.

Important

Cross-database queries are read-only for tables in other databases. You can SELECT from a table in another database, but you cannot INSERT, UPDATE, or DELETE into it. Write operations must be performed while connected to the database that owns the table.

2. Datashare Objects

Datashares enable sharing data between separate Redshift clusters (or between provisioned clusters and Redshift Serverless workgroups) within the same AWS account or across different accounts. The producer cluster creates a datashare and adds schemas and tables to it. The consumer cluster creates a database from the datashare and queries the shared data as if it were local.

sql 
-- ═══ ON THE PRODUCER CLUSTER ═══

-- Step 1: Create the datashare
CREATE DATASHARE adtech_analytics_share
SET PUBLICACCESSIBLE FALSE;

-- Step 2: Add schemas to share
ALTER DATASHARE adtech_analytics_share
ADD SCHEMA analytics;

-- Step 3: Add specific tables (or all tables in schema)
ALTER DATASHARE adtech_analytics_share
ADD ALL TABLES IN SCHEMA analytics;

-- Step 4: Grant usage to consumer cluster's namespace
GRANT USAGE ON DATASHARE adtech_analytics_share
TO NAMESPACE 'a1b2c3d4-e5f6-7890-abcd-ef1234567890';

-- Verify the share
SELECT * FROM svv_datashares;
SELECT * FROM svv_datashare_objects;
sql 
-- ═══ ON THE CONSUMER CLUSTER ═══

-- Step 1: Create a local database from the datashare
CREATE DATABASE adtech_shared
FROM DATASHARE adtech_analytics_share
OF NAMESPACE 'f9e8d7c6-b5a4-3210-fedc-ba9876543210';

-- Step 2: Query shared data as if it were local
SELECT
    campaign_id,
    event_date,
    SUM(impressions) AS total_impressions,
    SUM(clicks) AS total_clicks
FROM adtech_shared.analytics.fact_impressions
WHERE event_date >= '2025-06-01'
GROUP BY campaign_id, event_date
ORDER BY total_impressions DESC
LIMIT 20;

3. Producer/Consumer Architecture

The producer/consumer pattern is the foundation of data sharing in Redshift. The producer cluster owns the data and controls what is shared. Multiple consumer clusters can access the same datashare simultaneously. This architecture supports several real-world scenarios:

  • Team isolation — Each team has their own cluster (or Serverless workgroup) with independent compute, but shares a common set of curated analytics tables from the central data warehouse.
  • Multi-tenant analytics — A SaaS platform shares different subsets of data with different client clusters, each seeing only their own data through filtered views in the datashare.
  • Dev/staging/production — Production data can be shared read-only with development and staging clusters for realistic testing without data copying.
  • Cross-account sharing — Data can be shared between AWS accounts, enabling partner integrations or subsidiary data access without S3 exports.

4. Federated Queries to Aurora and RDS

Federated queries let you run SQL from Redshift that reaches into Amazon Aurora PostgreSQL, Aurora MySQL, or Amazon RDS databases. This is invaluable when you need to join real-time operational data (user profiles, campaign configurations, billing records) with warehouse analytics without building ETL pipelines to replicate that data into Redshift.

sql 
-- Step 1: Store credentials in Secrets Manager
-- (done via AWS Console or CLI, not SQL)

-- Step 2: Create external schema for Aurora PostgreSQL
CREATE EXTERNAL SCHEMA aurora_campaigns
FROM POSTGRES
DATABASE 'campaign_manager'
SCHEMA 'public'
URI 'aurora-cluster-prod.cluster-abc123.us-east-1.rds.amazonaws.com'
PORT 5432
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftFederatedRole'
SECRET_ARN 'arn:aws:secretsmanager:us-east-1:123456789012:secret:aurora-creds-xyz';

-- Step 3: Query Aurora data directly from Redshift
SELECT
    ac.campaign_id,
    ac.campaign_name,
    ac.daily_budget,
    ac.status,
    SUM(fi.impressions) AS today_impressions,
    SUM(fi.clicks) AS today_clicks,
    ROUND(
        SUM(fi.win_price_cpm) / 1000.0, 2
    ) AS today_spend
FROM aurora_campaigns.campaigns ac
JOIN analytics.fact_impressions fi
    ON ac.campaign_id = fi.campaign_id
WHERE fi.event_date = CURRENT_DATE
  AND ac.status = 'active'
GROUP BY ac.campaign_id, ac.campaign_name,
         ac.daily_budget, ac.status
HAVING SUM(fi.win_price_cpm) / 1000.0 > ac.daily_budget * 0.9
ORDER BY today_spend DESC;
Pro Tip

Federated queries push simple predicates down to the remote database to reduce data transfer. Structure your queries so that filters on Aurora/RDS columns appear directly in the WHERE clause rather than in a HAVING or subquery. This allows Redshift to send the filter to Aurora, which returns only the matching rows instead of the entire table.

5. Practical Aurora/RDS Integration Patterns

Federated queries are best used for small-to-medium lookups against operational databases — not for scanning millions of rows from Aurora. Here are the recommended patterns:

  • Configuration lookups — Join Redshift analytics with Aurora campaign configuration tables (budgets, targeting rules, flight dates) that change in real-time.
  • User profile enrichment — Enrich Redshift impression data with user profile attributes stored in RDS MySQL without replicating the user table.
  • Budget pacing alerts — Compare Redshift spend aggregates against Aurora budget records to detect campaigns approaching their daily or lifetime caps.
  • Reconciliation queries — Compare Redshift analytics totals against Aurora billing records to identify discrepancies.
Warning

Federated queries create a live connection to the remote database for each query execution. Do not run heavy aggregations that scan millions of remote rows — this will slow down both Redshift and the operational database. If you need large-scale analysis of Aurora data, use AWS DMS or Glue to replicate it into Redshift first.

6. Data Mesh Patterns with Datashares

Datashares enable a data mesh architecture where each team maintains ownership of their domain data while making curated datasets available to the rest of the organization. The key principle is domain-oriented decentralized data ownership with self-serve data infrastructure.

sql 
-- Producer: Ad Operations team publishes curated metrics
CREATE DATASHARE adops_metrics_share;

-- Share only the curated analytics schema, not raw data
ALTER DATASHARE adops_metrics_share
ADD SCHEMA curated_analytics;

ALTER DATASHARE adops_metrics_share
ADD TABLE curated_analytics.daily_campaign_performance;

ALTER DATASHARE adops_metrics_share
ADD TABLE curated_analytics.weekly_publisher_report;

ALTER DATASHARE adops_metrics_share
ADD TABLE curated_analytics.dim_campaign_current;

-- Grant to the Data Science team's cluster
GRANT USAGE ON DATASHARE adops_metrics_share
TO NAMESPACE 'ds-team-namespace-id';

-- Grant to the Finance team's cluster
GRANT USAGE ON DATASHARE adops_metrics_share
TO NAMESPACE 'finance-team-namespace-id';

-- Monitor who is using the shared data
SELECT
    share_name,
    consumer_namespace,
    consumer_account,
    share_date
FROM svv_datashare_consumers
ORDER BY share_date DESC;

Assignment 1
Level 1 — Foundation

Cross-Database Query Setup

Create two databases on your cluster and write queries that join tables across them.

  • Create an analytics database with a fact table and a billing database with an invoices table
  • Populate both tables with sample data that shares a common campaign_id column
  • Write a cross-database join query using three-part naming (database.schema.table)
  • Create a view in the analytics database that abstracts the cross-database join
  • Verify the view returns correct results and document any limitations you observe
Done when

You can run a single query that joins data from two different databases and returns a combined result set.

Assignment 2
Level 2 — Professional

Implement Producer/Consumer Data Sharing

Set up a datashare between two Redshift clusters or workgroups (use Serverless for cost-effective testing).

  • Create a datashare on the producer cluster with a curated analytics schema
  • Add specific tables (not all tables) to the datashare to practice selective sharing
  • Grant usage to a consumer cluster or Serverless workgroup
  • On the consumer, create a database from the datashare and query the shared data
  • Verify that inserting data on the producer is immediately visible on the consumer
Done when

Data inserted into the producer table is queryable from the consumer within seconds, with zero ETL or data copying.

Assignment 3
Level 3 — Production-Ready

Build a Federated Query for Budget Pacing

Create a federated query that compares Redshift analytics data with Aurora campaign configurations for real-time budget monitoring.

  • Create an Aurora PostgreSQL instance (or use an existing one) with a campaigns table containing budget data
  • Store Aurora credentials in AWS Secrets Manager
  • Create an external schema in Redshift pointing to Aurora
  • Write a budget pacing query that joins Redshift spend data with Aurora budget configurations
  • Create a stored procedure that runs the pacing check and inserts alerts into an alerts table
Done when

Your stored procedure identifies campaigns that have spent over 90% of their daily budget by querying live Aurora budget data joined with Redshift spend aggregates.

Self-Check

Before moving to Chapter 07, honestly answer these questions.

What is the difference between cross-database queries and datashares?

Cross-database queries let you join tables across different databases on the same Redshift cluster. All databases share the same compute resources. Datashares let you share data between separate Redshift clusters (or Serverless workgroups), potentially in different AWS accounts. Each consumer has its own independent compute resources, so heavy queries on the consumer do not affect the producer's performance.

Is shared data via datashare copied to the consumer cluster?

No. Datashares provide live, read-only access to the producer's data. The consumer reads directly from the producer's storage layer. There is no data duplication, no synchronization delay, and no additional storage cost on the consumer. Changes made on the producer are immediately visible to the consumer.

When should you use federated queries versus ETL replication?

Use federated queries for small-to-medium lookups against operational databases (hundreds of thousands of rows or less) where you need real-time data without ETL lag. Use ETL replication (via AWS DMS, Glue, or stored procedures) when you need to analyze large volumes of operational data, when query performance is critical, or when you want to transform the data before analysis. Federated queries add latency and load to the remote database, so they are not suitable for scanning millions of rows.

What are the key security considerations for data sharing?

The producer controls exactly which schemas and tables are shared — consumers cannot access unshared objects. Datashares can be restricted to specific consumer cluster namespaces (identified by UUID). Cross-account sharing requires explicit authorization from both the producer and consumer accounts. Federated queries require credentials stored in AWS Secrets Manager and an IAM role with the minimum necessary permissions to access the remote database. Always apply the principle of least privilege: share only what is needed, grant only the permissions required.
