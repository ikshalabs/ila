Chapter 07 · Redshift Basic

Data Transformation

Build robust data transformation pipelines using CTEs, CTAS, temporary tables, user-defined functions, and stored procedures — the essential tools for turning raw ad-tech event logs into clean, analysis-ready datasets.

Estimated: 60–90 min
Key Concepts
3 Assignments

Why This Matters

Raw impression logs from bid servers are messy: timestamps in different formats, null device types, duplicate events from retry logic, and bid prices that need currency conversion. Data transformation is the process of cleaning, enriching, and reshaping this raw data into the star schema that powers your dashboards and reports.

Redshift provides several transformation mechanisms, each optimized for different use cases. CTEs and subqueries handle complex query logic. CTAS (CREATE TABLE AS SELECT) materializes intermediate results for reuse. Temporary tables provide session-scoped scratch space. Stored procedures encapsulate multi-step ETL logic into reusable, schedulable units.

Key Insight

In ad-tech ETL, the most common transformation pattern is the incremental daily load: delete today's partition from the target table, then insert the transformed result. This pattern is idempotent (safe to re-run) and efficient (processes only one day of data). Wrap it in a transaction for atomicity.

Key Concepts

1. Common Table Expressions (CTEs)

CTEs break complex transformations into named, readable steps. They are essential for multi-stage ad-tech calculations like attribution windows and funnel analysis:

sql
-- Multi-step campaign performance calculation using CTEs
WITH impression_metrics AS (
    SELECT
        campaign_id,
        event_date,
        COUNT(*) AS impressions,
        SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks,
        SUM(win_price_cpm) / 1000.0 AS spend
    FROM analytics.fact_impressions
    WHERE event_date = '2025-01-15'
    GROUP BY campaign_id, event_date
),
with_derived AS (
    SELECT *,
        ROUND(clicks * 100.0 / NULLIF(impressions, 0), 4) AS ctr,
        ROUND(spend / NULLIF(clicks, 0), 2) AS cpc,
        ROUND(spend / NULLIF(impressions, 0) * 1000, 2) AS effective_cpm
    FROM impression_metrics
)
SELECT wd.*, dc.campaign_name, dc.advertiser_name
FROM with_derived wd
JOIN analytics.dim_campaign dc ON wd.campaign_id = dc.campaign_id
ORDER BY spend DESC;

2. CREATE TABLE AS SELECT (CTAS)

CTAS creates a new table from a query result. It is the fastest way to materialize transformed data because it bypasses the leader node for data distribution and inherits optimal compression automatically:

sql
-- Materialize daily summary as a new table
CREATE TABLE analytics.fact_daily_summary
DISTSTYLE KEY DISTKEY(campaign_id)
COMPOUND SORTKEY(event_date, campaign_id)
AS
SELECT
    event_date,
    campaign_id,
    COUNT(*) AS impressions,
    SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(win_price_cpm) / 1000.0 AS spend,
    AVG(win_price_cpm) AS avg_cpm
FROM analytics.fact_impressions
GROUP BY event_date, campaign_id;

3. Temporary Tables

Temporary tables exist only for the duration of your session. They are ideal for staging intermediate results during multi-step ETL transformations:

sql
-- Stage cleaned impression data before final insert
CREATE TEMP TABLE stg_impressions AS
SELECT
    event_timestamp,
    event_date,
    campaign_id,
    creative_id,
    UPPER(TRIM(country_code)) AS country_code,
    COALESCE(device_type, 'unknown') AS device_type,
    ABS(win_price_cpm) AS win_price_cpm
FROM analytics.raw_impressions
WHERE event_date = '2025-01-15'
  AND campaign_id IS NOT NULL
  AND win_price_cpm >= 0;

4. Stored Procedures for ETL

Stored procedures encapsulate multi-step ETL logic into reusable, schedulable units. They support variables, conditionals, loops, and transaction control:

sql
CREATE OR REPLACE PROCEDURE analytics.load_daily_impressions(
    p_load_date DATE
)
LANGUAGE plpgsql
AS $$
BEGIN
    -- Step 1: Delete existing data for idempotency
    DELETE FROM analytics.fact_impressions
    WHERE event_date = p_load_date;

    -- Step 2: Insert transformed data
    INSERT INTO analytics.fact_impressions
    SELECT
        event_timestamp, event_date, campaign_id,
        creative_id, publisher_id, device_type,
        country_code, bid_price_cpm, win_price_cpm,
        is_viewable, is_clicked
    FROM analytics.stg_raw_impressions
    WHERE event_date = p_load_date;

    -- Step 3: Update statistics
    ANALYZE analytics.fact_impressions;

    RAISE INFO 'Loaded impressions for %', p_load_date;
END;
$$;

-- Execute the procedure
CALL analytics.load_daily_impressions('2025-01-15');
Pro Tip

Always make stored procedures idempotent by deleting the target partition before inserting. This means you can safely re-run a failed load without creating duplicates. In ad-tech billing, duplicate impressions directly cause overbilling — idempotency is not optional.

Self-Check

Before moving to the next chapter, honestly answer these questions.

When should you use a CTE vs a temp table vs CTAS?

Use CTEs for readability within a single query — they do not persist data. Use temp tables when you need to stage data for multiple queries within a session (e.g., cleaning data, then running quality checks, then inserting). Use CTAS when you need a permanent materialized table (e.g., daily summary fact). CTAS is fastest because it parallelizes the write across all slices.

Why is idempotency critical for ad-tech ETL?

Ad-tech impression data directly drives billing. If a daily load is retried (due to timeout, network error, or partial failure), a non-idempotent INSERT would create duplicate impressions. Duplicate impressions mean advertisers are billed twice for the same ad view. The delete-then-insert pattern in a transaction ensures exactly-once loading.

What are the advantages of stored procedures over external ETL scripts?

Stored procedures run inside Redshift (no data transfer overhead), support transaction control (COMMIT/ROLLBACK), can use variables and conditionals, and are callable from any scheduler (AWS Step Functions, Airflow, cron). External scripts must round-trip data through the leader node. For transformations that do not leave Redshift, stored procedures are 2-10x faster.
