Chapter 07 · Redshift Foundations

Data Transformation

Master the art of reshaping, cleaning, and enriching raw data inside Amazon Redshift — from type casting and string manipulation to pivoting, unpivoting, and building incremental transformation pipelines that turn raw event logs into analytics-ready tables.

Estimated: 55–70 min
6 Key Concepts
3 Assignments

Why This Matters

Raw data is never ready for analysis. Impression logs arrive with inconsistent timestamp formats, nullable fields where business logic demands defaults, string-encoded numbers, and nested JSON payloads that must be flattened into relational columns. Before a single dashboard chart can be drawn or a campaign performance report generated, the data must be transformed.

In Redshift, data transformation is not a separate tool or service — it happens directly inside the warehouse using SQL. This is both a strength and a responsibility. Redshift gives you powerful built-in functions for type casting, string manipulation, date arithmetic, conditional logic, and JSON extraction. But writing transformations that are correct, performant, and maintainable requires understanding the specific functions Redshift provides and the patterns that work best with its columnar, MPP architecture.

This chapter teaches you to build transformation logic that turns raw staging tables into clean, enriched, analytics-ready fact and dimension tables. You will learn to handle every common data quality issue encountered in ad-tech pipelines: missing values, format mismatches, duplicate records, and schema evolution.

Key Insight

The goal of data transformation is not just to make data "look right." It is to encode business rules into repeatable SQL logic so that every analyst, every dashboard, and every downstream model sees the same consistent view of the data. A campaign's click-through rate must be calculated the same way everywhere, or your organization will spend more time arguing about numbers than acting on them.

Key Concepts

1. Type Casting and Data Coercion

Data loaded from CSV files, S3 JSON payloads, or external Spectrum tables often arrives as strings. Before you can perform arithmetic, date filtering, or aggregation, you must cast values to their correct data types. Redshift supports several casting syntaxes, but they behave differently when encountering invalid data.

The CAST() function and the :: shorthand will raise an error if the value cannot be converted. For resilient ETL pipelines, you need defensive strategies that handle bad data gracefully without killing the entire transformation.

sql 
-- Basic type casting syntaxes in Redshift
SELECT
    -- CAST function (ANSI standard)
    CAST(bid_price_str AS DECIMAL(10,4))   AS bid_price,

    -- Double-colon shorthand (PostgreSQL style)
    event_ts_str::TIMESTAMP                    AS event_timestamp,

    -- Safe casting with CASE to prevent errors on bad data
    CASE
        WHEN impressions_str ~ '^[0-9]+$'
        THEN impressions_str::INTEGER
        ELSE 0
    END                                         AS impressions,

    -- NVL for null handling with default values
    NVL(country_code, 'XX')                   AS country_code,

    -- COALESCE for multi-column fallback chains
    COALESCE(win_price, bid_price, 0.00)      AS effective_price

FROM staging.raw_impressions;
Important

Redshift does not have a TRY_CAST() function like SQL Server or Snowflake. If you need safe casting, you must use a CASE expression with regex validation (~ operator) to test the string before casting. Always build this defensive pattern into your staging-to-production transformation queries to prevent pipeline failures.

2. String Manipulation and Cleaning

Ad-tech data is notoriously inconsistent. Publisher names arrive in mixed case, URLs contain tracking parameters that must be stripped, creative identifiers embed metadata that needs extraction, and user-agent strings require parsing. Redshift provides a comprehensive set of string functions to handle all of these transformations efficiently.

sql 
-- String transformation functions for ad-tech data cleaning
SELECT
    -- Standardize case for consistent grouping
    LOWER(TRIM(publisher_name))                AS publisher_clean,

    -- Extract domain from full URL
    REGEXP_SUBSTR(
        page_url,
        'https?://([^/]+)',
        1, 1, 'e'
    )                                            AS domain,

    -- Replace characters for normalization
    REPLACE(
        REPLACE(device_model, ' ', '-'),
        '_', '-'
    )                                            AS device_normalized,

    -- Extract creative size from tag "banner_300x250_v2"
    REGEXP_SUBSTR(
        creative_tag,
        '(\d+x\d+)',
        1, 1, 'e'
    )                                            AS creative_size,

    -- Pad IDs to consistent length for display
    LPAD(campaign_id::VARCHAR, 8, '0')          AS campaign_display_id,

    -- Build composite key from multiple fields
    advertiser_id::VARCHAR || '-' ||
    campaign_id::VARCHAR || '-' ||
    TO_CHAR(event_date, 'YYYYMMDD')              AS composite_key,

    -- Substring for fixed-width encoded fields
    SUBSTRING(bid_request_id, 1, 8)             AS exchange_prefix,
    SUBSTRING(bid_request_id, 10, 16)            AS auction_id

FROM staging.raw_impressions
WHERE LEN(page_url) > 0;
Pro Tip

Redshift's REGEXP_SUBSTR with the 'e' flag extracts only the captured group (the part inside parentheses), not the entire match. This differs from standard regex behavior and is extremely useful for parsing URLs and creative tags. Always use the 'e' flag when you need to extract a sub-pattern.

3. Date and Timestamp Transformations

Time is the most important dimension in ad-tech analytics. Every report is time-bound — daily campaign performance, hourly bid analysis, weekly trend reports. Raw event timestamps arrive in UTC epoch milliseconds, ISO 8601 strings, or inconsistent timezone formats. Transforming these into usable date and time columns is a critical ETL skill.

sql 
-- Date and timestamp transformation patterns
SELECT
    -- Convert epoch milliseconds to timestamp
    TIMESTAMP 'epoch' +
        (event_epoch_ms / 1000) * INTERVAL '1 second'
                                                 AS event_timestamp_utc,

    -- Convert UTC to Eastern for US advertiser reporting
    CONVERT_TIMEZONE(
        'UTC',
        'US/Eastern',
        event_timestamp
    )                                            AS event_timestamp_et,

    -- Truncate to various granularities
    DATE_TRUNC('hour', event_timestamp)          AS event_hour,
    DATE_TRUNC('day', event_timestamp)           AS event_date,
    DATE_TRUNC('week', event_timestamp)          AS event_week,

    -- Extract individual date components
    EXTRACT(YEAR FROM event_timestamp)          AS event_year,
    EXTRACT(MONTH FROM event_timestamp)         AS event_month,
    EXTRACT(DOW FROM event_timestamp)           AS day_of_week,
    EXTRACT(HOUR FROM event_timestamp)          AS hour_of_day,

    -- Campaign duration in days
    DATEDIFF('day', campaign_start, campaign_end) AS campaign_days,

    -- Format for reporting output
    TO_CHAR(event_timestamp, 'YYYY-MM-DD HH24:MI') AS formatted_ts,

    -- Flag weekend traffic for bid optimization
    CASE
        WHEN EXTRACT(DOW FROM event_timestamp) IN (0, 6)
        THEN 'weekend'
        ELSE 'weekday'
    END                                          AS day_type

FROM staging.raw_bid_events;

The CONVERT_TIMEZONE function is critical for multi-region advertisers. Bid events are recorded in UTC by exchanges, but US advertisers expect Eastern Time, European advertisers expect CET, and APAC advertisers expect their local zones. Build timezone conversion into your transformation layer so downstream consumers never need to think about time zones.

Important

Redshift stores TIMESTAMP without timezone information. If your source column is TIMESTAMPTZ, Redshift silently converts it to UTC during storage. Always be explicit: store raw timestamps in UTC, and apply CONVERT_TIMEZONE at the transformation layer for reporting. Document your timezone policy in the schema comments.

4. Conditional Logic and Business Rules

Real-world data transformation is not just about changing data types. It is about encoding business rules into SQL. What counts as a valid impression? How do you classify traffic quality? What revenue model applies to each campaign? These rules are implemented using CASE expressions, DECODE, and conditional aggregation patterns.

sql 
-- Business rule transformations for ad-tech metrics
SELECT
    campaign_id,
    event_date,

    -- Classify traffic quality based on viewability
    CASE
        WHEN viewability_pct >= 70 THEN 'premium'
        WHEN viewability_pct >= 50 THEN 'standard'
        WHEN viewability_pct >= 30 THEN 'below_standard'
        WHEN viewability_pct IS NULL THEN 'unmeasured'
        ELSE 'non_viewable'
    END                                      AS viewability_tier,

    -- Revenue calculation based on pricing model
    CASE pricing_model
        WHEN 'CPM'  THEN impressions * bid_price / 1000.0
        WHEN 'CPC'  THEN clicks * bid_price
        WHEN 'CPA'  THEN conversions * bid_price
        WHEN 'CPCV' THEN completed_views * bid_price
        ELSE 0
    END                                      AS gross_revenue,

    -- Flag suspicious activity for fraud detection
    CASE
        WHEN clicks > impressions          THEN TRUE
        WHEN time_on_page_ms < 100         THEN TRUE
        WHEN ip_frequency_1h > 50          THEN TRUE
        ELSE FALSE
    END                                      AS is_suspicious,

    -- DECODE for simple value mapping (Redshift-specific)
    DECODE(device_type_code,
        1, 'Desktop',
        2, 'Mobile',
        3, 'Tablet',
        4, 'Connected TV',
        5, 'Console',
        'Unknown'
    )                                        AS device_type_name

FROM staging.enriched_events;
sql 
-- Conditional aggregation: multiple metrics in one pass
SELECT
    campaign_id,
    event_date,
    COUNT(*)                                         AS total_impressions,
    SUM(CASE WHEN is_viewable THEN 1 ELSE 0 END)     AS viewable_impressions,
    SUM(CASE WHEN is_clicked  THEN 1 ELSE 0 END)     AS clicks,
    SUM(CASE WHEN converted   THEN 1 ELSE 0 END)     AS conversions,
    SUM(CASE
        WHEN device_type = 'Mobile' THEN 1 ELSE 0
    END)                                             AS mobile_impressions,
    ROUND(
        100.0 * SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END)
        / NULLIF(COUNT(*), 0),
    4)                                               AS ctr_pct,
    ROUND(AVG(win_price_cpm), 4)                     AS avg_cpm
FROM analytics.fact_impressions
GROUP BY campaign_id, event_date;
Pro Tip

Redshift's DECODE function is more concise than CASE for simple value lookups and can be slightly faster because it is optimized internally. Use DECODE for direct value mapping, and reserve CASE for range comparisons and complex multi-condition logic.

5. JSON Parsing and Semi-Structured Data

Many ad-tech data feeds deliver event payloads as JSON strings stored in a single VARCHAR(MAX) column. Bid request objects from OpenRTB, attribution payloads from mobile measurement partners, and event metadata from tag managers all arrive as JSON. Redshift provides functions to parse and flatten this semi-structured data into relational columns suitable for analysis.

sql 
-- Extract fields from JSON payloads in Redshift
SELECT
    event_id,

    -- Top-level string field extraction
    JSON_EXTRACT_PATH_TEXT(
        payload, 'device', 'ua'
    )                                          AS user_agent,

    -- Nested numeric value with type casting
    JSON_EXTRACT_PATH_TEXT(
        payload, 'bid', 'price'
    )::DECIMAL(10,4)                           AS bid_price,

    -- Geographic data from deeply nested objects
    JSON_EXTRACT_PATH_TEXT(
        payload, 'device', 'geo', 'country'
    )                                          AS country_code,

    JSON_EXTRACT_PATH_TEXT(
        payload, 'device', 'geo', 'city'
    )                                          AS city_name,

    -- Safe extraction with NULLIF for missing keys
    NVL(
        NULLIF(
            JSON_EXTRACT_PATH_TEXT(
                payload, 'app', 'bundle'
            ), ''
        ),
        'unknown_bundle'
    )                                          AS app_bundle,

    -- Boolean extraction from JSON
    CASE
        WHEN JSON_EXTRACT_PATH_TEXT(
            payload, 'regs', 'gdpr'
        ) = '1'
        THEN TRUE
        ELSE FALSE
    END                                        AS gdpr_applies,

    -- Count array elements in JSON
    JSON_ARRAY_LENGTH(
        JSON_EXTRACT_PATH_TEXT(payload, 'imp')
    )                                          AS num_imp_slots

FROM staging.raw_bid_requests
WHERE IS_VALID_JSON(payload);
Watch Out

JSON_EXTRACT_PATH_TEXT returns an empty string ('') — not NULL — when a key is missing. This is a common source of bugs. Always wrap extractions in NULLIF(value, '') if you need proper NULL semantics for downstream COALESCE or NVL logic.

6. CTAS, INSERT-SELECT, and Incremental Transforms

Writing transformation queries is only half the job. You also need to materialize the results into destination tables. Redshift provides two primary mechanisms: CREATE TABLE AS SELECT (CTAS) for creating new tables from query results, and INSERT INTO ... SELECT for appending to existing tables. Each has important implications for distribution, sorting, and compression that affect downstream query performance.

sql 
-- CTAS: Create a transformed table with physical design
CREATE TABLE analytics.fact_daily_metrics
DISTSTYLE KEY
DISTKEY (campaign_id)
SORTKEY (event_date, campaign_id)
AS
SELECT
    fi.event_date,
    fi.campaign_id,
    dc.campaign_name,
    dc.advertiser_id,
    dc.advertiser_name,
    COUNT(*)                                       AS impressions,
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN fi.is_viewable THEN 1 ELSE 0 END) AS viewable,
    ROUND(AVG(fi.win_price_cpm), 4)                AS avg_cpm,
    ROUND(SUM(fi.win_price_cpm) / 1000.0, 2)       AS total_spend,
    CURRENT_TIMESTAMP                              AS etl_loaded_at
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc
    ON fi.campaign_id = dc.campaign_id
GROUP BY 1, 2, 3, 4, 5;
sql 
-- Incremental transform: idempotent DELETE + INSERT
BEGIN;

-- Step 1: Remove stale data for the target partition
DELETE FROM analytics.fact_daily_metrics
WHERE event_date = CURRENT_DATE - 1;

-- Step 2: Insert fresh transformed data
INSERT INTO analytics.fact_daily_metrics
SELECT
    fi.event_date,
    fi.campaign_id,
    dc.campaign_name,
    dc.advertiser_id,
    dc.advertiser_name,
    COUNT(*),
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END),
    SUM(CASE WHEN fi.is_viewable THEN 1 ELSE 0 END),
    ROUND(AVG(fi.win_price_cpm), 4),
    ROUND(SUM(fi.win_price_cpm) / 1000.0, 2),
    CURRENT_TIMESTAMP
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc
    ON fi.campaign_id = dc.campaign_id
WHERE fi.event_date = CURRENT_DATE - 1
GROUP BY 1, 2, 3, 4, 5;

COMMIT;

-- Step 3: Update table statistics after significant INSERT
ANALYZE analytics.fact_daily_metrics;
Important

The DELETE + INSERT pattern (sometimes called "upsert by partition") is the idiomatic Redshift approach for incremental transformations. Redshift does not support MERGE as a single atomic statement. Always wrap DELETE and INSERT in an explicit transaction (BEGIN; ... COMMIT;) to ensure atomicity — if the INSERT fails, the DELETE also rolls back, preventing data loss.

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is unclear, revisit the relevant section.

Why does Redshift lack TRY_CAST, and how do you safely cast data types?

Redshift's PostgreSQL heritage does not include TRY_CAST. To safely cast, use a CASE expression that first validates the string format using the regex match operator (~). For example, check that a string matches '^[0-9]+$' before casting to INTEGER. For dates, verify the format with REGEXP_INSTR before casting. This pattern is essential for ETL pipelines that must not fail on bad records.

What is the difference between NVL, COALESCE, and NULLIF?

NVL(expr, default) returns the default if expr is NULL — exactly two arguments. COALESCE(expr1, expr2, ..., exprN) returns the first non-NULL value from a list — use for fallback chains. NULLIF(expr1, expr2) returns NULL if both are equal, otherwise returns expr1 — use to convert sentinel values (empty strings, zeros) back to NULL for proper aggregate behavior.

Why must you wrap DELETE+INSERT in a transaction for incremental transforms?

Without an explicit transaction, if the INSERT fails after the DELETE commits, you lose data for that partition permanently. By wrapping both in BEGIN; ... COMMIT;, if the INSERT fails the entire transaction rolls back, preserving original data. This atomicity guarantee is critical for production data pipelines where data loss is unacceptable.

How does JSON_EXTRACT_PATH_TEXT handle missing keys differently from NULL?

JSON_EXTRACT_PATH_TEXT returns an empty string ('') when a key does not exist, not NULL. This means NVL(...) alone will not trigger the fallback. You must use NVL(NULLIF(JSON_EXTRACT_PATH_TEXT(...), ''), 'default') to properly handle missing JSON keys with default values.
