Chapter 08 · Redshift Foundations

Visualization and Reporting

Connect Amazon Redshift to business intelligence tools, design SQL-backed reporting views, build UNLOAD pipelines for data delivery, and create materialized layers that power real-time ad-tech dashboards with sub-second query response times.

Estimated: 50–65 min
5 Key Concepts
3 Assignments

Why This Matters

A data warehouse is only as valuable as the decisions it enables. You can have perfectly modeled schemas, optimized distribution keys, and blazing-fast queries — but none of it matters if campaign managers cannot see their performance data in a dashboard, if finance teams cannot generate billing reconciliation reports, or if the executive team cannot monitor media spend in real time.

Visualization and reporting is the final mile of the data pipeline. It is where raw numbers become actionable insights. In ad-tech, this means connecting Redshift to tools like Amazon QuickSight, Tableau, Looker, or Mode Analytics, designing reporting views that are both fast and flexible, and building data export pipelines for clients who consume data outside your BI platform.

This chapter completes the Redshift Foundations track by teaching you to build the reporting layer that sits on top of everything you have learned. You will create SQL views optimized for BI tools, design materialized reporting tables for dashboard performance, configure UNLOAD operations for data delivery, and understand the access patterns that make or break a production reporting system.

Key Insight

The reporting layer should never expose raw tables to BI tools directly. Always create an intermediate layer — views or materialized tables — that encodes business logic, handles naming conventions, and controls what data each user group can access. This semantic layer is the contract between the data warehouse and every consumer of the data.

Key Concepts

1. Reporting Views and the Semantic Layer

A reporting view is a SQL CREATE VIEW that presents clean, business-friendly column names and pre-computed metrics to downstream BI tools. Views do not store data — they are named queries that execute when accessed. For ad-tech reporting, views serve as the contract between your data engineering team and the analysts who build dashboards.

sql 
-- Reporting view: Campaign Performance Summary
-- Used by: Tableau, QuickSight, Looker dashboards
CREATE OR REPLACE VIEW reporting.v_campaign_performance AS
SELECT
    -- Dimensions with business-friendly names
    fm.event_date                              AS report_date,
    dc.advertiser_name                         AS advertiser,
    dc.campaign_name                           AS campaign,
    dc.campaign_status                         AS status,
    dd.device_category                         AS device,
    dg.country_name                            AS country,
    dg.region_name                             AS region,

    -- Core delivery metrics
    fm.impressions,
    fm.clicks,
    fm.viewable_impressions                    AS viewable_imps,
    fm.conversions,

    -- Computed KPIs with safe division
    ROUND(
        100.0 * fm.clicks /
        NULLIF(fm.impressions, 0), 4
    )                                          AS ctr_pct,

    ROUND(
        100.0 * fm.viewable_impressions /
        NULLIF(fm.impressions, 0), 2
    )                                          AS viewability_pct,

    ROUND(
        100.0 * fm.conversions /
        NULLIF(fm.clicks, 0), 4
    )                                          AS conversion_rate_pct,

    -- Financial metrics
    ROUND(fm.total_spend, 2)                   AS spend_usd,
    ROUND(fm.avg_cpm, 2)                       AS avg_cpm,
    ROUND(
        fm.total_spend /
        NULLIF(fm.clicks, 0), 2
    )                                          AS cost_per_click,
    ROUND(
        fm.total_spend /
        NULLIF(fm.conversions, 0), 2
    )                                          AS cost_per_acquisition

FROM analytics.fact_daily_metrics fm
JOIN analytics.dim_campaign dc
    ON fm.campaign_id = dc.campaign_id
LEFT JOIN analytics.dim_device dd
    ON fm.device_type_id = dd.device_type_id
LEFT JOIN analytics.dim_geography dg
    ON fm.geo_id = dg.geo_id
WHERE fm.event_date >= DATEADD('day', -90, CURRENT_DATE);
Pro Tip

Always use NULLIF(denominator, 0) in rate calculations. A campaign with zero impressions will produce a division-by-zero error without this guard. BI tools display NULL as "N/A" or blank, which is the correct business behavior for a campaign that has not yet delivered.

2. Materialized Reporting Tables

Views execute their underlying query every time they are accessed. For dashboards refreshed frequently by many analysts, this creates significant cluster load. Materialized reporting tables pre-compute results and store them physically, trading data freshness for speed.

sql 
-- Materialized table for executive dashboard
CREATE TABLE reporting.exec_campaign_summary (
    report_date       DATE           ENCODE az64,
    advertiser        VARCHAR(256)   ENCODE lzo,
    campaign          VARCHAR(512)   ENCODE lzo,
    impressions       BIGINT         ENCODE az64,
    clicks            BIGINT         ENCODE az64,
    conversions       INTEGER        ENCODE az64,
    spend_usd         DECIMAL(12,2) ENCODE az64,
    ctr_pct           DECIMAL(8,4)  ENCODE az64,
    cpa_usd           DECIMAL(10,2) ENCODE az64,
    refreshed_at      TIMESTAMP      ENCODE az64
)
DISTSTYLE ALL
SORTKEY (report_date, advertiser);

-- Refresh procedure: TRUNCATE + INSERT in transaction
BEGIN;
TRUNCATE reporting.exec_campaign_summary;
INSERT INTO reporting.exec_campaign_summary
SELECT
    event_date,
    advertiser_name,
    campaign_name,
    SUM(impressions),
    SUM(clicks),
    SUM(conversions),
    ROUND(SUM(total_spend), 2),
    ROUND(100.0 * SUM(clicks) / NULLIF(SUM(impressions), 0), 4),
    ROUND(SUM(total_spend) / NULLIF(SUM(conversions), 0), 2),
    CURRENT_TIMESTAMP
FROM analytics.fact_daily_metrics
WHERE event_date >= DATEADD('day', -30, CURRENT_DATE)
GROUP BY 1, 2, 3;
COMMIT;
ANALYZE reporting.exec_campaign_summary;
Important

Use DISTSTYLE ALL for small materialized reporting tables (under a few million rows). This replicates data to every compute node, so BI queries that join reporting tables never require network redistribution. The storage cost is minimal for aggregated data, and the query speed improvement is substantial.

3. UNLOAD for Data Export and Client Delivery

Not all data consumers connect to Redshift directly. Many ad-tech clients expect daily CSV or Parquet files delivered to their S3 buckets. The UNLOAD command exports query results directly to Amazon S3 in parallel, leveraging Redshift's MPP architecture for high-throughput data delivery.

sql 
-- UNLOAD to S3 as Parquet for data lake consumption
UNLOAD ('
    SELECT
        event_date,
        campaign_name,
        advertiser_name,
        impressions,
        clicks,
        conversions,
        spend_usd,
        ctr_pct,
        cost_per_click
    FROM reporting.v_campaign_performance
    WHERE report_date >= DATEADD(''day'', -7, CURRENT_DATE)
    ORDER BY report_date, advertiser, campaign
')
TO 's3://my-adtech-reports/weekly/campaign_perf_'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
FORMAT AS PARQUET
ALLOWOVERWRITE
PARALLEL ON
MAXFILESIZE 256 MB;
sql 
-- UNLOAD as single CSV with headers for client delivery
UNLOAD ('
    SELECT report_date, campaign, impressions, clicks, spend_usd
    FROM reporting.exec_campaign_summary
    WHERE advertiser = ''Acme Corp''
      AND report_date >= ''2025-01-01''
    ORDER BY report_date
')
TO 's3://client-reports/acme/monthly_2025-01_'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftUnloadRole'
CSV
HEADER
PARALLEL OFF
ALLOWOVERWRITE;
Pro Tip

Use PARALLEL OFF for human-readable client deliverables (single file for Excel). Use PARALLEL ON for data lake exports where downstream Spark or Athena jobs benefit from reading multiple files simultaneously.

4. Connecting BI Tools to Redshift

Every major BI tool supports Redshift natively. The connection uses the PostgreSQL wire protocol (JDBC/ODBC on port 5439). However, there are important configuration considerations that affect dashboard performance, user experience, and security.

sql 
-- Create dedicated BI user with read-only access
CREATE USER bi_dashboard_user
PASSWORD 'StrongP@ss_2025!'
CONNECTION LIMIT 10;

-- Create reporting schema and grant appropriate permissions
CREATE SCHEMA IF NOT EXISTS reporting;
GRANT USAGE ON SCHEMA reporting TO bi_dashboard_user;
GRANT SELECT ON ALL TABLES IN SCHEMA reporting
    TO bi_dashboard_user;

-- Ensure future tables/views are also accessible
ALTER DEFAULT PRIVILEGES IN SCHEMA reporting
    GRANT SELECT ON TABLES TO bi_dashboard_user;

-- Create a group for all BI users
CREATE GROUP bi_reporting_group;
ALTER GROUP bi_reporting_group ADD USER bi_dashboard_user;

5. Scheduled Refreshes and Data Quality Gates

Production reporting requires automated refresh cycles. Materialized tables must be updated on schedule, data quality checks must run before dashboards go live, and UNLOAD exports must fire at predictable times. Redshift integrates with AWS scheduling services for automation.

sql 
-- Scheduled refresh with data quality gate
BEGIN;

-- Refresh last 3 days (handles late-arriving data)
DELETE FROM reporting.exec_campaign_summary
WHERE report_date >= CURRENT_DATE - 3;

INSERT INTO reporting.exec_campaign_summary
SELECT
    event_date, advertiser_name, campaign_name,
    SUM(impressions), SUM(clicks), SUM(conversions),
    ROUND(SUM(total_spend), 2),
    ROUND(100.0 * SUM(clicks) / NULLIF(SUM(impressions), 0), 4),
    ROUND(SUM(total_spend) / NULLIF(SUM(conversions), 0), 2),
    CURRENT_TIMESTAMP
FROM analytics.fact_daily_metrics
WHERE event_date >= CURRENT_DATE - 3
GROUP BY 1, 2, 3;

COMMIT;

-- Data quality gate: fail if yesterday has no data
SELECT
    CASE
        WHEN COUNT(*) = 0 THEN
            1/0  -- Force error to trigger alert
        ELSE 1
    END AS dq_check
FROM reporting.exec_campaign_summary
WHERE report_date = CURRENT_DATE - 1;

ANALYZE reporting.exec_campaign_summary;
Watch Out

Always include a data quality check after refreshing reporting tables. If upstream ETL failed and yesterday's data has not arrived, the reporting table should NOT silently show zero rows. Dashboards showing stale data are worse than dashboards showing an explicit "data unavailable" warning. Fail loudly and trigger alerts.

Self-Check

Congratulations on completing the Redshift Foundations track. Before moving to the Advanced track, verify your understanding.

Why should BI tools query reporting views instead of analytics tables directly?

Reporting views provide a semantic layer that abstracts the physical schema. They encode business logic (KPI calculations, naming conventions, access control) in a single place. If the underlying schema changes, you update the view once instead of every dashboard. Views also restrict which columns and rows are visible, enforcing data governance without relying on BI tool configuration.

When should you use a materialized table versus a view for reporting?

Use a view when the underlying query executes quickly (under 5 seconds) and data freshness is critical. Use a materialized table when queries involve complex joins or aggregations taking more than 5 seconds, or when many concurrent users access the same data. The tradeoff is freshness: materialized tables are only as current as their last refresh.

What is the difference between PARALLEL ON and PARALLEL OFF in UNLOAD?

PARALLEL ON (default) writes multiple files simultaneously, one per slice. This is faster and ideal for data lake consumption by Spark or Athena. PARALLEL OFF writes a single file, better for client-facing CSV delivery where humans expect one file they can open in Excel.

How do you prevent BI queries from impacting ETL performance?

Use Workload Management (WLM) to create separate queues. Assign BI users to a queue with higher concurrency but lower memory and a timeout limit. Assign ETL users to a queue with lower concurrency but higher memory. This ensures runaway dashboard queries cannot block critical pipelines. Use Concurrency Scaling for burst BI demand.
