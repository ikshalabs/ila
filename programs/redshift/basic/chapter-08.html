Chapter 08 · Redshift Basic

Visualization and Reporting

Connect BI tools to Redshift, design dashboard-optimized queries, implement materialized views for sub-second response times, and build the reporting layer that campaign managers and advertisers interact with daily.

Estimated: 60–90 min
Key Concepts
3 Assignments

Why This Matters

A data warehouse is only as valuable as the insights people extract from it. In ad-tech, the end users are campaign managers who need real-time spend monitoring, advertisers who want weekly performance reports, and data scientists who build bidding optimization models. The visualization and reporting layer is where all your schema design, distribution keys, and ETL pipelines come together to deliver business value.

The key challenge is making interactive dashboards fast enough for users who expect sub-3-second response times while simultaneously supporting heavy analytical queries that scan billions of rows. Materialized views, pre-aggregated summary tables, and smart query design are the tools that make this possible.

Key Insight

The #1 mistake in ad-tech BI is connecting Tableau or Looker directly to the raw fact_impressions table with billions of rows. Every dashboard interaction becomes a full table scan. Instead, create materialized views and summary tables that pre-aggregate data to the grain your dashboards need. A daily campaign summary with 50K rows renders instantly; a billion-row impression table does not.

Key Concepts

1. Connecting BI Tools to Redshift

Redshift uses the PostgreSQL wire protocol, which means virtually every BI tool supports it natively. Common connections include:

  • Looker — Uses JDBC/ODBC. Define LookML models that map to your star schema. Use PDTs (Persistent Derived Tables) for pre-aggregation.
  • Tableau — Native Redshift connector. Use extracts for dashboards and live connections for ad-hoc exploration.
  • Mode Analytics — SQL-first interface ideal for ad-tech analysts who prefer writing queries. Connects via JDBC.
  • Amazon QuickSight — Native AWS integration with SPICE in-memory engine for fast dashboards.
sql
-- Create a reporting user with read-only access
CREATE USER bi_reader PASSWORD 'StrongP@ssw0rd!';
GRANT USAGE ON SCHEMA analytics TO bi_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics TO bi_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA analytics
    GRANT SELECT ON TABLES TO bi_reader;

2. Materialized Views

Materialized views store the result of a query physically on disk and can be refreshed on demand. They are the primary tool for making dashboards fast:

sql
-- Materialized view for the campaign performance dashboard
CREATE MATERIALIZED VIEW analytics.mv_daily_campaign_performance AS
SELECT
    fi.event_date,
    dc.campaign_id,
    dc.campaign_name,
    dc.advertiser_name,
    dc.campaign_type,
    COUNT(*) AS impressions,
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN fi.is_viewable THEN 1 ELSE 0 END) AS viewable_impressions,
    SUM(fi.win_price_cpm) / 1000.0 AS spend,
    ROUND(AVG(fi.win_price_cpm), 2) AS avg_cpm
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc ON fi.campaign_id = dc.campaign_id
GROUP BY fi.event_date, dc.campaign_id, dc.campaign_name,
         dc.advertiser_name, dc.campaign_type;

-- Refresh after daily ETL completes
REFRESH MATERIALIZED VIEW analytics.mv_daily_campaign_performance;

3. Dashboard-Optimized Queries

Dashboard queries must be designed for speed. Key patterns include querying materialized views instead of raw facts, using pre-computed date dimensions, and limiting result sets:

sql
-- Dashboard query: campaign performance for selected date range
-- Runs against materialized view, not raw impressions
SELECT
    event_date,
    campaign_name,
    advertiser_name,
    impressions,
    clicks,
    ROUND(clicks * 100.0 / NULLIF(impressions, 0), 2) AS ctr_pct,
    spend,
    avg_cpm,
    ROUND(viewable_impressions * 100.0 / NULLIF(impressions, 0), 1) AS viewability_pct
FROM analytics.mv_daily_campaign_performance
WHERE event_date BETWEEN '2025-01-01' AND '2025-01-31'
  AND advertiser_name = 'Acme Corp'
ORDER BY event_date, spend DESC;
Pro Tip

Schedule REFRESH MATERIALIZED VIEW to run immediately after your daily ETL pipeline completes. Use Amazon EventBridge or Step Functions to orchestrate: ETL load -> ANALYZE -> REFRESH MVs -> notify dashboard users. This ensures dashboards show fresh data within minutes of the nightly load completing.

4. Common Reporting Patterns for Ad-Tech

Here are the standard report types every ad-tech platform needs:

  • Campaign Pacing Report — Daily spend vs budget, projected end-of-month spend, pace alerts.
  • Creative Performance Report — CTR and viewability by creative, A/B test comparison.
  • Publisher Quality Report — Viewability and click-fraud indicators by publisher domain.
  • Geo Performance Report — Impressions, CTR, and spend by country/region/city.
  • Day-Parting Analysis — Performance by hour-of-day and day-of-week for bid optimization.

Self-Check

Before moving to the next chapter, honestly answer these questions.

Why should BI tools query materialized views instead of raw fact tables?

Raw fact tables like fact_impressions can have billions of rows. Every dashboard interaction (filter change, date range update) triggers a full aggregation scan. Materialized views pre-compute the aggregation and store the result (typically thousands to millions of rows). Dashboard queries against MVs run in milliseconds instead of seconds/minutes.

How often should materialized views be refreshed?

Refresh frequency depends on data freshness requirements. For daily batch ETL, refresh once after the nightly load completes. For near-real-time dashboards, Redshift supports AUTO REFRESH which incrementally updates MVs as base tables change. For ad-tech, daily refresh after ETL is the standard pattern.

What is the difference between a materialized view and a summary table built with CTAS?

Both store pre-aggregated data. A materialized view can be refreshed with a single command (REFRESH MATERIALIZED VIEW) that incrementally recomputes only changed data. A CTAS table must be manually dropped and recreated, or managed with DELETE/INSERT logic. MVs are more convenient but have restrictions (limited SQL support in the defining query). Use MVs for standard reports; use CTAS for complex transformations that exceed MV limitations.
