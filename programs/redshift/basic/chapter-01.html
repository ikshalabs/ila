Chapter 01 · Redshift Foundations

Redshift Architecture

Understand the internals of Amazon Redshift — columnar storage, massively parallel processing, leader and compute nodes, slices, and how cluster topology determines query performance for ad-tech analytics at scale.

Estimated: 45–60 min
6 Key Concepts
3 Assignments

Why This Matters

In ad-tech, you are dealing with billions of impression events, click streams, and bid logs every day. A single programmatic advertising platform can generate 500 million to 2 billion events daily. Traditional row-oriented databases like PostgreSQL or MySQL simply cannot scan this volume within the sub-second latency that campaign analysts demand.

Amazon Redshift was designed specifically for this class of problem. It combines columnar storage (reading only the columns you need), massively parallel processing (distributing work across many CPU cores simultaneously), and aggressive compression (reducing I/O by 10x or more) into a managed cloud data warehouse. Understanding its architecture is not optional — it is the foundation every query, every distribution key choice, and every performance optimization builds upon.

Key Insight

The difference between a 3-second query and a 300-second query on the same data almost always comes down to how well you understand Redshift's architecture. Distribution keys, sort keys, and compression encodings are not "advanced tuning" — they are design decisions that must be made when you create the table, informed by how the engine actually works.

Key Concepts

1. Columnar Storage

Traditional databases store data row by row. When you run SELECT campaign_id, impressions FROM ad_metrics, a row-store must read the entire row — including every column you did not ask for — from disk. If your table has 40 columns but your query only needs 2, you are reading 20x more data than necessary.

Redshift stores data column by column. Each column is stored in its own set of 1 MB blocks on disk. When your query asks for campaign_id and impressions, Redshift reads only those two columns from storage, skipping the other 38 entirely. For ad-tech fact tables with wide schemas (timestamp, device type, geo, creative, bid price, win price, viewability, etc.), this reduction in I/O is enormous.

sql 
-- This query only reads 2 columns from disk, not all 40
SELECT
    campaign_id,
    SUM(impressions) AS total_impressions
FROM analytics.fact_impressions
WHERE event_date BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY campaign_id
ORDER BY total_impressions DESC
LIMIT 20;

Columnar storage also enables superior compression. Values in a single column tend to have similar data types and ranges — a column of country_code values will have far less entropy than an entire row containing strings, integers, timestamps, and floats mixed together. Redshift automatically selects the best compression encoding (LZO, Zstandard, Delta, Runlength, etc.) per column.

Pro Tip

Use ANALYZE COMPRESSION on your impression fact tables after loading sample data. Redshift will recommend the optimal encoding for each column. A well-compressed ad impressions table can achieve 4:1 to 10:1 compression ratios, dramatically reducing storage costs and scan times.

2. Massively Parallel Processing (MPP)

Redshift does not process your query on a single server. Instead, the leader node receives your SQL, creates an execution plan, compiles it into C++ code, and distributes that compiled plan to multiple compute nodes. Each compute node works on its own slice of the data in parallel. The leader node then aggregates the partial results and returns the final answer.

Think of it like splitting a 10-billion-row impressions table across 16 compute nodes. Each node scans only 625 million rows. If each node has 2 slices, each slice scans about 312 million rows. The wall-clock time is determined by the slowest slice, not the total data volume.

3. Nodes, Slices, and Node Types

Every Redshift cluster consists of one leader node and one or more compute nodes. Each compute node is further divided into slices. A slice is a partition of a node's memory and disk, and it operates as an independent processing unit. The number of slices per node depends on the node type:

  • dc2.large — 2 slices per node, 160 GB SSD. Best for datasets under 500 GB where speed is critical (hot ad-tech dashboards).
  • dc2.8xlarge — 16 slices per node, 2.56 TB SSD. High performance for medium-scale impression analytics.
  • ra3.xlplus — 2 slices per node, managed storage. Separates compute from storage — ideal for growing ad-tech data lakes.
  • ra3.4xlarge — 4 slices per node, managed storage. Balanced choice for production ad-tech workloads.
  • ra3.16xlarge — 16 slices per node, managed storage. Maximum performance for enterprise-scale bidding analytics.
sql 
-- Check your cluster's node configuration
SELECT
    node,
    type,
    num_slices
FROM stv_slices
ORDER BY node, slice;

-- Check overall cluster capacity
SELECT
    owner AS node_owner,
    host AS node_host,
    diskno,
    capacity AS capacity_mb,
    used AS used_mb
FROM stv_partitions
WHERE type = 0
ORDER BY owner, diskno;

4. The Leader Node

The leader node is the entry point for all client connections. When your BI tool (Looker, Tableau, Mode) sends a SQL query, the leader node handles the entire lifecycle:

  1. Parse — Validates SQL syntax and resolves table/column references.
  2. Plan — Creates an optimized execution plan using table statistics and distribution metadata.
  3. Compile — Translates the plan into optimized C++ code segments (this is why first-run queries are slower).
  4. Distribute — Sends compiled code segments to the relevant compute node slices.
  5. Aggregate — Collects partial results from all slices and merges them into the final result set.
Important

The leader node also handles queries that do not require compute node participation — for example, SELECT GETDATE() or queries against system catalog tables. However, if your query returns large result sets, the leader node becomes a bottleneck since all data must flow through it. Always use LIMIT or UNLOAD for large extracts rather than streaming millions of rows through the leader.

5. Cluster Setup and Configuration

Setting up a Redshift cluster for an ad-tech workload requires thinking about the data volumes you will store and the query concurrency you need to support. Here is a typical setup for a mid-market demand-side platform (DSP):

sql 
-- After cluster is provisioned via AWS Console or CLI,
-- create your ad-tech analytics schema
CREATE SCHEMA IF NOT EXISTS analytics;

-- Create a sample impression fact table
CREATE TABLE analytics.fact_impressions (
    impression_id     BIGINT        IDENTITY(1,1),
    event_timestamp   TIMESTAMP     NOT NULL ENCODE az64,
    event_date        DATE          NOT NULL ENCODE az64,
    campaign_id       INTEGER       NOT NULL ENCODE az64,
    ad_group_id       INTEGER       NOT NULL ENCODE az64,
    creative_id       INTEGER       NOT NULL ENCODE az64,
    placement_id      INTEGER       ENCODE az64,
    publisher_id      INTEGER       ENCODE az64,
    device_type       VARCHAR(20)   ENCODE bytedict,
    country_code      CHAR(2)       ENCODE bytedict,
    bid_price_cpm     DECIMAL(10,4) ENCODE az64,
    win_price_cpm     DECIMAL(10,4) ENCODE az64,
    is_viewable       BOOLEAN       ENCODE runlength,
    is_clicked        BOOLEAN       ENCODE runlength
)
DISTSTYLE KEY
DISTKEY (campaign_id)
SORTKEY (event_date, campaign_id);

Notice the explicit ENCODE directives. We are telling Redshift exactly how to compress each column based on its data characteristics: az64 for integers and timestamps (Amazon's proprietary compression), bytedict for low-cardinality strings like device type and country code, and runlength for boolean flags where consecutive values are often identical.

6. Data Flow Through the Architecture

Understanding how data moves through a Redshift cluster helps you diagnose performance issues. Here is the complete lifecycle of a typical ad-tech analytics query:

sql 
-- Analyst query: daily campaign performance for January
SELECT
    fi.event_date,
    dc.campaign_name,
    dc.advertiser_name,
    COUNT(*) AS impressions,
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    ROUND(
      SUM(CASE WHEN fi.is_clicked THEN 1.0 ELSE 0 END) /
      NULLIF(COUNT(*), 0) * 100, 4
    ) AS ctr_pct,
    ROUND(AVG(fi.win_price_cpm), 2) AS avg_cpm
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc
    ON fi.campaign_id = dc.campaign_id
WHERE fi.event_date BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY fi.event_date, dc.campaign_name, dc.advertiser_name
ORDER BY fi.event_date, impressions DESC;

Here is what happens internally:

  1. Zone map pruning — The sort key on event_date allows Redshift to skip entire 1 MB blocks that fall outside the January date range. If the table has 365 days of data, approximately 11/12ths of the blocks are never read.
  2. Column projection — Only the columns referenced in SELECT, WHERE, JOIN, and GROUP BY are read from disk. The placement_id, publisher_id, bid_price_cpm, and is_viewable columns are not touched.
  3. Parallel scan — Each slice scans its own portion of the data simultaneously. With 8 slices, the work is divided into 8 parallel streams.
  4. Hash join — If dim_campaign is distributed with DISTSTYLE ALL, each slice has a local copy, eliminating network redistribution for the join.
  5. Partial aggregation — Each slice computes its own GROUP BY aggregates locally, then sends the partial results to the leader node.
  6. Final merge — The leader node merges partial aggregates from all slices, applies the ORDER BY, and returns the result to the client.
Pro Tip

Use SVL_QUERY_SUMMARY to see exactly how many bytes were scanned, how many blocks were skipped via zone maps, and whether any redistribution (network transfer between nodes) occurred. This is the single most useful diagnostic view for ad-tech query optimization.

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer is unclear, revisit the relevant section.

What is the difference between columnar and row-based storage, and why does it matter for ad-tech queries?

Row-based storage reads entire rows from disk even if you only need a few columns. Columnar storage reads only the columns referenced in your query. For ad-tech fact tables with 20-40 columns where typical queries reference only 5-8 columns, columnar storage reduces I/O by 60-80%. This is the single biggest reason Redshift can scan billions of rows in seconds.

Can you explain what a "slice" is and how it affects query parallelism?

A slice is a partition of a compute node's memory and disk. Each slice operates as an independent processing unit. If you have 4 nodes with 4 slices each, you have 16 slices — meaning your query can run 16 parallel streams simultaneously. Data is distributed across slices based on the table's distribution key. Uneven distribution (data skew) means some slices do more work than others, making the overall query only as fast as the slowest slice.

When would you choose ra3 node types over dc2, and why?

Choose ra3 when your data volume is growing and you want to scale storage independently of compute. RA3 nodes use managed storage (backed by S3) with a local SSD cache, so you only pay for the compute you need and the storage you use. Choose dc2 when your dataset fits entirely in local SSD and you need maximum scan performance with no network latency to storage. For most ad-tech workloads with 90+ day retention, ra3 is the better long-term choice because impression data grows constantly.
