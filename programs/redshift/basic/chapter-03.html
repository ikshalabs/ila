Distribution and Sort Keys

The two most impactful performance decisions in Redshift — distribution styles control how data is spread across nodes, sort keys control how quickly the engine can skip irrelevant blocks. Get these right and queries fly; get them wrong and you waste 90% of your cluster.

Estimated: 60–90 min
5 Key Concepts
3 Assignments

Why This Matters

In a Redshift cluster with 8 compute nodes and 32 slices, every table's data must be physically distributed across those slices. The distribution key determines which slice owns which rows. When two tables share the same distribution key, JOINs between them happen locally on each slice without any network transfer. When they do not, Redshift must redistribute one or both tables across the network — a process that can make queries 10-100x slower.

Similarly, sort keys determine the physical order of rows on disk within each slice. Because Redshift stores metadata about the minimum and maximum values in each 1 MB block (called zone maps), a well-sorted table allows the engine to skip entire blocks that cannot contain matching rows. For ad-tech impression data filtered by event_date, this means scanning 1/30th of the data for a single-day query on a 30-day table.

Key Insight

Distribution and sort keys are immutable after table creation in Redshift (unless you use ALTER TABLE ... ALTER SORTKEY, or do a deep copy). This means you must design them correctly upfront. Changing them later requires creating a new table, copying all data, and renaming — a process that can take hours for multi-billion-row impression tables.

Key Concepts

1. Distribution Styles: KEY, ALL, EVEN, AUTO

Redshift offers four distribution styles. Each one determines how rows are assigned to slices:

  • KEY — Rows are distributed based on a hash of the specified column. All rows with the same key value end up on the same slice. Use this for large fact tables, distributing on the most frequently JOINed column (e.g., campaign_id).
  • ALL — A full copy of the entire table is placed on every slice. Use this for small dimension tables (under 3 million rows) like dim_campaign, dim_creative, dim_publisher. Every join becomes local.
  • EVEN — Rows are distributed round-robin across slices. Use this when no column is a good distribution key, or for staging tables. Guarantees even distribution but all joins require redistribution.
  • AUTO — Redshift chooses the style automatically. Small tables get ALL, larger tables get EVEN. This is the default but rarely optimal for ad-tech workloads where you know your join patterns.
sql
-- Fact table: distribute by the most-joined column
CREATE TABLE analytics.fact_impressions (
    impression_id   BIGINT IDENTITY(1,1),
    event_date      DATE NOT NULL,
    campaign_id     INTEGER NOT NULL,
    creative_id     INTEGER,
    publisher_id    INTEGER,
    device_type     VARCHAR(20),
    country_code    CHAR(2),
    bid_price_cpm   DECIMAL(10,4),
    win_price_cpm   DECIMAL(10,4),
    is_clicked      BOOLEAN,
    is_viewable     BOOLEAN
)
DISTSTYLE KEY
DISTKEY (campaign_id)
COMPOUND SORTKEY (event_date, campaign_id);

-- Small dimension: replicate to all slices
CREATE TABLE analytics.dim_campaign (
    campaign_id     INTEGER PRIMARY KEY,
    campaign_name   VARCHAR(256),
    advertiser_id   INTEGER,
    advertiser_name VARCHAR(256),
    campaign_status VARCHAR(20),
    start_date      DATE,
    end_date        DATE,
    daily_budget    DECIMAL(12,2)
)
DISTSTYLE ALL;

2. Sort Keys: Compound vs Interleaved

Redshift supports two types of sort keys:

  • Compound sort key — Data is sorted by the first column, then by the second within ties of the first, and so on. This is extremely effective when queries consistently filter on the leading column(s). For ad-tech data, COMPOUND SORTKEY (event_date, campaign_id) is ideal because virtually every query filters by date range first.
  • Interleaved sort key — Data is sorted equally across all key columns. This is useful when queries filter on any combination of columns without a predictable leading column. However, interleaved keys require much more maintenance (VACUUM REINDEX) and are slower to load.
Important

For 95% of ad-tech workloads, use compound sort keys. Interleaved sort keys are only justified when you have genuinely unpredictable query patterns across multiple columns and you are willing to pay the VACUUM REINDEX cost. AWS recommends compound as the default starting point.

3. Zone Maps and Block Skipping

Every 1 MB block in Redshift stores a zone map — metadata recording the minimum and maximum values in that block for each column. When you run WHERE event_date = '2025-01-15', Redshift checks each block's zone map. If a block's minimum date is 2025-02-01 and maximum is 2025-02-15, the entire block is skipped without reading a single byte from disk.

sql
-- Check zone map effectiveness: how many blocks were scanned vs skipped
SELECT
    query,
    segment,
    step,
    label,
    rows,
    bytes,
    rows_pre_filter,
    blocks_read,
    blocks_touched
FROM svl_query_summary
WHERE query = 12345  -- replace with your query ID
  AND label LIKE 'scan%'
ORDER BY segment, step;

4. Detecting and Fixing Data Skew

Data skew occurs when one distribution key value has dramatically more rows than others. If campaign_id 42 has 50% of all impressions (a retargeting mega-campaign), the slice holding that campaign does 50% of the work while other slices sit idle:

sql
-- Detect data skew across slices
SELECT
    'analytics.fact_impressions' AS table_name,
    slice,
    COUNT(*) AS row_count,
    ROUND(
        COUNT(*) * 100.0 /
        SUM(COUNT(*)) OVER (),
    2) AS pct_of_total
FROM analytics.fact_impressions
GROUP BY slice
ORDER BY row_count DESC;

-- Alternative: use system table for skew analysis
SELECT
    trim(tablename) AS table_name,
    MAX(num_values) AS max_rows_on_slice,
    MIN(num_values) AS min_rows_on_slice,
    ROUND(
        MAX(num_values)::FLOAT / NULLIF(MIN(num_values), 0),
    2) AS skew_ratio
FROM svv_diskusage
WHERE name = 'fact_impressions'
GROUP BY tablename;
Pro Tip

A skew ratio above 1.5 is a warning sign; above 4.0 is an emergency. If you see high skew on a campaign_id distribution key, consider distributing on a higher-cardinality column like ad_group_id or switching to EVEN distribution for that table.

5. Decision Framework for Ad-Tech Tables

Here is a practical decision framework for choosing distribution and sort keys for common ad-tech tables:

  • fact_impressions — DISTKEY(campaign_id), COMPOUND SORTKEY(event_date, campaign_id). Date is the leading sort because every dashboard query filters by date range.
  • fact_clicks — DISTKEY(campaign_id), COMPOUND SORTKEY(event_date, campaign_id). Colocated with impressions for fast joins.
  • fact_bid_requests — DISTKEY(campaign_id) or EVEN if bid data has extreme skew toward a few campaigns.
  • dim_campaign — DISTSTYLE ALL. Small table, every query joins to it.
  • dim_creative — DISTSTYLE ALL. Small table, frequently joined.
  • dim_publisher — DISTSTYLE ALL or KEY(publisher_id) if table exceeds 5 million rows.
  • dim_date — DISTSTYLE ALL. Always small, joined everywhere.

Self-Check

When would you choose DISTSTYLE ALL over DISTSTYLE KEY?

Use ALL for small dimension tables (under 3-5 million rows) that are frequently joined to large fact tables. ALL places a copy on every slice, making every join local. Use KEY for large fact tables where you want to colocate rows that are frequently joined together. If a dimension table grows beyond 5 million rows, switch to KEY distribution.

What happens during a "redistribution" step and why is it expensive?

Redistribution means Redshift must send rows across the network from one node to another to match join keys. This involves serializing data, sending it over the interconnect, deserializing on the receiving end, and building a hash table. For multi-billion row tables, this can transfer gigabytes of data and is often the single slowest step in a query plan. Colocation (same DISTKEY on both tables) eliminates redistribution entirely.

Why should event_date be the leading column in a compound sort key for ad-tech fact tables?

Virtually every ad-tech dashboard query includes a date range filter: "show me last 7 days" or "show me January results." When event_date is the leading sort key column, zone maps can skip all blocks outside the date range before examining any other filter. If campaign_id were the leading column, the engine could only skip blocks for single-campaign queries, which are less common than date-range queries.
