Chapter 03 · Phase 2: Query Patterns

Analytical Query Patterns

Master the SQL patterns that power analytics dashboards — window functions, CTEs, conditional aggregation, date-series generation, and approximate functions. These are the query idioms you will use daily in every Redshift data warehouse.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

Every dashboard, report, and data product in a modern analytics stack ultimately boils down to SQL queries running against your warehouse. The difference between an analyst who writes basic GROUP BY queries and one who fluently uses window functions, CTEs, and conditional aggregation is the difference between a 10-minute report and a 10-hour report. Mastering analytical SQL patterns means you can answer complex business questions in a single query rather than stitching together multiple extracts in a spreadsheet.

Redshift's SQL dialect extends PostgreSQL with powerful analytical capabilities: window functions with frame specifications, approximate count distinct via HyperLogLog, regular expressions, and date/time functions optimized for time-series data. These are not academic exercises — they are the building blocks of production dashboards that decision-makers rely on every day.

This chapter focuses on six query patterns that appear in virtually every Redshift warehouse. By the end, you should be able to write a single CTE-based query that generates a daily time series, fills gaps, computes rolling averages, and ranks results — all without leaving SQL.

Key Insight

In Redshift analytics, window functions are not optional. They are the fundamental tool for running totals (cumulative spend), ranking (top campaigns by ROAS), moving averages (7-day CTR trends), and lead/lag comparisons (day-over-day impression deltas). Every query pattern in this chapter appears in real production dashboards.

Key Concepts

1. Aggregation and Conditional Aggregation

The most common analytical query pattern is aggregating event-level data into summary metrics. Conditional aggregation using CASE inside aggregate functions lets you compute multiple metrics from a single table scan, avoiding multiple subqueries or self-joins.

sql 
-- Campaign performance report with conditional aggregation
SELECT
    campaign_id,
    event_date,
    COUNT(*) AS total_impressions,
    SUM(CASE WHEN is_viewable THEN 1 ELSE 0 END) AS viewable_impressions,
    SUM(CASE WHEN clicked THEN 1 ELSE 0 END) AS clicks,
    ROUND(
      SUM(CASE WHEN clicked THEN 1.0 ELSE 0 END) /
      NULLIF(COUNT(*), 0) * 100, 4
    ) AS ctr_pct,
    SUM(win_price) AS total_spend,
    ROUND(AVG(win_price) * 1000, 2) AS effective_cpm
FROM analytics.impressions
WHERE event_date BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY campaign_id, event_date
ORDER BY campaign_id, event_date;

The NULLIF pattern prevents division-by-zero errors. The CASE WHEN ... THEN 1.0 (not 1) forces decimal division — without the decimal, Redshift performs integer division and you get 0 for any CTR below 100%. These are common pitfalls that silently produce wrong numbers in dashboards.

2. Window Functions for Running Totals and Rankings

Window functions compute a value across a set of rows related to the current row without collapsing the result set. They are essential for running totals, rankings, moving averages, and lead/lag comparisons. Redshift supports the full SQL:2003 window function specification.

sql 
-- Running total of spend and rank by daily spend within each campaign
SELECT
    campaign_id,
    event_date,
    daily_spend,
    SUM(daily_spend) OVER (
      PARTITION BY campaign_id
      ORDER BY event_date
      ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS cumulative_spend,
    RANK() OVER (
      PARTITION BY campaign_id
      ORDER BY daily_spend DESC
    ) AS spend_rank,
    LAG(daily_spend, 1) OVER (
      PARTITION BY campaign_id
      ORDER BY event_date
    ) AS prev_day_spend,
    ROUND(
      (daily_spend - LAG(daily_spend, 1) OVER (
        PARTITION BY campaign_id ORDER BY event_date
      )) / NULLIF(LAG(daily_spend, 1) OVER (
        PARTITION BY campaign_id ORDER BY event_date
      ), 0) * 100, 2
    ) AS day_over_day_change_pct
FROM analytics.daily_campaign_metrics
ORDER BY campaign_id, event_date;
ROWS vs RANGE

Always use ROWS BETWEEN instead of RANGE BETWEEN for frame specifications. RANGE treats ties as a single group and can produce unexpected results with non-unique ORDER BY columns. ROWS operates on physical row positions and is both more predictable and more performant in Redshift.

3. Common Table Expressions (CTEs)

CTEs (WITH clauses) let you break complex queries into named, readable steps. In Redshift, CTEs are materialized during execution — the query planner executes each CTE once and references the result. This makes them both a readability tool and a performance optimization for queries that would otherwise repeat the same subquery multiple times.

sql 
-- Multi-step CTE: daily metrics -> 7-day rolling avg -> top campaigns
WITH daily_metrics AS (
    SELECT
        campaign_id,
        event_date,
        COUNT(*) AS impressions,
        SUM(CASE WHEN clicked THEN 1 ELSE 0 END) AS clicks,
        SUM(win_price) AS spend
    FROM analytics.impressions
    WHERE event_date >= DATEADD(day, -30, CURRENT_DATE)
    GROUP BY campaign_id, event_date
),
rolling_avg AS (
    SELECT
        campaign_id,
        event_date,
        impressions,
        clicks,
        spend,
        AVG(impressions) OVER (
            PARTITION BY campaign_id
            ORDER BY event_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ) AS avg_impressions_7d,
        AVG(clicks::FLOAT / NULLIF(impressions, 0)) OVER (
            PARTITION BY campaign_id
            ORDER BY event_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ) AS avg_ctr_7d
    FROM daily_metrics
),
ranked AS (
    SELECT
        *,
        ROW_NUMBER() OVER (
            PARTITION BY event_date
            ORDER BY spend DESC
        ) AS daily_spend_rank
    FROM rolling_avg
)
SELECT *
FROM ranked
WHERE daily_spend_rank <= 10
ORDER BY event_date DESC, daily_spend_rank;

4. Date Series Generation and Gap Filling

Analytics dashboards need continuous time series — every date must appear even if there is no data for that date. Redshift does not have a built-in generate_series for dates in all configurations, but you can create one using a numbers table or the system table approach.

sql 
-- Generate a continuous date series using ROW_NUMBER()
WITH date_series AS (
    SELECT
        ('2024-01-01'::DATE + (ROW_NUMBER() OVER (ORDER BY TRUE) - 1)) AS cal_date
    FROM stv_blocklist  -- any system table with enough rows
    LIMIT 366
),
campaign_dates AS (
    SELECT
        c.campaign_id,
        d.cal_date
    FROM (SELECT DISTINCT campaign_id FROM analytics.impressions) c
    CROSS JOIN date_series d
),
daily_actuals AS (
    SELECT
        campaign_id,
        event_date,
        COUNT(*) AS impressions,
        SUM(win_price) AS spend
    FROM analytics.impressions
    GROUP BY campaign_id, event_date
)
SELECT
    cd.campaign_id,
    cd.cal_date,
    COALESCE(da.impressions, 0) AS impressions,
    COALESCE(da.spend, 0) AS spend
FROM campaign_dates cd
LEFT JOIN daily_actuals da
    ON cd.campaign_id = da.campaign_id
    AND cd.cal_date = da.event_date
ORDER BY cd.campaign_id, cd.cal_date;
Pro Tip

Create a permanent analytics.dim_date dimension table with pre-computed columns: day_of_week, week_number, month_name, is_weekend, fiscal_quarter. JOIN to it instead of computing date attributes inline. This is both faster and more maintainable.

5. Approximate Functions and HyperLogLog

When you need distinct counts across billions of rows, exact COUNT(DISTINCT) is expensive because it requires sorting and deduplication. Redshift offers APPROXIMATE COUNT(DISTINCT) which uses the HyperLogLog algorithm to return an estimate within 2% accuracy at a fraction of the cost.

sql 
-- Exact vs approximate count distinct comparison
SELECT
    event_date,
    COUNT(DISTINCT user_id) AS exact_unique_users,
    APPROXIMATE COUNT(DISTINCT user_id) AS approx_unique_users
FROM analytics.impressions
WHERE event_date BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY event_date
ORDER BY event_date;

-- Pre-aggregate HLL sketches for fast unique-user analysis
CREATE TABLE analytics.daily_user_sketches AS
SELECT
    event_date,
    campaign_id,
    HLL_CREATE_SKETCH(user_id) AS user_sketch
FROM analytics.impressions
GROUP BY event_date, campaign_id;

-- Combine sketches across date ranges for weekly unique users
SELECT
    campaign_id,
    HLL_ESTIMATE(HLL_COMBINE(user_sketch)) AS weekly_unique_users
FROM analytics.daily_user_sketches
WHERE event_date BETWEEN '2024-01-01' AND '2024-01-07'
GROUP BY campaign_id;

The HLL sketch pattern is particularly powerful: pre-aggregate sketches daily, then combine them for any arbitrary date range. This turns an O(n) distinct count into an O(1) sketch merge, making "unique users over the last 30 days" nearly instantaneous regardless of data volume.

6. Correlated Subqueries and Percentile Analysis

Some analytical questions require checking the existence of related data or computing distribution metrics. The EXISTS pattern is more efficient than IN for large datasets, and PERCENTILE_CONT provides exact percentile calculations for quality analysis.

sql 
-- Campaigns with impressions but no clicks in the past 7 days
SELECT
    c.campaign_id,
    c.campaign_name,
    c.budget,
    c.start_date
FROM analytics.campaigns c
WHERE EXISTS (
    SELECT 1
    FROM analytics.impressions i
    WHERE i.campaign_id = c.campaign_id
      AND i.event_date >= DATEADD(day, -7, CURRENT_DATE)
)
AND NOT EXISTS (
    SELECT 1
    FROM analytics.clicks cl
    WHERE cl.campaign_id = c.campaign_id
      AND cl.event_date >= DATEADD(day, -7, CURRENT_DATE)
);

-- Percentile analysis of bid prices by campaign
SELECT
    campaign_id,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY win_price) AS median_cpm,
    PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY win_price) AS p90_cpm,
    PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY win_price) AS p99_cpm
FROM analytics.impressions
WHERE event_date >= DATEADD(day, -30, CURRENT_DATE)
GROUP BY campaign_id
ORDER BY median_cpm DESC;
When should I use LISTAGG for string aggregation?

LISTAGG concatenates values from multiple rows into a single string. It is useful for creating comma-separated lists in reports, but be cautious with large datasets as the result is limited to 65535 characters in Redshift.

sql 
-- List all creative names for each campaign
SELECT
    campaign_id,
    LISTAGG(creative_name, ', ')
      WITHIN GROUP (ORDER BY creative_name) AS creative_list
FROM analytics.creatives
GROUP BY campaign_id;
How do NULLs interact with window functions?

By default, aggregate window functions like SUM and AVG ignore NULLs. For LAG and LEAD, NULLs are returned when there is no preceding or following row. Use COALESCE to substitute defaults, or use the optional third argument: LAG(value, 1, 0) to return 0 instead of NULL.

Self-Check

Before moving to Chapter 04, honestly answer these questions.

Can you write a window function with a custom frame specification from memory?

You should be able to write SUM(x) OVER (PARTITION BY a ORDER BY b ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) without looking it up. Practice until the syntax is automatic.

Do you understand when to use APPROXIMATE COUNT vs exact COUNT(DISTINCT)?

Use approximate when: the result is for dashboards where 2% error is acceptable, the dataset exceeds millions of rows, and query speed matters more than precision. Use exact when: the result drives financial calculations or you need deterministic outputs for reconciliation.

Can you generate a gap-filled date series for dashboard consumption?

The pattern is: generate a date series (using ROW_NUMBER over a system table or a dim_date table), CROSS JOIN with your dimension, then LEFT JOIN to actual data and COALESCE NULLs to zero. This ensures every date-dimension combination appears in the output.
