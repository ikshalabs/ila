Chapter 04 · Redshift Basic

Data Loading

Master the COPY command, S3 integration, manifest files, error handling, and Redshift Spectrum — the essential data pipeline tools for loading billions of ad impression records into your warehouse efficiently.

Estimated: 60–90 min
Key Concepts
3 Assignments

Why This Matters

A demand-side platform (DSP) generates hundreds of millions of impression events daily. Loading this data into Redshift efficiently is the difference between a 15-minute nightly ETL and an 8-hour nightmare that blocks your morning dashboards. The COPY command is Redshift's most optimized data loading mechanism — it parallelizes reads from S3, automatically distributes data across slices, and can handle compressed files without manual decompression.

Understanding COPY is not optional for ad-tech data engineers. It is the single most critical skill for keeping your warehouse fresh with the latest bid, impression, click, and conversion data.

Key Insight

The COPY command is 10-100x faster than INSERT statements for bulk loading. It reads files in parallel from S3, one file per slice. If you have 16 slices, split your data into 16 (or a multiple of 16) files for maximum throughput. A single monolithic file forces serial loading through one slice while the other 15 sit idle.

Key Concepts

1. The COPY Command

The COPY command loads data from Amazon S3, DynamoDB, or remote hosts via SSH into a Redshift table. For ad-tech, S3 is the standard source because impression logs are typically written to S3 by your bid servers in real-time:

sql
-- Load impression data from S3 (gzip compressed, pipe-delimited)
COPY analytics.fact_impressions
FROM 's3://adtech-data-lake/impressions/2025/01/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftS3ReadRole'
FORMAT AS CSV
DELIMITER '|'
GZIP
IGNOREHEADER 1
DATEFORMAT 'auto'
TIMEFORMAT 'auto'
MAXERROR 100
COMPUPDATE ON
STATUPDATE ON;

2. S3 Integration and File Splitting

For maximum COPY performance, split your data files to match your cluster's slice count. If you have 8 nodes with 2 slices each (16 slices), aim for 16, 32, or 48 files:

sql
-- Using a manifest file for precise file control
COPY analytics.fact_impressions
FROM 's3://adtech-data-lake/manifests/impressions_20250115.manifest'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftS3ReadRole'
MANIFEST
FORMAT AS PARQUET;
Pro Tip

Use Parquet format instead of CSV whenever possible. Parquet files carry their own schema, support columnar reads (Redshift only reads columns referenced in the COPY), and compress 3-5x better than gzipped CSV. For ad-tech impression data, Parquet can reduce load times by 40-60% compared to CSV.

3. Manifest Files

A manifest file is a JSON document that explicitly lists which S3 files to load. This prevents Redshift from loading duplicate files if a COPY is retried, and allows you to load files from multiple S3 prefixes:

json
{{
  "entries": [
    {{"url": "s3://adtech-data-lake/impressions/2025/01/15/part-00000.parquet", "mandatory": true}},
    {{"url": "s3://adtech-data-lake/impressions/2025/01/15/part-00001.parquet", "mandatory": true}},
    {{"url": "s3://adtech-data-lake/impressions/2025/01/15/part-00002.parquet", "mandatory": true}},
    {{"url": "s3://adtech-data-lake/impressions/2025/01/15/part-00003.parquet", "mandatory": true}}
  ]
}}

4. Error Handling and STL_LOAD_ERRORS

When COPY encounters data quality issues (type mismatches, constraint violations, malformed rows), it logs errors to the STL_LOAD_ERRORS system table. The MAXERROR parameter controls how many errors are tolerated before the entire load aborts:

sql
-- Diagnose COPY load errors
SELECT
    starttime,
    filename,
    line_number,
    colname,
    type,
    col_length,
    raw_field_value,
    err_reason
FROM stl_load_errors
ORDER BY starttime DESC
LIMIT 20;
Important

Always check STL_LOAD_ERRORS after every COPY command, even if it reports success. MAXERROR allows partial loads where some rows are silently skipped. In ad-tech, missing impressions means incorrect billing reconciliation. Set MAXERROR to 0 for financial data and only increase it for non-critical analytics loads.

5. Redshift Spectrum for Query-in-Place

Redshift Spectrum allows you to query data directly in S3 without loading it into Redshift tables. This is ideal for historical ad data that is rarely accessed but must be available for audits or long-range trend analysis:

sql
-- Create an external schema pointing to your data lake
CREATE EXTERNAL SCHEMA spectrum_ads
FROM DATA CATALOG
DATABASE 'adtech_datalake'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftSpectrumRole'
CREATE EXTERNAL DATABASE IF NOT EXISTS;

-- Query historical impressions without loading
SELECT
    event_date,
    campaign_id,
    COUNT(*) AS impressions
FROM spectrum_ads.impressions_archive
WHERE year = '2024' AND month = '06'
GROUP BY event_date, campaign_id;

Self-Check

Before moving to the next chapter, honestly answer these questions.

Why is COPY faster than INSERT for bulk loading?

COPY reads files in parallel from S3, with each slice loading its own file simultaneously. INSERT processes rows serially through the leader node. For a 16-slice cluster loading 16 files, COPY achieves 16x parallelism. Additionally, COPY batches the writes and can skip the write-ahead log for better throughput.

When should you use a manifest file instead of an S3 prefix?

Use manifest files when you need idempotent loads (retries should not load duplicate files), when files span multiple S3 prefixes or buckets, or when you need to guarantee that specific files are present (mandatory:true). For ad-tech, manifests are essential for daily partition loads where you must ensure exactly one day of data is loaded.

What is the optimal number of files for a COPY operation?

The optimal number is equal to a multiple of your cluster slice count. If you have 16 slices, use 16, 32, or 48 files of approximately equal size. Each file should be between 1 MB and 1 GB after compression. Files that are too small create overhead; files that are too large limit parallelism.
