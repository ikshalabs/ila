Chapter 04 · Phase 2: Query Patterns

Schema Design for Analytics

Design star schemas and wide denormalized tables for Redshift. Learn when to normalize, when to flatten, how to build fact and dimension tables, and why Redshift's columnar storage changes every schema design rule you learned in OLTP databases.

Estimated: 90–120 min
5 Key Concepts
3 Assignments

Why This Matters

Schema design in Redshift is fundamentally different from schema design in PostgreSQL, MySQL, or any other row-oriented OLTP database. In OLTP systems, normalization reduces data redundancy and ensures consistency through referential integrity. In Redshift — a columnar, distributed, append-optimized warehouse — normalization often hurts performance because every JOIN requires data redistribution across the network.

The cost of a JOIN in Redshift depends on whether the joined tables are co-located on the same slices (collocated JOIN) or require data to be shuffled between nodes (redistribution JOIN). A poorly designed schema forces redistribution on every query, turning sub-second lookups into multi-minute shuffles. Getting schema design right means understanding when to denormalize, how to structure fact and dimension tables, and which columns deserve dedicated encoding.

This chapter teaches you to think in terms of query patterns first, schema second. Instead of designing tables in isolation and hoping queries perform well, you identify your top 10 query patterns and design tables that serve them directly. This inversion of the traditional design process is what separates a warehouse that handles terabytes effortlessly from one that chokes on gigabytes.

Key Insight

In Redshift, wide tables are not a code smell — they are a design pattern. A 200-column fact table with pre-joined dimensions avoids redistribution JOINs, compresses efficiently because columnar storage only reads accessed columns, and enables the query optimizer to apply zone maps and sort-key pruning across the full dataset.

Key Concepts

1. Star Schema Fundamentals

The star schema is the foundational pattern for analytical warehouses. It consists of a central fact table containing measurable events (impressions, clicks, transactions) surrounded by dimension tables that describe the context (campaigns, advertisers, dates, geographies).

sql 
-- Fact table: pre-aggregated daily campaign metrics
CREATE TABLE analytics.fact_daily_metrics (
    date_key         INTEGER       ENCODE az64       NOT NULL,
    campaign_key     INTEGER       ENCODE az64       NOT NULL,
    creative_key     INTEGER       ENCODE az64,
    geo_key          INTEGER       ENCODE az64,
    impressions      BIGINT        ENCODE az64,
    clicks           BIGINT        ENCODE az64,
    viewable_imps    BIGINT        ENCODE az64,
    spend            DECIMAL(18,4) ENCODE az64,
    conversions      INTEGER       ENCODE az64,
    revenue          DECIMAL(18,4) ENCODE az64
)
DISTSTYLE KEY
DISTKEY (campaign_key)
COMPOUND SORTKEY (date_key, campaign_key);

-- Dimension table: campaigns (small, replicated to all slices)
CREATE TABLE analytics.dim_campaign (
    campaign_key     INTEGER       ENCODE az64  NOT NULL,
    campaign_id      INTEGER       ENCODE az64,
    campaign_name    VARCHAR(200)  ENCODE lzo,
    advertiser_id    INTEGER       ENCODE az64,
    advertiser_name  VARCHAR(200)  ENCODE lzo,
    campaign_type    VARCHAR(50)   ENCODE bytedict,
    start_date       DATE          ENCODE az64,
    end_date         DATE          ENCODE az64,
    budget           DECIMAL(12,2) ENCODE az64,
    status           VARCHAR(20)   ENCODE bytedict
)
DISTSTYLE ALL;  -- replicate to every slice for collocated JOINs
DISTSTYLE ALL for Small Dimensions

Use DISTSTYLE ALL for dimension tables under ~2 million rows. This replicates the table to every slice, ensuring JOINs with fact tables never require redistribution. For large dimension tables, use DISTSTYLE KEY with the same DISTKEY as the fact table's foreign key.

2. Denormalization and Wide Tables

In Redshift, the "right" amount of denormalization depends on your query patterns. If 80% of your queries JOIN fact_impressions with dim_campaign and dim_date, consider pre-joining these into a single wide table. The storage overhead is minimal (columnar compression handles redundant dimension values efficiently) and the query performance gain from eliminating JOINs can be 5-10x.

sql 
-- Pre-joined wide table: eliminates JOINs for common queries
CREATE TABLE analytics.wide_daily_metrics AS
SELECT
    -- Date dimensions (from dim_date)
    d.cal_date,
    d.day_of_week,
    d.week_number,
    d.month_name,
    d.quarter,
    d.is_weekend,
    -- Campaign dimensions (from dim_campaign)
    c.campaign_id,
    c.campaign_name,
    c.advertiser_id,
    c.advertiser_name,
    c.campaign_type,
    c.budget AS campaign_budget,
    c.status AS campaign_status,
    -- Geo dimensions (from dim_geo)
    g.country_code,
    g.region,
    g.city,
    -- Metrics (from fact table)
    f.impressions,
    f.clicks,
    f.viewable_imps,
    f.spend,
    f.conversions,
    f.revenue,
    -- Derived metrics
    ROUND(f.clicks::FLOAT / NULLIF(f.impressions, 0) * 100, 4) AS ctr_pct,
    ROUND(f.spend / NULLIF(f.impressions, 0) * 1000, 2) AS ecpm,
    ROUND(f.revenue / NULLIF(f.spend, 0), 4) AS roas
FROM analytics.fact_daily_metrics f
JOIN analytics.dim_date d       ON f.date_key = d.date_key
JOIN analytics.dim_campaign c   ON f.campaign_key = c.campaign_key
LEFT JOIN analytics.dim_geo g   ON f.geo_key = g.geo_key;

-- Set optimal table properties after creation
ALTER TABLE analytics.wide_daily_metrics
    ALTER DISTSTYLE KEY
    ALTER DISTKEY campaign_id;

ALTER TABLE analytics.wide_daily_metrics
    ALTER COMPOUND SORTKEY (cal_date, campaign_id);

3. Column Encoding Strategies

Redshift stores data in compressed columnar format. Choosing the right encoding for each column can reduce storage by 50-80% and improve query performance by reducing I/O. The key encodings to understand are:

  • AZ64 — The default for numeric and date types. Best all-around compression for integers, decimals, timestamps.
  • BYTEDICT — Dictionary encoding for low-cardinality strings (under ~256 unique values). Ideal for status codes, country codes, device types.
  • LZO — General-purpose compression for high-cardinality strings. Good for names, URLs, user agents.
  • ZSTD — Higher compression ratio than LZO at the cost of slightly more CPU. Best for large VARCHAR columns you rarely filter on.
  • RAW — No compression. Only for BOOLEAN or very small fixed-width columns where compression overhead exceeds savings.
sql 
-- Analyze current encoding and recommend improvements
ANALYZE COMPRESSION analytics.fact_daily_metrics;

-- Check column encoding and storage stats
SELECT
    TRIM(tablename) AS table_name,
    TRIM("column") AS column_name,
    TRIM(type) AS data_type,
    TRIM(encoding) AS encoding,
    notnull
FROM pg_table_def
WHERE schemaname = 'analytics'
  AND tablename = 'fact_daily_metrics'
ORDER BY "column";

-- Check actual compression ratios per column
SELECT
    col,
    TRIM(attname) AS column_name,
    SUM(num_values) AS total_rows,
    SUM(size) AS compressed_mb,
    ROUND(SUM(size)::FLOAT / NULLIF(SUM(num_values), 0) * 1000000, 2) AS bytes_per_row
FROM SVV_DISKUSAGE
WHERE name = 'fact_daily_metrics'
  AND col >= 0
GROUP BY col, attname
ORDER BY compressed_mb DESC;
Pro Tip

Run ANALYZE COMPRESSION on a table with representative data to get Redshift's encoding recommendations. It tests multiple encodings against your actual data and recommends the one with the best compression ratio. This is far more reliable than guessing.

4. Staging Layer Design Patterns

A well-designed warehouse has three layers: raw (landing zone), staging (cleaned and transformed), and analytics (query-ready). The staging layer is where data type conversions, deduplication, and business logic transformations happen.

sql 
-- Staging transformation: deduplicate and type-cast raw data
CREATE TABLE staging.impressions AS
WITH deduplicated AS (
    SELECT
        *,
        ROW_NUMBER() OVER (
            PARTITION BY impression_id
            ORDER BY impression_ts DESC
        ) AS rn
    FROM raw.impressions
)
SELECT
    impression_id,
    campaign_id,
    advertiser_id,
    creative_id,
    impression_ts,
    impression_ts::DATE AS event_date,
    ROUND(bid_price, 4) AS bid_price,
    ROUND(win_price, 4) AS win_price,
    UPPER(country_code) AS country_code,
    LOWER(device_type) AS device_type,
    is_viewable
FROM deduplicated
WHERE rn = 1
  AND impression_id IS NOT NULL
  AND campaign_id IS NOT NULL;

5. Slowly Changing Dimensions (SCD Type 2)

Dimension attributes change over time — a campaign's budget gets increased, an advertiser's name changes. SCD Type 2 preserves the full history by creating a new row for each change, with valid_from and valid_to date columns. This lets you accurately attribute historical metrics to the correct dimension state.

sql 
-- SCD Type 2 dimension table for campaigns
CREATE TABLE analytics.dim_campaign_scd2 (
    campaign_key     INTEGER       IDENTITY(1,1),
    campaign_id      INTEGER       ENCODE az64  NOT NULL,
    campaign_name    VARCHAR(200)  ENCODE lzo,
    budget           DECIMAL(12,2) ENCODE az64,
    status           VARCHAR(20)   ENCODE bytedict,
    valid_from       TIMESTAMP     ENCODE az64  NOT NULL,
    valid_to         TIMESTAMP     ENCODE az64  DEFAULT '9999-12-31',
    is_current       BOOLEAN       DEFAULT TRUE
)
DISTSTYLE ALL
SORTKEY (campaign_id, valid_from);

-- Join fact table to SCD2 dimension using date range
SELECT
    f.event_date,
    c.campaign_name,
    c.budget,
    c.status,
    f.impressions,
    f.spend
FROM analytics.fact_daily_metrics f
JOIN analytics.dim_campaign_scd2 c
    ON f.campaign_key = c.campaign_id
    AND f.event_date >= c.valid_from::DATE
    AND f.event_date < c.valid_to::DATE
WHERE f.event_date >= '2024-01-01';
When should I use SCD Type 1 (overwrite) vs Type 2 (history)?

Use Type 1 (overwrite in place) for attributes where historical accuracy does not matter — for example, fixing a typo in a campaign name. Use Type 2 (new row with date range) for attributes where historical accuracy is critical — for example, budget changes that affect spend pacing analysis, or status changes that affect active campaign counts.

How do I handle late-arriving dimension updates?

Use an upsert pattern: stage incoming dimension data into a temp table, then use a DELETE + INSERT pattern (Redshift does not support native MERGE). Close existing current records by setting valid_to and is_current = FALSE, then insert new records with is_current = TRUE.

sql 
-- Upsert pattern for SCD Type 2
BEGIN TRANSACTION;

-- Close existing current records for changed campaigns
UPDATE analytics.dim_campaign_scd2
SET valid_to = GETDATE(), is_current = FALSE
WHERE campaign_id IN (SELECT campaign_id FROM staging.campaign_updates)
  AND is_current = TRUE;

-- Insert new current records
INSERT INTO analytics.dim_campaign_scd2
    (campaign_id, campaign_name, budget, status, valid_from, valid_to, is_current)
SELECT
    campaign_id, campaign_name, budget, status,
    GETDATE(), '9999-12-31', TRUE
FROM staging.campaign_updates;

COMMIT;

Self-Check

Before moving to Chapter 05, honestly answer these questions.

Can you explain why DISTSTYLE ALL is appropriate for dimension tables?

DISTSTYLE ALL replicates the entire table to every slice. When a fact table (distributed by key) JOINs to a dimension with DISTSTYLE ALL, the JOIN is collocated on every slice — no network redistribution is needed. The storage cost is minimal for small dimensions (under a few million rows).

Do you know when to denormalize vs keep a star schema?

Denormalize (create a wide table) when: a specific set of queries dominates your workload, the JOIN columns are not co-distributed, and the dimension data changes infrequently. Keep the star schema when: you have diverse query patterns, dimensions change frequently, or you need to minimize data refresh times.

Can you choose the right column encoding without guessing?

Use AZ64 for numeric/date columns, BYTEDICT for low-cardinality strings (under 256 unique values), LZO for high-cardinality strings, and ZSTD for large text columns you rarely filter on. When unsure, run ANALYZE COMPRESSION to get Redshift's recommendation based on actual data.
