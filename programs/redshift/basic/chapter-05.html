Chapter 05 · Phase 3: Performance

Distribution and Sort Keys

Master the two most impactful physical design decisions in Redshift: distribution keys that control how data is placed across slices, and sort keys that control the on-disk ordering for zone-map pruning. These choices determine whether your queries scan megabytes or terabytes.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

Distribution and sort keys are the two most impactful physical design decisions you make in Redshift. A well-chosen distribution key ensures that JOINs between related tables happen locally on each slice without expensive network redistribution. A well-chosen sort key ensures that range-filtered queries (like "show me data from the last 7 days") skip 95% of the disk blocks via zone-map pruning.

Together, these two decisions can make the difference between a query that scans 50MB and one that scans 500GB — a 10,000x difference in I/O. Getting them wrong does not produce errors; it produces slow queries that you may not notice until your data volume grows and performance degrades exponentially. This is why distribution and sort key design is the single most important performance optimization in Redshift.

The challenge is that distribution and sort key choices are interdependent: the best DISTKEY for JOIN performance may not produce even data distribution, and the best SORTKEY for one query pattern may hurt another. This chapter teaches you a systematic approach to making these decisions based on your actual query workload, not guesswork.

Key Insight

Think of DISTKEY as answering "where does each row live?" and SORTKEY as answering "in what order are rows stored on disk?" The DISTKEY determines which slice owns a row. The SORTKEY determines which 1MB blocks within that slice contain the row. Together, they let Redshift find your data with minimal I/O.

Key Concepts

1. Distribution Styles: KEY, ALL, EVEN, AUTO

Redshift offers four distribution styles that control how rows are assigned to slices. Each has distinct trade-offs that depend on your table size and JOIN patterns:

  • DISTSTYLE KEY — Rows with the same DISTKEY value are co-located on the same slice. This enables collocated JOINs when two tables share the same DISTKEY column. Best for large fact tables.
  • DISTSTYLE ALL — The entire table is replicated to every slice. JOINs are always collocated but storage and COPY time increase linearly with cluster size. Best for small dimension tables.
  • DISTSTYLE EVEN — Rows are distributed round-robin across slices. Guarantees even distribution but all JOINs require redistribution. Use only when no good DISTKEY exists.
  • DISTSTYLE AUTO — Redshift starts with ALL for small tables and switches to EVEN as they grow. A safe default but gives you less control.
sql 
-- Check current distribution style for all tables
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(pgc.relname) AS table_name,
    CASE pgc.reldiststyle
        WHEN 0 THEN 'EVEN'
        WHEN 1 THEN 'KEY'
        WHEN 8 THEN 'ALL'
        WHEN 10 THEN 'AUTO(ALL)'
        WHEN 11 THEN 'AUTO(EVEN)'
        WHEN 12 THEN 'AUTO(KEY)'
    END AS diststyle,
    TRIM(pga.attname) AS distkey
FROM pg_class pgc
JOIN pg_namespace pgn ON pgc.relnamespace = pgn.oid
LEFT JOIN pg_attribute pga ON pga.attrelid = pgc.oid AND pga.attisdistkey = TRUE
WHERE pgn.nspname = 'analytics'
ORDER BY table_name;

-- Check data distribution skew across slices
SELECT
    TRIM(name) AS table_name,
    slice,
    SUM(num_values) AS row_count
FROM STV_TBL_PERM
WHERE name = 'fact_daily_metrics'
GROUP BY name, slice
ORDER BY slice;
Distribution Skew is a Silent Killer

If you choose a DISTKEY column with low cardinality (e.g., country_code with only 10 distinct values), most rows end up on a few slices while others sit empty. Your query runs at the speed of the slowest slice. Always check STV_TBL_PERM after COPY to verify even distribution.

2. Choosing the Right Distribution Key

The ideal DISTKEY column has three properties: (1) high cardinality (many unique values for even distribution), (2) frequently used in JOIN conditions (for collocated JOINs), and (3) frequently used in GROUP BY clauses (for local aggregation). In practice, you often must prioritize one over the others.

sql 
-- Analyze DISTKEY candidate: check cardinality and skew
SELECT
    'campaign_id' AS candidate_column,
    COUNT(DISTINCT campaign_id) AS distinct_values,
    COUNT(*) AS total_rows,
    MAX(freq) AS max_frequency,
    MIN(freq) AS min_frequency,
    ROUND(STDDEV(freq), 2) AS frequency_stddev
FROM (
    SELECT campaign_id, COUNT(*) AS freq
    FROM analytics.fact_daily_metrics
    GROUP BY campaign_id
) freq_dist;

-- Collocated JOIN example: both tables use campaign_id as DISTKEY
SELECT
    f.event_date,
    c.campaign_name,
    SUM(f.impressions) AS total_impressions,
    SUM(f.spend) AS total_spend
FROM analytics.fact_daily_metrics f  -- DISTKEY(campaign_id)
JOIN analytics.dim_campaign c         -- DISTSTYLE ALL (collocated)
    ON f.campaign_key = c.campaign_key
GROUP BY f.event_date, c.campaign_name;

3. Compound Sort Keys

A compound sort key defines the physical ordering of rows on disk. Redshift uses this ordering to maintain zone maps — metadata that records the min/max values in each 1MB block. When a query filters on a sort key column, Redshift skips blocks whose min/max range does not overlap the filter, dramatically reducing I/O.

sql 
-- Create table with compound sort key (date first, then campaign)
CREATE TABLE analytics.fact_events (
    event_id         BIGINT        ENCODE az64,
    event_date       DATE          ENCODE az64       NOT NULL,
    campaign_id      INTEGER       ENCODE az64       NOT NULL,
    event_type       VARCHAR(20)   ENCODE bytedict,
    impressions      BIGINT        ENCODE az64,
    clicks           BIGINT        ENCODE az64,
    spend            DECIMAL(18,4) ENCODE az64
)
DISTSTYLE KEY
DISTKEY (campaign_id)
COMPOUND SORTKEY (event_date, campaign_id);

-- This query benefits from sort key pruning (filters on leading column)
SELECT campaign_id, SUM(impressions), SUM(spend)
FROM analytics.fact_events
WHERE event_date BETWEEN '2024-01-01' AND '2024-01-07'
GROUP BY campaign_id;

-- Check zone map effectiveness: how many blocks were skipped?
SELECT
    query,
    TRIM(tbl_name) AS table_name,
    rows_pre_filter,
    rows_pre_user_filter,
    rows,
    ROUND((1.0 - rows_pre_user_filter::FLOAT / NULLIF(rows_pre_filter, 0)) * 100, 1) AS zone_map_skip_pct
FROM SVL_QUERY_SUMMARY
WHERE query = pg_last_query_id()
  AND step_type = 'scan';
Sort Key Column Order Matters

In a compound sort key, zone-map pruning only works when you filter on the leading columns of the sort key. A sort key of (event_date, campaign_id) benefits queries that filter on event_date or event_date AND campaign_id, but NOT queries that filter only on campaign_id. Put your most commonly filtered column first.

4. Interleaved Sort Keys

Unlike compound sort keys which prioritize the leading column, interleaved sort keys give equal weight to all included columns. This is useful when queries filter on different columns unpredictably. However, interleaved sort keys have a significant cost: VACUUM REINDEX is much more expensive, and they should not be used on columns with monotonically increasing values (like timestamps).

sql 
-- Interleaved sort key: equal pruning on any column combination
CREATE TABLE analytics.fact_ad_events (
    event_id         BIGINT        ENCODE az64,
    campaign_id      INTEGER       ENCODE az64,
    creative_id      INTEGER       ENCODE az64,
    device_type      VARCHAR(20)   ENCODE bytedict,
    country_code     CHAR(2)       ENCODE bytedict,
    impressions      BIGINT        ENCODE az64,
    spend            DECIMAL(18,4) ENCODE az64
)
DISTSTYLE KEY
DISTKEY (campaign_id)
INTERLEAVED SORTKEY (campaign_id, creative_id, country_code);

-- After loading data, VACUUM REINDEX to rebuild interleaved index
VACUUM REINDEX analytics.fact_ad_events;

5. VACUUM and ANALYZE

When you DELETE or UPDATE rows in Redshift, the old rows are not physically removed — they are marked as deleted ("ghost rows"). VACUUM reclaims this space and re-sorts data to maintain sort key effectiveness. ANALYZE updates table statistics used by the query planner.

sql 
-- Check tables that need VACUUM (high percentage of unsorted rows)
SELECT
    TRIM(pgn.nspname) AS schema_name,
    TRIM(a.name) AS table_name,
    a.rows AS total_rows,
    a.unsorted_rows,
    ROUND(a.unsorted_rows::FLOAT / NULLIF(a.rows, 0) * 100, 2) AS pct_unsorted,
    a.tbl_rows_deleted AS ghost_rows
FROM SVV_TABLE_INFO a
JOIN pg_class pgc ON pgc.oid = a.table_id
JOIN pg_namespace pgn ON pgn.oid = pgc.relnamespace
WHERE a.unsorted_rows > 0
ORDER BY pct_unsorted DESC;

-- Run VACUUM to reclaim space and re-sort
VACUUM FULL analytics.fact_daily_metrics;

-- Update statistics for the query planner
ANALYZE analytics.fact_daily_metrics;

6. Monitoring Distribution and Sort Key Effectiveness

Use Redshift system tables to monitor whether your distribution and sort key choices are effective. The key metrics to track are: distribution skew (are rows evenly spread?), zone-map effectiveness (are blocks being skipped?), and redistribution events (are JOINs causing network shuffles?).

sql 
-- Detect redistribution in query plans (DS_BCAST or DS_DIST)
SELECT
    query,
    segment,
    step,
    label,
    rows,
    bytes
FROM SVL_QUERY_SUMMARY
WHERE query = pg_last_query_id()
  AND label LIKE '%dist%'
ORDER BY segment, step;

-- Check table skew: ratio of max to min rows per slice
SELECT
    TRIM(name) AS table_name,
    MAX(row_count) AS max_rows_per_slice,
    MIN(row_count) AS min_rows_per_slice,
    ROUND(MAX(row_count)::FLOAT / NULLIF(MIN(row_count), 0), 2) AS skew_ratio
FROM (
    SELECT name, slice, SUM(num_values) AS row_count
    FROM STV_TBL_PERM
    WHERE name IN ('fact_daily_metrics', 'fact_events')
    GROUP BY name, slice
)
GROUP BY name
ORDER BY skew_ratio DESC;
How do I change a table's DISTKEY or SORTKEY after creation?

Redshift supports ALTER TABLE ... ALTER DISTSTYLE and ALTER TABLE ... ALTER SORTKEY for in-place changes. However, for large tables, this operation can take hours because it requires physically redistributing or re-sorting all data. An alternative is the deep-copy pattern: create a new table with the desired properties, insert data from the old table, then rename.

sql 
-- Deep-copy pattern for changing table properties
CREATE TABLE analytics.fact_events_new (LIKE analytics.fact_events)
    DISTSTYLE KEY DISTKEY (campaign_id)
    COMPOUND SORTKEY (event_date, campaign_id);

INSERT INTO analytics.fact_events_new
SELECT * FROM analytics.fact_events;

ALTER TABLE analytics.fact_events RENAME TO fact_events_old;
ALTER TABLE analytics.fact_events_new RENAME TO fact_events;
DROP TABLE analytics.fact_events_old;
When should I use DISTSTYLE AUTO instead of explicit keys?

Use AUTO when you are prototyping and do not yet know your query patterns. Redshift will start with DISTSTYLE ALL for small tables and automatically switch to EVEN as they grow. However, AUTO never chooses KEY distribution, so for production workloads where collocated JOINs matter, always specify an explicit DISTKEY.

Self-Check

Before moving to Chapter 06, honestly answer these questions.

Can you explain the difference between DISTSTYLE KEY, ALL, and EVEN?

KEY distributes rows by hash of the DISTKEY column (co-locates matching values). ALL replicates the entire table to every slice (best for small dimensions). EVEN distributes rows round-robin (guarantees even distribution but all JOINs require redistribution).

Do you know when compound sort keys beat interleaved sort keys?

Compound wins when most queries filter on the same leading column (e.g., date). Interleaved wins when queries unpredictably filter on different columns. Compound is also much cheaper to maintain (VACUUM is faster). For time-series data with a dominant date filter, compound is almost always better.

Can you detect and fix distribution skew using system tables?

Query STV_TBL_PERM to get row counts per slice. Calculate the ratio of max to min rows. A ratio above 2.0 indicates significant skew. Fix by choosing a higher-cardinality DISTKEY or switching to DISTSTYLE EVEN. Use the deep-copy pattern to change the DISTKEY on an existing table.
