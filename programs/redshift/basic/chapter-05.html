Chapter 05 · Redshift Basic

Schema Design for Analytics

Design star and snowflake schemas optimized for ad-tech analytics — fact tables for impressions, clicks, and conversions; dimension tables for campaigns, creatives, and publishers; and slowly changing dimension patterns for tracking changes over time.

Estimated: 60–90 min
Key Concepts
3 Assignments

Why This Matters

Schema design is the architectural blueprint for your entire ad-tech analytics platform. A well-designed star schema makes queries intuitive for analysts, enables efficient joins through colocation, and supports the hierarchical drill-down patterns that campaign managers need: advertiser to campaign to ad group to creative to placement.

In ad-tech, your schema must handle three distinct types of facts: transactional facts (individual impressions, clicks, bids), periodic snapshot facts (daily campaign summaries), and accumulating snapshot facts (conversion funnels tracking impression-to-click-to-conversion lifecycles). Getting the grain right for each fact table is the most important design decision you will make.

Key Insight

The grain of a fact table defines what one row represents. For fact_impressions, the grain is one impression event. For fact_daily_campaign_summary, the grain is one campaign per day. Never mix grains in the same table — it makes aggregation ambiguous and leads to double-counting bugs that are extremely difficult to diagnose in ad-tech billing.

Key Concepts

1. Star Schema for Ad-Tech

A star schema places a central fact table surrounded by dimension tables. The fact table contains measurable events (impressions, clicks, spend) and foreign keys to dimensions. Dimensions contain descriptive attributes (campaign name, advertiser, creative size, publisher domain):

sql
-- Central fact table: one row per impression event
CREATE TABLE analytics.fact_impressions (
    impression_id     BIGINT IDENTITY(1,1),
    event_timestamp   TIMESTAMP NOT NULL,
    event_date        DATE NOT NULL,
    campaign_id       INTEGER NOT NULL REFERENCES analytics.dim_campaign(campaign_id),
    creative_id       INTEGER REFERENCES analytics.dim_creative(creative_id),
    publisher_id      INTEGER REFERENCES analytics.dim_publisher(publisher_id),
    geo_id            INTEGER REFERENCES analytics.dim_geo(geo_id),
    device_type_id    SMALLINT REFERENCES analytics.dim_device(device_type_id),
    bid_price_cpm     DECIMAL(10,4),
    win_price_cpm     DECIMAL(10,4),
    is_viewable       BOOLEAN DEFAULT FALSE,
    is_clicked        BOOLEAN DEFAULT FALSE,
    viewability_score DECIMAL(5,2)
)
DISTSTYLE KEY DISTKEY(campaign_id)
COMPOUND SORTKEY(event_date, campaign_id);

2. Dimension Tables for Ad-Tech Entities

Each dimension table represents a business entity with descriptive attributes. In ad-tech, the key dimensions are campaigns, creatives, publishers, geography, devices, and time:

sql
-- Campaign dimension with advertiser hierarchy
CREATE TABLE analytics.dim_campaign (
    campaign_id       INTEGER PRIMARY KEY,
    campaign_name     VARCHAR(256),
    advertiser_id     INTEGER,
    advertiser_name   VARCHAR(256),
    agency_id         INTEGER,
    agency_name       VARCHAR(256),
    campaign_type     VARCHAR(50),  -- 'brand', 'performance', 'retargeting'
    campaign_status   VARCHAR(20),  -- 'active', 'paused', 'completed'
    objective         VARCHAR(50),  -- 'awareness', 'clicks', 'conversions'
    start_date        DATE,
    end_date          DATE,
    daily_budget      DECIMAL(12,2),
    total_budget      DECIMAL(12,2),
    effective_from    TIMESTAMP DEFAULT GETDATE(),
    effective_to      TIMESTAMP DEFAULT '9999-12-31',
    is_current        BOOLEAN DEFAULT TRUE
)
DISTSTYLE ALL;

3. Fact Table Types

Ad-tech data warehouses typically contain three types of fact tables:

  • Transactional facts — One row per event (impression, click, bid request). Highest volume, finest grain. Use for detailed analysis and debugging.
  • Periodic snapshots — One row per entity per time period (campaign per day, publisher per hour). Pre-aggregated for dashboard performance. Use for standard reporting.
  • Accumulating snapshots — One row per lifecycle (impression-to-conversion funnel). Updated as events occur. Use for attribution and funnel analysis.

4. Slowly Changing Dimensions (SCD Type 2)

Campaign attributes change over time: budgets are adjusted, names are updated, statuses flip between active and paused. SCD Type 2 preserves history by creating a new row for each change with effective date ranges:

sql
-- SCD Type 2 update: campaign budget changed
-- Step 1: Close the current record
UPDATE analytics.dim_campaign
SET effective_to = GETDATE(), is_current = FALSE
WHERE campaign_id = 42 AND is_current = TRUE;

-- Step 2: Insert new record with updated budget
INSERT INTO analytics.dim_campaign
    (campaign_id, campaign_name, advertiser_id, advertiser_name,
     daily_budget, total_budget, effective_from, effective_to, is_current)
VALUES
    (42, 'Summer Sale 2025', 101, 'Acme Corp',
     5000.00, 150000.00, GETDATE(), '9999-12-31', TRUE);
Important

When joining fact tables to SCD Type 2 dimensions, always join on both the business key AND the date range: WHERE fi.event_timestamp BETWEEN dc.effective_from AND dc.effective_to. Joining only on campaign_id will produce duplicate rows because multiple historical records exist for the same campaign.

Level 1 — Foundation

Build the Ad-Tech Star Schema

Create the complete star schema for an ad-tech data warehouse.

  • Create dim_campaign, dim_creative, dim_publisher, dim_geo, and dim_date tables
  • Create fact_impressions and fact_clicks tables with foreign keys to all dimensions
  • Insert at least 50 rows into each dimension table with realistic ad-tech data
  • Verify the schema using SVV_TABLE_INFO and document the distribution strategy
Done when

Assignment 2
Level 2 — Professional

Periodic Snapshot Fact Table

Create and populate a daily campaign summary fact table for dashboard performance.

  • Design fact_daily_campaign_summary with pre-aggregated metrics (impressions, clicks, spend, CTR, viewability)
  • Write a SQL query that populates the summary from fact_impressions for a given date
  • Create a stored procedure that runs this aggregation for any date parameter
  • Compare query performance: ad-hoc aggregation on fact_impressions vs direct select on the summary table
Done when

You have a working two-tier fact architecture: detail for ad-hoc analysis, summary for dashboard performance.

Assignment 3
Level 3 — Production-Ready

SCD Type 2 Implementation

Implement slowly changing dimensions for campaign and creative attribute tracking.

  • Add SCD Type 2 columns (effective_from, effective_to, is_current) to dim_campaign
  • Write a stored procedure that handles SCD Type 2 updates (close old record, insert new)
  • Test with 3 campaign budget changes and verify historical records are preserved
  • Write a query that correctly joins impressions to the campaign dimension version that was active at impression time
  • Document the SCD Type 2 pattern with a process diagram
Done when

You can track campaign attribute changes over time and correctly attribute impressions to the right campaign version.

Self-Check

Before moving to the next chapter, honestly answer these questions.

What is the grain of fact_impressions and why does it matter?

The grain is one impression event — each row represents a single ad being served to a single user at a specific moment. The grain determines what COUNT(*) means: in fact_impressions it counts total impressions. If you accidentally stored one row per campaign-per-day, COUNT(*) would count days, not impressions, leading to incorrect billing and reporting.

When should you use a periodic snapshot instead of a transactional fact?

Use periodic snapshots when dashboards need fast access to pre-aggregated metrics. If your daily campaign report query scans 500M impression rows and takes 30 seconds, a pre-built summary table with one row per campaign per day scans 30K rows in under 1 second. The tradeoff is data freshness: summaries must be rebuilt when source data is corrected.

Why use DISTSTYLE ALL for dimension tables in ad-tech?

Dimension tables like dim_campaign, dim_creative, and dim_publisher are small (thousands to low millions of rows) but joined in virtually every query. DISTSTYLE ALL places a complete copy on every slice, ensuring that every join between a fact table and a dimension is local (no network redistribution). The storage overhead is minimal for small tables.
