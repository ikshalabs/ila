Chapter 02 · Phase 1: Data Warehouse Fundamentals

Cluster Setup and Data Loading

Provision a Redshift cluster from scratch, configure networking and IAM roles, create your first schemas and tables, and bulk-load data from S3 using the COPY command — the single most important data-loading mechanism in the Redshift ecosystem.

Estimated: 90–120 min
6 Key Concepts
3 Assignments

Why This Matters

A data warehouse is only as useful as the data it contains, and getting data into Redshift efficiently is a skill that separates production-grade pipelines from toy demos. The COPY command is not just "another INSERT" — it is a massively parallel ingestion engine that can load terabytes of compressed data from Amazon S3 in minutes. Misusing it (or worse, using row-by-row INSERTs instead) is the single most common performance mistake new Redshift users make.

Before you can load anything, you need a cluster. Redshift cluster provisioning involves decisions about node types, network topology, IAM permissions, and encryption that are difficult to change later. A cluster provisioned without VPC isolation or with the wrong node type will cost you either in security audit failures or in wasted compute spend. This chapter teaches you to get both the infrastructure and the data loading right from day one.

Beyond the mechanics, understanding how COPY distributes data across slices and how Redshift compresses columns during ingestion gives you a mental model that will inform every optimization decision in later chapters. The patterns you learn here — manifest files, split files, compression codecs, error handling with STL_LOAD_ERRORS — are the foundation of every production ETL pipeline running on Redshift today.

Key Insight

The COPY command is not just a data loader — it is Redshift's parallel ingestion engine. A single COPY from S3 fans out across all slices simultaneously. When your source files are properly split (one file per slice or a multiple thereof), you achieve maximum throughput. A single large file forces serial reads on the leader node — the most common beginner mistake in Redshift data loading.

Key Concepts

1. Cluster Provisioning and Node Types

Redshift offers two primary node families: RA3 (managed storage, compute decoupled from storage) and the legacy DC2 (dense compute with local SSD). For new workloads, RA3 is almost always the right choice because it allows you to scale compute and storage independently. DC2 remains useful for sub-terabyte workloads where you want deterministic local-disk performance.

sql 
-- Check your cluster's node configuration
SELECT
    node,
    TRIM(node_type) AS node_type,
    slices,
    disk_space_used,
    disk_space_free
FROM STV_NODE_STORAGE_CAPACITY
ORDER BY node;

-- View cluster configuration from system tables
SELECT
    TRIM(node) AS node,
    slicecount,
    is_leader
FROM STV_SLICES
GROUP BY node, slicecount, is_leader
ORDER BY node;

Each node contains multiple slices. A slice is a partition of the node's memory and disk, and each slice processes a portion of every query in parallel. An RA3.xlplus node has 2 slices, an RA3.4xlarge has 4, and an RA3.16xlarge has 16. Understanding your slice count is critical because it determines how to split your source files for optimal COPY performance.

Cluster Sizing Rule of Thumb

Start with the smallest RA3 configuration (2x RA3.xlplus) and resize based on actual query performance. Redshift Serverless is an even better starting point for experimentation — it auto-scales and you pay only for RPUs consumed. Never start with a 16-node cluster "just in case."

2. IAM Roles and VPC Configuration

Redshift needs an IAM role to access S3 for COPY and UNLOAD operations. This role is attached to the cluster at creation time (or added later via ALTER CLUSTER). The principle of least privilege applies: grant s3:GetObject and s3:ListBucket on specific bucket prefixes, not s3:* on *.

json 
// IAM policy for Redshift S3 access (least privilege)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:GetBucketLocation",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-data-lake",
        "arn:aws:s3:::my-data-lake/raw/*"
      ]
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": [
        "arn:aws:s3:::my-data-lake/unload/*"
      ]
    }
  ]
}

For networking, place your Redshift cluster in a private VPC subnet. Enable the Enhanced VPC Routing option so that all COPY and UNLOAD traffic between Redshift and S3 stays within the AWS network (via a VPC endpoint) rather than traversing the public internet. This is both a security requirement and a performance optimization.

sql 
-- Verify Enhanced VPC Routing is enabled
SELECT
    TRIM(cluster_identifier) AS cluster_id,
    enhanced_vpc_routing
FROM SVV_REDSHIFT_CLUSTERS;

-- List IAM roles attached to the cluster
SELECT
    TRIM(iam_role_name) AS iam_role,
    TRIM(iam_role_arn)  AS iam_role_arn
FROM SVV_ATTACHED_MASKING_POLICY;

3. Creating Schemas and Tables

Redshift schemas serve the same purpose as in PostgreSQL — they namespace tables and control access. A typical analytics warehouse uses schemas like raw (landing zone for COPY), staging (cleaned/transformed data), and analytics (final tables for BI tools).

sql 
-- Create schemas for a layered warehouse
CREATE SCHEMA IF NOT EXISTS raw
    AUTHORIZATION admin_user;

CREATE SCHEMA IF NOT EXISTS staging
    AUTHORIZATION admin_user;

CREATE SCHEMA IF NOT EXISTS analytics
    AUTHORIZATION admin_user;

-- Create an impressions landing table with column encoding
CREATE TABLE raw.impressions (
    impression_id    BIGINT       ENCODE az64,
    campaign_id      INTEGER      ENCODE az64,
    advertiser_id    INTEGER      ENCODE az64,
    creative_id      INTEGER      ENCODE az64,
    placement_id     INTEGER      ENCODE az64,
    impression_ts    TIMESTAMP    ENCODE az64,
    event_date       DATE         ENCODE az64,
    bid_price        DECIMAL(10,4) ENCODE az64,
    win_price        DECIMAL(10,4) ENCODE az64,
    country_code     CHAR(2)       ENCODE bytedict,
    device_type      VARCHAR(20)   ENCODE bytedict,
    browser          VARCHAR(50)   ENCODE lzo,
    user_agent       VARCHAR(500)  ENCODE lzo,
    is_viewable      BOOLEAN      ENCODE raw
)
DISTSTYLE KEY
DISTKEY (campaign_id)
SORTKEY (event_date, campaign_id);

Notice three Redshift-specific DDL features: ENCODE specifies the column compression algorithm, DISTSTYLE KEY with DISTKEY determines how rows are distributed across slices, and SORTKEY controls the on-disk ordering. We will deep-dive into distribution and sort keys in Chapter 05 — for now, understand that these decisions affect both storage efficiency and query performance.

Pro Tip

Use ENCODE AUTO on columns when you are unsure which compression to pick. Redshift will analyze a sample of the data during the first COPY and choose the optimal encoding automatically. You can review the chosen encodings later with SELECT * FROM pg_table_def WHERE tablename = 'impressions';

4. The COPY Command — Parallel Data Loading

The COPY command is the primary mechanism for loading data into Redshift. It reads files from S3, DynamoDB, or remote hosts and loads them in parallel across all slices. The key parameters you must understand are the source location, IAM role, file format, and error-handling options.

sql 
-- Basic COPY from S3 with gzip compression
COPY raw.impressions
FROM 's3://my-data-lake/raw/impressions/2024/01/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftLoadRole'
FORMAT AS CSV
IGNOREHEADER 1
GZIP
DATEFORMAT 'auto'
TIMEFORMAT 'auto'
MAXERROR 100
COMPUPDATE OFF
STATUPDATE OFF;

-- COPY with a manifest file (explicit file list)
COPY raw.impressions
FROM 's3://my-data-lake/manifests/impressions_2024_01.manifest'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftLoadRole'
MANIFEST
FORMAT AS PARQUET;

-- COPY from JSON with auto-mapping
COPY raw.click_events
FROM 's3://my-data-lake/raw/clicks/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftLoadRole'
FORMAT AS JSON 'auto'
GZIP
MAXERROR 50
TRUNCATECOLUMNS;

The COMPUPDATE OFF and STATUPDATE OFF flags are critical for performance. By default, COPY analyzes data to choose compression encodings and updates statistics after loading. On large loads, these steps add significant overhead. Turn them off during bulk loads and run ANALYZE manually afterward.

Never Use INSERT for Bulk Loads

Row-by-row INSERT statements bypass Redshift's parallel ingestion engine. A 10-million-row load that takes 30 seconds via COPY will take hours via individual INSERTs. If your ETL pipeline is generating INSERT statements, refactor it immediately to write files to S3 and use COPY instead.

5. File Splitting and Manifest Files

For optimal COPY performance, split your source data into multiple files. The ideal number of files is equal to the number of slices in your cluster, or a multiple thereof. Each slice reads one file (or a set of files) independently, achieving true parallelism.

json 
// Manifest file: explicitly list files for COPY
{
  "entries": [
    { "url": "s3://my-data-lake/raw/impressions/part-000.gz", "mandatory": true },
    { "url": "s3://my-data-lake/raw/impressions/part-001.gz", "mandatory": true },
    { "url": "s3://my-data-lake/raw/impressions/part-002.gz", "mandatory": true },
    { "url": "s3://my-data-lake/raw/impressions/part-003.gz", "mandatory": true }
  ]
}

A manifest file is a JSON document that enumerates the exact files to load. Using manifests gives you deterministic loads — you know precisely which files were loaded, and the "mandatory": true flag causes the COPY to fail if any file is missing. This is far safer than using an S3 prefix, where a missing file silently results in incomplete data.

sql 
-- Check how data was distributed across slices after COPY
SELECT
    TRIM(name) AS table_name,
    slice,
    SUM(num_values) AS row_count,
    SUM(capacity)   AS disk_mb
FROM STV_TBL_PERM
WHERE name = 'impressions'
GROUP BY name, slice
ORDER BY slice;

-- Verify row count after loading
SELECT COUNT(*) AS total_rows
FROM raw.impressions;

6. Error Handling and STL_LOAD_ERRORS

When COPY encounters malformed rows, it does not immediately fail (unless MAXERROR 0 is set). Instead, it logs errors to the STL_LOAD_ERRORS system table and continues loading. After every COPY, you should query this table to identify and remediate data quality issues.

sql 
-- Check for errors from the most recent COPY
SELECT
    starttime,
    TRIM(filename)  AS source_file,
    line_number,
    TRIM(colname)   AS column_name,
    TRIM(type)      AS column_type,
    TRIM(raw_field_value) AS bad_value,
    TRIM(err_reason) AS error_reason
FROM STL_LOAD_ERRORS
ORDER BY starttime DESC
LIMIT 20;

-- Check COPY command history and row counts
SELECT
    query,
    TRIM(filename) AS filename,
    lines_scanned,
    TRIM(curtime) AS load_time,
    status
FROM STL_LOAD_COMMITS
ORDER BY curtime DESC
LIMIT 10;

-- Summarize errors by type
SELECT
    TRIM(err_reason) AS error_reason,
    COUNT(*) AS occurrences,
    MIN(starttime) AS first_seen,
    MAX(starttime) AS last_seen
FROM STL_LOAD_ERRORS
WHERE starttime > DATEADD(hour, -24, GETDATE())
GROUP BY err_reason
ORDER BY occurrences DESC;

Common error patterns include date format mismatches (the source uses MM/DD/YYYY but the column expects YYYY-MM-DD), character encoding issues (non-UTF8 data in VARCHAR columns), and numeric overflow (a value exceeds the column's DECIMAL precision). Diagnosing these from STL_LOAD_ERRORS is far faster than guessing.

How do I reload data after fixing errors?

Use a DELETE + COPY pattern or, for full table reloads, use TRUNCATE followed by COPY. TRUNCATE is preferred because it reclaims disk space immediately and does not produce dead rows that require VACUUM.

sql 
-- Full reload pattern
BEGIN TRANSACTION;
TRUNCATE TABLE raw.impressions;
COPY raw.impressions
FROM 's3://my-data-lake/raw/impressions/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftLoadRole'
FORMAT AS CSV
IGNOREHEADER 1
GZIP;
COMMIT;
What is the UNLOAD command?

UNLOAD is the inverse of COPY — it exports query results to S3. It is useful for creating data extracts, feeding downstream systems, or archiving historical data.

sql 
-- Export data to S3 as Parquet
UNLOAD ('SELECT * FROM analytics.daily_campaign_metrics
         WHERE event_date >= $$2024-01-01$$')
TO 's3://my-data-lake/unload/campaign_metrics/'
IAM_ROLE 'arn:aws:iam::123456789012:role/RedshiftLoadRole'
FORMAT AS PARQUET
ALLOWOVERWRITE;

Self-Check

Before moving to Chapter 03, honestly answer these questions. If any answer is "no", go back and fix it.

Can you explain why COPY is faster than INSERT for bulk loading?

COPY reads source files in parallel across all slices simultaneously. Each slice independently parses and loads its portion of the data. INSERT statements, by contrast, are processed sequentially by the leader node and distributed one row at a time. For a 10-million-row dataset, COPY can be 100x faster than individual INSERTs.

Do you know how many source files to create for optimal COPY performance?

The ideal number is equal to the number of slices in your cluster, or a multiple thereof. For a 2-node RA3.xlplus cluster (4 slices), use 4, 8, or 12 source files. Each file should be roughly equal in size (within 10%). Uneven file sizes cause some slices to finish early and sit idle while others continue processing.

Can you diagnose a COPY failure using STL_LOAD_ERRORS?

Query STL_LOAD_ERRORS and look at the err_reason, colname, and raw_field_value columns. Common errors include "Delimiter not found" (wrong delimiter specified), "Invalid digit" (non-numeric data in a numeric column), and "String length exceeds DDL length" (data too long for the VARCHAR). Fix the source data or adjust the table DDL and reload.
