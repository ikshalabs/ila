Chapter 02 · Redshift Foundations

SQL for Analytics

Master the analytical SQL patterns that power ad-tech reporting — aggregations, window functions, date arithmetic, and the query idioms that transform raw impression logs into actionable campaign insights.

Estimated: 60–90 min
5 Key Concepts
3 Assignments

Why This Matters

Every dashboard your campaign managers use, every report your advertisers receive, and every optimization algorithm your bidding engine runs is ultimately powered by SQL queries against your impression and click data. The difference between an analyst who writes basic GROUP BY queries and one who fluently uses window functions, date series, and conditional aggregation is the difference between a 10-minute report and a 10-hour report.

Redshift's SQL dialect is based on PostgreSQL 8.0 with significant extensions for analytics. It supports window functions, common table expressions (CTEs), approximate count distinct via HyperLogLog, and date/time functions specifically useful for time-series ad-tech data. Mastering these patterns means you can answer complex questions — "What is the 7-day rolling average CTR by campaign?" or "Which creatives showed a significant lift in viewability last week?" — in a single query.

Key Insight

In ad-tech analytics, window functions are not optional. They are the fundamental tool for running totals (cumulative spend), ranking (top campaigns by ROAS), moving averages (7-day CTR trends), and lead/lag comparisons (day-over-day impression changes). Every query pattern in this chapter appears in real production dashboards.

Key Concepts

1. Aggregation Patterns for Campaign Reporting

The most common ad-tech query pattern is aggregating impression-level events into campaign-level metrics. This involves GROUP BY with multiple aggregate functions and conditional logic:

sql 
-- Daily campaign performance report
SELECT
    fi.event_date,
    dc.campaign_name,
    dc.advertiser_name,
    COUNT(*) AS impressions,
    SUM(CASE WHEN fi.is_clicked THEN 1 ELSE 0 END) AS clicks,
    SUM(CASE WHEN fi.is_viewable THEN 1 ELSE 0 END) AS viewable_impressions,
    SUM(fi.win_price_cpm) / 1000.0 AS total_spend,
    ROUND(
        SUM(CASE WHEN fi.is_clicked THEN 1.0 ELSE 0 END) * 100.0 /
        NULLIF(COUNT(*), 0),
    4) AS ctr_pct,
    ROUND(
        SUM(CASE WHEN fi.is_viewable THEN 1.0 ELSE 0 END) * 100.0 /
        NULLIF(COUNT(*), 0),
    2) AS viewability_pct
FROM analytics.fact_impressions fi
JOIN analytics.dim_campaign dc ON fi.campaign_id = dc.campaign_id
WHERE fi.event_date BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY fi.event_date, dc.campaign_name, dc.advertiser_name
ORDER BY fi.event_date, total_spend DESC;

Notice the use of NULLIF to prevent division by zero — a common issue when campaigns have zero impressions on a given day. The CASE WHEN pattern for conditional counting is the standard idiom in Redshift because there is no native COUNT_IF function.

2. Window Functions for Time-Series Analysis

Window functions compute values across a set of rows related to the current row without collapsing them into a single output row. They are essential for calculating running totals, rankings, and moving averages over time-series ad-tech data:

sql 
-- 7-day rolling average CTR by campaign
WITH daily_stats AS (
    SELECT
        event_date,
        campaign_id,
        COUNT(*) AS impressions,
        SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks
    FROM analytics.fact_impressions
    WHERE event_date >= '2025-01-01'
    GROUP BY event_date, campaign_id
)
SELECT
    event_date,
    campaign_id,
    impressions,
    clicks,
    ROUND(clicks * 100.0 / NULLIF(impressions, 0), 4) AS daily_ctr,
    ROUND(
        AVG(clicks * 100.0 / NULLIF(impressions, 0))
        OVER (
            PARTITION BY campaign_id
            ORDER BY event_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ),
    4) AS rolling_7d_ctr,
    SUM(impressions) OVER (
        PARTITION BY campaign_id
        ORDER BY event_date
        ROWS UNBOUNDED PRECEDING
    ) AS cumulative_impressions
FROM daily_stats
ORDER BY campaign_id, event_date;
Pro Tip

Use ROWS BETWEEN instead of RANGE BETWEEN for rolling window calculations. ROWS is based on physical row position and is faster, while RANGE is based on value comparison, which is slower and can produce unexpected results with duplicate dates.

3. Ranking and Top-N Analysis

Ad-tech reporting frequently requires finding "top 10 campaigns by spend" or "rank publishers by viewability within each advertiser." Redshift supports ROW_NUMBER(), RANK(), and DENSE_RANK():

sql 
-- Top 5 creatives per campaign by click-through rate
WITH creative_stats AS (
    SELECT
        campaign_id,
        creative_id,
        COUNT(*) AS impressions,
        SUM(CASE WHEN is_clicked THEN 1 ELSE 0 END) AS clicks,
        ROW_NUMBER() OVER (
            PARTITION BY campaign_id
            ORDER BY SUM(CASE WHEN is_clicked THEN 1.0 ELSE 0 END) /
                NULLIF(COUNT(*), 0) DESC
        ) AS ctr_rank
    FROM analytics.fact_impressions
    WHERE event_date >= DATEADD('day', -30, GETDATE())
    GROUP BY campaign_id, creative_id
    HAVING COUNT(*) >= 1000
)
SELECT * FROM creative_stats
WHERE ctr_rank <= 5
ORDER BY campaign_id, ctr_rank;

4. Date Functions and Time-Series Generation

Ad-tech reports always have a time dimension. Redshift provides powerful date functions including DATEADD, DATEDIFF, DATE_TRUNC, and EXTRACT. A common pattern is generating a complete date series to ensure reports show zero-value days:

sql 
-- Generate a date series and LEFT JOIN to fill gaps with zeros
WITH date_series AS (
    SELECT
        ('2025-01-01'::DATE + row_number OVER () - 1) AS report_date
    FROM stl_scan
    LIMIT 31
),
daily_metrics AS (
    SELECT
        event_date,
        COUNT(*) AS impressions,
        SUM(win_price_cpm) / 1000.0 AS spend
    FROM analytics.fact_impressions
    WHERE campaign_id = 42
      AND event_date BETWEEN '2025-01-01' AND '2025-01-31'
    GROUP BY event_date
)
SELECT
    ds.report_date,
    COALESCE(dm.impressions, 0) AS impressions,
    COALESCE(dm.spend, 0) AS spend,
    TO_CHAR(ds.report_date, 'Day') AS day_of_week
FROM date_series ds
LEFT JOIN daily_metrics dm ON ds.report_date = dm.event_date
ORDER BY ds.report_date;
Important

Redshift does not have a native generate_series() function like PostgreSQL. The pattern above uses ROW_NUMBER() OVER () on an existing system table to generate sequential numbers. For production use, create a permanent dim_date dimension table with pre-computed attributes (fiscal week, quarter, holiday flags, etc.).

5. Approximate Count Distinct (HyperLogLog)

Counting unique users across billions of impressions is one of the most expensive operations in ad-tech. Redshift offers APPROXIMATE COUNT(DISTINCT ...) which uses the HyperLogLog algorithm to deliver results within 2% accuracy at 10-100x the speed of exact counts:

sql 
-- Exact unique users (slow on billions of rows)
SELECT
    campaign_id,
    COUNT(DISTINCT user_cookie_id) AS exact_uniques
FROM analytics.fact_impressions
GROUP BY campaign_id;

-- Approximate unique users (fast, ~2% error margin)
SELECT
    campaign_id,
    APPROXIMATE COUNT(DISTINCT user_cookie_id) AS approx_uniques
FROM analytics.fact_impressions
GROUP BY campaign_id;

For ad-tech reach and frequency analysis, approximate counts are almost always sufficient. Reserve exact distinct counts for financial reconciliation where precision is legally required.

Self-Check

Can you explain the difference between ROW_NUMBER(), RANK(), and DENSE_RANK()?

ROW_NUMBER() assigns a unique sequential integer to each row. RANK() assigns the same rank to tied values but skips subsequent ranks (1,2,2,4). DENSE_RANK() gives tied values the same rank without skipping (1,2,2,3). For ad-tech top-N queries, ROW_NUMBER() is usually preferred because it guarantees exactly N results.

Why should you use NULLIF() in CTR calculations?

CTR is calculated as clicks / impressions * 100. If a campaign has zero impressions on a given day, dividing by zero causes an error. NULLIF(impressions, 0) returns NULL when impressions is zero, which makes the division return NULL instead of an error. Use COALESCE to convert NULL to 0 for display.

When would you use APPROXIMATE COUNT DISTINCT instead of exact COUNT DISTINCT?

Use approximate counts when the table has hundreds of millions or billions of rows and you need unique user counts for dashboards where 2% error margin is acceptable. For financial reporting (unique advertisers, exact campaign counts), use exact COUNT(DISTINCT). For reach/frequency analysis on impression data, the approximate version can be 10-100x faster.
