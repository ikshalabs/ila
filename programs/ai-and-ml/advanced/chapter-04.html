Chapter 04 · Phase 2: Generative AI

Generative AI Foundations

Understand the core principles that power modern generative AI systems — from autoregressive language modeling and instruction tuning to Reinforcement Learning from Human Feedback (RLHF), prompt engineering, and the emergent capabilities that appear at scale. This chapter connects the theoretical foundations of generative models to the practical systems that power ChatGPT, Claude, and Gemini.

Estimated: 4–5 days
6 Core Concepts
3 Assignments

Why This Matters

The generative AI revolution is built on a surprisingly simple idea: train a model to predict the next token, give it enough data and parameters, and it develops capabilities that were never explicitly programmed. Understanding why this works — and where it fails — is the difference between using generative AI tools effectively and being mystified by their behavior.

This chapter covers the foundational concepts that every AI/ML engineer must internalize: how autoregressive language models work, why instruction tuning and RLHF transform raw next-token predictors into useful assistants, how prompt engineering exploits the latent structure of pretrained models, and what emergent capabilities mean for the future of AI systems.

Whether you are building applications on top of foundation models, fine-tuning models for specific tasks, or evaluating whether generative AI is appropriate for a use case, the concepts in this chapter are prerequisites for informed decision-making.

Key Insight

Generative AI systems are not databases that store and retrieve facts. They are probabilistic models that learn statistical patterns in text. This means they can generate fluent, coherent text about topics they have "seen" during training — but they can also confidently produce plausible-sounding nonsense. Understanding this distinction is the single most important concept in this chapter.

Key Concepts

1. Autoregressive Language Modeling

At the heart of every modern LLM is autoregressive language modeling: the task of predicting the next token given all previous tokens. Given a sequence of tokens [t1, t2, ..., tn], the model learns to estimate P(tn+1 | t1, ..., tn). The entire training process optimizes the model to minimize the cross-entropy loss between its predicted token distribution and the actual next token in the training data.

This seemingly simple objective produces remarkably capable models because natural language is information-dense. To predict the next word in "The capital of France is ___", the model must encode the fact that France's capital is Paris. To predict the next token in a code snippet, it must understand programming syntax and semantics. Next-token prediction is a proxy for a vast range of cognitive tasks. 

  Autoregressive Generation (next-token prediction):

  Input:   "The capital of France is"
                                        ┌─────────────┐
  Tokens:  [The] [capital] [of] [France] [is] ──▶ │  Transformer  │
                                                    │  (decoder)    │
                                                    └──────┬────────┘
                                                           │
                                                    ┌──────▼────────┐
                                                    │  Softmax over │
                                                    │  vocabulary   │
                                                    │  (50,000+     │
                                                    │   tokens)     │
                                                    └──────┬────────┘
                                                           │
  Probabilities:  "Paris": 0.87, "Lyon": 0.03, "the": 0.02, ...
                                                           │
  Sampling:  ──────────────────────────────────▶  [Paris]

  Then: [The] [capital] [of] [France] [is] [Paris] ──▶ next token...
Figure 1: Autoregressive generation via iterative next-token prediction
python
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

# Load a pretrained autoregressive language model
model_name = "meta-llama/Llama-2-7b-hf"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,
    device_map="auto",
)

# Tokenize input prompt
prompt = "The three laws of thermodynamics are:"
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

# Examine the next-token probability distribution
with torch.no_grad():
    outputs = model(**inputs)
    logits = outputs.logits[:, -1, :]  # logits for the last position
    probs = torch.softmax(logits, dim=-1)

# Get top-10 most likely next tokens
top_probs, top_indices = torch.topk(probs, 10)
for prob, idx in zip(top_probs[0], top_indices[0]):
    token = tokenizer.decode(idx)
    print(f"  {token!r:15s} p={prob:.4f}")

# Generate full response with different sampling strategies
# Greedy: always pick the most likely token
greedy_output = model.generate(**inputs, max_new_tokens=100, do_sample=False)

# Top-k sampling: sample from top k tokens
topk_output = model.generate(**inputs, max_new_tokens=100,
                             do_sample=True, top_k=50, temperature=0.7)

# Nucleus (top-p) sampling: sample from smallest set with cumulative prob >= p
nucleus_output = model.generate(**inputs, max_new_tokens=100,
                                do_sample=True, top_p=0.9, temperature=0.8)
Temperature Controls Randomness

Temperature scales the logits before softmax: T < 1.0 makes the distribution sharper (more deterministic, more repetitive), T > 1.0 makes it flatter (more creative, more random). At T=0, generation is greedy (always picks the top token). For factual tasks use T=0–0.3; for creative writing use T=0.7–1.0.

2. Pretraining, Fine-Tuning, and Instruction Tuning

Modern LLMs follow a three-stage training pipeline. First, pretraining on massive corpora (trillions of tokens from the internet, books, and code) creates a "foundation model" that understands language structure and has absorbed broad world knowledge. This stage requires thousands of GPUs and costs millions of dollars.

Second, instruction tuning (also called supervised fine-tuning or SFT) trains the model on curated (prompt, response) pairs that demonstrate the desired behavior. This transforms the base model from a text completer into an instruction follower. The training data is much smaller (thousands to hundreds of thousands of examples) but must be carefully curated for quality.

Third, alignment via RLHF or similar techniques adjusts the model to produce outputs that humans prefer, making it more helpful, harmless, and honest.

python
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer
from datasets import load_dataset
from peft import LoraConfig

# Load base model and tokenizer
model_name = "meta-llama/Llama-2-7b-hf"
model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype=torch.float16)
tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = tokenizer.eos_token

# Instruction tuning dataset format
# Each example has a prompt (instruction) and a completion (response)
dataset = load_dataset("json", data_files="instruction_data.jsonl")

def format_instruction(example):
    """Format as: ### Instruction:\n{prompt}\n\n### Response:\n{response}"""
    return {
        "text": f"""### Instruction:
{example['instruction']}

### Response:
{example['response']}{tokenizer.eos_token}"""
    }

dataset = dataset.map(format_instruction)

# Use LoRA for parameter-efficient fine-tuning
lora_config = LoraConfig(
    r=16,                    # rank of update matrices
    lora_alpha=32,            # scaling factor
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj"],
    bias="none",
    task_type="CAUSAL_LM",
)

# Training configuration
training_args = TrainingArguments(
    output_dir="./sft_output",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-4,
    warmup_steps=100,
    logging_steps=10,
    save_strategy="epoch",
    fp16=True,
)

# Supervised Fine-Tuning with TRL library
trainer = SFTTrainer(
    model=model,
    train_dataset=dataset["train"],
    args=training_args,
    peft_config=lora_config,
    dataset_text_field="text",
    max_seq_length=2048,
    tokenizer=tokenizer,
)
trainer.train()

3. RLHF and Alignment

Reinforcement Learning from Human Feedback (RLHF) is the technique that makes LLMs helpful rather than merely capable. The process works in three steps: (1) collect human comparisons of model outputs ("response A is better than response B"), (2) train a reward model that predicts human preferences, and (3) use RL (typically PPO) to optimize the language model to produce outputs that score highly according to the reward model.

Direct Preference Optimization (DPO) is a simpler alternative to RLHF that eliminates the need for a separate reward model and RL training. DPO directly optimizes the policy using a binary cross-entropy loss on preference pairs, making it more stable and easier to implement. 

  RLHF Pipeline:

  Step 1: Collect Preferences
  ┌─────────────────────────────────────────────────┐
  │  Prompt: "Explain quantum entanglement"          │
  │                                                  │
  │  Response A: [detailed, accurate explanation]    │
  │  Response B: [vague, slightly incorrect]         │
  │                                                  │
  │  Human annotator: A > B                          │
  └──────────────────────┬──────────────────────────┘
                         ▼
  Step 2: Train Reward Model
  ┌─────────────────────────────────────────────────┐
  │  Reward Model = f(prompt, response) -> score     │
  │  Trained on thousands of human comparisons       │
  │  Loss: -log(sigmoid(r(preferred) - r(rejected))) │
  └──────────────────────┬──────────────────────────┘
                         ▼
  Step 3: RL Fine-Tuning (PPO)
  ┌─────────────────────────────────────────────────┐
  │  Objective: maximize E[reward(response)]         │
  │             - beta * KL(policy || reference)     │
  │                                                  │
  │  KL penalty prevents model from drifting too     │
  │  far from the SFT checkpoint (avoids "reward     │
  │  hacking" where model exploits reward model)     │
  └─────────────────────────────────────────────────┘
Figure 2: The three-step RLHF pipeline for aligning language models
python
from trl import DPOTrainer, DPOConfig
from transformers import AutoModelForCausalLM, AutoTokenizer
from datasets import load_dataset

# DPO: Direct Preference Optimization (simpler than RLHF)
# Requires paired preference data: (prompt, chosen, rejected)

model = AutoModelForCausalLM.from_pretrained("./sft_checkpoint")
ref_model = AutoModelForCausalLM.from_pretrained("./sft_checkpoint")
tokenizer = AutoTokenizer.from_pretrained("./sft_checkpoint")

# Preference dataset format:
# {"prompt": "...", "chosen": "good response", "rejected": "bad response"}
dataset = load_dataset("json", data_files="preferences.jsonl")

# DPO loss: -log sigmoid(beta * (log pi(chosen) - log pi(rejected)
#           - log ref(chosen) + log ref(rejected)))

training_args = DPOConfig(
    output_dir="./dpo_output",
    num_train_epochs=1,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=5e-7,
    beta=0.1,                 # KL penalty strength
    loss_type="sigmoid",       # standard DPO loss
    warmup_steps=50,
    fp16=True,
)

# Train with DPO
dpo_trainer = DPOTrainer(
    model=model,
    ref_model=ref_model,
    args=training_args,
    train_dataset=dataset["train"],
    tokenizer=tokenizer,
)
dpo_trainer.train()

# The resulting model prefers outputs similar to "chosen"
# and avoids outputs similar to "rejected"

4. Prompt Engineering

Prompt engineering is the practice of designing inputs that elicit the desired behavior from a pretrained LLM without modifying its weights. It works because the model has learned to condition on context during pretraining: if the prompt looks like an expert explanation, the model will generate text in that style. Key techniques include:

  • Zero-shot prompting — describe the task directly with no examples. Relies entirely on the model's pretraining knowledge.
  • Few-shot prompting — include 2–5 input/output examples before the actual query. The model learns the pattern from the examples.
  • Chain-of-thought (CoT) — instruct the model to reason step by step. Dramatically improves performance on math, logic, and multi-step reasoning tasks.
  • System prompts — set the model's role, constraints, and output format upfront. Essential for building reliable applications.
python
from openai import OpenAI

client = OpenAI()

# ── Zero-shot prompting ──
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "user", "content": "Classify the sentiment: 'This product is terrible.' -> "}
    ]
)

# ── Few-shot prompting ──
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": "Classify text sentiment as POSITIVE, NEGATIVE, or NEUTRAL."},
        {"role": "user", "content": "'I love this!' -> POSITIVE\n'Not great.' -> NEGATIVE\n'It arrived on Tuesday.' -> NEUTRAL\n'This product is terrible.' -> "}
    ]
)

# ── Chain-of-thought prompting ──
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": "You are a math tutor. Think step by step."},
        {"role": "user", "content": """A store sells apples for $2 each and oranges for $3 each.
If Sarah buys 5 apples and 3 oranges, and she has a 20% discount coupon,
how much does she pay?

Let's solve this step by step:"""}
    ]
)

# ── Structured output with system prompt ──
response = client.chat.completions.create(
    model="gpt-4",
    messages=[
        {"role": "system", "content": """You are a data extraction assistant.
Extract entities from the text and return valid JSON:
{"people": [...], "organizations": [...], "locations": [...]}
Only include entities explicitly mentioned. No explanations."""},
        {"role": "user", "content": "Tim Cook announced that Apple will open a new office in Berlin next quarter."}
    ],
    response_format={"type": "json_object"},
)
Pro Tip

The most effective prompts are specific about format and explicit about constraints. Instead of "Summarize this article", write "Summarize this article in exactly 3 bullet points, each under 20 words, focusing on actionable findings. Do not include background information." The more precise your instructions, the more consistent the output.

5. Scaling Laws and Emergent Capabilities

Scaling laws describe the predictable relationship between model performance and three factors: the number of parameters (N), the amount of training data (D), and the compute budget (C). Kaplan et al. (2020) showed that loss decreases as a power law with each of these factors, and the Chinchilla paper (Hoffmann et al., 2022) showed that parameters and data should scale roughly equally for optimal compute efficiency.

More surprising are emergent capabilities: abilities that appear suddenly at specific scale thresholds rather than improving gradually. For example, chain-of-thought reasoning, multi-step arithmetic, and word unscrambling show near-zero performance in smaller models but jump to high accuracy once the model exceeds a critical size. This makes it difficult to predict what a larger model will be capable of.

python
import numpy as np
import matplotlib.pyplot as plt

# Chinchilla scaling law: optimal ratio of parameters to tokens
# For a compute budget C, optimal N ~ C^0.5, optimal D ~ C^0.5
# Rule of thumb: train on ~20 tokens per parameter

def chinchilla_optimal(compute_flops):
    """Estimate optimal model size and training tokens for a compute budget."""
    # From Hoffmann et al. (2022)
    optimal_params = 0.6 * (compute_flops / 6) ** 0.5
    optimal_tokens = compute_flops / (6 * optimal_params)
    return optimal_params, optimal_tokens

# Example: given 10^24 FLOPs of compute
params, tokens = chinchilla_optimal(1e24)
print(f"Optimal model: {params/1e9:.1f}B parameters")
print(f"Optimal data:  {tokens/1e12:.1f}T tokens")
print(f"Tokens/param:  {tokens/params:.0f}x")

# Scaling law: loss as a function of compute
# L(C) = A * C^(-alpha) + L_irreducible
# where alpha ~ 0.05 and L_irreducible is the entropy of natural language

compute_budgets = np.logspace(18, 25, 100)
losses = 3.5 * compute_budgets ** (-0.05) + 1.7  # approximate

# Key insight: doubling compute reduces loss by ~3.4%
# This predictable relationship enables planning massive training runs
Emergent Capabilities are Unpredictable

You cannot reliably predict which capabilities will emerge at a given scale. A model that cannot do 3-digit multiplication at 10B parameters might suddenly succeed at 70B parameters. This means small-scale experiments cannot fully predict large-scale behavior — and safety evaluations must be conducted at the full deployment scale.

6. Tokenization and Context Windows

LLMs do not process raw text — they operate on tokens, which are sub-word units learned from the training corpus. Common tokenization algorithms include Byte Pair Encoding (BPE), WordPiece, and SentencePiece. Understanding tokenization is critical because it determines the model's vocabulary, how it handles different languages and domains, and how much text fits in the context window.

The context window (or context length) is the maximum number of tokens a model can process in a single forward pass. It bounds how much information the model can attend to when generating a response. Modern models range from 4K tokens (early GPT-3.5) to 128K+ tokens (GPT-4 Turbo, Claude 3). Longer contexts enable working with entire documents but increase computational cost quadratically with standard attention.

python
import tiktoken

# GPT-4 uses the cl100k_base tokenizer (100,000 token vocabulary)
enc = tiktoken.get_encoding("cl100k_base")

# Tokenize a string
text = "Generative AI is transforming how we build software."
tokens = enc.encode(text)
print(f"Text: {text}")
print(f"Tokens: {tokens}")
print(f"Token count: {len(tokens)}")

# Decode individual tokens to see how text is split
for t in tokens:
    print(f"  {t:6d} -> {enc.decode([t])!r}")

# Token counting for context window management
def count_tokens(text, model="gpt-4"):
    enc = tiktoken.encoding_for_model(model)
    return len(enc.encode(text))

# Practical considerations:
# - English text: ~1 token per 4 characters, ~0.75 tokens per word
# - Code: more tokens per character (special characters, indentation)
# - Non-English: often 2-4x more tokens per word (less efficient)
# - GPT-4 Turbo: 128K context = ~300 pages of text

# Context window budget for a RAG application:
system_prompt_tokens = count_tokens(system_prompt)
context_tokens = count_tokens(retrieved_documents)
query_tokens = count_tokens(user_query)
max_response_tokens = 1000

total = system_prompt_tokens + context_tokens + query_tokens + max_response_tokens
print(f"Total token budget: {total} / 128000")
if total > 128000:
    print("WARNING: exceeds context window! Reduce retrieved docs.")
Why does BPE tokenization handle rare words better than word-level tokenization?

BPE builds its vocabulary by iteratively merging the most frequent character pairs. Common words like "the" become single tokens, while rare words are split into known sub-word pieces. For example, "unfathomable" might become ["un", "fath", "om", "able"]. This means the model never encounters a true "unknown" token — every word can be represented as a sequence of sub-word units. Word-level tokenization would assign rare words to a single [UNK] token, losing all information.

How does the context window affect application architecture?

The context window is a hard limit on what the model can "see" in a single inference call. Applications that need to process more text than the context window must use techniques like: (1) chunking and summarization (process document sections independently and aggregate), (2) RAG (retrieve only the most relevant sections), (3) sliding window approaches (process overlapping chunks), or (4) use models with very long contexts (128K+). Cost also scales with context length, so minimizing input tokens reduces API costs.

Self-Check

Before moving to Chapter 05, honestly answer these questions.

Can you explain how autoregressive language modeling creates general intelligence?

It does not create general intelligence. Autoregressive LMs learn statistical patterns in text. Because natural language encodes vast amounts of human knowledge, predicting next tokens requires learning factual knowledge, reasoning patterns, coding ability, and more. The model develops these as byproducts of compression: to predict text well, it must build internal representations of the concepts described in that text. This is powerful but fundamentally different from true understanding.

What is the difference between SFT and RLHF, and why do you need both?

SFT (Supervised Fine-Tuning) teaches the model what to say by training on high-quality (instruction, response) pairs. RLHF/DPO teaches the model how to say it better by learning from human preferences between alternative responses. SFT alone produces a model that follows instructions but may generate unhelpful, verbose, or unsafe responses. RLHF/DPO refines the model to prefer the style, safety, and quality that humans actually want.

Why does chain-of-thought prompting improve reasoning performance?

LLMs generate tokens left to right. Without CoT, the model must compute the answer in the "first token" — it has no scratchpad. CoT gives the model intermediate tokens to "think through" the problem step by step. Each generated step provides context for the next, effectively expanding the model's effective computation budget. This is analogous to a human showing their work on a math problem rather than guessing the answer directly.

How do you decide between prompt engineering and fine-tuning for a specific task?

Use prompt engineering when: (1) you have no training data, (2) the task is straightforward for the base model, (3) you need to iterate quickly, or (4) you want to avoid the cost of fine-tuning. Use fine-tuning when: (1) prompt engineering consistently fails, (2) you need domain-specific knowledge or style, (3) you have high-quality training data, (4) you need consistent formatting that prompting cannot reliably produce, or (5) you need to reduce token costs (a fine-tuned model with a short prompt can replace a base model with a long few-shot prompt).

Can you explain what Chinchilla-optimal training means?

Chinchilla-optimal means scaling model parameters and training data equally for a given compute budget. The Chinchilla paper showed that previous models (like GPT-3) were under-trained: they had too many parameters relative to the amount of training data. The optimal ratio is roughly 20 tokens per parameter. So a 7B model should be trained on ~140B tokens, and a 70B model on ~1.4T tokens. Training smaller models on more data (Chinchilla approach) consistently outperforms training larger models on less data.
