Chapter 09 · Phase 4: Production Operations

Production ML Systems

Design and operate end-to-end ML systems at scale: feature stores, model serving patterns, batch vs real-time inference, latency optimization, caching strategies, and system design for high-availability ML platforms.

Estimated: 4–5 days

Why This Matters

Training a model is perhaps 10% of the work in a production ML system. The remaining 90% is everything that surrounds it: data pipelines, feature engineering, serving infrastructure, monitoring, and operational tooling. Many teams can train an accurate model in a notebook, but struggle to serve it reliably at scale with consistent latency, high availability, and reasonable cost.

Production ML systems differ from research prototypes in every dimension. They must handle variable traffic patterns, degrade gracefully under load, serve predictions within strict latency budgets (often under 50ms), and maintain consistency between training and serving feature computations. A model that is 2% more accurate but 10x slower is usually the wrong choice for production.

This chapter covers the architecture patterns and engineering practices that separate production ML platforms from notebook experiments. You will learn to design systems that are fast, reliable, and maintainable — systems that other engineers can operate at 3 AM without needing to understand the mathematics behind the model.

Key Insight

The fundamental challenge of production ML is bridging two worlds: the data science world (experimentation, iteration, accuracy metrics) and the software engineering world (reliability, latency, maintainability, on-call rotations). Every architectural decision in this chapter is about making that bridge robust.

Key Concepts

1. Feature Stores

A feature store is the central repository for computed features in a production ML system. Without one, feature engineering logic is duplicated between training pipelines and serving code, creating training-serving skew — one of the most insidious bugs in ML systems. The training pipeline might compute a feature as a 30-day rolling average, while the serving code computes a 28-day average, and the model silently degrades.

Feature stores solve this by providing a single source of truth for feature definitions. Features are computed once, stored consistently, and served with low latency. Modern feature stores like Feast provide both an offline store (for historical features used in training) and an online store (for low-latency serving).

python 
# feature_store.py — Feature store with Feast for online/offline consistency
from datetime import timedelta
from feast import (
    Entity, Feature, FeatureView, FileSource,
    ValueType, FeatureStore,
)

# Define an entity (the "primary key" for feature lookup)
user = Entity(
    name="user_id",
    value_type=ValueType.INT64,
    description="Unique identifier for a user",
)

# Define the data source (offline store)
user_features_source = FileSource(
    path="data/user_features.parquet",
    event_timestamp_column="event_timestamp",
    created_timestamp_column="created_timestamp",
)

# Define a feature view (a group of related features)
user_activity_view = FeatureView(
    name="user_activity",
    entities=["user_id"],
    ttl=timedelta(days=1),
    features=[
        Feature(name="session_count_7d", dtype=ValueType.INT64),
        Feature(name="avg_session_duration_7d", dtype=ValueType.FLOAT),
        Feature(name="last_purchase_days_ago", dtype=ValueType.INT64),
        Feature(name="total_spend_30d", dtype=ValueType.FLOAT),
        Feature(name="page_views_24h", dtype=ValueType.INT64),
    ],
    online=True,
    source=user_features_source,
)


class MLFeatureService:
    """Unified feature retrieval for training and serving."""

    def __init__(self, repo_path: str = "feature_repo/"):
        self.store = FeatureStore(repo_path=repo_path)

    def get_training_features(self, entity_df):
        """Get point-in-time correct features for training.

        Prevents data leakage by only using features available
        at the time each training example was generated.
        """
        return self.store.get_historical_features(
            entity_df=entity_df,
            features=[
                "user_activity:session_count_7d",
                "user_activity:avg_session_duration_7d",
                "user_activity:last_purchase_days_ago",
                "user_activity:total_spend_30d",
                "user_activity:page_views_24h",
            ],
        ).to_df()

    def get_serving_features(self, user_ids: list):
        """Get features for real-time inference from online store.

        Returns features with sub-10ms latency from Redis/DynamoDB.
        """
        entity_rows = [{"user_id": uid} for uid in user_ids]
        return self.store.get_online_features(
            features=[
                "user_activity:session_count_7d",
                "user_activity:avg_session_duration_7d",
                "user_activity:last_purchase_days_ago",
                "user_activity:total_spend_30d",
                "user_activity:page_views_24h",
            ],
            entity_rows=entity_rows,
        ).to_dict()

    def materialize_features(self, start_date, end_date):
        """Push features from offline to online store.

        Run this on a schedule (e.g., every hour) to keep
        the online store fresh.
        """
        self.store.materialize(
            start_date=start_date,
            end_date=end_date,
        )
        print(f"Materialized features from {start_date} to {end_date}")
Point-in-Time Correctness

When retrieving historical features for training, you must use point-in-time joins. If you simply join on user_id without respecting timestamps, you will leak future information into your training data. The model will appear to perform well in offline evaluation but fail in production because it was trained on features that would not have been available at prediction time.

2. Model Serving Patterns

There are three primary patterns for serving ML predictions, and each fits different use cases. Choosing the wrong pattern is one of the most expensive architectural mistakes you can make, because it is difficult to change once deployed.

  • Synchronous REST/gRPC — Client sends a request, waits for a prediction. Best for user-facing applications with sub-100ms latency requirements (search ranking, recommendation, fraud detection).
  • Asynchronous message queue — Client publishes a message, consumer generates predictions. Best for workflows where the caller does not need an immediate response (email classification, content moderation queues).
  • Batch inference — Predictions are precomputed for all entities on a schedule. Best for large-scale scoring where freshness requirements are measured in hours, not milliseconds (daily churn scores, weekly product recommendations).

@app.post("/predict", response_model=PredictionResponse)

    """Synchronous prediction — blocks until result is ready."""
    start = time.perf_counter()
    features = np.array(request.features).reshape(1, -1)
    prediction = model.predict(features)[0]

# ── Pattern 3: Batch inference ───────────────────
class BatchInferenceRunner:
    """Precompute predictions for all entities on a schedule."""

    def __init__(self, model, feature_store, output_store):
        self.model = model
        self.feature_store = feature_store
        self.output_store = output_store

    def run_batch(self, entity_ids: List[int], batch_size: int = 1024):
        """Score all entities in batches for throughput."""
        results = {}
        for i in range(0, len(entity_ids), batch_size):
            batch_ids = entity_ids[i:i + batch_size]
            features = self.feature_store.get_serving_features(batch_ids)
            X = np.array([features[uid] for uid in batch_ids])
            predictions = self.model.predict(X)
            for uid, pred in zip(batch_ids, predictions):
                results[uid] = float(pred)
        self.output_store.bulk_write(results)
        print(f"Batch scored {len(results)} entities")
        return results

3. Latency Optimization

In production ML, latency is often the binding constraint. A recommendation model that takes 500ms to respond makes the page feel sluggish. A fraud detection model that takes 2 seconds blocks the payment flow. Latency optimization happens at every layer of the stack: model architecture, feature retrieval, inference runtime, and network.

The most effective optimizations, in roughly decreasing order of impact, are: model distillation (replace a large model with a smaller one trained to mimic it), quantization (reduce weight precision from float32 to int8), batching (group multiple requests for GPU efficiency), and caching (skip inference entirely for repeated inputs).

python 
# latency_optimization.py — Techniques for fast inference
import numpy as np
import time
import hashlib
import onnxruntime as ort
from functools import lru_cache
from typing import List, Dict
from collections import deque
import asyncio


class ONNXOptimizedModel:
    """Serve models via ONNX Runtime for 2-10x speedup over native."""

    def __init__(self, model_path: str):
        # Configure ONNX Runtime for optimal performance
        sess_options = ort.SessionOptions()
        sess_options.graph_optimization_level = (
            ort.GraphOptimizationLevel.ORT_ENABLE_ALL
        )
        sess_options.intra_op_num_threads = 4
        sess_options.inter_op_num_threads = 1
        sess_options.execution_mode = ort.ExecutionMode.ORT_SEQUENTIAL

        self.session = ort.InferenceSession(
            model_path, sess_options,
            providers=["CPUExecutionProvider"],
        )
        self.input_name = self.session.get_inputs()[0].name

    def predict(self, X: np.ndarray) -> np.ndarray:
        """Run inference with ONNX Runtime."""
        X = X.astype(np.float32)
        outputs = self.session.run(None, {self.input_name: X})
        return outputs[0]


class PredictionCache:
    """Cache predictions to avoid redundant inference.

    For many applications, the same user requests predictions
    multiple times with identical features. Caching eliminates
    redundant computation.
    """

    def __init__(self, max_size: int = 10000, ttl_seconds: int = 300):
        self.max_size = max_size
        self.ttl = ttl_seconds
        self.cache: Dict[str, dict] = {}
        self.access_order = deque()
        self.hits = 0
        self.misses = 0

    def _hash_features(self, features: np.ndarray) -> str:
        """Create a deterministic hash of the feature vector."""
        return hashlib.md5(features.tobytes()).hexdigest()

    def get(self, features: np.ndarray):
        """Look up cached prediction."""
        key = self._hash_features(features)
        if key in self.cache:
            entry = self.cache[key]
            if time.time() - entry["timestamp"] < self.ttl:
                self.hits += 1
                return entry["prediction"]
            else:
                del self.cache[key]
        self.misses += 1
        return None

    def put(self, features: np.ndarray, prediction: float):
        """Store a prediction in the cache."""
        if len(self.cache) >= self.max_size:
            oldest = self.access_order.popleft()
            self.cache.pop(oldest, None)
        key = self._hash_features(features)
        self.cache[key] = {
            "prediction": prediction,
            "timestamp": time.time(),
        }
        self.access_order.append(key)

    @property
    def hit_rate(self) -> float:
        total = self.hits + self.misses
        return self.hits / total if total > 0 else 0.0


class DynamicBatcher:
    """Batch incoming requests for GPU efficiency.

    Individual requests are queued and batched together
    to maximize GPU utilization. This trades a small amount
    of latency for much higher throughput.
    """

    def __init__(self, model, max_batch: int = 32, max_wait_ms: float = 10):
        self.model = model
        self.max_batch = max_batch
        self.max_wait = max_wait_ms / 1000
        self.queue: List = []
        self.lock = asyncio.Lock()

    async def predict(self, features: np.ndarray) -> float:
        """Submit a request and wait for batched result."""
        future = asyncio.get_event_loop().create_future()
        async with self.lock:
            self.queue.append((features, future))
            if len(self.queue) >= self.max_batch:
                await self._flush()
        # Wait for max_wait_ms then flush remaining
        await asyncio.sleep(self.max_wait)
        async with self.lock:
            if self.queue:
                await self._flush()
        return await future

    async def _flush(self):
        """Process all queued requests as a single batch."""
        batch = self.queue[:]
        self.queue.clear()
        X = np.vstack([f for f, _ in batch])
        predictions = self.model.predict(X)
        for (_, future), pred in zip(batch, predictions):
            future.set_result(float(pred))
Measure Before You Optimize

Profile your serving pipeline end-to-end before optimizing. In many systems, the bottleneck is not model inference — it is feature retrieval, network serialization, or database lookups. Optimizing the model from 5ms to 2ms is wasted effort if feature retrieval takes 200ms. Use distributed tracing (Chapter 07) to find the real bottleneck.

4. Batch vs Real-Time Inference

The choice between batch and real-time inference is not binary — most production systems use a hybrid architecture called lambda architecture (or more recently, kappa architecture). Batch predictions handle the base load cheaply, and real-time inference handles cases where freshness matters.

The key tradeoff is freshness vs cost. Batch inference processes millions of predictions for pennies per prediction, but the results may be hours old. Real-time inference gives fresh predictions but costs 10-100x more per prediction due to infrastructure overhead.

 │  │   Store      │     │  (FastAPI/TF)   │     │   Layer      │──▶ Client
 │  │  (Online)    │     │  (per-request)   │     │  (fallback)  │  │

 │  otherwise fall through to real-time model server.              │

Hybrid architecture: batch for throughput, real-time for freshness
python 
# hybrid_serving.py — Combine batch and real-time inference
import time
import redis
import json
import numpy as np
from typing import Optional


class HybridServingLayer:
    """Serve predictions from batch store with real-time fallback.

    1. Check if a fresh batch prediction exists
    2. If yes, return it (fast, cheap)
    3. If no, compute real-time prediction (slower, expensive)
    """

    def __init__(
        self,
        prediction_store,
        realtime_model,
        feature_service,
        freshness_threshold_seconds: int = 3600,
    ):
        self.prediction_store = prediction_store
        self.realtime_model = realtime_model
        self.feature_service = feature_service
        self.freshness_threshold = freshness_threshold_seconds
        self.batch_hits = 0
        self.realtime_hits = 0

    def predict(self, user_id: int) -> dict:
        """Get prediction, preferring batch with real-time fallback."""
        # Try batch prediction first
        cached = self.prediction_store.get(f"pred:{user_id}")
        if cached:
            entry = json.loads(cached)
            age = time.time() - entry["timestamp"]
            if age < self.freshness_threshold:
                self.batch_hits += 1
                return {
                    "prediction": entry["prediction"],
                    "source": "batch",
                    "age_seconds": round(age),
                }

        # Fall through to real-time
        self.realtime_hits += 1
        features = self.feature_service.get_serving_features([user_id])
        X = np.array(features[user_id]).reshape(1, -1)
        prediction = self.realtime_model.predict(X)[0]

        return {
            "prediction": float(prediction),
            "source": "realtime",
            "age_seconds": 0,
        }

    @property
    def batch_ratio(self) -> float:
        """What fraction of requests are served from batch?"""
        total = self.batch_hits + self.realtime_hits
        return self.batch_hits / total if total > 0 else 0.0

5. ML System Design at Scale

Designing an ML system for scale requires thinking about failure modes that do not exist in single-machine setups. What happens when the model server is down? What happens when features are stale? What happens when a new model version produces worse results than the previous one? Every production ML system needs answers to these questions.

The core design principles are: graceful degradation (serve stale predictions or a fallback model rather than failing), canary deployments (route a small percentage of traffic to the new model before full rollout), and circuit breakers (automatically stop sending traffic to a failing component).

python 
# ml_system_design.py — Production resilience patterns
import time
import random
import logging
from enum import Enum
from typing import Optional, Callable

logger = logging.getLogger(__name__)


class CircuitState(Enum):
    CLOSED = "closed"       # Normal operation
    OPEN = "open"           # Failing, reject requests
    HALF_OPEN = "half_open" # Testing recovery


class CircuitBreaker:
    """Prevent cascading failures in ML serving pipelines.

    When the model server starts failing, the circuit breaker
    trips open and returns fallback values instead of waiting
    for timeouts.
    """

    def __init__(
        self,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        fallback_fn: Optional[Callable] = None,
    ):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.fallback_fn = fallback_fn
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0

    def call(self, fn: Callable, *args, **kwargs):
        """Execute fn with circuit breaker protection."""
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                logger.info("Circuit breaker: testing recovery")
            else:
                logger.warning("Circuit open: returning fallback")
                return self.fallback_fn(*args, **kwargs)

        try:
            result = fn(*args, **kwargs)
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                logger.info("Circuit breaker: recovered")
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                logger.error(
                    f"Circuit OPEN after {self.failure_count} failures"
                )
            return self.fallback_fn(*args, **kwargs)


class CanaryRouter:
    """Route traffic between model versions for safe rollouts.

    Gradually shift traffic from the current model to the
    candidate model while monitoring key metrics.
    """

    def __init__(self, current_model, candidate_model):
        self.current = current_model
        self.candidate = candidate_model
        self.canary_percent = 0.0
        self.current_count = 0
        self.candidate_count = 0

    def predict(self, features, request_id: str = ""):
        """Route request based on canary percentage.

        Uses deterministic routing based on request_id
        so the same user always gets the same model version.
        """
        if request_id:
            use_canary = (hash(request_id) % 100) < self.canary_percent * 100
        else:
            use_canary = random.random() < self.canary_percent

        if use_canary:
            self.candidate_count += 1
            return self.candidate.predict(features), "candidate"
        else:
            self.current_count += 1
            return self.current.predict(features), "current"

    def set_canary_percent(self, percent: float):
        """Adjust traffic split. 0.0 = all current, 1.0 = all candidate."""
        self.canary_percent = max(0.0, min(1.0, percent))
        logger.info(f"Canary traffic: {self.canary_percent:.0%} to candidate")

    def promote_candidate(self):
        """Make the candidate model the new current model."""
        self.current = self.candidate
        self.canary_percent = 0.0
        self.candidate = None
        logger.info("Candidate promoted to current")

# ml_platform.py — Wiring the platform components together
import mlflow
import numpy as np
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List
import logging

logger = logging.getLogger(__name__)
app = FastAPI(title="ML Platform Server")


class ModelManager:
    """Manage model lifecycle: load, version, swap.

    Loads the production model from MLflow model registry
    and supports hot-swapping to new versions without
    restarting the server.
    """

    def __init__(self, model_name: str, stage: str = "Production"):
        self.model_name = model_name
        self.stage = stage
        self.model = None
        self.version = None
        self.load_model()

    def load_model(self):
        """Load the latest production model from MLflow."""
        model_uri = f"models:/{self.model_name}/{self.stage}"
        self.model = mlflow.pyfunc.load_model(model_uri)
        client = mlflow.tracking.MlflowClient()
        versions = client.get_latest_versions(
            self.model_name, stages=[self.stage]
        )
        self.version = versions[0].version if versions else "unknown"
        logger.info(
            f"Loaded {self.model_name} v{self.version} ({self.stage})"
        )

    def predict(self, features: np.ndarray) -> np.ndarray:
        """Generate predictions using the loaded model."""
        return self.model.predict(features)

    def check_for_updates(self) -> bool:
        """Check if a newer model version is available."""
        client = mlflow.tracking.MlflowClient()
        versions = client.get_latest_versions(
            self.model_name, stages=[self.stage]
        )
        if versions and versions[0].version != self.version:
            logger.info(f"New version available: v{versions[0].version}")
            self.load_model()
            return True
        return False

model_manager = ModelManager("recommendation_model")
feature_service = MLFeatureService()
cache = PredictionCache(max_size=50000, ttl_seconds=600)
circuit = CircuitBreaker(
    failure_threshold=5,
    fallback_fn=lambda *a, **k: {"prediction": 0.5, "source": "fallback"},
)


class Request(BaseModel):
    user_id: int

@app.post("/recommend")
async def recommend(req: Request):
    """Production prediction endpoint with all resilience layers."""
    def _predict():
        features = feature_service.get_serving_features([req.user_id])
        X = np.array(features[req.user_id]).reshape(1, -1)
        cached = cache.get(X[0])
        if cached is not None:
            return {"prediction": cached, "source": "cache"}
        pred = model_manager.predict(X)[0]
        cache.put(X[0], float(pred))
        return {
            "prediction": float(pred),
            "source": "model",
            "model_version": model_manager.version,
        }

    return circuit.call(_predict)

async def health():
    return {
        "status": "healthy",
        "model_version": model_manager.version,
        "circuit_state": circuit.state.value,
        "cache_hit_rate": f"{cache.hit_rate:.1%}",
    }
Do Not Build Your Own Feature Store

It is tempting to build a custom feature store with Redis and some Python scripts. Do not do this. Feature stores solve subtle problems (point-in-time correctness, entity versioning, materialization scheduling) that take months to get right. Use an existing solution like Feast, Tecton, or Hopsworks and spend your engineering effort on model quality instead.

When to choose gRPC over REST for model serving

gRPC outperforms REST for model serving in three scenarios: (1) high-throughput internal microservices where HTTP/2 multiplexing and binary serialization reduce latency by 2-5x, (2) streaming predictions where the server pushes multiple results over a single connection, and (3) polyglot environments where auto-generated client stubs from .proto files eliminate serialization bugs. Use REST when you need browser compatibility, simple debugging with curl, or when latency differences do not matter.

Cost optimization for ML inference at scale

The three highest-impact cost optimizations are: (1) Right-sizing instances — most ML models are CPU-bound, not GPU-bound. Running a gradient boosted tree on a GPU instance wastes money. (2) Autoscaling — ML traffic is spiky. Scale down during off-peak hours. Use predictive autoscaling if your traffic patterns are regular. (3) Batch precomputation — for any entity that will be scored multiple times with the same features, precompute and cache the result. A well-tuned batch + cache system can reduce real-time inference volume by 80-90%.

Self-Check

Before moving to Chapter 10, honestly answer these questions.

Can you explain why a feature store prevents training-serving skew?

A feature store provides a single definition for each feature that is used identically in training and serving. Without it, the training pipeline might compute avg_spend_30d using one SQL query while the serving code computes it differently. The feature store ensures the same transformation logic produces both offline (historical) and online (real-time) features, eliminating this class of bugs.

When would you choose batch inference over real-time inference?

Batch inference is the right choice when: (1) predictions do not need to be fresh — hourly or daily updates are acceptable (e.g., daily churn scores), (2) the entity set is known in advance (you can precompute for all users), and (3) cost matters — batch processing is 10-100x cheaper per prediction than real-time serving. The tradeoff is freshness: batch predictions may be hours old.

What does a circuit breaker do in an ML serving pipeline?

A circuit breaker monitors the failure rate of a downstream service (like the model server). When failures exceed a threshold, it "trips open" and immediately returns a fallback response instead of waiting for timeouts. This prevents cascading failures where a slow model server causes all upstream services to queue requests and eventually fail. The circuit breaker periodically tests recovery by allowing a single request through ("half-open" state).

How does ONNX Runtime improve inference performance?

ONNX Runtime applies graph-level optimizations (operator fusion, constant folding, memory planning) that the original framework cannot perform at runtime. It also uses optimized BLAS/LAPACK kernels and can target specific hardware (AVX-512, NEON). For tree-based models, ONNX Runtime converts the decision tree traversal into vectorized operations. The result is typically 2-10x speedup over native PyTorch or scikit-learn inference.

Can you design a canary deployment for a new model version?

A canary deployment routes a small percentage of traffic (e.g., 5%) to the new model version while monitoring key metrics (latency, error rate, prediction distribution). If metrics remain healthy, gradually increase the percentage (5% to 10% to 25% to 50% to 100%). Use deterministic routing based on user ID so the same user always gets the same model version, which prevents inconsistent experiences. If any metric degrades, automatically roll back to 0% canary traffic.
