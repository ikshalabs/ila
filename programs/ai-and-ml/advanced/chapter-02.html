Transformers and Large Language Models

Understand the architecture that redefined NLP and is now conquering vision, audio, and robotics. From self-attention mechanics to fine-tuning billion-parameter models with LoRA, this chapter gives you the working knowledge to build on top of LLMs rather than just calling APIs.

Why This Matters

The Transformer architecture, introduced in the 2017 paper "Attention Is All You Need," replaced recurrent networks as the dominant sequence model almost overnight. Its key innovation — replacing sequential computation with parallel self-attention — enables training on sequences of unprecedented length and unlocks massive parallelism on modern GPUs.

Today, Transformers are not just an NLP architecture. They power vision models (ViT), protein folding (AlphaFold), code generation (Codex), and multimodal systems (GPT-4, Gemini). Understanding the Transformer deeply — not just at the API level, but at the weight-matrix level — is the single most important skill for a modern ML engineer.

Key Insight

Transformers replaced recurrence with attention, making every position in a sequence directly accessible to every other position. This removes the sequential bottleneck of RNNs/LSTMs and makes training embarrassingly parallel. The cost is quadratic memory in sequence length — a trade-off that drives much of current research (FlashAttention, ring attention, sparse attention).

Key Concepts

1. Self-Attention Mechanism

Self-attention computes relationships between all pairs of positions in a sequence. Each token generates three vectors: a query (what am I looking for?), a key (what do I represent?), and a value (what information do I carry?). The attention weight between two positions is the softmax-normalized dot product of the query of one position with the key of the other, scaled by the square root of the key dimension.

python 
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class SelfAttention(nn.Module):
    """Single-head self-attention with causal masking support."""

    def __init__(self, d_model, dropout=0.1):
        super().__init__()
        self.d_model = d_model
        self.qkv = nn.Linear(d_model, 3 * d_model)
        self.proj = nn.Linear(d_model, d_model)
        self.dropout = nn.Dropout(dropout)

    def forward(self, x, causal=False):
        B, T, C = x.shape

        # Compute Q, K, V in a single matrix multiply
        qkv = self.qkv(x).reshape(B, T, 3, C).permute(2, 0, 1, 3)
        q, k, v = qkv[0], qkv[1], qkv[2]

        # Scaled dot-product attention
        scores = torch.matmul(q, k.transpose(-2, -1)) / math.sqrt(C)

        # Causal mask: prevent attending to future tokens
        if causal:
            mask = torch.triu(torch.ones(T, T, device=x.device), diagonal=1).bool()
            scores = scores.masked_fill(mask, float('-inf'))

        attn = F.softmax(scores, dim=-1)
        attn = self.dropout(attn)

        out = torch.matmul(attn, v)
        return self.proj(out)

2. Transformer Architecture

The full Transformer consists of an encoder (bidirectional attention over input) and a decoder (causal attention over output, with cross-attention to encoder). Each layer contains multi-head self-attention followed by a position-wise feed-forward network, with layer normalization and residual connections wrapping both sub-layers. 

  ┌─────────────────────────────────────────────────┐
  │               Transformer Block                  │
  │                                                  │
  │  Input ──▶ LayerNorm ──▶ Multi-Head Attention    │
  │    │                          │                  │
  │    └────────── (+) ◀──────────┘  (residual)      │
  │                 │                                │
  │                 ▼                                │
  │         LayerNorm ──▶ Feed-Forward (MLP)         │
  │    │                          │                  │
  │    └────────── (+) ◀──────────┘  (residual)      │
  │                 │                                │
  │                 ▼                                │
  │              Output                              │
  └─────────────────────────────────────────────────┘

  Feed-Forward: Linear(d_model, 4*d_model) -> GELU -> Linear(4*d_model, d_model)
  Positional encoding added to input embeddings (sinusoidal or learned)
Figure 1: A single Transformer block with pre-norm architecture (used in GPT-2+)
python 
import torch
import torch.nn as nn

class TransformerBlock(nn.Module):
    """Pre-norm Transformer block (GPT-2 style)."""

    def __init__(self, d_model, n_heads, dropout=0.1):
        super().__init__()
        self.ln1 = nn.LayerNorm(d_model)
        self.attn = nn.MultiheadAttention(d_model, n_heads,
                                          dropout=dropout, batch_first=True)
        self.ln2 = nn.LayerNorm(d_model)
        self.mlp = nn.Sequential(
            nn.Linear(d_model, 4 * d_model),
            nn.GELU(),
            nn.Dropout(dropout),
            nn.Linear(4 * d_model, d_model),
            nn.Dropout(dropout),
        )

    def forward(self, x, attn_mask=None):
        # Self-attention with residual
        normed = self.ln1(x)
        attn_out, _ = self.attn(normed, normed, normed, attn_mask=attn_mask)
        x = x + attn_out

        # Feed-forward with residual
        x = x + self.mlp(self.ln2(x))
        return x


class GPTModel(nn.Module):
    """Minimal GPT-style decoder-only Transformer."""

    def __init__(self, vocab_size, d_model=768, n_heads=12,
                 n_layers=12, max_seq_len=1024, dropout=0.1):
        super().__init__()
        self.token_emb = nn.Embedding(vocab_size, d_model)
        self.pos_emb = nn.Embedding(max_seq_len, d_model)
        self.drop = nn.Dropout(dropout)

        self.blocks = nn.ModuleList([
            TransformerBlock(d_model, n_heads, dropout)
            for _ in range(n_layers)
        ])

        self.ln_f = nn.LayerNorm(d_model)
        self.head = nn.Linear(d_model, vocab_size, bias=False)

        # Weight tying: share embedding and output projection
        self.head.weight = self.token_emb.weight

    def forward(self, idx):
        B, T = idx.shape
        positions = torch.arange(T, device=idx.device).unsqueeze(0)

        x = self.drop(self.token_emb(idx) + self.pos_emb(positions))

        # Causal mask for autoregressive generation
        causal_mask = torch.triu(
            torch.ones(T, T, device=idx.device), diagonal=1
        ).bool()

        for block in self.blocks:
            x = block(x, attn_mask=causal_mask)

        x = self.ln_f(x)
        logits = self.head(x)
        return logits

3. BERT and Masked Language Modeling

BERT (Bidirectional Encoder Representations from Transformers) uses the encoder side of the Transformer. Its pre-training objective is Masked Language Modeling (MLM): randomly mask 15% of input tokens and train the model to predict them. Because attention is bidirectional, BERT can use both left and right context to fill in the blanks, producing rich contextual embeddings.

BERT also uses a Next Sentence Prediction (NSP) task during pre-training, though later work (RoBERTa) showed this was unnecessary. The key insight is that bidirectional context produces better representations for understanding tasks (classification, NER, QA) than unidirectional models, at the cost of not being able to generate text autoregressively.

python 
from transformers import BertTokenizer, BertForMaskedLM
import torch

# Load pre-trained BERT
tokenizer = BertTokenizer.from_pretrained('bert-base-uncased')
model = BertForMaskedLM.from_pretrained('bert-base-uncased')

# Masked language modeling example
text = "The capital of France is [MASK]."
inputs = tokenizer(text, return_tensors='pt')

with torch.no_grad():
    outputs = model(**inputs)
    predictions = outputs.logits

# Get the predicted token for [MASK]
mask_idx = (inputs['input_ids'] == tokenizer.mask_token_id).nonzero(as_tuple=True)[1]
predicted_id = predictions[0, mask_idx].argmax(dim=-1)
predicted_token = tokenizer.decode(predicted_id)
print(f"Predicted: {predicted_token}")  # "paris"


# Fine-tuning BERT for classification
from transformers import BertForSequenceClassification

classifier = BertForSequenceClassification.from_pretrained(
    'bert-base-uncased', num_labels=3
)

# The [CLS] token embedding is used as the sentence representation
# A classification head (linear layer) is added on top
inputs = tokenizer("This movie was excellent!", return_tensors='pt')
outputs = classifier(**inputs)
logits = outputs.logits  # shape: (1, 3) for 3 classes

4. GPT and Autoregressive Generation

GPT (Generative Pre-trained Transformer) uses the decoder side with causal masking: each token can only attend to tokens that came before it. This enables autoregressive text generation — predicting one token at a time, feeding each prediction back as input for the next step.

The pre-training objective is simply next-token prediction: given a sequence of tokens, predict the next one. This seemingly simple objective, applied to massive datasets with massive models, produces emergent capabilities including reasoning, coding, and following complex instructions.

python 
import torch

def generate(model, tokenizer, prompt, max_tokens=100, temperature=0.8, top_k=50):
    """Autoregressive text generation with top-k sampling."""
    model.eval()
    input_ids = tokenizer.encode(prompt, return_tensors='pt')

    for _ in range(max_tokens):
        with torch.no_grad():
            logits = model(input_ids).logits[:, -1, :]  # last position

        # Temperature scaling
        logits = logits / temperature

        # Top-k filtering
        top_k_logits, top_k_indices = torch.topk(logits, top_k)
        probs = torch.softmax(top_k_logits, dim=-1)

        # Sample from the filtered distribution
        next_token_idx = torch.multinomial(probs, num_samples=1)
        next_token = top_k_indices.gather(-1, next_token_idx)

        input_ids = torch.cat([input_ids, next_token], dim=-1)

        # Stop if we generate end-of-text token
        if next_token.item() == tokenizer.eos_token_id:
            break

    return tokenizer.decode(input_ids[0])
Temperature and Sampling

Temperature controls randomness: lower values (0.1–0.3) produce more deterministic outputs, while higher values (0.8–1.2) produce more creative but potentially incoherent text. Top-k limits sampling to the k most likely tokens, preventing the model from choosing extremely unlikely tokens. Top-p (nucleus sampling) adaptively selects the smallest set of tokens whose cumulative probability exceeds p.

5. Fine-Tuning Strategies (LoRA, QLoRA)

Fine-tuning a full LLM requires updating billions of parameters, which demands enormous GPU memory. Parameter-Efficient Fine-Tuning (PEFT) methods solve this by updating only a small subset of parameters while keeping the original weights frozen.

LoRA (Low-Rank Adaptation) injects trainable low-rank matrices into attention layers. Instead of updating the full weight matrix W, LoRA adds a decomposition BA where B and A are much smaller. QLoRA goes further by quantizing the frozen weights to 4-bit precision, reducing memory by 4x while maintaining fine-tuning quality.

python 
from peft import LoraConfig, get_peft_model, TaskType
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

# Load base model
model_name = "meta-llama/Llama-2-7b-hf"
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,
    device_map="auto",
)

# Configure LoRA
lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                  # rank of the low-rank matrices
    lora_alpha=32,          # scaling factor
    lora_dropout=0.05,      # dropout on LoRA layers
    target_modules=[        # which layers to adapt
        "q_proj", "v_proj",
        "k_proj", "o_proj",
    ],
)

# Apply LoRA — only 0.1% of parameters are trainable!
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 4,194,304 || all params: 6,742,609,920 || 0.06%

# QLoRA: load in 4-bit and fine-tune with LoRA
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",           # NormalFloat4 quantization
    bnb_4bit_compute_dtype=torch.float16,
    bnb_4bit_use_double_quant=True,    # nested quantization
)

model_4bit = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=bnb_config,
    device_map="auto",
)

6. Prompt Engineering

Prompt engineering is the art of crafting inputs that elicit the best possible outputs from LLMs. Unlike fine-tuning, it requires no parameter updates — you shape the model's behavior entirely through the text you provide. Effective prompt engineering combines clear instructions, relevant context, and structured output formatting.

  • Zero-shot — provide only instructions, no examples. Works for simple tasks with capable models.
  • Few-shot — include 2–5 input/output examples before your query. Dramatically improves consistency.
  • Chain-of-thought (CoT) — ask the model to reason step-by-step. Unlocks complex reasoning in large models.
  • System prompts — set the model's persona, constraints, and output format at the system level.
python 
import openai

# Few-shot prompt with chain-of-thought
messages = [
    {"role": "system", "content": """You are a data analyst.
When analyzing data, think step-by-step:
1. Identify the key variables
2. Note any patterns or anomalies
3. Draw a conclusion with confidence level"""},
    {"role": "user", "content": """Revenue: Q1=$10M, Q2=$12M, Q3=$9M, Q4=$15M.
Analyze the trend."""},
    {"role": "assistant", "content": """Step 1: Key variables — quarterly revenue over one year.
Step 2: Q1->Q2 grew 20%, Q2->Q3 dropped 25%, Q3->Q4 surged 67%.
Step 3: Not a linear trend — looks seasonal with a strong Q4.
Confidence: Medium (one year of data is insufficient for robust trend)."""},
    {"role": "user", "content": """Users: Jan=1000, Feb=1200, Mar=1500, Apr=2000, May=2800.
Analyze the trend."""},
]

# Structured output with JSON mode
response = openai.chat.completions.create(
    model="gpt-4",
    messages=messages,
    temperature=0.3,  # low temperature for analytical tasks
)

print(response.choices[0].message.content)
Pro Tip

The most common prompt engineering mistake is being vague. Instead of "summarize this text," say "summarize in 3 bullet points, each under 20 words, focusing on actionable insights for a product manager." Specificity is the single biggest lever for prompt quality.

How does positional encoding work in Transformers?

Unlike RNNs, Transformers have no built-in notion of sequence order. Positional encodings inject position information by adding a position-dependent vector to each token embedding. The original paper used sinusoidal functions; modern models (GPT-2+) typically learn position embeddings. RoPE (Rotary Position Embeddings) encodes position in the attention computation itself, enabling better extrapolation to longer sequences.

What is the KV cache and why does it matter for inference?

During autoregressive generation, each new token requires attending to all previous tokens. Without caching, you would recompute the key and value projections for the entire sequence at every step. The KV cache stores previously computed keys and values, so each generation step only computes K and V for the new token and appends them to the cache. This reduces generation from O(n^2) to O(n) per step, but consumes significant GPU memory for long sequences.

Self-Check

Before moving to Chapter 03, honestly answer these questions.

Can you explain why Transformers use scaled dot-product attention?

Scaling by sqrt(d_k) prevents the dot products from becoming large, which would push the softmax into regions with extremely small gradients. Without scaling, training becomes unstable as model dimension increases.

What is the difference between BERT and GPT at the architecture level?

BERT uses the Transformer encoder with bidirectional attention (every token sees every other token). GPT uses the Transformer decoder with causal masking (each token only sees previous tokens). This makes BERT better for understanding tasks and GPT better for generation tasks.

Can you explain how LoRA reduces the number of trainable parameters?

Instead of updating a full d x d weight matrix (d^2 parameters), LoRA decomposes the update into two matrices: A (d x r) and B (r x d), where r is much smaller than d. This reduces trainable parameters from d^2 to 2*d*r. With d=4096 and r=16, that is a 128x reduction.

When should you use fine-tuning vs. prompt engineering?

Use prompt engineering first — it is faster, cheaper, and requires no training data. Use fine-tuning when: (1) you need consistent formatting or style, (2) you have domain-specific knowledge not in the base model, (3) prompt engineering cannot achieve the required quality, or (4) you need to reduce per-query token costs with shorter prompts.
