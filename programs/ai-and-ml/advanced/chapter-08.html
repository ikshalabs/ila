Model Monitoring and Drift

Models degrade silently. Learn to detect data drift, concept drift, and prediction quality decay using statistical tests and automated pipelines. Build systems that catch model failures before your users do — and trigger retraining when the evidence demands it.

Estimated: 3–4 days
6 Core Concepts
3 Assignments

Why This Matters

Every ML model has an expiration date. The world changes, user behavior shifts, new products launch, and the data your model learned from becomes a fading snapshot of a reality that no longer exists. A fraud detection model trained on 2023 transaction patterns will miss 2025 fraud tactics. A recommendation engine trained on pre-pandemic shopping behavior will misfire in a post-pandemic world. This is not a hypothetical risk — it is a certainty.

The challenge is that model degradation is invisible to traditional monitoring. Your service stays up, latency stays low, error rates stay flat — but predictions quietly become less accurate, less calibrated, and less useful. The only way to catch this is to monitor the statistical properties of your model's inputs, outputs, and performance metrics over time. This chapter teaches you to build that monitoring layer.

Chapter 07 gave you the observability infrastructure: Prometheus metrics, structured logs, and distributed traces. This chapter builds on that foundation with drift-specific statistical tests, automated detection pipelines, and retraining triggers. Together, they form a closed-loop system that detects degradation and responds automatically.

Key Insight

There are two fundamentally different kinds of drift. Data drift (covariate shift) means the inputs change but the mapping from inputs to outputs stays the same — you can detect this without labels. Concept drift means the mapping itself changes — you can only detect this with ground truth labels. A complete monitoring system needs both detection mechanisms because they catch different failure modes.

Key Concepts

1. Data Drift Detection with PSI and KS Tests

Data drift occurs when the statistical properties of production input data diverge from the training data distribution. The Population Stability Index (PSI) quantifies this divergence by comparing bin proportions between a reference distribution and a production distribution. PSI below 0.1 indicates no significant drift, 0.1–0.2 suggests moderate drift worth investigating, and above 0.2 indicates significant drift that likely requires model retraining.

The Kolmogorov-Smirnov (KS) test is a non-parametric test that measures the maximum distance between two cumulative distribution functions. Unlike PSI, it does not require binning, making it more sensitive to localized distribution changes. The KS test produces a p-value: if it falls below your significance threshold (typically 0.05), you reject the null hypothesis that the two distributions are identical.

python 
import numpy as np
from scipy import stats
from typing import Dict, Tuple, List
from dataclasses import dataclass

@dataclass
class DriftResult:
    """Result of a drift detection test for a single feature."""
    feature_name: str
    psi_score: float
    ks_statistic: float
    ks_p_value: float
    is_drifted: bool
    drift_severity: str  # "none", "moderate", "severe"
    ref_mean: float
    prod_mean: float
    ref_std: float
    prod_std: float


def calculate_psi(
    reference: np.ndarray,
    production: np.ndarray,
    n_bins: int = 10,
    epsilon: float = 1e-6,
) -> float:
    """Calculate Population Stability Index.

    PSI = SUM( (P_i - Q_i) * ln(P_i / Q_i) )

    Where P_i = proportion in bin i for production data
          Q_i = proportion in bin i for reference data
    """
    # Create bins from reference distribution
    breakpoints = np.percentile(
        reference, np.linspace(0, 100, n_bins + 1)
    )
    breakpoints[0] = -np.inf
    breakpoints[-1] = np.inf

    # Calculate proportions in each bin
    ref_counts = np.histogram(reference, bins=breakpoints)[0]
    prod_counts = np.histogram(production, bins=breakpoints)[0]

    ref_pct = ref_counts / len(reference) + epsilon
    prod_pct = prod_counts / len(production) + epsilon

    # PSI formula
    psi = np.sum((prod_pct - ref_pct) * np.log(prod_pct / ref_pct))
    return float(psi)


def detect_drift(
    reference: np.ndarray,
    production: np.ndarray,
    feature_name: str,
    psi_threshold: float = 0.2,
    ks_alpha: float = 0.05,
) -> DriftResult:
    """Run both PSI and KS tests on a single feature."""
    psi = calculate_psi(reference, production)
    ks_stat, ks_p = stats.ks_2samp(reference, production)

    # Determine severity
    if psi < 0.1 and ks_p >= ks_alpha:
        severity = "none"
    elif psi < psi_threshold:
        severity = "moderate"
    else:
        severity = "severe"

    return DriftResult(
        feature_name=feature_name,
        psi_score=round(psi, 4),
        ks_statistic=round(ks_stat, 4),
        ks_p_value=round(ks_p, 6),
        is_drifted=(psi >= psi_threshold or ks_p < ks_alpha),
        drift_severity=severity,
        ref_mean=round(np.mean(reference), 4),
        prod_mean=round(np.mean(production), 4),
        ref_std=round(np.std(reference), 4),
        prod_std=round(np.std(production), 4),
    )


# Example: detect drift across all features
reference_data = {
    "age": np.random.normal(35, 10, 10000),
    "income": np.random.lognormal(10.5, 0.8, 10000),
    "session_count": np.random.poisson(5, 10000).astype(float),
}

# Simulate drift: income distribution shifts
production_data = {
    "age": np.random.normal(35, 10, 5000),
    "income": np.random.lognormal(11.0, 0.9, 5000),  # DRIFTED
    "session_count": np.random.poisson(5, 5000).astype(float),
}

for feature in reference_data:
    result = detect_drift(
        reference_data[feature],
        production_data[feature],
        feature
    )
    print(f"{result.feature_name}: PSI={result.psi_score}, "
          f"KS p={result.ks_p_value}, drift={result.drift_severity}")

2. Concept Drift and Label Delay

Concept drift is harder to detect because it requires ground truth labels. In many production systems, labels arrive with significant delay: fraud labels come when transactions are reviewed (days), ad conversion labels come when users complete purchases (hours to weeks), and medical diagnosis labels come after lab results (days to months). This label delay creates a blind spot where your model might be failing but you cannot confirm it yet.

The ADWIN (Adaptive Windowing) algorithm is a streaming approach to concept drift detection. It maintains a variable-length window of recent observations and detects when the mean of a subwindow changes significantly compared to the rest. When drift is detected, old data is discarded, adapting the window to the new distribution. For ML systems, feed ADWIN a stream of binary correctness indicators (was the prediction correct?) and it will alert when accuracy shifts.

python 
import numpy as np
from collections import deque
from typing import Optional

class ADWINDriftDetector:
    """Adaptive Windowing method for concept drift detection.

    Maintains a variable-length window. When drift is detected,
    the old portion is dropped. Feed it a stream of 0/1 values
    representing prediction correctness.
    """

    def __init__(self, delta: float = 0.002):
        self.delta = delta
        self.window: List[float] = []
        self.drift_detected = False
        self.drift_count = 0

    def add_element(self, value: float) -> bool:
        """Add observation. Returns True if concept drift detected."""
        self.window.append(value)
        self.drift_detected = False

        if len(self.window) < 30:
            return False

        # Check for drift by comparing sub-windows
        for split in range(10, len(self.window) - 10):
            left = self.window[:split]
            right = self.window[split:]

            mean_diff = abs(np.mean(left) - np.mean(right))
            n_l, n_r = len(left), len(right)

            # Hoeffding bound for the confidence interval
            epsilon = np.sqrt(
                np.log(2.0 / self.delta) /
                (2.0 * ((1.0 / n_l) + (1.0 / n_r)))
            )

            if mean_diff >= epsilon:
                # Drop old data before the split point
                self.window = self.window[split:]
                self.drift_detected = True
                self.drift_count += 1
                return True

        return False

    @property
    def current_mean(self) -> Optional[float]:
        if self.window:
            return np.mean(self.window)
        return None


# Simulate concept drift in a classification task
np.random.seed(42)
detector = ADWINDriftDetector(delta=0.01)

# Phase 1: Model is accurate (90% correct)
for _ in range(500):
    correct = np.random.binomial(1, 0.9)
    detector.add_element(correct)

# Phase 2: Concept drift (accuracy drops to 60%)
for i in range(300):
    correct = np.random.binomial(1, 0.6)
    if detector.add_element(correct):
        print(f"DRIFT at step {500 + i}: "
              f"accuracy = {detector.current_mean:.3f}")
        break
Label Delay Trap

Never wait for labels to start monitoring. Use proxy signals that are available immediately: prediction confidence distributions, output class balance, feature distributions, and user engagement metrics. A sudden drop in average confidence or a shift in predicted class ratios often signals concept drift days before labels confirm it.

3. Automated Drift Reports with Evidently

Evidently is an open-source library that generates comprehensive data and model quality reports. It computes dozens of statistical tests across all features simultaneously, produces visual reports, and integrates with monitoring pipelines. For production use, Evidently can run as a batch job that compares each day's data against the reference distribution and exports results to Prometheus or a database.

python 
import pandas as pd
from evidently.report import Report
from evidently.metric_preset import DataDriftPreset
from evidently.metric_preset import TargetDriftPreset
from evidently.metrics import (
    DataDriftTable,
    DatasetDriftMetric,
    ColumnDriftMetric,
)
from evidently.test_suite import TestSuite
from evidently.tests import (
    TestShareOfDriftedColumns,
    TestColumnDrift,
)


def run_drift_report(
    reference_df: pd.DataFrame,
    production_df: pd.DataFrame,
    output_path: str = "drift_report.html",
) -> dict:
    """Generate comprehensive drift report."""

    # Create a drift report with multiple metrics
    report = Report(metrics=[
        DatasetDriftMetric(),
        DataDriftTable(),
    ])

    report.run(
        reference_data=reference_df,
        current_data=production_df,
    )

    # Save HTML report for visual inspection
    report.save_html(output_path)

    # Extract results programmatically
    results = report.as_dict()
    return results


def run_drift_tests(
    reference_df: pd.DataFrame,
    production_df: pd.DataFrame,
    critical_features: list,
    max_drift_share: float = 0.3,
) -> bool:
    """Run automated drift tests. Returns True if all pass."""

    tests = [
        # No more than 30% of columns should be drifted
        TestShareOfDriftedColumns(lte=max_drift_share),
    ]

    # Add specific tests for critical features
    for feature in critical_features:
        tests.append(TestColumnDrift(column_name=feature))

    suite = TestSuite(tests=tests)
    suite.run(
        reference_data=reference_df,
        current_data=production_df,
    )

    # Check if all tests passed
    result = suite.as_dict()
    all_passed = all(
        test["status"] == "SUCCESS"
        for test in result["tests"]
    )
    return all_passed


# Usage in a daily monitoring pipeline
reference = pd.read_parquet("data/reference_features.parquet")
today = pd.read_parquet("data/today_features.parquet")

all_ok = run_drift_tests(
    reference, today,
    critical_features=["income", "age", "transaction_count"],
)

if not all_ok:
    print("ALERT: Drift detected! Generating detailed report...")
    run_drift_report(reference, today, "reports/drift_report.html")

4. Prediction Quality Monitoring

Beyond input drift, you must monitor the quality of your model's outputs. This involves tracking proxy metrics (available immediately) and ground-truth metrics (available after label delay). Proxy metrics include: mean prediction confidence, confidence calibration (does a 90% confidence prediction actually have 90% accuracy?), prediction class distribution, and output value distribution for regression models.

python 
import numpy as np
from sklearn.calibration import calibration_curve
from sklearn.metrics import (
    accuracy_score, precision_score, recall_score,
    f1_score, roc_auc_score, log_loss,
)
from collections import deque
from dataclasses import dataclass, field
from typing import Dict, List


@dataclass
class QualitySnapshot:
    """A snapshot of model quality metrics at a point in time."""
    timestamp: float
    sample_count: int
    accuracy: float
    precision: float
    recall: float
    f1: float
    auc_roc: float
    log_loss_val: float
    calibration_error: float


class PredictionQualityMonitor:
    """Track model prediction quality over sliding windows."""

    def __init__(self, window_size: int = 5000):
        self.predictions = deque(maxlen=window_size)
        self.labels = deque(maxlen=window_size)
        self.probabilities = deque(maxlen=window_size)
        self.history: List[QualitySnapshot] = []

    def add_labeled_prediction(
        self, prediction: int, probability: float, label: int
    ):
        """Record a prediction that now has a ground truth label."""
        self.predictions.append(prediction)
        self.probabilities.append(probability)
        self.labels.append(label)

    def compute_quality(self) -> QualitySnapshot:
        """Compute all quality metrics over the current window."""
        import time

        preds = np.array(self.predictions)
        probs = np.array(self.probabilities)
        labels = np.array(self.labels)

        # Compute calibration error
        prob_true, prob_pred = calibration_curve(
            labels, probs, n_bins=10, strategy="uniform"
        )
        ece = np.mean(np.abs(prob_true - prob_pred))

        snapshot = QualitySnapshot(
            timestamp=time.time(),
            sample_count=len(preds),
            accuracy=accuracy_score(labels, preds),
            precision=precision_score(labels, preds, zero_division=0),
            recall=recall_score(labels, preds, zero_division=0),
            f1=f1_score(labels, preds, zero_division=0),
            auc_roc=roc_auc_score(labels, probs),
            log_loss_val=log_loss(labels, probs),
            calibration_error=ece,
        )
        self.history.append(snapshot)
        return snapshot

    def detect_degradation(
        self, baseline_accuracy: float, threshold: float = 0.05
    ) -> bool:
        """Check if current accuracy dropped below baseline by threshold."""
        if len(self.predictions) < 100:
            return False

        current = accuracy_score(
            np.array(self.labels), np.array(self.predictions)
        )
        return (baseline_accuracy - current) > threshold

5. Training-Serving Skew Detection

Training-serving skew occurs when the features used at training time differ from those used at inference time. This can happen due to bugs in feature engineering pipelines, differences between batch and streaming feature computation, stale feature store entries, or timezone handling differences. Skew is particularly insidious because the model appears to be working correctly in offline evaluation but performs poorly in production.

To detect skew, log the feature vectors used at inference time and periodically compare their distributions against the training data. The comparison should happen at the transformed feature level (after preprocessing), not the raw data level, because skew often occurs in the transformation pipeline itself.

python 
import numpy as np
import pandas as pd
from typing import Dict, List

class TrainingServingSkewDetector:
    """Detect differences between training and serving features."""

    def __init__(self, training_stats: Dict[str, Dict]):
        """
        training_stats: dict of feature_name -> {
            'mean': float, 'std': float,
            'min': float, 'max': float,
            'null_rate': float, 'dtype': str
        }
        """
        self.training_stats = training_stats

    def check_skew(self, serving_features: pd.DataFrame) -> Dict:
        """Compare serving features against training statistics."""
        skew_report = {}

        for col, train_stats in self.training_stats.items():
            if col not in serving_features.columns:
                skew_report[col] = {"issue": "MISSING_FEATURE"}
                continue

            serving_col = serving_features[col]
            issues = []

            # Check dtype mismatch
            if str(serving_col.dtype) != train_stats["dtype"]:
                issues.append(f"dtype: train={train_stats['dtype']}, "
                              f"serve={serving_col.dtype}")

            # Check null rate increase
            serve_null_rate = serving_col.isnull().mean()
            if serve_null_rate > train_stats["null_rate"] * 2 + 0.01:
                issues.append(
                    f"null_rate: train={train_stats['null_rate']:.3f}, "
                    f"serve={serve_null_rate:.3f}")

            # Check distribution shift (z-score of mean difference)
            if train_stats["std"] > 0:
                z_score = abs(
                    serving_col.mean() - train_stats["mean"]
                ) / train_stats["std"]
                if z_score > 3:
                    issues.append(
                        f"mean_shift: z={z_score:.2f} "
                        f"(train={train_stats['mean']:.4f}, "
                        f"serve={serving_col.mean():.4f})")

            # Check for out-of-range values
            oor_count = (
                (serving_col < train_stats["min"]) |
                (serving_col > train_stats["max"])
            ).sum()
            if oor_count > 0:
                issues.append(
                    f"out_of_range: {oor_count} values")

            skew_report[col] = {
                "issues": issues,
                "has_skew": len(issues) > 0,
            }

        return skew_report
Pro Tip

Log feature schemas alongside feature values. Store the expected data types, valid ranges, and allowed categories for each feature. Then validate every serving request against this schema. This catches a large class of training-serving skew bugs (type mismatches, missing features, new categories) immediately at inference time rather than waiting for drift detection.

6. Automated Retraining Pipelines

When drift is confirmed, the natural response is to retrain the model on fresh data. But automated retraining is surprisingly tricky to get right. You need to answer several questions: How much new data is enough? Should you retrain from scratch or fine-tune? How do you validate the new model before deploying it? What happens if the new model is worse than the old one?

The safest pattern is a validation gate: after retraining, evaluate the new model against a held-out set that represents the current distribution. Only promote the new model if it improves on the production model by a minimum margin (e.g., 0.5% accuracy improvement). This prevents deploying models that regressed due to data quality issues in the retraining data.

python 
import logging
from dataclasses import dataclass
from typing import Callable, Dict, Optional
from datetime import datetime

logger = logging.getLogger(__name__)

@dataclass
class RetrainingConfig:
    model_name: str
    psi_threshold: float = 0.2        # Trigger on feature drift
    accuracy_drop: float = 0.05       # Trigger on quality drop
    min_improvement: float = 0.005    # Min required improvement
    min_samples: int = 5000           # Min labeled data for retrain
    cooldown_hours: int = 24          # Min time between retrains


class RetrainingOrchestrator:
    """Orchestrates the detect-retrain-validate-deploy cycle."""

    def __init__(
        self,
        config: RetrainingConfig,
        train_fn: Callable,
        evaluate_fn: Callable,
        deploy_fn: Callable,
    ):
        self.config = config
        self.train_fn = train_fn
        self.evaluate_fn = evaluate_fn
        self.deploy_fn = deploy_fn
        self.last_retrain: Optional[datetime] = None
        self.retrain_log = []

    def should_retrain(self, drift_metrics: Dict, quality_metrics: Dict) -> bool:
        """Decide whether to trigger retraining."""

        # Check cooldown
        if self.last_retrain:
            hours_since = (
                datetime.now() - self.last_retrain
            ).total_seconds() / 3600
            if hours_since < self.config.cooldown_hours:
                return False

        # Check drift triggers
        max_psi = drift_metrics.get("max_psi", 0)
        drifted = max_psi > self.config.psi_threshold

        # Check quality triggers
        accuracy = quality_metrics.get("accuracy")
        baseline = quality_metrics.get("baseline_accuracy")
        degraded = False
        if accuracy and baseline:
            degraded = (baseline - accuracy) > self.config.accuracy_drop

        # Check data sufficiency
        has_data = quality_metrics.get(
            "labeled_count", 0
        ) >= self.config.min_samples

        return (drifted or degraded) and has_data

    def execute(self, training_data, current_metrics: Dict) -> bool:
        """Run the full retrain-validate-deploy cycle."""
        logger.info("Starting retraining for %s", self.config.model_name)

        # Step 1: Train new model
        new_model = self.train_fn(training_data)

        # Step 2: Evaluate on held-out validation set
        new_metrics = self.evaluate_fn(new_model)
        current_acc = current_metrics.get("accuracy", 0)
        new_acc = new_metrics.get("accuracy", 0)
        improvement = new_acc - current_acc

        # Step 3: Validation gate
        if improvement < self.config.min_improvement:
            logger.warning(
                "REJECTED: new=%.4f, old=%.4f, improvement=%.4f < %.4f",
                new_acc, current_acc, improvement,
                self.config.min_improvement,
            )
            self.retrain_log.append({
                "status": "rejected",
                "reason": "insufficient_improvement",
                "improvement": improvement,
            })
            return False

        # Step 4: Deploy
        logger.info(
            "APPROVED: %.4f -> %.4f (+%.4f). Deploying.",
            current_acc, new_acc, improvement,
        )
        self.deploy_fn(new_model)
        self.last_retrain = datetime.now()
        self.retrain_log.append({
            "status": "deployed",
            "improvement": improvement,
            "new_accuracy": new_acc,
        })
        return True

How do you choose between scheduled and triggered retraining?

Scheduled retraining (e.g., weekly) is simpler and predictable. Use it when your domain changes gradually and you have a steady stream of labeled data. Triggered retraining (on drift detection) is more responsive but requires reliable drift detection. Use it when your domain changes unpredictably. In practice, most teams use both: a scheduled cadence as a baseline with drift-triggered retraining for unexpected shifts.

What sample size do you need for reliable drift detection?

For PSI, you need at least 200–500 samples per bin, so with 10 bins that means 2,000–5,000 production samples minimum. For the KS test, sensitivity increases with sample size — with very large samples (100K+), the test becomes overly sensitive and will flag trivial differences as significant. In that case, use an effect-size threshold alongside the p-value, or use PSI which is naturally calibrated to practical significance.

Can you explain the difference between data drift and concept drift?

Data drift (covariate shift) means inputs change but the input-output relationship stays the same. Concept drift means the input-output relationship changes. Data drift is detectable without labels (PSI, KS tests on inputs). Concept drift requires ground truth labels to confirm.

Can you implement PSI from scratch and interpret the results?

PSI compares bin proportions between two distributions using the formula SUM((P_i - Q_i) * ln(P_i / Q_i)). Thresholds: below 0.1 is stable, 0.1–0.2 is moderate drift, above 0.2 is significant drift. You should be able to write this from memory and explain why the epsilon correction prevents division by zero.

Do you understand how to handle label delay in monitoring?

Use proxy metrics (confidence scores, class distributions) as early warning signals. When labels arrive, compute ground-truth metrics on sliding windows. Use streaming detectors like ADWIN for real-time concept drift detection. Never wait for labels to start monitoring — data drift detection is your first line of defense.

Can you design a safe automated retraining pipeline with a validation gate?

Your pipeline should: (1) detect drift or quality degradation, (2) trigger retraining with a cooldown period, (3) evaluate the new model on held-out data, (4) compare against production model, (5) deploy only if improvement exceeds a minimum threshold, (6) log all decisions for audit. If any of these steps are unclear, review the RetrainingOrchestrator code.

Can you detect training-serving skew in a feature pipeline?

Log feature vectors at inference time and compare their statistics (mean, std, null rate, dtype, value range) against the training data. Flag dtype mismatches, increased null rates, mean shifts beyond 3 standard deviations, and out-of-range values. Use feature schemas to catch issues at request time rather than waiting for drift detection.
