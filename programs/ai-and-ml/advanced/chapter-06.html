Model Serving and Scaling

Deploy trained models into production with TorchServe, TensorFlow Serving, and NVIDIA Triton. Implement A/B testing and canary deployment strategies to safely roll out model updates at scale.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

A model that lives only in a Jupyter notebook delivers zero business value. The gap between model.predict(x) in a notebook and a production endpoint serving thousands of requests per second is enormous — and it is where most ML projects fail. Model serving is the bridge between research and revenue.

Production serving demands more than wrapping a model in a Flask app. You need batching to maximize GPU utilization, versioning to roll back broken models, health checks to route traffic away from unhealthy replicas, and deployment strategies that let you test new models on real traffic without risking the entire user base.

Key Insight

The best model serving architecture is one that lets you deploy a new model version in under 5 minutes and roll it back in under 30 seconds. Speed of iteration — not raw inference throughput — is the primary competitive advantage in production ML.

Key Concepts

1. Model Serving Architectures

There are three primary patterns for serving ML models in production. Each makes different tradeoffs between latency, throughput, and operational complexity:

  • Synchronous REST/gRPC — The client sends a request and waits for a prediction. Simple to implement, but latency-sensitive. Used for real-time recommendations, fraud detection, and chatbots.
  • Asynchronous batch — Predictions are queued and processed in bulk. Higher throughput, but higher latency. Used for email classification, content moderation queues, and offline scoring.
  • Streaming — Models consume from event streams (Kafka, Kinesis) and emit predictions in near-real-time. Used for IoT sensor processing, financial tick data, and real-time anomaly detection.

The choice of architecture depends on your latency requirements. For P99 latency under 50ms, you need synchronous serving with GPU batching. For throughput over 10,000 predictions per second, you need horizontal scaling with a load balancer that understands model health.

# serving_config.py — Abstract serving configuration
from dataclasses import dataclass
from enum import Enum
from typing import Optional

class ServingMode(Enum):
    SYNC = "synchronous"
    ASYNC = "asynchronous"
    STREAMING = "streaming"

@dataclass
class ServingConfig:
    model_name: str
    model_version: str
    mode: ServingMode = ServingMode.SYNC
    max_batch_size: int = 32
    batch_timeout_ms: int = 100
    max_concurrent_requests: int = 128
    gpu_memory_fraction: float = 0.8
    health_check_interval_s: int = 10
    warmup_requests: int = 5

    def validate(self):
        assert self.max_batch_size > 0, "Batch size must be positive"
        assert 0 < self.gpu_memory_fraction <= 1.0
        return self

2. TorchServe Setup and Configuration

TorchServe is the official model serving framework for PyTorch. It handles model loading, request batching, worker management, and provides REST and gRPC endpoints out of the box. The key abstraction is the handler — a Python class that defines how to preprocess inputs, run inference, and postprocess outputs.

python
# handler.py — Custom TorchServe handler for a text classifier
import json
import logging
import torch
from ts.torch_handler.base_handler import BaseHandler
from transformers import AutoTokenizer, AutoModelForSequenceClassification

logger = logging.getLogger(__name__)

class TextClassifierHandler(BaseHandler):
    def initialize(self, context):
        """Load model and tokenizer from the model archive."""
        self.manifest = context.manifest
        properties = context.system_properties
        model_dir = properties.get("model_dir")

        # Load the tokenizer and model
        self.tokenizer = AutoTokenizer.from_pretrained(model_dir)
        self.model = AutoModelForSequenceClassification.from_pretrained(
            model_dir
        )
        self.model.eval()

        # Move to GPU if available
        self.device = torch.device(
            "cuda" if torch.cuda.is_available() else "cpu"
        )
        self.model.to(self.device)
        self.label_map = {0: "negative", 1: "positive"}
        logger.info("Model loaded on %s", self.device)

    def preprocess(self, requests):
        """Tokenize incoming text requests into model inputs."""
        texts = []
        for request in requests:
            data = request.get("data") or request.get("body")
            if isinstance(data, (bytes, bytearray)):
                data = data.decode("utf-8")
            if isinstance(data, str):
                data = json.loads(data)
            texts.append(data["text"])

        inputs = self.tokenizer(
            texts, padding=True, truncation=True,
            max_length=512, return_tensors="pt",
        )
        return {k: v.to(self.device) for k, v in inputs.items()}

    def inference(self, inputs):
        """Run the model forward pass."""
        with torch.no_grad():
            outputs = self.model(**inputs)
        return outputs.logits

    def postprocess(self, logits):
        """Convert logits to human-readable predictions."""
        probabilities = torch.softmax(logits, dim=-1)
        predictions = []
        for prob in probabilities:
            label_idx = prob.argmax().item()
            predictions.append({
                "label": self.label_map[label_idx],
                "confidence": round(prob[label_idx].item(), 4),
            })
        return predictions

# package_model.py — Create a TorchServe model archive (.mar)
import subprocess
import os

def create_model_archive(
    model_name: str,
    model_dir: str,
    handler_path: str,
    version: str = "1.0",
    export_path: str = "./model_store",
):
    """Package model artifacts into a .mar archive."""
    os.makedirs(export_path, exist_ok=True)

    cmd = [
        "torch-model-archiver",
        "--model-name", model_name,
        "--version", version,
        "--serialized-file", f"{model_dir}/pytorch_model.bin",
        "--handler", handler_path,
        "--extra-files", (
            f"{model_dir}/config.json,"
            f"{model_dir}/tokenizer_config.json,"
            f"{model_dir}/vocab.txt"
        ),
        "--export-path", export_path,
        "--force",
    ]
    subprocess.run(cmd, check=True)
    print(f"Archive created: {export_path}/{model_name}.mar")

# Package the model
create_model_archive(
    model_name="text_classifier",
    model_dir="./trained_model",
    handler_path="./handler.py",
    version="2.0",
)
Batching Is Critical

Without dynamic batching, a GPU serving model will use only 5–10% of its capacity. TorchServe's batch_size and max_batch_delay in config.properties can increase throughput by 10–20x. Always measure P50 and P99 latency to find the sweet spot between throughput and tail latency.

3. TensorFlow Serving

TensorFlow Serving is a high-performance serving system designed for production environments. It uses the SavedModel format and provides native support for model versioning, automatic model loading, and gRPC/REST APIs. The key advantage is its battle-tested gRPC implementation with sub-millisecond overhead.

python
# export_savedmodel.py — Export a Keras model for TF Serving
import tensorflow as tf
import os
import time

def export_for_serving(
    model: tf.keras.Model,
    export_dir: str,
    version: int = None,
):
    """Export a Keras model in SavedModel format for TF Serving."""
    if version is None:
        version = int(time.time())

    versioned_path = os.path.join(export_dir, str(version))

    # Define serving signatures with explicit shapes
    @tf.function(input_signature=[
        tf.TensorSpec(shape=[None, 224, 224, 3], dtype=tf.float32)
    ])
    def serve(images):
        predictions = model(images, training=False)
        return {"predictions": predictions}

    tf.saved_model.save(
        model, versioned_path,
        signatures={"serving_default": serve},
    )
    print(f"Model exported to {versioned_path}")
    return versioned_path


# Build and export
model = tf.keras.applications.ResNet50(weights="imagenet")
export_for_serving(model, "./models/image_classifier", version=1)
python
# tf_serving_client.py — gRPC client for TF Serving
import grpc
import numpy as np
import tensorflow as tf
from tensorflow_serving.apis import (
    predict_pb2,
    prediction_service_pb2_grpc,
)

def predict_grpc(
    image: np.ndarray,
    model_name: str = "image_classifier",
    host: str = "localhost",
    port: int = 8500,
    version: int = None,
):
    """Send a prediction request via gRPC to TF Serving."""
    channel = grpc.insecure_channel(f"{host}:{port}")
    stub = prediction_service_pb2_grpc.PredictionServiceStub(channel)

    request = predict_pb2.PredictRequest()
    request.model_spec.name = model_name
    if version:
        request.model_spec.version.value = version

    request.inputs["images"].CopyFrom(
        tf.make_tensor_proto(image, dtype=tf.float32)
    )

    response = stub.Predict(request, timeout=10.0)
    predictions = tf.make_ndarray(response.outputs["predictions"])
    return predictions

# Usage
image = np.random.rand(1, 224, 224, 3).astype(np.float32)
result = predict_grpc(image)
print(f"Top prediction: class {np.argmax(result)}")

4. NVIDIA Triton Inference Server

NVIDIA Triton is the most versatile inference server available. It supports TensorFlow, PyTorch, ONNX, TensorRT, and custom Python backends — all in a single server. Its killer feature is concurrent model execution: Triton can run multiple models on a single GPU simultaneously, and it supports dynamic batching, model ensembles, and model pipelines natively.

python
# triton_config.py — Generate Triton model repository config
import os

def create_triton_config(
    model_name: str,
    platform: str,
    input_dims: list,
    output_dims: list,
    max_batch_size: int = 32,
    instance_count: int = 1,
    repo_path: str = "./model_repository",
):
    """Create Triton model repository structure and config."""
    model_dir = os.path.join(repo_path, model_name)
    version_dir = os.path.join(model_dir, "1")
    os.makedirs(version_dir, exist_ok=True)

    config = f"""name: "{model_name}"
platform: "{platform}"
max_batch_size: {max_batch_size}

input [
  {{
    name: "input"
    data_type: TYPE_FP32
    dims: {input_dims}
  }}
]
output [
  {{
    name: "output"
    data_type: TYPE_FP32
    dims: {output_dims}
  }}
]
instance_group [
  {{
    count: {instance_count}
    kind: KIND_GPU
  }}
]
dynamic_batching {{
  preferred_batch_size: [ 8, 16, 32 ]
  max_queue_delay_microseconds: 100000
}}"""

    config_path = os.path.join(model_dir, "config.pbtxt")
    with open(config_path, "w") as f:
        f.write(config)
    print(f"Triton config: {config_path}")
    return config_path

create_triton_config(

# triton_client.py — Python client for Triton Inference Server
import numpy as np
import tritonclient.http as httpclient
from tritonclient.utils import InferenceServerException

class TritonModelClient:
    def __init__(self, url: str = "localhost:8000"):
        self.client = httpclient.InferenceServerClient(url=url)

    def check_health(self) -> bool:
        """Verify the server is live and ready."""
        return (
            self.client.is_server_live()
            and self.client.is_server_ready()
        )

    def predict(
        self,
        model_name: str,
        input_data: np.ndarray,
        input_name: str = "input",
        output_name: str = "output",
    ) -> np.ndarray:
        """Send inference request to Triton."""
        inputs = [
            httpclient.InferInput(input_name, input_data.shape, "FP32")
        ]
        inputs[0].set_data_from_numpy(input_data)
        outputs = [httpclient.InferRequestedOutput(output_name)]

        try:
            result = self.client.infer(
                model_name=model_name,
                inputs=inputs,
                outputs=outputs,
            )
            return result.as_numpy(output_name)
        except InferenceServerException as e:
            print(f"Inference failed: {e}")
            raise

    def get_model_metadata(self, model_name: str) -> dict:
        """Get model metadata including input/output shapes."""
        metadata = self.client.get_model_metadata(model_name)
        return {
            "name": metadata["name"],
            "versions": metadata["versions"],
            "inputs": metadata["inputs"],
            "outputs": metadata["outputs"],
        }

# Usage
client = TritonModelClient()
if client.check_health():
    image = np.random.rand(1, 3, 224, 224).astype(np.float32)
    preds = client.predict("resnet50", image)
    print(f"Top class: {np.argmax(preds)}")
Pro Tip

When choosing between TorchServe, TF Serving, and Triton: if your team uses a single framework, use its native server. If you have models across multiple frameworks, Triton is the clear winner — it can serve PyTorch, TensorFlow, ONNX, and TensorRT models from a single deployment.

5. A/B Testing for Models

A/B testing for ML models is fundamentally different from A/B testing for UI changes. Model A/B tests must account for prediction quality, latency impact, business metric movement, and the statistical challenges of measuring small improvements. The key principle: always test with a holdout group that receives the current production model.

python
# ab_testing.py — A/B testing framework for model serving
import hashlib
import time
from dataclasses import dataclass, field
from typing import Dict, Any
import numpy as np
from scipy import stats

@dataclass
class ModelVariant:
    name: str
    model_version: str
    traffic_weight: float  # 0.0 to 1.0
    predictions: list = field(default_factory=list)
    latencies_ms: list = field(default_factory=list)
    outcomes: list = field(default_factory=list)

class ABTestRouter:
    """Route requests to model variants based on user ID hash."""

    def __init__(self, experiment_name: str):
        self.experiment_name = experiment_name
        self.variants: Dict[str, ModelVariant] = {}

    def add_variant(self, variant: ModelVariant):
        self.variants[variant.name] = variant

    def route(self, user_id: str) -> ModelVariant:
        """Deterministic routing based on user ID hash."""
        hash_val = hashlib.sha256(
            f"{self.experiment_name}:{user_id}".encode()
        ).hexdigest()
        bucket = int(hash_val[:8], 16) / 0xFFFFFFFF

        cumulative = 0.0
        for variant in self.variants.values():
            cumulative += variant.traffic_weight
            if bucket <= cumulative:
                return variant
        return list(self.variants.values())[-1]

    def record_outcome(self, variant_name: str, outcome: float, latency_ms: float):
        variant = self.variants[variant_name]
        variant.outcomes.append(outcome)
        variant.latencies_ms.append(latency_ms)

    def analyze(self) -> Dict[str, Any]:
        """Run statistical analysis on the experiment."""
        names = list(self.variants.keys())
        control = self.variants[names[0]]
        treatment = self.variants[names[1]]

        t_stat, p_value = stats.ttest_ind(
            control.outcomes, treatment.outcomes
        )

        return {
            "control_mean": np.mean(control.outcomes),
            "treatment_mean": np.mean(treatment.outcomes),
            "lift": (np.mean(treatment.outcomes) - np.mean(control.outcomes))
                / np.mean(control.outcomes),
            "p_value": p_value,
            "significant": p_value < 0.05,
            "control_p99_latency": np.percentile(control.latencies_ms, 99),
            "treatment_p99_latency": np.percentile(treatment.latencies_ms, 99),
        }

# Setup experiment
router = ABTestRouter("recommender_v2_test")
router.add_variant(ModelVariant("control", "v1.0", 0.8))
router.add_variant(ModelVariant("treatment", "v2.0", 0.2))

6. Canary Deployments and Rollback

Canary deployments expose a new model version to a small fraction of traffic before full rollout. Unlike A/B tests (which run to statistical significance), canary deployments are safety mechanisms — they catch catastrophic failures like crashes, latency spikes, or wildly wrong predictions before they affect all users.

python
# canary_deployer.py — Canary deployment with automatic rollback
import time
import logging
from dataclasses import dataclass
from typing import Callable
import numpy as np

logger = logging.getLogger(__name__)

@dataclass
class CanaryConfig:
    model_name: str
    new_version: str
    old_version: str
    initial_traffic_pct: float = 5.0
    increment_pct: float = 10.0
    max_traffic_pct: float = 100.0
    observation_window_s: int = 300
    error_rate_threshold: float = 0.01
    latency_p99_threshold_ms: float = 200.0

class CanaryDeployer:
    """Progressive canary deployment with guardrails."""

    def __init__(self, config: CanaryConfig, metrics_fn: Callable, set_traffic_fn: Callable):
        self.config = config
        self.get_metrics = metrics_fn
        self.set_traffic = set_traffic_fn
        self.current_pct = 0.0
        self.rolled_back = False

    def check_guardrails(self) -> bool:
        """Check if canary metrics are within thresholds."""
        metrics = self.get_metrics(self.config.model_name, self.config.new_version)
        error_rate = metrics.get("error_rate", 0)
        p99_latency = metrics.get("p99_latency_ms", 0)

        if error_rate > self.config.error_rate_threshold:
            logger.error("Error rate %.4f exceeds %.4f", error_rate, self.config.error_rate_threshold)
            return False
        if p99_latency > self.config.latency_p99_threshold_ms:
            logger.error("P99 latency %.1fms exceeds %.1fms", p99_latency, self.config.latency_p99_threshold_ms)
            return False
        return True

    def rollback(self):
        """Immediately route all traffic to old version."""
        logger.warning("ROLLING BACK to %s", self.config.old_version)
        self.set_traffic(self.config.new_version, 0.0)
        self.set_traffic(self.config.old_version, 100.0)
        self.rolled_back = True

    def deploy(self):
        """Execute the progressive canary deployment."""
        self.current_pct = self.config.initial_traffic_pct
        logger.info("Starting canary: %s at %.1f%%", self.config.new_version, self.current_pct)

        while self.current_pct <= self.config.max_traffic_pct:
            self.set_traffic(self.config.new_version, self.current_pct)
            logger.info("Traffic at %.1f%%, observing %ds...", self.current_pct, self.config.observation_window_s)
            time.sleep(self.config.observation_window_s)

            if not self.check_guardrails():
                self.rollback()
                return False

            self.current_pct += self.config.increment_pct

        logger.info("Canary successful. Full rollout complete.")
        return True
Common Mistake

Never canary on percentage of requests — always canary on percentage of users. If you split by requests, a single user might see predictions from two different model versions within the same session, creating an inconsistent experience and corrupting your metrics.

Blue-green deployments for models

Blue-green deployment runs two complete environments (blue = current, green = new), tests the green environment thoroughly, then flips all traffic at once. This is useful when you cannot tolerate any period of mixed model versions. The tradeoff is that you lose the gradual rollout safety of canary deployments.

Shadow mode: testing without serving predictions

In shadow mode, the new model receives a copy of production traffic and generates predictions, but those predictions are never returned to the user. Instead, they are logged for offline comparison against the production model. This lets you evaluate real-world performance without any user impact. Shadow mode is expensive (double the compute) but invaluable for high-stakes models like fraud detection or medical diagnosis.

Self-Check

Before moving to Chapter 07, honestly answer these questions. If any answer is "no", go back and fix it.

Can you explain the tradeoffs between TorchServe, TF Serving, and Triton?

TorchServe is simplest for PyTorch-only teams. TF Serving has the lowest overhead for TensorFlow models. Triton supports all frameworks and has the best GPU utilization via concurrent model execution. Know when to use each.

Do you understand why dynamic batching matters for GPU inference?

Without batching, each request uses only a fraction of GPU compute. Dynamic batching collects requests over a short window and processes them together, achieving near-linear throughput scaling. The tradeoff is slightly higher latency for individual requests.

Can you design a canary deployment that detects a 2% accuracy regression?

You need enough traffic to achieve statistical significance. For a 2% difference with 95% confidence, you need roughly 5,000 labeled outcomes per variant. If your labeling pipeline has a 24-hour delay, your canary observation window must be at least that long.
