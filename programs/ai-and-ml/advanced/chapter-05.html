Why This Matters

Large Language Models have become the most commercially significant development in AI since deep learning itself. GPT-4, Claude, Gemini, and Llama are deployed across thousands of applications generating billions of dollars in revenue. But using an LLM API is not the same as understanding how these systems work at an engineering level.

This chapter takes you inside the machine. You will understand the decoder-only transformer architecture that powers modern LLMs, learn how KV-caching makes autoregressive generation efficient, master quantization techniques that fit 70B parameter models on consumer GPUs, build function-calling agents that use LLMs as reasoning engines, and evaluate LLMs using systematic benchmarks.

These skills are not just academic — they are the difference between an engineer who calls openai.chat.completions.create() and an engineer who can optimize latency by 10x, reduce costs by 5x, and debug the subtle failures that arise in production LLM systems.

Key Insight

An LLM is a compressed representation of its training data. A 70B parameter model (140 GB in float16) was trained on ~2 trillion tokens (~10 TB of text). The model achieves roughly 70x compression of its training data. This compression is lossy — the model captures patterns and associations, not exact facts. This explains both its remarkable capabilities (generalization) and its failure modes (hallucination).

Key Concepts

1. Decoder-Only Transformer Architecture

Modern LLMs use a decoder-only transformer architecture (as opposed to the encoder-decoder architecture of the original "Attention Is All You Need" paper). The key components are: token embeddings, positional encoding (typically RoPE), a stack of identical transformer blocks (each containing multi-head causal self-attention, layer normalization, and a feed-forward network), and a final linear projection to the vocabulary.

Causal attention (also called masked self-attention) is the critical constraint: each token can only attend to tokens at earlier positions (not future tokens). This enables autoregressive generation — the model generates one token at a time, conditioning only on what it has already produced. 

  Decoder-Only Transformer (e.g., Llama, GPT):

  Input tokens: [The] [cat] [sat] [on] [the]
       │
       ▼
  ┌─────────────────────────────────────────┐
  │  Token Embeddings + RoPE Positional Enc │
  └──────────────────┬──────────────────────┘
                     │
                     ▼  (repeated N times, e.g., N=32 for 7B)
  ┌─────────────────────────────────────────┐
  │  Transformer Block:                      │
  │  ┌────────────────────────────────────┐ │
  │  │  RMSNorm                           │ │
  │  │  Multi-Head Causal Self-Attention  │ │
  │  │  (Grouped Query Attention in Llama)│ │
  │  │  Residual Connection               │ │
  │  ├────────────────────────────────────┤ │
  │  │  RMSNorm                           │ │
  │  │  SwiGLU Feed-Forward Network       │ │
  │  │  Residual Connection               │ │
  │  └────────────────────────────────────┘ │
  └──────────────────┬──────────────────────┘
                     │
                     ▼
  ┌─────────────────────────────────────────┐
  │  RMSNorm -> Linear (hidden -> vocab)    │
  │  Output: logits over vocabulary          │
  └─────────────────────────────────────────┘
Figure 1: Decoder-only transformer architecture used by Llama, GPT, and Claude
python
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class RMSNorm(nn.Module):
    """Root Mean Square Layer Normalization (used in Llama, Gemma)."""
    def __init__(self, dim, eps=1e-6):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(dim))
        self.eps = eps

    def forward(self, x):
        norm = torch.rsqrt(x.pow(2).mean(-1, keepdim=True) + self.eps)
        return x * norm * self.weight


class CausalSelfAttention(nn.Module):
    """Multi-head causal (masked) self-attention with KV cache support."""
    def __init__(self, dim, n_heads, n_kv_heads=None):
        super().__init__()
        self.n_heads = n_heads
        self.n_kv_heads = n_kv_heads or n_heads  # GQA if n_kv_heads < n_heads
        self.head_dim = dim // n_heads

        self.q_proj = nn.Linear(dim, n_heads * self.head_dim, bias=False)
        self.k_proj = nn.Linear(dim, self.n_kv_heads * self.head_dim, bias=False)
        self.v_proj = nn.Linear(dim, self.n_kv_heads * self.head_dim, bias=False)
        self.o_proj = nn.Linear(n_heads * self.head_dim, dim, bias=False)

    def forward(self, x, kv_cache=None):
        B, T, C = x.shape
        q = self.q_proj(x).view(B, T, self.n_heads, self.head_dim).transpose(1, 2)
        k = self.k_proj(x).view(B, T, self.n_kv_heads, self.head_dim).transpose(1, 2)
        v = self.v_proj(x).view(B, T, self.n_kv_heads, self.head_dim).transpose(1, 2)

        # Append to KV cache for efficient generation
        if kv_cache is not None:
            k_cache, v_cache = kv_cache
            k = torch.cat([k_cache, k], dim=2)
            v = torch.cat([v_cache, v], dim=2)
        new_cache = (k, v)

        # Grouped Query Attention: repeat KV heads to match Q heads
        if self.n_kv_heads != self.n_heads:
            rep = self.n_heads // self.n_kv_heads
            k = k.repeat_interleave(rep, dim=1)
            v = v.repeat_interleave(rep, dim=1)

        # Scaled dot-product attention with causal mask
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        out = out.transpose(1, 2).contiguous().view(B, T, -1)
        return self.o_proj(out), new_cache

2. KV-Caching and Inference Optimization

Without optimization, generating a sequence of N tokens requires N forward passes, each recomputing attention over all previous tokens — an O(N^2) operation. KV-caching stores the key and value tensors from previous tokens so they do not need to be recomputed. This reduces each generation step from O(N) to O(1) in terms of new computation, making inference dramatically faster.

Other critical inference optimizations include Flash Attention (memory-efficient exact attention using tiling), continuous batching (dynamically add/remove requests from a batch to maximize GPU utilization), and speculative decoding (use a small "draft" model to generate candidate tokens, then verify them in parallel with the large model).

python
from vllm import LLM, SamplingParams

# vLLM: high-throughput LLM serving with PagedAttention
# Automatically handles KV-caching, continuous batching, and memory management

llm = LLM(
    model="meta-llama/Llama-2-13b-chat-hf",
    tensor_parallel_size=2,      # distribute across 2 GPUs
    gpu_memory_utilization=0.90,  # use 90% of GPU memory for KV cache
    max_model_len=4096,           # max context length
)

# Sampling parameters control generation behavior
sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=512,
    stop=["", "###"],  # stop sequences
)

# Batch inference: vLLM handles continuous batching automatically
prompts = [
    "Explain quantum computing in simple terms.",
    "Write a Python function to merge two sorted lists.",
    "What are the key differences between TCP and UDP?",
]

outputs = llm.generate(prompts, sampling_params)
for output in outputs:
    print(f"Prompt: {output.prompt[:50]}...")
    print(f"Output: {output.outputs[0].text[:200]}...")
    print(f"Tokens/sec: {len(output.outputs[0].token_ids) / output.metrics.finished_time:.1f}")
    print()
KV Cache Memory is the Bottleneck

For long contexts, the KV cache can consume more memory than the model weights themselves. A 13B model with 128K context requires ~50 GB just for the KV cache (at float16). This is why techniques like PagedAttention (vLLM), Grouped Query Attention (GQA), and KV cache compression are critical for production deployment.

3. Quantization and Efficient Inference

Quantization reduces model weight precision from float16 (16 bits) to lower bit widths (8-bit, 4-bit, or even 2-bit), dramatically reducing memory requirements and increasing inference speed. A 70B model that requires 140 GB in float16 fits in ~35 GB at 4-bit quantization — making it runnable on a single high-end GPU.

  • GPTQ — post-training quantization using second-order information. One-time calibration on a small dataset, then fast inference. Best for 4-bit quantization.
  • AWQ (Activation-Aware Weight Quantization) — preserves salient weights that have high activation magnitudes. Often better quality than GPTQ at the same bit width.
  • GGUF (llama.cpp) — quantization format for CPU inference. Enables running LLMs on laptops and edge devices without a GPU.
  • bitsandbytes — Hugging Face integration for 8-bit and 4-bit loading. Easy to use but generally slower than GPTQ/AWQ for serving.
python
# Load a 4-bit quantized model with bitsandbytes
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig

    load_in_4bit=True,

    bnb_4bit_compute_dtype=torch.float16,
    bnb_4bit_use_double_quant=True,   # quantize the quantization constants
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-70b-chat-hf",
    quantization_config=bnb_config,
    device_map="auto",  # distribute across available GPUs
)

# Model size comparison:
# float16:  140 GB (2x A100 80GB)
# 4-bit:    ~35 GB (1x A100 80GB or 1x A6000 48GB)
# Quality:  ~97-99% of float16 performance on most benchmarks

# GPTQ: faster inference with pre-quantized weights

from awq import AutoAWQForCausalLM

    "TheBloke/Llama-2-70B-Chat-AWQ",
    fuse_layers=True,  # fuse QKV projections for speed

4. Function Calling and Tool Use

Function calling transforms LLMs from text generators into reasoning engines that can take actions in the real world. The model receives a list of available functions (with their schemas), determines when a function call is needed to answer the user's question, generates the appropriate function call with arguments, and then incorporates the function's return value into its response.

python
from openai import OpenAI
import json

client = OpenAI()

# Define available tools
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "Get current weather for a location",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {"type": "string", "description": "City name"},
                    "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]},
                },
                "required": ["location"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "search_database",
            "description": "Search the product database",
            "parameters": {
                "type": "object",
                "properties": {
                    "query": {"type": "string"},
                    "category": {"type": "string"},
                    "max_results": {"type": "integer", "default": 5},
                },
                "required": ["query"],
            },
        },
    },
]

# Step 1: Send user message with available tools
messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]
response = client.chat.completions.create(
    model="gpt-4", messages=messages, tools=tools, tool_choice="auto",
)

# Step 2: Model decides to call a function
tool_call = response.choices[0].message.tool_calls[0]
args = json.loads(tool_call.function.arguments)
print(f"Function: {tool_call.function.name}, Args: {args}")
# Function: get_weather, Args: {"location": "Tokyo", "unit": "celsius"}

# Step 3: Execute function and send result back
weather_result = get_weather(**args)  # your actual function
messages.append(response.choices[0].message)
messages.append({
    "role": "tool",
    "tool_call_id": tool_call.id,
    "content": json.dumps(weather_result),
})

# Step 4: Model generates final response using function output
final = client.chat.completions.create(model="gpt-4", messages=messages)
print(final.choices[0].message.content)

5. Agentic Systems and ReAct

Agents are LLM-powered systems that can reason about tasks, plan multi-step solutions, call tools, and iterate until the task is complete. The most influential agent pattern is ReAct (Reasoning + Acting), where the model alternates between generating a reasoning trace ("I need to find the current stock price") and taking an action (calling a search API), then observing the result and deciding the next step. 

  ReAct Agent Loop:

  User: "What is Apple's PE ratio compared to Microsoft's?"

  ┌─────────────────────────────────────────────────┐
  │ Thought: I need to look up Apple's PE ratio      │
  │ Action:  search_financials("AAPL")               │
  │ Observation: AAPL PE ratio = 32.5                │
  │                                                  │
  │ Thought: Now I need Microsoft's PE ratio          │
  │ Action:  search_financials("MSFT")               │
  │ Observation: MSFT PE ratio = 37.2                │
  │                                                  │
  │ Thought: I have both values, I can compare        │
  │ Answer:  Apple's PE ratio (32.5) is lower than   │
  │          Microsoft's (37.2), meaning Apple is     │
  │          valued at a lower earnings multiple.     │
  └─────────────────────────────────────────────────┘
Figure 2: ReAct agent alternating between reasoning and tool use
python
from openai import OpenAI
import json

class ReActAgent:
    """Simple ReAct agent with tool use and iterative reasoning."""

    def __init__(self, tools, max_iterations=10):
        self.client = OpenAI()
        self.tools = tools
        self.tool_map = {t["function"]["name"]: t["_callable"] for t in tools}
        self.max_iterations = max_iterations

    def run(self, user_query):
        """Execute the agent loop until completion or max iterations."""
        messages = [
            {"role": "system", "content": """You are a helpful assistant with access to tools.
Use tools when you need real-time data or computations.
Always explain your reasoning before and after using tools."""},
            {"role": "user", "content": user_query},
        ]

        openai_tools = [{"type": "function", "function": t["function"]} for t in self.tools]

        for i in range(self.max_iterations):
            response = self.client.chat.completions.create(
                model="gpt-4",
                messages=messages,
                tools=openai_tools,
                tool_choice="auto",
            )
            msg = response.choices[0].message

            # If no tool calls, the agent has finished
            if not msg.tool_calls:
                return msg.content

            # Execute each tool call
            messages.append(msg)
            for tool_call in msg.tool_calls:
                fn_name = tool_call.function.name
                fn_args = json.loads(tool_call.function.arguments)
                print(f"  [Tool] {fn_name}({fn_args})")

                result = self.tool_map[fn_name](**fn_args)
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": json.dumps(result),
                })

        return "Agent reached max iterations without completing."

6. LLM Evaluation and Benchmarks

Evaluating LLMs is fundamentally harder than evaluating traditional ML models because the output space is vast and quality is subjective. The field uses a combination of automated benchmarks, human evaluation, and LLM-as-judge approaches:

  • MMLU — Massive Multitask Language Understanding. 57 subjects from STEM to humanities. Tests knowledge breadth.
  • HumanEval / MBPP — Code generation benchmarks. Measure whether the model can write correct Python functions from docstrings.
  • MT-Bench — Multi-turn conversation benchmark. Uses GPT-4 as a judge to score response quality on a 1–10 scale.
  • GSM8K — Grade school math problems. Tests multi-step mathematical reasoning.
  • TruthfulQA — Tests whether models generate truthful answers to questions where humans are often wrong.
  • HELM — Holistic Evaluation of Language Models. Evaluates across 42 scenarios covering accuracy, robustness, fairness, and efficiency.
python
from lm_eval import evaluator, tasks

# lm-evaluation-harness: standardized LLM benchmarking
# https://github.com/EleutherAI/lm-evaluation-harness

results = evaluator.simple_evaluate(
    model="hf",
    model_args="pretrained=meta-llama/Llama-2-13b-chat-hf",
    tasks=["mmlu", "gsm8k", "truthfulqa_mc2", "hellaswag"],
    num_fewshot=5,           # 5-shot prompting
    batch_size="auto",        # auto-detect optimal batch size
    device="cuda",
)

# Print results
for task, metrics in results["results"].items():
    acc = metrics.get("acc,none", metrics.get("acc_norm,none", "N/A"))
    print(f"  {task:20s} accuracy: {acc}")

# LLM-as-judge evaluation for open-ended responses
def llm_judge(question, response, criteria="helpfulness, accuracy, clarity"):
    """Use GPT-4 to score a model's response on a 1-10 scale."""
    judge_prompt = f"""Rate the following response on a scale of 1-10 for: {criteria}

Question: {question}
Response: {response}

Provide your rating as JSON: {{"score": N, "reasoning": "..."}}"""

    result = client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": judge_prompt}],
        response_format={"type": "json_object"},
        temperature=0,
    )
    return json.loads(result.choices[0].message.content)
Benchmarks Are Not Reality

Benchmark scores are useful for comparing models but do not predict production performance. A model that scores 90% on MMLU might fail catastrophically on your specific domain. Always build custom evaluation sets that reflect your actual use case, and test on adversarial inputs that probe known failure modes (hallucination, instruction following, edge cases).

What is Grouped Query Attention (GQA) and why is it used in modern LLMs?

Standard multi-head attention gives each head its own Q, K, V projections. Multi-Query Attention (MQA) shares a single K,V across all heads, reducing KV cache memory but hurting quality. GQA is a middle ground: it groups query heads into clusters that share K,V projections. For example, Llama-2 70B uses 64 query heads grouped into 8 KV head groups. This reduces KV cache memory by 8x while maintaining nearly full quality.

How does speculative decoding speed up LLM inference?

Speculative decoding uses a small "draft" model (e.g., 1B params) to generate N candidate tokens quickly. Then the large "target" model verifies all N tokens in a single forward pass (which is as fast as verifying 1 token due to parallelism). If the draft model guessed correctly, you get N tokens for the price of one large-model forward pass. Typical acceptance rates are 70-80%, giving 2-3x speedups. The output is mathematically identical to the large model alone.

Self-Check

Before moving to Chapter 06, honestly answer these questions.

Can you explain the decoder-only transformer architecture from input to output?

Input tokens are converted to embeddings and combined with positional encoding (RoPE). These pass through N transformer blocks, each containing: (1) RMSNorm, (2) multi-head causal self-attention with residual connection, (3) RMSNorm, (4) SwiGLU feed-forward network with residual connection. The final hidden states are normalized and projected to vocabulary logits via a linear layer. Causal masking ensures each token only attends to earlier positions.

How does KV-caching make autoregressive generation efficient?

Without KV-caching, generating token N requires recomputing the key and value projections for all N-1 previous tokens. With KV-caching, the K and V tensors from previous tokens are stored in memory. When generating token N, only the new token's Q, K, V are computed, and the new K, V are appended to the cache. This reduces each generation step from O(N) to O(1) in terms of new K,V computation, though the attention still takes O(N) to attend over all cached keys.

When would you choose 4-bit quantization vs. serving the full float16 model?

Use 4-bit quantization when: (1) you are GPU-memory constrained (e.g., fitting a 70B model on a single GPU), (2) the quality loss of 1-3% is acceptable, (3) you need to reduce serving costs, or (4) you are deploying on edge devices. Use float16 when: (1) maximum quality is required (medical, legal), (2) you have sufficient GPU memory, (3) you are doing fine-tuning (quantized models are harder to train), or (4) benchmarking requires exact precision.

Can you design a function-calling agent for a specific business use case?

A good agent design requires: (1) clear tool definitions with JSON schemas that constrain arguments, (2) a system prompt that explains the agent's role and when to use tools vs. respond directly, (3) error handling for tool failures and malformed calls, (4) a maximum iteration limit to prevent infinite loops, (5) logging of all reasoning steps and tool calls for debugging, and (6) guardrails to prevent the agent from taking dangerous actions (e.g., deleting data without confirmation).

How do you systematically evaluate an LLM for a production use case?

Start with standard benchmarks (MMLU, HumanEval) to establish a baseline. Then build a custom evaluation set of 100+ examples that represent your actual use case, including edge cases and adversarial inputs. Use LLM-as-judge for open-ended responses. Measure: (1) task accuracy, (2) hallucination rate, (3) instruction following, (4) latency and throughput, (5) cost per query, (6) safety (toxicity, bias, refusal). Run evaluations before every model change and compare against the previous version.
