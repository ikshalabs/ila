Chapter 07 · Phase 3: Production ML Skills

Model Evaluation and Tuning

Learn how to rigorously evaluate machine learning models, select the right metrics for your problem, avoid common validation pitfalls, and systematically tune hyperparameters to maximize performance on unseen data.

Estimated: 3–4 days
6 Key Concepts
3 Assignments

Why This Matters

A model that scores 99% accuracy on your training data might perform terribly in the real world. Proper evaluation is the discipline that tells you whether your model has actually learned useful patterns or has merely memorized the training set. Without it, you are flying blind — deploying models that could make costly, even dangerous, mistakes in production.

Evaluation is also where the business meets the math. A medical diagnostic model and a spam filter need completely different evaluation criteria, even if both are binary classifiers. Choosing the wrong metric can lead you to optimize for the wrong thing, producing a model that is technically accurate but practically useless.

Hyperparameter tuning is the other half of this chapter. Every model has knobs that control its behavior — learning rates, regularization strengths, tree depths, network widths. Systematic tuning, rather than manual guesswork, is what separates reproducible ML engineering from ad-hoc experimentation.

Key Insight

The goal of machine learning is not to fit the training data — it is to generalize to unseen data. Every evaluation technique in this chapter serves that single goal. Cross-validation estimates generalization error. Proper metrics measure what matters. Hyperparameter tuning finds the sweet spot between underfitting and overfitting. Keep generalization as your north star.

Key Concepts

1. Train, Validation, and Test Splits

The most fundamental principle of model evaluation is that you must never evaluate on data the model has seen during training. The standard approach splits your data into three non-overlapping sets: the training set (typically 60-80%) for learning parameters, the validation set (10-20%) for tuning hyperparameters, and the test set (10-20%) for the final, unbiased performance estimate.

python 
from sklearn.model_selection import train_test_split
import numpy as np

# Load your features X and labels y
# Step 1: Hold out the test set FIRST — never touch it during development
X_temp, X_test, y_temp, y_test = train_test_split(
    X, y, test_size=0.15, random_state=42, stratify=y
)

# Step 2: Split remaining data into train and validation
X_train, X_val, y_train, y_val = train_test_split(
    X_temp, y_temp, test_size=0.18, random_state=42, stratify=y_temp
)
# Result: ~70% train, ~15% validation, ~15% test

print(f"Training:   {len(X_train)} samples ({len(X_train)/len(X)*100:.0f}%)")
print(f"Validation: {len(X_val)} samples ({len(X_val)/len(X)*100:.0f}%)")
print(f"Test:       {len(X_test)} samples ({len(X_test)/len(X)*100:.0f}%)")

# CRITICAL: Use stratify=y for classification to maintain class proportions
# For time series: use temporal splits, not random splits!
Data Leakage: The Silent Killer

Data leakage occurs when information from the test set contaminates the training process. Common forms include: fitting a scaler on the full dataset before splitting, using future data in time series features, and including features derived from the target variable. Leakage produces unrealistically good validation scores that collapse in production. Always fit preprocessors on the training set only, then transform the validation and test sets.

2. Cross-Validation: Robust Performance Estimation

A single train/validation split gives you a single performance estimate, which may be unreliable due to the randomness of the split. K-Fold Cross-Validation solves this by splitting the data into K folds, training K models (each using a different fold as validation), and averaging the results. This gives you both a more accurate estimate and a measure of variance.

python 
from sklearn.model_selection import (
    cross_val_score, StratifiedKFold, TimeSeriesSplit
)
from sklearn.ensemble import RandomForestClassifier

model = RandomForestClassifier(n_estimators=100, random_state=42)

# Standard K-Fold Cross-Validation (K=5)
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=42)
scores = cross_val_score(model, X_train, y_train, cv=cv, scoring='accuracy')

print(f"CV Accuracy: {scores.mean():.4f} (+/- {scores.std():.4f})")
print(f"Per-fold scores: {scores.round(4)}")

# For time series data — use TimeSeriesSplit instead
tscv = TimeSeriesSplit(n_splits=5)
ts_scores = cross_val_score(model, X_train, y_train, cv=tscv, scoring='accuracy')
print(f"Time Series CV: {ts_scores.mean():.4f} (+/- {ts_scores.std():.4f})")

# Multiple metrics at once
from sklearn.model_selection import cross_validate

results = cross_validate(
    model, X_train, y_train, cv=cv,
    scoring=['accuracy', 'f1_weighted', 'roc_auc_ovr'],
    return_train_score=True
)

for metric in ['accuracy', 'f1_weighted', 'roc_auc_ovr']:
    train_score = results[f'train_{metric}'].mean()
    val_score = results[f'test_{metric}'].mean()
    print(f"{metric}: train={train_score:.4f}, val={val_score:.4f}")

  K-FOLD CROSS-VALIDATION (K=5)

  Fold 1:  [  VAL  |  TRAIN  |  TRAIN  |  TRAIN  |  TRAIN  ]  -> Score 1
  Fold 2:  [ TRAIN |   VAL   |  TRAIN  |  TRAIN  |  TRAIN  ]  -> Score 2
  Fold 3:  [ TRAIN |  TRAIN  |   VAL   |  TRAIN  |  TRAIN  ]  -> Score 3
  Fold 4:  [ TRAIN |  TRAIN  |  TRAIN  |   VAL   |  TRAIN  ]  -> Score 4
  Fold 5:  [ TRAIN |  TRAIN  |  TRAIN  |  TRAIN  |   VAL   ]  -> Score 5

  Final Score = mean(Score 1..5) +/- std(Score 1..5)
Fig 1. Each fold serves as validation exactly once. The final score is the average across all folds.

3. Classification Metrics: Beyond Accuracy

Accuracy is the first metric everyone learns, but it is often the wrong metric. Consider a fraud detection system: if 99.5% of transactions are legitimate, a model that always predicts "not fraud" achieves 99.5% accuracy while catching zero fraudsters. The right metric depends on the cost of different types of errors.

python 
from sklearn.metrics import (
    classification_report, confusion_matrix, roc_auc_score,
    precision_recall_curve, roc_curve, f1_score, average_precision_score
)
import matplotlib.pyplot as plt
import numpy as np

# Train a model and get predictions
model.fit(X_train, y_train)
y_pred = model.predict(X_val)
y_proba = model.predict_proba(X_val)[:, 1]

# Confusion Matrix: the foundation of all classification metrics
cm = confusion_matrix(y_val, y_pred)
print("Confusion Matrix:")
print(f"  TN={cm[0,0]}  FP={cm[0,1]}")
print(f"  FN={cm[1,0]}  TP={cm[1,1]}")

# Full classification report: precision, recall, F1 per class
print("\nClassification Report:")
print(classification_report(y_val, y_pred, digits=4))

# Key metric definitions:
# Precision = TP / (TP + FP) — "Of predicted positives, how many are correct?"
# Recall    = TP / (TP + FN) — "Of actual positives, how many did we catch?"
# F1-Score  = 2 * (P * R) / (P + R) — harmonic mean of precision and recall

# AUC-ROC: measures ranking quality across all thresholds
auc = roc_auc_score(y_val, y_proba)
print(f"AUC-ROC: {auc:.4f}")

# Average Precision: better for imbalanced datasets than AUC-ROC
ap = average_precision_score(y_val, y_proba)
print(f"Average Precision: {ap:.4f}")
Choosing the Right Metric

Fraud detection / disease screening: Optimize for recall (minimize false negatives, even at the cost of more false positives). Spam filtering / content moderation: Optimize for precision (minimize false positives, because blocking legitimate content is costly). Balanced problems: Use F1-score as a balanced trade-off. Ranking / scoring: Use AUC-ROC or Average Precision to evaluate probability calibration.

python 
# Plotting ROC and Precision-Recall curves
fig, axes = plt.subplots(1, 2, figsize=(14, 5))

# ROC Curve
fpr, tpr, thresholds = roc_curve(y_val, y_proba)
axes[0].plot(fpr, tpr, 'b-', label=f'Model (AUC={auc:.3f})')
axes[0].plot([0, 1], [0, 1], 'k--', label='Random')
axes[0].set_xlabel('False Positive Rate')
axes[0].set_ylabel('True Positive Rate')
axes[0].set_title('ROC Curve')
axes[0].legend()

# Precision-Recall Curve
precision, recall, pr_thresholds = precision_recall_curve(y_val, y_proba)
axes[1].plot(recall, precision, 'r-', label=f'Model (AP={ap:.3f})')
axes[1].set_xlabel('Recall')
axes[1].set_ylabel('Precision')
axes[1].set_title('Precision-Recall Curve')
axes[1].legend()

plt.tight_layout()
plt.savefig('evaluation_curves.png', dpi=150)
plt.show()

4. Regression Metrics and Residual Analysis

For regression problems, the core metrics are Mean Squared Error (MSE), its square root RMSE (which has the same units as the target), Mean Absolute Error (MAE) (more robust to outliers), and R-squared (the proportion of variance explained). Beyond single-number metrics, residual plots reveal systematic errors that aggregate metrics can hide.

python 
from sklearn.metrics import mean_squared_error, mean_absolute_error, r2_score
import numpy as np
import matplotlib.pyplot as plt

# Compute regression metrics
y_pred = model.predict(X_val)
mse = mean_squared_error(y_val, y_pred)
rmse = np.sqrt(mse)
mae = mean_absolute_error(y_val, y_pred)
r2 = r2_score(y_val, y_pred)

print(f"MSE:  {mse:.4f}")
print(f"RMSE: {rmse:.4f}")
print(f"MAE:  {mae:.4f}")
print(f"R2:   {r2:.4f}")

# Residual analysis — reveals systematic patterns
residuals = y_val - y_pred

fig, axes = plt.subplots(1, 3, figsize=(18, 5))

# 1. Residuals vs Predicted — should show no pattern
axes[0].scatter(y_pred, residuals, alpha=0.3, s=10)
axes[0].axhline(y=0, color='red', linestyle='--')
axes[0].set_xlabel('Predicted Values')
axes[0].set_ylabel('Residuals')
axes[0].set_title('Residuals vs Predicted')

# 2. Distribution of residuals — should be approximately normal
axes[1].hist(residuals, bins=50, edgecolor='black')
axes[1].set_xlabel('Residual')
axes[1].set_title('Residual Distribution')

# 3. Actual vs Predicted — should align on the diagonal
axes[2].scatter(y_val, y_pred, alpha=0.3, s=10)
axes[2].plot([y_val.min(), y_val.max()], [y_val.min(), y_val.max()], 'r--')
axes[2].set_xlabel('Actual')
axes[2].set_ylabel('Predicted')
axes[2].set_title('Actual vs Predicted')

plt.tight_layout()
plt.show()

5. Hyperparameter Tuning: Grid Search, Random Search, and Bayesian Optimization

Hyperparameters are settings you choose before training (learning rate, number of trees, regularization strength). Unlike model parameters (weights), they are not learned from data. Finding good hyperparameters requires a systematic search strategy combined with proper cross-validation.

python 
from sklearn.model_selection import GridSearchCV, RandomizedSearchCV
from sklearn.ensemble import GradientBoostingClassifier
from scipy.stats import randint, uniform

# Grid Search — exhaustive, good for small search spaces
param_grid = {
    'n_estimators': [100, 200, 300],
    'max_depth': [3, 5, 7],
    'learning_rate': [0.01, 0.05, 0.1],
    'subsample': [0.8, 1.0]
}

grid_search = GridSearchCV(
    GradientBoostingClassifier(random_state=42),
    param_grid,
    cv=5,
    scoring='f1_weighted',
    n_jobs=-1,             # Use all CPU cores
    verbose=1,
    return_train_score=True
)
grid_search.fit(X_train, y_train)

print(f"Best params: {grid_search.best_params_}")
print(f"Best CV F1:  {grid_search.best_score_:.4f}")

# Random Search — more efficient for large search spaces
param_distributions = {
    'n_estimators': randint(50, 500),
    'max_depth': randint(2, 10),
    'learning_rate': uniform(0.005, 0.2),
    'subsample': uniform(0.6, 0.4),
    'min_samples_leaf': randint(1, 20)
}

random_search = RandomizedSearchCV(
    GradientBoostingClassifier(random_state=42),
    param_distributions,
    n_iter=50,              # Try 50 random combinations
    cv=5,
    scoring='f1_weighted',
    n_jobs=-1,
    random_state=42
)
random_search.fit(X_train, y_train)

print(f"Best params: {random_search.best_params_}")
print(f"Best CV F1:  {random_search.best_score_:.4f}")
Pro Tip

Research by Bergstra and Bengio (2012) showed that random search is almost always more efficient than grid search when you have a limited compute budget. Grid search wastes trials on unimportant hyperparameters, while random search explores each dimension more broadly. For even better efficiency, use Bayesian optimization with libraries like Optuna or scikit-optimize.

python 
import optuna

# Bayesian Optimization with Optuna — the modern approach
def objective(trial):
    params = {
        'n_estimators': trial.suggest_int('n_estimators', 50, 500),
        'max_depth': trial.suggest_int('max_depth', 2, 10),
        'learning_rate': trial.suggest_float('learning_rate', 0.005, 0.2, log=True),
        'subsample': trial.suggest_float('subsample', 0.6, 1.0),
        'min_samples_leaf': trial.suggest_int('min_samples_leaf', 1, 20),
    }

    model = GradientBoostingClassifier(**params, random_state=42)
    scores = cross_val_score(model, X_train, y_train, cv=5, scoring='f1_weighted')
    return scores.mean()

# Run optimization
study = optuna.create_study(direction='maximize')
study.optimize(objective, n_trials=100, show_progress_bar=True)

print(f"Best F1: {study.best_value:.4f}")
print(f"Best params: {study.best_params}")

# Optuna visualizations
optuna.visualization.plot_optimization_history(study)
optuna.visualization.plot_param_importances(study)

6. Bias-Variance Tradeoff and Learning Curves

The bias-variance tradeoff is the fundamental tension in machine learning. High bias (underfitting) means the model is too simple to capture the data's patterns. High variance (overfitting) means the model is so complex it memorizes noise. The goal is to find the sweet spot where both are balanced.

python 
from sklearn.model_selection import learning_curve
import matplotlib.pyplot as plt
import numpy as np

def plot_learning_curve(estimator, X, y, title, cv=5):
    """Plot learning curves to diagnose bias vs variance."""
    train_sizes, train_scores, val_scores = learning_curve(
        estimator, X, y,
        train_sizes=np.linspace(0.1, 1.0, 10),
        cv=cv,
        scoring='accuracy',
        n_jobs=-1
    )

    train_mean = train_scores.mean(axis=1)
    train_std = train_scores.std(axis=1)
    val_mean = val_scores.mean(axis=1)
    val_std = val_scores.std(axis=1)

    plt.figure(figsize=(10, 6))
    plt.fill_between(train_sizes, train_mean - train_std,
                     train_mean + train_std, alpha=0.1, color='blue')
    plt.fill_between(train_sizes, val_mean - val_std,
                     val_mean + val_std, alpha=0.1, color='orange')
    plt.plot(train_sizes, train_mean, 'o-', color='blue', label='Training score')
    plt.plot(train_sizes, val_mean, 'o-', color='orange', label='Validation score')

    plt.xlabel('Training Set Size')
    plt.ylabel('Score')
    plt.title(title)
    plt.legend(loc='best')
    plt.grid(True, alpha=0.3)
    plt.show()

# Diagnose: High bias (underfitting)
# Both curves plateau at a LOW score → need more complex model

# Diagnose: High variance (overfitting)
# Training score HIGH, validation score LOW, big gap → need regularization

# Diagnose: Good fit
# Both curves converge to a HIGH score → model generalizes well
Diagnostic Decision Tree

High training error + high validation error = underfitting. Try: more features, more complex model, less regularization. Low training error + high validation error = overfitting. Try: more data, more regularization, simpler model, dropout, early stopping. Low training error + low validation error = good fit. Ship it (after final test set evaluation).

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is "no," go back and practice.

Can you explain why accuracy alone is insufficient for imbalanced classification?

With a 99:1 class ratio, a model that always predicts the majority class achieves 99% accuracy while being completely useless for detecting the minority class. Metrics like precision, recall, F1-score, and AUC-ROC account for the types of errors (false positives vs. false negatives) and give a much more informative picture of model quality.

Do you understand the difference between the validation set and the test set?

The validation set is used during development to tune hyperparameters and select between models. The test set is used exactly once at the very end to get an unbiased estimate of final performance. If you repeatedly evaluate on the test set and adjust your approach based on the results, it becomes a validation set and your final estimate is biased.

Can you diagnose underfitting vs. overfitting from a learning curve?

Underfitting: both training and validation curves plateau at a low score, often converging together. Overfitting: training score is high but validation score is significantly lower, with a persistent gap. More training data helps overfitting (curves converge) but not underfitting (you need a more expressive model).

Do you know when to use ROC-AUC vs. Precision-Recall AUC?

ROC-AUC works well for balanced datasets and measures overall discrimination ability. However, it can be overly optimistic for imbalanced data because the false positive rate denominator includes many true negatives. Precision-Recall AUC (Average Precision) focuses on the positive class and is more informative when the positive class is rare (fraud, disease, etc.).

Can you set up a hyperparameter search with Optuna from scratch?

You should be able to define an Optuna objective function that: creates a model with trial-suggested hyperparameters, evaluates it with cross-validation, and returns the score. Then create a study, call study.optimize(), and extract the best parameters. Bonus: use pruning callbacks to terminate unpromising trials early, saving compute time.
