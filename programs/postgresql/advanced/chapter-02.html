Advanced Indexing Strategies

Master GIN, GiST, BRIN, expression indexes, partial indexes, covering indexes, and index-only scans to keep every ad portal query fast at scale.

Estimated: 4–5 hours
6 Key Concepts
3 Assignments

Why This Matters

In the ad portal, queries filter campaigns by status, search leads by geographic region, match ad groups by bid ranges, and look up accounts by JSONB metadata. A single missing index on a 50-million-row leads table can turn a 2ms lookup into a 12-second sequential scan. Advanced indexing strategies — GIN, GiST, BRIN, partial indexes, expression indexes, and covering indexes — are the tools that keep every dashboard responsive and every bid evaluation under budget.

Key Insight

B-tree is the default, but it is not always the best. PostgreSQL ships with multiple index types, each optimized for a different access pattern. Choosing the right index type is as important as choosing to index at all.

Key Concepts

1. GIN Indexes (Generalized Inverted Index)

GIN indexes are designed for values that contain multiple elements — arrays, JSONB documents, and full-text search vectors. They create an inverted index where each element points back to the rows that contain it. In the ad portal, GIN is essential for querying JSONB campaign metadata and array-typed targeting fields.

sql
-- GIN index on JSONB column for campaign settings
CREATE INDEX idx_campaigns_settings_gin
  ON campaigns USING gin (settings);

-- Query that benefits: find campaigns with specific feature flags
SELECT id, name, monthly_budget
FROM campaigns
WHERE settings @> '{"auto_bid": true, "rtb_enabled": true}'::jsonb;

-- GIN index on array column for ad group targeting
CREATE INDEX idx_ad_group_locations_zip_gin
  ON ad_group_locations USING gin (zip_codes);

-- Find ad groups targeting a specific zip code
SELECT ag.id, ag.name
FROM ad_groups ag
JOIN ad_group_locations agl ON agl.ad_group_id = ag.id
WHERE agl.zip_codes @> ARRAY['90210'];

-- GIN index for full-text search on lead notes
CREATE INDEX idx_leads_notes_fts
  ON leads USING gin (to_tsvector('english', notes));

SELECT id, first_name, last_name
FROM leads
WHERE to_tsvector('english', notes) @@ to_tsquery('insurance & quote');
GIN Write Overhead

GIN indexes are expensive to update. Each INSERT into a JSONB column may update dozens of index entries. Use gin_pending_list_limit to batch updates and consider fastupdate = off for read-heavy workloads where you need consistent read performance at the cost of slower writes.

2. GiST Indexes (Generalized Search Tree)

GiST indexes support geometric, range, and proximity queries. In ad-tech, they power geographic targeting (finding ad groups serving a specific latitude/longitude) and range queries (finding campaigns with overlapping budget periods). GiST is also used for exclusion constraints.

sql
-- GiST index for range queries on campaign date ranges
CREATE TABLE campaign_schedules (
  id          BIGSERIAL PRIMARY KEY,
  campaign_id BIGINT REFERENCES campaigns(id),
  active_range tstzrange NOT NULL
);

CREATE INDEX idx_sched_range_gist
  ON campaign_schedules USING gist (active_range);

-- Find campaigns active at a specific timestamp
SELECT cs.campaign_id, c.name
FROM campaign_schedules cs
JOIN campaigns c ON c.id = cs.campaign_id
WHERE cs.active_range @> NOW();

-- Exclusion constraint: no overlapping schedules per campaign
ALTER TABLE campaign_schedules
  ADD CONSTRAINT no_overlap
  EXCLUDE USING gist (campaign_id WITH =, active_range WITH &&);

-- GiST for geographic point queries (with PostGIS or built-in point type)
CREATE INDEX idx_leads_geo_gist
  ON leads USING gist (geo_point);

SELECT id, first_name
FROM leads
WHERE geo_point <@> circle '((40.7128,-74.006),0.1)';

3. BRIN Indexes (Block Range Index)

BRIN indexes are extremely small and efficient for columns whose values correlate with physical row order — typically timestamps and auto-incrementing IDs. The ad portal's ad_group_stats table, with its dt (date) column, is a textbook BRIN candidate: rows are inserted chronologically, so nearby rows have similar dates.

sql
-- BRIN index on the stats date column (extremely compact)
CREATE INDEX idx_stats_dt_brin
  ON ad_group_stats USING brin (dt)
  WITH (pages_per_range = 32);

-- This query scans only blocks containing the target date range
SELECT campaign_id,
       SUM(impressions) AS total_impressions,
       SUM(clicks)      AS total_clicks,
       SUM(spend)       AS total_spend
FROM ad_group_stats
WHERE dt BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY campaign_id;

-- Compare index sizes: BRIN vs B-tree on same column
-- BRIN: ~48 KB for 100M rows
-- B-tree: ~2.1 GB for 100M rows
SELECT pg_size_pretty(pg_relation_size('idx_stats_dt_brin'));
BRIN Sweet Spot

BRIN shines on append-only tables where data arrives in order. The pages_per_range parameter controls granularity: lower values (16-32) give better selectivity but slightly larger indexes. For the ad_group_stats table with hundreds of millions of rows, BRIN saves gigabytes of disk and memory compared to B-tree.

4. Expression and Partial Indexes

Expression indexes index the result of a function or expression, not the raw column value. Partial indexes index only a subset of rows matching a WHERE condition. Together, they create focused, efficient indexes that serve specific query patterns in the ad portal.

sql
-- Expression index: case-insensitive company name search
CREATE INDEX idx_accounts_company_lower
  ON accounts (LOWER(company_name));

SELECT * FROM accounts
WHERE LOWER(company_name) = 'acme insurance';

-- Expression index: extract month from lead created_at
CREATE INDEX idx_leads_month
  ON leads (EXTRACT(MONTH FROM created_at));

-- Partial index: only active campaigns (excludes soft-deleted)
CREATE INDEX idx_campaigns_active
  ON campaigns (account_id, monthly_budget)
  WHERE discarded_at IS NULL;

-- Partial index: only high-value leads (disposition = 'sold')
CREATE INDEX idx_leads_sold
  ON leads (campaign_id, created_at)
  WHERE disposition = 'sold';

-- This query uses the partial index (much smaller than full index)
SELECT campaign_id, COUNT(*) AS sold_count
FROM leads
WHERE disposition = 'sold'
  AND created_at >= '2024-01-01'
GROUP BY campaign_id;

5. Covering Indexes and Index-Only Scans

A covering index includes all columns a query needs, enabling an Index Only Scan that never touches the table heap. PostgreSQL uses the INCLUDE clause to add non-key columns to an index. This is transformative for dashboard queries that aggregate a few columns from huge tables.

sql
-- Covering index for campaign dashboard query
CREATE INDEX idx_campaigns_dashboard
  ON campaigns (account_id)
  INCLUDE (name, monthly_budget, status)
  WHERE discarded_at IS NULL;

-- This query is satisfied entirely from the index
EXPLAIN ANALYZE
SELECT name, monthly_budget, status
FROM campaigns
WHERE account_id = 42
  AND discarded_at IS NULL;
-- Expected: Index Only Scan using idx_campaigns_dashboard

-- Covering index for stats aggregation
CREATE INDEX idx_stats_covering
  ON ad_group_stats (campaign_id, dt)
  INCLUDE (impressions, clicks, spend);

-- Dashboard totals without touching the heap
SELECT dt,
       SUM(impressions) AS impressions,
       SUM(clicks)      AS clicks,
       SUM(spend)       AS spend
FROM ad_group_stats
WHERE campaign_id = 7
  AND dt BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY dt
ORDER BY dt;
Visibility Map Requirement

Index Only Scans require the visibility map to be up to date. If VACUUM has not run recently, PostgreSQL may fall back to a regular Index Scan to check row visibility. Monitor idx_scan vs idx_tup_fetch in pg_stat_user_indexes to verify Index Only Scans are working.

6. Index Maintenance and Monitoring

Indexes are not fire-and-forget. They bloat over time, can become corrupt, and consume memory. Regular monitoring and maintenance keep them efficient. The ad portal runs nightly REINDEX jobs on high-churn tables like ad_group_stats and bid_change_histories.

sql
-- Check index usage statistics
SELECT schemaname, tablename, indexname,
       idx_scan, idx_tup_read, idx_tup_fetch,
       pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY idx_scan ASC;

-- Find unused indexes (candidates for removal)
SELECT indexrelname, idx_scan,
       pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;

-- Check index bloat estimate
SELECT nspname, relname,
       round((100 * pg_relation_size(indexrelid) /
         pg_relation_size(indrelid))::numeric, 1) AS idx_to_tbl_pct
FROM pg_stat_user_indexes
JOIN pg_index USING (indexrelid)
WHERE pg_relation_size(indrelid) > 0
ORDER BY idx_to_tbl_pct DESC;

-- Rebuild bloated index without locking
REINDEX INDEX CONCURRENTLY idx_leads_sold;

Self-Check

When should you choose GIN over GiST for a JSONB column?

Use GIN when you need exact containment queries (@>, ?, ?|, ?&) — GIN is faster for lookups but slower to update. Use GiST when you need ordering or nearest-neighbor queries, or when you need faster writes at the cost of slightly slower reads. For the ad portal JSONB settings column, GIN is usually the right choice because reads dominate writes.

Why can BRIN indexes be thousands of times smaller than B-tree indexes?

BRIN stores only the min/max values for each block range (e.g., 32 pages), not individual row pointers. For a 100M-row table, a B-tree stores 100M index entries, while BRIN stores summary data for perhaps 50,000 block ranges. The tradeoff is that BRIN may read extra blocks if data is not perfectly ordered within ranges.

What is the visibility map and why does it matter for Index Only Scans?

The visibility map tracks which heap pages contain only tuples visible to all transactions. An Index Only Scan can skip heap access for pages marked as all-visible. If VACUUM has not run, many pages may not be marked visible, forcing PostgreSQL to check the heap anyway — negating the benefit. Run VACUUM regularly on tables where you rely on Index Only Scans.
