Chapter 08 · Phase 4: Resilience Engineering

Backup and Disaster Recovery

Master every PostgreSQL backup strategy from logical dumps to continuous archiving with PITR. Design recovery plans that meet real RPO/RTO targets, automate backup verification, and build confidence that your data survives any failure scenario.

Estimated: 4–5 days
6 Key Concepts
3 Assignments

Why This Matters

Data is the most valuable asset an organization owns, and databases are where that data lives. Hardware fails, disks corrupt, operators make mistakes, and ransomware encrypts files. The difference between a minor inconvenience and a business-ending catastrophe is whether you have a tested, reliable backup and recovery strategy. In PostgreSQL, "backup" is not a single tool — it is an ecosystem of complementary approaches, each with distinct trade-offs in speed, granularity, and recovery flexibility.

Many teams discover their backup strategy is broken only when they need it most. A backup that has never been restored is not a backup — it is a hope. Production incidents have shown that corrupt WAL archives, misconfigured retention policies, and untested restore procedures are alarmingly common. This chapter teaches you to design, implement, and continuously verify backup systems that you can trust under pressure.

Beyond just creating backups, disaster recovery planning requires understanding Recovery Point Objective (RPO) — how much data loss is acceptable — and Recovery Time Objective (RTO) — how quickly the system must be restored. A proper DR plan maps every failure scenario to a documented, rehearsed runbook so that recovery is methodical rather than panicked.

Key Insight

Your backup strategy must be defined by business requirements, not by technical convenience. Start with the question: "How much data can we afford to lose, and how long can we be down?" Then work backwards to choose the right combination of pg_dump, pg_basebackup, WAL archiving, and third-party tools like pgBackRest.

Key Concepts

1. Logical Backups with pg_dump and pg_dumpall

Logical backups export database objects as SQL statements or a custom archive format. pg_dump backs up a single database, while pg_dumpall captures all databases plus cluster-wide objects like roles and tablespaces. Logical backups are portable across PostgreSQL major versions and allow selective restoration of individual tables, schemas, or data subsets.

The trade-off is speed: logical backups must read every row through the SQL engine, which is slow for multi-terabyte databases. They also represent a snapshot taken at the start of the dump transaction, meaning any changes made during the dump window are not captured. For databases under 100 GB, logical backups are often sufficient as the primary strategy. For larger databases, they serve as a complement to physical backups.

bash 
# Custom format: compressed, supports parallel restore
pg_dump -Fc -Z6 -j4 -d mydb -f /backups/mydb_$(date +%Y%m%d_%H%M%S).dump

# Directory format: parallel dump with 8 workers
pg_dump -Fd -j8 -d mydb -f /backups/mydb_directory/

# Dump specific schemas only
pg_dump -Fc -n public -n analytics -d mydb -f /backups/schemas.dump

# Dump global objects separately (roles, tablespaces)
pg_dumpall --globals-only -f /backups/globals.sql

# Schema-only and data-only dumps for flexibility
pg_dump -Fc --schema-only -d mydb -f /backups/mydb_schema.dump
pg_dump -Fc --data-only -d mydb -f /backups/mydb_data.dump
bash 
# Restore from custom format with parallel workers
pg_restore -d mydb_restored -Fc -j4 -c /backups/mydb.dump

# Restore specific tables only
pg_restore -d mydb -t orders -t customers /backups/mydb.dump

# List dump contents before restoring
pg_restore -l /backups/mydb.dump | head -50

# Selective restore via table of contents
pg_restore -l /backups/mydb.dump > /tmp/toc.list
# Edit toc.list to comment out unwanted items, then:
pg_restore -d mydb -L /tmp/toc.list /backups/mydb.dump
pg_dumpall Limitation

pg_dumpall only outputs plain SQL — no custom or directory format, no parallel restore, no selective object restore. For production, dump globals separately with pg_dumpall --globals-only and use per-database pg_dump -Fc for each database.

2. Physical Backups with pg_basebackup

Physical backups copy the raw data files of a PostgreSQL cluster via the replication protocol. pg_basebackup creates a consistent snapshot of the entire data directory. Unlike logical backups, physical backups capture everything — all databases, indexes, WAL position — and restore much faster because no SQL parsing is needed. They are the foundation of point-in-time recovery (PITR).

When combined with WAL archiving, physical backups allow you to restore the database to any moment between the backup timestamp and the last archived WAL segment. This makes them essential for near-zero data loss requirements.

bash 
# Compressed tar backup with WAL streaming
pg_basebackup -D /backups/base_$(date +%Y%m%d) \
  -Ft -z -P \
  -X stream \
  -c fast \
  -l "daily_backup_$(date +%Y%m%d)"

# Plain format with standby signal for immediate replica
pg_basebackup -D /var/lib/postgresql/16/standby \
  -Fp -R -X stream -c fast -P

# Remote backup from a primary server
pg_basebackup -h primary.db.internal \
  -U replication_user \
  -D /backups/remote_base \
  -Ft -z -P -X stream

# Verify backup integrity (PostgreSQL 13+)
pg_verifybackup /backups/base_20250224

3. WAL Archiving and Continuous Archiving

Write-Ahead Logging (WAL) is PostgreSQL's durability mechanism. Every change is written to WAL before being applied to data files. WAL archiving copies completed segments to an external location, creating a continuous record of all changes since the last base backup. This is the key ingredient for point-in-time recovery.

PostgreSQL offers the traditional archive_command which runs a shell command for each completed WAL segment, and the newer archive_library (PostgreSQL 15+). The archive_timeout setting forces archiving even if a segment is not full, controlling your maximum data loss window.

conf 
# postgresql.conf — WAL archiving configuration
wal_level = replica
archive_mode = on

# Copy WAL segments to archive directory
archive_command = 'test ! -f /wal_archive/%f && cp %p /wal_archive/%f'

# Force archiving every 5 minutes (max data loss window)
archive_timeout = 300

# Retain WAL for streaming replicas
max_wal_senders = 5
wal_keep_size = '2GB'
sql 
-- Verify WAL archiving is working
SELECT * FROM pg_stat_archiver;

-- Force a WAL segment switch for testing
SELECT pg_switch_wal();

-- Check archive health and lag
SELECT
    archived_count, failed_count,
    last_archived_wal, last_archived_time,
    CASE
        WHEN last_archived_time < now() - interval '10 minutes'
        THEN 'STALE - check archive_command'
        ELSE 'OK'
    END AS archive_status
FROM pg_stat_archiver;
Critical Warning

If archive_command fails persistently, PostgreSQL retains all unarchived WAL segments, eventually filling pg_wal and crashing the server. Always monitor pg_stat_archiver.failed_count and disk usage of pg_wal. A full WAL directory is one of the most common production outages.

4. Point-in-Time Recovery (PITR)

PITR allows restoring a database to any specific moment in time. This is critical for recovering from accidental data deletion or corruption — you can roll forward to the moment just before the destructive event. PITR requires a base backup plus a continuous chain of archived WAL segments.

bash 
# Step 1: Stop PostgreSQL and restore the base backup
systemctl stop postgresql-16
rm -rf /var/lib/postgresql/16/main/*
tar xzf /backups/base_20250220/base.tar.gz \
  -C /var/lib/postgresql/16/main/
tar xzf /backups/base_20250220/pg_wal.tar.gz \
  -C /var/lib/postgresql/16/main/pg_wal/
chown -R postgres:postgres /var/lib/postgresql/16/main/
conf 
# postgresql.conf — PITR recovery configuration
restore_command = 'cp /wal_archive/%f %p'
recovery_target_time = '2025-02-23 14:30:00 UTC'
recovery_target_action = 'pause'
recovery_target_inclusive = false

# Alternative targets:
# recovery_target_xid = '12345678'
# recovery_target_name = 'before_migration'
# recovery_target_lsn = '0/1A2B3C4D'
bash 
# Step 2: Create recovery signal and start
touch /var/lib/postgresql/16/main/recovery.signal
systemctl start postgresql-16

# Step 3: Monitor recovery progress
psql -c "SELECT pg_is_in_recovery();"
psql -c "SELECT pg_last_wal_replay_lsn(),
       pg_last_xact_replay_timestamp();"

# Step 4: When satisfied, promote to read-write
psql -c "SELECT pg_wal_replay_resume();"

# Create named restore points before risky operations
psql -c "SELECT pg_create_restore_point('before_migration_v42');"
Pro Tip

Before any risky operation, create a named restore point with pg_create_restore_point('before_migration'). Target this exact point in PITR using recovery_target_name instead of guessing timestamps.

5. Enterprise Backup with pgBackRest

pgBackRest is the de facto standard for production PostgreSQL backup management. It provides parallel backup/restore, full/differential/incremental backup types, backup rotation, integrity verification, and support for S3, GCS, and Azure Blob repositories. It replaces the manual combination of pg_basebackup plus archive scripts with a single, battle-tested tool.

conf 
# /etc/pgbackrest/pgbackrest.conf
[global]
repo1-path=/var/lib/pgbackrest
repo1-retention-full=4
repo1-retention-diff=7
repo1-cipher-type=aes-256-cbc
repo1-cipher-pass=your-encryption-passphrase
process-max=4
compress-type=zst
compress-level=6
log-level-console=info
log-level-file=detail

[mydb]
pg1-path=/var/lib/postgresql/16/main
pg1-port=5432
bash 
# Initialize and verify stanza
pgbackrest --stanza=mydb stanza-create
pgbackrest --stanza=mydb check

# Full / differential / incremental backups
pgbackrest --stanza=mydb --type=full backup
pgbackrest --stanza=mydb --type=diff backup
pgbackrest --stanza=mydb --type=incr backup

# List backups and verify integrity
pgbackrest --stanza=mydb info
pgbackrest --stanza=mydb verify

# PITR restore to specific time
pgbackrest --stanza=mydb \
  --type=time \
  --target="2025-02-23 14:30:00 UTC" \
  --target-action=promote \
  restore

6. Disaster Recovery Planning and Testing

A DR plan covers every plausible failure scenario: single disk failure, complete server loss, data center outage, accidental data deletion, and data corruption. Each scenario maps to a recovery procedure with clear RPO/RTO targets. The most critical aspect is regular testing.

bash 
#!/bin/bash
# Automated backup verification script
set -euo pipefail

STANZA="mydb"
RESTORE_DIR="/tmp/backup_verify"
VERIFY_PORT=5433

log() { echo "$(date '+%Y-%m-%d %H:%M:%S') $1"; }

log "Starting backup verification"
rm -rf "$RESTORE_DIR" && mkdir -p "$RESTORE_DIR"

# Restore latest backup
pgbackrest --stanza=$STANZA \
  --pg1-path=$RESTORE_DIR \
  --target-action=promote restore

# Start on verification port and run checks
pg_ctl start -D "$RESTORE_DIR" -o "-p $VERIFY_PORT" -w
psql -p "$VERIFY_PORT" -d mydb -c "SELECT count(*) FROM orders;"

# Index integrity check with amcheck
psql -p "$VERIFY_PORT" -d mydb -c \
  "CREATE EXTENSION IF NOT EXISTS amcheck;
   SELECT bt_index_check(c.oid)
   FROM pg_index i
   JOIN pg_class c ON c.oid = i.indexrelid
   WHERE indisvalid;"

log "Verification PASSED"
pg_ctl stop -D "$RESTORE_DIR" -m fast
rm -rf "$RESTORE_DIR"
RPO vs RTO

RPO (Recovery Point Objective) — maximum acceptable data loss, controlled by archive_timeout and WAL archiving frequency. RTO (Recovery Time Objective) — how quickly the system must be back online, controlled by backup type, size, and infrastructure readiness.

Backup strategy matrix for different RPO/RTO targets

RPO = 0: Synchronous replication + WAL archiving. Requires synchronous standby in a different AZ.

RPO < 5 min: Asynchronous replication + continuous WAL archiving. Standard for most production systems.

RPO < 24 hours: Nightly pg_dump + WAL archiving. Acceptable for development/staging.

RTO < 1 min: Patroni with automatic failover. Standby always ready.

RTO < 1 hour: Base backup restore + WAL replay. Requires pre-staged infrastructure.

Self-Check

Before moving to Chapter 09, honestly answer these questions. If any answer is uncertain, go back and practice hands-on.

Can you explain the difference between logical and physical backups?

Logical backups (pg_dump) export SQL representations — portable across versions, allow selective restore, but slow for large databases. Physical backups (pg_basebackup) copy raw data files — faster restore, required for PITR, but tied to the same major version. Use both in production.

What happens if your archive_command fails repeatedly?

PostgreSQL retains unarchived WAL segments in pg_wal. If failures persist, the directory fills the disk and PostgreSQL halts writes (PANIC shutdown). Monitor pg_stat_archiver.failed_count and disk usage continuously.

Can you perform a PITR to recover from an accidental DROP TABLE?

Yes, if you have a base backup plus continuous WAL archives covering the time between the backup and just before the DROP. Restore the base backup, set recovery_target_time to the moment before, and use recovery_target_inclusive = false.

What is the difference between full, differential, and incremental backups?

Full copies all data. Differential copies pages changed since the last full. Incremental copies pages changed since the last backup of any type. Restoring incremental requires chaining: full + all incrementals. pgBackRest handles this automatically.

Have you actually tested your backup by restoring it?

If not, your backup is unverified. Set up automated nightly verification that restores to a separate instance and runs integrity checks. A backup you have never restored is just a file, not a guarantee.
