Chapter 09 · Phase 5: Production & Capstone

Monitoring and Performance Tuning

Instrument your PostgreSQL instance with pg_stat views for deep observability, identify and fix slow queries, tune autovacuum for table health, and configure memory parameters to match your workload. A database you cannot observe is a database you cannot manage.

Estimated: 4–5 days
6 Key Concepts
3 Assignments

Why This Matters

Performance problems are rarely sudden — they creep in over weeks or months as data grows, traffic patterns shift, and dead tuples accumulate. By the time a user complains about slowness, the root cause is often weeks old. The only way to catch performance degradation early is continuous monitoring with clear baselines and alerts.

PostgreSQL exposes an extraordinary amount of internal state through its pg_stat_* views. Learning to read these views is like learning to read an X-ray — it reveals problems that are invisible from the application layer. This chapter teaches you to build a monitoring dashboard, diagnose slow queries, and tune the database for your specific workload.

Without observability, every performance investigation becomes guesswork. With it, you can pinpoint the exact query, table, or configuration parameter responsible for degradation. The difference between a junior DBA and a senior DBA is not the ability to write faster SQL — it is the ability to diagnose where time is being lost and why.

Key Insight

The three pillars of PostgreSQL health are: query performance (pg_stat_statements), table health (pg_stat_user_tables + VACUUM), and resource utilization (memory, connections, disk I/O). Monitor all three, and most performance problems become visible before they become incidents.

Key Concepts

1. pg_stat_statements and Slow Query Detection

The pg_stat_statements extension is the single most important monitoring tool for PostgreSQL. It tracks execution statistics for every normalized query, including call count, total time, mean time, rows returned, and shared block hits/reads. Without it, you are completely blind to what your database is actually spending time on.

The extension normalizes queries by replacing literal values with parameter placeholders, so SELECT * FROM users WHERE id = 42 and SELECT * FROM users WHERE id = 99 are tracked as a single entry. This lets you see aggregate behavior across thousands of executions of the same query pattern.

sql 
-- Enable the extension (requires superuser, add to shared_preload_libraries)
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Top 10 queries by total execution time
SELECT
  left(query, 80) AS short_query,
  calls,
  round(total_exec_time::numeric, 2) AS total_ms,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  round((100 * shared_blks_hit /
    nullif(shared_blks_hit + shared_blks_read, 0))::numeric, 2)
    AS cache_hit_pct,
  rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
sql 
-- Queries with the worst cache hit ratio (potential index candidates)
SELECT
  left(query, 80) AS short_query,
  calls,
  shared_blks_read,
  shared_blks_hit,
  round((100 * shared_blks_hit /
    nullif(shared_blks_hit + shared_blks_read, 0))::numeric, 2)
    AS cache_hit_pct
FROM pg_stat_statements
WHERE calls > 100
ORDER BY shared_blks_read DESC
LIMIT 10;

-- Queries with the highest stddev (inconsistent performance)
SELECT
  left(query, 80) AS short_query,
  calls,
  round(mean_exec_time::numeric, 2) AS mean_ms,
  round(stddev_exec_time::numeric, 2) AS stddev_ms,
  round(min_exec_time::numeric, 2) AS min_ms,
  round(max_exec_time::numeric, 2) AS max_ms
FROM pg_stat_statements
WHERE calls > 50
ORDER BY stddev_exec_time DESC
LIMIT 10;

-- Reset statistics (do this periodically to get fresh data)
SELECT pg_stat_statements_reset();
shared_preload_libraries Required

pg_stat_statements must be added to shared_preload_libraries in postgresql.conf and requires a PostgreSQL restart. Simply running CREATE EXTENSION is not enough. Without the shared library loaded, the extension will track nothing.

2. pg_stat_user_tables and Table Health

The pg_stat_user_tables view reveals how each table is being used: sequential scan frequency, index scan frequency, dead tuple count, and when the last vacuum and analyze ran. It is the primary tool for identifying tables that need indexes and tables where autovacuum is falling behind.

A high ratio of sequential scans to index scans on a large table usually means a missing index. A large n_dead_tup value relative to n_live_tup means autovacuum is not keeping up. Stale last_autoanalyze timestamps mean the planner is making decisions on outdated statistics.

sql 
-- Tables with the most sequential scans (missing index candidates)
SELECT
  schemaname, relname,
  seq_scan, seq_tup_read,
  idx_scan, idx_tup_fetch,
  CASE WHEN seq_scan + idx_scan > 0
    THEN round((100 * idx_scan / (seq_scan + idx_scan))::numeric, 2)
    ELSE 0
  END AS idx_scan_pct,
  n_live_tup, n_dead_tup,
  last_vacuum, last_autovacuum,
  last_analyze, last_autoanalyze
FROM pg_stat_user_tables
ORDER BY seq_scan DESC
LIMIT 20;

-- Tables with the highest dead tuple ratio (need vacuum)
SELECT
  schemaname, relname,
  n_live_tup, n_dead_tup,
  round((100 * n_dead_tup /
    nullif(n_live_tup + n_dead_tup, 0))::numeric, 2)
    AS dead_pct,
  last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY n_dead_tup DESC;

-- Index usage report: which indexes are never used?
SELECT
  schemaname, relname, indexrelname,
  idx_scan AS times_used,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE '%pkey'
ORDER BY pg_relation_size(indexrelid) DESC;

3. VACUUM and Autovacuum Tuning

PostgreSQL uses MVCC (Multi-Version Concurrency Control), which means updates and deletes create dead tuples instead of modifying data in place. VACUUM reclaims space from dead tuples, updates the visibility map, and freezes old transaction IDs. Autovacuum does this automatically, but its default settings are conservative and often need tuning for high-write workloads.

The default autovacuum trigger formula is: threshold + scale_factor * n_live_tup. With defaults of threshold=50 and scale_factor=0.2, a table with 10 million rows will not trigger autovacuum until it accumulates 2,000,050 dead tuples. That is an enormous amount of bloat before cleanup begins.

sql 
-- Check current autovacuum settings
SHOW autovacuum_vacuum_threshold;        -- default: 50
SHOW autovacuum_vacuum_scale_factor;     -- default: 0.2 (20%)
SHOW autovacuum_analyze_threshold;       -- default: 50
SHOW autovacuum_analyze_scale_factor;    -- default: 0.1 (10%)
SHOW autovacuum_naptime;                 -- default: 1min
SHOW autovacuum_max_workers;             -- default: 3

-- Tune per-table for high-write tables
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor = 0.01,    -- 1% instead of 20%
  autovacuum_vacuum_threshold = 1000,       -- trigger earlier
  autovacuum_analyze_scale_factor = 0.005,  -- 0.5% for fresh stats
  autovacuum_vacuum_cost_delay = 2          -- faster vacuum (ms)
);

-- Global tuning for write-heavy workloads
ALTER SYSTEM SET autovacuum_max_workers = 6;
ALTER SYSTEM SET autovacuum_vacuum_cost_limit = 2000;
ALTER SYSTEM SET autovacuum_naptime = '15s';

-- Manual vacuum with verbose output for debugging
VACUUM (VERBOSE, ANALYZE) orders;
Transaction ID Wraparound

PostgreSQL uses 32-bit transaction IDs that wrap around at approximately 4 billion. If autovacuum cannot keep up with dead tuple cleanup, you risk a transaction ID wraparound emergency where PostgreSQL shuts down to prevent data corruption. Monitor age(datfrozenxid) from pg_database and ensure it stays well below 2 billion. Set up alerting when it exceeds 500 million.

sql 
-- Monitor transaction ID age per database
SELECT
  datname,
  age(datfrozenxid) AS xid_age,
  pg_size_pretty(pg_database_size(datname)) AS db_size,
  CASE
    WHEN age(datfrozenxid) > 1000000000 THEN 'CRITICAL'
    WHEN age(datfrozenxid) > 500000000  THEN 'WARNING'
    ELSE 'OK'
  END AS status
FROM pg_database
ORDER BY age(datfrozenxid) DESC;

-- Monitor transaction ID age per table
SELECT
  schemaname, relname,
  age(relfrozenxid) AS xid_age,
  pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_stat_user_tables
ORDER BY age(relfrozenxid) DESC
LIMIT 20;

4. Memory Configuration

PostgreSQL's memory parameters have the single largest impact on performance after proper indexing. The three most important settings are shared_buffers, work_mem, and effective_cache_size. Getting these wrong can mean the difference between a database that handles 10,000 transactions per second and one that struggles with 500.

shared_buffers is PostgreSQL's own buffer pool, typically set to 25% of total RAM. effective_cache_size tells the planner how much memory is available between PostgreSQL buffers and the OS page cache, typically 75% of RAM. work_mem is the per-operation memory for sorts and hash joins — it is allocated per sort, not per connection.

conf 
# postgresql.conf — Memory tuning for a server with 32GB RAM

# Shared buffer pool: 25% of total RAM
shared_buffers = 8GB

# Per-operation sort/hash memory (per connection!)
# Be careful: 100 connections * 256MB = 25GB potential usage
work_mem = 256MB

# Hint to planner about OS cache size: 75% of total RAM
effective_cache_size = 24GB

# Maintenance operations (VACUUM, CREATE INDEX)
maintenance_work_mem = 2GB

# WAL settings for write-heavy workloads
wal_buffers = 64MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB
min_wal_size = 1GB

# I/O concurrency for SSDs
effective_io_concurrency = 200
random_page_cost = 1.1

# Connection limits
max_connections = 200
Pro Tip

work_mem is allocated per sort/hash operation per connection, not per connection. A complex query with 5 sort operations uses 5x work_mem. Start conservative (64MB) and increase only for specific workloads. Use SET LOCAL work_mem = '512MB' for individual analytical queries that need more memory, rather than raising the global default.

sql 
-- Check current memory settings
SELECT name, setting, unit, short_desc
FROM pg_settings
WHERE name IN (
  'shared_buffers', 'work_mem', 'effective_cache_size',
  'maintenance_work_mem', 'wal_buffers', 'max_connections'
);

-- Check buffer cache hit ratio (should be > 99%)
SELECT
  round(
    100 * sum(blks_hit) /
    nullif(sum(blks_hit) + sum(blks_read), 0)
  , 2) AS cache_hit_ratio
FROM pg_stat_database
WHERE datname = current_database();

-- Per-table cache hit ratio
SELECT
  schemaname, relname,
  heap_blks_read, heap_blks_hit,
  round(
    100 * heap_blks_hit /
    nullif(heap_blks_hit + heap_blks_read, 0)
  , 2) AS cache_hit_pct
FROM pg_statio_user_tables
WHERE heap_blks_read + heap_blks_hit > 0
ORDER BY heap_blks_read DESC
LIMIT 20;

5. Connection and Lock Monitoring

Connection exhaustion and lock contention are two of the most common causes of production outages. Monitoring active connections, idle-in-transaction sessions, and lock waits gives you early warning before these become incidents. A single idle-in-transaction session can hold locks for hours, preventing autovacuum from cleaning up dead tuples on that table.

PostgreSQL's pg_stat_activity view shows every current connection: its state, the query it is running, how long it has been running, and what it is waiting on. Combined with pg_locks, you can trace exactly which connection is blocking which other connection, forming a complete dependency tree.

sql 
-- Connection state overview
SELECT
  state,
  count(*) AS connections,
  max(now() - state_change) AS longest_duration
FROM pg_stat_activity
WHERE pid != pg_backend_pid()
GROUP BY state
ORDER BY connections DESC;

-- Find long-running queries (potential problems)
SELECT
  pid, usename, datname, state,
  now() - query_start AS duration,
  wait_event_type, wait_event,
  left(query, 100) AS short_query
FROM pg_stat_activity
WHERE state = 'active'
  AND now() - query_start > INTERVAL '30 seconds'
ORDER BY duration DESC;

-- Kill idle-in-transaction sessions older than 1 hour
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND now() - state_change > INTERVAL '1 hour'
  AND pid != pg_backend_pid();
sql 
-- Find blocking locks (who is blocking whom)
SELECT
  blocked_locks.pid AS blocked_pid,
  blocked_activity.usename AS blocked_user,
  blocking_locks.pid AS blocking_pid,
  blocking_activity.usename AS blocking_user,
  now() - blocked_activity.query_start AS blocked_duration,
  left(blocked_activity.query, 60) AS blocked_query,
  left(blocking_activity.query, 60) AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity
  ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks
  ON blocking_locks.locktype = blocked_locks.locktype
  AND blocking_locks.relation = blocked_locks.relation
  AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity
  ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

-- Automatic idle-in-transaction timeout (PostgreSQL 9.6+)
ALTER SYSTEM SET idle_in_transaction_session_timeout = '5min';
SELECT pg_reload_conf();

6. Bloat Detection and Remediation

Table and index bloat occurs when VACUUM cannot reclaim space fast enough, or when it can only mark space as reusable without returning it to the operating system. Bloated tables waste disk space, slow down sequential scans, and degrade cache efficiency because every buffer page may contain a mix of live and dead data.

PostgreSQL's standard VACUUM marks dead space as reusable but does not shrink the table file. Only VACUUM FULL (which takes an exclusive lock) or tools like pg_repack (which works online) can actually reclaim disk space. For indexes, REINDEX CONCURRENTLY rebuilds without blocking writes.

sql 
-- Estimate table bloat using pgstattuple extension
CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT
  table_len,
  pg_size_pretty(table_len) AS table_size,
  tuple_count AS live_tuples,
  dead_tuple_count AS dead_tuples,
  round((100 * dead_tuple_len /
    nullif(table_len, 0))::numeric, 2)
    AS dead_pct,
  free_space,
  pg_size_pretty(free_space) AS free_space_pretty
FROM pgstattuple('orders');

-- Estimate index bloat
SELECT
  pg_size_pretty(avg_leaf_density * leaf_pages * 8192
    / 100) AS estimated_live,
  pg_size_pretty(leaf_pages * 8192) AS total_size,
  avg_leaf_density AS density_pct,
  leaf_fragmentation
FROM pgstatindex('idx_orders_created_at');
bash 
# Remediate bloat without locking using pg_repack
# Install: CREATE EXTENSION pg_repack; (in database)
# Run from command line:
pg_repack -d app_production -t orders

# Repack specific indexes
pg_repack -d app_production -i idx_orders_created_at

# For indexes, REINDEX CONCURRENTLY (PG 12+)
# Does not block writes, but uses extra disk space
sql 
-- REINDEX CONCURRENTLY for zero-downtime index rebuild (PG 12+)
REINDEX INDEX CONCURRENTLY idx_orders_created_at;

-- Rebuild all indexes on a table concurrently
REINDEX TABLE CONCURRENTLY orders;

-- Create a comprehensive monitoring summary view
CREATE OR REPLACE VIEW monitoring_summary AS
SELECT
  (SELECT count(*) FROM pg_stat_activity) AS total_connections,
  (SELECT count(*) FROM pg_stat_activity
   WHERE state = 'active') AS active_queries,
  (SELECT count(*) FROM pg_stat_activity
   WHERE state = 'idle in transaction') AS idle_in_txn,
  (SELECT sum(n_dead_tup) FROM pg_stat_user_tables) AS total_dead_tuples,
  (SELECT count(*) FROM pg_locks
   WHERE NOT granted) AS blocked_queries,
  pg_database_size(current_database()) AS db_size_bytes,
  pg_size_pretty(
    pg_database_size(current_database())
  ) AS db_size;
Why does VACUUM not return disk space to the OS?

Standard VACUUM marks dead tuple space as reusable within the table file, but it does not shrink the file itself. The file can only shrink if dead space is at the end of the file, which is rare. This is by design — truncating files is expensive and requires an exclusive lock. For actual space reclamation, use VACUUM FULL (exclusive lock, rewrites entire table) or pg_repack (online, no exclusive lock except briefly at the end).

How to detect if shared_buffers is too small

Check the buffer cache hit ratio: query pg_stat_database and compute blks_hit / (blks_hit + blks_read). If this is below 99%, PostgreSQL is frequently reading from disk instead of cache. Increase shared_buffers incrementally (requires a restart) and re-measure. Also check per-table hit ratios using pg_statio_user_tables to find which specific tables are causing cache misses.

Self-Check

Before moving to Chapter 10, honestly answer these questions. If any answer is uncertain, go back and experiment.

Can you find the top 5 slowest queries in your database right now?

If not, install pg_stat_statements immediately. It is the single most important observability tool for PostgreSQL. Without it, you are flying blind. Add it to shared_preload_libraries, restart PostgreSQL, and run CREATE EXTENSION pg_stat_statements.

Do you know the dead tuple ratio for your busiest table?

Query pg_stat_user_tables for n_dead_tup / (n_live_tup + n_dead_tup). If any table is over 10%, autovacuum is not keeping up and you need to tune its per-table settings or investigate long-running transactions that are preventing cleanup.

Have you tuned shared_buffers and work_mem for your workload?

The default settings (128MB shared_buffers, 4MB work_mem) are designed for a tiny development machine. On a production server with 32GB+ RAM, these defaults leave massive performance on the table. Use PGTune as a starting point, then benchmark and adjust.

Can you trace a lock contention chain from blocked query to blocker?

Combine pg_stat_activity with pg_locks to find which session holds the lock that another session is waiting for. In production, set idle_in_transaction_session_timeout to automatically terminate sessions that hold transactions open too long. Also consider lock_timeout to prevent queries from waiting indefinitely for locks.

Do you have a buffer cache hit ratio above 99%?

Check with: SELECT round(100 * sum(blks_hit) / nullif(sum(blks_hit) + sum(blks_read), 0), 2) FROM pg_stat_database WHERE datname = current_database(). If below 99%, you are reading too much from disk. Increase shared_buffers, add indexes to reduce the working set, or investigate which tables are causing the most cache misses using pg_statio_user_tables.
