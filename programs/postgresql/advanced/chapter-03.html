Chapter 03 · Phase 1: Data Architecture

Table Partitioning

Learn to split massive tables into manageable partitions using declarative partitioning, implement range, list, and hash strategies, manage partition lifecycle at scale, and unlock partition pruning for dramatic query speedups.

Estimated: 3–4 days
6 Key Concepts
3 Assignments

Why This Matters

When a table grows beyond a few hundred million rows, even well-indexed queries start to slow down. Vacuum operations take longer, index bloat accelerates, and bulk deletes become multi-hour full-table rewrites. Table partitioning is the foundational technique for managing data at scale in PostgreSQL. It solves problems that no amount of indexing or query tuning alone can address.

Partitioning divides a single logical table into multiple physical sub-tables. The query planner knows the boundaries of each partition and can skip entire partitions that cannot contain matching rows — a technique called partition pruning. For time-series data, this means a query filtering on WHERE created_at >= '2025-01-01' only touches the 2025 partitions, completely ignoring years of historical data.

Beyond query performance, partitioning enables critical operational wins: you can DROP an entire partition of old data in milliseconds instead of running a multi-hour DELETE, you can run VACUUM on individual partitions without locking the whole table, and you can place hot partitions on fast SSDs while archiving cold partitions to cheaper storage using tablespaces. These are not theoretical benefits — they are daily realities for any team operating PostgreSQL at scale.

Key Insight

Partitioning is not about making individual row lookups faster. It is about making entire segments of data invisible to queries that do not need them. Think of it as physically organizing a warehouse so each aisle only contains one category of goods. You never walk down the wrong aisle.

Key Concepts

1. Declarative Partitioning Fundamentals

PostgreSQL 10 introduced declarative partitioning, replacing the older inheritance-based method. You define a parent table with a PARTITION BY clause, then create child partitions that specify their boundary constraints. The parent table stores no data itself — all rows are routed to the appropriate child partition automatically during INSERT.

PostgreSQL supports three partitioning strategies: range (for continuous values like dates or numeric IDs), list (for discrete categorical values like region codes or status enums), and hash (for uniform distribution when no natural range or list exists). Your choice depends on your most common query patterns and data management requirements.

sql 
-- Create a range-partitioned table for time-series data
CREATE TABLE events (
    id          bigserial,
    event_type  text NOT NULL,
    payload     jsonb,
    created_at  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at);

-- Create monthly partitions
CREATE TABLE events_2025_01 PARTITION OF events
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE events_2025_02 PARTITION OF events
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

CREATE TABLE events_2025_03 PARTITION OF events
    FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');

-- Always create a default partition for safety
CREATE TABLE events_default PARTITION OF events DEFAULT;
Partition Key in Primary Key

The partition key column must be included in any unique index or primary key constraint. PostgreSQL enforces uniqueness per-partition, not globally. If your primary key is (id) alone, change it to (id, created_at) when partitioning by created_at.

2. Range Partitioning for Time-Series Data

Range partitioning is the most common strategy, ideal for data with natural ordering — timestamps, dates, numeric IDs, or version numbers. Each partition covers a contiguous range, and ranges must not overlap. The upper bound is exclusive following the half-open interval convention: [lower, upper).

For time-series workloads, the partition granularity should match your retention and query patterns. If you frequently query "last 7 days," daily partitions give the best pruning. If your retention is "keep 2 years, drop older," monthly partitions make drops fast and keep the partition count manageable. As a rule, keep the total number of partitions under a few hundred to avoid excessive planning overhead.

sql 
-- Automate partition creation with a DO block
CREATE TABLE orders (
    id           bigserial,
    customer_id  bigint NOT NULL,
    total        numeric(12,2) NOT NULL,
    status       text NOT NULL DEFAULT 'pending',
    ordered_at   timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (id, ordered_at)
) PARTITION BY RANGE (ordered_at);

DO $$
DECLARE
    start_date date := '2024-01-01';
    end_date   date := '2026-01-01';
    curr       date := start_date;
    part_name  text;
BEGIN
    WHILE curr < end_date LOOP
        part_name := 'orders_' || to_char(curr, 'YYYY_MM');
        EXECUTE format(
            'CREATE TABLE %I PARTITION OF orders
             FOR VALUES FROM (%L) TO (%L)',
            part_name, curr,
            curr + interval '1 month'
        );
        curr := curr + interval '1 month';
    END LOOP;
END $$;

-- Verify partition pruning
EXPLAIN (COSTS OFF)
SELECT * FROM orders
WHERE ordered_at >= '2025-06-01'
  AND ordered_at < '2025-07-01';
-- Should scan only orders_2025_06

  orders (parent — stores no data)
      |
      +-- orders_2024_01   [2024-01-01 .. 2024-02-01)
      +-- orders_2024_02   [2024-02-01 .. 2024-03-01)
      +-- ...
      +-- orders_2025_06   [2025-06-01 .. 2025-07-01)  <-- query lands here
      +-- ...
      +-- orders_2025_12   [2025-12-01 .. 2026-01-01)
      +-- orders_default   (catch-all for unmatched rows)
Figure 1: Range partitioning with monthly boundaries — planner prunes to a single partition

3. List Partitioning for Categorical Data

List partitioning is ideal when data is categorized by a discrete set of values — regions, countries, tenant IDs in multi-tenant systems, or status codes. Each partition holds rows matching specific listed values. This lets you isolate tenants for security, route regions to different storage tiers, or instantly drop all data for a specific category.

sql 
-- Multi-tenant table partitioned by region
CREATE TABLE customer_data (
    id          bigserial,
    tenant_id   integer NOT NULL,
    region      text NOT NULL,
    data        jsonb,
    created_at  timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (id, region)
) PARTITION BY LIST (region);

CREATE TABLE customer_data_us PARTITION OF customer_data
    FOR VALUES IN ('us-east-1', 'us-west-2');

CREATE TABLE customer_data_eu PARTITION OF customer_data
    FOR VALUES IN ('eu-west-1', 'eu-central-1');

CREATE TABLE customer_data_apac PARTITION OF customer_data
    FOR VALUES IN ('ap-southeast-1', 'ap-northeast-1');

CREATE TABLE customer_data_default
    PARTITION OF customer_data DEFAULT;

-- Query only touches the EU partition
EXPLAIN (COSTS OFF)
SELECT * FROM customer_data
WHERE region = 'eu-west-1';

-- GDPR compliance: drop all EU data instantly
ALTER TABLE customer_data DETACH PARTITION customer_data_eu;
DROP TABLE customer_data_eu;
Pro Tip

For multi-tenant SaaS systems, list partitioning by tenant_id provides data isolation at the physical level. Combined with Row-Level Security (RLS), this gives you two independent layers of tenant isolation. You can place high-value tenants on dedicated tablespaces backed by premium storage.

4. Hash Partitioning for Even Distribution

Hash partitioning distributes rows evenly across a fixed number of partitions using a hash function on the partition key. It is useful when there is no natural range or list boundary, and you simply need to break a large table into smaller, equally-sized pieces for operational benefits — parallelized vacuum, reduced lock contention, and improved I/O throughput.

The number of hash partitions should be a power of 2 for optimal distribution. Hash partitioning does not support partition pruning for range queries — only exact-match lookups on the partition key benefit. This makes it less suitable for analytical workloads but excellent for OLTP systems with high-concurrency point lookups.

sql 
-- Hash partitioning for session storage
CREATE TABLE sessions (
    id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id    bigint NOT NULL,
    token      text NOT NULL,
    data       jsonb,
    expires_at timestamptz NOT NULL
) PARTITION BY HASH (id);

-- Create 8 hash partitions (power of 2)
CREATE TABLE sessions_p0 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 0);
CREATE TABLE sessions_p1 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 1);
CREATE TABLE sessions_p2 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 2);
CREATE TABLE sessions_p3 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 3);
CREATE TABLE sessions_p4 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 4);
CREATE TABLE sessions_p5 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 5);
CREATE TABLE sessions_p6 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 6);
CREATE TABLE sessions_p7 PARTITION OF sessions
    FOR VALUES WITH (MODULUS 8, REMAINDER 7);

-- Check distribution across partitions
SELECT
    tableoid::regclass AS partition_name,
    count(*) AS row_count
FROM sessions
GROUP BY tableoid
ORDER BY partition_name;

5. Partition Pruning and Query Planning

Partition pruning is the mechanism by which the query planner eliminates partitions that cannot contain relevant rows. It operates at two stages: during planning time (when filter values are constants) and at execution time (when filter values come from parameters, subqueries, or prepared statements). Execution-time pruning was added in PostgreSQL 11 and is critical for prepared statement performance.

sql 
-- Ensure partition pruning is enabled
SHOW enable_partition_pruning;  -- should be 'on'

-- Planning-time pruning: constant filter value
EXPLAIN (ANALYZE, COSTS OFF)
SELECT * FROM events
WHERE created_at >= '2025-03-01'
  AND created_at < '2025-04-01';
-- Look for: "Partitions removed during planning: N"

-- Execution-time pruning: parameterized query
PREPARE get_events(timestamptz, timestamptz) AS
SELECT * FROM events
WHERE created_at >= $1 AND created_at < $2;

EXPLAIN (ANALYZE, COSTS OFF)
EXECUTE get_events('2025-03-01', '2025-04-01');
-- Look for: "Partitions removed during execution: N"

-- Aggregation with pruning
SELECT
    date_trunc('day', created_at) AS day,
    count(*) AS event_count,
    count(DISTINCT event_type) AS unique_types
FROM events
WHERE created_at >= '2025-03-01'
  AND created_at < '2025-04-01'
GROUP BY 1
ORDER BY 1;

-- Sub-partitioning: year at top, month within
CREATE TABLE metrics (
    id         bigserial,
    sensor_id  integer NOT NULL,
    value      double precision,
    ts         timestamptz NOT NULL,
    PRIMARY KEY (id, ts)
) PARTITION BY RANGE (ts);

-- Year-level partition that is itself partitioned by month
CREATE TABLE metrics_2025 PARTITION OF metrics
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01')
    PARTITION BY RANGE (ts);

CREATE TABLE metrics_2025_01 PARTITION OF metrics_2025
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
CREATE TABLE metrics_2025_02 PARTITION OF metrics_2025
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');
Warning

If your WHERE clause uses a function on the partition key (e.g., WHERE EXTRACT(MONTH FROM created_at) = 3), partition pruning will not work. The planner cannot infer which partitions to skip. Always use direct comparisons on the partition key column.

6. Partition Maintenance and Lifecycle

Production partitioned tables require ongoing maintenance: creating future partitions before data arrives, detaching and dropping old partitions for retention policies, and monitoring partition sizes for balance. Automating this lifecycle is essential — if a partition does not exist when a row arrives and there is no default partition, the insert will fail with an error.

sql 
-- Detach a partition non-blocking (PG 14+)
ALTER TABLE events
    DETACH PARTITION events_2024_01 CONCURRENTLY;

-- Archive to cheaper storage before dropping
ALTER TABLE events_2024_01
    SET TABLESPACE archive_hdd;

-- Or just drop for instant space reclaim
DROP TABLE events_2024_01;

-- Attach an existing table as a new partition
ALTER TABLE events
    ATTACH PARTITION events_2026_07
    FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');

-- Automated partition management function
CREATE OR REPLACE FUNCTION maintain_partitions(
    p_table       text,
    p_ahead_months integer DEFAULT 3,
    p_retention    interval DEFAULT interval '24 months'
) RETURNS void AS $$
DECLARE
    curr       date;
    target     date;
    part_name  text;
    cutoff     date;
    rec        record;
BEGIN
    -- Create future partitions
    curr := date_trunc('month', now())::date;
    target := (curr + (p_ahead_months || ' months')::interval)::date;
    WHILE curr < target LOOP
        part_name := p_table || '_' || to_char(curr, 'YYYY_MM');
        IF NOT EXISTS (
            SELECT 1 FROM pg_class WHERE relname = part_name
        ) THEN
            EXECUTE format(
                'CREATE TABLE %I PARTITION OF %I FOR VALUES FROM (%L) TO (%L)',
                part_name, p_table, curr, curr + interval '1 month'
            );
            RAISE NOTICE 'Created: %', part_name;
        END IF;
        curr := curr + interval '1 month';
    END LOOP;

    -- Drop old partitions beyond retention
    cutoff := (now() - p_retention)::date;
    FOR rec IN
        SELECT inhrelid::regclass::text AS pname
        FROM pg_inherits
        WHERE inhparent = p_table::regclass
    LOOP
        IF rec.pname ~ '_\d{4}_\d{2}$' THEN
            IF to_date(
                right(rec.pname, 7), 'YYYY_MM'
            ) < cutoff THEN
                EXECUTE format(
                    'ALTER TABLE %I DETACH PARTITION %s CONCURRENTLY',
                    p_table, rec.pname
                );
                EXECUTE 'DROP TABLE ' || rec.pname;
                RAISE NOTICE 'Dropped: %', rec.pname;
            END IF;
        END IF;
    END LOOP;
END $$ LANGUAGE plpgsql;

-- Run monthly via pg_cron or crontab
SELECT maintain_partitions('events', 3, interval '18 months');

-- Monitor partition sizes
SELECT
    inhrelid::regclass AS partition,
    pg_size_pretty(pg_total_relation_size(inhrelid)) AS total_size,
    (SELECT reltuples::bigint FROM pg_class WHERE oid = inhrelid) AS est_rows
FROM pg_inherits
WHERE inhparent = 'events'::regclass
ORDER BY partition;
What happens if a row does not match any partition and there is no default?

PostgreSQL raises: ERROR: no partition of relation "events" found for row. A default partition is recommended in most setups. However, if you intentionally want to reject data outside known ranges (e.g., refusing future-dated events), omitting the default partition acts as a data validation constraint.

Can you partition an existing table without downtime?

Not directly with a single DDL statement. You must create a new partitioned table, migrate data in batches using INSERT INTO new_table SELECT * FROM old_table WHERE ..., then rename the tables in a transaction. The pg_partman extension and tools like pg_repack can help. Plan for a brief lock during the final rename swap.

How many partitions is too many?

In PostgreSQL 14+, the planner uses binary search for partition matching, so hundreds of partitions are fine. However, exceeding a few thousand partitions can cause noticeable planning overhead. For daily partitions spanning many years, consider sub-partitioning (partition by year at the top level, then by month within each year).

Self-Check

Before moving to Chapter 04, honestly answer these questions. If any answer is uncertain, go back and experiment.

Can you explain the difference between range, list, and hash partitioning?

Range: for continuous values with natural ordering (dates, numbers). List: for discrete categories (regions, statuses). Hash: for even distribution when no natural boundary exists. Range and list support pruning for their respective query patterns; hash only prunes on exact-match lookups.

Do you know why the partition key must be part of the primary key?

PostgreSQL enforces uniqueness per-partition, not globally across all partitions. Without the partition key in the unique constraint, two different partitions could each contain a row with the same primary key value.

Can you verify partition pruning is working by reading EXPLAIN output?

Look for "Partitions removed during planning" or "Partitions removed during execution." If all partitions are scanned (Append node listing every partition), pruning is not working — check that your WHERE clause uses direct comparisons on the partition key.

When should you use DETACH PARTITION CONCURRENTLY vs regular DETACH?

Regular DETACH takes an ACCESS EXCLUSIVE lock, blocking all reads and writes. CONCURRENTLY (PG 14+) uses a weaker lock allowing concurrent queries. Always prefer CONCURRENTLY in production.
