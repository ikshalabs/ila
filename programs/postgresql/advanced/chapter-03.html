Chapter 03 · PostgreSQL Advanced

Table Partitioning

Implement range, list, and hash partitioning strategies for billion-row ad-tech tables. Master partition pruning, lifecycle automation, and zero-downtime migration.

Estimated: 4–5 hours
6 Key Concepts
3 Assignments

Why This Matters

The ad portal's ad_group_stats table grows by millions of rows every day. Without partitioning, queries for a single day's data must scan the entire table. Inserts slow down as indexes grow. Deleting old data requires scanning billions of rows. Table partitioning solves all three problems by splitting one logical table into many physical tables, each containing a slice of the data.

Key Insight

Partitioning is not just about performance — it transforms maintenance. Dropping a month of stats data goes from a 4-hour DELETE to a sub-second DROP TABLE. VACUUM runs on small partitions instead of a monolithic table. Backups can skip historical partitions that never change.

Key Concepts

1. Range Partitioning

Range partitioning splits data by continuous ranges of a column — typically dates or IDs. This is the most common strategy for time-series data. The ad portal partitions ad_group_stats by month so that each partition contains one month of performance data.

sql
-- Create the partitioned parent table
CREATE TABLE ad_group_stats (
  id            BIGSERIAL,
  dt            DATE NOT NULL,
  campaign_id   BIGINT NOT NULL,
  ad_group_id   BIGINT NOT NULL,
  impressions   INTEGER DEFAULT 0,
  clicks        INTEGER DEFAULT 0,
  spend         NUMERIC(20,2) DEFAULT 0,
  conversions   INTEGER DEFAULT 0,
  created_at    TIMESTAMP NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (dt);

-- Create monthly partitions
CREATE TABLE ad_group_stats_2024_01 PARTITION OF ad_group_stats
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');
CREATE TABLE ad_group_stats_2024_02 PARTITION OF ad_group_stats
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');
CREATE TABLE ad_group_stats_2024_03 PARTITION OF ad_group_stats
  FOR VALUES FROM ('2024-03-01') TO ('2024-04-01');

-- Create a default partition for data outside defined ranges
CREATE TABLE ad_group_stats_default PARTITION OF ad_group_stats
  DEFAULT;

-- Queries automatically target only relevant partitions
EXPLAIN ANALYZE
SELECT campaign_id, SUM(spend)
FROM ad_group_stats
WHERE dt BETWEEN '2024-02-01' AND '2024-02-29'
GROUP BY campaign_id;
-- Only scans ad_group_stats_2024_02

2. List Partitioning

List partitioning splits data by discrete values. This is ideal for the leads table, which can be partitioned by lead_type_id or by geographic region. Each partition contains all leads of a specific type or region.

sql
-- Partition leads by lead_type_id
CREATE TABLE leads (
  id              BIGSERIAL,
  account_id      BIGINT NOT NULL,
  campaign_id     BIGINT NOT NULL,
  lead_type_id    INTEGER NOT NULL,
  product_type_id INTEGER,
  first_name      TEXT,
  last_name       TEXT,
  email           TEXT,
  disposition     TEXT,
  created_at      TIMESTAMP NOT NULL DEFAULT NOW()
) PARTITION BY LIST (lead_type_id);

-- Each lead type gets its own partition
CREATE TABLE leads_auto_insurance PARTITION OF leads
  FOR VALUES IN (1);
CREATE TABLE leads_home_insurance PARTITION OF leads
  FOR VALUES IN (2);
CREATE TABLE leads_health_insurance PARTITION OF leads
  FOR VALUES IN (3);
CREATE TABLE leads_life_insurance PARTITION OF leads
  FOR VALUES IN (4);
CREATE TABLE leads_other PARTITION OF leads
  DEFAULT;

-- This query scans only the auto insurance partition
SELECT COUNT(*), disposition
FROM leads
WHERE lead_type_id = 1
GROUP BY disposition;

3. Hash Partitioning

Hash partitioning distributes data evenly across partitions using a hash function. This is useful when there is no natural range or list key, but you want to split a massive table for parallel scan performance. The ad portal uses hash partitioning for the account_balances table to spread load evenly.

sql
-- Hash partition account_balances by account_id
CREATE TABLE account_balances (
  id         BIGSERIAL,
  account_id BIGINT NOT NULL,
  balance    NUMERIC(20,2) NOT NULL,
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
) PARTITION BY HASH (account_id);

-- Create 4 hash partitions (power of 2 is recommended)
CREATE TABLE account_balances_p0 PARTITION OF account_balances
  FOR VALUES WITH (MODULUS 4, REMAINDER 0);
CREATE TABLE account_balances_p1 PARTITION OF account_balances
  FOR VALUES WITH (MODULUS 4, REMAINDER 1);
CREATE TABLE account_balances_p2 PARTITION OF account_balances
  FOR VALUES WITH (MODULUS 4, REMAINDER 2);
CREATE TABLE account_balances_p3 PARTITION OF account_balances
  FOR VALUES WITH (MODULUS 4, REMAINDER 3);

4. Partition Pruning and Query Optimization

Partition pruning is PostgreSQL's ability to skip partitions that cannot contain matching rows. It happens at plan time (static pruning) and at execution time (dynamic pruning). Understanding pruning is critical for getting the performance benefits of partitioning.

sql
-- Verify partition pruning is enabled
SHOW enable_partition_pruning; -- should be 'on'

-- Static pruning: planner knows which partitions to skip
EXPLAIN (COSTS OFF)
SELECT * FROM ad_group_stats
WHERE dt = '2024-02-15';
-- Shows: Seq Scan on ad_group_stats_2024_02
-- Other partitions are NOT scanned

-- Dynamic pruning: works with parameterized queries
PREPARE stats_query(date) AS
  SELECT campaign_id, SUM(spend)
  FROM ad_group_stats
  WHERE dt = $1
  GROUP BY campaign_id;

EXPLAIN ANALYZE EXECUTE stats_query('2024-03-10');
-- Shows: dynamic partition pruning at execution

-- Join pruning: even joins benefit from partitioning
SELECT c.name, SUM(s.spend) AS total_spend
FROM campaigns c
JOIN ad_group_stats s ON s.campaign_id = c.id
WHERE s.dt BETWEEN '2024-01-01' AND '2024-01-31'
GROUP BY c.name;
Partition Key in WHERE Clause

Always include the partition key in your WHERE clause. Without it, PostgreSQL must scan all partitions, which is often slower than a single unpartitioned table due to per-partition overhead. This is the number one mistake teams make after implementing partitioning.

5. Partition Lifecycle Management

Creating, detaching, and dropping partitions is a regular operational task. The ad portal automates this with a cron job that creates future partitions and archives old ones. Proper lifecycle management ensures no data falls into the default partition and old data is cleaned up efficiently.

sql
-- Create a new partition for next month (run monthly via cron)
CREATE TABLE ad_group_stats_2024_04 PARTITION OF ad_group_stats
  FOR VALUES FROM ('2024-04-01') TO ('2024-05-01');

-- Create indexes on the new partition
CREATE INDEX idx_stats_2024_04_campaign
  ON ad_group_stats_2024_04 (campaign_id);
CREATE INDEX idx_stats_2024_04_ad_group
  ON ad_group_stats_2024_04 (ad_group_id);

-- Detach an old partition (makes it a standalone table)
ALTER TABLE ad_group_stats
  DETACH PARTITION ad_group_stats_2023_01 CONCURRENTLY;

-- Archive: dump to S3 then drop
-- pg_dump -t ad_group_stats_2023_01 | gzip | aws s3 cp ...
DROP TABLE ad_group_stats_2023_01;

-- Automate partition creation with PL/pgSQL
CREATE OR REPLACE FUNCTION create_monthly_partition(
  parent_table text, target_month date
) RETURNS void AS $$
DECLARE
  part_name text;
  start_dt  date;
  end_dt    date;
BEGIN
  start_dt  := date_trunc('month', target_month);
  end_dt    := start_dt + interval '1 month';
  part_name := parent_table || '_' || to_char(start_dt, 'YYYY_MM');
  EXECUTE format(
    'CREATE TABLE IF NOT EXISTS %I PARTITION OF %I FOR VALUES FROM (%L) TO (%L)',
    part_name, parent_table, start_dt, end_dt
  );
END;
$$ LANGUAGE plpgsql;

6. Migrating to Partitioned Tables

Converting an existing unpartitioned table to a partitioned one requires careful planning. PostgreSQL does not support in-place conversion, so you must create a new partitioned table, migrate data, and swap. The ad portal migrated its 2-billion-row stats table with zero downtime using this strategy.

sql
-- Step 1: Create new partitioned table
CREATE TABLE ad_group_stats_new (
  LIKE ad_group_stats INCLUDING ALL
) PARTITION BY RANGE (dt);

-- Step 2: Create partitions for all needed date ranges
-- (use generate_series or the function above)

-- Step 3: Copy data in batches to avoid long locks
INSERT INTO ad_group_stats_new
SELECT * FROM ad_group_stats
WHERE dt BETWEEN '2024-01-01' AND '2024-01-31';

-- Step 4: Swap tables (brief exclusive lock)
BEGIN;
ALTER TABLE ad_group_stats RENAME TO ad_group_stats_old;
ALTER TABLE ad_group_stats_new RENAME TO ad_group_stats;
COMMIT;

-- Step 5: Backfill any rows inserted during migration
INSERT INTO ad_group_stats
SELECT * FROM ad_group_stats_old
WHERE id > (
  SELECT MAX(id) FROM ad_group_stats
);
Primary Key Constraint

The partition key must be part of any primary key or unique constraint on the table. You cannot have PRIMARY KEY (id) alone on a partitioned table — it must be PRIMARY KEY (id, dt). This is a PostgreSQL requirement that ensures uniqueness can be enforced per-partition.

Self-Check

Why must the partition key be included in the primary key?

PostgreSQL enforces unique constraints per-partition, not globally. Including the partition key in the primary key allows each partition to independently enforce uniqueness. Without it, PostgreSQL would need to check every partition for duplicates on every insert, which would negate the performance benefits of partitioning.

What happens if you insert a row that does not match any partition range?

If a DEFAULT partition exists, the row goes there. If no DEFAULT partition exists, PostgreSQL raises an error: “no partition of relation matches the given value.” Always create a DEFAULT partition as a safety net, then monitor it — rows in the default partition indicate missing partition definitions.

When is hash partitioning better than range or list partitioning?

Hash partitioning is best when there is no natural range or category to split on, but you want even data distribution for parallel query performance. It is also useful when you want to avoid hotspots — range partitioning on dates can create hot partitions for the current month. Hash partitioning distributes writes evenly across all partitions.
