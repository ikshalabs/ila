Chapter 04 · Phase 2: Analytics Architecture

Fact Tables and Analytics Schemas

Design data warehouse schemas that power dashboards and reports. Learn star schemas, fact and dimension tables, slowly changing dimensions, and materialized views that make analytical queries fast.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Transactional databases are optimized for fast reads and writes of individual rows — inserting an order, updating an account balance, fetching a user profile. But the moment someone asks "what were our sales by region last quarter?", the OLTP schema grinds to a halt. The joins are expensive, the aggregations scan millions of rows, and the report blocks live transactions.

This is why analytics engineering exists as a discipline. By separating analytical workloads into purpose-built schemas — star schemas with fact tables and dimension tables — you can serve business intelligence queries in seconds instead of minutes. PostgreSQL is perfectly capable of serving as both your OLTP and OLAP database when the schemas are designed correctly.

Understanding dimensional modeling is essential for any data engineer, analytics engineer, or backend developer who builds reporting systems. Every BI tool, every dashboard, and every data pipeline ultimately reads from schemas built on these principles.

Key Insight

The star schema is not old technology that has been replaced by NoSQL or data lakes. It is the foundation that modern tools like dbt, Looker, Metabase, and even Snowflake are built upon. Ralph Kimball's dimensional modeling techniques from the 1990s remain the gold standard because they solve a fundamental problem: humans think in dimensions (time, geography, product) and measures (revenue, count, average).

Key Concepts

1. Star Schema Fundamentals

A star schema organizes data into two types of tables: fact tables at the center that contain measurable events (sales, clicks, shipments) and dimension tables around them that describe the context of those events (who, what, when, where). The name "star" comes from the shape of the entity-relationship diagram — the fact table in the center with dimension tables radiating outward. 

                      +----------------+
                      |   dim_date     |
                      |----------------|
                      | date_key  (PK) |
                      | full_date      |
                      | year           |
                      | quarter        |
                      | month          |
                      | day_of_week    |
                      +-------+--------+
                              |
  +----------------+   +------+----------+   +------------------+
  |  dim_product   |   |  fact_sales     |   |   dim_customer   |
  |----------------|   |-----------------|   |------------------|
  | product_key(PK)|---| sale_id    (PK) |---| customer_key(PK) |
  | product_name   |   | date_key   (FK) |   | customer_name    |
  | category       |   | product_key(FK) |   | email            |
  | brand          |   | customer_key(FK)|   | segment          |
  | unit_cost      |   | store_key  (FK) |   | city             |
  +----------------+   | quantity        |   | state            |

                      | discount_pct    |
                      | total_amount    |
                      +-------+---------+

sql 
-- Create the analytics schema
CREATE SCHEMA analytics;
SET search_path TO analytics, public;

-- Dimension: Date
CREATE TABLE dim_date (
    date_key     integer PRIMARY KEY,  -- YYYYMMDD format
    full_date    date NOT NULL UNIQUE,
    year         smallint NOT NULL,
    quarter      smallint NOT NULL,
    month        smallint NOT NULL,
    month_name   text NOT NULL,
    day_of_week  smallint NOT NULL,
    day_name     text NOT NULL,
    is_weekend   boolean NOT NULL,
    fiscal_year  smallint NOT NULL
);

-- Dimension: Product
CREATE TABLE dim_product (
    product_key   serial PRIMARY KEY,
    product_id    integer NOT NULL,  -- natural key from OLTP
    product_name  text NOT NULL,
    category      text NOT NULL,
    subcategory   text,
    brand         text,
    unit_cost     numeric(10,2),
    effective_from date NOT NULL DEFAULT current_date,
    effective_to   date DEFAULT '9999-12-31',
    is_current    boolean DEFAULT true
);

-- Dimension: Customer
CREATE TABLE dim_customer (
    customer_key  serial PRIMARY KEY,
    customer_id   integer NOT NULL,
    customer_name text NOT NULL,
    email         text,
    segment       text NOT NULL,  -- 'Enterprise', 'SMB', 'Consumer'
    city          text,
    state         text,
    country       text DEFAULT 'US'
);

-- Dimension: Store
CREATE TABLE dim_store (
    store_key   serial PRIMARY KEY,
    store_id    integer NOT NULL UNIQUE,
    store_name  text NOT NULL,
    city        text NOT NULL,
    state       text NOT NULL,
    region      text NOT NULL
);

-- Fact: Sales
CREATE TABLE fact_sales (
    sale_id       bigserial PRIMARY KEY,
    date_key      integer NOT NULL REFERENCES dim_date(date_key),
    product_key   integer NOT NULL REFERENCES dim_product(product_key),
    customer_key  integer NOT NULL REFERENCES dim_customer(customer_key),
    store_key     integer NOT NULL REFERENCES dim_store(store_key),
    quantity      integer NOT NULL,
    unit_price    numeric(10,2) NOT NULL,
    discount_pct  numeric(5,2) DEFAULT 0,
    total_amount  numeric(12,2) NOT NULL
);

-- Create indexes for common join patterns
CREATE INDEX idx_fact_sales_date ON fact_sales(date_key);
CREATE INDEX idx_fact_sales_product ON fact_sales(product_key);
CREATE INDEX idx_fact_sales_customer ON fact_sales(customer_key);
CREATE INDEX idx_fact_sales_store ON fact_sales(store_key);

2. Fact Table Design Principles

Fact tables are the engine of your analytics schema. They contain the measures — the numbers you aggregate (sum, count, average). Each row represents a single event or transaction. Fact tables are typically the largest tables in the warehouse, often containing billions of rows.

  • Grain — The level of detail for each row. A sales fact might be at "one row per line item" or "one row per transaction." Always define the grain before designing the table.
  • Additive measures — Values that can be summed across all dimensions (revenue, quantity).
  • Semi-additive measures — Values that can be summed across some dimensions but not others (account balance can be summed across customers but not across time).
  • Non-additive measures — Values that cannot be summed at all (unit price, discount percentage). Compute them using averages or weighted averages.
sql 
-- Analytical queries become simple and fast with star schemas

-- Revenue by quarter and product category
SELECT
    d.year,
    d.quarter,
    p.category,
    sum(f.total_amount)   AS revenue,
    sum(f.quantity)        AS units_sold,
    round(avg(f.discount_pct), 2) AS avg_discount
FROM fact_sales f
JOIN dim_date d    ON d.date_key = f.date_key
JOIN dim_product p ON p.product_key = f.product_key
WHERE d.year = 2024
GROUP BY d.year, d.quarter, p.category
ORDER BY d.quarter, revenue DESC;

-- Top 10 customers by lifetime value
SELECT
    c.customer_name,
    c.segment,
    sum(f.total_amount) AS lifetime_value,
    count(f.sale_id)    AS total_orders,
    min(d.full_date)    AS first_purchase,
    max(d.full_date)    AS last_purchase
FROM fact_sales f
JOIN dim_customer c ON c.customer_key = f.customer_key
JOIN dim_date d     ON d.date_key = f.date_key
GROUP BY c.customer_key, c.customer_name, c.segment
ORDER BY lifetime_value DESC
LIMIT 10;

3. Dimension Table Best Practices

Dimension tables are typically wide (many columns) and short (thousands to millions of rows, rarely billions). They provide the descriptive context that makes fact table data meaningful. Good dimension design uses surrogate keys (auto-incrementing integers) as primary keys, preserving the natural business key as a separate indexed column.

sql 
-- Populate the date dimension for 10 years
INSERT INTO dim_date
SELECT
    to_char(d, 'YYYYMMDD')::integer  AS date_key,
    d                                AS full_date,
    extract(year FROM d)::smallint  AS year,
    extract(quarter FROM d)::smallint AS quarter,
    extract(month FROM d)::smallint  AS month,
    to_char(d, 'Month')              AS month_name,
    extract(dow FROM d)::smallint    AS day_of_week,
    to_char(d, 'Day')                AS day_name,
    extract(dow FROM d) IN (0, 6)   AS is_weekend,
    CASE
        WHEN extract(month FROM d) >= 7
        THEN extract(year FROM d)::smallint + 1
        ELSE extract(year FROM d)::smallint
    END                              AS fiscal_year
FROM generate_series(
    '2020-01-01'::date,
    '2029-12-31'::date,
    '1 day'::interval
) AS d;
Always Pre-populate dim_date

The date dimension should be pre-populated for the full range of dates your warehouse will ever need. Generating it dynamically on each query wastes computation. Pre-populate 10+ years of dates including fiscal year mappings, holidays, and business day flags. This is a one-time operation that makes every time-based query trivially fast.

4. Slowly Changing Dimensions (SCD)

Dimension data changes over time — a customer moves to a new city, a product's price changes, an employee transfers departments. Slowly Changing Dimensions (SCD) define strategies for handling these changes while preserving historical accuracy.

  • SCD Type 1 — Overwrite the old value. No history preserved. Simple but loses historical context.
  • SCD Type 2 — Add a new row with effective dates. Full history preserved. The most common approach for analytics.
  • SCD Type 3 — Add a "previous_value" column. Limited history (only one prior value).
sql 
-- SCD Type 2: Handle a product price change
-- Step 1: Close the current record
UPDATE dim_product
SET
    effective_to = current_date - 1,
    is_current = false
WHERE product_id = 42
  AND is_current = true;

-- Step 2: Insert the new version
INSERT INTO dim_product (
    product_id, product_name, category, subcategory,
    brand, unit_cost, effective_from, effective_to, is_current
)
VALUES (
    42, 'Wireless Keyboard Pro', 'Electronics', 'Input Devices',
    'TechBrand', 35.00, current_date, '9999-12-31', true
);

-- Query: What was the product's cost when each sale happened?
SELECT
    d.full_date,
    p.product_name,
    p.unit_cost  AS cost_at_time_of_sale,
    f.unit_price AS sold_price,
    f.total_amount
FROM fact_sales f
JOIN dim_product p ON p.product_key = f.product_key
JOIN dim_date d    ON d.date_key = f.date_key
WHERE p.product_id = 42
ORDER BY d.full_date;

5. Materialized Views for Analytics

Even with a well-designed star schema, aggregating millions of fact rows on every dashboard load is wasteful. Materialized views pre-compute common aggregations and store the results as physical tables, turning expensive scans into instant reads.

sql 
-- Pre-compute monthly revenue by category and region
CREATE MATERIALIZED VIEW mv_monthly_revenue AS
SELECT
    d.year,
    d.month,
    d.month_name,
    p.category,
    s.region,
    sum(f.total_amount)   AS revenue,
    sum(f.quantity)        AS units_sold,
    count(DISTINCT f.customer_key) AS unique_customers,
    round(avg(f.total_amount), 2) AS avg_order_value
FROM fact_sales f
JOIN dim_date d    ON d.date_key = f.date_key
JOIN dim_product p ON p.product_key = f.product_key
JOIN dim_store s   ON s.store_key = f.store_key
GROUP BY d.year, d.month, d.month_name, p.category, s.region
WITH DATA;

CREATE UNIQUE INDEX idx_mv_monthly_rev
ON mv_monthly_revenue (year, month, category, region);

-- Dashboard query: instant results
SELECT category, region,
    sum(revenue) AS total_revenue,
    sum(units_sold) AS total_units
FROM mv_monthly_revenue
WHERE year = 2024
GROUP BY category, region
ORDER BY total_revenue DESC;

-- Refresh the materialized view (run from cron or scheduler)
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_monthly_revenue;
Layered Materialized Views

Build a hierarchy of materialized views: a daily grain view refreshed every hour, a monthly grain view refreshed daily, and a yearly grain view refreshed weekly. Each layer reads from the one below it, reducing computation at every level. This is the same pattern that dbt uses with its "staging, intermediate, mart" model layers.

6. ETL Patterns for Populating Analytics Tables

Getting data from your OLTP tables into your analytics schema requires an ETL (Extract, Transform, Load) process. In PostgreSQL, you can do this entirely with SQL — no external tools needed for simple cases.

sql 
-- ETL: Load new sales from OLTP into the analytics fact table
INSERT INTO analytics.fact_sales (
    date_key, product_key, customer_key, store_key,
    quantity, unit_price, discount_pct, total_amount
)
SELECT
    to_char(o.order_date, 'YYYYMMDD')::integer,
    dp.product_key,
    dc.customer_key,
    ds.store_key,
    oi.quantity,
    oi.unit_price,
    COALESCE(oi.discount_pct, 0),
    oi.quantity * oi.unit_price * (1 - COALESCE(oi.discount_pct, 0) / 100)
FROM public.orders o
JOIN public.order_items oi ON oi.order_id = o.id
JOIN analytics.dim_product dp
    ON dp.product_id = oi.product_id AND dp.is_current = true
JOIN analytics.dim_customer dc
    ON dc.customer_id = o.customer_id
JOIN analytics.dim_store ds
    ON ds.store_id = o.store_id
WHERE o.order_date >= current_date - interval '1 day'
  AND o.status = 'completed'
  AND NOT EXISTS (
      SELECT 1 FROM analytics.fact_sales fs
      WHERE fs.date_key = to_char(o.order_date, 'YYYYMMDD')::integer
        AND fs.product_key = dp.product_key
        AND fs.customer_key = dc.customer_key
  );
Idempotent ETL

Always make your ETL processes idempotent — running them twice should produce the same result as running them once. Use NOT EXISTS checks, ON CONFLICT clauses, or date-range-based deletes followed by inserts. If your ETL fails halfway through, you should be able to safely re-run it without duplicating data.

Self-Check

Before moving to Chapter 05, verify your understanding of these dimensional modeling concepts.

What is the difference between a fact table and a dimension table?

Fact tables contain measurable events (sales, clicks, shipments) with numeric values that you aggregate (sum, average, count). Dimension tables contain descriptive attributes that provide context (who, what, when, where). Fact tables are typically tall and narrow; dimension tables are wide and short.

Why use surrogate keys instead of natural keys in dimension tables?

Surrogate keys (auto-incrementing integers) decouple the warehouse from the source system. If the OLTP system changes its key format, the warehouse is unaffected. Surrogate keys also enable SCD Type 2 — the same natural key can have multiple rows (one per historical version) with different surrogate keys.

When would you choose SCD Type 2 over Type 1?

Use SCD Type 2 when you need to preserve history and answer questions like "what was the product's category when this sale happened?" Use Type 1 when history is irrelevant and you only care about the current value (e.g., fixing a typo in a customer name).

What is the "grain" of a fact table and why does it matter?

The grain defines what a single row in the fact table represents — "one row per line item per order" or "one row per daily page view per user." Defining the grain before designing the table prevents ambiguity, ensures measures are additive, and makes it clear which dimensions apply. Mixing grains in one fact table leads to double-counting and incorrect aggregations.

How do materialized views improve analytics query performance?

Materialized views pre-compute and store the results of expensive aggregation queries. Instead of scanning millions of fact rows and joining dimension tables on every dashboard load, the query reads from a small, pre-aggregated table. The trade-off is data staleness — you must refresh the view periodically. Using REFRESH ... CONCURRENTLY avoids blocking reads during the refresh.
