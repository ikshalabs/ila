Chapter 05 · Phase 2: Server-Side Programming

Advanced PL/pgSQL

Move beyond basic SQL queries into the world of server-side programming with PL/pgSQL — PostgreSQL's native procedural language. Learn to write custom functions, stored procedures, cursors, exception handling, dynamic SQL, composite types, and table-returning functions that push logic closer to your data.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Every application developer writes SQL. But the most effective database engineers write PL/pgSQL — PostgreSQL's procedural language that lets you embed control flow, variables, loops, and exception handling directly inside the database. When you move complex business logic from your application layer into PL/pgSQL functions, you eliminate network round trips, reduce data transfer, and execute operations at the speed of the database engine itself.

Consider a scenario where you need to calculate tiered pricing for 100,000 orders. In application code, you would fetch all orders, loop through them, apply pricing rules, and write the results back. That is 100,000 rows transferred over the network, processed in a slower language, and written back with 100,000 UPDATE statements. In PL/pgSQL, the same operation runs inside PostgreSQL with zero network overhead and direct access to indexes and query optimization.

PL/pgSQL is also the foundation for triggers, which you will learn in the next chapter. Before you can write a trigger function that automatically updates audit logs or enforces complex constraints, you need to master the procedural building blocks: variable declaration, control structures, cursors for row-by-row processing, and structured exception handling. This chapter gives you those building blocks.

Key Insight

PL/pgSQL functions are pre-compiled and cached by PostgreSQL. The first time a function is called in a session, the query plans for its SQL statements are prepared. Subsequent calls reuse those plans, making PL/pgSQL functions significantly faster than sending individual SQL statements from your application. This is one of the key performance advantages of server-side programming.

Key Concepts

1. Custom Functions: Structure and Syntax

A PL/pgSQL function is a named block of procedural code stored in the database catalog. Every function has a name, optional parameters, a return type, and a body enclosed in a $$ dollar-quoted string. Understanding the anatomy of a function is the first step to writing effective server-side logic.

sql 
-- Basic function: calculate discounted price
CREATE OR REPLACE FUNCTION calculate_discount(
    p_price    NUMERIC,
    p_discount NUMERIC DEFAULT 0.10
)
RETURNS NUMERIC
LANGUAGE plpgsql
IMMUTABLE           -- same inputs always produce same output
STRICT              -- returns NULL if any input is NULL
AS $$
DECLARE
    v_discounted NUMERIC;
BEGIN
    -- Calculate the discounted price
    v_discounted := p_price * (1 - p_discount);

    -- Ensure price never goes below zero
    IF v_discounted < 0 THEN
        v_discounted := 0;
    END IF;

    RETURN ROUND(v_discounted, 2);
END;
$$;

-- Usage
SELECT calculate_discount(100.00, 0.15);  -- 85.00
SELECT calculate_discount(49.99);          -- 44.99 (default 10%)
Volatility Categories Matter

Mark your functions with the correct volatility: IMMUTABLE (same inputs, same output, always), STABLE (same output within a single query), or VOLATILE (output can change anytime, the default). PostgreSQL uses this to optimize query plans. An incorrectly marked function can produce wrong results or prevent index usage.

2. Stored Procedures and Transaction Control

PostgreSQL 11 introduced PROCEDURE as distinct from FUNCTION. The critical difference: procedures can manage transactions using COMMIT and ROLLBACK inside the procedure body. Functions cannot. This makes procedures the right choice for long-running batch operations where you want to commit work in chunks rather than risking a single massive transaction.

sql 
-- Stored procedure with transaction control
CREATE OR REPLACE PROCEDURE archive_old_orders(
    p_cutoff_date DATE,
    p_batch_size  INT DEFAULT 1000
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_archived  INT := 0;
    v_batch     INT;
    v_total     INT;
BEGIN
    -- Count total records to archive
    SELECT COUNT(*) INTO v_total
    FROM orders
    WHERE created_at < p_cutoff_date
      AND status = 'completed';

    RAISE NOTICE 'Archiving % orders older than %', v_total, p_cutoff_date;

    LOOP
        -- Move a batch to the archive table
        WITH batch AS (
            SELECT id FROM orders
            WHERE created_at < p_cutoff_date
              AND status = 'completed'
            LIMIT p_batch_size
            FOR UPDATE SKIP LOCKED
        )
        INSERT INTO orders_archive
        SELECT o.* FROM orders o
        JOIN batch b ON o.id = b.id;

        GET DIAGNOSTICS v_batch = ROW_COUNT;

        -- Delete the moved records
        DELETE FROM orders
        WHERE id IN (
            SELECT id FROM orders_archive
            ORDER BY id DESC
            LIMIT v_batch
        );

        v_archived := v_archived + v_batch;

        -- Commit each batch independently
        COMMIT;

        RAISE NOTICE 'Archived % / % orders', v_archived, v_total;

        -- Exit when no more rows to process
        EXIT WHEN v_batch = 0;
    END LOOP;
END;
$$;

-- Call the procedure
CALL archive_old_orders('2024-01-01', 500);

3. Cursors: Row-by-Row Processing

Cursors let you process query results one row at a time instead of loading the entire result set into memory. While set-based operations should always be your first choice (they are faster), cursors are necessary when you need to perform complex row-level logic that cannot be expressed in a single SQL statement, or when the result set is too large to hold in memory.

sql 
-- Function using a cursor for complex row processing
CREATE OR REPLACE FUNCTION apply_tiered_pricing()
RETURNS INT
LANGUAGE plpgsql
AS $$
DECLARE
    order_cursor CURSOR FOR
        SELECT id, quantity, unit_price
        FROM order_items
        WHERE final_price IS NULL;
    v_record     RECORD;
    v_discount   NUMERIC;
    v_updated    INT := 0;
BEGIN
    OPEN order_cursor;

    LOOP
        FETCH order_cursor INTO v_record;
        EXIT WHEN NOT FOUND;

        -- Tiered discount logic
        v_discount := CASE
            WHEN v_record.quantity >= 100 THEN 0.20
            WHEN v_record.quantity >= 50  THEN 0.15
            WHEN v_record.quantity >= 10  THEN 0.10
            ELSE 0.00
        END;

        UPDATE order_items
        SET final_price = (unit_price * quantity) * (1 - v_discount)
        WHERE id = v_record.id;

        v_updated := v_updated + 1;
    END LOOP;

    CLOSE order_cursor;
    RETURN v_updated;
END;
$$;

4. Exception Handling and Error Management

Production PL/pgSQL code must handle errors gracefully. The EXCEPTION block catches runtime errors and lets you log them, return meaningful error messages, or attempt recovery. Without exception handling, a single error inside a function aborts the entire transaction, which can be catastrophic in long-running batch operations.

sql 
-- Function with structured exception handling
CREATE OR REPLACE FUNCTION safe_transfer_funds(
    p_from_account INT,
    p_to_account   INT,
    p_amount       NUMERIC
)
RETURNS TABLE(success BOOLEAN, message TEXT)
LANGUAGE plpgsql
AS $$
DECLARE
    v_balance NUMERIC;
BEGIN
    -- Validate input
    IF p_amount <= 0 THEN
        RETURN QUERY SELECT false, 'Amount must be positive';
        RETURN;
    END IF;

    -- Lock and check balance
    SELECT balance INTO v_balance
    FROM accounts
    WHERE id = p_from_account
    FOR UPDATE;

    IF NOT FOUND THEN
        RETURN QUERY SELECT false, 'Source account not found';
        RETURN;
    END IF;

    IF v_balance < p_amount THEN
        RETURN QUERY SELECT false, 'Insufficient funds';
        RETURN;
    END IF;

    -- Perform the transfer
    UPDATE accounts SET balance = balance - p_amount
    WHERE id = p_from_account;

    UPDATE accounts SET balance = balance + p_amount
    WHERE id = p_to_account;

    RETURN QUERY SELECT true, 'Transfer completed successfully';

EXCEPTION
    WHEN foreign_key_violation THEN
        RETURN QUERY SELECT false, 'Destination account not found';
    WHEN numeric_value_out_of_range THEN
        RETURN QUERY SELECT false, 'Amount exceeds allowed range';
    WHEN OTHERS THEN
        RETURN QUERY SELECT false,
            'Unexpected error: ' || SQLERRM;
END;
$$;

5. Dynamic SQL with EXECUTE

Sometimes you need to build SQL statements at runtime — the table name, column name, or WHERE clause is not known until the function is called. PL/pgSQL's EXECUTE statement runs dynamically constructed SQL strings. This is powerful but requires careful handling to prevent SQL injection. Always use format() with %I for identifiers and %L for literals.

sql 
-- Dynamic SQL: flexible search across any table
CREATE OR REPLACE FUNCTION count_rows_where(
    p_table   TEXT,
    p_column  TEXT,
    p_value   TEXT
)
RETURNS BIGINT
LANGUAGE plpgsql
AS $$
DECLARE
    v_count BIGINT;
    v_sql   TEXT;
BEGIN
    -- Build safe SQL using format() with %I (identifier) and %L (literal)
    v_sql := format(
        'SELECT COUNT(*) FROM %I WHERE %I = %L',
        p_table,
        p_column,
        p_value
    );

    RAISE NOTICE 'Executing: %', v_sql;

    -- Execute the dynamic query and capture the result
    EXECUTE v_sql INTO v_count;

    RETURN v_count;
END;
$$;

-- Usage: count users in a specific city
SELECT count_rows_where('users', 'city', 'New York');

-- Dynamic table partitioning utility
CREATE OR REPLACE FUNCTION create_monthly_partition(
    p_parent_table TEXT,
    p_year         INT,
    p_month        INT
)
RETURNS VOID
LANGUAGE plpgsql
AS $$
DECLARE
    v_partition_name TEXT;
    v_start_date     DATE;
    v_end_date       DATE;
BEGIN
    v_partition_name := format('%s_y%sm%s',
        p_parent_table, p_year, lpad(p_month::text, 2, '0'));
    v_start_date := make_date(p_year, p_month, 1);
    v_end_date   := v_start_date + INTERVAL '1 month';

    EXECUTE format(
        'CREATE TABLE IF NOT EXISTS %I PARTITION OF %I
         FOR VALUES FROM (%L) TO (%L)',
        v_partition_name, p_parent_table,
        v_start_date, v_end_date
    );

    RAISE NOTICE 'Created partition: %', v_partition_name;
END;
$$;
SQL Injection in Dynamic SQL

Never concatenate user input directly into dynamic SQL strings. Always use format('%I', identifier) for table and column names and format('%L', value) for literal values. Alternatively, use parameterized EXECUTE ... USING for value parameters. A single unescaped input can let an attacker drop your tables.

6. Composite Types and Table Functions

PL/pgSQL functions can return complex structures beyond simple scalar values. You can define custom composite types to return structured data, or use RETURNS TABLE to create functions that behave like virtual tables — you can SELECT from them, join them, and filter their output. This is incredibly powerful for encapsulating complex reporting queries.

sql 
-- Custom composite type
CREATE TYPE sales_summary AS (
    product_name  TEXT,
    total_sold    BIGINT,
    total_revenue NUMERIC,
    avg_price     NUMERIC,
    rank          INT
);

-- Table-returning function using the composite type
CREATE OR REPLACE FUNCTION get_sales_summary(
    p_start_date DATE,
    p_end_date   DATE,
    p_limit      INT DEFAULT 10
)
RETURNS SETOF sales_summary
LANGUAGE plpgsql
STABLE
AS $$
BEGIN
    RETURN QUERY
        SELECT
            p.name,
            COUNT(oi.id),
            SUM(oi.quantity * oi.unit_price),
            ROUND(AVG(oi.unit_price), 2),
            ROW_NUMBER() OVER (
                ORDER BY SUM(oi.quantity * oi.unit_price) DESC
            )::INT
        FROM order_items oi
        JOIN products p ON p.id = oi.product_id
        JOIN orders o ON o.id = oi.order_id
        WHERE o.created_at BETWEEN p_start_date AND p_end_date
        GROUP BY p.name
        ORDER BY 3 DESC
        LIMIT p_limit;
END;
$$;

-- Use it like a table
SELECT * FROM get_sales_summary('2025-01-01', '2025-12-31');
SELECT product_name, total_revenue
FROM get_sales_summary('2025-01-01', '2025-06-30', 5)
WHERE total_sold > 100;
Pro Tip

Table-returning functions are a powerful alternative to views when you need parameterized queries. Unlike views, which cannot accept parameters, a RETURNS TABLE function lets callers pass date ranges, limits, and filter criteria. They compose well with other SQL clauses — you can JOIN, WHERE, and ORDER BY on their output.

Self-Check

Before moving to Chapter 06, honestly answer these questions. If any answer is "no", go back and practice more.

Can you write a PL/pgSQL function with parameters, variables, and control flow from memory?

Write a function that accepts an employee ID and returns their bonus amount based on their department and years of service, using IF/ELSIF logic. If you need to look up the syntax for DECLARE, BEGIN, or RETURN, spend more time practicing the basics before moving forward.

Do you understand the difference between FUNCTION and PROCEDURE?

Functions return a value and cannot control transactions. Procedures can use COMMIT and ROLLBACK inside their body, making them suitable for batch operations. Functions can be used in SELECT statements; procedures are called with CALL. If this distinction is not clear, review the stored procedures section.

Can you use cursors for row-by-row processing when set-based operations are insufficient?

Cursors should be your last resort — set-based SQL is almost always faster. But when you genuinely need row-level logic (e.g., complex conditional updates that depend on the previous row), you need to declare a cursor, loop through it with FETCH, and close it properly. Practice the OPEN-FETCH-CLOSE pattern until it is second nature.

Can you write dynamic SQL safely using format() with %I and %L?

Dynamic SQL is powerful but dangerous. You should be able to construct a safe dynamic query that accepts a table name, column name, and filter value as parameters. Use %I for identifiers (table and column names) and %L for literal values. Never use string concatenation with user input.

Can you create a table-returning function and use it in a FROM clause?

Write a function that returns a set of rows using RETURNS TABLE and RETURN QUERY. Then use it in a SELECT statement with WHERE and ORDER BY clauses. This pattern is essential for parameterized views and reusable reporting functions in PostgreSQL.
