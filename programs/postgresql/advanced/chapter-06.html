Chapter 06 · Phase 3: Programmability (06/10)

Triggers and Event Systems

Move business logic closer to the data with advanced triggers, LISTEN/NOTIFY for real-time event propagation, audit logging that never misses a change, and row-level security policies that enforce access at the database layer itself.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Application-layer business logic is where most teams put their event handling — but it is also where most bugs hide. A background worker can crash silently, a microservice can miss a webhook, and an ORM can bypass validation rules with a raw UPDATE. When you push critical event logic into PostgreSQL itself, you get guarantees that application code alone cannot provide.

Triggers fire within the same transaction as the data change. LISTEN/NOTIFY delivers events without polling. Row-level security enforces access rules even when someone connects directly with psql. These are not replacements for application logic — they are the last line of defense that ensures your invariants hold no matter how the data is accessed.

Key Insight

The best event systems are defense-in-depth. Application code handles the happy path, triggers enforce invariants, audit logs capture everything, and row-level security prevents unauthorized access — even from your own engineers running ad-hoc queries.

Key Concepts

1. Advanced Trigger Patterns

PostgreSQL triggers can fire BEFORE, AFTER, or INSTEAD OF a data modification. They can operate at the row level or statement level, and they can modify the incoming data, reject the operation, or cascade changes to other tables. Understanding when to use each pattern is the difference between a clean system and a debugging nightmare.

sql 
-- BEFORE trigger: Validate and transform data before it hits the table
CREATE OR REPLACE FUNCTION validate_order_amount()
RETURNS TRIGGER AS $$
BEGIN
  -- Reject negative amounts at the database level
  IF NEW.amount < 0 THEN
    RAISE EXCEPTION 'Order amount cannot be negative: %', NEW.amount;
  END IF;

  -- Auto-set the updated_at timestamp
  NEW.updated_at := now();

  -- Normalize the currency code to uppercase
  NEW.currency := upper(NEW.currency);

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_validate_order
  BEFORE INSERT OR UPDATE ON orders
  FOR EACH ROW
  EXECUTE FUNCTION validate_order_amount();

The BEFORE trigger above does three things in a single function: validation, timestamp management, and data normalization. Because it runs before the row is written, it can modify NEW directly or abort the transaction with RAISE EXCEPTION.

sql 
-- AFTER trigger: Cascade side effects after the row is committed
CREATE OR REPLACE FUNCTION update_account_balance()
RETURNS TRIGGER AS $$
BEGIN
  UPDATE accounts
  SET balance = balance + NEW.amount,
      last_transaction_at = now()
  WHERE id = NEW.account_id;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_update_balance
  AFTER INSERT ON transactions
  FOR EACH ROW
  EXECUTE FUNCTION update_account_balance();
Trigger Ordering

When multiple triggers exist on the same table and event, PostgreSQL fires them in alphabetical order by name. Name your triggers deliberately — trg_01_validate, trg_02_audit, trg_03_notify — to control execution sequence.

2. LISTEN/NOTIFY for Real-Time Events

PostgreSQL has a built-in pub/sub system that lets database clients receive notifications without polling. NOTIFY sends a message on a named channel; any client that has issued LISTEN on that channel receives it immediately. Combined with triggers, this creates a reactive event pipeline entirely within the database.

sql 
-- Trigger function that sends a NOTIFY with the changed row as JSON
CREATE OR REPLACE FUNCTION notify_order_change()
RETURNS TRIGGER AS $$
DECLARE
  payload TEXT;
BEGIN
  payload := json_build_object(
    'operation', TG_OP,
    'table',     TG_TABLE_NAME,
    'id',        COALESCE(NEW.id, OLD.id),
    'data',      row_to_json(COALESCE(NEW, OLD)),
    'timestamp', now()
  )::TEXT;

  PERFORM pg_notify('order_changes', payload);

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_notify_orders
  AFTER INSERT OR UPDATE OR DELETE ON orders
  FOR EACH ROW
  EXECUTE FUNCTION notify_order_change();
sql 
-- Client-side: listen for events on the channel
LISTEN order_changes;

-- When an order is inserted/updated/deleted, this client
-- receives the JSON payload without polling.
-- In application code, use your driver's async notification API.
Pro Tip

The NOTIFY payload is limited to 8000 bytes. For large payloads, send only the primary key and operation type, then have the listener fetch the full row. This also avoids sending stale data if multiple updates happen in quick succession.

3. Event-Driven Workflows

Combining triggers with LISTEN/NOTIFY creates powerful event-driven workflows. A common pattern is to maintain an outbox table that records events transactionally, then process them asynchronously. This is known as the transactional outbox pattern.

sql 
-- Create the outbox table for reliable event delivery
CREATE TABLE event_outbox (
  id           BIGSERIAL PRIMARY KEY,
  event_type   TEXT NOT NULL,
  aggregate    TEXT NOT NULL,
  aggregate_id BIGINT NOT NULL,
  payload      JSONB NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  processed_at TIMESTAMPTZ
);

CREATE INDEX idx_outbox_unprocessed
  ON event_outbox (created_at)
  WHERE processed_at IS NULL;

-- Trigger: write to outbox whenever an order changes status
CREATE OR REPLACE FUNCTION outbox_order_status()
RETURNS TRIGGER AS $$
BEGIN
  IF OLD.status IS DISTINCT FROM NEW.status THEN
    INSERT INTO event_outbox (event_type, aggregate, aggregate_id, payload)
    VALUES (
      'order.status_changed',
      'orders',
      NEW.id,
      jsonb_build_object(
        'old_status', OLD.status,
        'new_status', NEW.status,
        'changed_by', current_user
      )
    );

    PERFORM pg_notify('outbox', NEW.id::TEXT);
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_outbox_order_status
  AFTER UPDATE ON orders
  FOR EACH ROW
  EXECUTE FUNCTION outbox_order_status();

4. Audit Logging with Triggers

Every production database needs an audit trail. A generic audit trigger captures every change to every tracked table, recording who changed what, when, and from what value. This is essential for compliance (SOC 2, HIPAA, GDPR) and invaluable for debugging.

sql 
-- Generic audit log table
CREATE TABLE audit_log (
  id             BIGSERIAL PRIMARY KEY,
  table_name     TEXT NOT NULL,
  operation      TEXT NOT NULL,
  row_id         TEXT,
  old_data       JSONB,
  new_data       JSONB,
  changed_fields TEXT[],
  db_user        TEXT NOT NULL DEFAULT current_user,
  app_user       TEXT,
  ip_address     INET,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_audit_table_time ON audit_log (table_name, created_at);
CREATE INDEX idx_audit_row ON audit_log (table_name, row_id);

-- Generic audit function that works with any table
CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS TRIGGER AS $$
DECLARE
  old_json  JSONB;
  new_json  JSONB;
  diff_keys TEXT[];
  row_pk    TEXT;
BEGIN
  IF TG_OP = 'DELETE' THEN
    old_json := to_jsonb(OLD);
    row_pk   := OLD.id::TEXT;
  ELSIF TG_OP = 'INSERT' THEN
    new_json := to_jsonb(NEW);
    row_pk   := NEW.id::TEXT;
  ELSE
    old_json := to_jsonb(OLD);
    new_json := to_jsonb(NEW);
    row_pk   := NEW.id::TEXT;

    -- Calculate which fields actually changed
    SELECT array_agg(key) INTO diff_keys
    FROM jsonb_each(new_json) AS n(key, value)
    WHERE n.value IS DISTINCT FROM old_json -> n.key;
  END IF;

  INSERT INTO audit_log
    (table_name, operation, row_id, old_data, new_data, changed_fields, app_user)
  VALUES (
    TG_TABLE_NAME, TG_OP, row_pk,
    old_json, new_json, diff_keys,
    current_setting('app.current_user', TRUE)
  );

  RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

-- Apply the audit trigger to any table
CREATE TRIGGER trg_audit_orders
  AFTER INSERT OR UPDATE OR DELETE ON orders
  FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();

CREATE TRIGGER trg_audit_users
  AFTER INSERT OR UPDATE OR DELETE ON users
  FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();
Pro Tip

Pass the application-level user through SET LOCAL app.current_user = 'jane@acme.com' at the beginning of each request. The audit trigger reads it with current_setting('app.current_user', TRUE). This bridges the gap between the database user (usually app_production) and the actual human who initiated the change.

5. Row-Level Security Policies

Row-Level Security (RLS) lets you define policies that control which rows each database role can see and modify. Once enabled on a table, every query is filtered through these policies — including direct psql sessions, data exports, and ORM queries.

sql 
-- Enable RLS on a multi-tenant table
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

-- Policy: users can only see projects belonging to their organization
CREATE POLICY org_isolation ON projects
  FOR ALL
  USING (
    org_id = current_setting('app.current_org_id')::INTEGER
  );

-- Policy: admins can see all rows
CREATE POLICY admin_full_access ON projects
  FOR ALL
  TO admin_role
  USING (TRUE);

-- Policy: read-only analysts can SELECT but not modify
CREATE POLICY analyst_read_only ON projects
  FOR SELECT
  TO analyst_role
  USING (
    org_id = current_setting('app.current_org_id')::INTEGER
  );

-- Force RLS even for table owner
ALTER TABLE projects FORCE ROW LEVEL SECURITY;
Critical Warning

RLS does not apply to table owners or superusers by default. If your application connects as the table owner, you must set FORCE ROW LEVEL SECURITY on the table or connect as a separate, less-privileged role. This is the most common RLS misconfiguration in production systems.

6. Change Data Capture

Change Data Capture (CDC) is the practice of identifying and capturing changes made to a database so that downstream systems can react. PostgreSQL supports CDC through logical decoding, which reads the Write-Ahead Log (WAL) and converts row changes into a consumable stream.

sql 
-- Enable logical replication in postgresql.conf
-- wal_level = logical

-- Create a replication slot for CDC
SELECT * FROM pg_create_logical_replication_slot(
  'cdc_slot',
  'pgoutput'
);

-- Create a publication for specific tables
CREATE PUBLICATION cdc_publication
  FOR TABLE orders, users, transactions;

-- Read changes from the slot (usually done by Debezium or similar)
SELECT * FROM pg_logical_slot_peek_changes(
  'cdc_slot', NULL, 10
);

-- Monitor replication slot status
SELECT slot_name, plugin, slot_type,
       pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS lag_bytes
FROM pg_replication_slots;

When to use triggers vs. CDC

Triggers are best for same-transaction logic: validation, denormalization, audit logging, and NOTIFY events. They add latency to the write path because they execute synchronously within the transaction.

CDC is best for asynchronous, external consumption: feeding a search index, syncing a cache, streaming to Kafka, or replicating to a data warehouse. It does not affect write latency since it reads from the WAL after the transaction commits.

Use both when you need both: triggers for immediate invariants, CDC for eventual consistency with external systems.

Conditional triggers with WHEN clauses

You can add a WHEN clause to a trigger definition to avoid invoking the function for every row. This is more efficient than checking the condition inside the function:

sql 
-- Only fire when status actually changes
CREATE TRIGGER trg_status_change
  AFTER UPDATE ON orders
  FOR EACH ROW
  WHEN (OLD.status IS DISTINCT FROM NEW.status)
  EXECUTE FUNCTION handle_status_change();

Self-Check

Before moving to Chapter 07, honestly answer these questions. If any answer is "no", go back and fix it.

Can you explain the difference between BEFORE and AFTER triggers?

BEFORE triggers fire before the row is written and can modify NEW or abort the operation with RAISE EXCEPTION. Use them for validation and data normalization. AFTER triggers fire after the row is committed and are used for side effects like audit logging, notifications, and cascading updates to other tables.

What happens to NOTIFY events if no one is listening?

They are silently discarded. NOTIFY is fire-and-forget — there is no queue, no retry, and no persistence. This is exactly why the transactional outbox pattern is important: the outbox table provides durability, while NOTIFY provides low-latency wake-up signals.

Does RLS protect against superusers and table owners?

No. By default, table owners and superusers bypass all RLS policies. To force RLS on the table owner, use ALTER TABLE t FORCE ROW LEVEL SECURITY. Superusers always bypass RLS and cannot be restricted. This is why your application should connect as a least-privilege role, not as the table owner.
