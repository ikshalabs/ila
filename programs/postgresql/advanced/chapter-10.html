Chapter 10 · Phase 5: Production & Capstone

Security, Compliance, and Capstone

Lock down your PostgreSQL deployment with role-based access control, row-level security policies, SSL/TLS encryption, audit logging for compliance, and data encryption at rest. Then bring everything together in a capstone project that demonstrates production-grade database engineering.

Estimated: 5–7 days
6 Key Concepts
3 Assignments

Why This Matters

Security is not a feature you add at the end — it is a property of every layer of your system. A data breach does not just cost money in fines; it destroys trust, erodes your brand, and can end careers. PostgreSQL provides powerful security primitives — roles, privileges, RLS, SSL, and audit logging — but they only work if you configure and test them systematically.

This final chapter brings together every skill you have learned across the entire program. The capstone project is not a toy exercise — it is a complete, production-grade database system that demonstrates mastery of schema design, query optimization, triggers, replication, backup, monitoring, and security. It is the artifact you will present to prove your PostgreSQL engineering competence.

Compliance frameworks like SOC 2, HIPAA, PCI-DSS, and GDPR mandate specific technical controls for data access, encryption, and audit trails. Understanding how to implement these controls in PostgreSQL is not optional for any organization that handles sensitive data — it is a legal and regulatory requirement.

Key Insight

Security in PostgreSQL follows the principle of least privilege: every role should have exactly the permissions it needs and nothing more. A web application should never connect as a superuser. An analyst should never have write access. An admin should never bypass audit logging. Defense in depth means layering multiple controls so that no single failure compromises the system.

Key Concepts

1. Role Management and Privileges

PostgreSQL uses a role-based access control (RBAC) system. Roles can own objects, have login privileges, and inherit permissions from other roles. A well-designed role hierarchy is the foundation of database security. The key principle is to create base roles for permission grouping (without login) and then create login roles that inherit from the base roles.

One of the most common security mistakes is leaving the default public schema permissions intact. By default, every role can create objects in the public schema and access any table without explicit grants. Always revoke these defaults in production.

sql 
-- Create a role hierarchy for a production application

-- Base roles (no login — used for privilege grouping)
CREATE ROLE app_readonly;
CREATE ROLE app_readwrite;
CREATE ROLE app_admin;

-- Grant schema-level access
GRANT USAGE ON SCHEMA public TO app_readonly;
GRANT USAGE ON SCHEMA public TO app_readwrite;
GRANT ALL ON SCHEMA public TO app_admin;

-- Grant table-level privileges
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public
  TO app_readwrite;
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO app_admin;

-- Ensure future tables inherit these grants
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO app_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_readwrite;

-- Grant sequence usage for INSERT operations
GRANT USAGE ON ALL SEQUENCES IN SCHEMA public TO app_readwrite;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT USAGE ON SEQUENCES TO app_readwrite;

-- Create login roles that inherit from base roles
CREATE ROLE web_app LOGIN PASSWORD 'strong_password_here';
GRANT app_readwrite TO web_app;

CREATE ROLE analyst_user LOGIN PASSWORD 'another_strong_password';
GRANT app_readonly TO analyst_user;

CREATE ROLE dba_user LOGIN PASSWORD 'dba_strong_password';
GRANT app_admin TO dba_user;

-- Revoke dangerous defaults
REVOKE ALL ON SCHEMA public FROM public;
REVOKE CREATE ON SCHEMA public FROM public;
Password Authentication

Always use scram-sha-256 authentication instead of the older md5 method. Set password_encryption = scram-sha-256 in postgresql.conf and use hostssl entries in pg_hba.conf with scram-sha-256 as the auth method. The md5 hash is considered cryptographically broken and should not be used in production.

2. Row-Level Security Deep Dive

Row-Level Security (RLS) allows you to control which rows a user can see or modify based on their identity or session variables. This is essential for multi-tenant applications where multiple organizations share the same tables. RLS policies are enforced at the database level, meaning even direct SQL access respects the policies — no application-layer bypass is possible.

RLS uses two types of expressions: USING controls which rows are visible (for SELECT, UPDATE, DELETE), and WITH CHECK controls which rows can be written (for INSERT, UPDATE). When both are specified on an UPDATE policy, the USING clause filters the rows that can be selected for update, and WITH CHECK validates the new row values.

sql 
-- Advanced RLS: separate read and write policies
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;
ALTER TABLE documents FORCE ROW LEVEL SECURITY;

-- Read policy: users see their own org's documents
CREATE POLICY doc_read ON documents
  FOR SELECT
  USING (
    org_id = current_setting('app.current_org_id')::INT
  );

-- Write policy: users can only create documents in their own org
CREATE POLICY doc_insert ON documents
  FOR INSERT
  WITH CHECK (
    org_id = current_setting('app.current_org_id')::INT
    AND created_by = current_setting('app.current_user_id')::INT
  );

-- Update policy: only document owners can update
CREATE POLICY doc_update ON documents
  FOR UPDATE
  USING (
    org_id = current_setting('app.current_org_id')::INT
    AND created_by = current_setting('app.current_user_id')::INT
  )
  WITH CHECK (
    org_id = current_setting('app.current_org_id')::INT
  );

-- Delete policy: only admins can delete
CREATE POLICY doc_delete ON documents
  FOR DELETE
  USING (
    org_id = current_setting('app.current_org_id')::INT
    AND current_setting('app.current_role') = 'admin'
  );

-- Test: set session variables and verify isolation
SET app.current_org_id = '1';
SET app.current_user_id = '42';
SET app.current_role = 'member';
SELECT * FROM documents; -- Only shows org 1 documents
RLS Performance

RLS policies are evaluated for every row access, so keep them simple. Avoid subqueries in policy expressions — use session variables (current_setting) or current_user instead. Create indexes on the columns used in RLS policies (e.g., org_id) so the planner can push the filter down to an index scan rather than applying it as a post-filter on every row.

3. SSL/TLS Configuration

All connections between your application and PostgreSQL should be encrypted with TLS. This prevents credential interception, data eavesdropping, and man-in-the-middle attacks on the network. Even within a private VPC, TLS is recommended as a defense-in-depth measure.

conf 
# postgresql.conf — Enable SSL
ssl = on
ssl_cert_file = '/etc/ssl/certs/server.crt'
ssl_key_file = '/etc/ssl/private/server.key'
ssl_ca_file = '/etc/ssl/certs/ca.crt'
ssl_min_protocol_version = 'TLSv1.3'
password_encryption = scram-sha-256

# pg_hba.conf — Require SSL for all remote connections
# TYPE    DATABASE  USER  ADDRESS       METHOD
hostssl  all       all   0.0.0.0/0    scram-sha-256
hostssl  all       all   ::0/0        scram-sha-256

# Local connections (no SSL needed for unix sockets)
local    all       all                    peer

# Reject non-SSL remote connections explicitly
hostnossl all      all   0.0.0.0/0    reject
bash 
# Generate self-signed certificate for development
openssl req -new -x509 -days 365 -nodes \
  -out server.crt -keyout server.key \
  -subj "/CN=postgres-server"

chmod 600 server.key
chown postgres:postgres server.crt server.key

# Verify SSL is working from the client side
psql "host=localhost dbname=app sslmode=verify-full sslrootcert=ca.crt"

# Check current SSL status from within psql
# \conninfo
# Shows: SSL connection (protocol: TLSv1.3, cipher: TLS_AES_256_GCM_SHA384)

4. Audit Logging for Compliance

Compliance frameworks like SOC 2, HIPAA, PCI-DSS, and GDPR require detailed audit trails of who accessed what data, when, and from where. PostgreSQL supports this through the pgAudit extension, which provides session-level and object-level audit logging far more detailed than the built-in log_statement parameter.

pgAudit distinguishes between session auditing (logs all statements from a session) and object auditing (logs only statements that access specific objects). Object auditing is more targeted and generates less noise, making it ideal for compliance on sensitive tables.

sql 
-- Install and configure pgAudit
CREATE EXTENSION IF NOT EXISTS pgaudit;

-- Session-level audit: log all DDL and write operations
ALTER SYSTEM SET pgaudit.log = 'ddl, write, role';
ALTER SYSTEM SET pgaudit.log_catalog = off;
ALTER SYSTEM SET pgaudit.log_relation = on;
ALTER SYSTEM SET pgaudit.log_statement_once = on;

-- Per-role audit logging for targeted monitoring
ALTER ROLE web_app SET pgaudit.log = 'write';
ALTER ROLE analyst_user SET pgaudit.log = 'read, write';

-- Object-level auditing: track all access to PII tables
CREATE ROLE auditor NOLOGIN;
GRANT SELECT, INSERT, UPDATE, DELETE ON
  users, payments, addresses TO auditor;
ALTER SYSTEM SET pgaudit.role = 'auditor';

-- Reload configuration
SELECT pg_reload_conf();

-- Enable connection logging for compliance
ALTER SYSTEM SET log_connections = on;
ALTER SYSTEM SET log_disconnections = on;
ALTER SYSTEM SET log_line_prefix = '%m [%p] %q%u@%d ';
Audit Log Storage

Audit logs must be stored separately from the database they are auditing. If an attacker gains database access, they should not be able to modify or delete the audit trail. Ship logs to an external SIEM (Splunk, ELK), an S3 bucket with object lock enabled, or a separate write-only database with different credentials.

5. Data Encryption (at Rest and in Transit)

Encryption at rest protects data on disk from physical theft or unauthorized access to storage. Encryption in transit (SSL/TLS) protects data on the network. Together with application-level column encryption for sensitive fields, they form a complete encryption strategy that satisfies most compliance requirements.

PostgreSQL does not natively support transparent data encryption (TDE) for the entire data directory. Instead, you can use filesystem-level encryption (LUKS, dm-crypt, AWS EBS encryption) for encryption at rest, and pgcrypto for column-level encryption of specific sensitive fields like SSNs, credit card numbers, and API keys.

sql 
-- Application-level column encryption using pgcrypto
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Create a table for encrypted secrets
CREATE TABLE user_secrets (
  id SERIAL PRIMARY KEY,
  user_id INT NOT NULL REFERENCES users(id),
  encrypted_ssn BYTEA NOT NULL,
  encrypted_card BYTEA,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Encrypt sensitive data before storage
INSERT INTO user_secrets (user_id, encrypted_ssn)
VALUES (
  1,
  pgp_sym_encrypt(
    '123-45-6789',
    current_setting('app.encryption_key')
  )
);

-- Decrypt when needed (only authorized roles)
SELECT user_id,
  pgp_sym_decrypt(
    encrypted_ssn,
    current_setting('app.encryption_key')
  ) AS ssn
FROM user_secrets
WHERE user_id = 1;

-- Hash passwords with bcrypt (never store plaintext)
UPDATE users
SET password_hash = crypt('user_password', gen_salt('bf', 12))
WHERE id = 1;

-- Verify password at login
SELECT id, email FROM users
WHERE email = 'user@example.com'
  AND password_hash = crypt('user_password', password_hash);
Key Management

Never store encryption keys in the same database as the encrypted data. Use a secrets manager (HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager) to provide keys at runtime through session variables. If an attacker dumps your database, the encrypted columns should be completely unreadable without the key from the external secrets manager.

6. Capstone: Production-Grade Database System

The capstone project integrates every concept from Chapters 01–10 into a single, deployable database system. It must demonstrate schema design, indexing, stored procedures, triggers, replication, backup, monitoring, and security — all working together as they would in a real production environment.

Choose a domain that requires complex data modeling: e-commerce, healthcare, financial services, or a multi-tenant SaaS platform. The system should handle at least three user roles, enforce row-level security, maintain audit trails, and include automated backup with point-in-time recovery capability.

Security checklist for production deployments
  • Application connects as a non-superuser role with minimal privileges
  • All remote connections require SSL (hostssl in pg_hba.conf)
  • Passwords use scram-sha-256 authentication (not md5)
  • Row-level security enabled on all multi-tenant tables
  • Audit logging captures all DDL, DML on sensitive tables, and role changes
  • PII columns encrypted with pgcrypto, keys stored in external secrets manager
  • Backup encryption enabled (pgBackRest with encryption or wal-g)
  • Default privileges revoked from public role
  • log_connections and log_disconnections enabled
  • Regular security audits using pg_hba_file_rules view
  • idle_in_transaction_session_timeout configured
  • statement_timeout set to prevent runaway queries
GDPR compliance considerations for PostgreSQL

Right to be forgotten: Implement a DELETE CASCADE or soft-delete pattern with a retention cleanup job. RLS ensures users only see active records. Create a stored procedure that removes all PII for a given user across all tables.

Data portability: Create a stored procedure that exports all data for a given user as JSON using row_to_json or jsonb_build_object.

Consent tracking: Add a consent_log table with trigger-based audit trail tracking when consent was given, modified, or withdrawn.

Data minimization: Use views to expose only necessary columns to application roles. The underlying table may contain more data, but the role cannot access restricted columns without explicit grants.

Self-Check

Before completing the program, honestly answer these final questions. They represent the minimum competence expected of a PostgreSQL engineer.

Does your application connect as a non-superuser role?

If your application connects as postgres or any superuser, you have a critical security gap. Superusers bypass all permission checks, RLS policies, and audit controls. Create a dedicated application role with only the permissions it needs. Never use a superuser for application connections.

Is every connection to your database encrypted with TLS?

Check your pg_hba.conf: are there any host entries (without ssl)? These allow unencrypted connections. Replace them with hostssl and set ssl_min_protocol_version = 'TLSv1.3'. Add a hostnossl ... reject line to explicitly deny unencrypted connections.

Can you recover your database to any point in the last 7 days?

This requires continuous WAL archiving plus regular base backups. If you only have nightly pg_dump files, your RPO is 24 hours — and that may not be acceptable for a production system handling financial data or user-facing transactions. Configure pgBackRest or wal-g for continuous archiving with at least 7 days retention.

Do you have a documented runbook for failover to a standby?

A standby server is only useful if your team knows exactly how to promote it. Document the exact steps: verify replication lag, promote the standby with pg_ctl promote or pg_promote(), update the connection pooler configuration, verify the new primary is accepting writes, and set up a new standby from the promoted server. Practice this procedure at least quarterly.

Can you demonstrate that your audit trail captures all PII access?

Run a SELECT on a PII table and verify the access appears in the PostgreSQL logs with pgAudit formatting. The log entry should show the timestamp, user, database, the exact SQL statement, and the tables accessed. If pgAudit is not installed, you have no compliance-grade audit trail — install and configure it immediately.
