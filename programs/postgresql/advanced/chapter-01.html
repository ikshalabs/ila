Query Planner Deep Dive

Master EXPLAIN ANALYZE output, understand cost estimation, join algorithms (nested loop, hash join, merge join), and planner configuration for optimal ad portal query performance.

Estimated: 4–5 hours
5 Key Concepts
3 Assignments

Why This Matters

When a dashboard query takes 3 seconds instead of 30ms, EXPLAIN ANALYZE is the diagnostic tool that tells you exactly why. Understanding the query planner is the difference between guessing at performance and knowing. In the ad portal, the query planner makes thousands of decisions per second about how to join campaigns to leads to accounts.

Key Insight

In production ad-tech systems, query performance is not optional — it is a business requirement. A 3-second dashboard load means lost productivity. A 500ms bid evaluation means lost revenue. Every topic in this advanced course addresses real performance and reliability challenges.

Key Concepts

1. PostgreSQL Cost Model

PostgreSQL assigns a cost to every possible query plan and chooses the cheapest one. The cost is measured in arbitrary units where seq_page_cost = 1.0 is the baseline. An index scan might cost 50 units while a sequential scan costs 5000. Understanding these costs helps you predict why the planner makes certain choices.

sql
-- Show current cost parameters
SHOW seq_page_cost;        -- 1.0 (baseline)
SHOW random_page_cost;     -- 4.0 (random I/O is 4x slower)
SHOW cpu_tuple_cost;       -- 0.01 (per-row CPU cost)
SHOW effective_cache_size; -- how much memory for caching

-- EXPLAIN with all details
EXPLAIN (ANALYZE, BUFFERS, COSTS, TIMING, FORMAT TEXT)
SELECT c.name, COUNT(l.id)
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.id
WHERE c.account_id = 7
GROUP BY c.id;

2. Join Algorithms

PostgreSQL uses three join strategies: Nested Loop (best for small result sets with indexed lookups), Hash Join (best for large unindexed joins), and Merge Join (best when both sides are already sorted). The planner chooses based on table sizes, available indexes, and work_mem.

sql
-- Force different join strategies for comparison (DO NOT use in production)
SET enable_hashjoin = off;
EXPLAIN ANALYZE SELECT * FROM campaigns c JOIN leads l ON l.campaign_id = c.id;
SET enable_hashjoin = on;

-- Increase work_mem to enable larger hash tables
SET work_mem = '256MB';
EXPLAIN ANALYZE SELECT * FROM campaigns c JOIN leads l ON l.campaign_id = c.id;

3. Scan Types

Sequential Scan: reads every row. Good for small tables or when selecting most rows. Index Scan: uses B-tree to find rows. Best for selective queries. Bitmap Index Scan: builds a bitmap of matching pages, then reads them. Good for moderate selectivity. Index Only Scan: returns data from the index itself without touching the table.

4. Planner Configuration

sql
-- Key settings that affect query planning
SET work_mem = '64MB';            -- Memory for sorts and hashes
SET effective_cache_size = '4GB';  -- OS + PG cache estimate
SET random_page_cost = 1.1;       -- Lower for SSD storage

-- Update table statistics for accurate planning
ANALYZE campaigns;
ANALYZE leads;

5. Reading Real Query Plans

A real query plan from the ad portal might show a Hash Join between campaigns and leads, with an Index Scan on campaigns.account_id and a Sequential Scan on leads (if no index on campaign_id). The key metrics are: actual time (ms), actual rows (vs estimated), and buffers hit/read.

Pro Tip: Use explain.dalibo.com

Paste your EXPLAIN ANALYZE output into explain.dalibo.com for a visual, interactive breakdown. It highlights the most expensive nodes, shows estimated vs actual row counts, and makes complex plans understandable.

Self-Check

What does it mean when estimated rows differ significantly from actual rows?

It means the table statistics are stale or the planner cannot estimate the selectivity of the WHERE condition. Run ANALYZE to update statistics. For complex conditions, consider CREATE STATISTICS for multi-column correlations.

When would PostgreSQL choose a Seq Scan even when an index exists?

When the query selects a large portion of the table (e.g., 50%+), a Seq Scan is faster than an Index Scan because sequential I/O is cheaper than random I/O. The planner correctly determines that reading the whole table sequentially is faster than bouncing around via the index.

How does work_mem affect join performance?

work_mem controls how much memory each sort or hash operation can use. With more work_mem, hash joins can build larger hash tables in memory (faster) instead of spilling to disk. Increasing work_mem can transform a slow disk-based sort into a fast in-memory sort.
