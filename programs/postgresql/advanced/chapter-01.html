Chapter 01 · Phase 1: Query Mastery

Advanced Query Optimization

Understand how PostgreSQL transforms your SQL into execution plans, learn to read cost estimates, master the three join algorithms, and harness parallel query execution to slash query times on large datasets.

Estimated: 3–4 days
6 Key Concepts
3 Assignments

Why This Matters

Every query you write is just a request. PostgreSQL's query planner decides how to actually execute it. Two queries that return identical results can differ by a factor of 1,000x in execution time depending on the plan the optimizer chooses. Understanding the planner is the single highest-leverage skill a database engineer can develop.

In production systems handling millions of rows, a single missing index or a poorly structured join can turn a 5ms query into a 30-second table scan. The difference between a junior and senior DBA is not SQL syntax — it is the ability to read an EXPLAIN ANALYZE output and know exactly where time is being wasted.

Key Insight

The goal of query optimization is not to write faster SQL. It is to give the planner enough information — through statistics, indexes, and query structure — so it can choose the fastest execution strategy on its own. You are training the optimizer, not replacing it.

Key Concepts

1. How the Query Planner Works

When PostgreSQL receives a query, it passes through several stages: parsing, rewriting, planning, and execution. The planner is the most critical stage — it generates multiple candidate execution plans, estimates the cost of each, and selects the cheapest one.

The planner considers factors like available indexes, table sizes (from pg_class.reltuples), column statistics (from pg_statistic), and join ordering. It uses a cost model where sequential I/O, random I/O, and CPU processing each have configurable cost weights.

sql 
-- View the execution plan without running the query
EXPLAIN SELECT o.id, c.name, o.total
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.created_at >= '2025-01-01'
ORDER BY o.total DESC
LIMIT 100;

-- Run the query and show actual timings + buffer usage
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT o.id, c.name, o.total
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.created_at >= '2025-01-01'
ORDER BY o.total DESC
LIMIT 100;

EXPLAIN vs EXPLAIN ANALYZE

EXPLAIN shows the estimated plan without running the query. EXPLAIN ANALYZE actually executes the query and shows real timings. Never run EXPLAIN ANALYZE on a destructive statement (DELETE, UPDATE) in production without wrapping it in a transaction you roll back.

2. Cost Estimation and Statistics

The planner relies heavily on table statistics collected by ANALYZE (or autovacuum). These statistics include the number of rows, the number of distinct values per column, the most common values, and histogram bounds for value distribution. When statistics are stale, the planner makes bad decisions.

sql 
-- Force a statistics refresh on a specific table
ANALYZE orders;

-- Check current statistics for a column
SELECT
    attname,
    n_distinct,
    most_common_vals,
    most_common_freqs,
    correlation
FROM pg_stats
WHERE tablename = 'orders'
  AND attname = 'status';

-- Increase statistics target for skewed data columns
ALTER TABLE orders
  ALTER COLUMN status SET STATISTICS 1000;

-- Check estimated vs actual row counts
SELECT
    relname,
    reltuples::bigint AS estimated_rows,
    relpages AS disk_pages
FROM pg_class
WHERE relname = 'orders';

The cost model uses configurable parameters. seq_page_cost (default 1.0) is the cost of reading a page sequentially. random_page_cost (default 4.0) reflects the higher cost of random disk access. On SSDs, you should lower random_page_cost to around 1.1 since random and sequential reads are nearly the same speed.

sql 
-- Tune cost parameters for SSD storage
SET random_page_cost = 1.1;
SET effective_cache_size = '8GB';

-- View all planner cost settings
SELECT name, setting, short_desc
FROM pg_settings
WHERE name LIKE '%cost%'
   OR name LIKE '%cache%';

3. Join Algorithms (Nested Loop, Hash, Merge)

PostgreSQL implements three join algorithms, and the planner selects the best one based on table sizes, available indexes, and sort order. Understanding when each is chosen helps you structure queries and indexes to get the optimal plan.

  • Nested Loop Join — For each row in the outer table, scan the inner table. Best when the outer set is small and the inner table has an index. O(n*m) worst case, but fast with indexes.
  • Hash Join — Build a hash table from the smaller relation, then probe it for each row in the larger one. Best for large unsorted joins. Requires memory for the hash table (work_mem).
  • Merge Join — Both inputs must be sorted on the join key. Walks through both in parallel. Extremely efficient when data is pre-sorted (e.g., joining on a primary key with a btree index).
sql 
-- Force specific join methods to compare plans
-- Disable hash join to see if merge or nested loop is better
SET enable_hashjoin = off;
EXPLAIN ANALYZE
SELECT o.*, p.name
FROM orders o
JOIN products p ON p.id = o.product_id
WHERE o.region = 'US';

-- Re-enable and try hash join only
SET enable_hashjoin = on;
SET enable_mergejoin = off;
SET enable_nestloop = off;
EXPLAIN ANALYZE
SELECT o.*, p.name
FROM orders o
JOIN products p ON p.id = o.product_id
WHERE o.region = 'US';

-- Always reset after testing
RESET enable_hashjoin;
RESET enable_mergejoin;
RESET enable_nestloop;
Pro Tip

If you see a nested loop join on two large tables (both over 100k rows) with no index lookup on the inner side, the planner is likely making a mistake due to stale statistics. Run ANALYZE on both tables and re-check the plan.

4. Parallel Query Execution

PostgreSQL can split a single query across multiple CPU cores using parallel workers. This is especially powerful for sequential scans, aggregations, and hash joins on large tables. The planner automatically considers parallel plans when the estimated cost exceeds parallel_tuple_cost thresholds.

sql 
-- Check current parallel settings
SHOW max_parallel_workers_per_gather;
SHOW max_parallel_workers;
SHOW parallel_tuple_cost;
SHOW min_parallel_table_scan_size;

-- Allow up to 4 parallel workers per query
SET max_parallel_workers_per_gather = 4;

-- Force parallel for testing (do NOT use in production)
SET parallel_tuple_cost = 0;
SET parallel_setup_cost = 0;
SET min_parallel_table_scan_size = 0;

-- Observe parallel workers in EXPLAIN output
EXPLAIN ANALYZE
SELECT region, COUNT(*), AVG(total)
FROM orders
GROUP BY region;

-- Look for "Workers Planned: 4" and "Workers Launched: 4"

Operations that support parallelism include: sequential scans, index scans (in PostgreSQL 12+), hash joins, merge joins, nested loops, aggregations, and CREATE INDEX CONCURRENTLY. Operations that do not parallelize include: writes (INSERT/UPDATE/DELETE), cursors, and functions marked PARALLEL UNSAFE.

5. Plan Hints and pg_hint_plan

PostgreSQL does not natively support query hints (unlike Oracle or MySQL). This is by design — the optimizer is trusted to make the right choice given accurate statistics. However, when you genuinely need to override the planner, the pg_hint_plan extension provides hints via SQL comments.

sql 
-- Install pg_hint_plan (requires superuser)
CREATE EXTENSION pg_hint_plan;

-- Force a specific join method via hint comment
/*+ HashJoin(o c) Leading(o c) */
SELECT o.id, c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id;

-- Force an index scan on a specific index
/*+ IndexScan(orders idx_orders_created_at) */
SELECT * FROM orders
WHERE created_at >= '2025-01-01';

-- Force sequential scan (useful when index is slower)
/*+ SeqScan(orders) */
SELECT * FROM orders
WHERE status = 'active';
Warning

Plan hints are a last resort. They hard-code decisions that should adapt to changing data distributions. If you find yourself using hints frequently, the real problem is likely stale statistics, missing indexes, or poorly structured queries. Fix the root cause first.

6. Common Query Anti-Patterns

Certain SQL patterns consistently defeat the query planner. Recognizing them is faster than running EXPLAIN on every query. Here are the most common offenders:

sql 
-- ANTI-PATTERN 1: Function on indexed column prevents index use
-- BAD: forces sequential scan
SELECT * FROM orders
WHERE EXTRACT(YEAR FROM created_at) = 2025;

-- GOOD: range scan uses btree index on created_at
SELECT * FROM orders
WHERE created_at >= '2025-01-01'
  AND created_at < '2026-01-01';

-- ANTI-PATTERN 2: Implicit type casting
-- BAD: phone is varchar, comparing to integer
SELECT * FROM customers
WHERE phone = 5551234567;

-- GOOD: match the column type
SELECT * FROM customers
WHERE phone = '5551234567';

-- ANTI-PATTERN 3: SELECT * fetches unnecessary columns
-- BAD: loads all columns from disk
SELECT * FROM orders WHERE id = 42;

-- GOOD: enables index-only scans
SELECT id, total, status FROM orders WHERE id = 42;

-- ANTI-PATTERN 4: OR conditions that prevent index use
-- BAD: may force sequential scan
SELECT * FROM orders
WHERE customer_id = 10 OR product_id = 20;

-- GOOD: UNION ALL for separate index scans
SELECT * FROM orders WHERE customer_id = 10
UNION ALL
SELECT * FROM orders WHERE product_id = 20
  AND customer_id != 10;
Why does wrapping a column in a function prevent index use?

A B-tree index stores values in sorted order. When you apply a function to the column (e.g., LOWER(email)), the planner cannot look up the transformed value in the index because the index stores the original values. The fix is either to rewrite the query to avoid the function, or to create a functional index: CREATE INDEX idx_email_lower ON users (LOWER(email));

When is a sequential scan actually faster than an index scan?

When the query returns a large percentage of the table (typically over 10–15%), a sequential scan reads pages in order from disk, which is much faster than the random I/O pattern of an index scan bouncing between index pages and heap pages. The planner accounts for this via random_page_cost vs seq_page_cost.

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer is uncertain, go back and experiment.

Can you read an EXPLAIN ANALYZE output and identify the bottleneck in under 60 seconds?

Look for the node with the highest "actual time" or the largest gap between estimated and actual rows. That is almost always where the problem is. Practice with explain.depesz.com until this becomes instinctive.

Do you understand when PostgreSQL chooses a sequential scan over an index scan?

A sequential scan is preferred when the query returns a large fraction of the table, when the table is very small (fits in a few pages), or when random_page_cost is set much higher than seq_page_cost. If you are on SSDs, lower random_page_cost to 1.1 and re-test.

Can you explain the difference between the three join algorithms to a colleague?

Nested loop: for each row in A, look up matching rows in B (best with index on B). Hash join: build a hash table from the smaller table, probe with the larger (best for unsorted large joins). Merge join: sort both inputs and walk through in parallel (best when both are already sorted on the join key).
