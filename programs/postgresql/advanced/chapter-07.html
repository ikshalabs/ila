Chapter 07 · Phase 4: Operations (07/10)

Replication and Failover

Build a highly available PostgreSQL architecture with streaming replication, logical replication for selective table sync, connection pooling with pgBouncer, and automatic failover using Patroni. A single-node database is a single point of failure — this chapter eliminates it.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

A database that cannot survive a hardware failure is a liability. Every minute of unplanned downtime costs money, erodes trust, and creates a cascading failure across every service that depends on your data. Replication is how you turn a single database into a resilient cluster that can survive node failures, scale read traffic, and provide zero-data-loss guarantees.

But replication alone is not enough. Without connection pooling, your application will exhaust PostgreSQL's connection limit under load. Without automatic failover, a primary failure requires manual intervention — at 3 AM, by an on-call engineer who may not have rehearsed the procedure. This chapter builds the complete high-availability stack.

Key Insight

High availability is not a feature you add later — it is an architecture you design from the start. Retrofitting replication onto a running production database is orders of magnitude harder than setting it up before you go live.

Key Concepts

1. Streaming Replication Setup

Streaming replication is PostgreSQL's built-in mechanism for maintaining an exact byte-for-byte copy of the primary database on one or more standby servers. The primary ships WAL (Write-Ahead Log) records to the standby in real-time, and the standby replays them to stay in sync.

conf 
# postgresql.conf — Primary server configuration
wal_level = replica
max_wal_senders = 5
wal_keep_size = 1GB
synchronous_commit = on

# For synchronous replication (zero data loss):
# synchronous_standby_names = 'standby1'

# pg_hba.conf — Allow replication connections
host  replication  replicator  10.0.0.0/24  scram-sha-256
bash 
# Create a replication user on the primary
psql -c "CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'secure_password';"

# Take a base backup on the standby server
pg_basebackup -h primary-host -U replicator \
  -D /var/lib/postgresql/16/main \
  -Fp -Xs -P -R

# The -R flag creates standby.signal and configures
# primary_conninfo in postgresql.auto.conf automatically

                         WAL Stream
 ┌──────────────┐  ──────────────────▶  ┌──────────────┐
 │   PRIMARY    │                       │   STANDBY    │
 │  (read/write)│                       │  (read-only) │
 │              │  ◀──────────────────  │              │
 └──────┬───────┘   feedback/status     └──────────────┘
        │
        │  writes
        ▼
 ┌──────────────┐
 │     WAL      │
 │   Segments   │
 └──────────────┘
Figure 1: Streaming replication — primary ships WAL to standby in real-time

2. Logical Replication

While streaming replication copies the entire database cluster, logical replication lets you selectively replicate individual tables or a subset of data. This is invaluable for cross-version upgrades, data federation, and feeding analytics systems.

sql 
-- On the PUBLISHER (source database)
ALTER SYSTEM SET wal_level = logical;
-- Restart PostgreSQL after this change

-- Create a publication for specific tables
CREATE PUBLICATION analytics_pub
  FOR TABLE orders, users, products
  WITH (publish = 'insert, update, delete');

-- On the SUBSCRIBER (destination database)
CREATE SUBSCRIPTION analytics_sub
  CONNECTION 'host=primary-host dbname=app user=replicator password=secure'
  PUBLICATION analytics_pub;

-- Monitor replication status
SELECT subname, received_lsn, latest_end_lsn,
       latest_end_time
FROM pg_stat_subscription;
Logical vs. Streaming

Streaming replication replicates the entire cluster and is used for HA/failover. Logical replication replicates specific tables and allows writes on both sides. They serve different purposes and can be used together. Logical replication requires wal_level = logical (which is a superset of replica).

3. pgBouncer and Connection Pooling

PostgreSQL forks a new process for every connection. At 100+ concurrent connections, this becomes a serious resource bottleneck. pgBouncer sits between your application and PostgreSQL, multiplexing many client connections onto a smaller pool of database connections.

ini 
; pgbouncer.ini — Connection pooler configuration
[databases]
app_production = host=127.0.0.1 port=5432 dbname=app_production

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt

; Pool mode: transaction is the most common for web apps
pool_mode = transaction

; Connection limits
default_pool_size = 20
max_client_conn = 1000
max_db_connections = 50

; Timeouts
server_idle_timeout = 300
client_idle_timeout = 0
query_timeout = 30
Pro Tip

Use pool_mode = transaction for web applications. This releases the server connection back to the pool after each transaction completes. Avoid session mode unless you need LISTEN/NOTIFY, prepared statements, or session-level variables. Use statement mode only for simple load balancing with no multi-statement transactions.

4. Read Replicas for Scaling

Once you have streaming replication, your standby servers can serve read-only queries. This offloads analytics, reporting, and search traffic from the primary, freeing it to handle writes. The key challenge is routing: your application must know which queries go to the primary and which go to a replica.

sql 
-- Enable hot standby reads on the replica
-- postgresql.conf on standby:
-- hot_standby = on

-- Check if current server is in recovery (is a replica)
SELECT pg_is_in_recovery();
-- Returns true on replicas, false on primary

-- Check replication lag on the replica
SELECT
  now() - pg_last_xact_replay_timestamp() AS replication_lag,
  pg_is_in_recovery() AS is_replica,
  pg_last_wal_receive_lsn() AS received_lsn,
  pg_last_wal_replay_lsn() AS replayed_lsn;

-- Check replication status from the primary
SELECT client_addr, state, sent_lsn, write_lsn,
       flush_lsn, replay_lsn,
       pg_wal_lsn_diff(sent_lsn, replay_lsn) AS byte_lag
FROM pg_stat_replication;

5. Automatic Failover with Patroni

Patroni is an open-source tool that manages PostgreSQL high availability with automatic failover. It uses a distributed consensus store (etcd, Consul, or ZooKeeper) to elect a leader and automatically promotes a standby to primary when the current primary fails.

yaml 
# patroni.yml — Patroni configuration
scope: production-cluster
name: node1

restapi:
  listen: 0.0.0.0:8008
  connect_address: 10.0.1.10:8008

etcd:
  hosts: 10.0.1.50:2379,10.0.1.51:2379,10.0.1.52:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1MB
    postgresql:
      parameters:
        max_connections: 200
        shared_buffers: 2GB
        wal_level: replica
        max_wal_senders: 5
        max_replication_slots: 5

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 10.0.1.10:5432
  data_dir: /var/lib/postgresql/16/main
  authentication:
    superuser:
      username: postgres
      password: secure_password
    replication:
      username: replicator
      password: replicator_password
bash 
# Check cluster status with patronictl
patronictl -c /etc/patroni/patroni.yml list

# Output:
# + Cluster: production-cluster ----+---------+---------+
# | Member | Host       | Role    | State   | Lag     |
# +--------+------------+---------+---------+---------+
# | node1  | 10.0.1.10  | Leader  | running |         |
# | node2  | 10.0.1.11  | Replica | running | 0 bytes |
# | node3  | 10.0.1.12  | Replica | running | 0 bytes |
# +--------+------------+---------+---------+---------+

# Manual switchover (planned maintenance)
patronictl -c /etc/patroni/patroni.yml switchover \
  --master node1 --candidate node2

6. Monitoring Replication Lag

Replication lag is the delay between when a change is written on the primary and when it becomes visible on the replica. High lag means stale reads, potential data loss during failover, and unhappy users who see inconsistent data.

sql 
-- Comprehensive replication monitoring query (run on primary)
SELECT
  client_addr,
  application_name,
  state,
  sync_state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  pg_wal_lsn_diff(sent_lsn, replay_lsn) AS replay_lag_bytes,
  pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn) AS send_lag_bytes,
  write_lag,
  flush_lag,
  replay_lag
FROM pg_stat_replication
ORDER BY client_addr;

-- Alert when lag exceeds threshold (use in monitoring)
SELECT client_addr,
       pg_wal_lsn_diff(sent_lsn, replay_lsn) AS lag_bytes
FROM pg_stat_replication
WHERE pg_wal_lsn_diff(sent_lsn, replay_lsn) > 10485760;
-- Alerts when any replica is more than 10MB behind
Critical Warning

If a replication slot is not consumed (e.g., a replica is offline for days), WAL files accumulate on the primary and will eventually fill the disk. Always monitor pg_replication_slots and drop inactive slots before they cause a disk-full outage on the primary itself.

Synchronous vs. asynchronous replication

Asynchronous (default): The primary does not wait for the standby to confirm receipt of WAL data. Fastest writes, but a primary failure can lose the last few milliseconds of committed transactions.

Synchronous: The primary waits for at least one standby to confirm it has written the WAL data to disk. Zero data loss, but write latency increases by the network round-trip time. Enable with synchronous_standby_names in postgresql.conf.

Connection routing with HAProxy

Use HAProxy to route connections to the current primary, leveraging Patroni's REST API for health checks:

conf 
# HAProxy configuration for Patroni
frontend postgres_write
  bind *:5000
  default_backend postgres_primary

backend postgres_primary
  option httpchk GET /primary
  http-check expect status 200
  server node1 10.0.1.10:5432 check port 8008
  server node2 10.0.1.11:5432 check port 8008
  server node3 10.0.1.12:5432 check port 8008

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is "no", go back and fix it.

Can you explain the difference between streaming and logical replication?

Streaming replication copies the entire WAL byte stream and creates an exact replica of the primary. Logical replication decodes the WAL into logical changes and can replicate specific tables to different databases, even across PostgreSQL versions. Streaming is for HA; logical is for selective data distribution.

What pool mode should you use for a web application?

transaction mode. It releases the server connection after each transaction, allowing many clients to share a small pool. session mode holds the connection for the entire client session and defeats the purpose of pooling for most web apps. statement mode breaks multi-statement transactions.

What happens if a replication slot consumer goes offline for too long?

WAL files accumulate on the primary because PostgreSQL retains them for the slot. This will eventually fill the disk and crash the primary. Always monitor pg_replication_slots for inactive slots and set max_slot_wal_keep_size to limit retention.
