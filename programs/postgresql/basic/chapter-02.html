Chapter 02 · Phase 1: SQL Fundamentals

CRUD Operations and Filtering

Every database interaction boils down to four operations: Create, Read, Update, Delete. Mastering these with proper filtering is the foundation of all SQL work.

Estimated: 2–3 days
3 Advancement Levels
3 Assignments

Why This Matters

CRUD — Create, Read, Update, Delete — sounds simple, but the details matter enormously. A poorly written UPDATE without a WHERE clause can wipe an entire table. A SELECT without proper filtering can bring a production database to its knees. An INSERT that violates constraints silently corrupts your data model.

Every API endpoint, every form submission, every background job ultimately translates to one or more CRUD operations. The difference between a junior and senior engineer is not whether they know these operations exist — it is whether they understand the edge cases: What happens when you update zero rows? How do you handle NULLs in comparisons? What is the performance difference between IN and EXISTS?

This chapter drills into each operation with the precision required for production work. By the end, you will write CRUD operations that are safe, efficient, and correct under all conditions.

Key Insight

SQL is a declarative language — you describe what you want, not how to get it. The database optimizer figures out the "how." This means the same logical query can execute thousands of different ways internally. Writing clear, well-structured SQL helps the optimizer choose the best plan.

Key Concepts

1. SELECT and Column Expressions

SELECT is the most frequently used SQL statement. It retrieves data from one or more tables and can transform that data using expressions, functions, and aliases. Mastering SELECT means mastering how to ask precise questions of your data.

sql 
-- Setup: Create and populate an employees table
CREATE TABLE employees (
    id          serial       PRIMARY KEY,
    first_name  text         NOT NULL,
    last_name   text         NOT NULL,
    email       text         UNIQUE NOT NULL,
    department  varchar(50),
    salary      numeric(10,2),
    hire_date   date         NOT NULL,
    is_active   boolean      DEFAULT true,
    manager_id  integer      REFERENCES employees(id)
);

INSERT INTO employees (first_name, last_name, email, department, salary, hire_date) VALUES
    ('Alice',   'Chen',     'alice@co.io',   'Engineering', 135000, '2021-03-15'),
    ('Bob',     'Martinez', 'bob@co.io',     'Engineering', 128000, '2021-06-01'),
    ('Carol',   'Johnson',  'carol@co.io',   'Marketing',   95000,  '2022-01-10'),
    ('David',   'Kim',      'david@co.io',   'Marketing',   88000,  '2022-09-20'),
    ('Eve',     'Williams', 'eve@co.io',     'Sales',       105000, '2020-11-05'),
    ('Frank',   'Brown',    'frank@co.io',   'Sales',       92000,  '2023-02-14'),
    ('Grace',   'Lee',      'grace@co.io',   'Engineering', 142000, '2019-08-22'),
    ('Hank',    'Davis',    'hank@co.io',    NULL,           75000,  '2023-07-01');
sql 
-- Basic column selection
SELECT first_name, last_name, salary FROM employees;

-- String concatenation
SELECT
    first_name || ' ' || last_name AS full_name,
    email
FROM employees;

-- Computed columns with math
SELECT
    first_name,
    salary,
    salary * 12          AS annual_salary,
    salary * 12 * 0.30   AS estimated_tax,
    round(salary / 160, 2) AS hourly_rate
FROM employees;

-- CASE expressions (SQL's if/else)
SELECT
    first_name,
    salary,
    CASE
        WHEN salary >= 130000 THEN 'Senior'
        WHEN salary >= 100000 THEN 'Mid-Level'
        WHEN salary >= 80000  THEN 'Junior'
        ELSE 'Entry'
    END AS level
FROM employees;

-- DISTINCT to remove duplicates
SELECT DISTINCT department FROM employees;

2. INSERT — Single and Bulk

INSERT adds new rows to a table. PostgreSQL supports single-row inserts, multi-row inserts, and INSERT with conflict handling. The RETURNING clause is a PostgreSQL-specific feature that returns the inserted data, which is extremely useful for getting auto-generated IDs.

sql 
-- Single row insert
INSERT INTO employees (first_name, last_name, email, department, salary, hire_date)
VALUES ('Irene', 'Taylor', 'irene@co.io', 'Engineering', 115000, '2024-01-15');

-- Single row with RETURNING (get the auto-generated id back)
INSERT INTO employees (first_name, last_name, email, department, salary, hire_date)
VALUES ('Jack', 'Wilson', 'jack@co.io', 'Sales', 98000, '2024-02-01')
RETURNING id, first_name, email;

-- Multi-row (bulk) insert — much faster than individual inserts
INSERT INTO employees (first_name, last_name, email, department, salary, hire_date) VALUES
    ('Kate',  'Adams',   'kate@co.io',  'Marketing',   91000,  '2024-03-01'),
    ('Leo',   'Garcia',  'leo@co.io',   'Engineering', 125000, '2024-03-15'),
    ('Mia',   'Patel',   'mia@co.io',   'Sales',       102000, '2024-04-01')
RETURNING id, first_name;

-- INSERT ... ON CONFLICT (upsert) — insert or update if exists
INSERT INTO employees (first_name, last_name, email, department, salary, hire_date)
VALUES ('Alice', 'Chen', 'alice@co.io', 'Engineering', 140000, '2021-03-15')
ON CONFLICT (email) DO UPDATE SET
    salary = EXCLUDED.salary,
    department = EXCLUDED.department;
Pro Tip

Always use RETURNING when you need the inserted data back. Without it, your application would need a separate SELECT query to fetch the new row's ID — that is two round trips instead of one. In high-throughput systems, this doubles your INSERT latency.

3. UPDATE with Conditions

UPDATE modifies existing rows. The most critical rule: always include a WHERE clause unless you intentionally want to update every row. An unfiltered UPDATE is one of the fastest ways to cause a production incident.

sql 
-- Update a single row by primary key
UPDATE employees
SET    salary = 145000
WHERE  id = 1;

-- Update multiple columns at once
UPDATE employees
SET    department = 'Product',
       salary = salary * 1.10    -- 10% raise
WHERE  email = 'bob@co.io';

-- Conditional update: give 5% raise to Engineering
UPDATE employees
SET    salary = round(salary * 1.05, 2)
WHERE  department = 'Engineering'
  AND  is_active = true
RETURNING first_name, salary;

-- Update with a subquery
UPDATE employees
SET    manager_id = (
           SELECT id FROM employees
           WHERE email = 'grace@co.io'
       )
WHERE  department = 'Engineering'
  AND  email != 'grace@co.io';
The Missing WHERE Catastrophe

Running UPDATE employees SET salary = 0 without a WHERE clause sets every employee's salary to zero. In production, always wrap destructive operations in a transaction: BEGIN; UPDATE ...; SELECT ... (verify); COMMIT; or ROLLBACK;. This gives you a safety net before the changes become permanent.

4. DELETE and Soft Deletes

DELETE permanently removes rows from a table. In production systems, you almost never want to actually delete data — instead, you use a "soft delete" pattern where you mark rows as inactive. This preserves audit trails and makes recovery possible.

sql 
-- Hard delete: permanently remove a row
DELETE FROM employees
WHERE id = 8
RETURNING *;

-- Delete with multiple conditions
DELETE FROM employees
WHERE department = 'Sales'
  AND hire_date < '2022-01-01'
RETURNING id, first_name, last_name;

-- ── Soft Delete Pattern ──
-- Add a deleted_at column to the table
ALTER TABLE employees
ADD COLUMN deleted_at timestamptz;

-- "Delete" by setting deleted_at
UPDATE employees
SET    deleted_at = now()
WHERE  id = 6;

-- Query only active (non-deleted) rows
SELECT * FROM employees
WHERE  deleted_at IS NULL;

-- "Restore" a soft-deleted row
UPDATE employees
SET    deleted_at = NULL
WHERE  id = 6;

-- TRUNCATE: delete ALL rows (much faster than DELETE for full wipe)
-- TRUNCATE TABLE employees;  ← be very careful with this!

  Hard Delete vs Soft Delete
  ==========================

  Hard Delete:                       Soft Delete:
  +----+-------+                     +----+-------+------------+
  | id | name  |                     | id | name  | deleted_at |
  +----+-------+                     +----+-------+------------+
  |  1 | Alice |   DELETE WHERE      |  1 | Alice | NULL       |
  |  2 | Bob   |   id = 3           |  2 | Bob   | NULL       |
  |  3 | Carol | ← row gone forever  |  3 | Carol | 2024-03-15 | ← still in DB
  +----+-------+                     +----+-------+------------+

  Recovery: impossible               Recovery: SET deleted_at = NULL
  Audit:    no trail                  Audit:    full history preserved
Figure 1: Hard delete destroys data permanently; soft delete preserves it for recovery and auditing

5. WHERE Clause Operators

The WHERE clause is where SQL becomes truly powerful. PostgreSQL supports a rich set of comparison operators, logical operators, and pattern matching that let you express complex filtering conditions in a single query.

sql 
-- Comparison operators: =, !=, <, >, <=, >=
SELECT * FROM employees WHERE salary >= 100000;
SELECT * FROM employees WHERE department != 'Sales';

-- AND / OR — combine conditions
SELECT * FROM employees
WHERE  department = 'Engineering'
  AND  salary > 130000;

SELECT * FROM employees
WHERE  department = 'Engineering'
   OR  department = 'Marketing';

-- IN — match against a list (cleaner than multiple ORs)
SELECT * FROM employees
WHERE  department IN ('Engineering', 'Marketing', 'Sales');

-- NOT IN — exclude values
SELECT * FROM employees
WHERE  department NOT IN ('Sales');

-- BETWEEN — inclusive range
SELECT * FROM employees
WHERE  salary BETWEEN 90000 AND 120000;

SELECT * FROM employees
WHERE  hire_date BETWEEN '2022-01-01' AND '2023-12-31';

-- LIKE — pattern matching (case-sensitive)
SELECT * FROM employees
WHERE  email LIKE '%@co.io';        -- ends with @co.io

SELECT * FROM employees
WHERE  first_name LIKE 'A%';         -- starts with A

-- ILIKE — case-insensitive pattern matching (PostgreSQL extension)
SELECT * FROM employees
WHERE  first_name ILIKE '%alice%';   -- matches Alice, alice, ALICE

-- EXISTS — check if a subquery returns any rows
SELECT * FROM employees e
WHERE  EXISTS (
    SELECT 1 FROM employees m
    WHERE m.id = e.manager_id
);
Operator Precedence

AND has higher precedence than OR. The query WHERE a = 1 OR b = 2 AND c = 3 is evaluated as WHERE a = 1 OR (b = 2 AND c = 3). If you mean (a = 1 OR b = 2) AND c = 3, you must use explicit parentheses. When in doubt, always use parentheses to make your intent clear.

6. NULL Handling and Sorting

NULL in SQL is not a value — it is the absence of a value. This distinction matters because NULL behaves differently from every other value: it is not equal to anything, not even itself. Misunderstanding NULL is the source of countless bugs.

sql 
-- NULL comparisons: these return NO rows (NULL != NULL)
SELECT * FROM employees WHERE department = NULL;    -- WRONG!
SELECT * FROM employees WHERE department != NULL;   -- WRONG!

-- Correct way: use IS NULL / IS NOT NULL
SELECT * FROM employees WHERE department IS NULL;
SELECT * FROM employees WHERE department IS NOT NULL;

-- COALESCE: replace NULL with a default value
SELECT
    first_name,
    COALESCE(department, 'Unassigned') AS department
FROM employees;

-- NULLIF: returns NULL if two values are equal (useful for avoiding division by zero)
SELECT
    first_name,
    salary / NULLIF(department, '') -- avoid divide-by-empty-string errors
FROM employees;

-- ── Sorting ──
-- Basic ORDER BY
SELECT * FROM employees ORDER BY salary DESC;

-- Multi-column sort
SELECT * FROM employees
ORDER BY department ASC, salary DESC;

-- Control where NULLs appear in sort
SELECT * FROM employees
ORDER BY department ASC NULLS LAST;

SELECT * FROM employees
ORDER BY department ASC NULLS FIRST;

-- Pagination with LIMIT and OFFSET
SELECT   first_name, salary
FROM     employees
ORDER BY salary DESC
LIMIT    5           -- return only 5 rows
OFFSET   0;          -- start from the first row

-- Page 2 (rows 6-10)
SELECT   first_name, salary
FROM     employees
ORDER BY salary DESC
LIMIT    5
OFFSET   5;
Three-Valued Logic

SQL uses three-valued logic: TRUE, FALSE, and UNKNOWN. Any comparison involving NULL produces UNKNOWN, not TRUE or FALSE. NULL = NULL is UNKNOWN. NULL != 5 is UNKNOWN. WHERE only includes rows where the condition is TRUE, so rows with NULL values in the compared column are silently excluded. This is why NOT IN with a subquery that might return NULLs can produce unexpected empty results.

Self-Check

Before moving to Chapter 03, verify your understanding of these critical concepts.

What does RETURNING do and why should you always use it?

RETURNING makes INSERT, UPDATE, and DELETE return the affected rows, just like a SELECT. Without it, INSERT returns only the count of inserted rows; you would need a separate SELECT to get auto-generated IDs or computed defaults. Using RETURNING eliminates one database round trip, which halves latency for write operations. It also guarantees you see exactly what was written, which is important when triggers or DEFAULT values are involved.

Why is NOT IN (subquery) dangerous when NULLs might be present?

If a subquery returns any NULL values, NOT IN returns no rows at all. For example: WHERE department NOT IN (SELECT dept FROM other_table) — if other_table.dept contains even one NULL, the entire WHERE clause evaluates to UNKNOWN for every row, returning nothing. The safe alternative is NOT EXISTS, which correctly handles NULLs because it checks for row existence, not value equality.

When should you use a hard delete versus a soft delete?

Use soft deletes when you need audit trails, when regulations require data retention (GDPR right to erasure is handled differently from actual deletion), when users might want to undo deletions, or when other tables reference the row with foreign keys. Use hard deletes for truly temporary data (session tokens, OTP codes), for data that must be permanently destroyed (privacy compliance after retention period), or when table size is a concern and you have no archival strategy. Most production systems default to soft deletes.
