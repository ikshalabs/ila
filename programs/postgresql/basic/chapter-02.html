Chapter 02 · PostgreSQL Basics

CRUD Operations and Filtering

Master SELECT, INSERT, UPDATE, and DELETE — the four operations that drive every interaction with your ad portal's data. Learn to filter with WHERE, handle NULLs, and sort result sets for real-world reporting.

Estimated: 3–4 hours
5 Key Concepts
3 Assignments

Why This Matters

Every screen in an ad portal is backed by CRUD operations. The campaign list page is a SELECT with filters. Creating a new campaign is an INSERT. Toggling a campaign from active to paused is an UPDATE. Removing a test ad group is a DELETE (or more commonly in ad-tech, a soft delete that sets discarded_at). These four operations are the verbs of your database vocabulary.

The difference between a junior and senior engineer often comes down to how precisely they can filter data. "Show me all active campaigns with a monthly budget over $5,000 that were created in the last 30 days" — that is a WHERE clause with multiple conditions, and if you cannot write it fluently, you will struggle with every debugging session and every feature you build.

Key Insight

In production ad-tech, you will write far more SELECT statements than any other operation. A typical ratio is 90% reads to 10% writes. Mastering SELECT with precise filtering is the highest-leverage SQL skill you can develop. Every support ticket, every analytics dashboard, every debugging session starts with a well-crafted SELECT.

Key Concepts

1. SELECT — Reading Data

The SELECT statement retrieves rows from one or more tables. In an ad portal, you use it constantly: listing campaigns for an account, pulling lead statistics, checking bid prices. The basic structure is SELECT columns FROM table WHERE condition ORDER BY column.

sql 
-- Select all columns from campaigns
SELECT * FROM campaigns;

-- Select specific columns (always preferred in production)
SELECT id, name, monthly_budget, active
FROM campaigns;

-- Use aliases for clarity in complex queries
SELECT
    c.name        AS campaign_name,
    c.monthly_budget AS budget,
    c.base_price  AS price_per_lead,
    c.active      AS is_active
FROM campaigns c;

-- Limit results (essential when exploring large tables)
SELECT * FROM leads LIMIT 10;

-- Distinct values — what statuses exist in accounts?
SELECT DISTINCT status FROM accounts;
Avoid SELECT * in Production

SELECT * is fine for exploration, but never use it in application code. It fetches every column — including large TEXT fields like bullet_points or ad_image_url — wasting bandwidth and memory. Always list the specific columns your application needs.

2. INSERT — Creating Records

INSERT adds new rows to a table. When an advertiser creates a new campaign through the portal UI, the backend executes an INSERT. PostgreSQL offers a powerful RETURNING clause that gives you back the generated ID without a separate query.

sql 
-- Insert a single campaign
INSERT INTO campaigns (account_id, user_id, lead_type_id, product_type_id, brand_id,
                        name, active, monthly_budget, base_price, timezone,
                        created_at, updated_at)
VALUES (
    1, 1, 1, 1, 1,
    'Auto Insurance — California',
    TRUE,
    15000.00,
    12.50,
    'America/Los_Angeles',
    NOW(), NOW()
)
RETURNING id, name;

-- Insert multiple rows in one statement
INSERT INTO accounts (company_name, status, timezone, created_at, updated_at)
VALUES
    ('Acme Insurance Group',  'active',     'America/New_York',    NOW(), NOW()),
    ('Pacific Shield Agents', 'active',     'America/Los_Angeles', NOW(), NOW()),
    ('Midwest Coverage LLC',  'onboarding', 'America/Chicago',     NOW(), NOW())
RETURNING id, company_name, status;

-- INSERT with ON CONFLICT (upsert) — prevent duplicate entries
INSERT INTO ad_group_stats (campaign_id, ad_group_id, dt, total_matches, total_searches,
                            created_at, updated_at)
VALUES (1, 5, '2025-03-15', 142, 1580, NOW(), NOW())
ON CONFLICT (campaign_id, dt, ad_group_id)
DO UPDATE SET
    total_matches  = EXCLUDED.total_matches,
    total_searches = EXCLUDED.total_searches,
    updated_at     = NOW();
Pro Tip: RETURNING

The RETURNING clause is a PostgreSQL superpower. Instead of doing INSERT + SELECT to get the new row's ID, do INSERT ... RETURNING id. This is atomic and faster — the ad portal uses it on every record creation to immediately return the new ID to the frontend.

3. UPDATE and DELETE

UPDATE modifies existing rows. DELETE removes them. In the ad portal, UPDATE is far more common than DELETE because the platform uses soft deletes — setting discarded_at to the current timestamp instead of physically removing the row.

sql 
-- Pause a campaign (set active to false)
UPDATE campaigns
SET active = FALSE, updated_at = NOW()
WHERE id = 42;

-- Increase all bid prices by 10% for a specific account
UPDATE ad_groups
SET
    base_bid_price = base_bid_price * 1.10,
    updated_at = NOW()
WHERE account_id = 7
  AND active = TRUE
RETURNING id, ad_group_name, base_bid_price;

-- Soft delete — the ad portal pattern
UPDATE campaigns
SET discarded_at = NOW(), updated_at = NOW()
WHERE id = 42;

-- Hard delete (rarely used in production ad-tech)
DELETE FROM ad_group_notes
WHERE ad_group_id = 15
  AND created_at < '2024-01-01';
Always Use WHERE with UPDATE/DELETE

An UPDATE or DELETE without a WHERE clause affects every row in the table. Running UPDATE campaigns SET active = FALSE; without a WHERE would deactivate all campaigns for all advertisers. In production, always wrap destructive operations in a transaction and verify with a SELECT first.

4. WHERE Clause and Operators

The WHERE clause is where SQL becomes powerful. It supports comparison operators, logical operators, pattern matching, range checks, and NULL handling. In ad-tech, you combine these to build precise data queries for dashboards, reports, and debugging.

sql 
-- Comparison operators: =, !=, <, >, <=, >=
SELECT * FROM campaigns
WHERE monthly_budget >= 5000.00;

-- AND / OR — combine conditions
SELECT name, monthly_budget, active
FROM campaigns
WHERE active = TRUE
  AND monthly_budget > 5000.00
  AND created_at >= NOW() - INTERVAL '30 days';

-- IN — match against a list of values
SELECT * FROM accounts
WHERE status IN ('active', 'onboarding');

-- BETWEEN — range check (inclusive)
SELECT * FROM ad_groups
WHERE base_bid_price BETWEEN 5.00 AND 25.00;

-- LIKE — pattern matching
SELECT * FROM campaigns
WHERE name LIKE '%Auto Insurance%';

-- ILIKE — case-insensitive (PostgreSQL extension)
SELECT * FROM campaigns
WHERE name ILIKE '%auto insurance%';

-- NOT — negate any condition
SELECT * FROM campaigns
WHERE status NOT IN ('paused', 'archived');

-- EXISTS — check for related rows (efficient for large tables)
SELECT c.name
FROM campaigns c
WHERE EXISTS (
    SELECT 1 FROM leads l
    WHERE l.campaign_id = c.id
      AND l.created_at >= NOW() - INTERVAL '7 days'
);

5. NULL Handling

NULL represents "unknown" or "missing" in SQL. It is not the same as empty string or zero. In the ad portal, discarded_at IS NULL means "this record is active" while discarded_at IS NOT NULL means "this record was soft-deleted." Mishandling NULLs is one of the most common sources of bugs.

sql 
-- Find active (non-discarded) campaigns
SELECT * FROM campaigns
WHERE discarded_at IS NULL;

-- Find soft-deleted campaigns
SELECT * FROM campaigns
WHERE discarded_at IS NOT NULL;

-- WRONG: This will NEVER return rows! NULL != NULL
SELECT * FROM campaigns
WHERE discarded_at = NULL;  -- BUG! Always use IS NULL

-- COALESCE — replace NULL with a default
SELECT
    name,
    COALESCE(monthly_budget, 0.00) AS budget,
    COALESCE(timezone, 'America/New_York') AS tz
FROM campaigns;

-- NULLIF — return NULL if two values are equal
-- Useful to avoid division by zero
SELECT
    total_matches::numeric / NULLIF(total_searches, 0) AS match_rate
FROM ad_group_stats;
The Soft Delete Pattern

The ad portal never physically deletes campaigns, accounts, or ad groups. Instead, it sets discarded_at = NOW(). This preserves historical data for reporting and enables "undo" functionality. Every query in the application includes WHERE discarded_at IS NULL to filter out deleted records. This is why the schema has discarded_at columns on nearly every table with corresponding indexes.

6. Sorting and Pagination

sql 
-- ORDER BY — sort results
SELECT name, monthly_budget, created_at
FROM campaigns
WHERE active = TRUE
ORDER BY monthly_budget DESC;

-- Multiple sort columns
SELECT name, active, monthly_budget
FROM campaigns
ORDER BY active DESC, monthly_budget DESC;

-- NULLS FIRST / NULLS LAST — control NULL ordering
SELECT name, max_price
FROM campaigns
ORDER BY max_price DESC NULLS LAST;

-- Pagination with LIMIT + OFFSET
-- Page 1 (rows 1-20)
SELECT * FROM campaigns
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- Page 2 (rows 21-40)
SELECT * FROM campaigns
ORDER BY created_at DESC
LIMIT 20 OFFSET 20;
Keyset Pagination for Large Tables

OFFSET becomes slow on large tables because PostgreSQL must skip N rows. For the leads table with millions of rows, use keyset pagination instead: WHERE id > :last_seen_id ORDER BY id LIMIT 20. This uses the index directly and performs consistently regardless of page number.

Self-Check

What happens if you run UPDATE without a WHERE clause?

It updates every row in the table. This is almost never intentional and can be catastrophic in production. Always include a WHERE clause and test with a SELECT first.

Why does WHERE status = NULL never match any rows?

NULL represents "unknown." Any comparison with NULL (including =, !=, <) returns NULL (unknown), not TRUE. You must use IS NULL or IS NOT NULL. This is the SQL standard three-valued logic.

When would you use ON CONFLICT instead of a regular INSERT?

Use ON CONFLICT (upsert) when you might be inserting a row that violates a unique constraint. For example, the ad portal inserts daily stats per ad group. If stats for that day already exist, it updates the existing row instead of failing. This is atomic and avoids race conditions.
