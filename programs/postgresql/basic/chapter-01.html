Chapter 01 · Phase 1: SQL Fundamentals

Introduction to PostgreSQL

PostgreSQL powers some of the world's most demanding applications. Before you can build anything, you need to understand its architecture, install it correctly, and write your first queries with confidence.

Estimated: 2–3 days
3 Advancement Levels
3 Assignments

Why This Matters

PostgreSQL is not just another database — it is the most advanced open-source relational database in the world. It powers Instagram, Spotify, Apple, and the US Federal Aviation Administration. When companies need a database that will not lose data, will not buckle under concurrency, and will not lock them into a vendor, they choose PostgreSQL.

Understanding PostgreSQL from the ground up means understanding how data actually works in production systems. Every web framework, every ORM, every migration tool is an abstraction over SQL. When those abstractions leak — and they always do — the engineer who understands the database underneath is the one who fixes the outage at 2 AM.

This chapter gives you the foundation: installing PostgreSQL, navigating the psql command-line interface, understanding databases and schemas, learning core data types, and writing your very first queries. Everything in the remaining seven chapters builds on top of what you learn here.

Key Insight

PostgreSQL is not a "beginner database" that you graduate from. Companies run multi-petabyte PostgreSQL clusters in production. Learning it properly now means you are learning the same tool you will use at scale. There is no "real" database you switch to later — this is the real database.

Key Concepts

1. What Is PostgreSQL

PostgreSQL (often called "Postgres") is an open-source, object-relational database management system (ORDBMS) that has been in active development since 1986. It originated from the POSTGRES project at UC Berkeley, led by Professor Michael Stonebraker, and has evolved into one of the most reliable and feature-rich databases available.

At its core, PostgreSQL is an ACID-compliant database. ACID stands for Atomicity, Consistency, Isolation, and Durability — four properties that guarantee your data remains correct even when things go wrong:

  • Atomicity — A transaction either completes entirely or has no effect at all. No half-written records.
  • Consistency — Every transaction moves the database from one valid state to another. Constraints are always enforced.
  • Isolation — Concurrent transactions do not interfere with each other. PostgreSQL uses MVCC (Multi-Version Concurrency Control) to achieve this without heavy locking.
  • Durability — Once a transaction is committed, it survives power failures, crashes, and errors. Data is written to the Write-Ahead Log (WAL) before being confirmed.
PostgreSQL vs MySQL

MySQL is popular because it was historically easier to set up. PostgreSQL is chosen when correctness, advanced features (JSON, arrays, window functions, CTEs), and standards compliance matter. If you are building anything beyond a toy application, PostgreSQL is the safer long-term bet. MySQL has improved significantly, but PostgreSQL's extensibility and SQL standard adherence remain unmatched.

2. Installation and Setup

There are multiple ways to install PostgreSQL depending on your operating system. We will cover the three most common approaches: Homebrew on macOS, apt on Ubuntu/Debian, and Docker for any platform.

bash 
# ── macOS with Homebrew ──
brew install postgresql@16
brew services start postgresql@16

# Verify installation
psql --version
# psql (PostgreSQL) 16.x
bash 
# ── Ubuntu/Debian with apt ──
sudo apt update
sudo apt install postgresql postgresql-contrib

# The service starts automatically
sudo systemctl status postgresql

# Switch to the postgres user and open psql
sudo -u postgres psql
bash 
# ── Docker (any platform) ──
docker run --name pg-learn \
  -e POSTGRES_PASSWORD=learnpg \
  -e POSTGRES_USER=student \
  -e POSTGRES_DB=learn_db \
  -p 5432:5432 \
  -d postgres:16-alpine

# Connect to it
psql -h localhost -U student -d learn_db
Pro Tip

Docker is the recommended approach for learning. It keeps your system clean, lets you destroy and recreate databases in seconds, and ensures you are running the exact version used in this course. If something breaks, docker rm -f pg-learn and start fresh.

3. psql CLI Basics

psql is PostgreSQL's interactive terminal. It is the most powerful way to interact with your database — faster than any GUI for everyday tasks. Learning the key meta-commands will save you hours over the course of this program.

sql 
-- Connect to a database
-- psql -h localhost -U student -d learn_db

-- Once inside psql, use backslash commands:

\l              -- List all databases
\c learn_db     -- Connect to a specific database
\dt             -- List all tables in current schema
\d users        -- Describe the 'users' table structure
\dn             -- List all schemas
\du             -- List all roles/users
\di             -- List all indexes
\df             -- List all functions
\timing         -- Toggle query execution timing on/off
\x              -- Toggle expanded display (vertical output)
\q              -- Quit psql

  psql Session Workflow
  =====================

  $ psql -h localhost -U student -d learn_db
         |
         v
  +----------------------------------+
  |  learn_db=#                      |
  |                                  |
  |  \l    --> list databases        |
  |  \dt   --> list tables           |
  |  \d t  --> describe table t      |
  |  \x    --> toggle expanded view  |
  |                                  |
  |  SELECT * FROM users;            |
  |  --------------------            |
  |  (results appear here)           |
  |                                  |
  |  \q    --> quit                  |
  +----------------------------------+
Figure 1: Typical psql session showing meta-commands and query execution

The \timing command is especially useful. It shows how long each query takes, which will become critical when you start optimizing performance in later chapters. Get in the habit of turning it on at the start of every session.

4. Databases and Schemas

PostgreSQL has a three-level namespace hierarchy: cluster → database → schema → object. A cluster is a single PostgreSQL server instance. Each cluster can contain multiple databases, and each database can contain multiple schemas. Tables, views, functions, and other objects live inside schemas.

sql 
-- Create a new database
CREATE DATABASE ecommerce_dev;

-- Connect to it (\c in psql)
\c ecommerce_dev

-- Create schemas for organizing tables
CREATE SCHEMA inventory;
CREATE SCHEMA billing;

-- Create a table inside a specific schema
CREATE TABLE inventory.products (
    id    serial PRIMARY KEY,
    name  text   NOT NULL,
    price numeric(10, 2)
);

-- Set search_path so you can omit the schema prefix
SET search_path TO inventory, public;

-- Now you can query without the schema prefix
SELECT * FROM products;

-- Check current search_path
SHOW search_path;
Why Schemas?

Schemas let you organize tables into logical groups without creating separate databases. This is especially useful in multi-tenant applications where each tenant might have their own schema, or in large applications where you want to separate billing tables from analytics tables. The default schema is public, and that is where tables go if you do not specify one.

5. Data Types

PostgreSQL has one of the richest type systems of any database. Choosing the right data type prevents bugs, saves storage, and enables the query optimizer to make better decisions. Here are the types you will use most often:

sql 
-- ── Numeric Types ──
CREATE TABLE type_demo_numeric (
    -- Integers: use the smallest type that fits
    small_val   smallint,        -- 2 bytes: -32768 to 32767
    regular_val integer,         -- 4 bytes: -2.1B to 2.1B (most common)
    big_val     bigint,          -- 8 bytes: +/- 9.2 quintillion

    -- Auto-incrementing shortcuts
    id          serial,          -- integer + auto-increment sequence
    big_id      bigserial,       -- bigint + auto-increment sequence

    -- Exact decimal: ALWAYS use for money
    price       numeric(10, 2), -- 10 total digits, 2 after decimal

    -- Floating point: for scientific data only
    score       real,            -- 4-byte float
    ratio       double precision  -- 8-byte float
);

-- ── Text Types ──
CREATE TABLE type_demo_text (
    country_code char(2),        -- fixed-length, padded with spaces
    username     varchar(50),    -- variable-length with max limit
    bio          text            -- unlimited length (preferred)
);

-- ── Date/Time Types ──
CREATE TABLE type_demo_time (
    created_at  timestamptz,    -- with time zone (ALWAYS prefer this)
    logged_at   timestamp,      -- without time zone (avoid if possible)
    birth_date  date,           -- date only: 2024-03-15
    start_time  time,           -- time only: 14:30:00
    duration    interval        -- span: '2 hours 30 minutes'
);

-- ── Boolean ──
CREATE TABLE type_demo_bool (
    is_active   boolean DEFAULT true,
    is_verified boolean DEFAULT false
);

-- ── Arrays and JSON ──
CREATE TABLE type_demo_advanced (
    tags        text[],           -- array of text values
    scores      integer[],        -- array of integers
    metadata    jsonb,            -- binary JSON (indexable, preferred)
    config      json              -- text JSON (preserves key order)
);

-- ── UUID ──
CREATE TABLE type_demo_uuid (
    id uuid DEFAULT gen_random_uuid() PRIMARY KEY
);
Never Use float for Money

Floating point numbers introduce rounding errors. 0.1 + 0.2 = 0.30000000000000004 in IEEE 754 arithmetic. For any financial data — prices, balances, transaction amounts — always use numeric or decimal. They are exact types that store every digit precisely.

6. Your First Queries

Now let us put everything together. We will create a real table, insert data, and query it. This is the foundation of every database interaction you will ever write.

sql 
-- Create a table for an e-commerce product catalog
CREATE TABLE products (
    id          serial       PRIMARY KEY,
    name        text         NOT NULL,
    category    varchar(50)  NOT NULL,
    price       numeric(10,2) NOT NULL,
    in_stock    boolean      DEFAULT true,
    created_at  timestamptz  DEFAULT now()
);

-- Insert sample products
INSERT INTO products (name, category, price) VALUES
    ('Mechanical Keyboard', 'Electronics', 149.99),
    ('Standing Desk',       'Furniture',   599.00),
    ('USB-C Hub',           'Electronics', 49.99),
    ('Ergonomic Mouse',     'Electronics', 79.50),
    ('Monitor Arm',         'Furniture',   129.00),
    ('Webcam HD',           'Electronics', 89.99);
sql 
-- Query all products
SELECT * FROM products;

-- Query specific columns
SELECT name, price FROM products;

-- Filter with WHERE
SELECT name, price
FROM   products
WHERE  category = 'Electronics';

-- Multiple conditions with AND
SELECT name, price
FROM   products
WHERE  category = 'Electronics'
  AND  price < 100;

-- Sort results
SELECT   name, price
FROM     products
ORDER BY price DESC;

-- Count rows
SELECT count(*) AS total_products FROM products;

-- Computed columns with aliases
SELECT
    name,
    price,
    price * 1.1 AS price_with_tax
FROM     products
ORDER BY price_with_tax DESC;

-- LIMIT and OFFSET for pagination
SELECT   name, price
FROM     products
ORDER BY price ASC
LIMIT    3
OFFSET   0;

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer feels uncertain, revisit the relevant section.

Can you explain the difference between text, varchar(n), and char(n)?

char(n) is fixed-length and pads with spaces — use it only for codes like ISO country codes (e.g., char(2)). varchar(n) is variable-length with a maximum — use it when you want to enforce a length limit. text is variable-length with no limit — it is the preferred type for most string data in PostgreSQL. Internally, PostgreSQL stores text and varchar identically, so there is no performance difference.

What is the difference between a database and a schema?

A database is a completely isolated container — you cannot query across databases in a single SQL statement (without extensions like dblink). A schema is a namespace within a database — you can freely JOIN tables across schemas. Use separate databases for unrelated applications. Use schemas to organize tables within a single application (e.g., billing, analytics, public).

Why should you always use timestamptz instead of timestamp?

timestamp (without time zone) stores the literal date and time you give it — it has no concept of time zones. If a user in Tokyo and a user in New York both insert "2024-01-15 09:00:00", the database stores the same value even though they mean different instants in time. timestamptz converts every value to UTC on storage and converts back to the session's time zone on retrieval. This means you always know exactly when something happened, regardless of where the query runs.
