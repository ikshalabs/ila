Chapter 01 · PostgreSQL Basics

Introduction to PostgreSQL

Install PostgreSQL, connect with psql and pgAdmin, create your first database and schema, and understand the data types that power every table in a production ad-tech platform.

Estimated: 2–3 hours
6 Key Concepts
3 Assignments

Why This Matters

Every ad-tech platform is fundamentally a data platform. When an advertiser sets a bid of $4.25 per lead, when a click event fires at 2:03:17.482 AM, when a campaign's daily budget ticks down from $500.00 to $496.75 — all of that lives in PostgreSQL. The database is not a detail; it is the system.

PostgreSQL (often called "Postgres") is the most advanced open-source relational database in the world. It powers companies from Apple and Instagram to Stripe and the ad bidding platforms you will build in this course. Understanding how to install it, connect to it, and speak its language is the non-negotiable foundation for everything that follows.

Key Insight

In an ad bidding system, a single mistyped data type — say, storing bid prices as integer instead of numeric(20,2) — can silently truncate $4.25 to $4, costing advertisers real money and destroying trust. Getting the fundamentals right here prevents entire categories of production bugs.

Key Concepts

1. What Is PostgreSQL?

PostgreSQL is an object-relational database management system (ORDBMS) that has been actively developed for over 35 years. Unlike MySQL, which was originally designed for speed over correctness, PostgreSQL was built from the ground up for data integrity, extensibility, and SQL standards compliance.

For an ad-tech platform, PostgreSQL offers critical capabilities that simpler databases lack:

  • ACID transactions — When a bid is placed and a budget is decremented, both happen or neither does. No half-states.
  • Rich data types — JSONB for flexible lead data, arrays for targeting zip codes, numeric for precise monetary values.
  • Advanced indexing — GIN indexes for JSONB queries, partial indexes for active-only campaigns, expression indexes for case-insensitive search.
  • Concurrency (MVCC) — Hundreds of simultaneous bid requests never block each other. Readers never block writers.

2. Installation

There are multiple ways to install PostgreSQL depending on your operating system. For development on an ad-tech platform, we recommend Docker for consistent environments across the team, though native installations are fine for learning.

bash 
# macOS — Install with Homebrew
brew install postgresql@16
brew services start postgresql@16

# Ubuntu / Debian
sudo apt update
sudo apt install postgresql postgresql-contrib
sudo systemctl start postgresql
sudo systemctl enable postgresql

# Docker — Recommended for team consistency
docker run --name ad_portal_pg \
  -e POSTGRES_USER=ad_portal \
  -e POSTGRES_PASSWORD=dev_password \
  -e POSTGRES_DB=ad_portal_development \
  -p 5432:5432 \
  -d postgres:16-alpine
Version Matters

Always pin your PostgreSQL version (e.g., postgres:16-alpine, not postgres:latest). In a real ad portal, a surprise upgrade from Postgres 15 to 16 on one developer's machine can cause subtle query plan differences that waste hours of debugging.

3. Connecting with psql

psql is the official command-line client for PostgreSQL. It is the tool you will use most often during development and debugging. Every ad-tech engineer should be comfortable navigating databases, running queries, and inspecting schemas directly from the terminal.

bash 
# Connect to your local PostgreSQL
psql -U ad_portal -d ad_portal_development

# Connect to Docker PostgreSQL
psql -h localhost -p 5432 -U ad_portal -d ad_portal_development

# Using a connection string (common in application configs)
psql "postgresql://ad_portal:dev_password@localhost:5432/ad_portal_development"

Once inside psql, these meta-commands become second nature:

psql 
-- List all databases
\l

-- Connect to a specific database
\c ad_portal_development

-- List all tables in current schema
\dt

-- Describe a table's columns and types
\d campaigns

-- List all schemas
\dn

-- Show current connection info
\conninfo

-- Execute a SQL file
\i /path/to/seed_campaigns.sql

-- Toggle expanded display (useful for wide tables)
\x

-- Show timing for queries
\timing

-- Quit psql
\q
Pro Tip

Set PSQL_PAGER='less -S' in your shell profile to get horizontal scrolling instead of wrapping. This makes wide result sets from tables like ad_group_stats much easier to read. Also add \pset null '(null)' to your ~/.psqlrc so NULL values are visually distinct from empty strings.

4. pgAdmin — The GUI Alternative

While psql is the power tool, pgAdmin provides a graphical interface that is especially helpful when exploring unfamiliar schemas. When you first join an ad-tech team and need to understand how campaigns, ad_groups, and leads relate, pgAdmin's ERD view is invaluable.

bash 
# Run pgAdmin alongside PostgreSQL with Docker
docker run --name pgadmin \
  -e PGADMIN_DEFAULT_EMAIL=dev@adportal.com \
  -e PGADMIN_DEFAULT_PASSWORD=admin \
  -p 5050:80 \
  --link ad_portal_pg:postgres \
  -d dpage/pgadmin4

Navigate to http://localhost:5050, add a server connection with host postgres, port 5432, and the credentials from your database container. The query tool supports syntax highlighting, auto-complete, and result export to CSV — features you will use constantly when debugging campaign performance queries.

5. Databases and Schemas

PostgreSQL has a three-level namespace: cluster (server) → database → schema. Every database has a public schema by default.

sql 
-- Create a database for our ad portal
CREATE DATABASE ad_portal_development;

-- Connect to it
\c ad_portal_development

-- Create a schema for reporting (separate from operational tables)
CREATE SCHEMA reports;

-- Create a table in the reports schema
CREATE TABLE reports.daily_campaign_summary (
    campaign_id   BIGINT NOT NULL,
    report_date   DATE NOT NULL,
    total_leads   INTEGER DEFAULT 0,
    total_spend   NUMERIC(12,2) DEFAULT 0.00,
    avg_bid_price NUMERIC(8,2),
    PRIMARY KEY (campaign_id, report_date)
);

-- List tables across all schemas
\dt *.*

-- Set search path so you don't need to prefix schema names
SET search_path TO public, reports;
Why Schemas Matter in Ad-Tech

In a mature ad portal, operational tables (public.campaigns, public.leads) coexist with analytics tables (reports.daily_stats). Schemas let you grant different permissions — analysts get read-only access to reports while the application has full access to public. The real ad portal uses the public schema for all operational tables and enables extensions like btree_gin and plpgsql.

6. Essential Data Types for Ad-Tech

Choosing the right data type is one of the most consequential decisions in table design. Here are the types you will use constantly in the ad portal:

sql 
-- NUMERIC TYPES
BIGINT              -- 8 bytes. IDs, foreign keys. Range: up to 9.2 quintillion
INTEGER             -- 4 bytes. Counters: lead_count, monthly_volume
NUMERIC(p, s)       -- Exact decimal. ALWAYS for money: bids, budgets, balances
BOOLEAN             -- true/false. Flags: active, locked_price, enterprise

-- TEXT TYPES
VARCHAR(n)          -- Bounded string: status, timezone, disposition
TEXT                -- Unlimited: notes, bullet_points, ad_image_url

-- DATE/TIME TYPES
TIMESTAMP           -- Date+time, no timezone. created_at, updated_at
TIMESTAMPTZ         -- Date+time WITH timezone. Better for multi-tz portals
DATE                -- Date only. report_date, budget period boundaries

-- SPECIAL TYPES (covered in Advanced course)
JSONB               -- Binary JSON: lead data, smart_view_filters
TEXT[]              -- Array of text: languages, filter_value_array
UUID                -- Universally unique identifier: external-facing IDs

Here is a real table from the ad portal that demonstrates these types in action:

sql 
-- Mirrors the real campaigns table from the ad portal
CREATE TABLE campaigns (
    id                  BIGSERIAL PRIMARY KEY,
    account_id          BIGINT NOT NULL,
    user_id             BIGINT NOT NULL,
    lead_type_id        BIGINT NOT NULL,
    product_type_id     BIGINT NOT NULL,
    brand_id            BIGINT NOT NULL,
    name                VARCHAR(255),
    active              BOOLEAN,
    monthly_budget      NUMERIC(20,2),
    monthly_volume      INTEGER,
    base_price          NUMERIC(12,2) DEFAULT 0.00,
    min_price           NUMERIC(12,2),
    max_price           NUMERIC(12,2),
    timezone            VARCHAR(50),
    shared_lead_type    BOOLEAN DEFAULT TRUE,
    locked_price        BOOLEAN DEFAULT FALSE,
    discarded_at        TIMESTAMP,
    created_at          TIMESTAMP NOT NULL,
    updated_at          TIMESTAMP NOT NULL
);

-- BIGSERIAL = auto-incrementing BIGINT (1, 2, 3, ...)
-- NUMERIC(20,2) for budget: up to $999,999,999,999,999,999.99
-- discarded_at: soft-delete pattern (NULL = active, timestamp = deleted)
Never Use FLOAT for Money

FLOAT and DOUBLE PRECISION use binary floating-point which cannot exactly represent most decimal fractions. 0.1 + 0.2 becomes 0.30000000000000004. In an ad portal processing millions of dollars, tiny errors compound into real discrepancies. Always use NUMERIC for bids, budgets, balances.

Self-Check

Before moving to Chapter 02, honestly answer these questions.

Can you explain why NUMERIC is used instead of FLOAT for bid prices?

FLOAT uses binary floating-point which cannot exactly store most decimal fractions. 0.1 + 0.2 becomes 0.30000000000000004. When summing millions of bid values, errors compound into real financial discrepancies. NUMERIC(20,2) stores exact decimal values — essential for an ad bidding platform.

What is the difference between a database and a schema?

A database is a completely isolated container — you cannot join tables across databases. A schema is a namespace within a database — tables in different schemas can be joined freely. Use databases for truly independent applications and schemas for logical groupings within the same application.

When should you use BIGINT vs INTEGER for a primary key?

INTEGER tops out at ~2.1 billion. A high-volume leads table processing millions per month could exhaust it. BIGINT supports 9.2 quintillion values. The 4 extra bytes per row are trivial compared to the cost of running out of IDs in production. The ad portal uses BIGSERIAL for all primary keys.
