Chapter 04 · PostgreSQL Basic

Joins and Subqueries

Connect normalized tables with INNER, LEFT, RIGHT, and FULL joins. Build complex queries with self-joins, correlated subqueries, CTEs, and UNION for real ad-tech analytics.

Estimated: 3–4 hours
5 Key Concepts
3 Assignments

Why This Matters

Real-world ad-tech queries almost never touch a single table. When a campaign manager asks "show me all active campaigns for Acme Insurance with their total lead count," that answer lives across three tables: accounts, campaigns, and leads. JOINs are the mechanism that reunites normalized data into useful results.

Subqueries and CTEs (Common Table Expressions) let you break complex analytical questions into readable, composable steps. "Which campaigns have a higher-than-average bid price?" or "Find accounts with no leads in the last 30 days" — these require thinking in layers, and CTEs make that thinking explicit.

Key Insight

In the ad portal, the most common query pattern is a multi-table JOIN filtering by account_id with a WHERE clause on discarded_at IS NULL. Understanding INNER JOIN vs LEFT JOIN determines whether you see "campaigns with leads" or "all campaigns, including those with zero leads." That distinction changes business decisions.

Key Concepts

1. INNER JOIN

INNER JOIN returns only rows where the join condition matches in both tables. If a campaign has no leads, it will not appear in an INNER JOIN between campaigns and leads.

sql
-- List campaigns with their account names
SELECT
    a.company_name,
    c.name AS campaign_name,
    c.monthly_budget,
    c.active
FROM campaigns c
INNER JOIN accounts a ON a.id = c.account_id
WHERE c.discarded_at IS NULL
  AND a.discarded_at IS NULL
ORDER BY c.monthly_budget DESC;

-- Three-table join: accounts -> campaigns -> ad_groups
SELECT
    a.company_name,
    c.name AS campaign_name,
    ag.ad_group_name,
    ag.base_bid_price
FROM ad_groups ag
INNER JOIN campaigns c ON c.id = ag.campaign_id
INNER JOIN accounts a ON a.id = ag.account_id
WHERE ag.active = TRUE
  AND ag.discarded_at IS NULL;

2. LEFT JOIN and RIGHT JOIN

LEFT JOIN returns all rows from the left table, even if there is no match in the right table. This is essential when you want to see "all campaigns, including those with zero leads."

sql
-- All campaigns with lead count (including campaigns with 0 leads)
SELECT
    c.name,
    c.monthly_budget,
    COUNT(l.id) AS lead_count
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.id
WHERE c.discarded_at IS NULL
GROUP BY c.id, c.name, c.monthly_budget
ORDER BY lead_count DESC;

-- Find campaigns with NO leads (orphaned campaigns)
SELECT c.name, c.created_at
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.id
WHERE l.id IS NULL
  AND c.discarded_at IS NULL;
LEFT JOIN vs INNER JOIN

The choice between LEFT and INNER JOIN is a business decision, not a technical one. Use INNER JOIN when missing relationships mean the row is invalid. Use LEFT JOIN when you need to count zeroes — campaigns with no leads, accounts with no campaigns. In ad-tech dashboards, showing zero is often more informative than hiding the row entirely.

3. Self-Joins

A self-join joins a table to itself. In the ad portal, the admin_users table has a manager_id that references another admin_user. To list employees with their manager's name, you self-join.

sql
-- List admin users with their manager's name
SELECT
    emp.first_name || ' ' || emp.last_name AS employee,
    emp.job_title,
    mgr.first_name || ' ' || mgr.last_name AS manager
FROM admin_users emp
LEFT JOIN admin_users mgr ON mgr.id = emp.manager_id
WHERE emp.discarded_at IS NULL;

-- Compare bid prices between ad groups in the same campaign
SELECT
    ag1.ad_group_name AS group_a,
    ag1.base_bid_price AS bid_a,
    ag2.ad_group_name AS group_b,
    ag2.base_bid_price AS bid_b
FROM ad_groups ag1
INNER JOIN ad_groups ag2
    ON ag1.campaign_id = ag2.campaign_id
    AND ag1.id < ag2.id
WHERE ag1.active = TRUE AND ag2.active = TRUE;

4. Subqueries and Correlated Subqueries

sql
-- Subquery in WHERE: campaigns with above-average budget
SELECT name, monthly_budget
FROM campaigns
WHERE monthly_budget > (
    SELECT AVG(monthly_budget) FROM campaigns
    WHERE discarded_at IS NULL
);

-- Correlated subquery: each campaign's lead count vs account average
SELECT
    c.name,
    (SELECT COUNT(*) FROM leads l WHERE l.campaign_id = c.id) AS my_leads,
    (SELECT AVG(cnt) FROM (
        SELECT COUNT(*) AS cnt FROM leads
        WHERE account_id = c.account_id
        GROUP BY campaign_id
    ) sub) AS account_avg
FROM campaigns c
WHERE c.discarded_at IS NULL;

-- IN subquery: accounts that have at least one active campaign
SELECT company_name
FROM accounts
WHERE id IN (
    SELECT DISTINCT account_id
    FROM campaigns
    WHERE active = TRUE AND discarded_at IS NULL
);

5. CTEs and UNION

sql
-- CTE: break complex logic into named steps
WITH campaign_stats AS (
    SELECT
        c.id,
        c.name,
        c.account_id,
        c.monthly_budget,
        COUNT(l.id) AS lead_count,
        COALESCE(SUM(c.base_price), 0) AS total_revenue
    FROM campaigns c
    LEFT JOIN leads l ON l.campaign_id = c.id
    WHERE c.discarded_at IS NULL
    GROUP BY c.id
),
account_summary AS (
    SELECT
        a.company_name,
        SUM(cs.lead_count) AS total_leads,
        SUM(cs.monthly_budget) AS total_budget,
        COUNT(cs.id) AS campaign_count
    FROM campaign_stats cs
    JOIN accounts a ON a.id = cs.account_id
    GROUP BY a.company_name
)
SELECT * FROM account_summary
ORDER BY total_leads DESC;

-- UNION: combine results from different queries
SELECT 'active' AS segment, COUNT(*) AS total
FROM campaigns WHERE active = TRUE AND discarded_at IS NULL
UNION ALL
SELECT 'paused', COUNT(*)
FROM campaigns WHERE active = FALSE AND discarded_at IS NULL
UNION ALL
SELECT 'deleted', COUNT(*)
FROM campaigns WHERE discarded_at IS NOT NULL;
Pro Tip: CTEs vs Subqueries

CTEs are generally more readable than deeply nested subqueries. They also make debugging easier — you can run each CTE independently. However, PostgreSQL materializes CTEs in some versions, so for performance-critical queries, check the query plan with EXPLAIN ANALYZE (Chapter 06).

Self-Check

When should you use LEFT JOIN instead of INNER JOIN?

Use LEFT JOIN when you need to include rows from the left table that have no matching rows in the right table. In ad-tech, this is common when showing "all campaigns including those with zero leads" or "all accounts including those with no campaigns." INNER JOIN would hide those zero-count rows.

What is the difference between a CTE and a subquery?

A CTE (WITH clause) gives a name to a subquery, making it reusable and more readable. CTEs are especially valuable when the same derived result is referenced multiple times. Subqueries are inline and anonymous. For single-use cases, either works; for multi-step analysis, CTEs are strongly preferred.

Why is UNION ALL faster than UNION?

UNION removes duplicate rows, which requires sorting or hashing the entire result set. UNION ALL returns all rows without deduplication. In the ad portal, when you know segments are mutually exclusive (active/paused/deleted campaigns), use UNION ALL to avoid the unnecessary dedup cost.
