Chapter 04 · Phase 2: Multi-Table Queries

Joins and Subqueries

Real databases spread data across many tables. Joins let you recombine that data in powerful ways, while subqueries let you compose queries inside other queries. Together, they are the backbone of every non-trivial SQL operation you will ever write.

Estimated: 3–4 days
6 Key Concepts
3 Assignments

Why This Matters

In Chapter 03, you learned to separate data into normalized tables to eliminate redundancy. That normalization comes at a cost: the data you need for any single question is now spread across multiple tables. Joins are how you reassemble that data. Without joins, a normalized database is a collection of disconnected fragments. With joins, it becomes a richly interconnected system where any question can be answered with a single query.

Subqueries take this further by letting you nest one query inside another. Need to find customers who have placed orders above the average order value? That requires computing the average first (an inner query) and then filtering customers against it (an outer query). Subqueries make this kind of multi-step reasoning possible in a single SQL statement.

Understanding joins and subqueries is not optional for any engineer working with databases. Every ORM generates joins behind the scenes. Every reporting dashboard, every analytics query, every data migration — they all rely on joins. When the ORM-generated SQL is slow, the engineer who understands joins is the one who can diagnose and fix it.

Key Insight

A join is conceptually a Cartesian product (every row paired with every row) followed by a filter. PostgreSQL never actually materializes the full Cartesian product — the query planner uses hash joins, merge joins, or nested loops to compute only the matching rows. Understanding this mental model helps you write correct joins and reason about performance.

Key Concepts

1. INNER JOIN: The Default Join

An INNER JOIN returns only the rows that have matching values in both tables. If a customer has no orders, they will not appear in the result. If an order references a customer that does not exist (which foreign keys prevent), it will not appear either. This is the most common type of join and is the default when you write just JOIN.

sql 
-- Setup: our e-commerce tables
CREATE TABLE customers (
    id    serial PRIMARY KEY,
    name  text   NOT NULL,
    email text   UNIQUE NOT NULL
);

CREATE TABLE orders (
    id          serial      PRIMARY KEY,
    customer_id integer     NOT NULL REFERENCES customers(id),
    total       numeric(10,2) NOT NULL,
    ordered_at  timestamptz DEFAULT now()
);

INSERT INTO customers (name, email) VALUES
  ('Alice Chen',   'alice@example.com'),
  ('Bob Smith',    'bob@example.com'),
  ('Carol Davis',  'carol@example.com'),
  ('Dave Wilson',  'dave@example.com');

INSERT INTO orders (customer_id, total) VALUES
  (1, 250.00), (1, 89.99),
  (2, 450.00),
  (3, 120.50), (3, 75.00), (3, 310.00);
-- Dave has no orders

-- INNER JOIN: only customers WITH orders appear
SELECT
    c.name,
    c.email,
    o.id   AS order_id,
    o.total
FROM   customers c
INNER JOIN orders o ON o.customer_id = c.id
ORDER BY c.name, o.id;

2. LEFT JOIN and RIGHT JOIN

A LEFT JOIN returns all rows from the left table, even if there is no matching row in the right table. Non-matching rows get NULL values for the right table's columns. This is essential when you want to find "missing" data — customers without orders, products without reviews, employees without managers.

sql 
-- LEFT JOIN: ALL customers, even those without orders
SELECT
    c.name,
    c.email,
    o.id    AS order_id,
    o.total
FROM   customers c
LEFT JOIN orders o ON o.customer_id = c.id
ORDER BY c.name, o.id;
-- Dave now appears with NULL for order_id and total

-- Find customers who have NEVER placed an order
SELECT c.name, c.email
FROM   customers c
LEFT JOIN orders o ON o.customer_id = c.id
WHERE  o.id IS NULL;

-- Count orders per customer (including zero-order customers)
SELECT
    c.name,
    COUNT(o.id) AS order_count,
    COALESCE(SUM(o.total), 0) AS total_spent
FROM      customers c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY  c.id, c.name
ORDER BY  total_spent DESC;
LEFT JOIN + WHERE Pitfall

Putting a condition on the right table in the WHERE clause effectively converts a LEFT JOIN into an INNER JOIN, because NULL values from non-matching rows will fail the condition. If you need to filter the right table while preserving the LEFT JOIN behavior, put the condition in the ON clause instead: LEFT JOIN orders o ON o.customer_id = c.id AND o.total > 100.

3. CROSS JOIN and FULL OUTER JOIN

A CROSS JOIN produces every combination of rows from both tables (the Cartesian product). With 100 rows in table A and 50 rows in table B, you get 5,000 result rows. This is rarely used in application queries, but it is useful for generating test data or creating all possible combinations (like all products in all sizes).

A FULL OUTER JOIN returns all rows from both tables. Rows that match are combined. Rows from the left table with no match get NULLs for right columns, and vice versa. This is useful for reconciliation queries where you want to see everything that matches plus everything that does not.

sql 
-- CROSS JOIN: generate all product-size combinations
SELECT p.name, s.size_label
FROM   products p
CROSS JOIN (VALUES ('S'), ('M'), ('L'), ('XL')) AS s(size_label);

-- FULL OUTER JOIN: reconcile two data sources
CREATE TABLE system_a_users (email text PRIMARY KEY, name text);
CREATE TABLE system_b_users (email text PRIMARY KEY, name text);

SELECT
    COALESCE(a.email, b.email) AS email,
    a.name AS system_a_name,
    b.name AS system_b_name,
    CASE
        WHEN a.email IS NULL THEN 'Only in System B'
        WHEN b.email IS NULL THEN 'Only in System A'
        ELSE 'In Both Systems'
    END AS status
FROM      system_a_users a
FULL OUTER JOIN system_b_users b ON a.email = b.email;

4. Self Joins and Multi-Table Joins

A self join is when a table is joined to itself. This is common for hierarchical data like org charts (employees with managers), threaded comments, or category trees. You must use table aliases to distinguish the two "copies" of the table.

sql 
-- Self join: employees and their managers
CREATE TABLE employees (
    id         serial  PRIMARY KEY,
    name       text    NOT NULL,
    manager_id integer REFERENCES employees(id)
);

INSERT INTO employees (name, manager_id) VALUES
  ('CEO Sarah',    NULL),
  ('VP Mike',      1),
  ('VP Lisa',      1),
  ('Dev Alex',     2),
  ('Dev Jordan',   2),
  ('Design Kim',   3);

SELECT
    e.name       AS employee,
    m.name       AS manager
FROM      employees e
LEFT JOIN employees m ON e.manager_id = m.id
ORDER BY  m.name NULLS FIRST, e.name;

-- Multi-table join: orders with customer and product details
SELECT
    c.name       AS customer,
    o.id         AS order_id,
    p.name       AS product,
    oi.quantity,
    oi.unit_price,
    oi.quantity * oi.unit_price AS line_total
FROM       customers c
JOIN       orders o      ON o.customer_id = c.id
JOIN       order_items oi ON oi.order_id = o.id
JOIN       products p     ON p.id = oi.product_id
ORDER BY   c.name, o.id;
Pro Tip

When joining three or more tables, always use short, meaningful table aliases (c for customers, o for orders, oi for order_items). Without aliases, your queries become unreadable walls of text. With aliases, the intent is immediately clear: c.name is the customer name, p.name is the product name.

5. Subqueries: Scalar, Row, and Table

A subquery is a SELECT statement embedded inside another SQL statement. Subqueries come in three flavors based on what they return: a scalar subquery returns a single value, a row subquery returns a single row, and a table subquery returns a full result set.

sql 
-- Scalar subquery in WHERE: orders above average
SELECT id, customer_id, total
FROM   orders
WHERE  total > (
    SELECT AVG(total) FROM orders
);

-- Scalar subquery in SELECT: show each order's % of total revenue
SELECT
    id,
    total,
    ROUND(total / (SELECT SUM(total) FROM orders) * 100, 2) AS pct_of_revenue
FROM   orders
ORDER BY total DESC;

-- Table subquery in FROM (derived table)
SELECT
    cs.name,
    cs.order_count,
    cs.total_spent
FROM (
    SELECT
        c.name,
        COUNT(o.id) AS order_count,
        SUM(o.total) AS total_spent
    FROM      customers c
    JOIN      orders o ON o.customer_id = c.id
    GROUP BY  c.id, c.name
) cs
WHERE cs.total_spent > 200;

-- Subquery with IN: customers who ordered specific products
SELECT name, email
FROM   customers
WHERE  id IN (
    SELECT DISTINCT o.customer_id
    FROM   orders o
    JOIN   order_items oi ON oi.order_id = o.id
    WHERE  oi.product_id IN (1, 5, 12)
);
Correlated Subquery Performance

A correlated subquery references columns from the outer query, so it executes once per outer row. This can be extremely slow on large tables. For example, WHERE total > (SELECT AVG(total) FROM orders WHERE customer_id = o.customer_id) runs the inner query for every row in the outer query. When you can, rewrite correlated subqueries as JOINs or use CTEs with window functions instead.

6. Common Table Expressions (CTEs)

A Common Table Expression (CTE) is a named temporary result set defined with the WITH keyword. CTEs make complex queries readable by breaking them into logical steps. Think of them as declaring variables in a programming language — each CTE gives a name to an intermediate result that you can reference later.

sql 
-- CTE: find high-value customers and their most recent order
WITH customer_totals AS (
    SELECT
        customer_id,
        COUNT(*) AS order_count,
        SUM(total) AS lifetime_value
    FROM     orders
    GROUP BY customer_id
),
latest_orders AS (
    SELECT DISTINCT ON (customer_id)
        customer_id,
        id AS last_order_id,
        total AS last_order_total,
        ordered_at AS last_order_date
    FROM     orders
    ORDER BY customer_id, ordered_at DESC
)
SELECT
    c.name,
    ct.order_count,
    ct.lifetime_value,
    lo.last_order_total,
    lo.last_order_date
FROM      customer_totals ct
JOIN      customers c       ON c.id = ct.customer_id
JOIN      latest_orders lo  ON lo.customer_id = ct.customer_id
WHERE     ct.lifetime_value > 300
ORDER BY  ct.lifetime_value DESC;

-- CTE with EXISTS: find products that have been ordered
WITH ordered_products AS (
    SELECT DISTINCT product_id FROM order_items
)
SELECT p.name, p.price
FROM   products p
WHERE  EXISTS (
    SELECT 1 FROM ordered_products op
    WHERE op.product_id = p.id
);

-- Recursive CTE: walk the org chart
WITH RECURSIVE org_tree AS (
    -- Base case: start from the CEO (no manager)
    SELECT id, name, manager_id, 0 AS depth
    FROM   employees
    WHERE  manager_id IS NULL

    UNION ALL

    -- Recursive step: find employees of current level
    SELECT e.id, e.name, e.manager_id, ot.depth + 1
    FROM   employees e
    JOIN   org_tree ot ON e.manager_id = ot.id
)
SELECT
    REPEAT('  ', depth) || name AS org_chart,
    depth
FROM   org_tree
ORDER BY depth, name;
CTE vs Subquery

CTEs and subqueries produce the same results. The advantage of CTEs is readability: each CTE has a name and can be referenced multiple times. In PostgreSQL 12+, CTEs are automatically inlined by the optimizer (just like subqueries), so there is typically no performance difference. Use CTEs when they make your query easier to understand; use subqueries for simple, one-off filters.

Self-Check

Before moving to Chapter 05, honestly answer these questions. If any answer feels uncertain, revisit the relevant section.

What is the difference between INNER JOIN and LEFT JOIN?

INNER JOIN returns only rows that have matches in both tables. LEFT JOIN returns all rows from the left table, filling in NULLs for the right table columns when there is no match. Use LEFT JOIN when you need to include "missing" data — such as customers without orders or products without reviews.

When would you use a CTE instead of a subquery?

Use a CTE when you want to name an intermediate result for readability, when you need to reference the same intermediate result multiple times in the query, or when you need recursion (WITH RECURSIVE). For simple, one-off filters, a subquery in the WHERE clause is often more concise and equally readable.

What is a correlated subquery and why is it potentially slow?

A correlated subquery references columns from the outer query, causing the inner query to execute once per row of the outer query. For a table with 100,000 rows, this means 100,000 executions of the inner query. The fix is usually to rewrite it as a JOIN, which lets the optimizer choose an efficient strategy (hash join, merge join) that processes both tables in a single pass.

How does DISTINCT ON differ from DISTINCT?

DISTINCT removes entire duplicate rows. DISTINCT ON (expression) is a PostgreSQL extension that keeps only the first row for each unique value of the expression, as determined by the ORDER BY clause. It is extremely useful for "latest per group" queries: SELECT DISTINCT ON (customer_id) * FROM orders ORDER BY customer_id, ordered_at DESC returns the most recent order for each customer.

Can you write a recursive CTE? What are its two required parts?

A recursive CTE has two parts joined by UNION ALL: the base case (a non-recursive query that provides the starting rows) and the recursive step (a query that references the CTE itself to produce additional rows). Execution alternates between the two until the recursive step produces no new rows. Always include a depth counter or cycle detection to prevent infinite recursion.
