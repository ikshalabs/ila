Chapter 08 · Phase 4: Server-Side Logic

Views, Functions, and Triggers

Move logic into the database layer with views that simplify queries, PL/pgSQL functions that encapsulate business rules, and triggers that enforce data integrity automatically — no application code required.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

Every production database eventually needs logic that lives inside the database rather than in application code. Views abstract complex joins into simple table-like interfaces. Functions encapsulate calculations, validations, and multi-step operations into reusable units. Triggers respond automatically to data changes — enforcing audit trails, maintaining denormalized counters, and validating business rules that span multiple tables.

When you push the right logic into PostgreSQL, you get consistency guarantees that are impossible to achieve from application code alone. A trigger that records every row change into an audit table will fire regardless of whether the change came from your Rails app, a data migration script, or a DBA running a manual fix at 3 AM. No application middleware can provide that level of guarantee.

This is the final chapter of the PostgreSQL Fundamentals track. By the end, you will understand how to build database-layer abstractions that make your application code simpler, your data more consistent, and your system more resilient to bugs and operational mistakes.

Key Insight

The debate about "thin database vs. fat database" is a false dichotomy. The best systems use each layer for what it does best: the database enforces constraints, maintains audit trails, and provides pre-computed views. The application handles user interaction, business workflows, and orchestration. Learning where to draw that line is what separates senior engineers from juniors.

Key Concepts

1. CREATE VIEW — Virtual Tables

A view is a named SQL query stored in the database catalog. When you SELECT from a view, PostgreSQL executes the underlying query and returns the results as if it were a real table. Views do not store data themselves — they are purely virtual. They are invaluable for simplifying complex joins, restricting access to sensitive columns, and providing stable interfaces even as underlying table structures change.

sql 
-- Create tables for our examples
CREATE TABLE departments (
    id   serial PRIMARY KEY,
    name text UNIQUE NOT NULL
);

CREATE TABLE employees (
    id         serial PRIMARY KEY,
    first_name text NOT NULL,
    last_name  text NOT NULL,
    email      text UNIQUE NOT NULL,
    dept_id    integer REFERENCES departments(id),
    salary     numeric(10,2) NOT NULL,
    hired_at   date NOT NULL DEFAULT current_date
);

-- A simple view: employee directory with department names
CREATE VIEW v_employee_directory AS
SELECT
    e.id,
    e.first_name || ' ' || e.last_name AS full_name,
    e.email,
    d.name AS department,
    e.hired_at
FROM employees e
JOIN departments d ON d.id = e.dept_id;

-- Use the view exactly like a table
SELECT * FROM v_employee_directory
WHERE department = 'Engineering'
ORDER BY hired_at DESC;

Views can reference other views, and PostgreSQL will inline and optimize the entire query chain. You can also create updatable views — simple views on a single table that allow INSERT, UPDATE, and DELETE operations to pass through to the underlying table automatically.

sql 
-- A view that hides salary data for non-HR users
CREATE VIEW v_employee_public AS
SELECT id, first_name, last_name, email, dept_id, hired_at
FROM employees;

-- This view is auto-updatable (single table, no aggregation)
INSERT INTO v_employee_public (first_name, last_name, email, dept_id, hired_at)
VALUES ('Ada', 'Lovelace', 'ada@example.com', 1, '2024-01-15');

-- Replace a view definition (can add columns but not remove)
CREATE OR REPLACE VIEW v_employee_directory AS
SELECT
    e.id,
    e.first_name || ' ' || e.last_name AS full_name,
    e.email,
    d.name AS department,
    e.hired_at,
    current_date - e.hired_at AS days_employed
FROM employees e
JOIN departments d ON d.id = e.dept_id;

-- Drop a view safely
DROP VIEW IF EXISTS v_employee_public;
Views Are Not Caches

A regular view re-executes its query every time you select from it. If the underlying query is expensive (large joins, aggregations across millions of rows), the view will be slow every time. For caching behavior, use materialized views instead.

2. Materialized Views — Cached Query Results

A materialized view stores the results of its query physically on disk, just like a real table. The first query populates the data, and subsequent reads are fast because they come from the stored snapshot. The trade-off is staleness — you must explicitly refresh the materialized view to pick up changes from the underlying tables.

sql 
-- Create a materialized view for department statistics
CREATE MATERIALIZED VIEW mv_dept_stats AS
SELECT
    d.name                              AS department,
    count(e.id)                         AS employee_count,
    round(avg(e.salary), 2)              AS avg_salary,
    min(e.salary)                        AS min_salary,
    max(e.salary)                        AS max_salary,
    round(sum(e.salary), 2)              AS total_payroll
FROM departments d
LEFT JOIN employees e ON e.dept_id = d.id
GROUP BY d.name
WITH DATA;

-- Query it like a regular table (reads from stored data)
SELECT * FROM mv_dept_stats
ORDER BY total_payroll DESC;

-- Create an index on the materialized view for fast lookups
CREATE UNIQUE INDEX idx_mv_dept_stats_name
ON mv_dept_stats (department);

-- Refresh the data (blocks reads during refresh)
REFRESH MATERIALIZED VIEW mv_dept_stats;

-- Refresh CONCURRENTLY (allows reads during refresh)
-- Requires a unique index on the materialized view
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_dept_stats;

  Regular View vs. Materialized View
  ====================================

  Regular View:
  SELECT * FROM v_report
       |
       v
  [Re-execute full query]  <-- every single time
       |
       v
  [Return results]

  Materialized View:
  SELECT * FROM mv_report
       |
       v
  [Read stored snapshot]   <-- fast, like a table
       |
       v
  [Return results]

  REFRESH MATERIALIZED VIEW mv_report;
       |
       v
  [Re-execute query, replace stored data]
Figure 1: Regular views re-execute on every read; materialized views serve cached snapshots
Refreshing on a Schedule

Use pg_cron or your application's job scheduler (like Sidekiq or cron) to refresh materialized views periodically. A common pattern is to refresh analytics views every 15 minutes during business hours. REFRESH MATERIALIZED VIEW CONCURRENTLY is preferred in production because it does not block read queries during the refresh.

3. PL/pgSQL Functions

PL/pgSQL is PostgreSQL's built-in procedural language. It extends SQL with variables, control flow (IF/ELSE, loops), exception handling, and the ability to return tables, records, or scalar values. Functions written in PL/pgSQL run inside the database server, eliminating network round-trips and enabling atomic multi-step operations.

sql 
-- Simple scalar function: calculate annual bonus
CREATE OR REPLACE FUNCTION calc_bonus(
    emp_salary numeric,
    perf_rating integer  -- 1 to 5
)
RETURNS numeric AS $$
DECLARE
    bonus_pct numeric;
BEGIN
    CASE perf_rating
        WHEN 5 THEN bonus_pct := 0.20;
        WHEN 4 THEN bonus_pct := 0.15;
        WHEN 3 THEN bonus_pct := 0.10;
        WHEN 2 THEN bonus_pct := 0.05;
        ELSE          bonus_pct := 0.00;
    END CASE;

    RETURN round(emp_salary * bonus_pct, 2);
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Use the function in queries
SELECT
    first_name,
    salary,
    calc_bonus(salary, 4) AS bonus
FROM employees;
sql 
-- Function returning a table (set-returning function)
CREATE OR REPLACE FUNCTION get_dept_employees(
    dept_name text
)
RETURNS TABLE (
    employee_id   integer,
    full_name     text,
    salary        numeric,
    tenure_years  numeric
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        e.id,
        e.first_name || ' ' || e.last_name,
        e.salary,
        round(
            extract(epoch FROM age(e.hired_at)) / 31557600, 1
        )
    FROM employees e
    JOIN departments d ON d.id = e.dept_id
    WHERE d.name = dept_name
    ORDER BY e.salary DESC;
END;
$$ LANGUAGE plpgsql STABLE;

-- Call the function like a table in FROM clause
SELECT * FROM get_dept_employees('Engineering');

-- Join function results with other tables
SELECT de.full_name, de.salary, p.project_name
FROM get_dept_employees('Engineering') de
JOIN project_assignments pa ON pa.employee_id = de.employee_id
JOIN projects p ON p.id = pa.project_id;
Volatility Categories

Every function has a volatility: IMMUTABLE (always returns the same result for the same inputs — can be constant-folded), STABLE (returns consistent results within a single query — safe for index scans), or VOLATILE (default — can return different results on each call). Setting the correct volatility allows the query planner to optimize more aggressively. Marking a function IMMUTABLE when it is not can produce incorrect query results.

4. Triggers — Automatic Reactions to Data Changes

A trigger is a function that PostgreSQL executes automatically in response to INSERT, UPDATE, or DELETE events on a table. Triggers are the most powerful way to enforce complex business rules, maintain audit logs, synchronize denormalized data, and validate constraints that span multiple tables.

sql 
-- Step 1: Create the audit log table
CREATE TABLE employee_audit (
    id          bigserial PRIMARY KEY,
    employee_id integer NOT NULL,
    action      text NOT NULL,
    old_data    jsonb,
    new_data    jsonb,
    changed_at  timestamptz DEFAULT now(),
    changed_by  text DEFAULT current_user
);

-- Step 2: Create the trigger function
CREATE OR REPLACE FUNCTION fn_employee_audit()
RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        INSERT INTO employee_audit (employee_id, action, new_data)
        VALUES (NEW.id, 'INSERT', to_jsonb(NEW));
        RETURN NEW;

    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO employee_audit (employee_id, action, old_data, new_data)
        VALUES (NEW.id, 'UPDATE', to_jsonb(OLD), to_jsonb(NEW));
        RETURN NEW;

    ELSIF TG_OP = 'DELETE' THEN
        INSERT INTO employee_audit (employee_id, action, old_data)
        VALUES (OLD.id, 'DELETE', to_jsonb(OLD));
        RETURN OLD;
    END IF;
END;
$$ LANGUAGE plpgsql;

-- Step 3: Attach the trigger to the employees table
CREATE TRIGGER trg_employee_audit
    AFTER INSERT OR UPDATE OR DELETE ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_employee_audit();

Now every change to the employees table is automatically recorded in employee_audit with the full before-and-after state as JSON. No application code is required — even raw SQL updates from psql will be audited.

sql 
-- Test the trigger: update a salary
UPDATE employees
SET salary = salary * 1.10
WHERE email = 'ada@example.com';

-- Check the audit log
SELECT
    action,
    old_data->>'salary' AS old_salary,
    new_data->>'salary' AS new_salary,
    changed_at,
    changed_by
FROM employee_audit
WHERE employee_id = 1
ORDER BY changed_at DESC;

5. Common Trigger Patterns

Beyond audit logging, triggers are used extensively for maintaining timestamps, enforcing soft deletes, and keeping denormalized counters in sync. Here are production-tested patterns you will use repeatedly throughout your career.

sql 
-- Pattern 1: Auto-update updated_at timestamp
CREATE OR REPLACE FUNCTION fn_set_updated_at()
RETURNS trigger AS $$
BEGIN
    NEW.updated_at := now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_employees_updated_at
    BEFORE UPDATE ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_set_updated_at();


-- Pattern 2: Prevent salary decreases
CREATE OR REPLACE FUNCTION fn_prevent_salary_decrease()
RETURNS trigger AS $$
BEGIN
    IF NEW.salary < OLD.salary THEN
        RAISE EXCEPTION 'Salary cannot be decreased from % to %',
            OLD.salary, NEW.salary;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_no_salary_decrease
    BEFORE UPDATE OF salary ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_prevent_salary_decrease();


-- Pattern 3: Maintain denormalized employee count
ALTER TABLE departments ADD COLUMN employee_count integer DEFAULT 0;

CREATE OR REPLACE FUNCTION fn_update_dept_count()
RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'INSERT' THEN
        UPDATE departments
        SET employee_count = employee_count + 1
        WHERE id = NEW.dept_id;
    ELSIF TG_OP = 'DELETE' THEN
        UPDATE departments
        SET employee_count = employee_count - 1
        WHERE id = OLD.dept_id;
    ELSIF TG_OP = 'UPDATE' AND OLD.dept_id <> NEW.dept_id THEN
        UPDATE departments
        SET employee_count = employee_count - 1
        WHERE id = OLD.dept_id;
        UPDATE departments
        SET employee_count = employee_count + 1
        WHERE id = NEW.dept_id;
    END IF;
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_dept_employee_count
    AFTER INSERT OR UPDATE OR DELETE ON employees
    FOR EACH ROW
    EXECUTE FUNCTION fn_update_dept_count();
Trigger Performance Warning

Every trigger adds overhead to the triggering operation. An AFTER INSERT trigger on a high-write table can significantly slow down bulk inserts. Audit triggers that write to another table double the write I/O. Profile your triggers under realistic load and consider disabling them during large data migrations with ALTER TABLE ... DISABLE TRIGGER ALL.

6. Event Triggers — DDL Governance

While regular triggers respond to DML (data manipulation: INSERT/UPDATE/DELETE), event triggers respond to DDL (data definition: CREATE, ALTER, DROP). They fire on schema-level changes and are used for governance, compliance logging, and preventing accidental schema modifications in production environments.

sql 
-- Log all DDL operations
CREATE TABLE ddl_log (
    id          bigserial PRIMARY KEY,
    event       text NOT NULL,
    object_type text,
    object_name text,
    executed_by text DEFAULT current_user,
    executed_at timestamptz DEFAULT now(),
    command_tag text
);

-- Create the event trigger function
CREATE OR REPLACE FUNCTION fn_log_ddl()
RETURNS event_trigger AS $$
DECLARE
    obj record;
BEGIN
    FOR obj IN
        SELECT * FROM pg_event_trigger_ddl_commands()
    LOOP
        INSERT INTO ddl_log (event, object_type, object_name, command_tag)
        VALUES (
            TG_EVENT,
            obj.object_type,
            obj.object_identity,
            obj.command_tag
        );
    END LOOP;
END;
$$ LANGUAGE plpgsql;

-- Attach to ddl_command_end events
CREATE EVENT TRIGGER trg_log_ddl
    ON ddl_command_end
    EXECUTE FUNCTION fn_log_ddl();
sql 
-- Prevent DROP TABLE in production
CREATE OR REPLACE FUNCTION fn_prevent_drop()
RETURNS event_trigger AS $$
DECLARE
    obj record;
BEGIN
    FOR obj IN
        SELECT * FROM pg_event_trigger_dropped_objects()
    LOOP
        IF obj.object_type = 'table' THEN
            RAISE EXCEPTION
                'DROP TABLE is disabled in production. Contact the DBA team.';
        END IF;
    END LOOP;
END;
$$ LANGUAGE plpgsql;

CREATE EVENT TRIGGER trg_prevent_drop
    ON sql_drop
    EXECUTE FUNCTION fn_prevent_drop();

-- Check the DDL log
SELECT * FROM ddl_log
ORDER BY executed_at DESC
LIMIT 10;
Event Trigger Events

PostgreSQL supports four event trigger events: ddl_command_start (before DDL runs), ddl_command_end (after DDL completes), sql_drop (when objects are dropped), and table_rewrite (when a table is rewritten by ALTER COLUMN TYPE). Use ddl_command_start to prevent operations and ddl_command_end to log completed changes.

Self-Check

This is the final self-check of the PostgreSQL Fundamentals track. Answer honestly — if anything is unclear, revisit the relevant section before moving to the Advanced track.

What is the difference between a regular view and a materialized view?

A regular view is a stored query that re-executes every time you select from it — it stores no data. A materialized view executes the query once and stores the results on disk like a table. Subsequent reads are fast because they read stored data, but the data can become stale. You must explicitly call REFRESH MATERIALIZED VIEW to update it.

When should you use BEFORE vs AFTER triggers?

Use BEFORE triggers to modify or validate the incoming row before it is written — for example, setting updated_at or rejecting invalid changes. Use AFTER triggers to react to completed changes — for example, writing audit logs or updating counters. BEFORE triggers can modify NEW; AFTER triggers cannot change the already-written row.

What are the three volatility levels for PostgreSQL functions?

IMMUTABLE: always returns the same result for the same arguments (can be constant-folded). STABLE: consistent results within a single SQL statement (safe for index scans). VOLATILE (default): can return different results on each call. Setting the correct volatility lets the planner optimize more aggressively.

How do event triggers differ from regular triggers?

Regular triggers fire on DML (INSERT, UPDATE, DELETE) against a specific table. Event triggers fire on DDL (CREATE, ALTER, DROP) at the database level. Event triggers use event_trigger as their return type and access special functions like pg_event_trigger_ddl_commands() to inspect the DDL that was executed.

When is REFRESH MATERIALIZED VIEW CONCURRENTLY needed?

A normal REFRESH takes an exclusive lock, blocking all reads until complete. CONCURRENTLY builds new data alongside the old and swaps them atomically, allowing uninterrupted reads. It requires a unique index on the materialized view and is slightly slower, but is essential in production where queries cannot be blocked during refresh windows.
