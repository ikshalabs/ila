Chapter 08 · PostgreSQL Basic

Views, Functions, and Triggers

Encapsulate complex queries in views, build reusable PL/pgSQL functions for bid calculations, and automate audit trails with triggers that fire on every ad group bid change.

Estimated: 3–4 hours
4 Key Concepts
3 Assignments

Why This Matters

Views, functions, and triggers let you encapsulate complex logic inside the database itself. In the ad portal, a view like active_campaigns eliminates the need to repeat WHERE discarded_at IS NULL AND active = TRUE in every query. Functions encapsulate business logic like bid price calculations. Triggers automatically maintain audit trails and enforce complex business rules.

Key Insight

The ad portal uses materialized views for expensive analytical queries (refreshed periodically), PL/pgSQL functions for business logic that must be atomic with database operations, and triggers for automatically tracking bid change histories and updating timestamps.

Key Concepts

1. Views

sql
-- View: active campaigns (filters out soft-deleted and paused)
CREATE VIEW active_campaigns AS
SELECT
    c.id, c.name, c.account_id, c.monthly_budget,
    c.base_price, c.active, c.created_at,
    a.company_name
FROM campaigns c
JOIN accounts a ON a.id = c.account_id
WHERE c.discarded_at IS NULL
  AND c.active = TRUE
  AND a.discarded_at IS NULL;

-- Now queries are much simpler:
SELECT * FROM active_campaigns WHERE monthly_budget > 5000;

-- View: campaign performance summary
CREATE VIEW campaign_performance AS
SELECT
    c.id, c.name, c.monthly_budget,
    COUNT(l.id) AS total_leads,
    COUNT(ag.id) AS ad_group_count,
    AVG(ag.base_bid_price) AS avg_bid_price
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.id
LEFT JOIN ad_groups ag ON ag.campaign_id = c.id AND ag.discarded_at IS NULL
WHERE c.discarded_at IS NULL
GROUP BY c.id;

2. Materialized Views

sql
-- Materialized view: pre-computed, stored on disk
CREATE MATERIALIZED VIEW account_dashboard AS
SELECT
    a.id, a.company_name,
    COUNT(DISTINCT c.id) AS campaign_count,
    SUM(c.monthly_budget) AS total_budget,
    COUNT(DISTINCT l.id) AS total_leads,
    ab.current_balance
FROM accounts a
LEFT JOIN campaigns c ON c.account_id = a.id AND c.discarded_at IS NULL
LEFT JOIN leads l ON l.account_id = a.id
LEFT JOIN account_balances ab ON ab.user_id = a.id
WHERE a.discarded_at IS NULL
GROUP BY a.id, a.company_name, ab.current_balance;

-- Refresh the materialized view (e.g., every hour via cron)
REFRESH MATERIALIZED VIEW CONCURRENTLY account_dashboard;

3. PL/pgSQL Functions

sql
-- Function: calculate effective bid price with margin
CREATE OR REPLACE FUNCTION effective_bid(
    base_bid NUMERIC,
    rtb_cm NUMERIC DEFAULT 0.30
) RETURNS NUMERIC AS $$
BEGIN
    RETURN ROUND(base_bid * (1 - rtb_cm), 2);
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- Use the function in queries
SELECT
    ad_group_name,
    base_bid_price,
    effective_bid(base_bid_price, rtb_cm) AS net_bid
FROM ad_groups
WHERE active = TRUE;

-- Function: safely update bid with validation
CREATE OR REPLACE FUNCTION update_bid(
    p_ad_group_id BIGINT,
    p_new_bid NUMERIC
) RETURNS VOID AS $$
DECLARE
    v_old_bid NUMERIC;
BEGIN
    SELECT base_bid_price INTO v_old_bid
    FROM ad_groups WHERE id = p_ad_group_id;

    -- Record the change
    INSERT INTO ad_group_bid_change_histories (ad_group_id, old_bid, new_bid, created_at, updated_at)
    VALUES (p_ad_group_id, v_old_bid, p_new_bid, NOW(), NOW());

    -- Apply the change
    UPDATE ad_groups SET base_bid_price = p_new_bid, updated_at = NOW()
    WHERE id = p_ad_group_id;
END;
$$ LANGUAGE plpgsql;

4. Triggers

sql
-- Trigger function: auto-update the updated_at timestamp
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach trigger to campaigns table
CREATE TRIGGER campaigns_updated_at
    BEFORE UPDATE ON campaigns
    FOR EACH ROW
    EXECUTE FUNCTION set_updated_at();

-- Trigger: log bid changes automatically
CREATE OR REPLACE FUNCTION log_bid_change()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.base_bid_price IS DISTINCT FROM NEW.base_bid_price THEN
        INSERT INTO ad_group_bid_change_histories
            (ad_group_id, old_bid, new_bid, created_at, updated_at)
        VALUES
            (NEW.id, OLD.base_bid_price, NEW.base_bid_price, NOW(), NOW());
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER ad_groups_bid_audit
    AFTER UPDATE ON ad_groups
    FOR EACH ROW
    EXECUTE FUNCTION log_bid_change();

Self-Check

When should you use a view vs a materialized view?

Use a regular view when you need always-current data and the underlying query is fast. Use a materialized view when the query is expensive (aggregating millions of rows) and slightly stale data is acceptable. The ad portal uses materialized views for dashboard summaries refreshed hourly.

What is the difference between BEFORE and AFTER triggers?

BEFORE triggers run before the row is modified and can change the NEW values (e.g., setting updated_at). AFTER triggers run after the modification and are used for logging/auditing (e.g., recording bid changes). BEFORE triggers can prevent the operation by returning NULL.

Why use IMMUTABLE vs VOLATILE function volatility?

IMMUTABLE means the function always returns the same result for the same inputs (like effective_bid). PostgreSQL can cache results and use it in indexes. VOLATILE (default) means results may change (like functions using NOW()). Mislabeling a function as IMMUTABLE when it is not causes incorrect cached results.
