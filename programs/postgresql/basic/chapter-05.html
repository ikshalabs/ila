Aggregation and Window Functions

Master GROUP BY, HAVING, and aggregate functions for ad-tech reporting. Learn window functions for rankings, running totals, and moving averages that power campaign dashboards.

Estimated: 3–4 hours
5 Key Concepts
3 Assignments

Why This Matters

Every ad-tech dashboard is powered by aggregation. "Total leads this month," "average bid price per campaign," "top 5 accounts by spend" — these all require GROUP BY, aggregate functions, and increasingly, window functions. Without aggregation skills, you cannot build the reports that drive business decisions.

Window functions are PostgreSQL's secret weapon for analytics. They let you compute running totals, rankings, moving averages, and percentage-of-total calculations without collapsing rows — something that would otherwise require complex self-joins or application-level processing.

Key Insight

The difference between GROUP BY and window functions is fundamental: GROUP BY collapses rows (10 campaigns become 1 summary row per account), while window functions add computed columns to every row (10 campaigns remain 10 rows, each annotated with its rank, running total, or percentage of account budget).

Key Concepts

1. GROUP BY and Aggregate Functions

sql
-- Count campaigns per account
SELECT
    account_id,
    COUNT(*) AS campaign_count,
    SUM(monthly_budget) AS total_budget,
    AVG(base_price) AS avg_price,
    MIN(base_price) AS min_price,
    MAX(base_price) AS max_price
FROM campaigns
WHERE discarded_at IS NULL
GROUP BY account_id
ORDER BY total_budget DESC;

-- Daily lead volume for a campaign
SELECT
    DATE(created_at) AS lead_date,
    COUNT(*) AS daily_leads
FROM leads
WHERE campaign_id = 42
GROUP BY DATE(created_at)
ORDER BY lead_date DESC;

2. HAVING — Filtering Aggregated Results

sql
-- Accounts with more than 5 active campaigns
SELECT
    a.company_name,
    COUNT(c.id) AS campaign_count,
    SUM(c.monthly_budget) AS total_budget
FROM accounts a
JOIN campaigns c ON c.account_id = a.id
WHERE c.active = TRUE AND c.discarded_at IS NULL
GROUP BY a.id, a.company_name
HAVING COUNT(c.id) > 5
ORDER BY campaign_count DESC;

-- Campaigns where average bid price exceeds $20
SELECT
    c.name,
    AVG(ag.base_bid_price) AS avg_bid
FROM campaigns c
JOIN ad_groups ag ON ag.campaign_id = c.id
WHERE ag.discarded_at IS NULL
GROUP BY c.id, c.name
HAVING AVG(ag.base_bid_price) > 20.00;

3. Window Functions

Window functions perform calculations across a set of rows related to the current row, without collapsing the result set. They use the OVER() clause.

sql
-- Running total of monthly budget per account
SELECT
    account_id,
    name,
    monthly_budget,
    SUM(monthly_budget) OVER (
        PARTITION BY account_id
        ORDER BY created_at
    ) AS running_total
FROM campaigns
WHERE discarded_at IS NULL;

-- Each campaign's budget as percentage of its account's total
SELECT
    name,
    monthly_budget,
    SUM(monthly_budget) OVER (PARTITION BY account_id) AS account_total,
    ROUND(
        monthly_budget * 100.0 / SUM(monthly_budget) OVER (PARTITION BY account_id),
        1
    ) AS pct_of_account
FROM campaigns
WHERE discarded_at IS NULL;

4. Ranking Functions

sql
-- Rank ad groups by bid price within each campaign
SELECT
    campaign_id,
    ad_group_name,
    base_bid_price,
    ROW_NUMBER() OVER (PARTITION BY campaign_id ORDER BY base_bid_price DESC) AS row_num,
    RANK() OVER (PARTITION BY campaign_id ORDER BY base_bid_price DESC) AS rank,
    DENSE_RANK() OVER (PARTITION BY campaign_id ORDER BY base_bid_price DESC) AS dense_rank
FROM ad_groups
WHERE active = TRUE AND discarded_at IS NULL;

-- Top 3 campaigns per account by budget
SELECT * FROM (
    SELECT
        account_id, name, monthly_budget,
        ROW_NUMBER() OVER (
            PARTITION BY account_id
            ORDER BY monthly_budget DESC
        ) AS rn
    FROM campaigns
    WHERE discarded_at IS NULL
) ranked
WHERE rn <= 3;

5. LAG, LEAD, and Moving Averages

sql
-- Bid change history: show old and new bid side by side
SELECT
    ad_group_id,
    new_bid,
    LAG(new_bid) OVER (PARTITION BY ad_group_id ORDER BY created_at) AS previous_bid,
    new_bid - LAG(new_bid) OVER (PARTITION BY ad_group_id ORDER BY created_at) AS change,
    created_at
FROM ad_group_bid_change_histories
ORDER BY ad_group_id, created_at;

-- 7-day moving average of daily lead count
SELECT
    lead_date,
    daily_leads,
    AVG(daily_leads) OVER (
        ORDER BY lead_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) AS moving_avg_7d
FROM (
    SELECT DATE(created_at) AS lead_date, COUNT(*) AS daily_leads
    FROM leads
    GROUP BY DATE(created_at)
) daily;

Self-Check

What is the difference between WHERE and HAVING?

WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER aggregation. You cannot use aggregate functions in WHERE (e.g., WHERE COUNT(*) > 5 is invalid). Use HAVING COUNT(*) > 5 instead.

When would you use RANK vs ROW_NUMBER vs DENSE_RANK?

ROW_NUMBER assigns unique sequential numbers (1,2,3) even for ties. RANK assigns the same number for ties but skips (1,1,3). DENSE_RANK assigns the same for ties without skipping (1,1,2). Use ROW_NUMBER for pagination, RANK for competitive rankings, DENSE_RANK for grading.

How does PARTITION BY differ from GROUP BY?

GROUP BY collapses rows into summary groups (100 rows become 5 group rows). PARTITION BY in a window function divides rows into partitions for calculation but keeps all 100 original rows. Each row gets the computed value for its partition.
