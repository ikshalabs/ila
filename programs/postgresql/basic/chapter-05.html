Chapter 05 · Phase 3: Intermediate SQL

Aggregation and Window Functions

Move beyond row-level queries into the world of grouped summaries, running totals, and ranked result sets. Aggregation and window functions are the tools that turn raw data into business intelligence — the queries that power every dashboard, report, and analytics pipeline.

Estimated: 3–4 hours
3 Advancement Levels
3 Assignments

Why This Matters

Nearly every business question boils down to aggregation: "How many orders did we get last month?" "What is the average revenue per customer?" "Who are our top 10 sellers?" Without GROUP BY, SUM, and COUNT, you would need to pull every row into application code and aggregate manually — a pattern that is both slow and error-prone at scale.

Window functions take this a step further. They let you compute running totals, rank rows within partitions, and calculate moving averages — all without collapsing the result set. Before window functions existed in SQL, these calculations required self-joins or procedural code that was difficult to write, difficult to debug, and difficult to optimize.

PostgreSQL has had full window function support since version 8.4 (2009), and its implementation is one of the most complete in any relational database. Understanding these features is the dividing line between writing basic queries and writing queries that can drive dashboards, financial reports, and real-time analytics.

Key Insight

Aggregate functions collapse rows into a single result per group. Window functions compute a value for every row while still giving you access to the individual row data. The difference is OVER() — that single clause is what separates aggregation from windowing.

Key Concepts

1. GROUP BY Basics

The GROUP BY clause collapses rows that share the same values in the specified columns into a single summary row. Every column in your SELECT list must either appear in the GROUP BY clause or be wrapped in an aggregate function. Violating this rule produces an error in PostgreSQL (unlike MySQL in permissive mode, which silently returns indeterminate values).

sql 
-- Count orders per customer
SELECT
    customer_id,
    COUNT(*) AS order_count
FROM orders
GROUP BY customer_id
ORDER BY order_count DESC;

-- Revenue by product category and month
SELECT
    p.category,
    DATE_TRUNC('month', o.created_at) AS order_month,
    SUM(oi.quantity * oi.unit_price) AS total_revenue,
    COUNT(DISTINCT o.id) AS order_count
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
JOIN products p ON p.id = oi.product_id
GROUP BY p.category, order_month
ORDER BY p.category, order_month;

2. Aggregate Functions (COUNT, SUM, AVG, MIN, MAX)

PostgreSQL provides a rich set of aggregate functions. The five most commonly used are COUNT, SUM, AVG, MIN, and MAX. Each of these ignores NULL values except COUNT(*), which counts all rows regardless of NULLs. Understanding this NULL behavior is critical for accurate reporting.

sql 
-- Full aggregate summary for each department
SELECT
    department_id,
    COUNT(*) AS employee_count,
    COUNT(manager_id) AS with_manager,
    ROUND(SUM(salary), 2) AS total_payroll,
    ROUND(AVG(salary), 2) AS avg_salary,
    MIN(salary) AS lowest_salary,
    MAX(salary) AS highest_salary,
    MAX(salary) - MIN(salary) AS salary_range
FROM employees
GROUP BY department_id
ORDER BY total_payroll DESC;

-- Using FILTER clause for conditional aggregation (PostgreSQL 9.4+)
SELECT
    COUNT(*) AS total_orders,
    COUNT(*) FILTER (WHERE status = 'completed') AS completed,
    COUNT(*) FILTER (WHERE status = 'cancelled') AS cancelled,
    ROUND(
        100.0 * COUNT(*) FILTER (WHERE status = 'completed')
        / NULLIF(COUNT(*), 0), 1
    ) AS completion_rate_pct
FROM orders
WHERE created_at >= '2025-01-01';
Pro Tip

The FILTER clause is a PostgreSQL extension that replaces the classic CASE WHEN ... THEN 1 END pattern inside aggregates. It is cleaner to read, easier to maintain, and the query planner can sometimes optimize it more efficiently. Always wrap denominators with NULLIF(expr, 0) to avoid division-by-zero errors.

3. HAVING vs WHERE

A common source of confusion is when to use WHERE versus HAVING. The rule is straightforward: WHERE filters individual rows before grouping. HAVING filters groups after aggregation. You cannot reference aggregate functions in a WHERE clause.

sql 
-- WHERE filters rows BEFORE grouping
-- HAVING filters groups AFTER aggregation
SELECT
    customer_id,
    COUNT(*) AS order_count,
    SUM(total_amount) AS lifetime_value
FROM orders
WHERE status != 'cancelled'       -- Row-level filter (runs before GROUP BY)
GROUP BY customer_id
HAVING COUNT(*) >= 5             -- Group-level filter (runs after GROUP BY)
   AND SUM(total_amount) > 500    -- Another group-level filter
ORDER BY lifetime_value DESC;

-- Find products ordered in more than 3 distinct months
SELECT
    p.name,
    COUNT(DISTINCT DATE_TRUNC('month', o.created_at)) AS active_months
FROM products p
JOIN order_items oi ON oi.product_id = p.id
JOIN orders o ON o.id = oi.order_id
GROUP BY p.id, p.name
HAVING COUNT(DISTINCT DATE_TRUNC('month', o.created_at)) > 3;

  SQL Logical Execution Order
 ┌──────────────────────────────────────────────────────────┐
 │  1. FROM / JOIN       ── assemble source rows             │
 │  2. WHERE             ── filter individual rows            │
 │  3. GROUP BY          ── collapse into groups              │
 │  4. HAVING            ── filter groups by aggregate result │
 │  5. WINDOW            ── compute window functions          │
 │  6. SELECT            ── compute output expressions        │
 │  7. DISTINCT          ── remove duplicate rows             │
 │  8. ORDER BY          ── sort the final result set         │
 │  9. LIMIT / OFFSET    ── paginate                          │
 └──────────────────────────────────────────────────────────┘

4. Window Functions Introduction (OVER, PARTITION BY)

Window functions perform a calculation across a set of rows related to the current row — without collapsing those rows into a single output. The key syntax element is the OVER() clause, which defines the "window" of rows the function sees. You can partition the window with PARTITION BY and order it with ORDER BY.

sql 
-- Compare each employee's salary to their department average
SELECT
    name,
    department_id,
    salary,
    AVG(salary) OVER (
        PARTITION BY department_id
    ) AS dept_avg_salary,
    salary - AVG(salary) OVER (
        PARTITION BY department_id
    ) AS diff_from_avg
FROM employees
ORDER BY department_id, salary DESC;

-- Each order with the customer's total and percentage of their total
SELECT
    id AS order_id,
    customer_id,
    total_amount,
    SUM(total_amount) OVER (
        PARTITION BY customer_id
    ) AS customer_lifetime,
    ROUND(
        100.0 * total_amount /
        SUM(total_amount) OVER (PARTITION BY customer_id), 2
    ) AS pct_of_lifetime
FROM orders
WHERE status = 'completed'
ORDER BY customer_id, total_amount DESC;
Common Mistake

You cannot use window functions in a WHERE or HAVING clause. Window functions are evaluated after WHERE, GROUP BY, and HAVING in the SQL execution order. If you need to filter on a window function result, wrap the query in a CTE or subquery and filter in the outer query.

5. Ranking Functions (ROW_NUMBER, RANK, DENSE_RANK, NTILE)

Ranking functions assign a position number to each row within its partition. The three main ranking functions differ only in how they handle ties:

  • ROW_NUMBER() — Always unique; tied rows get arbitrary sequential numbers.
  • RANK() — Ties share the same rank; the next rank skips (1, 2, 2, 4).
  • DENSE_RANK() — Ties share the same rank; no gaps (1, 2, 2, 3).
  • NTILE(n) — Distributes rows into n roughly equal-sized buckets.
sql 
-- Compare all three ranking functions side by side
SELECT
    name,
    department_id,
    salary,
    ROW_NUMBER() OVER (
        PARTITION BY department_id ORDER BY salary DESC
    ) AS row_num,
    RANK() OVER (
        PARTITION BY department_id ORDER BY salary DESC
    ) AS rnk,
    DENSE_RANK() OVER (
        PARTITION BY department_id ORDER BY salary DESC
    ) AS dense_rnk
FROM employees
ORDER BY department_id, salary DESC;

-- Top 3 products per category by revenue
WITH ranked_products AS (
    SELECT
        p.category,
        p.name,
        SUM(oi.quantity * oi.unit_price) AS revenue,
        ROW_NUMBER() OVER (
            PARTITION BY p.category
            ORDER BY SUM(oi.quantity * oi.unit_price) DESC
        ) AS rn
    FROM products p
    JOIN order_items oi ON oi.product_id = p.id
    GROUP BY p.category, p.name
)
SELECT category, name, revenue
FROM ranked_products
WHERE rn <= 3
ORDER BY category, revenue DESC;

-- NTILE: distribute customers into 4 spending quartiles
SELECT
    customer_id,
    SUM(total_amount) AS lifetime_value,
    NTILE(4) OVER (
        ORDER BY SUM(total_amount) DESC
    ) AS spending_quartile
FROM orders
WHERE status = 'completed'
GROUP BY customer_id
ORDER BY lifetime_value DESC;

6. Running Totals and Moving Averages

By adding an ORDER BY inside the OVER() clause, aggregate window functions compute a running (cumulative) result. You can further control the window frame with ROWS BETWEEN or RANGE BETWEEN to build moving averages and other sliding-window calculations.

sql 
-- Running total of daily revenue
WITH daily_totals AS (
    SELECT
        DATE(created_at) AS order_date,
        SUM(total_amount) AS daily_revenue
    FROM orders
    WHERE created_at >= '2025-01-01'
    GROUP BY DATE(created_at)
)
SELECT
    order_date,
    daily_revenue,
    SUM(daily_revenue) OVER (
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS cumulative_revenue
FROM daily_totals
ORDER BY order_date;

-- 7-day moving average of daily revenue
WITH daily_totals AS (
    SELECT
        DATE(created_at) AS order_date,
        SUM(total_amount) AS daily_revenue
    FROM orders
    GROUP BY DATE(created_at)
)
SELECT
    order_date,
    daily_revenue,
    ROUND(
        AVG(daily_revenue) OVER (
            ORDER BY order_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ), 2
    ) AS moving_avg_7d
FROM daily_totals
ORDER BY order_date;

-- LAG / LEAD: compare with previous and next row values
WITH daily_totals AS (
    SELECT
        DATE(created_at) AS order_date,
        SUM(total_amount) AS daily_revenue
    FROM orders
    GROUP BY DATE(created_at)
)
SELECT
    order_date,
    daily_revenue,
    LAG(daily_revenue, 1) OVER (ORDER BY order_date) AS prev_day_revenue,
    ROUND(
        100.0 * (daily_revenue - LAG(daily_revenue, 1) OVER (ORDER BY order_date))
        / NULLIF(LAG(daily_revenue, 1) OVER (ORDER BY order_date), 0),
        2
    ) AS day_over_day_pct
FROM daily_totals
ORDER BY order_date;
Frame Specification

The default window frame when ORDER BY is present is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. If two rows share the same ORDER BY value, RANGE includes both in the same frame. To get strictly row-by-row accumulation, use ROWS BETWEEN instead. For a 7-day moving average, use ROWS BETWEEN 6 PRECEDING AND CURRENT ROW.

Self-Check

Before moving to Chapter 06, honestly answer these questions. If any answer is uncertain, revisit the relevant concept.

What is the difference between WHERE and HAVING?

WHERE filters individual rows before grouping occurs. It cannot reference aggregate functions. HAVING filters groups after aggregation and can reference aggregate results like COUNT(*) > 5. Best practice is to use WHERE to exclude irrelevant rows early (reducing the data that needs grouping) and HAVING to filter the resulting groups.

When would you use RANK instead of ROW_NUMBER?

Use RANK() when ties matter and you want tied rows to share the same position (e.g., two students both scored 95 and should both be rank 1, with the next student at rank 3). Use ROW_NUMBER() when you need a unique sequential number regardless of ties — for example, to select exactly the top N rows from each partition. Use DENSE_RANK() when you want tie-sharing without gaps (1, 2, 2, 3 instead of 1, 2, 2, 4).

How does a window frame specification affect SUM() OVER()?

Without a frame specification, SUM() OVER(ORDER BY col) defaults to RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, producing a running total. But with RANGE, rows sharing the same ORDER BY value are lumped together. For strictly row-by-row accumulation, use ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. For sliding windows like a 7-day moving average, use ROWS BETWEEN 6 PRECEDING AND CURRENT ROW.
