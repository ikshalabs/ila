Chapter 06 · PostgreSQL Basic

Indexes and Query Plans

Understand B-tree indexes, read EXPLAIN ANALYZE output, and develop index strategies that make ad portal queries fast. Learn partial indexes, covering indexes, and when NOT to index.

Estimated: 3–4 hours
4 Key Concepts
3 Assignments

Why This Matters

Without indexes, every query scans the entire table. In the ad portal, the leads table can have millions of rows. A query like SELECT * FROM leads WHERE campaign_id = 42 without an index on campaign_id forces PostgreSQL to read every single row. With an index, it jumps directly to the matching rows — the difference between 500ms and 2ms.

Understanding EXPLAIN ANALYZE output is what separates engineers who guess at performance from those who know. When a dashboard query takes 3 seconds, the query plan tells you exactly why: a sequential scan on a 2-million-row table, a nested loop join that should be a hash join, or a missing index that forces a sort.

Key Insight

The ad portal schema has indexes on every foreign key column, every discarded_at column, and composite indexes on commonly queried combinations like (campaign_id, dt, ad_group_id). These indexes were not added randomly — each one was justified by a real query pattern identified through EXPLAIN ANALYZE.

Key Concepts

1. B-tree Indexes

B-tree is the default index type in PostgreSQL and handles equality and range comparisons. It is the workhorse for most ad-tech queries.

sql
-- Create index on foreign key (PostgreSQL does NOT do this automatically)
CREATE INDEX idx_campaigns_account_id ON campaigns (account_id);
CREATE INDEX idx_ad_groups_campaign_id ON ad_groups (campaign_id);
CREATE INDEX idx_leads_campaign_id ON leads (campaign_id);

-- Composite index for common query pattern
CREATE UNIQUE INDEX idx_campaign_dt_adgroup
    ON ad_group_stats (campaign_id, dt, ad_group_id);

-- Index on soft-delete column (speeds up WHERE discarded_at IS NULL)
CREATE INDEX idx_campaigns_discarded ON campaigns (discarded_at);

2. EXPLAIN ANALYZE

sql
-- See the query plan WITHOUT executing
EXPLAIN SELECT * FROM campaigns WHERE account_id = 7;

-- See the actual execution plan WITH timing
EXPLAIN ANALYZE SELECT * FROM campaigns WHERE account_id = 7;

-- Verbose output with buffers (most useful for optimization)
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT c.name, COUNT(l.id) AS lead_count
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.id
WHERE c.account_id = 7 AND c.discarded_at IS NULL
GROUP BY c.id;
Reading EXPLAIN Output

Look for: Seq Scan (full table scan — often bad on large tables), Index Scan (good — using an index), Bitmap Index Scan (good for moderate selectivity). The cost numbers are estimates; the actual time numbers are real. Focus on actual time and rows.

3. Partial and Covering Indexes

sql
-- Partial index: only index non-deleted campaigns
-- Much smaller than indexing the whole table
CREATE INDEX idx_active_campaigns
    ON campaigns (account_id, active)
    WHERE discarded_at IS NULL;

-- Covering index (INCLUDE): avoids table lookup for specific columns
CREATE INDEX idx_campaigns_account_covering
    ON campaigns (account_id)
    INCLUDE (name, monthly_budget, active);

4. Index Strategies for Ad-Tech

  • Always index foreign keys — account_id, campaign_id, ad_group_id on every child table.
  • Index soft-delete columns — discarded_at appears in every query.
  • Composite indexes for common patterns — The (campaign_id, dt, ad_group_id) index on ad_group_stats matches the exact query pattern for daily stats lookups.
  • Partial indexes for active-only queries — Most queries filter for non-deleted records. A partial index WHERE discarded_at IS NULL is smaller and faster.
Do Not Over-Index

Every index slows down INSERT, UPDATE, and DELETE because the index must be maintained. A table with 15 indexes will have noticeably slower writes. Add indexes based on actual query patterns identified through EXPLAIN ANALYZE, not speculation.

  • Create a partial index for active non-deleted campaigns and verify plan change

Done when

You have a documented index strategy explaining the purpose of each index. No unused indexes remain.

Self-Check

Why does PostgreSQL not automatically create indexes on foreign keys?

PostgreSQL separates referential integrity (FK constraint) from performance (index). The FK ensures correctness; the index improves query speed. Not every FK column is queried directly, so an automatic index might waste space. PostgreSQL lets you decide.

What is the difference between EXPLAIN and EXPLAIN ANALYZE?

EXPLAIN shows the estimated query plan without running the query. EXPLAIN ANALYZE actually executes the query and shows real timing and row counts. Use EXPLAIN for quick checks; use EXPLAIN ANALYZE for accurate performance measurement (but be careful with UPDATE/DELETE — it executes them!).

When should you use a partial index?

When most queries filter on the same condition. Since nearly every ad portal query includes WHERE discarded_at IS NULL, a partial index with that WHERE clause is smaller (excludes deleted rows) and faster than a full index.
