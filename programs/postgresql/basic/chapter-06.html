Chapter 06 · Phase 3: Intermediate SQL

Indexes and Query Plans

Understand why some queries finish in milliseconds while others crawl for seconds. Learn to read EXPLAIN ANALYZE output, choose the right index type, and build indexing strategies that scale with your data.

Estimated: 3–4 hours
3 Advancement Levels
3 Assignments

Why This Matters

A table with 10 million rows and no indexes forces PostgreSQL to read every single row for every query — a sequential scan. Adding the right index can reduce that from seconds to microseconds. But indexes are not free: they consume disk space, slow down writes, and the wrong index can actually make queries slower if the planner chooses it when a sequential scan would be faster.

The skill is not just "add an index." It is understanding when an index helps, which type of index to use, and how to verify your assumption with EXPLAIN ANALYZE. This chapter gives you that complete toolkit.

Every production database eventually hits a slow-query wall. The engineers who can read a query plan, identify the bottleneck, and fix it with the right index are the ones who keep systems running smoothly under load.

Key Insight

An index is a trade-off: you spend extra disk space and slower writes to get dramatically faster reads. The art is choosing which columns to index and which type of index to use, guided by your actual query patterns and validated by EXPLAIN ANALYZE.

Key Concepts

1. Why Indexes Matter (Sequential Scan vs Index Scan)

Without an index, PostgreSQL must perform a sequential scan (Seq Scan) — reading every row in the table from disk, checking each one against the WHERE condition. This is O(n) in the number of rows. With a B-tree index, lookups become O(log n), which is the difference between scanning 10 million rows and traversing 4 tree levels.

sql 
-- Without an index: sequential scan (slow on large tables)
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42;

-- Output (example):
-- Seq Scan on orders  (cost=0.00..184723.45 rows=52 width=96)
--   Filter: (customer_id = 42)
--   Rows Removed by Filter: 9999948
--   Planning Time: 0.085 ms
--   Execution Time: 1247.332 ms

-- Create an index on customer_id
CREATE INDEX idx_orders_customer_id ON orders (customer_id);

-- With the index: index scan (fast)
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42;

-- Output (example):
-- Index Scan using idx_orders_customer_id on orders
--   (cost=0.43..164.20 rows=52 width=96)
--   Index Cond: (customer_id = 42)
--   Planning Time: 0.112 ms
--   Execution Time: 0.287 ms

2. B-Tree Index Internals

B-tree (balanced tree) is the default and most commonly used index type in PostgreSQL. It stores keys in sorted order across a hierarchy of pages: a root page, internal pages, and leaf pages. The leaf pages are linked together, allowing efficient range scans.

sql 
-- B-tree indexes support these operators: <  <=  =  >=  >  BETWEEN  IN  IS NULL

-- Single-column index
CREATE INDEX idx_orders_created_at
    ON orders (created_at);

-- Multi-column (composite) index — column order matters!
CREATE INDEX idx_orders_customer_status
    ON orders (customer_id, status);

-- The composite index above efficiently supports:
--   WHERE customer_id = 42                      (uses first column)
--   WHERE customer_id = 42 AND status = 'active' (uses both columns)
-- But NOT:
--   WHERE status = 'active'  (skips leading column = index not used)

-- Check index size
SELECT
    indexrelname AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE relname = 'orders'
ORDER BY pg_relation_size(indexrelid) DESC;
Column Order in Composite Indexes

In a composite index (a, b, c), the index can be used for queries filtering on a, a AND b, or a AND b AND c. It cannot efficiently serve queries that skip the leading column. Think of it like a phone book sorted by last name then first name — you can look up by last name alone, but not by first name alone.

3. Reading EXPLAIN ANALYZE Output

EXPLAIN ANALYZE is your primary diagnostic tool. It runs the query and shows the actual execution plan with real timing data. Learning to read this output is the single most valuable performance skill in PostgreSQL.

sql 
-- EXPLAIN shows the plan without running the query
EXPLAIN
SELECT * FROM orders WHERE customer_id = 42;

-- EXPLAIN ANALYZE actually runs the query and shows real timings
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT
    o.id,
    o.total_amount,
    c.name
FROM orders o
JOIN customers c ON c.id = o.customer_id
WHERE o.created_at >= '2025-01-01'
  AND o.status = 'completed'
ORDER BY o.total_amount DESC
LIMIT 10;

-- Key fields to look for in the output:
-- > Scan type: Seq Scan, Index Scan, Index Only Scan, Bitmap Scan
-- > cost=startup..total  (estimated, in arbitrary units)
-- > rows=N              (estimated row count)
-- > actual time=X..Y    (real milliseconds with ANALYZE)
-- > Buffers: shared hit/read  (cache hit ratio)
-- > Planning Time / Execution Time (total wall clock)

  EXPLAIN ANALYZE Output Anatomy
 ┌──────────────────────────────────────────────────────────────┐
 │ Sort  (cost=1234..1240 rows=100 width=64)                   │
 │       (actual time=2.34..2.41 rows=100 loops=1)             │
 │   Sort Key: total_amount DESC                               │
 │   Sort Method: top-N heapsort  Memory: 32kB                 │
 │   -> Nested Loop                                            │
 │        (cost=0.85..1200 rows=100 width=64)                  │

4. Index Types (B-tree, Hash, GiST, GIN)

PostgreSQL supports multiple index types, each optimized for different kinds of queries. Choosing the right type is as important as choosing the right columns. The default is B-tree, but understanding when to reach for GIN, GiST, or Hash can unlock significant performance improvements for specialized workloads.

  • B-tree — Default. Supports =, <, >, BETWEEN, IN, IS NULL. Best for most equality and range queries.
  • Hash — Only supports =. Slightly faster than B-tree for pure equality, but no range support. Rarely used in practice.
  • GiST — Generalized Search Tree. For geometric data, full-text search, and range types. Supports operators like &&, @>, <@.
  • GIN — Generalized Inverted Index. For arrays, JSONB, and full-text search. Excellent for containment queries (@>, ?, ?&).
sql 
-- B-tree (default) — equality and range queries
CREATE INDEX idx_orders_date ON orders (created_at);

-- Hash — only for exact equality (rarely needed)
CREATE INDEX idx_sessions_token ON sessions USING hash (session_token);

-- GIN — for JSONB containment queries
CREATE INDEX idx_products_metadata ON products USING gin (metadata);

-- Query using the GIN index
SELECT * FROM products
WHERE metadata @> '{"color": "red"}'::jsonb;

-- GIN for array containment
CREATE INDEX idx_products_tags ON products USING gin (tags);

SELECT * FROM products
WHERE tags @> ARRAY['organic', 'vegan'];

-- GiST — for full-text search with tsvector
CREATE INDEX idx_articles_fts ON articles
    USING gist (to_tsvector('english', title || ' ' || body));

-- GIN — also for full-text (faster lookups, slower updates than GiST)
CREATE INDEX idx_articles_fts_gin ON articles
    USING gin (to_tsvector('english', title || ' ' || body));

-- Full-text search query using the index
SELECT title, ts_rank(
    to_tsvector('english', title || ' ' || body),
    plainto_tsquery('english', 'database performance')
) AS relevance
FROM articles
WHERE to_tsvector('english', title || ' ' || body)
    @@ plainto_tsquery('english', 'database performance')
ORDER BY relevance DESC
LIMIT 10;

 ┌──────────────┬──────────────────────┬────────────────────────────┐
 │  Index Type  │  Best For            │  Operators Supported       │
 ├──────────────┼──────────────────────┼────────────────────────────┤
 │  B-tree      │  Most queries        │  =  <  >  <=  >=  BETWEEN │
 │              │  ORDER BY, range     │  IN  IS NULL  IS NOT NULL  │
 ├──────────────┼──────────────────────┼────────────────────────────┤
 │  Hash        │  Exact equality only │  =                         │
 ├──────────────┼──────────────────────┼────────────────────────────┤

 │              │  Full-text search    │  (inverted lookups)        │
 ├──────────────┼──────────────────────┼────────────────────────────┤

 └──────────────┴──────────────────────┴────────────────────────────┘
Figure 3: Quick reference for choosing the right index type based on your query operators
GIN vs GiST for Full-Text Search

GIN indexes are faster for lookups but slower to build and update, making them ideal for read-heavy workloads. GiST indexes are faster to update but slower for lookups, and they can be lossy (requiring a recheck). For most production full-text search, GIN is the better choice unless your table has very frequent writes and infrequent reads.

sql 
-- Bitmap Scan: PostgreSQL combines multiple indexes
-- When a single index is not selective enough,
-- PostgreSQL may use a Bitmap Heap Scan

EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42
   OR status = 'urgent';

-- Output may show:
-- Bitmap Heap Scan on orders
--   Recheck Cond: ((customer_id = 42) OR (status = 'urgent'))
--   -> BitmapOr
--        -> Bitmap Index Scan on idx_orders_customer_id
--             Index Cond: (customer_id = 42)
--        -> Bitmap Index Scan on idx_orders_status
--             Index Cond: (status = 'urgent')

-- Monitor index hit rates to evaluate effectiveness
SELECT
    relname AS table_name,
    ROUND(
        100.0 * idx_scan / NULLIF(seq_scan + idx_scan, 0), 2
    ) AS index_usage_pct,
    seq_scan,
    idx_scan
FROM pg_stat_user_tables
WHERE seq_scan + idx_scan > 0
ORDER BY index_usage_pct ASC;

5. Covering Indexes and Index-Only Scans

A covering index includes all the columns a query needs, allowing PostgreSQL to satisfy the query entirely from the index without ever visiting the table heap. This is called an Index Only Scan and is the fastest possible data access pattern. PostgreSQL 11 introduced the INCLUDE clause specifically for building covering indexes efficiently.

The key difference between putting a column in the index key vs the INCLUDE list: key columns are sorted and can be used for filtering and ordering. INCLUDE columns are just stored alongside the leaf entries — they cannot be used for filtering, but they make the index covering without bloating the internal tree structure.

sql 
-- Query: we only need customer_id and total_amount for completed orders
SELECT customer_id, total_amount
FROM orders
WHERE status = 'completed';

-- Covering index using INCLUDE (PostgreSQL 11+)
CREATE INDEX idx_orders_status_covering
    ON orders (status)
    INCLUDE (customer_id, total_amount);

-- EXPLAIN ANALYZE will show "Index Only Scan" instead of "Index Scan"
-- This avoids heap fetches entirely — significant speedup on large tables
EXPLAIN ANALYZE
SELECT customer_id, total_amount
FROM orders
WHERE status = 'completed';

-- Check visibility map coverage (needed for index-only scans)
SELECT
    relname,
    n_live_tup,
    n_dead_tup,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables
WHERE relname = 'orders';
Pro Tip

Index-Only Scans require the visibility map to be up to date, which means VACUUM must have run recently. If you see "Index Scan" instead of "Index Only Scan" on a covering index, run VACUUM orders and try again. In production, ensure autovacuum is properly configured.

6. Partial and Expression Indexes

A partial index only indexes rows that match a WHERE condition. This makes the index smaller and faster for queries that always filter on that condition. An expression index indexes the result of an expression rather than a raw column.

sql 
-- Partial index: only index active orders (90% of queries filter on this)
CREATE INDEX idx_orders_active
    ON orders (customer_id, created_at)
    WHERE status = 'active';

-- This query uses the partial index
SELECT * FROM orders
WHERE status = 'active'
  AND customer_id = 42;

-- Expression index: index on lowercase email for case-insensitive lookups
CREATE INDEX idx_users_email_lower
    ON users (LOWER(email));

-- The query must use the same expression to hit the index
SELECT * FROM users
WHERE LOWER(email) = 'alice@example.com';

-- Expression index on JSONB field
CREATE INDEX idx_products_brand
    ON products ((metadata ->> 'brand'));

-- Find unused indexes to clean up
SELECT
    schemaname,
    relname AS table_name,
    indexrelname AS index_name,
    idx_scan AS times_used,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
Warning: Index Bloat

Every index must be updated on every INSERT, UPDATE, and DELETE. Too many indexes on a heavily-written table can dramatically slow down writes. Always audit your indexes periodically and drop any that are unused. The query above using pg_stat_user_indexes with idx_scan = 0 identifies candidates for removal.

sql 
-- Concurrent index creation (zero downtime in production)
-- Regular CREATE INDEX locks the table for writes
-- CONCURRENTLY avoids that lock at the cost of slower build

CREATE INDEX CONCURRENTLY idx_orders_customer_id_new
    ON orders (customer_id);

-- If CONCURRENTLY fails mid-way, it leaves an INVALID index
-- Check for invalid indexes:
SELECT
    indexrelname AS index_name,
    indisvalid AS is_valid
FROM pg_stat_user_indexes sui
JOIN pg_index i ON i.indexrelid = sui.indexrelid
WHERE NOT indisvalid;

-- Drop invalid index and retry
DROP INDEX CONCURRENTLY idx_orders_customer_id_new;

-- REINDEX to rebuild a bloated index
REINDEX INDEX CONCURRENTLY idx_orders_customer_id;

-- Check total index size for a table
SELECT
    relname AS table_name,
    pg_size_pretty(pg_table_size(relid)) AS table_size,
    pg_size_pretty(pg_indexes_size(relid)) AS all_indexes_size,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size
FROM pg_stat_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 10;
Pro Tip: Statistics and the Planner

The query planner relies on table statistics (collected by ANALYZE) to decide whether to use an index. After loading a large batch of data or creating a new index, run ANALYZE table_name to update statistics. Stale statistics are the number one reason the planner makes poor choices.

Self-Check

Before moving to Chapter 07, honestly answer these questions.

When would PostgreSQL choose a sequential scan over an index scan?

PostgreSQL chooses a sequential scan when the query returns a large fraction of the table (typically more than 5-10%). In that case, the overhead of random I/O from an index scan is more expensive than reading the entire table sequentially. The planner also prefers sequential scans when the table is very small (fits in a few pages), when statistics are outdated, or when enable_indexscan is disabled.

What is the difference between an Index Scan and an Index Only Scan?

An Index Scan reads the index to find matching row pointers (TIDs), then fetches the actual rows from the table heap to get all requested columns. An Index Only Scan can satisfy the query entirely from the index because the index contains all needed columns (a covering index). The Index Only Scan avoids heap fetches entirely, making it significantly faster — but it requires the visibility map to be up to date (recent VACUUM).

Why is column order important in a composite index?

A composite B-tree index on (a, b, c) stores data sorted first by a, then by b within equal a values, then by c. This means the index can efficiently serve queries on a, a + b, or a + b + c, but not queries on b alone or c alone. Put the most commonly filtered column first, and the most selective column (highest cardinality) next.
