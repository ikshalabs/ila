Chapter 03 · PostgreSQL Basics

Table Design and Normalization

Learn to design tables with proper primary keys, foreign keys, and constraints. Understand normalization forms and how the ad portal's entity relationships prevent data anomalies.

Estimated: 3–4 hours
5 Key Concepts
3 Assignments

Why This Matters

A poorly designed database schema is technical debt that compounds with every feature. In an ad portal, if campaign data is scattered across denormalized tables, changing a bid price might require updating five different places — and missing one means inconsistent data, incorrect billing, and angry advertisers.

Table design is the architectural blueprint of your data. Get it right, and every query is simple, every update is atomic, and every report is trustworthy. Get it wrong, and you will spend the next two years fighting data integrity issues that no amount of application code can solve.

Key Insight

The ad portal schema has over 100 tables with carefully designed foreign key relationships. An account has many campaigns, each campaign has many ad_groups, and each ad group has many ad_group_locations, ad_group_filters, and ad_group_ads. This hierarchy is not accidental — it is normalized design that prevents data anomalies.

Key Concepts

1. Primary Keys

A primary key uniquely identifies every row in a table. PostgreSQL enforces this with a unique index and a NOT NULL constraint. In the ad portal, every table uses BIGSERIAL (auto-incrementing BIGINT) as the primary key.

sql
-- BIGSERIAL automatically creates a sequence and NOT NULL constraint
CREATE TABLE accounts (
    id            BIGSERIAL PRIMARY KEY,
    company_name  VARCHAR(255),
    status        VARCHAR(50) DEFAULT 'onboarding',
    timezone      VARCHAR(50),
    enterprise    BOOLEAN DEFAULT FALSE,
    created_at    TIMESTAMP NOT NULL,
    updated_at    TIMESTAMP NOT NULL
);

-- Composite primary key for stats tables (no single unique column)
CREATE TABLE ad_group_stats (
    dt            TIMESTAMP,
    account_id    BIGINT,
    ad_group_id   BIGINT,
    campaign_id   BIGINT,
    total_matches BIGINT,
    total_searches BIGINT,
    created_at    TIMESTAMP NOT NULL,
    updated_at    TIMESTAMP NOT NULL,
    UNIQUE (campaign_id, dt, ad_group_id)
);

2. Foreign Keys and Referential Integrity

Foreign keys create relationships between tables and enforce referential integrity — you cannot insert a campaign for an account that does not exist. This is the database's guarantee that your data relationships are always valid.

sql
CREATE TABLE campaigns (
    id              BIGSERIAL PRIMARY KEY,
    account_id      BIGINT NOT NULL
        REFERENCES accounts(id) ON DELETE CASCADE,
    name            VARCHAR(255),
    active          BOOLEAN DEFAULT TRUE,
    monthly_budget  NUMERIC(20,2),
    base_price      NUMERIC(12,2) DEFAULT 0.00,
    created_at      TIMESTAMP NOT NULL,
    updated_at      TIMESTAMP NOT NULL
);

CREATE TABLE ad_groups (
    id              BIGSERIAL PRIMARY KEY,
    campaign_id     BIGINT NOT NULL
        REFERENCES campaigns(id),
    account_id      BIGINT NOT NULL
        REFERENCES accounts(id),
    ad_group_name   VARCHAR(255),
    base_bid_price  NUMERIC(20,2),
    active          BOOLEAN,
    created_at      TIMESTAMP NOT NULL,
    updated_at      TIMESTAMP NOT NULL
);

-- Try inserting a campaign for a non-existent account:
INSERT INTO campaigns (account_id, name, created_at, updated_at)
VALUES (99999, 'Ghost Campaign', NOW(), NOW());
-- ERROR: insert or update on table "campaigns" violates
-- foreign key constraint "campaigns_account_id_fkey"
ON DELETE Strategies

ON DELETE CASCADE automatically deletes child rows when a parent is deleted (dangerous for ad data). ON DELETE RESTRICT (the default) prevents deleting a parent with children. ON DELETE SET NULL sets the foreign key to NULL. The ad portal rarely uses CASCADE — instead it uses soft deletes on parent records, which leaves children intact for historical reporting.

3. Constraints

Constraints are rules the database enforces automatically. They catch bad data at the database level, which is more reliable than application-level validation alone.

sql
CREATE TABLE campaigns_v2 (
    id              BIGSERIAL PRIMARY KEY,
    account_id      BIGINT NOT NULL REFERENCES accounts(id),
    name            VARCHAR(255) NOT NULL,

    -- CHECK: budget must be positive
    monthly_budget  NUMERIC(20,2)
        CHECK (monthly_budget > 0),

    -- CHECK: min_price must be less than max_price
    min_price       NUMERIC(12,2),
    max_price       NUMERIC(12,2),
    CHECK (min_price <= max_price),

    -- DEFAULT: new campaigns default to active
    active          BOOLEAN DEFAULT TRUE NOT NULL,

    -- UNIQUE: no duplicate campaign names per account
    UNIQUE (account_id, name),

    created_at      TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMP NOT NULL DEFAULT NOW()
);

4. Normalization Forms

Normalization eliminates data redundancy and prevents anomalies. The ad portal schema follows third normal form (3NF) for operational tables.

  UNNORMALIZED (bad):
  ┌────────────────────────────────────────────────────────────┐
  │ campaigns                                                  │
  │ id │ name    │ account_name    │ account_email │ budget    │
  │  1 │ Auto CA │ Acme Insurance  │ a@acme.com   │ 15000.00 │
  │  2 │ Home TX │ Acme Insurance  │ a@acme.com   │ 8000.00  │
  │  3 │ Life NY │ Pacific Shield  │ p@pacific.com│ 22000.00 │
  └────────────────────────────────────────────────────────────┘
  Problem: "Acme Insurance" and "a@acme.com" are duplicated!

  NORMALIZED (good):
  ┌──────────────────────────┐     ┌──────────────────────┐
  │ accounts                 │     │ campaigns            │
  │ id │ name           │ email       │     │ id │ account_id │ name │ budget  │
  │  1 │ Acme Insurance │ a@acme.com  │     │  1 │     1      │ Auto │ 15000  │
  │  2 │ Pacific Shield │ p@pacific.. │     │  2 │     1      │ Home │  8000  │
  └──────────────────────────┘     │  3 │     2      │ Life │ 22000  │
                                    └──────────────────────┘
Figure 1: Normalization eliminates data duplication
  • 1NF — Every column contains atomic (single) values. No arrays or comma-separated lists in a column (except PostgreSQL arrays for specific use cases like languages).
  • 2NF — Every non-key column depends on the entire primary key, not just part of it.
  • 3NF — No non-key column depends on another non-key column. If account_name depends on account_id, it belongs in the accounts table, not campaigns.

5. Entity Relationships in the Ad Portal

Understanding the ad portal's entity relationships is essential for writing correct queries:

  accounts (1) ──────< (many) campaigns
  accounts (1) ──────< (many) ad_groups
  campaigns (1) ─────< (many) ad_groups
  ad_groups (1) ─────< (many) ad_group_locations
  ad_groups (1) ─────< (many) ad_group_filters
  ad_groups (1) ─────< (many) ad_group_ads
  campaigns (1) ─────< (many) leads
  accounts (1) ──────< (many) account_balances
  ad_groups (1) ─────< (many) ad_group_bid_change_histories
Figure 2: Core entity relationships in the ad portal
sql
-- One-to-many: account has many campaigns
-- The campaigns table has account_id referencing accounts.id

-- Many-to-many: ad_groups <-> ads (through ad_group_ads)
CREATE TABLE ad_group_ads (
    id          BIGSERIAL PRIMARY KEY,
    ad_id       BIGINT NOT NULL REFERENCES ads(id),
    ad_group_id BIGINT NOT NULL REFERENCES ad_groups(id),
    platform    VARCHAR(50),
    weight      INTEGER DEFAULT 100,
    created_at  TIMESTAMP NOT NULL,
    updated_at  TIMESTAMP NOT NULL
);

-- Self-referencing: admin_users can reference their manager
CREATE TABLE admin_users (
    id          BIGSERIAL PRIMARY KEY,
    email       VARCHAR(255) NOT NULL,
    first_name  VARCHAR(100),
    last_name   VARCHAR(100),
    manager_id  BIGINT REFERENCES admin_users(id),
    team_lead_id BIGINT REFERENCES admin_users(id),
    created_at  TIMESTAMP NOT NULL,
    updated_at  TIMESTAMP NOT NULL
);
Pro Tip: Index Foreign Keys

PostgreSQL does NOT automatically create indexes on foreign key columns. Always create them manually. Without an index on campaigns.account_id, a query like SELECT * FROM campaigns WHERE account_id = 7 does a full table scan. The ad portal has explicit indexes on every foreign key column.

Add business rule constraints and performance indexes to your schema.

  • Add CHECK constraint: monthly_budget > 0
  • Add CHECK constraint: min_price <= max_price

Self-Check

Why does PostgreSQL not auto-index foreign key columns?

Unlike some databases, PostgreSQL separates the concern of referential integrity (foreign key constraint) from performance optimization (index). The constraint ensures data correctness; the index improves query speed. You might have a foreign key you rarely query by, so an automatic index would waste space. PostgreSQL gives you the choice.

When would you denormalize intentionally?

Denormalization is acceptable for read-heavy reporting tables where join performance matters. The ad_group_stats table includes both campaign_id AND account_id even though campaign_id alone could determine the account. This redundancy avoids a join in the hot path of analytics queries. Denormalize only when you have measured a performance problem.

What is the soft-delete pattern and why does the ad portal use it?

Instead of DELETE FROM campaigns WHERE id = 42, the ad portal sets UPDATE campaigns SET discarded_at = NOW() WHERE id = 42. This preserves the record for historical reporting, audit trails, and the ability to "undelete." Every query filters with WHERE discarded_at IS NULL. The trade-off is slightly more complex queries but dramatically better data preservation.
