Chapter 03 · Phase 1: SQL Fundamentals

Table Design and Normalization

A well-designed schema is the difference between an application that scales effortlessly and one that collapses under its own weight. Learn to model real-world data using primary keys, foreign keys, constraints, and the principles of normalization that prevent data anomalies before they happen.

Estimated: 3–4 days
6 Key Concepts
3 Assignments

Why This Matters

Every application is only as good as the data model underneath it. You can have beautiful front-end code, a sophisticated API layer, and a perfectly configured deployment pipeline — but if your database schema is poorly designed, you will spend your career fighting data bugs, writing compensating queries, and apologizing for incorrect reports. Table design is the foundation that everything else rests on.

Normalization is the systematic process of organizing columns and tables to minimize data redundancy and prevent update anomalies. When the same piece of information is stored in multiple places, it inevitably drifts out of sync. A customer's address updated in one table but not another. A product price corrected in the catalog but stale in the orders history. These are not hypothetical problems — they are the daily reality of teams working with poorly normalized schemas.

PostgreSQL gives you powerful tools to enforce good design at the database level: primary keys, foreign keys, unique constraints, check constraints, not-null constraints, and exclusion constraints. When you use these correctly, the database itself becomes a guardian of data integrity. Invalid data is rejected before it ever reaches your application logic, eliminating entire categories of bugs.

Key Insight

The goal of normalization is not to create the maximum number of tables. It is to ensure that every fact is stored exactly once. When you need to update a customer's email, you change it in one row of one table — and every query in the system automatically reflects the change. That is the power of a properly normalized schema.

Key Concepts

1. Primary Keys and Identity

A primary key uniquely identifies every row in a table. It is the single most important constraint you will define. PostgreSQL enforces that primary key values are unique and never null. Behind the scenes, it automatically creates a B-tree index on the primary key columns, making lookups by ID extremely fast.

You have several options for generating primary key values. The serial type creates an auto-incrementing integer sequence. The bigserial type does the same with a larger range. The uuid type generates universally unique identifiers. In modern PostgreSQL (version 10+), the SQL-standard GENERATED ALWAYS AS IDENTITY syntax is preferred over serial.

sql 
-- Option 1: SERIAL (legacy but widely used)
CREATE TABLE customers_v1 (
    id    serial PRIMARY KEY,
    name  text   NOT NULL,
    email text   UNIQUE NOT NULL
);

-- Option 2: IDENTITY (SQL standard, preferred)
CREATE TABLE customers_v2 (
    id    integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name  text    NOT NULL,
    email text    UNIQUE NOT NULL
);

-- Option 3: UUID (distributed-friendly, no sequence contention)
CREATE TABLE customers_v3 (
    id    uuid DEFAULT gen_random_uuid() PRIMARY KEY,
    name  text NOT NULL,
    email text UNIQUE NOT NULL
);

-- Composite primary key (for junction/join tables)
CREATE TABLE order_items (
    order_id   integer NOT NULL,
    product_id integer NOT NULL,
    quantity   integer NOT NULL CHECK (quantity > 0),
    PRIMARY KEY (order_id, product_id)
);
Serial vs Identity

The serial type is a shortcut that creates a separate sequence object and sets a default. The problem is that it allows explicit inserts that can conflict with the sequence. GENERATED ALWAYS AS IDENTITY prevents manual ID assignment unless you explicitly override it with OVERRIDING SYSTEM VALUE. This is safer for production systems where accidental ID collisions can cause data corruption.

2. Foreign Keys and Referential Integrity

A foreign key creates a link between two tables by ensuring that values in one table correspond to values in another. This is how you model relationships: a customer has many orders, an order belongs to a customer. Without foreign keys, your application must manually verify every relationship — and inevitably, orphaned records creep in.

sql 
-- Parent table
CREATE TABLE customers (
    id    integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name  text    NOT NULL,
    email text    UNIQUE NOT NULL
);

-- Child table with foreign key
CREATE TABLE orders (
    id          integer     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id integer     NOT NULL REFERENCES customers(id),
    order_date  timestamptz DEFAULT now(),
    total       numeric(12,2) NOT NULL
);

-- Foreign key with explicit ON DELETE behavior
CREATE TABLE order_items (
    id         integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_id   integer NOT NULL,
    product_id integer NOT NULL,
    quantity   integer NOT NULL CHECK (quantity > 0),
    unit_price numeric(10,2) NOT NULL,

    CONSTRAINT fk_order
      FOREIGN KEY (order_id)
      REFERENCES orders(id)
      ON DELETE CASCADE,

    CONSTRAINT fk_product
      FOREIGN KEY (product_id)
      REFERENCES products(id)
      ON DELETE RESTRICT
);

The ON DELETE clause controls what happens when a referenced row is deleted. The options are: CASCADE (delete child rows automatically), RESTRICT (prevent deletion if children exist), SET NULL (set the foreign key column to null), and SET DEFAULT (set it to the column's default value). Choose carefully — CASCADE is convenient but can silently destroy data if misused.

3. Constraints: Your Data Safety Net

Constraints are rules that PostgreSQL enforces at the database level. They are your strongest tool for data integrity because they cannot be bypassed by application bugs, race conditions, or manual SQL scripts run during emergencies. If a constraint says an email must be unique, then no two rows can ever have the same email — regardless of how the data was inserted.

sql 
CREATE TABLE employees (
    id         integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    first_name text    NOT NULL,
    last_name  text    NOT NULL,
    email      text    UNIQUE NOT NULL,
    department text    NOT NULL,
    salary     numeric(10,2),
    hire_date  date    NOT NULL DEFAULT CURRENT_DATE,

    -- CHECK constraint: salary must be positive
    CONSTRAINT positive_salary CHECK (salary > 0),

    -- CHECK constraint: hire date cannot be in the future
    CONSTRAINT valid_hire_date CHECK (hire_date <= CURRENT_DATE),

    -- CHECK constraint: department must be from a known list
    CONSTRAINT valid_department CHECK (
        department IN ('Engineering', 'Sales', 'Marketing',
                       'Finance', 'Operations', 'HR')
    )
);

-- Add a constraint after table creation
ALTER TABLE employees
  ADD CONSTRAINT email_format
  CHECK (email ~* '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$');

-- Unique constraint across multiple columns
ALTER TABLE employees
  ADD CONSTRAINT unique_name_dept
  UNIQUE (first_name, last_name, department);
Pro Tip

Always name your constraints explicitly. When a constraint violation occurs, PostgreSQL reports the constraint name in the error message. ERROR: new row violates check constraint "positive_salary" is infinitely more useful than a generic unnamed constraint error. Your future self (and your on-call teammates) will thank you.

4. Normalization: First Through Third Normal Form

Normalization is a step-by-step process for eliminating redundancy. Each "normal form" builds on the previous one. In practice, most production schemas should achieve Third Normal Form (3NF). Going beyond 3NF is rarely necessary and can make queries unnecessarily complex.

First Normal Form (1NF) requires that every column contains only atomic (single) values and every row is unique. No arrays, no comma-separated lists, no repeated groups.

sql 
-- VIOLATION of 1NF: phone_numbers stores multiple values
CREATE TABLE contacts_bad (
    id            serial PRIMARY KEY,
    name          text,
    phone_numbers text  -- '555-0101, 555-0102, 555-0103'
);

-- CORRECT 1NF: separate table for phone numbers
CREATE TABLE contacts (
    id   serial PRIMARY KEY,
    name text NOT NULL
);

CREATE TABLE contact_phones (
    id         serial  PRIMARY KEY,
    contact_id integer NOT NULL REFERENCES contacts(id),
    phone      text    NOT NULL,
    label      text    DEFAULT 'mobile'
);

Second Normal Form (2NF) requires 1NF plus: every non-key column must depend on the entire primary key, not just part of it. This only matters for composite primary keys. If a column depends on only one part of a composite key, it belongs in a separate table.

sql 
-- VIOLATION of 2NF: product_name depends only on product_id,
-- not on the full composite key (order_id, product_id)
CREATE TABLE order_details_bad (
    order_id     integer,
    product_id   integer,
    product_name text,       -- depends only on product_id!
    quantity     integer,
    PRIMARY KEY (order_id, product_id)
);

-- CORRECT 2NF: product_name lives in the products table
CREATE TABLE products (
    id   integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text    NOT NULL
);

CREATE TABLE order_details (
    order_id   integer REFERENCES orders(id),
    product_id integer REFERENCES products(id),
    quantity   integer NOT NULL CHECK (quantity > 0),
    PRIMARY KEY (order_id, product_id)
);

Third Normal Form (3NF) requires 2NF plus: no non-key column depends on another non-key column. This eliminates transitive dependencies. If column C depends on column B, and column B depends on the primary key, then C should be in a separate table keyed by B.

sql 
-- VIOLATION of 3NF: city and state depend on zip_code,
-- not directly on the primary key
CREATE TABLE customers_bad (
    id       serial PRIMARY KEY,
    name     text,
    zip_code text,
    city     text,   -- depends on zip_code, not on id
    state    text    -- depends on zip_code, not on id
);

-- CORRECT 3NF: zip_code details in their own table
CREATE TABLE zip_codes (
    zip_code text PRIMARY KEY,
    city     text NOT NULL,
    state    text NOT NULL
);

CREATE TABLE customers_good (
    id       serial PRIMARY KEY,
    name     text   NOT NULL,
    zip_code text   REFERENCES zip_codes(zip_code)
);
The Normalization Tradeoff

Normalization reduces redundancy but increases the number of JOINs needed to reconstruct data. In OLTP (transactional) systems, 3NF is ideal because writes are frequent and data integrity is paramount. In OLAP (analytical) systems, some denormalization is common to speed up read-heavy queries. The key is to normalize first, then selectively denormalize when you have measured performance evidence.

5. Modeling Relationships: One-to-Many, Many-to-Many, One-to-One

Real-world data has three fundamental relationship types. Understanding how to model each in PostgreSQL is essential for every schema you will ever design.

One-to-Many (1:N) is the most common relationship. One customer has many orders. One department has many employees. Model this with a foreign key on the "many" side pointing to the "one" side.

Many-to-Many (M:N) requires a junction table (also called a join table or bridge table). A student enrolls in many courses; a course has many students. The junction table holds pairs of foreign keys.

sql 
-- Many-to-Many: Students and Courses
CREATE TABLE students (
    id   integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text    NOT NULL
);

CREATE TABLE courses (
    id    integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    title text    NOT NULL,
    code  text    UNIQUE NOT NULL
);

-- Junction table: the relationship itself can carry data
CREATE TABLE enrollments (
    student_id  integer NOT NULL REFERENCES students(id) ON DELETE CASCADE,
    course_id   integer NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
    enrolled_at timestamptz DEFAULT now(),
    grade       char(2),
    PRIMARY KEY (student_id, course_id)
);

-- One-to-One: each user has exactly one profile
CREATE TABLE users (
    id    integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email text    UNIQUE NOT NULL
);

CREATE TABLE user_profiles (
    user_id integer PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    bio     text,
    avatar  text,
    theme   text DEFAULT 'light'
);
Pro Tip

One-to-one relationships are modeled by making the foreign key column also the primary key of the child table, or by adding a UNIQUE constraint to the foreign key. This ensures that each parent row has at most one child. Use one-to-one splits when a table has many nullable columns that are only relevant for a subset of rows.

6. Putting It All Together: A Complete Schema

Let us design a complete, normalized schema for a small e-commerce application. This schema demonstrates every concept from this chapter working together: identity columns, foreign keys, check constraints, junction tables, and proper normalization.

sql 
-- ── Complete E-Commerce Schema ──

CREATE TABLE categories (
    id   integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text    UNIQUE NOT NULL,
    slug text    UNIQUE NOT NULL
);

CREATE TABLE products (
    id          integer     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        text        NOT NULL,
    description text,
    price       numeric(10,2) NOT NULL CHECK (price >= 0),
    sku         text        UNIQUE NOT NULL,
    category_id integer     NOT NULL REFERENCES categories(id),
    is_active   boolean     DEFAULT true,
    created_at  timestamptz DEFAULT now()
);

CREATE TABLE customers (
    id         integer     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    first_name text        NOT NULL,
    last_name  text        NOT NULL,
    email      text        UNIQUE NOT NULL,
    created_at timestamptz DEFAULT now()
);

CREATE TABLE addresses (
    id          integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id integer NOT NULL REFERENCES customers(id) ON DELETE CASCADE,
    line1       text    NOT NULL,
    line2       text,
    city        text    NOT NULL,
    state       text    NOT NULL,
    zip         text    NOT NULL,
    is_default  boolean DEFAULT false
);

CREATE TABLE orders (
    id              integer     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id     integer     NOT NULL REFERENCES customers(id),
    shipping_addr   integer     REFERENCES addresses(id),
    status          text        NOT NULL DEFAULT 'pending',
    ordered_at      timestamptz DEFAULT now(),

    CONSTRAINT valid_status CHECK (
        status IN ('pending', 'confirmed', 'shipped',
                   'delivered', 'cancelled')
    )
);

CREATE TABLE order_items (
    id         integer     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_id   integer     NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
    product_id integer     NOT NULL REFERENCES products(id),
    quantity   integer     NOT NULL CHECK (quantity > 0),
    unit_price numeric(10,2) NOT NULL CHECK (unit_price >= 0),

    UNIQUE (order_id, product_id)
);

-- Many-to-many: products can have multiple tags
CREATE TABLE tags (
    id   integer GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name text    UNIQUE NOT NULL
);

CREATE TABLE product_tags (
    product_id integer NOT NULL REFERENCES products(id) ON DELETE CASCADE,
    tag_id     integer NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (product_id, tag_id)
);
Common Mistake: Storing Calculated Totals

Do not store order_total as a column in the orders table. This is a derived value that can be calculated as SUM(quantity * unit_price) from order_items. Storing it creates a redundancy that can drift out of sync. If you need fast access to the total, use a materialized view or a computed column, not a manually maintained duplicate.

Assignment 3
Level 3 — Production-Ready

Schema Migration Script with Rollback

Write a forward migration and a corresponding rollback script for evolving an existing schema without data loss.

  • Start with the e-commerce schema from this chapter and create a V1_baseline.sql
  • Write V2_up.sql: add a reviews table, add a phone column to customers, add an index on orders.customer_id

Self-Check

Before moving to Chapter 04, honestly answer these questions. If any answer feels uncertain, revisit the relevant section.

What is the difference between serial and GENERATED ALWAYS AS IDENTITY?

serial creates a sequence and sets a default, but allows explicit value insertion that can conflict with the sequence counter. GENERATED ALWAYS AS IDENTITY is the SQL-standard approach that prevents manual ID assignment unless you explicitly use OVERRIDING SYSTEM VALUE. Identity columns are safer because they prevent accidental sequence/value mismatches and make the intent clearer.

Can you explain what a transitive dependency is and why it violates 3NF?

A transitive dependency exists when column C depends on column B, which depends on the primary key A. For example, if zip_code determines city and state, then city and state transitively depend on the primary key through zip_code. This violates 3NF because updating a zip code's city name requires changing every row with that zip code. The fix is to create a separate zip_codes table.

When would you use ON DELETE CASCADE vs ON DELETE RESTRICT?

Use CASCADE when child rows have no meaning without their parent — for example, order_items when an order is deleted, or comments when a post is deleted. Use RESTRICT (or its sibling NO ACTION) when deletion should be blocked to prevent accidental data loss — for example, preventing deletion of a product that has been referenced in past orders.

Why should you name your constraints explicitly?

PostgreSQL generates automatic names like employees_salary_check, but these are cryptic and can change if you alter the table. Explicit names like positive_salary make error messages self-documenting: ERROR: new row violates check constraint "positive_salary" immediately tells you what went wrong. Named constraints are also easier to reference in ALTER TABLE DROP CONSTRAINT commands.

How do you model a many-to-many relationship in SQL?

Create a junction (join) table with two foreign key columns, each referencing one of the related tables. The primary key of the junction table is typically a composite of both foreign key columns. The junction table can also carry its own attributes — for example, an enrollments table linking students and courses can include a grade column and an enrolled_at timestamp.
