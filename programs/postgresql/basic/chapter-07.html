Chapter 07 · PostgreSQL Basic

Transactions and Concurrency

Understand ACID properties, isolation levels, row-level locking, and deadlock prevention. Learn how PostgreSQL MVCC enables concurrent bid processing without data corruption.

Estimated: 3–4 hours
5 Key Concepts
3 Assignments

Why This Matters

In an ad bidding platform, multiple operations must succeed or fail together. When a bid is accepted, the system must: (1) record the lead, (2) decrement the campaign budget, (3) update the account balance. If step 2 succeeds but step 3 fails, the account balance is wrong and the advertiser is overcharged. Transactions guarantee atomicity — all steps succeed or none do.

Concurrency is equally critical. Hundreds of bid requests arrive simultaneously. Without proper isolation, two requests might read the same budget balance, both approve a bid, and overspend the campaign. Understanding isolation levels and PostgreSQL's MVCC architecture prevents these race conditions.

Key Insight

ACID is not academic theory — it is the reason the ad portal can process thousands of concurrent bid requests without data corruption. Atomicity ensures complete operations, Consistency enforces constraints, Isolation prevents race conditions, and Durability guarantees nothing is lost after a crash.

Key Concepts

1. ACID Properties

  • Atomicity — A transaction is all-or-nothing. If any statement fails, all changes are rolled back.
  • Consistency — Transactions move the database from one valid state to another. Constraints are enforced.
  • Isolation — Concurrent transactions don't see each other's uncommitted changes (by default).
  • Durability — Once committed, data survives crashes. PostgreSQL uses WAL (Write-Ahead Log).
sql
-- Transaction: accept a bid and update balances atomically
BEGIN;

-- 1. Insert the lead record
INSERT INTO leads (account_id, campaign_id, lead_type_id, product_type_id, created_at, updated_at)
VALUES (7, 42, 1, 1, NOW(), NOW());

-- 2. Decrement the account balance
UPDATE account_balances
SET current_balance = current_balance - 12.50, updated_at = NOW()
WHERE user_id = 7;

-- 3. If everything succeeds, commit
COMMIT;

-- If anything fails, roll back EVERYTHING
-- ROLLBACK;

2. Isolation Levels

sql
-- Read Committed (default) — sees committed changes from other transactions
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Repeatable Read — snapshot at start, no phantom reads
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- Serializable — strongest isolation, transactions run as if sequential
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- Check current isolation level
SHOW transaction_isolation;

3. Row-Level Locking

sql
-- SELECT FOR UPDATE: lock the row to prevent concurrent modification
BEGIN;
SELECT current_balance
FROM account_balances
WHERE user_id = 7
FOR UPDATE;

-- Now no other transaction can modify this row until we COMMIT/ROLLBACK
UPDATE account_balances
SET current_balance = current_balance - 12.50
WHERE user_id = 7;
COMMIT;

-- SKIP LOCKED: skip rows locked by other transactions (useful for job queues)
SELECT * FROM leads
WHERE disposition IS NULL
LIMIT 1
FOR UPDATE SKIP LOCKED;

4. Deadlock Prevention

Deadlocks occur when two transactions each hold a lock the other needs. PostgreSQL detects deadlocks and kills one transaction. Prevent them by always locking resources in the same order.

Deadlock Example

Transaction A locks campaign 1, then tries to lock campaign 2. Transaction B locks campaign 2, then tries to lock campaign 1. Both are stuck. PostgreSQL will abort one. Prevention: always lock campaigns in ID order (lock campaign 1 first, then 2).

5. MVCC (Multi-Version Concurrency Control)

PostgreSQL uses MVCC to allow readers and writers to operate simultaneously without blocking each other. Instead of locking rows for reads, PostgreSQL keeps multiple versions of each row and shows each transaction the version that was current when the transaction started.

Pro Tip: MVCC and VACUUM

MVCC creates "dead tuples" — old row versions that no transaction needs anymore. VACUUM reclaims this space. The ad portal relies on autovacuum to keep tables healthy. We will cover VACUUM tuning in the Advanced course (Chapter 06).

Self-Check

What does ACID stand for and why does each property matter for ad-tech?

Atomicity (bid acceptance is all-or-nothing), Consistency (constraints prevent invalid balances), Isolation (concurrent bids do not interfere), Durability (committed transactions survive crashes). Without ACID, an ad portal would have incorrect balances, lost leads, and data corruption.

What is the difference between READ COMMITTED and REPEATABLE READ?

READ COMMITTED (default) sees committed changes from other transactions between statements. REPEATABLE READ takes a snapshot at the start and sees a consistent view throughout. Use REPEATABLE READ for reports that must see consistent data across multiple queries.

How does MVCC allow readers and writers to coexist?

Instead of blocking readers when a writer modifies a row, PostgreSQL keeps both the old and new version. Readers see the old version; the writer works on the new one. After commit, new readers see the updated version. This is why PostgreSQL rarely has read-write contention.
