Chapter 07 · Phase 4: Applied PostgreSQL

Transactions and Concurrency

Learn how PostgreSQL keeps data consistent when multiple users read and write simultaneously. Master ACID properties, isolation levels, MVCC internals, locking strategies, and deadlock prevention — the foundation of reliable data systems.

Estimated: 3–4 hours
3 Advancement Levels
3 Assignments

Why This Matters

In a single-user system, data consistency is trivial. But the moment two users try to update the same bank balance, book the same airline seat, or decrement the same inventory count, you face the fundamental challenge of concurrent data access. Without transactions and proper isolation, you get lost updates, phantom reads, and corrupted data.

PostgreSQL solves this with a sophisticated Multi-Version Concurrency Control (MVCC) system that allows readers and writers to operate simultaneously without blocking each other. Understanding how MVCC works — and what guarantees each isolation level provides — is essential for building applications that are both correct and performant under concurrent load.

This chapter covers the theory and practice of transactions: the ACID properties that define correctness, the isolation levels that control visibility, the locking mechanisms that prevent conflicts, and the deadlock detection system that keeps everything moving.

Key Insight

PostgreSQL never lets readers block writers or writers block readers. This is the core advantage of MVCC. While other databases use locking to serialize access, PostgreSQL maintains multiple versions of each row so concurrent transactions can each see a consistent snapshot of the data without waiting.

Key Concepts

1. ACID Properties Explained

ACID is the set of guarantees that every relational transaction must provide. These are not optional features — they are the contract between your application and the database.

  • Atomicity — A transaction is all-or-nothing. Either every statement succeeds, or none of them take effect. There is no partial commit.
  • Consistency — A transaction moves the database from one valid state to another. Constraints (NOT NULL, UNIQUE, FOREIGN KEY, CHECK) are enforced at commit time.
  • Isolation — Concurrent transactions behave as if they were running serially. The degree of isolation is configurable via isolation levels.
  • Durability — Once a transaction commits, its changes survive even if the server crashes immediately after. PostgreSQL achieves this through Write-Ahead Logging (WAL).
  ACID Guarantees in Practice
 ┌──────────────────────────────────────────────────────────────┐
 │                                                              │
 │  BEGIN;                                                      │
 │    UPDATE accounts SET balance = balance - 100 WHERE id = 1; │
 │    UPDATE accounts SET balance = balance + 100 WHERE id = 2; │
 │  COMMIT;                                                     │
 │                                                              │
 │  Atomicity:   Both UPDATEs succeed, or neither does          │
 │  Consistency: Total balance across accounts remains the same │

2. Transaction Syntax (BEGIN, COMMIT, ROLLBACK, SAVEPOINT)

PostgreSQL provides full transaction control with BEGIN, COMMIT, ROLLBACK, and SAVEPOINT. Savepoints allow you to roll back part of a transaction without aborting the entire thing — essential for error handling in complex business logic.

sql 
-- Basic transaction
BEGIN;
  INSERT INTO orders (customer_id, total_amount, status)
  VALUES (42, 199.99, 'pending');

  UPDATE inventory
  SET quantity = quantity - 1
  WHERE product_id = 101 AND quantity > 0;
COMMIT;

-- Rollback if something goes wrong
BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  -- Oops, wrong amount!
ROLLBACK;  -- Nothing happened. Balance unchanged.

-- Savepoints for partial rollback
BEGIN;
  INSERT INTO orders (customer_id, total_amount, status)
  VALUES (42, 199.99, 'pending');

  SAVEPOINT before_notification;

  INSERT INTO notifications (user_id, message)
  VALUES (42, 'Order placed!');
  -- If notification insert fails:
  ROLLBACK TO SAVEPOINT before_notification;
  -- The order INSERT is still intact!

COMMIT;  -- Order is committed even if notification failed
Error Handling in PostgreSQL

In PostgreSQL, any error inside a transaction puts it in an aborted state. No further commands will execute until you issue ROLLBACK (or ROLLBACK TO SAVEPOINT). This is different from MySQL, where you can continue after an error. Always use savepoints in application code when you want to handle errors gracefully without aborting the entire transaction.

3. Isolation Levels (Read Uncommitted through Serializable)

PostgreSQL supports four SQL standard isolation levels, though in practice it treats Read Uncommitted the same as Read Committed (it never allows dirty reads). Each level provides different protections against concurrency anomalies. 

  Isolation Level        Dirty Read  Non-Repeatable Read  Phantom Read  Serialization Anomaly
 ┌─────────────────────┬────────────┬────────────────────┬─────────────┬──────────────────────┐
 │ Read Uncommitted*   │  No**      │  Possible          │  Possible   │  Possible            │
 │ Read Committed      │  No        │  Possible          │  Possible   │  Possible            │
 │ Repeatable Read     │  No        │  No                │  No***      │  Possible            │
 │ Serializable        │  No        │  No                │  No         │  No                  │
 └─────────────────────┴────────────┴────────────────────┴─────────────┴──────────────────────┘
  *  PG treats Read Uncommitted as Read Committed
  ** PG never allows dirty reads at any level
  *** PG's Repeatable Read also prevents phantoms (stricter than SQL standard)
Figure 2: Isolation levels and the anomalies they prevent in PostgreSQL
sql 
-- Set isolation level for a single transaction
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
  SELECT balance FROM accounts WHERE id = 1;
  -- Even if another transaction commits a change to this row,
  -- we will see the same balance throughout our transaction
  SELECT balance FROM accounts WHERE id = 1;
  -- Same result as above — snapshot is frozen at BEGIN
COMMIT;

-- Set default isolation level for a session
SET default_transaction_isolation = 'serializable';

-- Serializable: prevents ALL anomalies (but may abort transactions)
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT SUM(balance) FROM accounts;
  UPDATE accounts SET balance = balance + 100 WHERE id = 1;
COMMIT;
-- If a concurrent serializable transaction conflicts,
-- one of them will get: ERROR: could not serialize access
-- Your app must retry the failed transaction!

-- Check the current isolation level
SHOW transaction_isolation;
Pro Tip: Read Committed in Practice

Read Committed is the default in PostgreSQL. Each statement within the transaction sees the latest committed data at the time that statement starts. This means two identical SELECT statements in the same transaction can return different results if another transaction commits between them. For most web applications, this is perfectly acceptable and provides the best performance/correctness trade-off.

sql 
-- Demonstrating non-repeatable read under READ COMMITTED
-- Session 1:                         Session 2:
-- BEGIN;                             BEGIN;
-- SELECT balance FROM accounts

--                                      SET balance=500
--                                      WHERE id=1;
--                                    COMMIT;
-- SELECT balance FROM accounts       
--   WHERE id=1;  -- returns 500!    
-- (non-repeatable read)              
-- COMMIT;                            

-- Under REPEATABLE READ, the second SELECT still returns 1000
-- because the snapshot was taken at BEGIN, not at each statement

4. MVCC — How PostgreSQL Handles Concurrency

MVCC (Multi-Version Concurrency Control) is the engine that makes PostgreSQL's concurrency model work. Instead of locking rows to prevent concurrent access, PostgreSQL keeps multiple versions of each row. Each version is tagged with the transaction ID (xid) that created it and the transaction ID that deleted (or updated) it. When a transaction reads data, it only sees row versions committed before its snapshot was taken.

sql 
-- See the hidden system columns that power MVCC
SELECT
    xmin,          -- Transaction ID that created this row version
    xmax,          -- Transaction ID that deleted/updated this version (0 = alive)
    ctid,          -- Physical location (page, tuple) on disk
    id,
    name,
    balance
FROM accounts
WHERE id = 1;

-- When you UPDATE a row, PostgreSQL:
-- 1. Marks the old version with xmax = current_xid
-- 2. Creates a NEW version with xmin = current_xid
-- 3. The old version remains on disk until VACUUM cleans it up

BEGIN;
  UPDATE accounts SET balance = 500 WHERE id = 1;
  -- At this point, two versions of row id=1 exist on disk:
  --   Old: xmin=100, xmax=200 (the current transaction)
  --   New: xmin=200, xmax=0   (visible after commit)
COMMIT;

-- Check for bloat from dead tuples
SELECT
    relname AS table_name,
    n_live_tup,
    n_dead_tup,
    ROUND(
        100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2
    ) AS dead_pct,
    last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC
LIMIT 10;

  MVCC: How an UPDATE Creates Two Row Versions
 ┌────────────────────────────────────────────────────────────────┐
 │                                                                │
 │  Before UPDATE (xid 200):                                      │
 │  ┌──────────────────────────────────────────┐                  │
 │  │ xmin=100  xmax=0  │ id=1  balance=1000   │  (live)         │
 │  └──────────────────────────────────────────┘                  │
 │                                                                │
 │  After UPDATE (xid 200), before VACUUM:                        │
 │  ┌──────────────────────────────────────────┐                  │
 │  │ xmin=100  xmax=200 │ id=1  balance=1000  │  (dead tuple)   │
 │  └──────────────────────────────────────────┘                  │
 │  ┌──────────────────────────────────────────┐                  │
 │  │ xmin=200  xmax=0   │ id=1  balance=500   │  (live)         │
 │  └──────────────────────────────────────────┘                  │
 │                                                                │
 │  VACUUM removes the dead tuple when no transaction needs it    │
 │                                                                │
 └────────────────────────────────────────────────────────────────┘
Figure 3: MVCC creates a new row version on UPDATE; the old version becomes a dead tuple
Pro Tip: VACUUM and Table Bloat

Dead tuples from MVCC accumulate over time and cause table bloat. Autovacuum runs automatically to clean them up, but heavy-write tables may need more aggressive autovacuum settings. Monitor n_dead_tup in pg_stat_user_tables and adjust autovacuum_vacuum_scale_factor if bloat becomes a problem.

5. Locking Types and Strategies

While MVCC eliminates most read/write conflicts, PostgreSQL still uses locks for write/write conflicts and DDL operations. Understanding lock types helps you design applications that avoid unnecessary contention.

sql 
-- Row-level locks: SELECT ... FOR UPDATE
BEGIN;
  SELECT * FROM accounts
  WHERE id = 1
  FOR UPDATE;  -- Lock this row; other FOR UPDATE on same row waits

  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;  -- Lock released

-- FOR UPDATE SKIP LOCKED — non-blocking queue pattern
BEGIN;
  SELECT * FROM job_queue
  WHERE status = 'pending'
  ORDER BY created_at
  LIMIT 1
  FOR UPDATE SKIP LOCKED;  -- Skip rows locked by other workers

  UPDATE job_queue SET status = 'processing' WHERE id = 123;
COMMIT;

-- FOR KEY SHARE — weakest lock, allows most concurrent access
SELECT * FROM products WHERE id = 42 FOR KEY SHARE;
-- Prevents DELETE and key updates, but allows non-key column updates

-- Advisory locks: application-level locking
SELECT pg_advisory_lock(12345);  -- Acquire named lock
-- ... do exclusive work ...
SELECT pg_advisory_unlock(12345);  -- Release lock

-- View current locks in the system
SELECT
    l.pid,
    l.locktype,
    l.mode,
    l.granted,
    a.query
FROM pg_locks l
JOIN pg_stat_activity a ON a.pid = l.pid
WHERE l.locktype = 'relation'
ORDER BY l.pid;

  PostgreSQL Lock Compatibility Matrix (Row-Level Locks)
 ┌─────────────────┬──────────┬──────────┬──────────┬──────────┐
 │ Requested \Held │ KEY SHARE│ SHARE    │NO KEY UPD│ UPDATE   │
 ├─────────────────┼──────────┼──────────┼──────────┼──────────┤
 │ FOR KEY SHARE   │    OK    │    OK    │    OK    │  BLOCK   │
 │ FOR SHARE       │    OK    │    OK    │  BLOCK   │  BLOCK   │
 │ FOR NO KEY UPD  │    OK    │  BLOCK   │  BLOCK   │  BLOCK   │
 │ FOR UPDATE      │  BLOCK   │  BLOCK   │  BLOCK   │  BLOCK   │
 └─────────────────┴──────────┴──────────┴──────────┴──────────┘
  OK = both locks can coexist | BLOCK = second transaction waits
Figure 4: Row-level lock compatibility — weaker locks allow more concurrency
FOR UPDATE SKIP LOCKED

The SKIP LOCKED clause is a game-changer for queue-based processing. Instead of blocking when a row is already locked, it simply skips that row and moves to the next one. This allows multiple workers to dequeue jobs concurrently without waiting for each other — no external queue system (Redis, RabbitMQ) required for many use cases.

6. Deadlock Detection and Prevention

A deadlock occurs when two or more transactions are each waiting for a lock held by the other, creating a circular dependency that can never resolve. PostgreSQL automatically detects deadlocks (checking every deadlock_timeout interval, default 1 second) and aborts the transaction that is easiest to roll back, returning an error to the client.

sql 
-- Deadlock example (DO NOT do this in production!)

-- Session 1:                          Session 2:
-- BEGIN;                              BEGIN;
-- UPDATE accounts SET balance=0       UPDATE accounts SET balance=0
--   WHERE id=1;  -- locks row 1         WHERE id=2;  -- locks row 2
--
-- UPDATE accounts SET balance=0       UPDATE accounts SET balance=0
--   WHERE id=2;  -- WAITS (row 2)       WHERE id=1;  -- WAITS (row 1)
--
-- DEADLOCK! PostgreSQL kills one session with:
-- ERROR: deadlock detected

-- Prevention strategy: always lock rows in the SAME ORDER
BEGIN;
  -- Always lock accounts in ascending ID order
  SELECT * FROM accounts
  WHERE id IN (1, 2)
  ORDER BY id  -- consistent ordering prevents deadlocks
  FOR UPDATE;

  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- Configure deadlock detection timeout (default: 1 second)
SHOW deadlock_timeout;
SET deadlock_timeout = '2s';

-- Set a statement timeout to prevent long-running lock waits
SET lock_timeout = '5s';  -- Abort if we can't get a lock within 5 seconds
SET statement_timeout = '30s';  -- Abort any statement running longer than 30s

-- Find blocked queries
SELECT
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid AND NOT bl.granted
JOIN pg_locks gl ON gl.locktype = bl.locktype
    AND gl.database IS NOT DISTINCT FROM bl.database
    AND gl.relation IS NOT DISTINCT FROM bl.relation
    AND gl.page IS NOT DISTINCT FROM bl.page
    AND gl.tuple IS NOT DISTINCT FROM bl.tuple
    AND gl.pid != bl.pid
    AND gl.granted
JOIN pg_stat_activity blocking ON blocking.pid = gl.pid;
Deadlock Prevention Rules

Rule 1: Always acquire locks in a consistent, deterministic order (e.g., ascending primary key). Rule 2: Keep transactions short. The longer a transaction holds locks, the higher the chance of a deadlock. Rule 3: Set lock_timeout to avoid indefinite waits. Rule 4: Your application must be prepared to retry transactions that fail due to deadlocks or serialization errors.

sql 
-- Optimistic concurrency control pattern
-- Instead of locking, use a version column to detect conflicts

CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    name TEXT NOT NULL,
    price NUMERIC(10,2),
    version INTEGER NOT NULL DEFAULT 1
);

-- Read the product and its version
SELECT id, name, price, version FROM products WHERE id = 1;
-- Returns: id=1, name='Widget', price=29.99, version=3

-- Update only if version matches (optimistic lock)
UPDATE products
SET price = 34.99, version = version + 1
WHERE id = 1 AND version = 3;
-- If 0 rows updated, another transaction changed it first — retry!

-- Long-running transaction monitoring
SELECT
    pid,
    NOW() - xact_start AS transaction_age,
    state,
    query
FROM pg_stat_activity
WHERE state != 'idle'
  AND xact_start IS NOT NULL
ORDER BY xact_start;

Self-Check

Before moving to Chapter 08, honestly answer these questions.

What happens if a transaction encounters an error in PostgreSQL?

Any error inside a transaction puts it into an aborted state. No further SQL commands will execute (they all return ERROR: current transaction is aborted) until you issue ROLLBACK or ROLLBACK TO SAVEPOINT. This is by design — PostgreSQL refuses to continue a transaction that may be in an inconsistent state. Use savepoints when you need to handle errors gracefully without aborting the entire transaction.

How does MVCC allow readers to not block writers?

When a writer updates a row, it creates a new version of that row (marked with its transaction ID) while leaving the old version intact. Readers running in a different transaction continue to see the old version based on their snapshot. They never need to wait for the writer to commit or roll back. This is why PostgreSQL can achieve high concurrency without read locks. The trade-off is dead tuple accumulation, which VACUUM cleans up.

When should you use SERIALIZABLE isolation?

Use SERIALIZABLE when your application requires that concurrent transactions produce the same result as some serial (one-at-a-time) execution. This is common in financial systems, reservation systems, and anywhere write-skew anomalies could cause data corruption. The trade-off is that PostgreSQL may abort transactions with serialization errors, so your application must implement retry logic. For most web applications, READ COMMITTED (the default) is sufficient.
