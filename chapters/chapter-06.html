Chapter 06 · Phase 3: Revenue Engine

Conversion Attribution and Background Jobs

Track conversion events and map them to campaigns and ad groups reliably. Move heavy tasks to async workers with robust retry logic and observability. Accurate attribution is the foundation for optimization and billing confidence.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Every ad platform exists to answer one question: "Did this campaign actually drive revenue?" Without reliable conversion attribution, advertisers cannot measure ROI, the bidding engine cannot optimize, and the billing system cannot charge with confidence. Attribution is not a reporting feature — it is the trust layer of the entire platform.

At the same time, conversion events arrive at high volume and unpredictable cadence. Processing them synchronously inside API request cycles would crush latency and create cascading failures under load. Background jobs solve this by decoupling ingestion from processing, but only if they are designed with retry safety, idempotency, and observability from day one.

The Cost of Getting This Wrong

Misattributed conversions lead to overbilling or underbilling — both destroy trust. Duplicate conversion records inflate metrics and corrupt optimization signals. Non-idempotent workers that retry on failure can double-charge advertisers or create phantom data. This chapter is where data integrity meets system reliability.

Key Concepts

1. Conversion Event Model

A conversion event represents a valuable action taken by a user after interacting with an ad — a purchase, a sign-up, a form submission. The challenge is reliably connecting that event back to the campaign and ad group that drove it.

Attribution requires attribution keys — identifiers that link the conversion to the original click or impression. These keys typically include a click_id, a campaign_id, and sometimes a device_fingerprint for cross-session matching. When the primary key is missing, fallback matching kicks in, using IP address, user agent, and timing windows to find the most likely source.

Key Insight

The match_confidence enum is critical for downstream systems. The billing engine may only count exact matches for invoicing, while the optimization engine uses all confidence levels with weighted scoring. Always record how you matched, not just that you matched.

2. Idempotency and Data Integrity

Conversion events can arrive more than once — pixel fires retry on network failures, webhook providers send duplicates, and users can double-click submit buttons. Without idempotency, each duplicate creates a phantom conversion that inflates metrics and corrupts billing.

The solution is an idempotency key — a unique identifier that the client generates and sends with each request. The server uses this key to detect duplicates: if a record with the same key already exists, the server returns the existing record instead of creating a new one.

ruby 
# app/controllers/api/v1/conversions_controller.rb
module Api
  module V1
    class ConversionsController < ApplicationController
      def create
        idempotency_key = request.headers["Idempotency-Key"] || params[:idempotency_key]

        unless idempotency_key.present?
          render json: { error: "Idempotency-Key header is required" },
                 status: :unprocessable_entity
          return
        end

        # Return existing record if idempotency key already used
        existing = current_account.conversion_events
                                  .find_by(idempotency_key: idempotency_key)
        if existing
          render json: ConversionSerializer.new(existing), status: :ok
          return
        end

        # Create new conversion event within a transaction
        conversion = nil
        ActiveRecord::Base.transaction(isolation: :serializable) do
          # Double-check inside transaction to prevent race conditions
          conversion = current_account.conversion_events
                                      .create_or_find_by!(idempotency_key: idempotency_key) do |event|
            event.assign_attributes(conversion_params)
            event.occurred_at ||= Time.current
          end
        end

        # Enqueue attribution matching as a background job
        AttributionMatchJob.perform_later(conversion.id)

        render json: ConversionSerializer.new(conversion),
               status: :created

      rescue ActiveRecord::SerializationFailure
        # Transaction conflict — safe to retry
        retry
      end

      private

      def conversion_params
        params.require(:conversion).permit(
          :event_type, :click_id, :revenue_cents,
          :occurred_at, :ip_address, :user_agent,
          metadata: {}
        )
      end
    end
  end
end
Transaction Isolation Matters

The :serializable isolation level prevents two concurrent requests with the same idempotency key from both inserting a record. Without it, a race condition between the find_by check and the create! call can produce duplicates. PostgreSQL's SERIALIZABLE mode detects this conflict and raises SerializationFailure, which we catch and retry safely.

How does create_or_find_by! differ from find_or_create_by!?

find_or_create_by! does a SELECT first, then INSERT if not found. Under concurrency, two threads can both see "not found" and both attempt to INSERT, causing a unique constraint violation. create_or_find_by! flips the order: it attempts the INSERT first, and if a unique constraint violation occurs, it falls back to a SELECT. This is inherently safer for concurrent idempotent writes because the database's unique index is the source of truth, not application-level checks.

For reconciliation, build a nightly job that compares conversion counts per campaign against external reporting sources. Log mismatches as structured events so your monitoring can alert when attribution drift exceeds a threshold:

ruby 
# app/services/attribution_reconciler.rb
class AttributionReconciler
  DRIFT_THRESHOLD = 0.05  # 5% mismatch tolerance

  def call(account, date_range)
    internal_counts = account.conversion_events
                             .where(occurred_at: date_range)
                             .group(:campaign_id)
                             .count

    internal_counts.each do |campaign_id, count|
      external = fetch_external_count(campaign_id, date_range)
      drift = (count - external).abs.to_f / [count, external, 1].max

      if drift > DRIFT_THRESHOLD
        Rails.logger.warn(
          "attribution.drift_detected",
          campaign_id: campaign_id,
          internal: count,
          external: external,
          drift_pct: (drift * 100).round(2)
        )
      end
    end
  end
end

3. Background Job Architecture

Attribution matching, reconciliation, report generation, email delivery — these tasks are too slow or too unreliable to run inside a web request. The solution is to enqueue them as background jobs that are processed asynchronously by dedicated worker processes.

Rails provides Active Job as a unified interface, with Sidekiq as the production backend. Sidekiq uses Redis as a job queue and processes jobs in threads, giving you high throughput without the memory overhead of forking.

Key Insight

Queue weights determine how often Sidekiq checks each queue, not strict priority ordering. With weights of 6, 4, 2, 1, the critical queue gets checked ~46% of the time (6/13), while mailers gets ~8% (1/13). This prevents low-priority queues from starving completely while ensuring critical jobs are processed first under load.

4. Retry Strategy and Dead Letters

Background jobs fail. Networks time out, databases go briefly unavailable, external APIs return 500s. A robust retry strategy uses exponential backoff with jitter to avoid thundering herd problems, where hundreds of failed jobs all retry at exactly the same moment.

After exhausting all retries, failed jobs move to a dead-letter queue where they can be inspected, debugged, and replayed manually. The key design constraint: every worker must be idempotent, because retries mean the same job may run multiple times.

ruby 
# app/jobs/attribution_match_job.rb
class AttributionMatchJob < ApplicationJob
  queue_as :critical

  # Sidekiq retry configuration
  sidekiq_options retry: 8,                    # Max 8 retries
                  dead: true,                  # Move to dead set after exhaustion
                  backtrace: 20,               # Store 20 backtrace lines
                  lock: :until_executed        # Prevent duplicate enqueues

  # Retry schedule: 15s, 30s, 1m, 2m, 4m, 8m, 16m, 32m
  sidekiq_retry_in do |count, _exception|
    delay = (15 * (2 ** count))           # Exponential backoff
    jitter = rand(delay * 0.15).to_i      # 15% jitter
    delay + jitter
  end

  def perform(conversion_event_id)
    event = ConversionEvent.find_by(id: conversion_event_id)
    return if event.nil?           # Record deleted — skip silently
    return if event.exact?          # Already matched — idempotent skip

    result = AttributionMatcher.new(event).call

    event.update!(
      campaign_id:      result.campaign_id,
      ad_group_id:      result.ad_group_id,
      match_confidence: result.confidence,
      metadata:         event.metadata.merge(
        "matched_at" => Time.current.iso8601,
        "match_strategy" => result.strategy
      )
    )

    Rails.logger.info(
      "attribution.matched",
      conversion_id: event.id,
      confidence: result.confidence,
      strategy: result.strategy
    )

  rescue ActiveRecord::RecordNotFound
    # Event was deleted between enqueue and execution — safe to ignore
    Rails.logger.info("attribution.event_missing", conversion_id: conversion_event_id)
  end
end
Pro Tip: Idempotent Worker Design

Notice the early returns: if the event is already exact?, we skip re-matching. If the record is missing, we log and return. These guards make the worker safe to run multiple times. A good rule of thumb: a worker should produce the same result whether it runs once or ten times. Never assume a job will execute exactly once.

What happens when a job exhausts all retries?

When Sidekiq has retried a job the maximum number of times and it still fails, the job is moved to the Dead Set (also called the dead-letter queue). Jobs in the Dead Set are retained for 6 months by default and can be viewed in the Sidekiq Web UI. From there, an operator can inspect the error, fix the root cause, and manually retry the job. You should also set up an alert that fires when any job enters the Dead Set — it means something is genuinely broken, not just temporarily unavailable.

Why add jitter to exponential backoff?

Without jitter, if 1,000 jobs fail at the same time (e.g., a database blip), they all retry at exactly 15 seconds, then 30 seconds, then 60 seconds. This creates retry storms that can overwhelm the downstream system just as it's recovering. Adding random jitter (typically 10–25% of the delay) spreads retries across time, smoothing the load and giving the system a chance to recover gracefully.

5. Observability

A background job system without observability is a black box. You need to know: How many jobs are enqueued? What is the processing latency? How many retries are happening? Are any queues backing up? This information drives SLOs and alerts.

ruby 
# config/initializers/sidekiq.rb
require "sidekiq"
require "sidekiq/middleware/server/retry_jobs"

Sidekiq.configure_server do |config|
  config.redis = { url: ENV.fetch("REDIS_URL", "redis://localhost:6379/1") }

  # Death handler — alert when a job hits the dead set
  config.death_handlers << ->(job, exception) do
    Rails.logger.error(
      "sidekiq.job_dead",
      job_class: job["class"],
      job_id: job["jid"],
      args: job["args"],
      error: exception.message,
      queue: job["queue"],
      retried_count: job["retry_count"]
    )

    # Notify alerting system (PagerDuty, Slack, etc.)
    AlertService.notify_dead_job(
      job_class: job["class"],
      error: exception.message,
      severity: :critical
    )
  end
end

Sidekiq.configure_client do |config|
  config.redis = { url: ENV.fetch("REDIS_URL", "redis://localhost:6379/1") }

ruby 
# app/services/queue_health_monitor.rb
class QueueHealthMonitor
  # SLOs: queue-specific latency targets
  SLOS = {
    "critical" => 30.seconds,   # Must process within 30s
    "default"  => 2.minutes,    # Must process within 2m
    "low"      => 15.minutes,   # Must process within 15m
    "mailers"  => 5.minutes     # Must process within 5m
  }.freeze

  def check_all
    stats = Sidekiq::Stats.new
    report = {
      processed: stats.processed,
      failed:    stats.failed,
      enqueued:  stats.enqueued,
      retry_size: stats.retry_size,
      dead_size:  stats.dead_size,
      queues:     {}
    }

    Sidekiq::Queue.all.each do |queue|
      latency = queue.latency  # Seconds since oldest job enqueued
      slo = SLOS.fetch(queue.name, 5.minutes)
      breaching = latency > slo

      report[:queues][queue.name] = {
        size: queue.size,
        latency_seconds: latency.round(2),
        slo_seconds: slo.to_i,
        slo_breaching: breaching
      }

      if breaching
        Rails.logger.warn(
          "queue.slo_breach",
          queue: queue.name,
          latency: latency.round(2),
          slo: slo.to_i,
          size: queue.size
        )
      end
    end

    report
  end
end
ruby 
# app/jobs/concerns/job_instrumentation.rb
module JobInstrumentation
  extend ActiveSupport::Concern

  included do
    around_perform do |job, block|
      start = Process.clock_gettime(Process::CLOCK_MONOTONIC)
      begin
        block.call
        duration = Process.clock_gettime(Process::CLOCK_MONOTONIC) - start
        Rails.logger.info(
          "job.completed",
          job_class: job.class.name,
          job_id: job.job_id,
          queue: job.queue_name,
          duration_ms: (duration * 1000).round(2),
          args: job.arguments
        )
      rescue => e
        duration = Process.clock_gettime(Process::CLOCK_MONOTONIC) - start
        Rails.logger.error(
          "job.failed",
          job_class: job.class.name,
          job_id: job.job_id,
          queue: job.queue_name,
          duration_ms: (duration * 1000).round(2),
          error: e.message,
          error_class: e.class.name
        )
        raise  # Re-raise so Sidekiq can retry
      end
    end
  end
end
Replay Tooling Is Not Optional

Build a rake task or admin endpoint that can replay failed jobs from the dead set. In production, you will need this. A common pattern is to filter dead jobs by class name, fix the root cause (deploy a fix, restore a dependency), and then bulk-retry all matching dead jobs. Sidekiq's DeadSet API supports this: Sidekiq::DeadSet.new.each { |job| job.retry if job.klass == "AttributionMatchJob" }.

Self-Check

Before moving to Chapter 07, honestly answer these questions. If any answer is "no", go back and fix it.

Does submitting the same conversion event twice produce exactly one database record?

Fire the same POST /api/v1/conversions request with the same Idempotency-Key header five times in rapid succession. Check your database — there should be exactly one conversion_events row. The first request returns 201 Created; subsequent requests return 200 OK with the same record. If you see duplicates, your transaction isolation or unique index is not working.

Can you explain the confidence level of every attributed conversion?

Query your conversion_events table and group by match_confidence. You should see events in each category: exact, fingerprint, probabilistic, and unmatched. For each category, you should be able to explain why it was classified that way. If all events are exact or all are unmatched, your fallback matching logic may not be working.

What happens when a background job fails 8 times in a row?

After exhausting all 8 retries with exponential backoff, the job should land in the Sidekiq Dead Set. Check the Sidekiq Web UI at /admin/sidekiq/morgue — your failed job should be there with the full error message and backtrace. Your death handler should have fired an alert. You should be able to fix the root cause and retry the job from the Dead Set or via your rake task.

Are your queue SLOs being monitored and alerted on?

Artificially slow down a worker (add a sleep 60) and watch the critical queue's latency climb past the 30-second SLO. Your QueueHealthMonitor should log a queue.slo_breach warning. If you have alerting configured, it should fire. Remove the sleep and verify the queue drains back to healthy latency. If none of this is observable, your monitoring is incomplete.
