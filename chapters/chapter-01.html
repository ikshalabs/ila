Chapter 01 · Phase 1: Platform Foundations

Platform Bootstrap

Create the project skeleton and shared developer workflow for a new ad bidding platform. Before you write a single feature, the team needs a project that boots cleanly, tests reliably, and welcomes new engineers without friction.

Estimated: 2–3 days
3 Advancement Levels
3 Assignments

Why This Matters

Every production system starts with a bootstrap moment — the first rails new, the first npx create-react-app, the first docker compose up. This moment seems trivial, but the decisions you make here cascade through every chapter that follows.

A badly bootstrapped project creates invisible drag: slow test suites, mysterious environment bugs, "works on my machine" conversations, and hours lost to configuration archaeology. A well-bootstrapped project does the opposite — it makes every subsequent feature faster to build, test, and ship.

Key Insight

The goal of bootstrap is not to build features. It's to build the machine that builds features. A team of 5 engineers who can each boot the project in 2 minutes will outperform a team of 10 who spend 45 minutes fighting environment setup.

Key Concepts

1. Service Boundaries for a Bidding MVP

An ad bidding platform has a minimum of three runtime concerns that should be separated early, even if they live in the same repository:

  • API server — Rails in API mode. Serves JSON to the frontend, accepts webhook callbacks, exposes health endpoints.
  • Background workers — Sidekiq (or Solid Queue). Processes async jobs like lead enrichment, conversion imports, billing events.
  • Frontend app — React SPA. The operator console where campaigns are managed, bids are tuned, and analytics are consumed.

All five services — React, Rails, Sidekiq, PostgreSQL, and Redis — should boot with a single command. If any one of them is down, the others should report a clear health status, not a cryptic stack trace.

2. Rails API Mode

When you run rails new with the --api flag, Rails strips out everything you don't need for a JSON API: views, helpers, assets pipeline, cookie-based sessions. What you get is a lean framework that's purpose-built for serving structured data.

bash 
# Create a new Rails API application
rails new bidding_api --api \
  --database=postgresql \
  --skip-action-mailer \
  --skip-action-mailbox \
  --skip-action-text \
  --skip-active-storage \
  --skip-action-cable

The --skip-* flags remove subsystems you won't need for a bidding API. You can always add them back later — it's easier to add than to untangle.

Why not skip Active Storage?

We skip it in the bootstrap because creative asset management (Chapter 06) will use a different storage pattern. If you find you need it later, rails active_storage:install takes 30 seconds. Don't cargo-cult gems you aren't using yet.

3. React Project Setup

For the operator console, we use Vite + React. Vite gives sub-second hot reloads during development and optimized production builds. The key decisions at bootstrap:

bash 
# Create React app with Vite and TypeScript
npm create vite@latest bidding_console -- --template react-ts

cd bidding_console
npm install

# Essential deps for what's ahead
npm install axios react-router-dom @tanstack/react-query
  • TypeScript — Catches type errors at compile time. In a bidding system where a mistyped budget field could cost real money, this is non-negotiable.
  • React Router — We'll need protected routes (Chapter 13) and deep-linking for campaign/ad-group detail pages.
  • TanStack Query — Manages server state, caching, and background refetching. Eliminates an entire category of bugs around stale data.

4. Docker Compose for Local Development

Docker Compose is the single most impactful bootstrap decision. It guarantees that PostgreSQL and Redis versions match across every developer machine, CI, and staging. No more "I'm on Postgres 14, you're on 16" bugs.

yaml 
# docker-compose.yml
version: '3.8'

services:
  db:
    image: postgres:16-alpine
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: bidding
      POSTGRES_PASSWORD: bidding_dev
      POSTGRES_DB: bidding_development
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U bidding"]
      interval: 5s
      timeout: 5s
      retries: 5

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  pgdata:
Pro Tip

Always include healthcheck directives. Without them, docker compose up will report services as "running" while they're still initializing. Your Rails app will crash trying to connect to a Postgres that isn't ready yet, and you'll waste 10 minutes debugging a timing issue.

5. Health Check Endpoints

Every service should answer the question "are you okay?" with a deterministic, machine-readable response. For the Rails API, this means a /health endpoint that checks database and Redis connectivity:

ruby 
# app/controllers/health_controller.rb
class HealthController < ApplicationController
  def show
    checks = {
      database: database_healthy?,
      redis:    redis_healthy?,
      time:     Time.current.iso8601
    }

    status = checks.values_at(:database, :redis).all? ? :ok : :service_unavailable
    render json: checks, status: status
  end

  private

  def database_healthy?
    ActiveRecord::Base.connection.execute("SELECT 1")
    true
  rescue StandardError
    false
  end

  def redis_healthy?
    Redis.new.ping == "PONG"
  rescue StandardError
    false
  end
end

This simple pattern — check each dependency, return a boolean per service, return HTTP 503 if any are down — will save you hours of debugging in every chapter that follows.

6. One-Command Boot

The "gold standard" for bootstrap is a single script that takes a developer from a fresh clone to a running system. Here's what bin/setup should do:

bash 
#!/usr/bin/env bash
# bin/setup — Boot the entire platform from zero
set -euo pipefail

echo "==> Starting infrastructure..."
docker compose up -d

echo "==> Waiting for Postgres..."
until docker compose exec db pg_isready -U bidding > /dev/null 2>&1; do
  sleep 1
done

echo "==> Setting up Rails..."
cd api
bundle install
bin/rails db:prepare
cd ..

echo "==> Setting up React..."
cd console
npm install
cd ..

echo "==> Done! Run bin/dev to start all services."
What should bin/dev look like?

Use Overmind or Foreman to run all processes from a single Procfile.dev:

procfile 
api:     cd api && bin/rails server -p 3000
worker:  cd api && bundle exec sidekiq
console: cd console && npm run dev -- --port 3001

Then bin/dev is simply: overmind start -f Procfile.dev

Environment variable management

Never commit .env files. Instead, commit a .env.example with placeholder values and document every variable:

env 
# .env.example — Copy to .env and fill in values
DATABASE_URL=postgresql://bidding:bidding_dev@localhost:5432/bidding_development
REDIS_URL=redis://localhost:6379/0
RAILS_ENV=development
SECRET_KEY_BASE=generate-with-rails-secret

Use the dotenv gem to load these automatically in development.

Assignments

Complete these in order. Each level builds on the previous one. Submit your work as a GitHub PR with docs/chapter-01.md containing your architecture notes and setup proof (screenshots or terminal output).

Assignment 1
Level 1 — Foundation

Local API + UI Boot with Health Checks

Get the full stack running locally. Both the Rails API and React app should be accessible, and health checks should return clean status.

  • Create a new Rails API app with --api --database=postgresql
  • Create a new React + TypeScript app with Vite
  • Write a docker-compose.yml for PostgreSQL and Redis with health checks
  • Add a /health endpoint to Rails that checks DB and Redis connectivity
  • Verify: curl localhost:3000/health returns {"database":true,"redis":true}
  • Verify: React app loads at localhost:3001
Done when

A mentor can clone your repo, run docker compose up -d and bin/rails server, and see a healthy API response and a React welcome page without editing any files.

Assignment 2
Level 2 — Professional

One-Command Startup/Shutdown and Environment Templates

Automate the entire boot process so no developer ever has to read a README to get started.

  • Create bin/setup that bootstraps everything from a fresh clone
  • Create bin/dev using Overmind/Foreman and a Procfile.dev
  • Add .env.example with every required variable documented
  • bin/setup should auto-copy .env.example to .env if missing
  • Add a bin/teardown script to cleanly stop and remove containers
Done when

A new engineer can clone the repo and run one command (bin/setup && bin/dev) to have the full system running. No manual steps. No "see the wiki."

Assignment 3
Level 3 — Production-Ready

Team-Ready Onboarding Guide with Troubleshooting Matrix

Write documentation that anticipates failure. The best setup guide is one that helps engineers help themselves when things go wrong.

  • Write docs/chapter-01.md with architecture diagram and service overview
  • Include a troubleshooting table: symptom → likely cause → fix command
  • Document coding conventions: linters, test commands, branch naming
  • Time a full setup from scratch — target under 20 minutes
  • Have another person follow your guide without your help and note where they get stuck
Done when

A colleague who has never seen the repo can complete setup in under 20 minutes by following your documentation alone. If they get stuck, the troubleshooting matrix resolves it without Slack.

Self-Check

Before moving to Chapter 02, honestly answer these questions. If any answer is "no", go back and fix it.

Can a new engineer run the project in under 20 minutes?

Time it yourself. If it takes longer, identify where the friction is: slow dependency installs? Missing documentation? A database migration that requires manual steps? Each pain point is a setup bug that needs fixing.

Are service health checks deterministic and easy to debug?

Try stopping Redis (docker compose stop redis) and hitting /health. Does it clearly say "redis": false? Or does it return a 500 with a stack trace? Health checks should diagnose, not crash.

Is the scope constrained enough for feature-by-feature delivery?

You should have zero business logic in the codebase right now. No models beyond what Rails generates. No routes beyond /health. If you've built anything else, you've over-scoped the bootstrap — move it to the appropriate chapter.
