Chapter 05 · Phase 3: Revenue Engine

Bid Strategy Engine and Lead Intake

Add bid suggestion and apply flow driven by target CPA and delivery signals. Ingest inbound leads with schema validation, deduplication, and abuse protection. Bid quality determines acquisition cost; lead quality drives revenue quality and partner trust.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

In an ad bidding platform, the bid is the single most consequential number you produce. Bid too high and you destroy margin. Bid too low and you lose volume to competitors. Every dollar of acquisition cost flows directly from the bid you submit, which means the bid strategy engine is where business logic meets financial reality.

On the other side of the equation sits lead intake — the pipeline that accepts inbound leads from partners, validates their structure, deduplicates them, and scores them for quality. If your bid engine is the cost side, lead intake is the revenue side. Bad leads burn money: they waste sales time, inflate conversion metrics, and erode partner trust. A quality gate at ingestion prevents garbage from ever entering the system.

The Cost of Getting This Wrong

An unclamped bid can spend an entire daily budget in minutes. A missing dedup window can charge you twice for the same lead. These are not edge cases — they are the most common failure modes in production bidding systems. Safety rails and validation are not optional; they are the foundation of financial stability.

Key Concepts

1. Bid Strategy Fundamentals

A bid strategy is the set of rules that translate business goals into a dollar amount attached to each ad impression or click. The most common anchor is target CPA (cost per acquisition): the maximum you're willing to pay for a single conversion.

  • CPA and conversion-rate tradeoffs — A higher bid wins more auctions but costs more per lead. A lower bid reduces cost but may starve the campaign of volume. The engine must balance these forces.
  • Static vs. dynamic bid suggestions — Static bids use a fixed formula (e.g., target CPA × historical conversion rate). Dynamic bids incorporate real-time signals: recent conversion rate, time of day, device type, and remaining daily budget.
  • Safety rails: max/min clamps — Every bid must pass through clamp logic that enforces an absolute floor and ceiling. Without clamps, a bug in dynamic logic could produce a $0.00 or $10,000 bid.
  • Explanation metadata — Every bid recommendation should carry a reason hash explaining why this number was chosen. This is critical for debugging, auditing, and operator trust.

Key Insight

Notice that preview and apply share the same computation pipeline. The only difference is what happens at step 5: preview returns the result without side effects, while apply persists the bid and writes an audit record. This design makes the recommendation logic fully testable without touching the database.

2. Recommendation Service

The recommendation service is a plain Ruby service object with no side effects. It takes campaign data as input and returns a bid suggestion with explanation metadata. Keeping the logic deterministic and stateless means you can test every edge case with unit tests — no database, no mocks, no flaky specs.

ruby 
# app/services/bid_recommendation_service.rb
class BidRecommendationService
  BidResult = Struct.new(:suggested_bid, :reason, :inputs, :clamped, keyword_init: true)

  MIN_BID_FLOOR   = 0.01
  MAX_BID_CEILING = 500.00

  def initialize(campaign:, conversion_rate:, modifier: 1.0)
    @campaign        = campaign
    @conversion_rate = conversion_rate
    @modifier        = modifier
  end

  def call
    raw_bid = compute_raw_bid
    clamped_bid = clamp(raw_bid)

    BidResult.new(
      suggested_bid: clamped_bid.round(2),
      reason:        determine_reason(raw_bid, clamped_bid),
      inputs:        build_inputs(raw_bid),
      clamped:       raw_bid != clamped_bid
    )
  end

  private

  def compute_raw_bid
    return 0.0 if @conversion_rate.zero?

    @campaign.target_cpa * @conversion_rate * @modifier
  end

  def clamp(bid)
    floor   = [@campaign.bid_floor   || MIN_BID_FLOOR,   MIN_BID_FLOOR].max
    ceiling = [@campaign.bid_ceiling || MAX_BID_CEILING, MAX_BID_CEILING].min

    bid.clamp(floor, ceiling)
  end

  def determine_reason(raw, clamped)
    if raw < MIN_BID_FLOOR
      "floor_clamped"
    elsif raw > (@campaign.bid_ceiling || MAX_BID_CEILING)
      "ceiling_clamped"
    else
      "cpa_target"
    end
  end

  def build_inputs(raw_bid)
    {
      target_cpa: @campaign.target_cpa,
      conversion_rate: @conversion_rate,
      modifier: @modifier,
      raw_bid: raw_bid.round(4),
      bid_floor: @campaign.bid_floor,
      bid_ceiling: @campaign.bid_ceiling
    }
  end
end

The preview endpoint calls this service and returns the result as JSON. The apply endpoint calls the same service, persists the bid on the ad group, and writes a BidChange audit record. Both endpoints validate that the operator has permission (via Pundit policies from Chapter 02) before executing.

ruby 
# app/controllers/api/v1/bid_recommendations_controller.rb
module Api
  module V1
    class BidRecommendationsController < ApplicationController
      before_action :set_ad_group

      # GET /api/v1/ad_groups/:ad_group_id/bid_recommendation
      def show
        result = BidRecommendationService.new(
          campaign:        @ad_group.campaign,
          conversion_rate: @ad_group.recent_conversion_rate,
          modifier:        params.fetch(:modifier, 1.0).to_f
        ).call

        render json: {
          ad_group_id:   @ad_group.id,
          suggested_bid: result.suggested_bid,
          reason:        result.reason,
          inputs:        result.inputs,
          clamped:       result.clamped
        }
      end

      # POST /api/v1/ad_groups/:ad_group_id/bid_recommendation/apply
      def apply
        result = BidRecommendationService.new(
          campaign:        @ad_group.campaign,
          conversion_rate: @ad_group.recent_conversion_rate
        ).call

        unless result.suggested_bid.between?(
          @ad_group.campaign.bid_floor, @ad_group.campaign.bid_ceiling
        )
          render json: { error: "Bid out of bounds" }, status: :unprocessable_entity
          return
        end

        ActiveRecord::Base.transaction do
          @ad_group.update!(current_bid: result.suggested_bid)
          BidChange.create!(
            ad_group:      @ad_group,
            previous_bid:  @ad_group.current_bid_before_last_save,
            new_bid:       result.suggested_bid,
            reason:        result.reason,
            metadata:      result.inputs,
            operator:      current_user
          )
        end

        render json: { applied: true, new_bid: result.suggested_bid }
      end

      private

      def set_ad_group
        @ad_group = current_account.ad_groups.find(params[:ad_group_id])
      end
    end
  end
end
Bounds Check on Apply, Not Just Preview

The apply endpoint re-validates that the computed bid falls within bounds before persisting. Never trust that the preview result is still valid by the time the operator clicks "apply" — campaign settings may have changed between the two requests.

3. Bid History and Audit

Every bid change must be auditable. Regulators, account managers, and debugging engineers all need to answer the same question: "Who changed this bid, when, and why?" The BidChange model captures this history as an append-only audit log.

ruby 
# app/models/bid_change.rb
class BidChange < ApplicationRecord
  belongs_to :ad_group
  belongs_to :operator, class_name: "User"

  validates :new_bid, presence: true, numericality: { greater_than: 0 }
  validates :reason,  presence: true

  scope :recent, -> { order(created_at: :desc).limit(50) }
  scope :for_ad_group, ->(ag_id) { where(ad_group_id: ag_id) }

  # Backtest helper: replay bid decisions against historical data
  def self.backtest_window(ad_group_id, from:, to:)
    where(ad_group_id: ad_group_id)
      .where(created_at: from..to)
      .order(:created_at)
  end
end
Pro Tip: Backtesting Framework

The backtest_window scope lets you replay historical bid decisions. Feed past conversion data through the BidRecommendationService and compare what it would have suggested vs. what was actually applied. This is how you validate a new bid strategy before deploying it to production campaigns.

Why append-only instead of updating a current_bid column directly?

If you only store the current bid, you lose all history. When an account manager asks "why did our CPA spike on Tuesday?", you need to see every bid change, who made it, and what reasoning was used. The append-only BidChange log gives you complete forensic capability. The current_bid on the ad group is a denormalized cache for fast reads; the audit log is the source of truth.

4. Lead Intake Pipeline

Leads arrive from external partners via API. Each lead must pass through three gates before entering the system: schema validation (does the payload have the required fields?), deduplication (have we seen this lead before?), and quality scoring (is this lead worth pursuing?). Reject early and reject loudly — partners need clear error contracts to fix integration issues.

The minimum valid lead payload includes: email, first_name, last_name, phone, and source_campaign_id. Everything else is optional metadata.

ruby 
# app/controllers/api/v1/leads_controller.rb
module Api
  module V1
    class LeadsController < ApplicationController
      skip_before_action :authenticate_request!, only: :create
      before_action :authenticate_partner!, only: :create
      before_action :enforce_rate_limit!,   only: :create

      # POST /api/v1/leads
      def create
        result = LeadIngestionService.new(
          params:  lead_params,
          partner: @current_partner
        ).call

        if result.success?
          render json: {
            lead_id:  result.lead.id,
            status:   result.lead.status,
            quality:  result.quality_score
          }, status: :created
        else
          render json: {
            error:    result.error_code,
            message:  result.error_message,
            details:  result.validation_errors
          }, status: result.http_status
        end
      end

      private

      def lead_params
        params.require(:lead).permit(
          :email, :first_name, :last_name, :phone,
          :source_campaign_id, :source_ad_group_id,
          :ip_address, :user_agent,
          metadata: {}
        )
      end

      def authenticate_partner!
        api_key = request.headers["X-Partner-Key"]
        @current_partner = Partner.find_by(api_key: api_key, active: true)

        unless @current_partner
          render json: { error: "invalid_api_key",
                         message: "Missing or invalid X-Partner-Key header" },
                 status: :unauthorized
        end
      end

      def enforce_rate_limit!
        key = "lead_intake:#{@current_partner.id}:#{Time.current.beginning_of_minute.to_i}"
        count = Rails.cache.increment(key, 1, expires_in: 60.seconds)

        if count > @current_partner.rate_limit_per_minute
          render json: { error: "rate_limit_exceeded",
                         message: "Rate limit: #{@current_partner.rate_limit_per_minute}/min" },
                 status: :too_many_requests
        end
      end
    end
  end
end
Key Insight

The lead intake endpoint uses partner authentication (API key), not user authentication (JWT). Partners are external systems pushing leads into your platform. They get their own identity, their own rate limits, and their own error contracts. This is a fundamentally different auth model from the operator-facing endpoints in previous chapters.

How does the dedup strategy work with configurable lookback windows?

Deduplication checks whether a lead with the same email (or phone) has been ingested within a configurable window. The default is 24 hours, but partners can set different windows depending on their use case. The check happens inside LeadIngestionService before the lead is persisted:

ruby 
# Inside LeadIngestionService
def duplicate?
  lookback = @partner.dedup_window_hours&.hours || 24.hours
  Lead.where(email: normalized_email)
      .where(partner_id: @partner.id)
      .where("created_at > ?", lookback.ago)
      .exists?
end

5. Quality Gate

Not all leads are equal. A quality gate scores each lead at ingestion and routes it accordingly. High-quality leads go to immediate sales follow-up. Medium-quality leads enter a nurture queue. Low-quality or suspicious leads are flagged for review or rejected outright.

The quality scoring service is a pluggable pipeline. Each scoring hook receives the lead data and returns a score component. The final quality score is the weighted sum of all components.

ruby 
# app/services/lead_quality_scorer.rb
class LeadQualityScorer
  ScoreResult = Struct.new(:total_score, :components, :tier, keyword_init: true)

  HOOKS = [
    Scoring::EmailValidator,
    Scoring::PhoneValidator,
    Scoring::CompletionChecker,
    Scoring::AbuseDetector,
    Scoring::GeoRelevance
  ].freeze

  TIER_THRESHOLDS = { high: 80, medium: 50, low: 0 }.freeze

  def initialize(lead_data:, partner:)
    @lead_data = lead_data
    @partner   = partner
  end

  def call
    components = HOOKS.map do |hook|
      result = hook.new(@lead_data, @partner).score
      { name: hook.name.demodulize, score: result.score, weight: result.weight }
    end

    total = components.sum { |c| c[:score] * c[:weight] }
    tier  = determine_tier(total)

    ScoreResult.new(total_score: total.round(1), components: components, tier: tier)
  end

  private

  def determine_tier(score)
    TIER_THRESHOLDS.each do |tier, threshold|
      return tier if score >= threshold
    end
    :rejected
  end
end
ruby 
# app/services/scoring/abuse_detector.rb
module Scoring
  class AbuseDetector
    Result = Struct.new(:score, :weight, keyword_init: true)

    DISPOSABLE_DOMAINS = %w[mailinator.com guerrillamail.com tempmail.com].freeze
    SUSPICIOUS_PATTERNS = [
      /(.)\1{4,}/,         # repeated chars: "aaaaaa"
      /test@/i,            # test emails
      /\d{8,}@/           # long numeric prefix
    ].freeze

    def initialize(lead_data, partner)
      @email   = lead_data[:email].to_s.downcase
      @phone   = lead_data[:phone].to_s
      @partner = partner
    end

    def score
      penalty = 0
      penalty += 40 if disposable_domain?
      penalty += 30 if suspicious_pattern?
      penalty += 20 if velocity_spike?

      Result.new(score: [100 - penalty, 0].max, weight: 0.3)
    end

    private

    def disposable_domain?
      domain = @email.split("@").last
      DISPOSABLE_DOMAINS.include?(domain)
    end

    def suspicious_pattern?
      SUSPICIOUS_PATTERNS.any? { |pat| @email.match?(pat) }
    end

    def velocity_spike?
      key = "abuse:email_velocity:#{Digest::SHA256.hexdigest(@email)}"
      count = Rails.cache.increment(key, 1, expires_in: 1.hour)
      count > 3
    end
  end
end
Error Contracts for Partners

Partners need machine-readable errors, not human-readable messages. Always return a stable error_code string (e.g., "duplicate_lead", "invalid_email", "rate_limit_exceeded") alongside a human-friendly message. Partners build retry logic against the code, not the message. Changing a message is cosmetic; changing a code is a breaking API change.

Self-Check

Before moving to Chapter 06, honestly answer these questions. If any answer is "no", go back and fix it.

Can the bid engine produce a bid outside the configured clamp range?

Set bid_floor to 1.00 and bid_ceiling to 5.00 on a campaign. Then configure inputs that would produce a raw bid of $0.02 and $500.00. The service should clamp both to the floor and ceiling respectively, and the clamped flag should be true in both cases. If you can produce a bid outside the range, your clamp logic has a bug.

Does every bid change have an audit trail with operator and reason?

Apply a bid via the API, then query BidChange.last. It should have: the operator who applied it, the previous bid, the new bid, the reason code, and the full inputs hash. If any of these fields are nil, your audit logging is incomplete. Run BidChange.where(operator: nil).count — it should be zero.

Can a partner submit the same lead twice within the dedup window?

Submit a lead with a specific email. Wait a moment and submit the exact same email again. You should receive a 409 Conflict with error_code: "duplicate_lead". Now wait until the dedup window expires (or shorten it for testing) and submit again — it should succeed. If duplicates slip through, check your dedup query and index.

Are error responses machine-readable with stable error codes?

Hit the lead endpoint with invalid data and inspect the response. You should see a JSON object with error (a stable string code), message (human-friendly), and optionally details (per-field errors). A partner should be able to build a switch statement on your error codes without worrying about message text changes.
