Chapter 04 · Phase 2: Core Business Logic

Ad Group Targeting and Creative Assets

Introduce ad groups with composable targeting dimensions and build a creative asset pipeline so media can be validated, uploaded, attached to ad groups, and safely cloned for A/B experimentation.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

A campaign without ad groups is a blunt instrument. It says "spend money" but not where, on whom, or with what creative. Ad groups are the operational unit where targeting strategy meets creative execution. They let advertisers slice a campaign into segments — by territory, audience filter, or traffic source — and attach specific creative assets to each segment.

Granular targeting enables better ROI and controlled experimentation. When you can split traffic by geography or source and attach different headlines, images, or landing pages to each split, you move from guessing to measuring. Faster creative iteration directly affects CTR and campaign performance. The difference between a 1.2% CTR and a 2.4% CTR can be a single headline change — but only if your platform makes that change safe, auditable, and instant.

The Cost of Getting This Wrong

A misconfigured targeting rule can drain an entire daily budget on irrelevant traffic in minutes. A creative asset with a broken URL or wrong dimensions will render as a blank rectangle — wasting every impression it receives. Both failures are silent: you won't know until the money is gone. Validation at the model layer is not optional.

Key Concepts

1. Ad Group Architecture

An ad group is a child of a campaign. It inherits defaults from the parent campaign (budget allocation, status, date range) but can override them. This parent-child relationship is fundamental: it means you can pause a campaign and all its ad groups stop, or you can pause a single ad group while the rest of the campaign continues running.

Each ad group carries its own targeting dimensions — territory restrictions, audience filters, source constraints — and its own set of creative attachments. The key design decision is override vs. inherit: when an ad group doesn't specify a value, it falls back to the campaign's default. When it does specify a value, the ad group's value wins.

ruby 
# app/models/ad_group.rb
class AdGroup < ApplicationRecord
  acts_as_tenant :account

  belongs_to :campaign
  has_many   :ad_group_creatives, dependent: :destroy
  has_many   :creatives, through: :ad_group_creatives
  has_many   :targeting_rules, as: :targetable, dependent: :destroy

  enum :status, { draft: 0, active: 1, paused: 2, archived: 3 }

  validates :name, presence: true, length: { maximum: 120 }
  validates :status, presence: true
  validate  :campaign_belongs_to_same_account

  # Override-or-inherit pattern
  def effective_daily_budget
    daily_budget_override || campaign.daily_budget
  end

  def effective_bid_cap
    bid_cap_override || campaign.default_bid_cap
  end

  private

  def campaign_belongs_to_same_account
    return unless campaign
    if campaign.account_id != account_id
      errors.add(:campaign, "must belong to the same account")
    end
  end
end

2. Targeting Rule Engine

Targeting rules define who sees the ads in a given ad group. A rule is a composable unit: it has a dimension (territory, source, device, etc.), an operator (include, exclude, matches), and a value. Rules are combined with AND/OR logic to create complex targeting expressions.

The key design principle is composability. Rather than having a single monolithic targeting configuration with dozens of fields, each rule is a standalone record. You can add, remove, or modify individual rules without touching the others. This makes A/B testing trivial: clone an ad group, change one rule, and compare performance.

ruby 
# app/models/targeting_rule.rb
class TargetingRule < ApplicationRecord
  acts_as_tenant :account

  belongs_to :targetable, polymorphic: true

  DIMENSIONS = %w[territory source device audience_segment].freeze
  OPERATORS  = %w[include exclude matches].freeze
  COMBINATORS = %w[and or].freeze

  validates :dimension, presence: true, inclusion: { in: DIMENSIONS }
  validates :operator,  presence: true, inclusion: { in: OPERATORS }
  validates :value,     presence: true
  validates :combinator, inclusion: { in: COMBINATORS }, allow_nil: true
  validate  :value_matches_dimension

  scope :for_dimension, ->(dim) { where(dimension: dim) }

  private

  def value_matches_dimension
    case dimension
    when "territory"
      unless value.match?(/\A[A-Z]{2}\z/)
        errors.add(:value, "must be a 2-letter ISO country code")
      end
    when "source"
      allowed = %w[google facebook organic direct email]
      unless allowed.include?(value)
        errors.add(:value, "must be one of: #{allowed.join(', ')}")
      end
    when "device"
      allowed = %w[mobile desktop tablet]
      unless allowed.include?(value)
        errors.add(:value, "must be one of: #{allowed.join(', ')}")
      end
    end
  end
end

With this model, building a compound targeting expression looks like assembling Lego blocks. Here is an example of combining rules with AND/OR logic:

ruby 
# Compose rules: (territory = US OR territory = CA) AND (source = google)
ad_group.targeting_rules.create!([
  { dimension: "territory", operator: "include", value: "US", combinator: nil,   position: 0 },
  { dimension: "territory", operator: "include", value: "CA", combinator: "or",  position: 1 },
  { dimension: "source",    operator: "include", value: "google", combinator: "and", position: 2 }
])

# Evaluate: does a request match this ad group's rules?
class TargetingEvaluator
  def initialize(ad_group)
    @rules = ad_group.targeting_rules.order(:position)
  end

  def match?(request_context)
    return true if @rules.empty?

    result = evaluate_rule(@rules.first, request_context)
    @rules.drop(1).each do |rule|
      case rule.combinator
      when "and" then result = result && evaluate_rule(rule, request_context)
      when "or"  then result = result || evaluate_rule(rule, request_context)
      end
    end
    result
  end

  private

  def evaluate_rule(rule, ctx)
    actual = ctx[rule.dimension.to_sym]
    case rule.operator
    when "include" then actual == rule.value
    when "exclude" then actual != rule.value
    when "matches" then actual&.match?(Regexp.new(rule.value)) || false
    else false
    end
  end
end
Validate Targeting Combinations

Never allow contradictory rules. An ad group that includes territory: US AND excludes territory: US will match zero traffic and silently waste budget. Add a custom validation that detects impossible combinations before saving.

Why use polymorphic associations for targeting rules?

Targeting rules use belongs_to :targetable, polymorphic: true because the same targeting system may apply to ad groups today and to campaigns or audiences tomorrow. A polymorphic association means the targeting_rules table has targetable_type and targetable_id columns, so a rule can belong to any model that declares has_many :targeting_rules, as: :targetable. This avoids creating separate tables for campaign targeting, ad group targeting, and audience targeting.

3. Creative Asset Pipeline

A creative asset is the actual content an end user sees: a headline, a description, a display URL, and optionally an image or video file. The creative model must enforce strict validation — a headline that is too long will be truncated by the ad network, a URL with a typo will 404, and a wrong-sized image will be rejected at serve time.

Creatives are account-owned, not ad-group-owned. This is a deliberate design choice: a single creative can be attached to multiple ad groups across different campaigns. This reusability means you don't duplicate assets when the same banner runs in three territories. But it also means you must be careful about who can edit a creative that is shared across many live ad groups.

ruby 
# app/models/creative.rb
class Creative < ApplicationRecord
  acts_as_tenant :account

  has_many :ad_group_creatives, dependent: :restrict_with_error
  has_many :ad_groups, through: :ad_group_creatives

  has_one_attached :media_file   # Active Storage

  enum :creative_type, { text: 0, image: 1, video: 2, html5: 3 }
  enum :status, { draft: 0, approved: 1, rejected: 2, archived: 3 }

  validates :name,     presence: true, length: { maximum: 80 }
  validates :headline, presence: true, length: { maximum: 90 }
  validates :body,     length: { maximum: 300 }
  validates :display_url, format: {
    with: /\Ahttps?:\/\/[^\s]+\z/,
    message: "must be a valid HTTP or HTTPS URL"
  }, allow_blank: true
  validates :destination_url, presence: true, format: {
    with: /\Ahttps:\/\/[^\s]+\z/,
    message: "must be a valid HTTPS URL"
  }

  validate :media_file_validation, if: -> { media_file.attached? }

  def attached_ad_group_count
    ad_group_creatives.count
  end

  def in_use?
    ad_group_creatives.joins(:ad_group).where(ad_groups: { status: :active }).exists?
  end

  private

  def media_file_validation
    unless media_file.content_type.in?(%w[image/png image/jpeg image/gif video/mp4])
      errors.add(:media_file, "must be PNG, JPEG, GIF, or MP4")
    end

    if media_file.blob.byte_size > 10.megabytes
      errors.add(:media_file, "must be less than 10 MB")
    end
  end
end

4. Creative-to-Ad-Group Attachment

The relationship between creatives and ad groups is many-to-many. A creative can run in multiple ad groups, and an ad group can display multiple creatives (rotating or weighted). The join table — AdGroupCreative — carries metadata about the attachment: its position in the rotation, its weight, and its status.

ruby 
# app/models/ad_group_creative.rb
class AdGroupCreative < ApplicationRecord
  acts_as_tenant :account

  belongs_to :ad_group
  belongs_to :creative

  validates :creative_id, uniqueness: { scope: :ad_group_id,
    message: "is already attached to this ad group" }
  validates :weight, numericality: {
    greater_than_or_equal_to: 0,
    less_than_or_equal_to: 100
  }, allow_nil: true
  validate :same_account_check

  scope :active, -> { joins(:creative).where(creatives: { status: :approved }) }

  private

  def same_account_check
    return unless ad_group && creative
    if ad_group.account_id != creative.account_id
      errors.add(:creative, "must belong to the same account as the ad group")
    end
  end
end

The controller for attaching creatives enforces permission-scoped access. Only operators and admins can attach or detach creatives. Viewers can list them but cannot modify the relationship. All changes are wrapped in a transaction for rollback safety.

ruby 
# app/controllers/api/v1/ad_group_creatives_controller.rb
module Api::V1
  class AdGroupCreativesController < ApplicationController
    before_action :set_ad_group

    def create
      authorize @ad_group, :update?

      creative = current_account.creatives.find(params[:creative_id])
      attachment = @ad_group.ad_group_creatives.build(
        creative: creative,
        weight: params[:weight],
        account: current_account
      )

      if attachment.save
        render json: attachment, status: :created
      else
        render json: { errors: attachment.errors.full_messages },
               status: :unprocessable_entity
      end
    end

    def destroy
      authorize @ad_group, :update?

      attachment = @ad_group.ad_group_creatives.find(params[:id])
      attachment.destroy!
      head :no_content
    end

    private

    def set_ad_group
      campaign = current_account.campaigns.find(params[:campaign_id])
      @ad_group = campaign.ad_groups.find(params[:ad_group_id])
    end
  end
end
Key Insight: Change History

Every attachment and detachment should be auditable. Consider adding a paper_trail or a custom change_logs table that records who attached which creative to which ad group, and when. This is critical for debugging performance drops — "CTR fell 40% after Tuesday" becomes actionable when you can see exactly which creative was swapped on Tuesday.

Why restrict_with_error instead of destroy on the Creative association?

The Creative model uses has_many :ad_group_creatives, dependent: :restrict_with_error instead of dependent: :destroy. This is intentional: if a creative is attached to active ad groups, deleting it would break those ad groups silently. restrict_with_error prevents deletion and returns a clear error message: "Cannot delete record because dependent ad group creatives exist." The operator must explicitly detach the creative from all ad groups before it can be deleted. This forces a conscious decision rather than a cascade.

5. Duplication and Experimentation

The most common workflow in ad operations is clone-and-edit. An operator sees a high-performing ad group, clones it, changes one targeting rule or one creative, and runs both side by side. This is the foundation of A/B testing at the ad group level.

The clone must be deep and safe. It must duplicate the ad group record, its targeting rules, and its creative attachments — but it must not copy the primary keys. Stale IDs from the source record leaking into the clone is the most common bug in clone implementations.

ruby 
# app/services/ad_group_clone_service.rb
class AdGroupCloneService
  def initialize(source_ad_group, account:)
    @source  = source_ad_group
    @account = account
  end

  def call(name_suffix: " (Copy)")
    ActiveRecord::Base.transaction do
      clone = @source.dup
      clone.assign_attributes(
        name: "#{@source.name}#{name_suffix}",
        status: :draft,          # Always start as draft
        account: @account
      )
      clone.save!

      # Deep-clone targeting rules (new IDs, no stale references)
      @source.targeting_rules.each do |rule|
        cloned_rule = rule.dup
        cloned_rule.targetable = clone
        cloned_rule.account = @account
        cloned_rule.save!
      end

      # Re-attach the same creatives (shared, not duplicated)
      @source.ad_group_creatives.each do |agc|
        clone.ad_group_creatives.create!(
          creative: agc.creative,
          weight: agc.weight,
          account: @account
        )
      end

      clone
    end
  end
end
ruby 
# Bulk creative swap across multiple ad groups
class BulkCreativeSwapService
  def initialize(account:, old_creative:, new_creative:)
    @account      = account
    @old_creative = old_creative
    @new_creative = new_creative
  end

  def call
    ActiveRecord::Base.transaction do
      attachments = AdGroupCreative
        .where(account: @account, creative: @old_creative)

      swapped_count = 0
      attachments.find_each do |attachment|
        # Skip if new creative is already attached
        next if AdGroupCreative.exists?(
          ad_group: attachment.ad_group,
          creative: @new_creative
        )

        attachment.update!(creative: @new_creative)
        swapped_count += 1
      end

      { swapped: swapped_count, total: attachments.size }
    end
  end
end
Clones Must Always Start as Draft

Notice the clone sets status: :draft regardless of the source's status. A clone of an active ad group must not go live automatically — that would double your spend instantly. The operator must explicitly review and activate the clone.

What about cloning creatives themselves?

The clone service above shares creatives rather than duplicating them. This is correct for most A/B tests (same creative, different targeting). But when the experiment is about the creative itself (different headline), you need to clone the creative too. Add a CreativeCloneService that duplicates the creative record and its Active Storage attachment. Use creative.media_file.blob to re-attach the same blob to the new record without downloading and re-uploading the file.

Self-Check

Before moving to Chapter 05, honestly answer these questions. If any answer is "no", go back and fix it.

Can an ad group access creatives from a different account?

Create two accounts. Create a creative in Account A and an ad group in Account B. Try to attach Account A's creative to Account B's ad group. The same_account_check validation should reject it with a clear error message. If it silently succeeds, you have a cross-tenant data leak.

Do cloned ad groups have entirely new IDs?

Clone an ad group and compare the source and clone. The clone's id must be different. Every targeting rule on the clone must have a new id. Every ad_group_creative record on the clone must have a new id. If any ID matches the source, your clone implementation is leaking stale references.

Does deleting a creative that is attached to active ad groups fail safely?

Attach a creative to two active ad groups. Try to delete the creative. The restrict_with_error dependency should prevent deletion and return an error explaining that dependent records exist. The creative should still be in the database, and both ad groups should still have it attached.

Are targeting rules properly validated per dimension?

Try creating a targeting rule with dimension: "territory" and value: "not-a-country-code". The value_matches_dimension validation should reject it. Try dimension: "source" with value: "invalid_source". Same result. Every dimension has its own validation rules, and all of them must be enforced.
