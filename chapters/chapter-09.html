Operations Console and Performance Dashboard

Build ad group operations UI with bid recommendation, edit, and history visibility. Ship dashboard cards, trends, and tables for spend, leads, conversions, and efficiency.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

The operations console is the day-to-day control center for performance operators. While the campaign builder (Chapter 08) handles setup, this console is where operators spend 90% of their time — adjusting bids, reviewing performance trends, and reacting to spend anomalies.

Actionable visibility closes the optimization loop every day. If an operator cannot see that a bid change from yesterday caused a 40% drop in conversions, they are flying blind. If they cannot quickly compare today's CPL against last week's, they cannot justify budget reallocations to clients. The dashboard is not a nice-to-have — it is the product.

The Cost of Stale Data

A dashboard that shows data 30 minutes old is worse than no dashboard at all — it creates false confidence. Operators make bid changes based on numbers they trust. If those numbers are stale, every optimization decision is wrong. This chapter teaches you how to build real-time patterns that keep data fresh and clearly indicate when it is not.

Key Concepts

1. Ad Group Operations UI

The ad group operations panel is a dense, data-rich interface. Operators need to see ad groups in the context of their parent campaign, preview bid suggestions from the engine (Chapter 06), and apply changes with confidence. The key requirement is context without clutter — every piece of information must earn its screen real estate.

  • Ad group list with campaign context — Each row shows the ad group name, current bid, suggested bid, daily spend, and conversion count. The parent campaign name appears as a grouped header.
  • Bid suggestion preview and safe apply flow — Before applying a bid change, operators see a diff: current vs. suggested, with projected impact. A confirmation step prevents accidental bulk changes.
  • Change history with compare mode — Every bid change is logged with timestamp, user, old value, and new value. Compare mode lets operators overlay two time periods to spot regression.

The [Apply] button should never immediately mutate data. Always show a confirmation modal with the diff: $2.40 → $2.85 (+18.75%). Bulk apply should require an additional "I've reviewed all changes" checkbox. This is a safe apply flow — operators trust the console because it never surprises them.

2. Real-Time Data Patterns

Performance data changes constantly. Spend accumulates throughout the day, leads arrive in bursts, and bid changes take effect within minutes. Your UI needs a strategy for keeping data fresh without overwhelming the server or the user.

There are two primary patterns: polling (client asks the server on a timer) and WebSockets (server pushes updates to the client). Each has tradeoffs:

  • Polling — Simple to implement, works with any infrastructure. But it wastes bandwidth when nothing has changed and introduces latency equal to the polling interval.
  • WebSockets — Instant updates, no wasted requests. But requires persistent connections, load balancer support, and reconnection logic.
  • Hybrid approach — Poll for dashboard-level aggregates (every 30s), use WebSockets for critical alerts (budget exceeded, bid change confirmation). This is the recommended pattern for our platform.
tsx 
// hooks/usePollingQuery.ts
// A custom hook that wraps TanStack Query with automatic polling
import { useQuery } from "@tanstack/react-query";

interface PollingOptions {
  queryKey: string[];
  fetchFn: () => Promise<any>;
  intervalMs?: number;
  enabled?: boolean;
}

export function usePollingQuery({
  queryKey,
  fetchFn,
  intervalMs = 30_000,
  enabled = true,
}: PollingOptions) {
  return useQuery({
    queryKey,
    queryFn: fetchFn,
    refetchInterval: enabled ? intervalMs : false,
    refetchIntervalInBackground: false, // pause when tab is hidden
    staleTime: intervalMs / 2,         // data is "fresh" for half the interval
  });
}
tsx 
// hooks/useStaleIndicator.ts
// Shows a visual indicator when data is older than threshold
import { useState, useEffect } from "react";

export function useStaleIndicator(
  dataUpdatedAt: number,
  thresholdMs: number = 60_000
) {
  const [isStale, setIsStale] = useState(false);

  useEffect(() => {
    const check = () => {
      setIsStale(Date.now() - dataUpdatedAt > thresholdMs);
    };
    check();
    const timer = setInterval(check, 5_000);
    return () => clearInterval(timer);
  }, [dataUpdatedAt, thresholdMs]);

  return { isStale, ageMs: Date.now() - dataUpdatedAt };
}
tsx 
// hooks/useOptimisticBidUpdate.ts
// Optimistically update bid in UI, rollback on server failure
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { updateAdGroupBid } from "../api/adGroups";

export function useOptimisticBidUpdate() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: updateAdGroupBid,
    onMutate: async (variables) => {
      // Cancel in-flight refetches so they don't overwrite our optimistic update
      await queryClient.cancelQueries({ queryKey: ["adGroups"] });

      // Snapshot previous data for rollback
      const previous = queryClient.getQueryData(["adGroups"]);

      // Optimistically update the cache
      queryClient.setQueryData(["adGroups"], (old: any) =>
        old.map((ag: any) =>
          ag.id === variables.adGroupId
            ? { ...ag, bid_amount: variables.newBid }
            : ag
        )
      );

      return { previous };
    },
    onError: (_err, _vars, context) => {
      // Rollback to snapshot on failure
      queryClient.setQueryData(["adGroups"], context?.previous);
    },
    onSettled: () => {
      // Always refetch to ensure consistency
      queryClient.invalidateQueries({ queryKey: ["adGroups"] });
    },
  });
}
Optimistic Updates Require Rollback

Never use optimistic updates without a rollback strategy. If the server rejects a bid change (e.g., bid exceeds budget cap), the UI must revert to the previous value and show a clear error message. The onError handler above restores the snapshot, and onSettled forces a fresh fetch regardless of success or failure.

3. Performance Dashboard Architecture

The performance dashboard is a grid of KPI cards at the top, trend charts in the middle, and breakdown tables at the bottom. This three-tier layout is standard across analytics products (Google Ads, Facebook Ads Manager, Datadog) because it maps to how operators think: summary first, trends second, details on demand.

  • KPI card components with sparklines — Four cards showing total spend, total leads, cost per lead (CPL), and conversion rate. Each includes a miniature 7-day sparkline and a delta badge ("+12% vs. last week").
  • Trend charts with date range selectors — A line chart showing daily spend and leads over the selected period. Date range presets: Today, Last 7 Days, Last 30 Days, Custom.
  • Breakdown tables with drill-down — A sortable table showing per-campaign or per-ad-group metrics. Clicking a row drills into that entity's detail view.

4. Data Visualization

For chart rendering, Recharts is the recommended library for our React stack. It is built on D3 primitives but exposes a declarative React component API. For teams that prefer a lighter bundle, Chart.js with react-chartjs-2 is a solid alternative.

tsx 
// components/KpiCard.tsx
import { Sparklines, SparklinesLine } from "react-sparklines";

interface KpiCardProps {
  title: string;
  value: string;
  delta: number;         // percent change vs. prior period
  sparkData: number[];   // last 7 data points
  isLoading: boolean;
}

export function KpiCard({ title, value, delta, sparkData, isLoading }: KpiCardProps) {
  if (isLoading) {
    return (
      <div className="kpi-card skeleton">
        <div className="skeleton-line w-24 h-4" />
        <div className="skeleton-line w-32 h-8 mt-2" />
        <div className="skeleton-line w-full h-10 mt-3" />
      </div>
    );
  }

  const isPositive = delta >= 0;

  return (
    <div className="kpi-card">
      <span className="kpi-title">{title}</span>
      <span className="kpi-value">{value}</span>
      <Sparklines data={sparkData} height={32} margin={4}>
        <SparklinesLine
          color={isPositive ? "#14b8a6" : "#ef4444"}
          style={{ strokeWidth: 2, fill: "none" }}
        />
      </Sparklines>
      <span className={`kpi-delta ${isPositive ? "positive" : "negative"}`}>
        {isPositive ? "+" : ""}{delta}%
      </span>
    </div>
  );
}

Pro Tip: Loading Skeletons Over Spinners

Never use a spinner for dashboard cards or charts. Skeletons (grey shimmer placeholders that match the layout of the real content) feel faster because they give the user a structural preview of what is coming. A spinner says "wait"; a skeleton says "almost there." Use CSS @keyframes with a linear-gradient shimmer effect for production-quality loading states.

5. Performance Optimization

Dashboards and operations consoles are among the most performance-sensitive UIs in any application. An operator might have 500+ ad groups across 40 campaigns. The breakdown table might show 200 rows. Charts might render 365 data points. Without optimization, the UI will lag, scroll will stutter, and operators will lose trust in the tool.

  • Virtualized lists for large datasets — Use @tanstack/react-virtual to only render the rows visible in the viewport. A list of 1,000 ad groups only renders ~20 DOM nodes at a time.
  • Debounced search and filtering — When operators type in a search box, debounce the input by 300ms. This prevents re-rendering the entire list on every keystroke.
  • React.memo and useMemo for expensive renders — Wrap pure display components in React.memo so they skip re-render when props haven't changed. Use useMemo for derived data (filtered lists, aggregated totals).
tsx 
// hooks/useDebouncedValue.ts
import { useState, useEffect } from "react";

export function useDebouncedValue<T>(value: T, delayMs: number = 300): T {
  const [debounced, setDebounced] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delayMs);
    return () => clearTimeout(timer);
  }, [value, delayMs]);

  return debounced;
}
tsx 
// components/VirtualizedAdGroupList.tsx
import { useVirtualizer } from "@tanstack/react-virtual";
import { useRef, useMemo, memo } from "react";
import { useDebouncedValue } from "../hooks/useDebouncedValue";

interface AdGroup {
  id: number;
  name: string;
  bid_amount: number;
  campaign_name: string;
  daily_spend: number;
  leads: number;
}

// Memoized row component — skips re-render if props are the same
const AdGroupRow = memo(function AdGroupRow({ adGroup }: { adGroup: AdGroup }) {
  return (
    <div className="ad-group-row">
      <span>{adGroup.name}</span>
      <span>${adGroup.bid_amount.toFixed(2)}</span>
      <span>${adGroup.daily_spend.toFixed(2)}</span>
      <span>{adGroup.leads}</span>
    </div>
  );
});

export function VirtualizedAdGroupList({
  adGroups,
  searchTerm,
}: {
  adGroups: AdGroup[];
  searchTerm: string;
}) {
  const parentRef = useRef<HTMLDivElement>(null);
  const debouncedSearch = useDebouncedValue(searchTerm);

  // useMemo: only recompute filtered list when inputs change
  const filtered = useMemo(
    () =>
      adGroups.filter((ag) =>
        ag.name.toLowerCase().includes(debouncedSearch.toLowerCase())
      ),
    [adGroups, debouncedSearch]
  );

  const virtualizer = useVirtualizer({
    count: filtered.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 48, // estimated row height in px
    overscan: 5,              // render 5 extra rows above/below viewport
  });

  return (
    <div ref={parentRef} style={{ height: 600, overflow: "auto" }}>
      <div style={{ height: virtualizer.getTotalSize(), position: "relative" }}>
        {virtualizer.getVirtualItems().map((vRow) => (
          <div
            key={vRow.key}
            style={{
              position: "absolute",
              top: 0,
              transform: `translateY(${vRow.start}px)`,
              width: "100%",
            }}
          >
            <AdGroupRow adGroup={filtered[vRow.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}
Key Insight

The combination of useDebouncedValue, useMemo, and React.memo creates three layers of protection against unnecessary work. The debounce prevents excessive filtering. The useMemo caches the filtered array. The memo wrapper on each row prevents re-rendering rows whose data has not changed. Together, they make a 1,000-row list scroll smoothly.

When should I use React.memo vs. useMemo?

React.memo wraps a component — it prevents the component from re-rendering when its props haven't changed. Use it for leaf components that receive stable data (table rows, list items, chart points).

useMemo wraps a computation — it caches the result of an expensive calculation and only recomputes when its dependencies change. Use it for filtering arrays, computing aggregates, or transforming API data into chart format.

Rule of thumb: if you are memoizing what to render, use React.memo. If you are memoizing what to compute, use useMemo.

How do I profile React performance issues?

Use the React DevTools Profiler tab. Record a session, then look for components that re-render more often than expected. The flamegraph view highlights wasted renders in yellow/red. For runtime performance, use Chrome DevTools' Performance tab to identify long tasks and layout thrashing.

A common anti-pattern: passing a new object or array literal as a prop on every render. For example, style={{ color: "red" }} creates a new object each time, breaking React.memo. Hoist constant objects outside the component or wrap them in useMemo.

Self-Check

Before moving to Chapter 10, honestly answer these questions. If any answer is "no", go back and fix it.

Does the dashboard clearly indicate when data is stale?

Open the dashboard and disconnect your network (DevTools → Network → Offline). Wait 60 seconds. The "last updated" indicator should turn yellow or red, clearly telling the operator that the numbers on screen may not reflect reality. Reconnect — the data should refresh automatically and the indicator should return to green.

Does a failed bid change roll back cleanly in the UI?

Mock a server error on the bid update endpoint (return 422). Click "Apply" on a bid suggestion. The UI should show the new bid optimistically, then revert to the old bid when the error arrives, and display a toast or inline error message. The ad group row should never show a "stuck" intermediate state.

Does the ad group list stay smooth with 500+ rows?

Seed your development database with 500+ ad groups. Open the operations console. Scroll rapidly through the list. There should be no visible jank, no dropped frames, and no blank spaces. Open React DevTools Profiler, record a scroll session, and verify that the number of rendered DOM nodes stays constant (roughly 20–30 rows, not 500+).
