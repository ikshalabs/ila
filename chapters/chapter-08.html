Chapter 08 · Phase 5: Frontend Application

React Shell and Campaign Builder UI

Create the frontend shell with authenticated routing and role-aware navigation. Implement campaign list/create/edit flows with robust form validation and optimistic updates.

Estimated: 5–7 days
3 Advancement Levels
3 Assignments

Why This Matters

A stable shell reduces UI regressions and speeds feature rollout. Every new page you build in later chapters — billing dashboards, creative managers, analytics views — will be mounted inside this shell. If the shell is brittle, every feature team pays the cost in broken layouts, inconsistent navigation, and auth edge cases.

Fast, clear campaign setup improves execution speed for growth teams. Campaign managers interact with the create/edit forms dozens of times a day. Poor validation feedback, missing draft saves, or sluggish list rendering cause frustration that compounds across hundreds of users. The UI you build here is the primary interface for the entire platform.

The Cost of Getting This Wrong

A leaky auth guard means users see pages they shouldn't. A broken navigation state means users get stuck. A form that silently drops data means campaigns launch with wrong targeting. These are not cosmetic bugs — they are revenue-impacting incidents that erode trust with every team that depends on the platform.

Key Concepts

1. React Application Shell

The application shell is the outermost layout component that persists across page navigations. It contains the header (brand, user menu, account switcher), the sidebar (role-aware navigation links), and the content area where routed pages render. The shell itself never unmounts — only the content area swaps as users navigate.

  • Layout architecture — Header at the top, collapsible sidebar on the left, scrollable content on the right. Use CSS Grid for the overall layout and flexbox for internal alignment.
  • Route protection with auth guards — Wrap protected routes in a component that checks for a valid token before rendering. Redirect to /login if the user is unauthenticated.
  • Role-aware navigation menus — Filter sidebar links based on the user's role. Viewers should not see "Settings" or "Billing." Admins see everything. Operators see campaign management but not account settings.

2. Authentication Flow in React

The frontend mirrors the backend JWT strategy from Chapter 02. On login, the server returns a JWT. The React app stores it, attaches it to every API request, and watches for expiration. When the token is about to expire, the app silently refreshes it. When it cannot refresh, it redirects to the login page without losing the user's current URL.

The auth context is the single source of truth for the current user, account, and role. Every component that needs auth state reads from this context — never from localStorage directly.

tsx 
// src/contexts/AuthContext.tsx
import { createContext, useContext, useState, useCallback, useEffect } from "react";
import { api } from "../lib/api";

interface User {
  id: number;
  email: string;
}

interface AuthState {
  user: User | null;
  account: { id: number; name: string } | null;
  role: "viewer" | "operator" | "admin" | null;
  token: string | null;
  isAuthenticated: boolean;
}

interface AuthContextValue extends AuthState {
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
}

const AuthContext = createContext<AuthContextValue | null>(null);

export function AuthProvider({ children }: { children: React.ReactNode }) {
  const [state, setState] = useState<AuthState>(() => {
    const token = localStorage.getItem("token");
    return {
      user: null, account: null, role: null,
      token, isAuthenticated: false,
    };
  });

  // Validate token on mount
  useEffect(() => {
    if (!state.token) return;
    api.get("/auth/me")
      .then(({ data }) => setState(s => ({
        ...s, user: data.user, account: data.account,
        role: data.role, isAuthenticated: true,
      })))
      .catch(() => { localStorage.removeItem("token"); setState(s => ({
        ...s, token: null, isAuthenticated: false,
      })); });
  }, [state.token]);

  const login = useCallback(async (email: string, password: string) => {
    const { data } = await api.post("/auth/login", { email, password });
    localStorage.setItem("token", data.token);
    setState({
      user: data.user, account: data.account,
      role: data.role, token: data.token, isAuthenticated: true,
    });
  }, []);

  const logout = useCallback(() => {
    localStorage.removeItem("token");
    setState({ user: null, account: null, role: null,
               token: null, isAuthenticated: false });
  }, []);

  return (
    <AuthContext.Provider value={{ ...state, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error("useAuth must be used within AuthProvider");
  return ctx;
}
tsx 
// src/components/AuthGuard.tsx
import { Navigate, useLocation } from "react-router-dom";
import { useAuth } from "../contexts/AuthContext";

interface AuthGuardProps {
  children: React.ReactNode;
  requiredRole?: "viewer" | "operator" | "admin";
}

const ROLE_HIERARCHY = { viewer: 0, operator: 1, admin: 2 };

export function AuthGuard({ children, requiredRole }: AuthGuardProps) {
  const { isAuthenticated, role } = useAuth();
  const location = useLocation();

  if (!isAuthenticated) {
    return <Navigate to="/login" state={{ from: location }} replace />;
  }

  if (requiredRole && role) {
    if (ROLE_HIERARCHY[role] < ROLE_HIERARCHY[requiredRole]) {
      return <Navigate to="/unauthorized" replace />;
    }
  }

  return <>{children}</>;
}
Key Insight

The AuthGuard preserves the user's intended destination via location.state.from. After a successful login, you can redirect them back to the page they were trying to reach — instead of always landing on the dashboard. This is a small detail that dramatically improves the user experience when session tokens expire mid-workflow.

Why store the token in localStorage instead of cookies?

Both approaches work. localStorage is simpler for SPAs because you control the Authorization header directly. HTTP-only cookies are more secure against XSS because JavaScript cannot read them, but they require CSRF protection and complicate cross-origin API calls. For this project, localStorage + short-lived tokens + refresh rotation is a pragmatic choice. In a production system handling financial data, you might prefer HTTP-only cookies with SameSite=Strict.

3. Campaign List View

The campaign list is the most-visited page in the platform. It must load fast, handle empty states gracefully, and support pagination, filtering, and sorting without full-page reloads. TanStack Query (formerly React Query) handles server state: caching, background refetching, stale-while-revalidate, and pagination — all without manual useEffect chains.

tsx 
// src/features/campaigns/CampaignListPage.tsx
import { useState } from "react";
import { useQuery } from "@tanstack/react-query";
import { Link } from "react-router-dom";
import { api } from "../../lib/api";
import { useAuth } from "../../contexts/AuthContext";

interface Campaign {
  id: number;
  name: string;
  status: "draft" | "active" | "paused" | "completed";
  budget_cents: number;
  created_at: string;
}

interface PaginatedResponse {
  campaigns: Campaign[];
  meta: { page: number; per_page: number; total: number };
}

export function CampaignListPage() {
  const { role } = useAuth();
  const [page, setPage] = useState(1);
  const [statusFilter, setStatusFilter] = useState<string>("");
  const [sortBy, setSortBy] = useState("created_at");

  const { data, isLoading, isError, error } = useQuery<PaginatedResponse>({
    queryKey: ["campaigns", { page, status: statusFilter, sort: sortBy }],
    queryFn: () =>
      api.get("/campaigns", {
        params: { page, status: statusFilter, sort: sortBy },
      }).then(res => res.data),
    staleTime: 30_000,          // 30s before refetch
    placeholderData: (prev) => prev,  // keep old data while fetching
  });

  if (isLoading) return <CampaignListSkeleton />;
  if (isError) return <ErrorBanner message={error.message} />;
  if (!data?.campaigns.length) return <EmptyState />;

  const totalPages = Math.ceil(data.meta.total / data.meta.per_page);

  return (
    <div>
      <div className="list-header">
        <h1>Campaigns</h1>
        {role !== "viewer" && (
          <Link to="/campaigns/new" className="btn-primary">
            + New Campaign
          </Link>
        )}
      </div>

      <Filters status={statusFilter} onStatusChange={setStatusFilter}
               sortBy={sortBy} onSortChange={setSortBy} />

      <table className="campaign-table">
        <thead>
          <tr>
            <th>Name</th><th>Status</th><th>Budget</th><th>Created</th>
          </tr>
        </thead>
        <tbody>
          {data.campaigns.map(c => (
            <tr key={c.id}>
              <td><Link to={`/campaigns/${c.id}`}>{c.name}</Link></td>
              <td><StatusBadge status={c.status} /></td>
              <td>${(c.budget_cents / 100).toFixed(2)}</td>
              <td>{new Date(c.created_at).toLocaleDateString()}</td>
            </tr>
          ))}
        </tbody>
      </table>

      <Pagination page={page} totalPages={totalPages}
                   onPageChange={setPage} />
    </div>
  );
}
Never Skip Loading and Error States

Every data-fetching component needs three UI branches: loading (skeleton or spinner), error (retry button + human-readable message), and empty (call to action). Skipping any of these creates a jarring experience. TanStack Query gives you isLoading, isError, and data to make this trivial.

4. Campaign Create/Edit Forms

Campaign forms are the most complex UI in the platform. They need to handle multiple field types, validate on blur and on submit, map server-side validation errors to specific fields, and support optimistic updates for a snappy user experience. react-hook-form provides performant, uncontrolled form state management with minimal re-renders.

tsx 
// src/features/campaigns/CampaignForm.tsx
import { useForm } from "react-hook-form";
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { useNavigate } from "react-router-dom";
import { api } from "../../lib/api";
import { TextField, SelectField, DateField } from "../../components/form";

interface CampaignFormData {
  name: string;
  status: "draft" | "active" | "paused";
  budget_cents: number;
  start_date: string;
  end_date: string;
}

interface CampaignFormProps {
  defaultValues?: Partial<CampaignFormData>;
  campaignId?: number;  // present for edit, absent for create
}

export function CampaignForm({ defaultValues, campaignId }: CampaignFormProps) {
  const navigate = useNavigate();
  const queryClient = useQueryClient();
  const isEditing = !!campaignId;

  const {
    register, handleSubmit, setError,
    formState: { errors, isSubmitting },
  } = useForm<CampaignFormData>({
    defaultValues: defaultValues ?? {
      name: "", status: "draft", budget_cents: 0,
      start_date: "", end_date: "",
    },
  });

  const mutation = useMutation({
    mutationFn: (data: CampaignFormData) =>
      isEditing
        ? api.patch(`/campaigns/${campaignId}`, { campaign: data })
        : api.post("/campaigns", { campaign: data }),

    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["campaigns"] });
      navigate("/campaigns");
    },

    // Map server validation errors to form fields
    onError: (err: any) => {
      const serverErrors = err.response?.data?.errors;
      if (serverErrors) {
        Object.entries(serverErrors).forEach(([field, messages]) => {
          setError(field as any, {
            type: "server",
            message: (messages as string[]).join(", "),
          });
        });
      }
    },
  });

  return (
    <form onSubmit={handleSubmit(data => mutation.mutate(data))}>
      <TextField
        label="Campaign Name"
        error={errors.name?.message}
        {...register("name", {
          required: "Name is required",
          minLength: { value: 3, message: "At least 3 characters" },
        })}
      />

      <SelectField
        label="Status"
        options={[
          { value: "draft", label: "Draft" },
          { value: "active", label: "Active" },
          { value: "paused", label: "Paused" },
        ]}
        error={errors.status?.message}
        {...register("status")}
      />

      <TextField
        label="Budget (cents)"
        type="number"
        error={errors.budget_cents?.message}
        {...register("budget_cents", {
          required: "Budget is required",
          min: { value: 100, message: "Minimum $1.00" },
        })}
      />

      <DateField label="Start Date" error={errors.start_date?.message}
                 {...register("start_date", { required: "Required" })} />
      <DateField label="End Date" error={errors.end_date?.message}
                 {...register("end_date", { required: "Required" })} />

      <div className="form-actions">
        <button type="button" onClick={() => navigate("/campaigns")}>
          Cancel
        </button>
        <button type="submit" disabled={isSubmitting}>
          {isSubmitting ? "Saving..." : isEditing ? "Update Campaign" : "Create Campaign"}
        </button>
      </div>
    </form>
  );
}
Pro Tip: Server Error Mapping

The onError handler maps server validation errors (like { name: ["has already been taken"] }) directly to react-hook-form's field errors using setError. This means users see server-side errors inline next to the field, not in a generic toast. This pattern eliminates the "which field has the error?" guessing game.

5. Reusable Form Primitives

Every form in the platform — campaigns, ad groups, creatives, billing settings — uses the same set of primitive components. Building them once with consistent validation patterns, error display, and accessibility attributes saves dozens of hours across later chapters.

tsx 
// src/components/form/TextField.tsx
import { forwardRef } from "react";

interface TextFieldProps extends React.InputHTMLAttributes<HTMLInputElement> {
  label: string;
  error?: string;
  hint?: string;
}

export const TextField = forwardRef<HTMLInputElement, TextFieldProps>(
  ({ label, error, hint, id, ...props }, ref) => {
    const fieldId = id ?? label.toLowerCase().replace(/\s+/g, "-");

    return (
      <div className={`field ${error ? "field--error" : ""}`}>
        <label htmlFor={fieldId} className="field__label">
          {label}
        </label>
        <input
          ref={ref}
          id={fieldId}
          aria-invalid={!!error}
          aria-describedby={error ? `${fieldId}-error` : undefined}
          className="field__input"
          {...props}
        />
        {hint && !error && (
          <p className="field__hint">{hint}</p>
        )}
        {error && (
          <p id={`${fieldId}-error`} role="alert" className="field__error">
            {error}
          </p>
        )}
      </div>
    );
  }
);
TextField.displayName = "TextField";

// src/components/form/SelectField.tsx
import { forwardRef } from "react";

interface Option { value: string; label: string }

interface SelectFieldProps extends React.SelectHTMLAttributes<HTMLSelectElement> {
  label: string;
  options: Option[];
  error?: string;
  placeholder?: string;
}

export const SelectField = forwardRef<HTMLSelectElement, SelectFieldProps>(
  ({ label, options, error, placeholder, id, ...props }, ref) => {
    const fieldId = id ?? label.toLowerCase().replace(/\s+/g, "-");

    return (
      <div className={`field ${error ? "field--error" : ""}`}>
        <label htmlFor={fieldId} className="field__label">{label}</label>
        <select ref={ref} id={fieldId} className="field__select"
                aria-invalid={!!error} {...props}>
          {placeholder && <option value="">{placeholder}</option>}
          {options.map(opt => (
            <option key={opt.value} value={opt.value}>{opt.label}</option>
          ))}
        </select>
        {error && (
          <p id={`${fieldId}-error`} role="alert" className="field__error">
            {error}
          </p>
        )}
      </div>
    );
  }
);
SelectField.displayName = "SelectField";

Each primitive follows the same pattern: a forwardRef wrapper that accepts a label, an error message, and spreads remaining props to the underlying HTML element. The aria-invalid and aria-describedby attributes ensure screen readers announce validation errors. The field--error CSS class adds a red border for visual feedback.

How to implement draft/save for incomplete forms?

Use react-hook-form's watch() to observe all field values, then debounce and persist them to localStorage (keyed by the campaign ID or a temporary draft ID). On mount, check if a draft exists and prompt the user to restore it. Clear the draft on successful submit. This ensures that if the browser crashes or the user navigates away accidentally, their work is recoverable.

tsx 
// Draft auto-save hook
function useDraftSave(formId: string, watch: () => any) {
  const values = watch();

  useEffect(() => {
    const timer = setTimeout(() => {
      localStorage.setItem(`draft:${formId}`, JSON.stringify(values));
    }, 1000); // debounce 1 second
    return () => clearTimeout(timer);
  }, [formId, values]);
}

function loadDraft(formId: string) {
  const raw = localStorage.getItem(`draft:${formId}`);
  return raw ? JSON.parse(raw) : null;
}

function clearDraft(formId: string) {
  localStorage.removeItem(`draft:${formId}`);
}
Key Insight

Building form primitives with forwardRef is essential for react-hook-form integration. The library registers fields by attaching a ref to the underlying DOM element. Without forwardRef, the register() function cannot control the input, and you lose uncontrolled-mode performance benefits.

Self-Check

Before moving to Chapter 09, honestly answer these questions. If any answer is "no", go back and fix it.

Does the auth guard redirect to login and preserve the original URL?

Open a browser in incognito mode. Navigate directly to /campaigns. You should be redirected to /login. Log in with valid credentials. You should land on /campaigns, not the dashboard. Now log out and try navigating to /campaigns/42/edit. After login, you should land on that exact edit page.

Does the campaign list handle all three states correctly?

Test all three states: (1) With campaigns in the database, the list renders with pagination. (2) With the API returning a 500, the error state appears with a working retry button. (3) With no campaigns, the empty state shows a "Create your first campaign" prompt. To test the loading state, throttle your network in DevTools to "Slow 3G" and verify the skeleton appears before data loads.

Do server-side validation errors appear inline on the correct fields?

Create a campaign with a name that already exists (or violates a server-side constraint). The error message should appear under the name field, not in a generic toast or alert. Try submitting with an end date before the start date — the server should reject it, and the error should map to the end_date field. If errors only appear in the console or a generic banner, your server-error mapping is incomplete.
