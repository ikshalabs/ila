Chapter 03 · Phase 2: Core Business Logic

Campaign Management and Budget Pacing

Build the core campaign entity with lifecycle states, validation, and API contracts. Add budget pacing logic so spend is distributed safely across a campaign's lifetime instead of burning through budget in the first hour of traffic.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

Campaign controls are the center of spend planning in any ad platform. Every dollar an advertiser commits flows through a campaign object — from the moment a draft is created to the final impression delivered. If campaign state transitions are unreliable, operators lose control of what's running. If budget pacing is absent or broken, money burns too fast and unit economics collapse.

Consider this scenario: an advertiser sets a $10,000 daily budget on a campaign targeting a high-traffic geo. Without pacing, the entire budget could be exhausted in the first 90 minutes of the day when bid competition is highest. The remaining 22.5 hours produce zero impressions. The advertiser sees terrible performance metrics and churns. Runaway spend destroys trust faster than any other bug.

This chapter combines two critical concerns — campaign lifecycle management and budget pacing guardrails — because they are deeply intertwined. You cannot pace a budget without understanding campaign states (is it active? paused? completed?), and you cannot safely manage campaign states without understanding their budget implications (can we activate a campaign with zero budget remaining?).

The Cost of Getting This Wrong

A missing guard clause on a state transition can allow a paused campaign to resume spending. A budget check that doesn't account for race conditions can allow double-spending under concurrent traffic. These aren't theoretical — they're the top two billing disputes in production ad platforms. Every dollar overspent comes out of your margin or your advertiser's trust.

Key Concepts

1. Campaign Data Model and State Machine

A campaign is the top-level spending entity. It belongs to an account (tenant), has a name, a budget, start and end dates, and most importantly — a status that controls whether it can spend money. The status field is not just a label; it's a state machine with enforced transitions.

The four canonical states are: draft, active, paused, and completed. Each transition has guard clauses — conditions that must be true before the transition is allowed. For example, a campaign cannot move from draft to active unless it has a positive budget and valid dates.

ruby 
# app/models/campaign.rb
class Campaign < ApplicationRecord
  belongs_to :account
  acts_as_tenant :account

  enum :status, {
    draft:     0,
    active:    1,
    paused:    2,
    completed: 3
  }, default: :draft

  validates :name,       presence: true, length: { maximum: 255 }
  validates :budget,     presence: true, numericality: { greater_than_or_equal_to: 0 }
  validates :daily_budget, numericality: { greater_than: 0 }, allow_nil: true
  validates :start_date, presence: true
  validates :end_date,   presence: true
  validate  :end_date_after_start_date

  # ── State Transitions ──

  VALID_TRANSITIONS = {
    "draft"     => ["active"],
    "active"    => ["paused", "completed"],
    "paused"    => ["active", "completed"],
    "completed" => []
  }.freeze

  def transition_to!(new_status)
    unless VALID_TRANSITIONS[status]&.include?(new_status.to_s)
      errors.add(:status, "cannot transition from #{status} to #{new_status}")
      raise ActiveRecord::RecordInvalid, self
    end

    # Run guard clauses for the target state
    send("guard_#{new_status}!")
    update!(status: new_status)
  end

  private

  def guard_active!
    raise ActiveRecord::RecordInvalid, self if budget <= 0
    raise ActiveRecord::RecordInvalid, self if end_date <= Date.current
  end

  def guard_paused!
    # No additional guards — always allowed from valid source states
  end

  def guard_completed!
    # No additional guards — completion is always allowed
  end

  def end_date_after_start_date
    return unless start_date && end_date
    errors.add(:end_date, "must be after start date") if end_date <= start_date
  end
end
Key Insight

The state machine is implemented with a simple hash and guard methods rather than a gem like aasm or statesman. For a four-state machine, a hash is more readable and debuggable than a DSL. If your state machine grows beyond 6–8 states, consider switching to a dedicated gem that provides history tracking and event callbacks.

2. Campaign CRUD API Design

The campaign API follows standard REST conventions. The key design decisions are: use separate serializers for list vs. detail views (list responses should be lightweight), validate all inputs through strong parameters, and always scope queries to the current tenant.

ruby 
# config/routes.rb
Rails.application.routes.draw do
  post "/auth/login", to: "auth#login"
  get  "/auth/me",    to: "auth#me"

  resources :campaigns, only: [:index, :show, :create, :update] do
    member do
      post :activate
      post :pause
      post :complete
    end
  end
end
ruby 
# app/controllers/campaigns_controller.rb
class CampaignsController < ApplicationController
  before_action :set_campaign, only: [:show, :update, :activate, :pause, :complete]

  # GET /campaigns
  def index
    campaigns = current_account.campaigns
                  .order(created_at: :desc)
                  .page(params[:page])
                  .per(25)

    render json: {
      campaigns: campaigns.map { |c| CampaignListSerializer.new(c).as_json },
      meta: { total: campaigns.total_count, page: campaigns.current_page }
    }
  end

  # GET /campaigns/:id
  def show
    render json: { campaign: CampaignDetailSerializer.new(@campaign).as_json }
  end

  # POST /campaigns
  def create
    campaign = current_account.campaigns.build(campaign_params)

    if campaign.save
      render json: { campaign: CampaignDetailSerializer.new(campaign).as_json },
             status: :created
    else
      render json: { errors: campaign.errors.full_messages },
             status: :unprocessable_entity
    end
  end

  # PATCH /campaigns/:id
  def update
    if @campaign.update(campaign_params)
      render json: { campaign: CampaignDetailSerializer.new(@campaign).as_json }
    else
      render json: { errors: @campaign.errors.full_messages },
             status: :unprocessable_entity
    end
  end

  # POST /campaigns/:id/activate
  def activate
    @campaign.transition_to!(:active)
    render json: { campaign: CampaignDetailSerializer.new(@campaign).as_json }
  rescue ActiveRecord::RecordInvalid
    render json: { errors: @campaign.errors.full_messages },
           status: :unprocessable_entity
  end

  # POST /campaigns/:id/pause
  def pause
    @campaign.transition_to!(:paused)
    render json: { campaign: CampaignDetailSerializer.new(@campaign).as_json }
  rescue ActiveRecord::RecordInvalid
    render json: { errors: @campaign.errors.full_messages },
           status: :unprocessable_entity
  end

  # POST /campaigns/:id/complete
  def complete
    @campaign.transition_to!(:completed)
    render json: { campaign: CampaignDetailSerializer.new(@campaign).as_json }
  rescue ActiveRecord::RecordInvalid
    render json: { errors: @campaign.errors.full_messages },
           status: :unprocessable_entity
  end

  private

  def set_campaign
    @campaign = current_account.campaigns.find(params[:id])
  end

  def campaign_params
    params.require(:campaign).permit(
      :name, :budget, :daily_budget,
      :start_date, :end_date
    )
  end
end

Notice the two serializer classes. The list serializer returns only the fields needed for a table view (id, name, status, budget, dates). The detail serializer adds computed fields like spent_today, remaining_budget, and pacing_status. This keeps list responses fast even when an account has thousands of campaigns.

ruby 
# app/serializers/campaign_list_serializer.rb
class CampaignListSerializer

    @campaign = campaign
  end

  def as_json
    {
      id:         @campaign.id,

      end_date:   @campaign.end_date,
      created_at: @campaign.created_at
    }

end

# app/serializers/campaign_detail_serializer.rb
class CampaignDetailSerializer < CampaignListSerializer
  def as_json
    super.merge(
      daily_budget:     @campaign.daily_budget&.to_f,
      spent_today:      @campaign.spent_today.to_f,
      remaining_budget: @campaign.remaining_budget.to_f,
      pacing_status:    PacingService.status_for(@campaign),
      updated_at:       @campaign.updated_at
    )
  end
end
Always Scope to the Tenant

Every query in the controller goes through current_account.campaigns, never Campaign.find(...). This is the tenant isolation pattern from Chapter 02. If acts_as_tenant is configured, this scoping happens automatically, but the explicit form is clearer and safer as a habit.

Why separate state-transition endpoints instead of using PATCH /campaigns/:id?

State transitions have side effects — activating a campaign may trigger bid registration, pausing may cancel pending bids, completing may fire billing reconciliation. A generic PATCH with status: "active" hides the intent and makes it harder to add callbacks, logging, and guards. Dedicated POST endpoints (/activate, /pause, /complete) make the intent explicit and give you a clean place to hang before/after hooks.

3. Budget Pacing Models

Budget pacing determines how fast a campaign is allowed to spend. Without pacing, a campaign with a $10,000 daily budget will bid aggressively in the morning auction and exhaust its budget before noon. Pacing smooths spend across the day or the campaign's lifetime.

There are two budget types and three pacing strategies:

  • Daily budget — Maximum spend per calendar day. Resets at midnight UTC. Most common for always-on campaigns.
  • Lifetime budget — Total spend for the entire campaign duration. The system calculates an implied daily budget based on remaining days.
  • Even distribution — Spread spend uniformly across the day. Simple, predictable, the default for most campaigns.
  • Front-loaded — Spend more aggressively in the first half of the day, then throttle. Useful for time-sensitive promotions.
  • Performance-based — Allocate more budget to hours with historically higher conversion rates. Requires data from prior campaigns.

ruby 
# app/services/pacing_service.rb
class PacingService
  # Returns the maximum amount this campaign can spend RIGHT NOW
  # based on its pacing strategy and elapsed time today.

  def self.allowed_spend(campaign)
    new(campaign).allowed_spend
  end

  def self.status_for(campaign)
    new(campaign).status
  end

  def initialize(campaign)
    @campaign = campaign
    @now = Time.current
  end

  def allowed_spend
    effective_daily = effective_daily_budget
    return 0 if effective_daily <= 0

    elapsed_fraction = elapsed_day_fraction
    ideal_spend = effective_daily * elapsed_fraction
    actual_spend = @campaign.spent_today

    # Allow up to ideal_spend — what's already been spent
    [ideal_spend - actual_spend, 0].max
  end

  def status
    return "paused"     unless @campaign.active?
    return "exhausted"  if remaining_lifetime <= 0
    return "throttled"  if allowed_spend <= 0
    "on_pace"
  end

  private

  def effective_daily_budget
    if @campaign.daily_budget.present?
      @campaign.daily_budget
    else
      # Lifetime budget: spread remaining budget over remaining days
      remaining_days = (@campaign.end_date - Date.current).to_i
      return 0 if remaining_days <= 0
      remaining_lifetime / remaining_days.to_f
    end
  end

  def remaining_lifetime
    @campaign.budget - @campaign.total_spent
  end

  def elapsed_day_fraction
    # What fraction of the day has passed (0.0 to 1.0)
    seconds_today = @now.seconds_since_midnight
    seconds_today / 86_400.to_f
  end
end
Pro Tip: Time Zones Matter

This example uses UTC for simplicity. In production, advertisers expect daily budgets to reset at midnight in their time zone. Store a timezone column on the Account model and use Time.current.in_time_zone(account.timezone) for all pacing calculations.

4. Budget Enforcement

Knowing the pacing limit is one thing. Enforcing it under concurrent traffic is another. When thousands of bid requests arrive per second, multiple workers may read the same spent_today value simultaneously, each concluding "there's budget remaining," and all proceeding to spend — blowing past the budget.

The solution involves three layers: optimistic checks for fast rejection, pessimistic locks for atomic updates, and emergency cutoffs for catastrophic failures. 

  ┌───────────────────────────────────────────────────────────┐
  │                  BID REQUEST ARRIVES                       │
  └──────────────────────┬────────────────────────────────────┘
                         │
                         ▼
              ┌──────────────────────┐
              │  Layer 1: FAST CHECK │   Read from Redis cache
              │  Is campaign active? │   "Is there likely budget?"
              │  Rough budget check  │   Rejects 90% of over-budget
              └──────────┬───────────┘
                         │ Pass
                         ▼
              ┌──────────────────────┐
              │  Layer 2: DB LOCK    │   SELECT ... FOR UPDATE
              │  Atomic spend check  │   Read true spent_today
              │  Deduct & commit     │   Single transaction
              └──────────┬───────────┘
                         │ Pass
                         ▼
              ┌──────────────────────┐
              │  Layer 3: POST-SPEND │   Async reconciliation
              │  Update Redis cache  │   Log budget decision
              │  Check daily totals  │   Trigger alerts if over
              └──────────────────────┘
Figure 3: Three-layer budget enforcement pipeline
ruby 
# app/services/budget_enforcer.rb
class BudgetEnforcer
  Result = Struct.new(:allowed, :reason, keyword_init: true)

  def self.can_spend?(campaign, amount)
    new(campaign, amount).check
  end

  def initialize(campaign, amount)
    @campaign = campaign
    @amount   = amount
  end

  def check
    # Layer 1: Fast rejection (no DB hit)
    return deny("campaign_not_active") unless @campaign.active?
    return deny("emergency_cutoff")    if emergency_cutoff?

    # Layer 2: Atomic budget check with row lock
    Campaign.transaction(isolation: :serializable) do
      locked = Campaign.lock("FOR UPDATE").find(@campaign.id)
      allowed = PacingService.allowed_spend(locked)

      if @amount > allowed
        return deny("pacing_limit_exceeded")
      end

      if locked.total_spent + @amount > locked.budget
        return deny("lifetime_budget_exceeded")
      end

      # Deduct budget atomically
      locked.increment!(:spent_today, @amount)
      locked.increment!(:total_spent, @amount)

      allow
    end
  rescue ActiveRecord::SerializationFailure
    # Serializable isolation can raise on conflict — retry or reject
    deny("transaction_conflict")
  end

  private

  def emergency_cutoff?
    Redis.current.get("cutoff:campaign:#{@campaign.id}") == "1"
  end

  def allow
    Result.new(allowed: true, reason: "approved")
  end

  def deny(reason)
    Result.new(allowed: false, reason: reason)
  end
end
Race Conditions Are the #1 Budget Bug

Without SELECT ... FOR UPDATE or serializable isolation, two concurrent bid requests can both read spent_today = $9,500 on a $10,000 budget, both conclude there's $500 remaining, and both spend $500 — resulting in $10,500 total. This happens under real traffic. Always use database-level locking for spend mutations.

PostgreSQL's SERIALIZABLE isolation level provides the strongest guarantee. Here's what's happening at the database level:

sql 
-- Transaction 1 (Worker A)
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;

SELECT spent_today, total_spent, budget
  FROM campaigns
  WHERE id = 42
  FOR UPDATE;
-- Result: spent_today=9500, total_spent=45000, budget=100000

-- Check: 9500 + 500 <= daily_limit? YES
-- Check: 45000 + 500 <= 100000?     YES

UPDATE campaigns
  SET spent_today = spent_today + 500,
      total_spent = total_spent + 500
  WHERE id = 42;

COMMIT;

-- Transaction 2 (Worker B) runs concurrently
-- The FOR UPDATE clause forces Worker B to WAIT until
-- Worker A commits. Worker B then sees the updated values
-- (spent_today=10000) and correctly rejects the spend.
When should I use Redis vs. PostgreSQL for budget checks?

Use Redis for the fast-check layer (Layer 1). It's an approximate, eventually-consistent check that rejects obviously over-budget requests without touching the database. Redis counters are updated asynchronously after each spend event.

Use PostgreSQL for the authoritative check (Layer 2). The database is the source of truth. Only PostgreSQL transactions with row-level locking can guarantee that two concurrent requests don't both pass the budget check. Never rely solely on Redis for budget enforcement — its counters can drift under failure.

Emergency Spend Cutoff

Sometimes you need to stop spending immediately — a billing dispute, a detected fraud pattern, or a client calling in panic. The emergency cutoff is a Redis flag that short-circuits all budget checks without requiring a database write:

ruby 
# app/services/emergency_cutoff.rb
class EmergencyCutoff
  def self.activate!(campaign, reason:, activated_by:)
    Redis.current.set(
      "cutoff:campaign:#{campaign.id}", "1",
      ex: 24.hours.to_i  # Auto-expire after 24h as safety net
    )

    BudgetAuditLog.create!(
      campaign: campaign,
      event:    "emergency_cutoff_activated",
      reason:   reason,
      user_id:  activated_by.id,
      metadata: { spent_at_cutoff: campaign.spent_today }
    )

    Rails.logger.warn(
      "budget.emergency_cutoff",
      campaign_id: campaign.id,
      reason: reason
    )
  end

  def self.deactivate!(campaign, deactivated_by:)
    Redis.current.del("cutoff:campaign:#{campaign.id}")

    BudgetAuditLog.create!(
      campaign: campaign,
      event:    "emergency_cutoff_deactivated",
      user_id:  deactivated_by.id
    )
  end

  def self.active?(campaign)
    Redis.current.get("cutoff:campaign:#{campaign.id}") == "1"
  end
end
Key Insight

The cutoff key has a 24-hour TTL as a safety net. If an operator activates a cutoff but forgets to deactivate it, it auto-expires rather than silently blocking the campaign forever. The audit log records both activation and deactivation, so there's always a paper trail for billing disputes.

5. Auditability

In an ad platform, every budget decision must be explainable. When an advertiser asks "why did my campaign stop spending at 2pm?", you need to pull up a log that says exactly what happened: "pacing limit exceeded at 14:02 UTC; spent_today was $8,450 against a $8,500 pacing target; next check passed at 14:15 UTC."

The audit trail has two components: a budget audit log for spend decisions and a campaign change history for state and configuration changes.

ruby

ruby 
# app/models/budget_audit_log.rb
class BudgetAuditLog < ApplicationRecord
  belongs_to :campaign
  belongs_to :account

  EVENTS = %w[
    spend_approved
    spend_rejected
    pacing_throttled
    lifetime_budget_exceeded
    daily_budget_exceeded
    emergency_cutoff_activated
    emergency_cutoff_deactivated
    budget_reconciled
    campaign_state_changed
  ].freeze

  validates :event, inclusion: { in: EVENTS }

  scope :for_date, ->(date) {
    where(created_at: date.beginning_of_day..date.end_of_day)
  }

  scope :rejections, -> {
    where(event: %w[spend_rejected pacing_throttled lifetime_budget_exceeded])
  }
end
ruby 
# app/models/concerns/auditable.rb
module Auditable
  extend ActiveSupport::Concern

  included do
    has_many :change_logs, as: :auditable, dependent: :destroy
    after_update :log_changes
  end

  private

  def log_changes
    return if saved_changes.blank?

    # Skip internal fields
    tracked = saved_changes.except("updated_at", "spent_today", "total_spent")
    return if tracked.blank?

    change_logs.create!(
      account_id:  account_id,
      field_changes: tracked,
      changed_by:   Current.user&.id
    )
  end
end

Audit Logs Are Append-Only

Never update or delete audit log records. They are an immutable ledger. If a correction is needed, create a new log entry with a "correction" event that references the original. This ensures the audit trail is trustworthy for billing disputes and compliance reviews.

How do you handle high-volume audit logging without impacting bid latency?

For spend decisions that happen on every bid request (potentially thousands per second), write audit logs asynchronously using a background job queue (Sidekiq). Buffer events in Redis and flush them to PostgreSQL in batches every few seconds. The critical path (bid response) should never wait for an audit log insert.

For state changes and manual actions (activate, pause, cutoff), write synchronously. These are low-frequency operations where the operator expects immediate confirmation that their action was recorded.

Self-Check

Before moving to Chapter 04, honestly answer these questions. If any answer is "no", go back and fix it.

Are invalid state transitions fully blocked and well-messaged?

Try every invalid transition: draft → completed, draft → paused, completed → active, active → draft. Each should return a 422 Unprocessable Entity with a clear error message like "status cannot transition from draft to completed". The error should tell the operator exactly what's wrong and what transitions are available. Check that the valid transition from draft → active is also blocked when the budget is zero or the end_date is in the past.

Does pacing prevent budget exhaustion under burst traffic?

Simulate a scenario: a campaign with a $10,000 daily budget at 25% through the day (6:00 AM). The pacing service should allow roughly $2,500 of spend. If spent_today is already $2,400, only $100 should be allowed. If you send 100 concurrent requests each trying to spend $100, at most one should succeed (the rest should be throttled). Test this with a multi-threaded spec or by checking that spent_today never exceeds the pacing target by more than one transaction amount.

Are budget decisions explainable from logs and evidence?

Run a sequence of operations: create a campaign, activate it, spend against it until pacing throttles, trigger an emergency cutoff, then deactivate the cutoff. Query BudgetAuditLog.where(campaign_id: campaign.id).order(:created_at) and verify you can reconstruct exactly what happened, when, and why. Every entry should have a reason field that a non-engineer (account manager, billing support) can understand.

Can operators recover quickly from validation errors?

Try to activate a campaign with a zero budget. The error message should say something like "Cannot activate: budget must be greater than 0", not a generic "Validation failed". The operator should know exactly what to fix. After updating the budget to a positive value, the activation should succeed on the next attempt without needing to refresh or re-create the campaign.
