Chapter 10 · Phase 6: Ship It

Testing, Launch Hardening and Capstone

Introduce automated checks across API, UI, and end-to-end critical paths. Run a full launch simulation, incident drill, and final architecture defense. This is the culmination of everything you have built — prove it is production-ready.

Estimated: 5–7 days
3 Advancement Levels
3 Assignments

Why This Matters

You have spent nine chapters building a real-time ad bidding platform — authentication, campaigns, bidding engines, billing, dashboards, and operational tooling. None of it matters if you cannot ship it reliably and defend it under pressure.

Reliable release velocity requires predictable quality gates. A green CI pipeline is not a vanity metric — it is the mechanism that prevents regressions from reaching production. Readiness is proven by operational behavior, not by code volume. A team that can deploy on demand, recover from incidents, and explain every architectural decision is a team that ships with confidence.

This final chapter combines two critical disciplines: the engineering rigor of automated testing with the operational maturity of launch hardening. Together, they form the bridge between "it works on my machine" and "it runs in production under load, with monitoring, and we can defend every decision."

The Cost of Skipping This

Teams that ship without test coverage and launch hardening do not move faster — they move in circles. Every production incident becomes an unplanned sprint. Every regression becomes a fire drill. Every deployment becomes a gamble. The time you invest here pays for itself within the first month of production operation.

Key Concepts

1. Testing Strategy

A mature testing strategy follows the test pyramid: many fast unit tests at the base, fewer integration tests in the middle, and a small number of end-to-end tests at the top. Each layer catches different classes of bugs at different costs.

  • Unit tests — Test individual methods and classes in isolation. Fastest to run, cheapest to maintain. Cover business logic, validations, and service objects.
  • Integration tests — Test how components work together. API request specs that hit the full Rails stack (routing, controllers, models, database) without a browser.
  • End-to-end tests — Test complete user flows through a real browser. Slowest and most expensive, but catch issues that no other layer can (JavaScript errors, CSS layout breaks, race conditions).
                        ┌───────────┐
                       /  E2E Tests  \            Slow, expensive, high confidence
                      /   (Cypress)   \           5-15 tests for critical paths
                     /─────────────────\
                    /  Integration Tests \         Medium speed, medium confidence
                   /    (RSpec Request)    \       50-200 request specs
                  /────────────────────────\
                 /      Unit Tests          \      Fast, cheap, high volume
                /    (RSpec + Jest)           \    500+ specs across models,
               /──────────────────────────────\   services, and components

  ┌─────────────────────────────────────────────────────────────────────┐
  │  RULE OF THUMB                                                      │
  │  70% unit  ·  20% integration  ·  10% end-to-end                   │
  │  If a bug can be caught by a unit test, don't write an E2E for it  │
  └─────────────────────────────────────────────────────────────────────┘
Figure 1: The test pyramid — catch bugs at the cheapest layer possible

For our Rails + React platform, the pyramid maps to concrete tools:

  • RSpec for all backend testing — model specs, service specs, request specs, and job specs
  • Jest + React Testing Library for frontend component and hook testing
  • Cypress or Playwright for browser-based end-to-end flows

# spec/models/campaign_spec.rb
RSpec.describe Campaign, type: :model do
  describe "validations" do
    it { is_expected.to validate_presence_of(:name) }
    it { is_expected.to validate_numericality_of(:daily_budget).is_greater_than(0) }
    it { is_expected.to belong_to(:account) }
  end

  describe "#activate!" do
    let(:campaign) { create(:campaign, status: :draft) }

    it "transitions from draft to active" do
      campaign.activate!
      expect(campaign.reload.status).to eq("active")
    end

    it "raises when budget is zero" do
      campaign.update!(daily_budget: 0)
      expect { campaign.activate! }.to raise_error(Campaign::InsufficientBudget)
    end
  end
end
Key Insight

Test behavior, not implementation. A test that asserts "calling activate! changes the status to active" is useful. A test that asserts "the third line of the method calls update_column" is brittle and breaks when you refactor. Good tests survive refactoring.

2. CI/CD Pipeline

Continuous Integration ensures every push is validated. Continuous Deployment automates the release process. Together, they eliminate the "works on my machine" problem and enforce quality gates before code reaches production.

# .github/workflows/ci.yml
name: CI Pipeline
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          bundler-cache: true
      - run: bundle exec rubocop --parallel
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "yarn"
      - run: yarn install --frozen-lockfile
      - run: yarn lint

  test-backend:
    runs-on: ubuntu-latest
    needs: lint
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_PASSWORD: postgres
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
      redis:
        image: redis:7
        ports: ["6379:6379"]
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          bundler-cache: true
      - run: bundle exec rails db:setup
      - run: bundle exec rspec --format progress --profile 10
      - run: bundle exec rails coverage:check  # Fail if <90%

  test-frontend:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "yarn"
      - run: yarn install --frozen-lockfile
      - run: yarn test --coverage --watchAll=false

  e2e:
    runs-on: ubuntu-latest
    needs: [test-backend, test-frontend]
    steps:
      - uses: actions/checkout@v4
      - uses: cypress-io/github-action@v6
        with:
          build: yarn build
          start: yarn start
          wait-on: "http://localhost:3000"
Parallel Execution is Essential

Backend and frontend tests run in parallel (both needs: lint). E2E tests run only after both pass. This structure means a lint failure blocks everything immediately, saving CI minutes. A typical pipeline should complete in under 10 minutes. If it takes longer, developers stop waiting and merge without green builds.

3. End-to-End Testing

End-to-end tests validate critical user flows through a real browser. They are the most expensive tests to write and maintain, so you should reserve them for paths where a failure would have the highest business impact: login, campaign creation, bid submission, and payment flows.

javascript 
// cypress/e2e/campaign-lifecycle.cy.js
describe("Campaign Lifecycle", () => {
  beforeEach(() => {
    cy.task("db:seed");   // Reset DB to known state
    cy.login("operator@example.com", "password123");
  });

  it("creates, activates, and pauses a campaign", () => {
    // Navigate to campaigns
    cy.visit("/campaigns");
    cy.findByRole("button", { name: /new campaign/i }).click();

    // Fill in campaign form
    cy.findByLabelText(/campaign name/i).type("Holiday Sale 2025");
    cy.findByLabelText(/daily budget/i).type("500");
    cy.findByLabelText(/bid strategy/i).select("maximize_clicks");
    cy.findByRole("button", { name: /create campaign/i }).click();

    // Verify creation
    cy.findByText("Holiday Sale 2025").should("be.visible");
    cy.findByTestId("campaign-status").should("contain", "Draft");

    // Activate
    cy.findByRole("button", { name: /activate/i }).click();
    cy.findByTestId("campaign-status").should("contain", "Active");

    // Pause
    cy.findByRole("button", { name: /pause/i }).click();
    cy.findByTestId("campaign-status").should("contain", "Paused");
  });

  it("blocks viewer from creating campaigns", () => {
    cy.login("viewer@example.com", "password123");
    cy.visit("/campaigns");
    cy.findByRole("button", { name: /new campaign/i }).should("not.exist");
  });
});

Pro Tip: Managing Flaky Tests

Flaky tests erode trust in the entire test suite. When a test fails intermittently, quarantine it — move it to a separate CI job that does not block merges, tag it with @flaky, and fix it within one sprint. Never leave a flaky test in the critical path. Track flakiness rates over time; a suite with >2% flakiness needs structural attention.

4. Launch Hardening

Launch hardening is the process of stress-testing your application against the conditions it will face in production. This is not about writing more features — it is about making the features you already have resilient, observable, and secure.

ruby 
# app/controllers/health_controller.rb
class HealthController < ApplicationController
  skip_before_action :authenticate_request!

  def show
    checks = {
      database:  database_healthy?,
      redis:     redis_healthy?,
      sidekiq:   sidekiq_healthy?,
      disk_space: disk_space_ok?
    }

    status = checks.values.all? ? :ok : :service_unavailable

    render json: {
      status:    status == :ok ? "healthy" : "degraded",
      checks:    checks,
      version:   ENV["GIT_SHA"] || "dev",
      timestamp: Time.current.iso8601
    }, status: status
  end

  private

  def database_healthy?
    ActiveRecord::Base.connection.active?
  rescue
    false
  end

  def redis_healthy?
    Redis.current.ping == "PONG"
  rescue
    false
  end

  def sidekiq_healthy?
    Sidekiq::ProcessSet.new.size > 0
  rescue
    false
  end

  def disk_space_ok?
    usage = `df -h / | tail -1 | awk '{print $5}'`.to_i
    usage < 90
  rescue
    false
  end
end
ruby 
# config/initializers/sentry.rb — Error Monitoring
Sentry.init do |config|
  config.dsn = ENV["SENTRY_DSN"]
  config.breadcrumbs_logger = [:active_support_logger, :http_logger]
  config.traces_sample_rate = 0.1  # 10% of transactions for performance monitoring
  config.profiles_sample_rate = 0.1

  config.before_send = lambda do |event, hint|
    # Scrub sensitive data before sending to Sentry
    event.request.data = "[FILTERED]" if event.request&.data
    event
  end
end

Load Testing is Non-Negotiable

Use a tool like k6, wrk, or Apache Bench to simulate production traffic patterns. Your bidding engine must handle burst traffic (ad auction spikes) without degrading. If p95 response time exceeds 500ms under 100 concurrent users, you have a bottleneck that will become a production incident.

5. Capstone Defense

The capstone defense is the final assessment. It is not a written exam — it is a live demonstration of your ability to build, operate, and defend a production-grade application. You will walk through your architecture decisions, run an incident drill under time pressure, and present your portfolio to a panel.

This mirrors real industry practice: senior engineers are not evaluated on what they memorized, but on how they reason under pressure, how they communicate tradeoffs, and how they respond to the unexpected.

The incident drill works as follows: the evaluator will introduce a failure into a running instance of your application (database connection drop, memory leak, misconfigured environment variable, or security vulnerability). You have 20 minutes to:

  1. Detect the failure using your monitoring tools (not by reading the injected code)
  2. Diagnose the root cause using logs, metrics, and health checks
  3. Resolve the issue or implement a mitigation
  4. Communicate the incident timeline to the panel as if they were stakeholders
Sample incident drill scenarios

The evaluator may choose from scenarios like these. You will not know which one in advance:

  • Database connection pool exhaustion — Too many long-running queries saturate the pool. New requests queue and timeout. How do you detect it? How do you fix it?
  • Redis memory limit reached — Sidekiq jobs start failing silently. Cached data becomes stale. The dashboard shows outdated numbers.
  • JWT secret rotation gone wrong — A new secret is deployed but existing tokens use the old secret. All authenticated users are suddenly logged out.
  • N+1 query under load — A campaign listing page that worked fine with 10 campaigns now takes 30 seconds with 1,000 campaigns.
  • Cross-tenant data leak — A new endpoint was added without tenant scoping. Account A can see Account B's bid history.
Portfolio presentation guidelines

Your portfolio should be a GitHub repository (or set of repositories) that demonstrates the complete platform you built across all 10 chapters. It should include:

  • A clear README with setup instructions that actually work (test them on a clean machine)
  • Architecture decision records (ADRs) for major choices (database schema, caching strategy, auth approach)
  • A running demo (deployed to a staging environment or Docker Compose setup)
  • Test coverage report visible in CI output
  • A 5-minute video walkthrough demonstrating the key user flows

This portfolio is what you show in job interviews. It is proof that you can build, test, deploy, and operate a non-trivial application — not just follow a tutorial.

Assignment 2
Level 2 — Professional

CI/CD Pipeline with Quality Gates and E2E Tests

Configure a GitHub Actions pipeline that enforces quality gates and add end-to-end tests for critical user flows.

  • Create .github/workflows/ci.yml with lint, backend test, frontend test, and E2E jobs

  • Add coverage threshold gate: pipeline fails if coverage drops below 90%

  • Write E2E tests for: login flow, campaign creation, bid submission, and dashboard rendering

Done when

Opening a PR triggers the full CI pipeline. All jobs pass. Introducing a linting violation, a failing test, or a coverage drop below 90% causes the pipeline to fail and blocks the merge. E2E tests run the complete campaign lifecycle in a real browser.

Assignment 3
Level 3 — Production-Ready

Capstone — Full Launch Simulation and Architecture Defense

Complete the launch hardening checklist, survive an incident drill, and defend your architecture in a live capstone presentation. This is the final evaluation of the ILA certification.

Self-Check

This is the final self-check of the entire program. Answer honestly. If any answer is "no", go back and fix it before scheduling your capstone defense.

Does your test suite catch real bugs, or does it just confirm happy paths?

Review your test suite. Count the number of tests that verify error conditions: invalid inputs, unauthorized access, network failures, race conditions. If more than 80% of your tests only check the "everything works perfectly" scenario, your suite will not catch the bugs that actually happen in production. Add edge case tests for every critical service.

Can you deploy to production with a single command and roll back in under 5 minutes?

Your deployment process should be documented and automated. A manual deployment that requires SSH access, running migrations by hand, and restarting services is a process that will break at 2 AM on a Saturday. Your rollback should be as simple as git revert and deploy — or ideally, a single "rollback" button in your CI/CD system.

Can you explain every architectural decision you made and the tradeoffs involved?

Pick any three major decisions from your codebase (database schema design, caching strategy, authentication approach, API versioning, background job architecture). For each one, can you explain: (1) what alternatives you considered, (2) why you chose this approach, and (3) what you would change if requirements shifted? If you cannot articulate tradeoffs, you do not understand the decision deeply enough.

If production went down right now, would you know within 60 seconds?

Check your monitoring setup. Is your health check endpoint being polled? Are error rates being tracked? Do you have alerting rules that fire when thresholds are breached? If the answer to any of these is "no," your observability is not production-ready. A silent failure is worse than a loud one — at least a loud failure gets fixed.
