Chapter 07 · Phase 4: Data and Money

Billing Ledger and Analytics API

Create immutable billing events and receipt records tied to monetizable actions. Build account-scoped analytics endpoints for KPI cards, trends, and breakdown tables.

Estimated: 4–5 days
3 Advancement Levels
3 Assignments

Why This Matters

In an ad bidding platform, every impression served, every lead captured, and every conversion attributed eventually becomes a line item on an invoice. If your billing data is wrong — even by a fraction of a cent per event — errors compound across millions of transactions into disputes, lost revenue, and broken trust.

Financial correctness underpins invoicing, trust, and audits. Good analytics enable faster, better bid decisions. This chapter builds both: an append-only billing ledger that guarantees auditability, and an analytics aggregation API that turns raw data into actionable KPI cards, trend lines, and breakdown tables for the frontend.

The Cost of Getting This Wrong

Mutable billing records are a compliance nightmare. If anyone can UPDATE or DELETE a billing row, you cannot prove what happened during an audit. Regulatory frameworks (SOX, PCI-DSS) require an immutable trail. Similarly, slow or inaccurate analytics cause operators to make bad bidding decisions — overbidding wastes budget, underbidding loses market share.

Key Concepts

1. Append-Only Billing Ledger

The billing ledger follows a single, inviolable rule: rows are only inserted, never updated or deleted. Every billable action — an impression served, a lead captured, a conversion attributed — creates a new BillingEvent record with the exact amount, currency, and timestamp. If a charge needs to be reversed, you insert a new adjustment entry with a negative amount that references the original event.

  • Immutable event model — amount, currency, timestamps, and event type are locked at creation.
  • Receipt creation — each billable event generates a receipt with a unique reference number.
  • No UPDATE/DELETE — corrections are always handled via adjustment entries that reference the original.
  • Full audit trail — every financial change is traceable to a specific event and actor.

ruby 
# app/models/billing_event.rb
class BillingEvent < ApplicationRecord
  acts_as_tenant :account

  belongs_to :account
  has_one    :receipt, dependent: :restrict_with_error
  belongs_to :adjustment_for, class_name: "BillingEvent", optional: true

  enum :event_type, {
    impression: "impression",
    lead:       "lead",
    conversion: "conversion",
    adjustment: "adjustment",
    refund:     "refund"
  }

  validates :event_type,   presence: true
  validates :amount_cents,  presence: true
  validates :currency,      presence: true
  validates :reference_id,  presence: true, uniqueness: true
  validate  :immutability_guard, on: :update

  before_create :generate_reference_id

  private

  def immutability_guard
    errors.add(:base, "Billing events are immutable")
  end

  def generate_reference_id
    self.reference_id ||= "BE-#{SecureRandom.uuid}"
  end
end
Key Insight

Notice the immutability_guard validation on update. Even though your application logic should never call update on a billing event, this guard catches accidental mutations at the model layer. Defense in depth — if the code has a bug, the model stops it. You can further enforce this at the database level with a PostgreSQL trigger that raises an exception on UPDATE or DELETE.

2. Billing API

The billing API provides receipt endpoints with account-scoping, reconciliation views for daily totals, and refund/adjustment entries with a full audit trail. All queries are scoped through current_account — no cross-tenant access is possible.

ruby 
# app/models/receipt.rb
class Receipt < ApplicationRecord
  belongs_to :billing_event

  enum :status, { issued: 0, voided: 1 }

  validates :receipt_number, presence: true, uniqueness: true
  validates :total_cents,    presence: true

  before_validation :set_receipt_number, on: :create

  private

  def set_receipt_number
    self.receipt_number ||= "RCT-#{Time.current.strftime('%Y%m%d')}-#{SecureRandom.hex(6).upcase}"
  end
end
ruby 
# app/controllers/api/v1/billing_events_controller.rb
module Api
  module V1
    class BillingEventsController < ApplicationController
      def index
        events = current_account.billing_events
                   .order(created_at: :desc)
                   .page(params[:page])
                   .per(25)

        render json: {
          billing_events: events.as_json(include: :receipt),
          meta: pagination_meta(events)
        }
      end

      def show
        event = current_account.billing_events.find(params[:id])
        render json: event.as_json(include: :receipt)
      end

      def reconciliation
        daily_totals = current_account.billing_events
          .where(created_at: params[:start_date]..params[:end_date])
          .group("DATE(created_at)")
          .group(:event_type)
          .select(
            "DATE(created_at) AS date",
            :event_type,
            "SUM(amount_cents) AS total_cents",
            "COUNT(*) AS event_count"
          )

        render json: { reconciliation: daily_totals }
      end

      def create_adjustment
        original = current_account.billing_events.find(params[:billing_event_id])

        adjustment = current_account.billing_events.create!(
          event_type:       :adjustment,
          amount_cents:     -original.amount_cents,
          currency:         original.currency,
          adjustment_for:   original,
          metadata:         { reason: params[:reason], adjusted_by: current_user.id }
        )

        render json: adjustment.as_json(include: :receipt), status: :created
      end
    end
  end
end
Adjustments, Not Edits

Never modify a billing event directly. The create_adjustment action creates a new row with a negative amount_cents that references the original event via adjustment_for_id. The ledger balance is always the sum of all events — originals and adjustments alike. This is the same pattern used by banks, payment processors, and accounting systems worldwide.

3. Analytics Service Layer

The analytics service provides a stable contract for frontend consumption. It exposes KPI card endpoints (spend, leads, conversions, efficiency), trend data for charts, and breakdown tables for detailed analysis. The service layer abstracts whether data comes from live aggregation or materialized snapshots.

ruby 
# app/services/analytics_service.rb
class AnalyticsService
  def initialize(account, date_range: 30.days.ago..Time.current)
    @account    = account
    @date_range = date_range
  end

  # KPI Cards — top-level numbers for the dashboard
  def kpi_cards
    {
      total_spend:    total_spend,
      total_leads:    total_leads,
      total_conversions: total_conversions,
      cost_per_lead:  cost_per_lead,
      cost_per_conversion: cost_per_conversion,
      conversion_rate: conversion_rate
    }
  end

  # Trend data — daily aggregates for line/bar charts
  def trends(group_by: :day)
    @account.billing_events
      .where(created_at: @date_range)
      .group("DATE(created_at)")
      .group(:event_type)
      .select(
        "DATE(created_at) AS date",
        :event_type,
        "SUM(amount_cents) AS total_cents",
        "COUNT(*) AS event_count"
      )
      .order("date ASC")
  end

  # Breakdown table — drill-down by source, campaign, etc.
  def breakdown(dimension: :event_type)
    @account.billing_events
      .where(created_at: @date_range)
      .group(dimension)
      .select(
        "#{dimension} AS dimension",
        "SUM(amount_cents) AS total_cents",
        "COUNT(*) AS event_count",
        "AVG(amount_cents) AS avg_cents"
      )
      .order("total_cents DESC")
  end

  private

  def total_spend
    scoped_events.sum(:amount_cents)
  end

  def total_leads
    scoped_events.where(event_type: :lead).count
  end

  def total_conversions
    scoped_events.where(event_type: :conversion).count
  end

  def cost_per_lead
    return 0 if total_leads.zero?
    total_spend.to_f / total_leads
  end

  def cost_per_conversion
    return 0 if total_conversions.zero?
    total_spend.to_f / total_conversions
  end

  def conversion_rate
    return 0.0 if total_leads.zero?
    (total_conversions.to_f / total_leads * 100).round(2)
  end

  def scoped_events
    @scoped_events ||= @account.billing_events.where(created_at: @date_range)
  end
end
ruby 
# app/controllers/api/v1/analytics_controller.rb
module Api
  module V1
    class AnalyticsController < ApplicationController
      def kpi
        service = AnalyticsService.new(current_account, date_range: parsed_range)
        render json: { kpi: service.kpi_cards }
      end

      def trends
        service = AnalyticsService.new(current_account, date_range: parsed_range)
        render json: { trends: service.trends }
      end

      def breakdown
        service = AnalyticsService.new(current_account, date_range: parsed_range)
        dimension = params[:dimension]&.to_sym || :event_type
        render json: { breakdown: service.breakdown(dimension: dimension) }
      end

      private

      def parsed_range
        start_date = Date.parse(params[:start_date] || 30.days.ago.to_s)
        end_date   = Date.parse(params[:end_date]   || Date.current.to_s)
        start_date.beginning_of_day..end_date.end_of_day
      end
    end
  end
end
Live aggregation vs. materialized snapshots — when to use which?

Live aggregation queries raw tables on every request. It is always accurate but can be slow on large datasets. Use it when data freshness matters more than speed — for example, the real-time spend counter on a campaign detail page.

Materialized snapshots pre-compute aggregates and store them in summary tables or PostgreSQL materialized views. They are fast to read but stale by the refresh interval. Use them for dashboard KPI cards and trend charts where a 5–15 minute lag is acceptable. The analytics service can seamlessly switch between the two strategies based on the query context.

4. Query Optimization

Analytics queries aggregate millions of rows. Without proper indexing and query planning, dashboard pages will time out. PostgreSQL materialized views, composite indexes, and EXPLAIN ANALYZE are your primary tools.

sql 
-- Materialized view for daily account summaries
CREATE MATERIALIZED VIEW account_daily_summaries AS
SELECT
  account_id,
  DATE(created_at)   AS event_date,
  event_type,
  SUM(amount_cents)   AS total_cents,
  COUNT(*)             AS event_count,
  AVG(amount_cents)   AS avg_cents
FROM billing_events
GROUP BY account_id, DATE(created_at), event_type
ORDER BY event_date DESC;

-- Create a unique index for concurrent refresh
CREATE UNIQUE INDEX idx_account_daily_summaries_unique
  ON account_daily_summaries (account_id, event_date, event_type);

-- Refresh without locking reads (requires unique index)
REFRESH MATERIALIZED VIEW CONCURRENTLY account_daily_summaries;
sql 
-- EXPLAIN ANALYZE: verify the query plan uses indexes
EXPLAIN ANALYZE
SELECT
  event_type,
  SUM(amount_cents) AS total,
  COUNT(*)          AS cnt
FROM billing_events
WHERE account_id = 42
  AND created_at BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY event_type;

-- Expected: Index Scan using idx_billing_events_account_id_created_at
-- If you see Seq Scan, add the composite index:
CREATE INDEX idx_billing_events_acct_date
  ON billing_events (account_id, created_at);
Pro Tip: EXPLAIN Before You Ship

Run EXPLAIN ANALYZE on every analytics query before deploying. Look for sequential scans on large tables — they indicate missing indexes. A query that runs in 2ms on your dev database with 100 rows can take 30 seconds in production with 10 million rows. The composite index (account_id, created_at) is the most important index in this chapter.

How do you schedule materialized view refreshes?

Use a recurring Sidekiq job (or solid_queue if you followed Chapter 06) to refresh the materialized view on a schedule. Every 5–15 minutes is typical for dashboard analytics. The CONCURRENTLY option lets reads continue during the refresh.

ruby 
# app/jobs/refresh_analytics_view_job.rb
class RefreshAnalyticsViewJob < ApplicationJob
  queue_as :low

  def perform
    ActiveRecord::Base.connection.execute(
      "REFRESH MATERIALIZED VIEW CONCURRENTLY account_daily_summaries"
    )
    Rails.logger.info("analytics.materialized_view.refreshed")
  end
end

5. Consistency and Reliability

Analytics dashboards must never show numbers that contradict each other. If the KPI card says "Total Spend: $4,230" but the breakdown table sums to $4,180, the user loses trust in the entire system. Consistency checks, graceful degradation, and caching are essential.

ruby 
# app/services/analytics_cache.rb
class AnalyticsCache
  TTL = 5.minutes

  def initialize(account)
    @account = account
  end

  def fetch_kpi(date_range:)
    cache_key = kpi_key(date_range)

    Rails.cache.fetch(cache_key, expires_in: TTL) do
      service = AnalyticsService.new(@account, date_range: date_range)
      result  = service.kpi_cards

      # Consistency check: verify summary matches detail
      verify_consistency!(result, date_range)

      result
    end
  end

  def invalidate!
    Rails.cache.delete_matched("analytics/#{@account.id}/*")
  end

  private

  def kpi_key(date_range)
    "analytics/#{@account.id}/kpi/#{date_range.first.to_date}-#{date_range.last.to_date}"
  end

  def verify_consistency!(result, date_range)
    breakdown_total = @account.billing_events
      .where(created_at: date_range)
      .group(:event_type)
      .sum(:amount_cents)
      .values
      .sum

    if result[:total_spend] != breakdown_total
      Rails.logger.warn(
        "analytics.consistency_mismatch",
        account_id:      @account.id,
        kpi_total:       result[:total_spend],
        breakdown_total: breakdown_total
      )
    end
  end
end
ruby 
# Graceful degradation in the controller
def kpi
  cache = AnalyticsCache.new(current_account)
  result = cache.fetch_kpi(date_range: parsed_range)
  render json: { kpi: result, cached: true }
rescue ActiveRecord::StatementInvalid, PG::Error => e
  Rails.logger.error("analytics.query_failed", error: e.message)

  # Fall back to last known good cache, even if expired
  stale = Rails.cache.read(cache.send(:kpi_key, parsed_range))
  if stale
    render json: { kpi: stale, stale: true }
  else
    render json: { error: "Analytics temporarily unavailable" }, status: :service_unavailable
  end
end
Key Insight

The verify_consistency! method cross-checks the KPI total against an independent breakdown sum. If they diverge, it logs a warning but does not fail the request. This is a canary — it tells you something is wrong before users notice. In production, wire this to your alerting system (Datadog, PagerDuty, etc.) so the on-call engineer investigates before it becomes a customer complaint.

What caching backend should I use?

For development and small deployments, Rails' built-in :memory_store works fine. For production, use Redis via the redis gem and config.cache_store = :redis_cache_store. Redis gives you shared caching across multiple app servers, TTL support, and the delete_matched pattern for selective invalidation. If you are already using Solid Cache (Rails 8), that is another excellent option that stores cache entries in your database.

Self-Check

Before moving to Chapter 08, honestly answer these questions. If any answer is "no", go back and fix it.

Can a billing event be modified after creation?

Try calling BillingEvent.first.update!(amount_cents: 0) in the Rails console. You should get a validation error from the immutability_guard. If the update succeeds, your guard is not working. Bonus: add a PostgreSQL trigger that raises an exception on UPDATE or DELETE for defense in depth.

Do the KPI card totals match the breakdown table sums?

Hit /api/v1/analytics/kpi and /api/v1/analytics/breakdown with the same date range. Sum the total_cents values from the breakdown response. It should exactly equal total_spend from the KPI response. If they diverge, check your verify_consistency! method and look for missing event types in the breakdown query.

Do your analytics queries use index scans in production-like conditions?

Seed your development database with at least 100,000 billing events across multiple accounts. Run EXPLAIN ANALYZE on your analytics queries. Look for Index Scan or Bitmap Index Scan in the output. If you see Seq Scan on the billing_events table, you need to add the composite index (account_id, created_at). Document the before/after query plans in your PR.
