Chapter 02 · Phase 1: Platform Foundations

Authentication and Tenant Isolation

Implement login and strict account scoping so customer data never crosses tenant boundaries. Before you build a single campaign or ad group, you need to know who is making the request and which account they belong to.

Estimated: 3–4 days
3 Advancement Levels
3 Assignments

Why This Matters

In a multi-tenant ad bidding platform, every row in every table belongs to someone. Campaigns, budgets, creatives, leads, invoices — they're all owned by an account. If Account A can read Account B's campaign data, you don't have a "bug" — you have a security breach.

Authentication answers the question "who are you?" Authorization answers "what are you allowed to do?" Tenant isolation answers "whose data can you see?" All three must be solved before a single business feature is safe to build. Every controller, every query, every background job in chapters 3–18 depends on the guarantees you establish here.

The Cost of Getting This Wrong

Cross-tenant data leaks are among the most expensive bugs in SaaS. They trigger mandatory breach notifications, destroy customer trust, and can result in regulatory fines. This chapter is not optional — it is the security foundation for everything that follows.

Key Concepts

1. Authentication vs. Authorization

These two words are often confused, but they solve completely different problems:

  • Authentication (AuthN) — Verifying identity. "This request is from user #42." Done via credentials (password, token, certificate).
  • Authorization (AuthZ) — Verifying permission. "User #42 is allowed to pause Campaign #99." Done via policies, roles, and scoping rules.

In our bidding platform, authentication happens once at login (or token refresh). Authorization happens on every single request. The two are layered — you cannot authorize a request from an unknown user.

2. JWT-Based Authentication

JSON Web Tokens (JWTs) are the standard way to authenticate API requests in a stateless architecture. Instead of server-side sessions, the server issues a signed token that the client stores and sends with every request.

A JWT has three parts, separated by dots: header.payload.signature. The header declares the algorithm. The payload carries claims (user ID, account ID, expiration). The signature proves the token wasn't tampered with.

ruby 
# Gemfile
gem "bcrypt", "~> 3.1"   # Password hashing
gem "jwt",    "~> 2.7"   # Token encoding/decoding
ruby 
# app/services/jwt_service.rb
class JwtService
  SECRET = Rails.application.credentials.secret_key_base
  ALGORITHM = "HS256"

  def self.encode(user)
    payload = {
      sub: user.id,
      acc: user.account_id,
      exp: 24.hours.from_now.to_i,
      iat: Time.current.to_i
    }
    JWT.encode(payload, SECRET, ALGORITHM)
  end

  def self.decode(token)
    decoded = JWT.decode(token, SECRET, true, algorithm: ALGORITHM)
    decoded.first.symbolize_keys
  rescue JWT::DecodeError, JWT::ExpiredSignature
    nil
  end
end
Key Insight

Notice the acc (account ID) claim in the payload. This is critical — it means every request carries both "who" and "which tenant" without an extra database lookup. The server verifies the signature is valid, trusts the claims, and scopes all queries accordingly.

3. The User-Account-Membership Model

A common mistake is putting account_id directly on the User model. This means a user can only belong to one account — ever. In reality, a person might manage campaigns for multiple clients. The solution is a three-table pattern:

ruby 
# app/models/user.rb
class User < ApplicationRecord
  has_secure_password
  has_many :memberships, dependent: :destroy
  has_many :accounts, through: :memberships

  validates :email, presence: true,
                     uniqueness: { case_sensitive: false },
                     format: { with: URI::MailTo::EMAIL_REGEXP }
end

# app/models/account.rb
class Account < ApplicationRecord
  has_many :memberships, dependent: :destroy
  has_many :users, through: :memberships
  has_many :campaigns  # Chapter 03

  validates :name, presence: true
end

# app/models/membership.rb
class Membership < ApplicationRecord
  belongs_to :user
  belongs_to :account

  enum :role, { viewer: 0, operator: 1, admin: 2 }

  validates :role, presence: true
  validates :user_id, uniqueness: { scope: :account_id }
end

 ┌──────────┐       ┌──────────────┐       ┌──────────┐
 │   User   │──────▶│  Membership  │◀──────│  Account  │
 │          │  1:N  │              │  N:1   │          │
 │ email    │       │ role:        │       │ name     │
 │ password │       │   viewer     │       │ slug     │
 │          │       │   operator   │       │          │
 │          │       │   admin      │       │ campaigns│
 └──────────┘       └──────────────┘       │ ad_groups│
                                           │ creatives│
                                           │ leads    │
                                           └──────────┘
Figure 2: User, Account, and Membership — the multi-tenancy foundation

The Membership join table carries the role column. This means the same person can be an admin on their own agency's account and a viewer on a client's account. Roles are per-membership, not per-user.

4. Tenant Resolution — Setting the Scope

Once you know who the user is, you need to determine which tenant's data they're working with. This is called tenant resolution. In our platform, the JWT's acc claim determines the current account. A before_action in the base controller sets the scope for every request:

ruby 
# app/controllers/application_controller.rb
class ApplicationController < ActionController::API
  before_action :authenticate_request!

  private

  def authenticate_request!
    token = request.headers["Authorization"]&.split(" ")&.last
    claims = JwtService.decode(token)

    unless claims
      render json: { error: "Unauthorized" }, status: :unauthorized
      return
    end

    @current_user = User.find_by(id: claims[:sub])
    @current_account = Account.find_by(id: claims[:acc])

    unless @current_user && @current_account
      render json: { error: "Unauthorized" }, status: :unauthorized
    end
  end

  def current_user    = @current_user
  def current_account = @current_account
end
Never Trust the Client to Scope Data

Don't accept account_id as a query parameter. Always resolve the tenant from the JWT's acc claim. A malicious user can change query params; they cannot forge a valid JWT signature.

5. Scoping Every Query to the Tenant

The most dangerous bug in a multi-tenant app is an unscoped query. If even one controller action calls Campaign.find(params[:id]) without scoping to the current account, any authenticated user can access any tenant's data by guessing IDs.

The fix is to always query through the account:

ruby 
# WRONG — Unscoped. Any user can access any campaign.
campaign = Campaign.find(params[:id])

# RIGHT — Scoped to current account. Cross-tenant access raises RecordNotFound.
campaign = current_account.campaigns.find(params[:id])

For extra safety, you can use a gem like acts_as_tenant which automatically adds a WHERE account_id = ? clause to every query for tenanted models. This way, even if a developer forgets to scope manually, the gem catches it.

How does acts_as_tenant work under the hood?

The gem uses a default_scope and a before_validation callback. When you call ActsAsTenant.current_tenant = account in a controller before_action, the gem stores it in a thread-local variable. Every model that declares acts_as_tenant :account then automatically adds WHERE account_id = #{current_tenant.id} to all queries and sets account_id on new records before validation.

ruby 
# app/models/campaign.rb
class Campaign < ApplicationRecord
  acts_as_tenant :account

  # Now Campaign.all automatically adds WHERE account_id = ?
  # and Campaign.create!(...) auto-sets account_id
end

6. Login, Logout, and Token Lifecycle

With the foundation in place, the auth controller is straightforward:

ruby 
# app/controllers/auth_controller.rb
class AuthController < ApplicationController
  skip_before_action :authenticate_request!, only: :login

  def login
    user = User.find_by(email: params[:email]&.downcase)

    unless user&.authenticate(params[:password])
      render json: { error: "Invalid credentials" }, status: :unauthorized
      return
    end

    membership = user.memberships.includes(:account).first
    unless membership
      render json: { error: "No account access" }, status: :forbidden
      return
    end

    token = JwtService.encode(user)
    render json: {
      token: token,
      user: { id: user.id, email: user.email },
      account: { id: membership.account.id, name: membership.account.name },
      role: membership.role
    }
  end

  def me
    membership = current_user.memberships
                             .find_by(account: current_account)
    render json: {
      user: { id: current_user.id, email: current_user.email },
      account: { id: current_account.id, name: current_account.name },
      role: membership&.role
    }
  end
end
ruby 
# config/routes.rb
Rails.application.routes.draw do
  post "/auth/login",  to: "auth#login"
  get  "/auth/me",     to: "auth#me"
  get  "/health",      to: "health#show"
end
Pro Tip: Secure Error Messages

Notice the login endpoint returns the same "Invalid credentials" message whether the email doesn't exist or the password is wrong. This prevents user enumeration attacks where an attacker probes for valid emails by observing different error messages.

7. Role-Based Authorization Policies

Once you know who the user is and which account they belong to, you still need to check what they're allowed to do. The Pundit gem provides a clean pattern for this using plain Ruby policy classes:

ruby 
# app/policies/campaign_policy.rb
class CampaignPolicy
  attr_reader :membership, :campaign

  def initialize(membership, campaign)
    @membership = membership
    @campaign   = campaign
  end

  def show?
    true  # Any member can view campaigns in their account
  end

  def create?
    membership.admin? || membership.operator?
  end

  def update?
    membership.admin? || membership.operator?
  end

  def destroy?
    membership.admin?
  end
end

Policy classes are pure Ruby — no DSL, no magic, easy to test. The convention is one policy per model, with a method for each action. A viewer can read but not modify. An operator can create and update campaigns. Only an admin can delete.

Why not use cancancan instead of Pundit?

Both work. cancancan centralizes all rules in a single Ability class, which can become unwieldy in a platform with many models. Pundit distributes policies across separate files (one per model), which scales better for team development. In a codebase where 5+ developers are working across campaigns, ad groups, creatives, and billing, Pundit's file-per-model approach causes fewer merge conflicts and is easier to review.

8. Secure Logging Without Leaking Secrets

Auth failures must be observable. If someone is brute-forcing passwords or probing for cross-tenant access, you need to know. But the logs must never contain passwords, tokens, or user PII.

ruby 
# config/initializers/filter_parameter_logging.rb
Rails.application.config.filter_parameters += [
  :password, :token, :secret, :authorization,
  :_key, :crypt, :salt, :certificate, :otp
]

# In your auth controller, log safely:
Rails.logger.info(
  "auth.login_failed",
  email_hash: Digest::SHA256.hexdigest(params[:email].to_s),
  ip: request.remote_ip
)
Key Insight

Log a hash of the email, not the email itself. This lets you correlate repeat failures ("same hash tried 47 times in 5 minutes") without storing PII in your log aggregator. Your incident response team can identify patterns without handling personal data.

Assignment 1
Level 1 — Foundation

JWT Login/Logout and Protected Routes

Build the authentication layer. Users should be able to log in with email and password and receive a JWT. All other endpoints should reject unauthenticated requests.

  • Add bcrypt and jwt gems to the Gemfile
  • Create User model with has_secure_password and email validation
  • Create Account and Membership models with role enum (viewer, operator, admin)
  • Implement JwtService with encode and decode methods
  • Build POST /auth/login that returns a JWT on valid credentials
  • Build GET /auth/me that returns current user info (requires valid token)
  • Verify: requests without a token receive 401 Unauthorized
  • Verify: requests with an expired token receive 401 Unauthorized
Done when

You can curl -X POST /auth/login with valid credentials and get a JWT. Using that token in the Authorization: Bearer header on /auth/me returns user info. Hitting /auth/me without a token returns 401.

Assignment 3
Level 3 — Production-Ready

Security Regression Tests for Cross-Tenant Attack Paths

Write automated tests that prove cross-tenant access is impossible. These tests become a safety net for every future chapter.

  • Write request specs: authenticated user from Account A hits Account B's campaign endpoint → receives 404 Not Found
  • Write request specs: expired JWT → 401
  • Write request specs: tampered JWT (modified payload, same signature) → 401
  • Write request specs: viewer attempts to create/delete campaigns → 403 Forbidden
  • Verify that failed login attempts are logged with hashed email and IP (no plaintext passwords)
  • Run the full test suite — all tests green, no cross-tenant leaks
Done when

The test suite includes at least 8 security-focused request specs. Cross-tenant access returns 404 (not 403 — don't reveal that the resource exists). JWT tampering and expiration are covered. A mentor can read your test names and understand every attack vector you've defended against.

Self-Check

Before moving to Chapter 03, honestly answer these questions. If any answer is "no", go back and fix it.

Do protected endpoints fail safely when auth context is missing?

Hit any endpoint without an Authorization header. You should get a clean 401 Unauthorized JSON response, not a 500 error or a stack trace. Now try with an invalid token (random string). Same result — 401. The error message should be generic ("Unauthorized") and never hint at why authentication failed.

Can an authenticated user access data from another tenant?

Create two accounts with different campaigns. Log in as Account A and try to access Account B's campaign by ID. You should get 404 Not Found, not 403 Forbidden. Why 404? Because returning 403 confirms the resource exists — it just says you can't access it. A 404 reveals nothing about other tenants' data.

Are auth failures observable in logs without leaking sensitive data?

Trigger a failed login. Check your Rails logs. You should see the event type (auth.login_failed), a hashed email, and the requester's IP address. You should not see the attempted password, the full email, or the JWT token. Run grep -i password log/development.log — if it returns anything, you have a leak.
